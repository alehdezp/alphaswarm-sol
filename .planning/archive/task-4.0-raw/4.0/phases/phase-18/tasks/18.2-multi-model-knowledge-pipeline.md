## Executive Summary

This task defines the **multi-model, multi-agent architecture** for building the world's most comprehensive Solidity vulnerability knowledge database. The system uses:

- **Opus 4.5**: High-level architect - organization, logic, semantic merging, quality decisions
- **Haiku**: Fast workers - crawling, summarization, extraction, initial processing
- **Parallel Subagents**: One per subcategory for maximum throughput

Every unique idea, scenario, and learning from all sources gets merged into a single, well-structured knowledge system optimized for LLM consumption.

## Model Responsibilities

### Opus 4.5 (Architect Model)

| Responsibility | Why Opus |
|---------------|----------|
| **Semantic Organization** | Requires deep understanding of vulnerability relationships |
| **Merge Conflict Resolution** | Needs judgment to decide which source is more accurate |
| **Quality Assessment** | Must evaluate if content adds new value |
| **Pattern Linkage** | Requires understanding of detection capabilities |
| **Taxonomy Design** | Needs to create logical category hierarchies |
| **Gap Identification** | Must recognize what's missing from coverage |
| **Final Document Structure** | Ensures coherence and completeness |

### Haiku (Worker Model)

| Responsibility | Why Haiku |
|---------------|-----------|
| **Content Crawling** | Fast, cheap, high volume |
| **Initial Summarization** | Extract key points from long documents |
| **Entity Extraction** | Pull out CVEs, function names, patterns |
| **Deduplication Check** | Quick similarity comparison |
| **Metadata Extraction** | Parse dates, authors, severity |
| **Code Block Extraction** | Identify vulnerable/safe code examples |
| **Category Classification** | Assign content to categories |

## Document Schema for LLM Consumption

### Core Document Structure

Every vulnerability knowledge document follows this schema, optimized for fast LLM understanding:

```python
@dataclass
class VulnKnowledgeDoc:
    """
    Single vulnerability knowledge document.
    Optimized for LLM consumption with clear sections.
    Target: 800-1500 tokens per document.
    """

    # === IDENTIFICATION (50 tokens) ===
    id: str                          # "reentrancy/classic/state-after-call"
    name: str                        # "State Write After External Call"
    category: str                    # "reentrancy"
    subcategory: str                 # "classic-reentrancy"
    severity: str                    # "critical" | "high" | "medium" | "low"
    prevalence: str                  # "very_common" | "common" | "rare"

    # === QUICK SUMMARY (100 tokens) ===
    one_liner: str                   # Single sentence description
    tldr: str                        # 2-3 sentence summary for quick understanding

    # === DETECTION (200 tokens) ===
    detection: DetectionSection

    # === EXPLOITATION (200 tokens) ===
    exploitation: ExploitationSection

    # === MITIGATION (200 tokens) ===
    mitigation: MitigationSection

    # === EXAMPLES (300 tokens) ===
    examples: ExamplesSection

    # === PATTERN LINKAGE (100 tokens) ===
    pattern_linkage: PatternLinkage

    # === METADATA (50 tokens) ===
    metadata: DocMetadata


@dataclass
class DetectionSection:
    """How to detect this vulnerability."""

    # Graph signals (what BSKG looks for)
    graph_signals: List[str]         # ["state_write_after_external_call", "no_reentrancy_guard"]

    # Operation sequence
    vulnerable_sequence: str         # "R:bal → X:out → W:bal (read, external, write)"
    safe_sequence: str               # "R:bal → W:bal → X:out (CEI pattern)"

    # Key indicators
    indicators: List[str]            # ["external call before state update", "no mutex"]

    # What to check
    checklist: List[str]             # ["Check call ordering", "Verify guard presence"]


@dataclass
class ExploitationSection:
    """How this vulnerability is exploited."""

    # Attack mechanism
    attack_vector: str               # How attacker exploits this

    # Prerequisites
    prerequisites: List[str]         # What attacker needs

    # Attack flow
    attack_steps: List[str]          # Step-by-step attack

    # Impact
    potential_impact: str            # What can be stolen/broken
    monetary_risk: str               # "high" | "critical" - funds at risk


@dataclass
class MitigationSection:
    """How to fix this vulnerability."""

    # Primary fix
    primary_fix: str                 # Main remediation approach

    # Alternative fixes
    alternative_fixes: List[str]     # Other valid approaches

    # Code pattern
    safe_pattern: str                # Name of safe pattern (e.g., "CEI")

    # Verification
    how_to_verify: List[str]         # How to confirm fix works


@dataclass
class ExamplesSection:
    """Real-world examples and code samples."""

    # Vulnerable code (minimal, illustrative)
    vulnerable_code: str             # 10-20 lines max
    vulnerable_code_explanation: str # What's wrong

    # Fixed code
    fixed_code: str                  # Same example, fixed
    fixed_code_explanation: str      # What changed

    # Real-world incidents
    real_exploits: List[RealExploit] # CVEs, hacks


@dataclass
class RealExploit:
    """A real-world exploit of this vulnerability."""
    name: str                        # "The DAO Hack"
    date: str                        # "2016-06-17"
    loss: str                        # "$60M"
    protocol: str                    # "The DAO"
    brief: str                       # One sentence description
    source_url: str                  # Link to analysis


@dataclass
class PatternLinkage:
    """Links this knowledge to BSKG detection patterns."""

    linkage_type: str                # "exact_match" | "partial_match" | "theoretical" | "requires_llm"

    # For exact/partial match
    pattern_ids: List[str]           # ["reentrancy-001", "reentrancy-002"]
    coverage_pct: float              # 0.95 = 95% of variants covered

    # For theoretical
    why_no_pattern: Optional[str]    # Why automated detection is hard
    manual_hints: List[str]          # What to look for manually

    # For requires_llm
    llm_context_needed: List[str]    # What context LLM needs


@dataclass
class DocMetadata:
    """Document metadata for tracking and retrieval."""

    # Sources
    sources: List[str]               # URLs this knowledge came from
    source_authority: float          # 0-1 weighted authority score

    # Freshness
    last_updated: str                # ISO date
    content_hash: str                # For change detection

    # Retrieval optimization
    keywords: List[str]              # For keyword search
    embedding_id: Optional[str]      # For vector search

    # Quality
    completeness_score: float        # 0-1 how complete
    confidence_score: float          # 0-1 how confident in accuracy
```

## Subagent Architecture

### Category Agent (Haiku)

Each of the 15 categories has a dedicated agent that:

```python
class CategoryAgent:
    """
    Haiku-powered agent for processing a single category.
    Spawns subcategory workers for parallel processing.
    """

    category: str                    # e.g., "reentrancy"
    model: str = "haiku"
    max_workers: int = 7             # One per subcategory

    async def process_category(self, sources: List[Source]) -> CategoryResult:
        """
        Process all sources for this category.

        1. Filter sources relevant to this category
        2. Spawn subcategory workers in parallel
        3. Collect and aggregate results
        4. Return to orchestrator for merging
        """

        # Spawn parallel workers for each subcategory
        workers = []
        for subcategory in self.subcategories:
            worker = SubcategoryWorker(
                category=self.category,
                subcategory=subcategory,
                model="haiku"
            )
            workers.append(worker.process(sources))

        # Run all workers in parallel
        results = await asyncio.gather(*workers)

        return CategoryResult(
            category=self.category,
            subcategory_results=results
        )
```

### Subcategory Worker (Haiku)

```python
class SubcategoryWorker:
    """
    Haiku-powered worker for processing a single subcategory.
    Does the heavy lifting of summarization and extraction.
    """

    category: str
    subcategory: str
    model: str = "haiku"

    async def process(self, sources: List[Source]) -> SubcategoryResult:
        """
        Process all sources for this subcategory.

        For each source:
        1. Extract relevant content
        2. Summarize key points
        3. Extract code examples
        4. Extract attack vectors
        5. Extract mitigations
        6. Compute content hash
        """

        summaries = []
        for source in sources:
            summary = await self.summarize_source(source)
            summaries.append(summary)

        return SubcategoryResult(
            subcategory=self.subcategory,
            summaries=summaries,
            unique_ideas=self.extract_unique_ideas(summaries)
        )

    async def summarize_source(self, source: Source) -> SourceSummary:
        """
        Use Haiku to create structured summary.

        Prompt template:
        ```
        Extract vulnerability knowledge from this content.
        Category: {category}
        Subcategory: {subcategory}

        Content:
        {content}

        Extract:
        1. Key points (5-10 bullets)
        2. Attack vector (how to exploit)
        3. Mitigation (how to fix)
        4. Code examples (vulnerable + fixed)
        5. Real incidents (CVEs, hacks, losses)

        Format as JSON.
        ```
        """
        pass
```

### Merge Orchestrator (Opus 4.5)

```python
class MergeOrchestrator:
    """
    Opus 4.5-powered orchestrator for semantic merging.
    Makes intelligent decisions about combining knowledge.
    """

    model: str = "opus"

    async def merge_subcategory(
        self,
        subcategory: str,
        summaries: List[SourceSummary]
    ) -> VulnKnowledgeDoc:
        """
        Merge all summaries into a single knowledge document.

        Opus responsibilities:
        1. Identify ALL unique ideas/scenarios across sources
        2. Detect conflicting information
        3. Resolve conflicts using source authority + reasoning
        4. Ensure no valuable information is lost
        5. Structure for optimal LLM consumption
        6. Validate completeness
        """

        # Step 1: Identify unique ideas
        unique_ideas = await self.identify_unique_ideas(summaries)

        # Step 2: Detect conflicts
        conflicts = await self.detect_conflicts(summaries)

        # Step 3: Resolve conflicts
        resolutions = await self.resolve_conflicts(conflicts)

        # Step 4: Semantic merge
        merged = await self.semantic_merge(
            unique_ideas=unique_ideas,
            resolutions=resolutions,
            summaries=summaries
        )

        # Step 5: Structure for LLM
        document = await self.structure_document(merged)

        # Step 6: Validate
        await self.validate_completeness(document, summaries)

        return document

    async def identify_unique_ideas(
        self,

    ) -> List[UniqueIdea]:
        """
        Use Opus to identify every distinct idea/scenario.

        Prompt template:
        ```
        Analyze these vulnerability knowledge summaries.

        Identify EVERY unique:
        - Attack scenario or variant
        - Detection approach
        - Mitigation technique
        - Edge case or gotcha
        - Real-world example

        Do NOT lose any valuable information.
        Even slightly different scenarios should be captured.

        Summaries:
        {summaries}

        Output unique ideas as JSON array.
        ```
        """
        pass

    async def resolve_conflicts(
        self,
        conflicts: List[Conflict]
    ) -> List[Resolution]:
        """

        Output:
        - resolution: Which claim to use
        - reasoning: Why this is correct
        - nuance: Any context where the other claim applies

class IdeaCaptureValidator:

class ScenarioDeduplicator:
    """
    Identifies truly duplicate vs subtly different scenarios.
    """

    async def are_scenarios_duplicate(
        self,
        scenario_a: str,
        scenario_b: str
    ) -> DuplicationResult:
        """
        Use Opus to determine if scenarios are truly duplicate.

        Prompt:
        ```
        Are these two vulnerability scenarios duplicates or distinct?

        Scenario A: {scenario_a}
        Scenario B: {scenario_b}

        Consider:
        - Different token types (ETH vs ERC20 vs ERC721)
        - Different contexts (DeFi vs NFT vs governance)
        - Different attack prerequisites
        - Different impact levels
        - Different code patterns
