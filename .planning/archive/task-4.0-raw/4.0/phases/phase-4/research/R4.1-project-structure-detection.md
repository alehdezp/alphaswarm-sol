## Objective

Research and document how to reliably detect Solidity project types (Foundry, Hardhat, Brownie, etc.) and their configuration.

## Research Questions

### 1. Foundry Detection
- [ ] What files indicate a Foundry project?
  - `foundry.toml` (primary indicator)
  - `lib/` directory with git submodules
  - `forge-std` dependency
- [ ] Where are remappings stored?
  - Inside `foundry.toml` under `[profile.default].remappings`
  - Separate `remappings.txt` file
  - Both can coexist (which takes precedence?)
- [ ] How does `forge` resolve imports?

### 2. Hardhat Detection
- [ ] What files indicate a Hardhat project?
  - `hardhat.config.js` or `hardhat.config.ts`
  - `package.json` with `hardhat` dependency
- [ ] Where are import paths configured?
  - `node_modules/@openzeppelin/contracts/`
  - Custom paths in `hardhat.config.js`
- [ ] How does Hardhat resolve `@openzeppelin/` imports?

### 3. Mixed/Unknown Projects
- [ ] How to handle projects with both `foundry.toml` and `hardhat.config.js`?
- [ ] What's the fallback when no config is found?
- [ ] How to detect Brownie projects (`brownie-config.yaml`)?

### 4. Remapping Patterns
- [ ] Common remapping patterns:
  ```
  @openzeppelin/=lib/openzeppelin-contracts/
  @openzeppelin/contracts/=node_modules/@openzeppelin/contracts/
  forge-std/=lib/forge-std/src/
  ```
- [ ] How to parse `foundry.toml` remappings section?
- [ ] How to resolve transitive dependencies?
