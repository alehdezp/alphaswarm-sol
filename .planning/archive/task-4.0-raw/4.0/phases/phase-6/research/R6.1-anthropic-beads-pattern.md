## What is the Beads Pattern?

The "Beads" pattern is a design approach for structuring LLM context that treats each unit of work as a self-contained "bead" with everything needed for the AI to act independently.

### Core Principles

1. **Self-Containment**: Each bead contains ALL information needed - no external lookups required
2. **Structured Context**: Information organized in predictable sections
3. **Task Clarity**: Clear objective and success criteria within the bead
4. **Evidence-Based**: Supporting data included, not just claims
5. **Handoff-Ready**: Can be passed to any capable agent without loss

### Bead Structure (General)

```
BEAD
├── Identity (what is this bead)
├── Context (background information)
├── Task (what needs to be done)
├── Resources (data/code/examples needed)
├── Guidance (how to approach the task)
├── Constraints (limits and requirements)
└── Expected Output (what success looks like)
```

## Research Questions

### Q1: How does the Beads pattern relate to vulnerability investigation?

**To Research:**
- What context does an auditor need to verify a finding?
- How do existing audit workflows structure this context?
- What makes investigation context "complete"?

**Initial Thoughts:**
- Auditors need: code, pattern explanation, similar examples, test scaffold
- Current tools provide findings but not investigation guidance
- "Complete" = LLM can decide TP/FP without asking questions

### Q2: What sections should a VulnerabilityBead contain?

**To Research:**
- Review audit report formats
- Look at how auditors document findings
- Identify minimum viable context

**Initial Mapping:**
| General Bead Section | BSKG Vulnerability Bead |
|---------------------|------------------------|
| Identity | Bead ID, pattern ID, severity |
| Context | Vulnerable code, related code, contract info |
| Task | Verify if TP/FP, assess exploitability |
| Resources | Similar exploits, test scaffold |
| Guidance | Investigation steps, questions to answer |
| Constraints | What's in scope, what to ignore |
| Expected Output | Verdict with confidence and reasoning |

### Q3: How to handle incomplete context?

**To Research:**
- What if code is too long?
- What if related code is missing?
- How to summarize without losing signal?

**Strategies to Explore:**
1. Progressive disclosure (start minimal, expand if needed)
2. Intelligent truncation (keep critical parts)
3. Summary + reference (summarize with pointer to full code)
4. Request-based expansion (LLM asks for more)

### Q4: How to validate bead completeness?

**To Research:**
- What metrics indicate sufficient context?
- How to test without human in loop?
- What's acceptable accuracy threshold?

**Validation Approaches:**
1. LLM accuracy on known-verdict findings
2. Follow-up question rate (fewer = better)
3. Confidence calibration
4. A/B test different bead structures

## Recommendations

### Bead Structure for VKG

Based on research, recommend the following VulnerabilityBead structure:

```
VulnerabilityBead
├── IDENTITY
│   ├── id: VKG-001
│   ├── vulnerability_class: reentrancy
│   ├── pattern_id: vm-001
│   ├── severity: critical
│   └── confidence: 0.95
│
├── CODE CONTEXT
│   ├── vulnerable_code: function + surrounding lines
│   ├── related_code: called functions, modifiers
│   ├── full_contract: if small enough
│   └── inheritance_chain: parent contracts
│
├── PATTERN CONTEXT
│   ├── pattern_name: "Classic Reentrancy"
│   ├── pattern_description: what it detects
│   ├── why_flagged: specific reason this code matched
│   └── evidence_lines: [48, 50]
│
├── INVESTIGATION GUIDANCE
│   ├── steps: ordered investigation steps
│   ├── questions: what to answer
│   ├── common_fps: known false positive patterns
│   └── key_indicators: what confirms real vuln
│
├── HISTORICAL CONTEXT
│   ├── similar_exploits: real-world examples
│   └── fix_recommendations: how to remediate
│
├── TEST CONTEXT
│   ├── test_scaffold: Foundry test template
│   └── attack_scenario: exploit description
│
└── STATUS
    ├── status: pending/investigating/confirmed/rejected
    ├── verdict: final determination
    └── notes: investigation notes
```

### Context Size Guidelines

| Section | Target Size | Max Size | Notes |
|---------|------------|----------|-------|
| vulnerable_code | 20-50 lines | 100 lines | Include full function |
| related_code | 5 snippets | 10 snippets | Prioritize by relevance |
| full_contract | Skip if > 500 lines | 500 lines | Include if small |
| investigation_steps | 5 steps | 10 steps | Class-specific |
| similar_exploits | 2-3 exploits | 5 exploits | Most relevant |

### Completeness Validation

Recommend validation via:
1. **Field completeness**: All required fields populated
2. **Size adequacy**: Enough context (not too truncated)
3. **LLM accuracy test**: 80%+ correct verdicts
