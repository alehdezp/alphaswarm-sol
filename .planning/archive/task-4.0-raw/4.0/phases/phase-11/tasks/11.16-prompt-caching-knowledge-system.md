## 1. OBJECTIVE

Implement a comprehensive prompt caching system with deep vulnerability knowledge that LLM agents can leverage for smart contract security analysis. The system must provide:

1. **Prompt Caching Infrastructure** - Cache stable context blocks to reduce token costs
2. **Vulnerability Knowledge Grimoires** - Per-category expert knowledge documents
3. **Business Logic Context** - Why vulnerabilities matter, attack scenarios, misconfigurations

**Key Principle:** The pattern YAML is not the only knowledge. LLMs need rich context about:
- Why this vulnerability category is important
- All known attack scenarios and variants
- Common misconfigurations in libraries/tokens
- Historical exploits with losses
- Best practices and safe patterns

### 4.3 Example: Reentrancy Knowledge Grimoire

```markdown
# Reentrancy Vulnerability Knowledge Grimoire

## Overview
Reentrancy occurs when an external call allows an attacker to re-enter the calling
contract before the first invocation is complete, potentially draining funds or
corrupting state.

## Business Impact
- **Historical Losses:** $3.6B+ (The DAO: $60M, Curve 2023: $70M, Euler: $197M)
- **Target Protocols:** Any protocol with ETH/token transfers before state updates
- **Severity Range:** Critical (fund drain) to Medium (state corruption)

## Attack Scenarios

### Scenario 1: Classic Withdraw Reentrancy
**Conditions:**
- Public/external function transfers ETH or tokens
- Balance/state updated AFTER the transfer
- No reentrancy guard

**Mechanism:**
1. Attacker calls withdraw()
2. Contract sends ETH to attacker
3. Attacker's receive() calls withdraw() again
4. Contract still shows old balance (not updated yet)
5. Repeat until drained

**Example Exploit:**
```solidity
// VULNERABLE
function withdraw() public {
    uint bal = balances[msg.sender];
    (bool success,) = msg.sender.call{value: bal}("");
    require(success);
    balances[msg.sender] = 0;  // Updated AFTER call - vulnerable!
}

// ATTACKER
receive() external payable {
    if (address(target).balance > 0) {
        target.withdraw();  // Re-enter!
    }
}
```

**Real-World Instance:** The DAO (June 2016, $60M)

### Scenario 2: Cross-Function Reentrancy
**Conditions:**
- Multiple functions share state
- External call in one function
- Another function reads/writes shared state

**Mechanism:**
1. Attacker calls functionA() which has external call
2. During callback, attacker calls functionB()
3. functionB() sees inconsistent state
4. Attacker exploits state inconsistency

**Example:**
```solidity
// Function A - has external call
function withdraw() public {
    uint bal = balances[msg.sender];
    msg.sender.call{value: bal}("");  // <-- callback here
    balances[msg.sender] = 0;
}

// Function B - reads same state
function transfer(address to, uint amount) public {
    require(balances[msg.sender] >= amount);  // Still shows old balance!
    balances[msg.sender] -= amount;
    balances[to] += amount;
}
```

**Real-World Instance:** Cream Finance (2021)

### Scenario 3: Read-Only Reentrancy
**Conditions:**
- Protocol reads state from external contract during callback
- External contract's state is stale during callback

**Mechanism:**
1. Attacker calls Protocol A which calls Protocol B
2. Protocol B has callback to attacker
3. Attacker reads Protocol A's view function
4. View function returns stale data
5. Attacker uses stale data in Protocol B

**Example:**
```solidity
// Protocol A - Curve pool
function remove_liquidity() {
    // Burns LP tokens
    // Sends ETH (callback!)
    // Updates internal accounting AFTER
}

// Protocol B - Uses Curve for pricing
function getPrice() view returns (uint) {
    return curve.get_virtual_price();  // Stale during reentrancy!
}
```

**Real-World Instance:** Curve/Vyper reentrancy (July 2023, $70M)

### Scenario 4: ERC777 Token Callback Reentrancy
**Conditions:**
- Contract accepts ERC777 tokens
- Token operations before state updates
- No reentrancy guard

**Mechanism:**
ERC777 tokens have hooks that call back to sender/receiver.

**Real-World Instance:** imBTC Uniswap (2020)

## False Positive Indicators

### 1. Has Reentrancy Guard
```solidity
modifier nonReentrant() {
    require(!locked);
    locked = true;
    _;
    locked = false;
}
```
**Why Safe:** Guard prevents re-entry entirely.

### 2. Follows CEI Pattern
```solidity
function withdraw() public {
    uint bal = balances[msg.sender];
    balances[msg.sender] = 0;        // Effect BEFORE
    msg.sender.call{value: bal}("");  // Interaction AFTER
}
```
**Why Safe:** State updated before external call.

### 3. Uses pull-over-push
```solidity
function claimReward() public {
    uint reward = pendingRewards[msg.sender];
    pendingRewards[msg.sender] = 0;
    token.transfer(msg.sender, reward);  // Safe: state already updated
}
```
**Why Safe:** User initiates transfer after state update.

### 4. View/Pure Function
View functions that only read state cannot be exploited for reentrancy
(but can READ stale state - see read-only reentrancy).

### 5. Uses transfer() or send()
```solidity
payable(msg.sender).transfer(amount);  // Only 2300 gas - can't reenter
```
**Why Safe:** 2300 gas insufficient for reentry.
**Warning:** May fail with contract wallets, considered deprecated pattern.

## Edge Cases

### 1. Low-Level Call with Insufficient Gas
```solidity
msg.sender.call{gas: 5000, value: amt}("");
```
If gas is too low for meaningful operations, reentry may be impossible.
**Severity:** Reduced but not eliminated.

### 2. Access Control on Reentrant Functions
If functions requiring owner/admin access, random users can't exploit.
**Severity:** Lower (requires compromised privileged account).

### 3. Single-Block Limitations
Some reentrancy attacks require multiple transactions in one block.
**Consideration:** L2s with fast blocks may have different risk profile.

## Library/Token-Specific Pitfalls

### OpenZeppelin ReentrancyGuard
- **Correct Usage:** Apply to ALL state-modifying external-calling functions
- **Common Mistake:** Only applying to "obvious" functions
- **Version Note:** v4.x uses `_status` variable

### ERC777 Tokens
- Have `tokensReceived()` hook - can reenter on any ERC777 transfer
- Must use reentrancy guard when handling ERC777

### Safe ERC20 (safeTransfer)
- Does NOT protect against reentrancy
- Only protects against missing return values
- Still need CEI or guard

### Uniswap V2
- `swap()` callbacks to `uniswapV2Call()`
- Must handle reentrancy in callback

### Uniswap V3
- Flash loans callback to `uniswapV3FlashCallback()`
- `swap()` callbacks to `uniswapV3SwapCallback()`

## Safe Patterns

### Pattern 1: Checks-Effects-Interactions
```solidity
function withdraw(uint amount) public {
    // CHECKS
    require(balances[msg.sender] >= amount, "Insufficient balance");

    // EFFECTS
    balances[msg.sender] -= amount;

    // INTERACTIONS
    (bool success,) = msg.sender.call{value: amount}("");
    require(success, "Transfer failed");
}
```

### Pattern 2: Reentrancy Guard
```solidity
import "@openzeppelin/contracts/security/ReentrancyGuard.sol";

contract Safe is ReentrancyGuard {
    function withdraw(uint amount) public nonReentrant {
        require(balances[msg.sender] >= amount);
        (bool success,) = msg.sender.call{value: amount}("");
        require(success);
        balances[msg.sender] -= amount;
    }
}
```

### Pattern 3: Pull Payment
```solidity
mapping(address => uint) public pendingWithdrawals;

function withdraw() public {
    uint amount = pendingWithdrawals[msg.sender];
    pendingWithdrawals[msg.sender] = 0;  // Clear BEFORE transfer
    payable(msg.sender).transfer(amount);
}
```

## Verification Checklist

1. [ ] Is there an external call (call, delegatecall, transfer, token transfer)?
2. [ ] Is state modified AFTER the external call?
3. [ ] Is there a reentrancy guard modifier?
4. [ ] Does the contract follow CEI pattern?
5. [ ] Can the callback reach other state-modifying functions?
6. [ ] Does the contract handle ERC777 tokens?
7. [ ] Are there view functions that could return stale data during callback?

## Related Patterns
- `vm-001-classic`: Classic reentrancy detection
- `vm-002-unprotected-transfer`: Value transfer without protection
- `token-002-erc777-reentrancy`: ERC777-specific reentrancy
```

### 5.1 Knowledge Loader

```python
# src/true_vkg/llm/knowledge.py

from dataclasses import dataclass
from typing import Dict, List, Optional
from pathlib import Path
import hashlib

@dataclass
class KnowledgeBlock:
    """A cacheable knowledge block."""
    id: str
    category: str
    content: str
    token_estimate: int
    cache_key: str

    @classmethod
    def from_markdown(cls, path: Path) -> "KnowledgeBlock":
        """Load knowledge block from markdown file."""
        content = path.read_text()
        category = path.stem
        cache_key = hashlib.sha256(content.encode()).hexdigest()[:16]

        # Estimate tokens (rough: 4 chars per token)
        token_estimate = len(content) // 4

        return cls(
            id=f"kb-{category}",
            category=category,
            content=content,
            token_estimate=token_estimate,
            cache_key=cache_key,
        )

class KnowledgeManager:
    """Manage vulnerability knowledge for LLM prompts."""

    def __init__(self, knowledge_dir: str = "knowledge/grimoires"):
        self.knowledge_dir = Path(knowledge_dir)
        self._blocks: Dict[str, KnowledgeBlock] = {}
        self._load_all()

    def _load_all(self) -> None:
        """Load all knowledge blocks."""
        for path in self.knowledge_dir.glob("*.md"):
            if path.name != "index.md":
                block = KnowledgeBlock.from_markdown(path)
                self._blocks[block.category] = block

    def get_for_pattern(self, pattern_id: str) -> Optional[KnowledgeBlock]:
        """Get knowledge block for a pattern."""
        # Map pattern ID to category
        category = self._pattern_to_category(pattern_id)
        return self._blocks.get(category)

    def get_cacheable_system_prompt(
        self,
        categories: List[str]
    ) -> str:
        """Get combined knowledge blocks for caching in system prompt."""
        blocks = []
        for cat in categories:
            if cat in self._blocks:
                blocks.append(self._blocks[cat].content)

        return "\n\n---\n\n".join(blocks)

    def _pattern_to_category(self, pattern_id: str) -> str:
        """Map pattern ID to knowledge category."""
        mappings = {
            "vm-": "reentrancy",
            "auth-": "access-control",
            "oracle-": "oracle-manipulation",
            "token-": "token-integration",
            "upgrade-": "upgrade-safety",
            "dos-": "dos",
            "mev-": "mev",
            "crypto-": "cryptographic",
        }
        for prefix, category in mappings.items():
            if pattern_id.startswith(prefix):
                return category
        return "general"
```

### 5.2 Cached Prompt Builder

```python
# src/true_vkg/llm/cached_prompt.py

from dataclasses import dataclass
from typing import List, Dict, Any, Optional
from true_vkg.llm.knowledge import KnowledgeManager

@dataclass
class CachedPromptContext:
    """Context for building cached prompts."""
    knowledge_categories: List[str]  # Which knowledge to cache
    pattern_context: str             # Pattern-specific context
    code_context: str                # Sliced code context
    query: str                       # Specific analysis query

class CachedPromptBuilder:
    """Build prompts optimized for API caching."""

    def __init__(self, knowledge_manager: KnowledgeManager):
        self.knowledge = knowledge_manager
        self._session_cache_key: Optional[str] = None

    def build_anthropic_format(
        self,
        ctx: CachedPromptContext
    ) -> Dict[str, Any]:
        """Build prompt in Anthropic's cache-friendly format."""

        # Get cacheable knowledge block
        knowledge_content = self.knowledge.get_cacheable_system_prompt(
            ctx.knowledge_categories
        )

        return {
            "system": [
                {
                    "type": "text",
                    "text": self._build_system_prompt(knowledge_content),
                    "cache_control": {"type": "ephemeral"}
                }
            ],
            "messages": [
                {
                    "role": "user",
                    "content": self._build_user_prompt(ctx)
                }
            ]
        }

    def build_openai_format(
        self,
        ctx: CachedPromptContext
    ) -> List[Dict[str, str]]:
        """Build prompt in OpenAI format (uses automatic prefix caching)."""

        knowledge_content = self.knowledge.get_cacheable_system_prompt(
            ctx.knowledge_categories
        )

        return [
            {
                "role": "system",
                "content": self._build_system_prompt(knowledge_content)
            },
            {
                "role": "user",
                "content": self._build_user_prompt(ctx)
            }
        ]

    def _build_system_prompt(self, knowledge: str) -> str:
        """Build the system prompt with knowledge."""
        return f"""You are an expert smart contract security auditor. You analyze Solidity code for vulnerabilities with deep expertise in DeFi security.

## Vulnerability Knowledge Base

{knowledge}

## Your Role

When analyzing code:
1. Consider ALL attack scenarios from the knowledge base
2. Check for false positive indicators before flagging
3. Consider edge cases that might change severity
4. Reference real-world exploits when relevant
5. Provide specific, actionable remediation

Your analysis must be:
- Precise: Only flag real vulnerabilities
- Complete: Check all relevant scenarios
- Actionable: Provide clear fix recommendations
"""

    def _build_user_prompt(self, ctx: CachedPromptContext) -> str:
        """Build the user prompt with context."""
        return f"""## Pattern Context
{ctx.pattern_context}

## Code Under Analysis
```solidity
{ctx.code_context}
```

## Analysis Request
{ctx.query}

Respond with a structured JSON analysis:
{{
    "verdict": "VULNERABLE" | "SAFE" | "UNCERTAIN",
    "confidence": 0-100,
    "severity": "CRITICAL" | "HIGH" | "MEDIUM" | "LOW" | "INFO",
    "attack_scenario": "Description of how this could be exploited",
    "false_positive_check": "Which FP indicators were checked and results",
    "evidence": ["List of specific code evidence"],
    "remediation": "Specific fix recommendation",
    "related_exploits": ["Any similar real-world exploits"]
}}
"""
```
