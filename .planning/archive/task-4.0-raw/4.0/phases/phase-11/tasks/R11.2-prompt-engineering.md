## Prompt Templates to Develop

### 1. Finding Verification Prompt

```
SYSTEM:
You are a smart contract security auditor. Your task is to verify whether
a potential vulnerability identified by static analysis is a true positive
or false positive.

RULES:
1. Only analyze the provided code - do not assume behavior not shown
2. Cite specific lines of code as evidence
3. Consider defensive patterns that may mitigate the issue
4. If uncertain, say so - do not guess
5. Respond ONLY with valid JSON

USER:
## Finding
- Pattern: {pattern_id}
- Location: {file}:{line}
- Description: {description}
- Confidence: {confidence}

## Vulnerable Function
```solidity
{function_code}
```

## Context
- Callers: {callers}
- State Reads: {state_reads}
- State Writes: {state_writes}
- External Calls: {external_calls}

## Question
Is this vulnerability valid? Explain why or why not.

## Required Response Format
{
  "verdict": "VULNERABLE" | "SAFE" | "UNCERTAIN",
  "confidence": 0-100,
  "reasoning": "explanation...",
  "evidence": ["line 45: ...", "line 52: ..."],
  "mitigations_found": ["list of protections if any"]
}
```

### 2. Business Logic Analysis Prompt

```
SYSTEM:
You are analyzing a smart contract function for business logic vulnerabilities
that static patterns cannot detect. Focus on:
- Incorrect state transitions
- Missing access controls
- Economic attacks
- Edge case handling

USER:
## Function Summary
- Name: {function_name}
- Visibility: {visibility}
- Purpose: {intent_annotation or "Unknown"}

## Full Function
```solidity
{full_code}
```

## State Variables Accessed
{state_variables}

## Question
Identify any business logic issues. Consider:
1. Are there missing checks?
2. Can state become inconsistent?
3. Are there economic exploits possible?

## Response Format
{
  "issues": [
    {
      "type": "economic_attack" | "state_inconsistency" | "missing_check" | "edge_case",
      "severity": "critical" | "high" | "medium" | "low",
      "description": "...",
      "evidence": ["line X: ..."],
      "exploit_scenario": "how attacker would exploit",
      "fix_suggestion": "how to fix"
    }
  ],
  "safe_aspects": ["list of things correctly implemented"]
}
```

### 3. Test Scaffold Generation Prompt

```
SYSTEM:
Generate a Foundry test that would trigger the vulnerability.
The test should:
1. Be compilable Solidity
2. Use Foundry test conventions
3. Include setup and assertions
4. Be safe to review (not actually exploit)

USER:
## Vulnerability
{finding_details}

## Contract Interface
{interface}

## Generate
A Foundry test that demonstrates this vulnerability.

## Response Format
{
  "test_code": "// SPDX-License-Identifier: MIT\npragma solidity ^0.8.0;\n...",
  "setup_notes": "explanation of test setup",
  "expected_result": "what the test proves"
}
```

## Few-Shot Examples to Create

Create examples for each major vulnerability class:

### Reentrancy Examples

```json
{
  "class": "reentrancy",
  "examples": [
    {
      "input": "function withdraw() { balances[msg.sender] = 0; payable(msg.sender).transfer(amount); }",
      "output": {
        "verdict": "SAFE",
        "reasoning": "State is updated before external call (CEI pattern)",
        "evidence": ["Line 1: balance zeroed before transfer"]
      }
    },
    {
      "input": "function withdraw() { payable(msg.sender).transfer(amount); balances[msg.sender] = 0; }",
      "output": {
        "verdict": "VULNERABLE",
        "reasoning": "External call before state update allows reentrancy",
        "evidence": ["Line 1: transfer before balance update"]
      }
    }
  ]
}
```

### Access Control Examples

```json
{
  "class": "access_control",
  "examples": [
    {
      "input": "function setOwner(address newOwner) public { owner = newOwner; }",
      "output": {
        "verdict": "VULNERABLE",
        "reasoning": "No access control - anyone can change owner",
        "evidence": ["Line 1: public function with no modifier"]
      }
    },
    {
      "input": "function setOwner(address newOwner) public onlyOwner { owner = newOwner; }",
      "output": {
        "verdict": "SAFE",
        "reasoning": "Protected by onlyOwner modifier",
        "evidence": ["Line 1: onlyOwner modifier present"]
      }
    }
  ]
}
```
