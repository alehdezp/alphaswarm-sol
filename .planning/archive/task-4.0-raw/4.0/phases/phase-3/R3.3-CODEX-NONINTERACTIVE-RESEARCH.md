## Executive Summary

OpenAI Codex CLI provides a robust noninteractive mode (`codex exec`) specifically designed for automation and CI/CD integration. Key findings:

- **`codex exec`** runs Codex without the interactive TUI, streaming progress to stderr and final output to stdout
- **`--output-schema`** enforces JSON Schema-validated structured output, ideal for downstream automation
- **`--json`** enables JSON Lines streaming for real-time event monitoring
- **MCP integration** allows BSKG to be exposed as a tool that Codex can invoke
- **GitHub Action** (`openai/codex-action@v1`) provides first-class CI/CD support

**VKG Integration Paths**:
1. **MCP Tool**: Expose BSKG as an MCP server that Codex can call during audits
2. **Output Schema Consumer**: BSKG consumes Codex structured output for further analysis
3. **Prompt Provider**: BSKG generates specialized audit prompts for Codex execution

## Codex CLI Overview

### What is Codex CLI?

OpenAI Codex CLI is a terminal-based AI coding agent that:
- Reads and understands repository context
- Edits files and executes shell commands
- Operates with configurable sandbox permissions
- Supports both interactive and noninteractive modes

### Installation

```bash
npm install -g @openai/codex
# or
npx @openai/codex
```

### Core Features

| Feature | Description |
|---------|-------------|
| **File Operations** | Read, write, and modify files in the working directory |
| **Command Execution** | Run shell commands with approval controls |
| **Code Review** | `/review` command for PR analysis |
| **MCP Integration** | Connect to external tools via Model Context Protocol |
| **Web Search** | Optional first-party search capability |
| **Session Resume** | Continue previous conversations with `resume` command |

### Model Selection

- Default: `gpt-5-codex` (macOS/Linux) or `gpt-5` (Windows)
- Switchable via `--model` flag or config
- Supports reasoning effort configuration (`minimal | low | medium | high | xhigh`)

## Noninteractive Mode (`codex exec`)

### Purpose

`codex exec` (alias: `codex e`) enables headless execution for:
- CI/CD pipelines and automation
- Pre-merge checks and quality gates
- Scripted workflows with piped output
- Batch processing with explicit sandbox settings

### Command Syntax

```bash
codex exec "<task-prompt>"
```

### Output Streams

| Stream | Content |
|--------|---------|
| **stderr** | Progress updates while Codex runs |
| **stdout** | Final agent message only (for piping) |

### Key Command Line Options

#### Permissions & Sandbox Settings

| Option | Purpose |
|--------|---------|
| `--full-auto` | Allow edits during execution without prompts |
| `--sandbox <mode>` | `read-only` (default), `workspace-write`, `danger-full-access` |
| `--skip-git-repo-check` | Allow running outside Git repositories |
| `--add-dir <path>` | Grant additional directories write access |

#### Output Formatting

| Option | Purpose |
|--------|---------|
| `--json` | Enable JSON Lines streaming output |
| `-o, --output-last-message <path>` | Write final message to file |
| `--output-schema <path>` | Validate response against JSON Schema |

#### Model & Configuration

| Option | Purpose |
|--------|---------|
| `--model, -m <model>` | Override configured model |
| `--profile, -p <name>` | Load configuration profile |
| `--config, -c <key=value>` | Override configuration values |

### Complete Example

```bash
# Run a security audit with structured output
codex exec "Analyze the Solidity contracts for reentrancy vulnerabilities" \
  --sandbox read-only \
  --output-schema audit-schema.json \
  -o audit-results.json \
  --json
```

### Session Resume

```bash
# Resume last session
codex exec resume --last "continue analyzing the remaining contracts"

# Resume specific session
codex exec resume <SESSION_ID> "focus on access control issues"
```

## Structured Output with `--output-schema`

### Overview

The `--output-schema` option enforces that Codex's final response conforms to a specified JSON Schema. This enables:
- Predictable output structure for automation
- Type-safe downstream processing
- Validation of required fields
- Elimination of response parsing ambiguity

### JSON Schema Format

Create a schema file defining the expected response structure:

**Example: `vkg-audit-schema.json`**

```json
{
  "$schema": "http://json-schema.org/draft-07/schema#",
  "type": "object",
  "required": ["findings", "summary", "risk_score"],
  "properties": {
    "findings": {
      "type": "array",
      "items": {
        "type": "object",
        "required": ["id", "severity", "title", "description", "location"],
        "properties": {
          "id": {
            "type": "string",
            "description": "Unique finding identifier"
          },
          "severity": {
            "type": "string",
            "enum": ["critical", "high", "medium", "low", "informational"]
          },
          "title": {
            "type": "string"
          },
          "description": {
            "type": "string"
          },
          "location": {
            "type": "object",
            "properties": {
              "file": { "type": "string" },
              "line_start": { "type": "integer" },
              "line_end": { "type": "integer" },
              "function": { "type": "string" }
            },
            "required": ["file"]
          },
          "confidence": {
            "type": "number",
            "minimum": 0,
            "maximum": 1
          },
          "recommendation": {
            "type": "string"
          },
          "cwe_id": {
            "type": "string"
          }
        }
      }
    },
    "summary": {
      "type": "object",
      "properties": {
        "total_findings": { "type": "integer" },
        "critical": { "type": "integer" },
        "high": { "type": "integer" },
        "medium": { "type": "integer" },
        "low": { "type": "integer" },
        "informational": { "type": "integer" }
      }
    },
    "risk_score": {
      "type": "number",
      "minimum": 0,
      "maximum": 100
    },
    "contracts_analyzed": {
      "type": "array",
      "items": { "type": "string" }
    },
    "analysis_metadata": {
      "type": "object",
      "properties": {
        "timestamp": { "type": "string" },
        "model": { "type": "string" },
        "vkg_version": { "type": "string" }
      }
    }
  }
}
```

### Usage

```bash
codex exec "Perform a security audit of the Solidity contracts" \
  --output-schema vkg-audit-schema.json \
  -o audit-results.json
```

### JSON Lines Mode (`--json`)

When `--json` is enabled, stdout becomes a JSON Lines stream with event types:

```json
{"type":"thread.started","thread_id":"0199a213-81c0-7800-8aa1-bbab2a035a53"}
{"type":"turn.started"}
{"type":"item.started","item":{"id":"item_1","type":"command_execution"}}
{"type":"item.completed","item":{"id":"item_1","type":"command_execution","exit_code":0}}
{"type":"item.completed","item":{"id":"item_3","type":"agent_message","text":"Analysis complete..."}}
{"type":"turn.completed"}
```

**Event Types**:
- `thread.started` / `thread.completed`
- `turn.started` / `turn.completed` / `turn.failed`
- `item.started` / `item.completed`
- `error`

**Item Types**:
- Agent messages
- Reasoning steps
- Command executions
- File changes
- MCP tool calls
- Web searches
- Plan updates

### Integration Approaches

#### Approach 1: BSKG as MCP Tool Server

VKG exposes its capabilities as an MCP server that Codex can invoke:

```toml
# ~/.codex/config.toml
[mcp_servers.vkg-security]
command = "uv"
args = ["run", "alphaswarm", "mcp-server", "--stdio"]
startup_timeout_sec = 30
tool_timeout_sec = 120
```

**Tools to Expose**:

| Tool | Description | Parameters |
|------|-------------|------------|
| `vkg_build_graph` | Build knowledge graph from contracts | `path: string` |
| `vkg_query` | Execute VQL2 or natural language query | `query: string, graph_path?: string` |
| `vkg_run_patterns` | Run pattern pack against graph | `patterns: string[], graph_path?: string` |
| `vkg_get_findings` | Get findings with evidence | `severity?: string, limit?: int` |
| `vkg_explain` | Get detailed explanation of finding | `finding_id: string` |

#### Approach 2: BSKG as Prompt Generator

VKG generates context-rich prompts for Codex execution:

```bash
# BSKG generates audit prompt with context
uv run alphaswarm generate-codex-prompt contracts/ > audit-prompt.md

# Codex executes with VKG-generated context
codex exec --output-schema vkg-schema.json < audit-prompt.md
```

#### Approach 3: Hybrid Pipeline

```bash
# Step 1: BSKG builds graph and identifies high-risk areas
uv run alphaswarm build-kg contracts/ --output graph.json
uv run alphaswarm query "functions with critical risk" --compact > high-risk.json

# Step 2: Codex performs deep analysis on high-risk areas
codex exec "Analyze these high-risk functions for exploitability: $(cat high-risk.json)" \
  --output-schema exploit-analysis-schema.json \
  -o exploit-analysis.json

# Step 3: BSKG incorporates Codex findings
uv run alphaswarm import-findings exploit-analysis.json --source codex
```

## Example Commands for BSKG Audits

### Basic Security Audit

```bash
codex exec "Perform a comprehensive security audit of the Solidity contracts in ./contracts/. \
Focus on: reentrancy, access control, oracle manipulation, and token handling. \
Use the BSKG knowledge graph for context." \
  --sandbox read-only \
  --output-schema vkg-audit-schema.json \
  -o audit-results.json
```

### Pattern-Focused Analysis

```bash
codex exec "Run BSKG pattern detection for the following categories: \
1. Reentrancy patterns (reentrancy-001 through reentrancy-005) \
2. Access control patterns (auth-001 through auth-010) \
3. Oracle patterns (oracle-001 through oracle-005) \
Provide evidence for each match." \
  --full-auto \
  --sandbox workspace-write \
  --output-schema pattern-results-schema.json \
  -o pattern-results.json
```

### Comparative Analysis

```bash
codex exec "Compare the security posture of contracts/v1/ and contracts/v2/. \
Identify: \
1. New vulnerabilities introduced in v2 \
2. Vulnerabilities fixed from v1 \
3. Unchanged security issues \
Use BSKG behavioral signatures for comparison." \
  --sandbox read-only \
  --output-schema comparison-schema.json \
  -o version-comparison.json
```

### Exploit Path Synthesis

```bash
codex exec "For each critical finding from VKG, synthesize a potential exploit path. \
Include: \
1. Entry point function \
2. Required preconditions \
3. Attack steps with transaction sequence \
4. Expected outcome \
Focus on findings with confidence > 0.8." \
  --sandbox read-only \
  --output-schema exploit-paths-schema.json \
  -o exploit-paths.json
```

## CI/CD Integration Patterns

### GitHub Actions Workflow

```yaml
name: BSKG Security Audit with Codex

on:
  pull_request:
    paths:
      - 'contracts/**/*.sol'
  push:
    branches: [main]

jobs:
  vkg-codex-audit:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      pull-requests: write

    steps:
      - name: Checkout repository
        uses: actions/checkout@v4

      - name: Setup Python
        uses: actions/setup-python@v5
        with:
          python-version: '3.11'

      - name: Install VKG
        run: |
          pip install uv
          uv sync

      - name: Build BSKG Knowledge Graph
        run: |
          uv run alphaswarm build-kg contracts/ --output .vrs/graph.json

      - name: Generate Audit Prompt
        run: |
          cat << 'EOF' > audit-prompt.md
          # Security Audit Request

          Analyze the Solidity contracts in this repository for security vulnerabilities.

          ## Focus Areas
          1. Reentrancy vulnerabilities
          2. Access control issues
          3. Oracle manipulation risks
          4. Token handling problems
          5. Upgrade safety

          ## BSKG Context
          A knowledge graph has been built at .vrs/graph.json with 50+ security properties
          per function. Use this for informed analysis.

          ## Output Requirements
          Provide findings with severity, location, and remediation recommendations.
          EOF

      - name: Create Output Schema
        run: |
          cat << 'EOF' > audit-schema.json
          {
            "$schema": "http://json-schema.org/draft-07/schema#",
            "type": "object",
            "required": ["findings", "summary"],
            "properties": {
              "findings": {
                "type": "array",
                "items": {
                  "type": "object",
                  "required": ["severity", "title", "location"],
                  "properties": {
                    "severity": { "enum": ["critical", "high", "medium", "low", "info"] },
                    "title": { "type": "string" },
                    "description": { "type": "string" },
                    "location": {
                      "type": "object",
                      "properties": {
                        "file": { "type": "string" },
                        "line": { "type": "integer" },
                        "function": { "type": "string" }
                      }
                    },
                    "recommendation": { "type": "string" }
                  }
                }
              },
              "summary": {
                "type": "object",
                "properties": {
                  "total": { "type": "integer" },
                  "by_severity": { "type": "object" }
                }
              }
            }
          }
          EOF

      - name: Run Codex Security Audit
        uses: openai/codex-action@v1
        with:
          prompt-file: audit-prompt.md
          codex-args: '--output-schema audit-schema.json'
          sandbox: read-only
          output-file: audit-results.json
        env:
          OPENAI_API_KEY: ${{ secrets.OPENAI_API_KEY }}

      - name: Process Results
        id: process
        run: |
          CRITICAL=$(jq '.summary.by_severity.critical // 0' audit-results.json)
          HIGH=$(jq '.summary.by_severity.high // 0' audit-results.json)
          echo "critical_count=$CRITICAL" >> $GITHUB_OUTPUT
          echo "high_count=$HIGH" >> $GITHUB_OUTPUT

          # Generate summary
          jq -r '.findings[] | "- [\(.severity)] \(.title) (\(.location.file):\(.location.line // "?"))"' \
            audit-results.json > findings-summary.txt

      - name: Comment on PR
        if: github.event_name == 'pull_request'
        uses: actions/github-script@v7
        with:
          script: |
            const fs = require('fs');
            const summary = fs.readFileSync('findings-summary.txt', 'utf8');
            const results = JSON.parse(fs.readFileSync('audit-results.json', 'utf8'));

            const body = `## BSKG + Codex Security Audit Results

            **Total Findings:** ${results.summary.total}
            - Critical: ${results.summary.by_severity?.critical || 0}
            - High: ${results.summary.by_severity?.high || 0}
            - Medium: ${results.summary.by_severity?.medium || 0}
            - Low: ${results.summary.by_severity?.low || 0}

            ### Findings
            ${summary}

            ---
            *Automated security analysis powered by BSKG + OpenAI Codex*`;

            github.rest.issues.createComment({
              owner: context.repo.owner,
              repo: context.repo.repo,
              issue_number: context.issue.number,
              body: body
            });

      - name: Fail on Critical Findings
        if: steps.process.outputs.critical_count > 0
        run: |
          echo "::error::Found ${{ steps.process.outputs.critical_count }} critical security findings"
          exit 1

      - name: Upload Artifacts
        uses: actions/upload-artifact@v4
        with:
          name: vkg-audit-results
          path: |
            audit-results.json
            .vrs/graph.json
```

```groovy
                script {
                    def results = readJSON file: 'audit-results.json'

                    }
                }
            }
        }
    }

    post {
        always {
            archiveArtifacts artifacts: 'audit-results.json, graph.json'
        }
    }
```

## Codex GitHub Action

### Overview

The official `openai/codex-action@v1` provides first-class GitHub Actions integration:

```yaml
- uses: openai/codex-action@v1
  with:
    prompt: "Analyze this code for security issues"
    sandbox: read-only
    output-file: results.json
  env:
    OPENAI_API_KEY: ${{ secrets.OPENAI_API_KEY }}
```

### Action Inputs

| Input | Required | Description |
|-------|----------|-------------|
| `prompt` | One of prompt/prompt-file | Inline task instructions |
| `prompt-file` | One of prompt/prompt-file | Path to prompt file |
| `codex-args` | No | Extra CLI flags (JSON array or shell string) |
| `model` | No | Model to use |
| `effort` | No | Reasoning effort level |
| `sandbox` | No | Access level: `read-only`, `workspace-write`, `danger-full-access` |
| `output-file` | No | Save final message to disk |
| `codex-version` | No | Pin specific CLI version |
| `codex-home` | No | Reuse config/MCP across steps |

### Action Outputs

| Output | Description |
|--------|-------------|
| `final-message` | Last Codex response, available for downstream steps |

### Security Controls

**Safety Strategies**:
- `drop-sudo` (default): Removes sudo access
- `unprivileged-user`: Runs as specific account
- `unsafe`: Unrestricted access (Windows only)

**Access Controls**:
- `allow-users` / `allow-bots`: Control who triggers workflows
- Combine with branch protection rules

### VKG-Specific Action Usage

```yaml
- name: BSKG Security Audit
  uses: openai/codex-action@v1
  with:
    prompt-file: .github/prompts/vkg-audit.md
    codex-args: |
      ["--output-schema", ".github/schemas/vkg-audit.json"]
```

## Security Considerations

### Sandbox Modes

| Mode | Permissions | Use Case |
|------|-------------|----------|
| `read-only` | Read files only, no execution | Security audits, analysis |
| `workspace-write` | Read/write workspace, limited network | Code fixes, patches |
| `danger-full-access` | Full system access | Only in isolated environments |

### Best Practices

1. **Always use `read-only` sandbox for audits** - Prevents accidental modifications
2. **Validate `--output-schema` strictly** - Ensures predictable output structure
3. **Rotate API keys regularly** - Use GitHub Secrets with rotation
4. **Limit workflow triggers** - Restrict to trusted events/users
5. **Sanitize inputs** - Never pass unsanitized PR content to prompts
6. **Run as final step** - Prevent state leakage to subsequent steps

### Authentication in CI

```bash
# Recommended: Environment variable (works only with codex exec)
CODEX_API_KEY="sk-..." codex exec "task"

# Alternative: GitHub Secrets
env:
  OPENAI_API_KEY: ${{ secrets.OPENAI_API_KEY }}
```

### Audit Trail

- All Codex actions are logged in transcripts
- Enable `history.persistence: save-all` for complete audit trail
- JSON Lines output captures all events for compliance

## Limitations and Constraints

### Technical Limitations

| Limitation | Description | Workaround |
|------------|-------------|------------|
| **Git Requirement** | Must run in Git repository | Use `--skip-git-repo-check` (reduces safety) |
| **Token Limits** | Model context window constraints | Use BSKG subgraph extraction for focused context |
| **Execution Timeout** | Long audits may timeout | Split into smaller, focused tasks |
| **No Interactive Mode** | Cannot handle user prompts in CI | Pre-configure all options |
| **Platform Support** | Windows requires special handling | Use Linux/macOS runners |

### Schema Limitations

- Schema must be valid JSON Schema draft-07
- Complex nested structures may increase token usage
- Large arrays may hit output limits

### MCP Limitations

- Startup timeout defaults to 10s (may need increase for VKG)
- Tool timeout defaults to 60s (VKG queries may need longer)
- Some MCP features require authentication

### Cost Considerations

- `codex exec` uses API tokens for each invocation
- Large codebases increase token consumption
- Consider caching BSKG graphs to reduce repeated analysis

## Appendix B: BSKG MCP Server Interface Specification

```typescript
// BSKG MCP Tool Definitions for Codex Integration

interface VKGBuildGraphParams {
  path: string;           // Path to Solidity contracts
  options?: {
    include_tests?: boolean;
    max_depth?: number;
    output_path?: string;
  };
}

interface VKGBuildGraphResult {
  success: boolean;
  graph_path: string;
  contracts_count: number;
  functions_count: number;
  properties_per_function: number;
  build_time_ms: number;
}

interface VKGQueryParams {
  query: string;          // VQL2 or natural language query
  graph_path?: string;    // Optional, uses default if not specified
  options?: {
    compact?: boolean;
    explain?: boolean;
    limit?: number;
  };
}

interface VKGQueryResult {
  matches: Array<{
    node_id: string;
    node_type: string;
    properties: Record<string, any>;
    evidence: string[];
  }>;
  query_time_ms: number;
  explanation?: string;
}

interface VKGRunPatternsParams {
  patterns: string[];     // Pattern IDs or category wildcards
  graph_path?: string;
  options?: {
    min_confidence?: number;
    include_draft?: boolean;
  };
}

interface VKGRunPatternsResult {
  findings: Array<{
    pattern_id: string;
    matches: Array<{
      function: string;
      contract: string;
      confidence: number;
      evidence: string[];
    }>;
  }>;
  patterns_evaluated: number;
  total_matches: number;
}

interface VKGExplainParams {
  finding_id: string;
  options?: {
    include_code?: boolean;
    include_fix?: boolean;
    include_exploits?: boolean;
  };
}

interface VKGExplainResult {
  finding_id: string;
  title: string;
  detailed_description: string;
  root_cause: string;
  exploitation_scenario?: string;
  recommended_fix?: string;
  code_before?: string;
  code_after?: string;
  related_cves?: string[];
  related_exploits?: Array<{
    name: string;
    protocol: string;
    loss_usd: number;
  }>;
}
```
