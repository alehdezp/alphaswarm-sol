## 3. VERIFICATION PROCEDURE

### 3.1 Manual Expert Procedure

When an auditor verifies access control:

1. **Identify the privileged operation**
   - What state is being modified?
   - Is it owner, admin, roles, critical params?

2. **Trace the access path**
   - Is there a modifier?
   - Does the modifier check msg.sender?
   - Can the check be bypassed?

3. **Test bypass scenarios**
   - Call as unauthorized user
   - Call through another contract
   - Call with manipulated state

4. **Determine impact**
   - Can attacker gain privileges?
   - Can attacker drain funds?
   - Can attacker disable the contract?

### 3.2 Grimoire Steps

```yaml
steps:
  - name: identify_privileged_state
    type: analysis
    prompt: |
      Analyze this function for privileged state modifications:

      ## Function Code
      {function_code}

      ## State Variables
      {state_variables}

      Identify:
      1. What state is being modified?
      2. Is this privileged state? (owner, admin, roles, fees, paused)
      3. What is the current access control mechanism?

      Return JSON:
      {
        "modified_state": ["var1", "var2"],
        "is_privileged": true/false,
        "access_control": "none" | "onlyOwner" | "onlyAdmin" | "custom",
        "access_control_description": "..."
      }

  - name: analyze_gate_strength
    type: analysis
    prompt: |
      Analyze the strength of this access control:

      ## Function
      {function_code}

      ## Access Control Identified
      {identify_privileged_state_output}

      ## Modifiers
      {modifiers}

      Evaluate:
      1. Does the gate actually protect the function?
      2. Are there bypass paths?
      3. Does it use tx.origin (weak)?
      4. Can the gate be manipulated?

      Return JSON:
      {
        "gate_present": true/false,
        "gate_strength": "none" | "weak" | "medium" | "strong",
        "weaknesses": ["tx.origin used", "no zero check", ...],
        "bypass_possible": true/false,
        "bypass_method": "..."
      }

  - name: generate_exploit_test
    type: code_generation
    condition: "gate_strength != strong"
    prompt: |
      Generate a Foundry test that demonstrates the access control issue:

      ## Function
      {function_code}

      ## Weakness Identified
      {analyze_gate_strength_output}

      Generate a test that:
      1. Deploys the contract
      2. Attempts to call the function as an unauthorized user
      3. Asserts that the call succeeds (vulnerability) or fails (safe)

      Include:
      - Setup with authorized and unauthorized accounts
      - Attempt by unauthorized account
      - Check for state modification

    output_file: test_{function_name}_access_control.t.sol

  - name: run_exploit_test
    type: bash
    command: |
      cd {working_dir} && forge test --match-test test_{function_name}_access_control -vvv

  - name: determine_verdict
    type: analysis
    prompt: |
      Based on the test results, determine if this is a real vulnerability:

      ## Test Output
      {run_exploit_test_output}

      ## Analysis
      {analyze_gate_strength_output}

      Determine:
      1. Did the unauthorized call succeed?
      2. Was privileged state modified?
      3. Is this exploitable in practice?

      Return JSON:
      {
        "verdict": "CONFIRMED" | "REJECTED" | "UNCERTAIN",
        "confidence": 0-100,
        "evidence": ["test showed...", "state was..."],
        "reasoning": "...",
        "severity": "critical" | "high" | "medium" | "low"
      }
```

        _;

    }

    // SAFE: Proper access control
