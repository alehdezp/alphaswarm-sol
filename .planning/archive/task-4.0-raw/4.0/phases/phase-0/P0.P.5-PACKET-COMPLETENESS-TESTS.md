## 1. Overview

This document specifies the **test requirements** for evidence packet completeness validation. These tests ensure that evidence packets produced by BSKG adhere to the contract defined in P0.P.1 and that bucket assignments follow the rules in P0.P.2.

**Key Principle:** These are TEST REQUIREMENTS (what to test), not test implementations. Actual test code will be written in Phase 2 and Phase 3.

**Test Scope:**
- Required field presence and validity
- Location validation
- Behavioral signature validation
- Confidence bucket computation
- Fallback behavior
- Integration (full packet assembly)
- Regression (determinism guarantees)

## 3. Required Fields Presence Tests

### 3.1 Test Scenarios

| Test ID | Scenario | Input | Expected | Acceptance Criteria |
|---------|----------|-------|----------|---------------------|
| RFP-001 | All required fields present | Complete finding | Valid packet | All 13 required fields present |
| RFP-002 | Missing `id` field | Finding without packet ID | Error OR auto-generate | ID generated per P0.P.1 Section 4.2.1 |
| RFP-003 | Missing `finding_id` | Packet without finding ref | Error | Must fail validation |
| RFP-004 | Missing `location` object | Finding without location | Fallback applied | Location uses defaults per P0.P.1 Section 6 |

| RFP-006 | Missing `location.contract` | Location without contract | Fallback | `"Unknown"` applied |
| RFP-007 | Missing `location.function` | Location without function | Fallback | `"unknown"` applied |
| RFP-008 | Missing `location.lines` | Location without lines | Fallback | `[0, 0]` applied |
| RFP-009 | Missing `behavioral_signature` | No signature computed | Fallback | Empty string `""` applied |
| RFP-010 | Missing `semantic_ops` | No ops detected | Fallback | `["UNKNOWN"]` applied |
| RFP-011 | Missing `properties` | No properties matched | Fallback | `["(pattern-based)"]` applied |
| RFP-012 | Missing `evidence_lines` | No code extracted | Fallback | `["(source unavailable)"]` applied |
| RFP-013 | Missing `tool_sources` | No tools listed | Fallback | `["vkg"]` applied |

### 3.2 Acceptance Criteria

```
PASS: Test passes if:
  - Valid packets contain all 13 required fields
  - Missing non-critical fields trigger fallback values per P0.P.1 Section 6
  - Missing critical fields (finding_id) cause validation failure
  - Fallback values match exactly those specified in P0.P.1
```

### 3.3 Test Data Requirements

- Fixtures: Complete findings, partial findings, empty findings
- Contract samples: `tests/contracts/BasicVault.sol`, minimal contract
- Graph outputs: Cached graph with known properties

## 4. Location Validation Tests

### 4.1 Test Scenarios

| Test ID | Scenario | Input | Expected | Acceptance Criteria |
|---------|----------|-------|----------|---------------------|
| LOC-001 | Valid location all fields | File, contract, function, lines | Valid | All location fields non-empty |
| LOC-002 | File path normalization | Relative path | Normalized | Absolute or consistent relative path |
| LOC-003 | Invalid line numbers (negative) | lines: [-1, 10] | Fallback | `[0, 0]` applied |
| LOC-004 | Invalid line range (end < start) | lines: [50, 10] | Corrected | Swap or use [start, start] |
| LOC-005 | Single line (start == end) | lines: [42, 42] | Valid | Single line is valid |
| LOC-006 | Zero line numbers | lines: [0, 0] | Flagged | `request_more_context` set |
| LOC-007 | Contract name extraction | Node without contract field | Derived | Extract from node label or file |
| LOC-008 | Function name extraction | Node without function field | Derived | Extract from node label |

### 4.2 Acceptance Criteria

```
PASS: Test passes if:
  - Valid locations pass validation without modification
  - Invalid line numbers trigger fallback or correction
  - Line range is always [start, end] where start <= end
  - Zero/fallback locations set request_more_context=true
  - Contract/function names derived from available data
```

### 4.3 Edge Cases

- Multi-contract files: Contract name must match specific contract
- Inline assembly: May lack precise line numbers
- Generated code: May have synthetic locations

## 5. Behavioral Signature Validation Tests

### 5.1 Test Scenarios

| Test ID | Scenario | Input | Expected | Acceptance Criteria |
|---------|----------|-------|----------|---------------------|
| SIG-001 | Valid signature with ordering | `R:bal->X:out->W:bal` | Valid | Contains valid codes, arrow-separated |
| SIG-002 | Single operation signature | `X:out` | Valid | Single code is valid |
| SIG-003 | Empty signature | `""` | Valid with flag | Empty string OK, `request_more_context` may be set |
| SIG-004 | Invalid operation code | `X:invalid->W:bal` | Error OR cleaned | Invalid codes rejected or stripped |
| SIG-005 | Duplicate operations | `R:bal->R:bal->W:bal` | Valid | Duplicates allowed (temporal ordering) |
| SIG-006 | All 20 operation codes | Each code individually | Valid | All codes from P0.P.1 Section 3.3 pass |
| SIG-007 | Signature from pre-computed | Node with behavioral_signature | Used directly | Node signature takes precedence |
| SIG-008 | Signature from operation_sequence | Node with ops list | Computed | Derived from ops in order |
| SIG-009 | Signature from individual flags | Node with property flags | Computed | Derived from flags per P0.P.1 Section 5.3 |
| SIG-010 | Signature ordering matches CFG | Known CFG ordering | Correct order | Signature reflects control flow |
| SIG-011 | Signature with auth prefix | `C:auth->M:crit` | Valid | Auth check first indicates safe pattern |
| SIG-012 | Complex multi-op signature | 5+ operations | Valid | Long signatures valid |

### 5.2 Valid Operation Codes (from P0.P.1)

```
X:out, X:in, R:bal, W:bal, C:auth, M:own, M:role, X:call, X:unk, R:ext,
M:crit, I:init, R:orc, L:arr, U:time, U:blk, A:div, A:mul, V:in, E:evt
```

### 5.3 Acceptance Criteria

```
PASS: Test passes if:
  - Valid signatures contain only valid operation codes
  - Codes are separated by "->" arrow
  - Empty signatures are allowed but flagged
  - Signature derivation follows priority: pre-computed > sequence > flags
  - Ordering matches CFG when available
```

## 12. Validation Matrix

### 12.1 Field Validation Matrix

| Field | Type | Required | Fallback | Validation Rule |
|-------|------|----------|----------|-----------------|
| id | string | YES | Generated | Format: `EP-{finding_id}-{hash}` |
| finding_id | string | YES | ERROR | Non-empty, matches finding |
| location.file | string | YES | "unknown" | Non-empty |
| location.contract | string | YES | "Unknown" | Non-empty |
| location.function | string | YES | "unknown" | Non-empty |
| location.lines | [int, int] | YES | [0, 0] | start <= end, both >= 0 |
| behavioral_signature | string | YES | "" | Valid codes or empty |
| semantic_ops | [string] | YES | ["UNKNOWN"] | Non-empty list |
| properties | [string] | YES | ["(pattern-based)"] | Non-empty list |
| evidence_lines | [string] | YES | ["(source unavailable)"] | Non-empty list |
| tool_sources | [string] | YES | ["vkg"] | Contains "vkg" |
| confidence_bucket | string | YES | "uncertain" | One of: likely, uncertain |
| rationale | string | YES | Pattern message | Length >= 10 |

### 12.2 Bucket Validation Matrix

| Score Range | No Override | With Override | Rationale Contains |
|-------------|-------------|---------------|-------------------|
| >= 0.75 | likely | uncertain | "strong" OR score |
| 0.40 - 0.74 | uncertain | uncertain | "verification" |
| < 0.40 | uncertain | uncertain | "verification" OR "weak" |

### 12.3 Override Condition Matrix

| Condition | Forces Bucket | Priority |
|-----------|---------------|----------|
| request_more_context=true | uncertain | 1 (highest) |
| disputed=true | uncertain | 2 |
| location.file="unknown" | uncertain | 3 |
| no props AND no signature | uncertain | 4 |
