### Pattern 1: Academic Security Paper Structure

**What:** Standard structure for CS security tool papers
**When to use:** Research paper core sections
**Why:** Readers expect familiar structure, reviewers know where to find claims

**Typical sections:**
1. **Abstract** (150-200 words): Problem, approach, key results
2. **Introduction** (1-2 pages): Problem motivation, contributions list, paper outline
3. **Related Work** (1-2 pages): Prior approaches, gaps, positioning
4. **Approach/Methodology** (2-3 pages): Technical approach, key insights
5. **Architecture/Implementation** (2-3 pages): System design, components
6. **Evaluation** (2-3 pages): Benchmarks, metrics, comparison
7. **Discussion** (0.5-1 page): Limitations, threats to validity
8. **Conclusion** (0.5 page): Summary, future work

### Pattern 2: Novel Contribution Framing

**What:** Frame behavioral signatures as differentiation from existing tools
**When to use:** Introduction, Related Work sections
**Why:** Establishes novelty, explains why this approach matters

**Key framing:**
```markdown
## Problem with Existing Approaches

Static analysis tools like Slither detect vulnerabilities by:
- Pattern matching on AST nodes (function names, modifiers)
- Taint analysis for data flow
- CFG analysis for control flow

**Limitation:** Name-based heuristics miss vulnerabilities when:
- Developers use non-standard naming
- Business logic vulnerabilities have no syntactic signature
- Cross-function patterns span multiple contracts

## Our Approach: Behavioral Signatures

AlphaSwarm.sol detects vulnerabilities through behavioral signatures:
- **Semantic operations** describe WHAT code does (TRANSFERS_VALUE_OUT, WRITES_USER_BALANCE)
- **Operation ordering** captures the dangerous pattern (transfer before write = reentrancy)
- **No name heuristics** - works regardless of function/variable names
```

### Pattern 4: Evidence-Based Claims

**What:** Every performance claim cites Phase 7.3 validation
**When to use:** Abstract, Evaluation sections
**Why:** Academic integrity, reproducibility

**Format:**
```markdown
## Bad (unvalidated claim)
AlphaSwarm.sol achieves high precision on real-world contracts.

## Good (evidence-based)
AlphaSwarm.sol achieved 84.6% detection rate on the DVDeFi
benchmark (11/13 challenges) with 90% precision on Tier A patterns
(Table 2, Section 5.2).
```

### Anti-Patterns to Avoid

- **Unvalidated claims**: Every metric must cite Phase 7.3 results
- **First person ("we")**: Use third person or passive voice
- **Marketing language**: No "revolutionary", "cutting-edge", "best-in-class"
- **Missing limitations**: Always include threats to validity
- **Incomplete related work**: Cite major tools even if they use different approaches

## Common Pitfalls

### Pitfall 1: Unvalidated Performance Claims

**What goes wrong:** Paper claims "high precision" without Phase 7.3 metrics
**Why it happens:** Writing before validation complete
**How to avoid:** Phase 7.4 depends on Phase 7.3; only cite validated results
**Warning signs:** Phrases like "expected to achieve", "preliminary results suggest"

### Pitfall 2: Missing Limitations Section

**What goes wrong:** Paper appears to overclaim, reviewers skeptical
**Why it happens:** Natural bias to emphasize strengths
**How to avoid:** Explicitly document:
- What the tool cannot detect (e.g., off-chain oracle manipulation)
- Known false positive categories
- Scalability limits
**Warning signs:** No "Discussion" or "Threats to Validity" section

### Pitfall 3: Incomplete Related Work

**What goes wrong:** Reviewers think authors don't know the field
**Why it happens:** Focus only on similar approaches
**How to avoid:** Cite tools from ALL categories:
- Static analysis (Slither, Aderyn, Semgrep)
- Symbolic execution (Mythril, Manticore)
- Formal verification (Certora, Halmos)
- Academic papers (relevant conferences)
**Warning signs:** Related work is 1 paragraph

### Pitfall 4: Architecture Diagram Too Complex

**What goes wrong:** Reader can't understand system
**Why it happens:** Including every component
**How to avoid:** Create layered diagrams:
1. High-level overview (5-6 boxes)
2. Detailed component diagram (per subsystem)
3. Data flow diagram (for specific workflow)
**Warning signs:** Single diagram with 20+ boxes

### Pitfall 5: Appendices Not Machine-Readable

**What goes wrong:** Tables can't be copy-pasted, hard to reference
**Why it happens:** Manual formatting
**How to avoid:** Generate appendices from source files:
```bash
# Generate operations table from reference
python scripts/generate_operations_table.py docs/reference/operations.md > appendices/A-operations.md

# Generate pattern summary from VulnDocs
python scripts/summarize_patterns.py vulndocs/ > appendices/B-patterns.md
```
**Warning signs:** Appendix tables don't match current codebase

## Code Examples

### Generate Operations Table from Reference

```python
# scripts/generate_operations_table.py
"""Generate Appendix A: Operations table from docs/reference/operations.md"""

import re
from pathlib import Path

def extract_operations(reference_path: Path) -> list[dict]:
    """Extract operation definitions from reference doc."""
    content = reference_path.read_text()
    operations = []

    # Parse operation definitions
    for match in re.finditer(r'\| `(\w+)` \| (.*?) \| (.*?) \|', content):
        operations.append({
            'name': match.group(1),
            'category': match.group(2).strip(),
            'description': match.group(3).strip(),
        })

    return operations

def format_appendix(operations: list[dict]) -> str:
    """Format as markdown appendix table."""
    lines = [
        "# Appendix A: Semantic Operations",
        "",
        "Complete catalog of semantic operations used for behavioral signature detection.",
        "",
        "| Operation | Category | Description |",
        "|-----------|----------|-------------|",
    ]

    for op in sorted(operations, key=lambda x: (x['category'], x['name'])):
        lines.append(f"| `{op['name']}` | {op['category']} | {op['description']} |")

    return '\n'.join(lines)

if __name__ == '__main__':
    ops = extract_operations(Path('docs/reference/operations.md'))
    print(format_appendix(ops))
```

### Generate Pattern Summary from VulnDocs

```python
# scripts/summarize_patterns.py
"""Generate Appendix B: Pattern catalog summary from VulnDocs."""

import yaml
from pathlib import Path

def summarize_patterns(vulndocs_dir: Path) -> list[dict]:
    """Collect pattern metadata from VulnDocs."""
    patterns = []

    for pattern_file in vulndocs_dir.rglob('patterns/*.yaml'):
        with open(pattern_file) as f:
            pattern = yaml.safe_load(f)
            patterns.append({
                'id': pattern.get('id', 'unknown'),
                'name': pattern.get('name', 'Unknown'),
                'severity': pattern.get('severity', 'medium'),
                'tier': 'A' if 'tier_a' in pattern.get('match', {}) else 'B',
            })

    return patterns

def format_appendix(patterns: list[dict]) -> str:
    """Format as markdown appendix table."""
    lines = [
        "# Appendix B: Pattern Catalog",
        "",
        f"Summary of {len(patterns)} vulnerability detection patterns.",
        "",
        "| ID | Name | Severity | Tier |",
        "|----|------|----------|------|",
    ]

    for p in sorted(patterns, key=lambda x: x['id']):
        lines.append(f"| `{p['id']}` | {p['name']} | {p['severity']} | {p['tier']} |")

    return '\n'.join(lines)

if __name__ == '__main__':
    patterns = summarize_patterns(Path('vulndocs'))
    print(format_appendix(patterns))
```
