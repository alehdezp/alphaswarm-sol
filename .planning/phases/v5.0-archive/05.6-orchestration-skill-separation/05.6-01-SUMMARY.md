## Decisions Made

### ORCH2-03: Context Mode Requirements
| Skill | context | disable-model-invocation | Rationale |
|-------|---------|--------------------------|-----------|
| audit | fork | true | User controls audit start, isolate from main context |
| investigate | fork | false | Orchestrators can spawn, needs isolation |
| verify | fork | false | Orchestrators can spawn, needs isolation |
| debate | fork | false | Orchestrators can spawn, needs isolation |
| health-check | (none) | true | Quick inline check, user-only |

**Rationale:**
- Fork context prevents audit/verification work from polluting main conversation
- Inline context for health-check enables quick status checks without context overhead
- User-controlled skills (audit, health-check) have disable-model-invocation: true to prevent Claude from invoking them autonomously
- Orchestrator-spawnable skills (investigate, verify, debate) have disable-model-invocation: false to allow programmatic invocation

### VRS vs BSKG Namespace
- **Product skills:** Use `vrs:` prefix (Vulnerability Research Skills)
- **Development skills:** Use `vkg:` prefix (Vulnerability Knowledge Graph)
- **Rationale:** Clear separation between user-facing product (VRS) and internal development (VKG)

### Write Boundaries
All shipped skills restricted to:
- `.claude/vrs/` - VRS working directory
- `.beads/` - Bead storage directory

**Rationale:** Prevent accidental modifications to user's codebase during audit workflows.

### Shipped vs Development Skills
- **Shipped:** `src/true_vkg/skills/shipped/` - Distributed to users via `alphaswarm init`
- **Development:** `.claude/skills/vkg/` - Used only in this repository for VRS development

**Rationale:** Clean separation between product distribution and internal tooling.
