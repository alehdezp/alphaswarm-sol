# Phase 5.6 Plan 05: Health Check CLI Command

Health check command for validating AlphaSwarm.sol installation and environment

## Objective

Implement `alphaswarm health-check` command for validating BSKG installation. Users need to verify their installation is working correctly, with clear pass/warn/fail status for each component (CLI, tools, vulndocs, skills) and actionable fixes for any issues.

## What Was Built

### Tool Availability API (src/true_vkg/core/availability.py)

Extended availability.py with simple tool checking functions:

1. **ToolStatus dataclass**: Captures name, availability, version, path, error, required flag
2. **check_tool_available(tool_name)**: Checks individual tool availability and version
3. **get_available_tools()**: Returns status dict for all known tools
4. **check_vulndocs_available()**: Searches standard locations for vulndocs
5. **check_skills_available(project_root)**: Validates .claude/vrs/ skill files

**Tool Support:**
- **Required**: slither
- **Optional**: aderyn, mythril, foundry, echidna, semgrep, halmos

**Version Detection:** Runs `tool --version` with 10s timeout, parses first line

### Health Check Command (src/true_vkg/cli/health.py)

New CLI command with comprehensive validation:

1. **run_health_check(project, verbose)**: Core health check logic
   - Returns structured dict with checks, fixes, healthy status
   - CLI availability (always pass if running)
   - Required tools (FAIL if missing)
   - Optional tools (WARN if missing)
   - Vulndocs (WARN if missing)
   - Skills (WARN if missing)
   - Beads directory (WARN if missing)

2. **display_health_results(results, verbose)**: Rich table display
   - Color-coded status (green PASS, yellow WARN, red FAIL)
   - Optional version info in verbose mode
   - Actionable fix suggestions
   - Overall health summary

3. **CLI Integration**: Added to main.py as `health-check` subcommand

### Status Semantics

- **PASS (green)**: Component available and working
- **WARN (yellow)**: Optional component missing or not critical
- **FAIL (red)**: Required component missing, blocks usage

## Decisions Made

### 1. CLI Command Name: `alphaswarm health-check`

**Context:** Plan specified `alphaswarm health-check` but codebase uses `alphaswarm`.

**Decision:** Use `alphaswarm health-check` to match existing CLI branding.

**Rationale:**
- pyproject.toml defines `alphaswarm` as the CLI entrypoint
- All documentation and code uses `alphaswarm` consistently
- No references to `alphaswarm` in codebase

**Alternatives Considered:**
- Use `alphaswarm` (rejected: not defined in codebase)
- Use `vkg` alias (rejected: not current convention)

### 2. Separate from `doctor` Command

**Context:** Existing `doctor` command does similar diagnostics.

**Decision:** Create separate `health-check` command.

**Rationale:**
- `doctor`: Diagnoses project state issues (graph.json, versions, findings)
- `health-check`: Validates installation environment (tools, vulndocs, skills)
- Different use cases: installation validation vs project debugging

**Alternatives Considered:**
- Extend doctor (rejected: different focus areas)
- Replace doctor (rejected: existing functionality needed)

### 3. WARN for Optional Tools

**Context:** Most tools (aderyn, mythril, echidna, halmos) are optional.

**Decision:** Show WARN status for missing optional tools, not FAIL.

**Rationale:**
- Users can use BSKG with just Slither
- Should inform about enhancement opportunities without blocking
- `healthy: true` if only required tools present

**Alternatives Considered:**
- FAIL for all missing tools (rejected: too strict)
- PASS for optional tools (rejected: user doesn't see what's available)

## Deviations from Plan

### CLI Command Name

**Planned:** `alphaswarm health-check`
**Implemented:** `alphaswarm health-check`
**Reason:** Plan used incorrect CLI name, codebase uses `alphaswarm` consistently throughout

### Skills Status

**Planned:** FAIL if skills missing
**Implemented:** WARN if skills missing
**Reason:** Skills are optional for BSKG core functionality, shouldn't block health check

Both deviations align with actual project structure and improve user experience.
