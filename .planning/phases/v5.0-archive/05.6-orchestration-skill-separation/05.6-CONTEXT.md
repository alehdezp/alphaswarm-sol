# Phase 5.6: Orchestration Skill Separation & Health - Context

**Gathered:** 2026-01-22
**Status:** Ready for planning

<domain>
## Phase Boundary

Clean separation between development-time tooling (skills/agents for building VKG) and shipped product tooling (skills/agents for auditing Solidity). Create health check skills, define installation model, ensure orchestrators delegate work properly with clear skill boundaries.

**Key constraint:** Development happens ONLY in alphaswarm repo. When VRS skills are installed in any other repo, it's always assessment mode. The separation is physical, not modal.

</domain>

<decisions>
## Implementation Decisions

### Skill Distribution & Installation

**Distribution method:** Project `.claude/` folder (not plugin marketplace)

**Folder structure:**
- `.claude/vrs/` - Shipped VRS skills in target projects
- `.claude/` (alphaswarm repo only) - Dev skills (pattern-forge, vrs:*, knowledge-aggregation-worker)
- Dev skills NEVER leave this repo; only assessment skills ship

**CLI installation:**
- Command: `alphaswarm init` copies VRS skills to target project
- Auto-configure: Creates `.claude/settings.local.json` with VRS enabled
- Conflict handling: Namespace isolation - all VRS content in `.claude/vrs/`, never touches root `.claude/` files
- Documentation: Creates `.claude/vrs/README.md` only (no CLAUDE.md modification)

**Tool installation priority:** brew → pipx → npm (Mac-first, with fallbacks)

### Global vs Local Data

**Global (installed once, updated centrally):**
- vulndocs, patterns → `~/.alphaswarm/`
- VRS skills → copied to each project's `.claude/vrs/` via CLI
- Update mechanism: `alphaswarm update` pulls from GitHub releases

**Local (per-project):**
- Graph, beads, findings, cache → created in target project
- Beads → `.beads/` (steveyegge pattern, JSONL, git-tracked by user choice)
- VRS cache/config → `.claude/vrs/cache/` (gitignore by user choice)

### Write Permissions & Boundaries

**Strict write boundary:** Agents ONLY write to:
- `.claude/vrs/` - VRS internals (cache, config)
- `.beads/` - Task/finding beads

**Never touch:** Other project files (Solidity, configs, etc.)

**Read permissions:** Agents CAN read Solidity files for assessment. Prefer graph/patterns over raw code when possible.

**Minimal noise principle:** Don't create artifacts unless instructed. No unprompted file creation.

### Package & Tool Access

**Hybrid access:** CLI commands for users, Python import for programmatic agent use
- CLI: `alphaswarm build-kg`, `alphaswarm query`, etc.
- Python: `import true_vkg` if package installed

**Required tools:** Slither only (core graph needs it)

**Optional tools:** Aderyn, Mythril, Foundry, Echidna, Semgrep, Halmos - app works without them, warns about missing

### Agent Context & Isolation

**Context sharing:** Skill-level control (not global rule)
- Orchestrators share necessary context with sub-orchestrators
- Fork when context would be noise
- Share when it helps avoid duplicate work
- Claude's discretion per skill

### Health Check Behavior

**Scope:** Full stack validation
- CLI installed
- Required tools available (Slither)
- Optional tools status
- Vulndocs present in `~/.alphaswarm/`
- Skills loaded in `.claude/vrs/`

**Failure handling:**
- Create skills to fix missing components
- Optional tools: if user declines, app works without them
- Graceful degradation: never crash due to missing optional tools

**Auto-check:** First `alphaswarm` command triggers health check if not done
**Caching:** Results in `.claude/vrs/health.json`, skip re-check until config changes

### Bead Management

**Bead creation skills:** Create skills for:
- Bead/task creation during tool deduplication
- Logical merge of findings per tool
- Subagents for complex merge tasks

**Bead scope:** ONLY security/Solidity-related work creates beads
- Non-security tasks return structured results, no bead

### Orchestrator Delegation Pattern

**Golden rule:** Orchestrators delegate as much as possible to subagents. Trivial low-noise tasks allowed directly. Subagents use condensed skills and report back.

**Hierarchy:** Three-tier preferred (orchestrator → sub-orch → workers), dynamic nesting allowed

**Concurrency limits:**
- Max 5 subagents running total
- Max 2 sub-orchestrators running
- Configurable via settings

**Failure recovery:**
- Beads track state (in-progress, completed, blocked)
- Smart prioritization on resume: auto-resume highest priority, pause others for user decision
- Must handle stopped work, resume in-progress tasks
- Test thoroughly for resilience

**Report format:** Hybrid
- Security work: bead + summary returned to orchestrator
- Non-security: structured return only

### Skill Organization

**Namespace:** Grouped prefixes
- `/vrs-bead-*` - Bead management (create, update, merge)
- `/vrs-tool-*` - Tool execution (run, check status)
- `/vrs-orch-*` - Orchestration (spawn-worker, check-beads, resume-work)

**Orchestrator skills (explicit):**
- `/vrs-orch-spawn` - Spawn worker agent
- `/vrs-orch-check-beads` - Check bead status
- `/vrs-orch-resume` - Resume interrupted work

</decisions>
