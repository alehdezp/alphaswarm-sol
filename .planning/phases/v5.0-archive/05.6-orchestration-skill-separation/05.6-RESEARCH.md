# Phase 5.6: Orchestration Skill Separation & Health - Research

**Researched:** 2026-01-22
**Domain:** Claude Code skills architecture, CLI installation patterns, beads management, context isolation
**Confidence:** HIGH

## Summary

This phase implements clean separation between development-time tooling (building VKG) and shipped product tooling (auditing with VKG). The research covers four key areas:

1. **Claude Code Skills Architecture** - How skills work, folder structure, context isolation, and deployment patterns
2. **CLI Installation Patterns** - How to ship skills to target projects via CLI
3. **Beads Management** - The steveyegge/beads pattern for AI-optimized task tracking
4. **Context Pollution Prevention** - Subagent isolation and settings inheritance

The user decisions in CONTEXT.md have locked the distribution method (`.claude/vrs/` folder via `alphaswarm init` CLI) and the beads storage pattern (`.beads/` with JSONL). This research provides implementation details for those decisions.

**Primary recommendation:** Use Claude Code's native skill scoping mechanisms (project-level `.claude/vrs/`) combined with `disable-model-invocation: true` for write operations. Implement beads following steveyegge pattern with JSONL storage in `.beads/`.

### Pattern 2: Skill Frontmatter for Access Control

**What:** Use Claude Code frontmatter to control skill invocation
**When to use:** All shipped skills that perform writes or have side effects

```yaml
# Source: https://code.claude.com/docs/en/skills
---
name: vrs-audit
description: Run BSKG security audit on Solidity contracts
context: fork              # Isolated subagent context
disable-model-invocation: true  # User-only invocation
allowed-tools: Read, Grep, Glob, Bash(alphaswarm:*)
---
```

**Invocation control matrix:**

| Skill Type | `disable-model-invocation` | `context` | Rationale |
|------------|---------------------------|-----------|-----------|
| `/vrs-audit` | `true` | `fork` | User controls when audits run |
| `/vrs-health-check` | `true` | inline | Quick check, no isolation needed |
| `/vrs-bead-create` | `false` | `fork` | Orchestrators can create beads |
| `/vrs-tool-run` | `false` | `fork` | Orchestrators can run tools |
| `/vrs-orch-spawn` | `false` | `fork` | Orchestrators manage workers |

### Pattern 3: CLI Installation with Namespace Isolation

**What:** CLI copies skills to `.claude/vrs/` without touching other `.claude/` files
**When to use:** `alphaswarm init` command

```python
# Source: User decision in CONTEXT.md
def install_skills(target_project: Path) -> None:
    """Install VRS skills to target project."""
    vrs_dir = target_project / ".claude" / "vrs"
    vrs_dir.mkdir(parents=True, exist_ok=True)

    # Copy skill files from package
    skills_source = get_package_skills_path()  # In package data
    for skill_file in skills_source.glob("*.md"):
        shutil.copy(skill_file, vrs_dir / skill_file.name)

    # Create local settings without modifying existing
    settings_local = target_project / ".claude" / "settings.local.json"
    if not settings_local.exists():
        settings_local.write_text(json.dumps({
            "vrs": {"enabled": True, "version": __version__}
        }, indent=2))
```

### Pattern 5: Beads JSONL Storage (steveyegge pattern)

**What:** Store beads as JSONL files in `.beads/` directory
**When to use:** All security-related investigation tasks

```python
# Source: github.com/steveyegge/beads
# Bead storage format
@dataclass
class BeadEntry:
    id: str              # "bd-{hash}" format
    title: str
    status: str          # "open", "in_progress", "complete", "blocked"
    priority: int        # 0 = highest
    parent_id: Optional[str]  # For hierarchical beads
    blockers: List[str]  # IDs of blocking beads
    created_at: str      # ISO timestamp
    updated_at: str
    agent_id: Optional[str]  # Last working agent
    work_state: Dict[str, Any]  # Resumption data

    def to_jsonl(self) -> str:
        return json.dumps(asdict(self))

# Index file: .beads/index.jsonl
# One line per bead, updated on status change
# Agents query with `bd ready` equivalent
```

### Anti-Patterns to Avoid

- **Mixing dev/product skills:** Never put pattern-forge or vrs:discover in shipped `.claude/vrs/`
- **Touching user's `.claude/` root:** Only write to `.claude/vrs/` and `.beads/`
- **Running tools in main context:** Always use `context: fork` for tool execution
- **Storing beads as JSON:** Use JSONL for git-friendliness and append-only updates
- **Hardcoding paths:** Use `~/.alphaswarm/` for global data, project-relative for local

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Skill invocation | Custom dispatcher | Claude Code native skills | Anthropic maintains, tested at scale |
| Context isolation | Manual context management | `context: fork` in frontmatter | Built-in, handles edge cases |
| Settings inheritance | Custom config loader | `settings.local.json` | Claude Code respects precedence |
| Tool detection | Custom check commands | `true_vkg.core.availability` | Already built, handles versions |
| Bead status tracking | Custom status enum | steveyegge/beads pattern | Proven for AI workflows |
| Hash-based IDs | UUID generation | Short hash with prefix | `bd-{hash[:4]}` prevents collisions |

**Key insight:** Claude Code's skill and subagent system already solves most orchestration problems. Don't rebuild what Anthropic provides.

## Common Pitfalls

### Pitfall 1: Settings Pollution in Subagents

**What goes wrong:** Hooks and settings from parent agent are inherited by subagents, causing infinite loops or unexpected behavior
**Why it happens:** Claude Code inherits settings by default for consistency
**How to avoid:**
- Use `--settings` flag with minimal config for subagents
- Disable hooks explicitly in subagent invocations
- Test subagent behavior in isolation
**Warning signs:** Recursive hook execution, unexpected permission prompts

```yaml
# Safe subagent invocation
claude --settings .claude/vrs/subagent-settings.json --agent vrs-worker
```

### Pitfall 2: Context Window Bloat

**What goes wrong:** Main conversation context fills with tool output, degrading reasoning quality
**Why it happens:** Running tools inline instead of in forked context
**How to avoid:** All tool executions use `context: fork`, return summaries not raw output
**Warning signs:** Long responses, degraded quality on complex reasoning, token budget exceeded

Quantified impact (from research):
- Without isolation: 169,000 tokens with 91% noise
- With subagents: 21,000 tokens with 76% useful signal

### Pitfall 3: Skill Not Found After Installation

**What goes wrong:** User installs skills but Claude doesn't see them
**Why it happens:** Skills in wrong location, missing frontmatter, or skill budget exceeded
**How to avoid:**
- Health check validates skill presence
- Use exact path `.claude/vrs/`
- Keep descriptions under 500 chars
**Warning signs:** `/vrs-*` commands not in autocomplete, skill budget warnings

### Pitfall 4: Beads Lost on Restart

**What goes wrong:** Agent work state not persisted, restarts lose progress
**Why it happens:** Work state stored in memory only
**How to avoid:**
- Every significant work step updates bead JSONL
- Use `work_state` field for resumption data
- Agent ID tracked in bead for handoff
**Warning signs:** "Starting fresh" messages after interruption

### Pitfall 5: MCP Server Conflicts

**What goes wrong:** User's MCP servers conflict with VRS requirements
**Why it happens:** Namespace collisions, incompatible tool definitions
**How to avoid:**
- VRS uses `alphaswarm:*` namespace for all tools
- Document required MCP config in README
- Health check detects conflicts
**Warning signs:** Tool not found errors, unexpected tool behavior

### CLI Init Command

```python
# Source: Based on CONTEXT.md decisions + existing CLI patterns
# Location: src/true_vkg/cli/init.py (new file or addition to main.py)

@app.command("init")
def init_alphaswarm(
    target: Path = typer.Argument(Path("."), help="Target project directory"),
    force: bool = typer.Option(False, "--force", help="Overwrite existing installation"),
) -> None:
    """Initialize VRS in a target project.

    Copies VRS skills to .claude/vrs/ and creates necessary directories.
    Does NOT modify existing .claude/ files or settings.

    Example:
        alphaswarm init ./my-defi-project
    """
    vrs_dir = target / ".claude" / "vrs"
    beads_dir = target / ".beads"

    # Check existing installation
    if vrs_dir.exists() and not force:
        console.print("[yellow]VRS already installed. Use --force to reinstall.[/yellow]")
        raise typer.Exit(1)

    # Create directories
    vrs_dir.mkdir(parents=True, exist_ok=True)
    beads_dir.mkdir(parents=True, exist_ok=True)

    # Copy skills from package data
    skills_source = Path(__file__).parent.parent / "skills" / "shipped"
    for skill_file in skills_source.glob("*.md"):
        shutil.copy(skill_file, vrs_dir / skill_file.name)

    # Create beads index
    (beads_dir / "index.jsonl").touch()

    # Create README
    readme = vrs_dir / "README.md"
    readme.write_text(VRS_README_TEMPLATE)

    # Run health check
    console.print("[green]VRS installed successfully.[/green]")
    console.print("Running health check...")
    subprocess.run(["alphaswarm", "health-check"], cwd=target)
```

### Orchestrator Delegation Pattern

```python
# Source: Based on CONTEXT.md + existing handlers.py patterns
# Pattern for ensuring orchestrators delegate work

class OrchestrationSkill:
    """Base for orchestrator skills that delegate to subagents."""

    MAX_SUBAGENTS = 5
    MAX_SUB_ORCHESTRATORS = 2

    def execute(self, context: SkillContext) -> SkillResult:
        """Execute skill by delegating to appropriate workers."""
        # Trivial work can be done directly
        if self._is_trivial(context):
            return self._execute_trivial(context)

        # Complex work must be delegated
        workers = self._select_workers(context)

        # Respect concurrency limits
        if len(workers) > self.MAX_SUBAGENTS:
            workers = self._prioritize_workers(workers)[:self.MAX_SUBAGENTS]

        # Spawn workers with forked context
        results = []
        for worker in workers:
            result = self._spawn_worker(worker, context)
            results.append(result)

        # Aggregate results
        return self._aggregate_results(results)

    def _is_trivial(self, context: SkillContext) -> bool:
        """Check if work is trivial enough to do directly."""
        # Trivial: single file read, status check, simple query
        # Non-trivial: tool execution, multi-file analysis, LLM reasoning
        return context.estimated_tokens < 1000 and not context.requires_tools
```
