# Phase 5.6 Plan 09: Bead CLI Commands Summary

**One-liner:** CLI layer for bead management with create/update/list/show commands, rich tables, and JSON output.

## Objective Achieved

Implemented the CLI wrapper for the JSONL bead storage backend created in 05.6-06. The `alphaswarm bead` commands enable both human users and skills to manage security investigation beads from the command line.

## What Was Built

### 1. Bead CLI Module (src/true_vkg/cli/bead.py)

**Four commands implemented:**

1. **`bead create`** - Create new investigation beads
   - Arguments: title (required)
   - Options: severity, pattern, location, priority, parent, json
   - Returns bead ID in `bd-{hash}` format
   - Example: `alphaswarm bead create "Check reentrancy" -s high`

2. **`bead update`** - Update bead status
   - Arguments: bead_id (required)
   - Options: status, notes, blocked-by
   - Updates status and tracks blockers
   - Example: `alphaswarm bead update bd-a1b2c3d4 --status in_progress`

3. **`bead list`** - List beads with filtering
   - Options: status, severity, ready (flag), json
   - Rich table display with colored severity
   - Filters by status/severity or shows only ready beads
   - Example: `alphaswarm bead list --ready`

4. **`bead show`** - Show bead details
   - Arguments: bead_id (required)
   - Options: json
   - Displays all bead fields including timestamps
   - Example: `alphaswarm bead show bd-a1b2c3d4`

**Features:**
- Rich table display with colors and formatting
- JSON output flag (`--json`) for all commands
- Error handling with exit codes
- Helper function `get_storage()` for consistent storage access
- Uses BeadJSONLStorage from 05.6-06

### 2. CLI Integration (src/true_vkg/cli/init.py)

**Added bead subcommand to alphaswarm CLI:**
- Import: `from true_vkg.cli import bead`
- Registration: `app.add_typer(bead.app, name="bead")`
- Makes bead commands available as `alphaswarm bead <command>`

## Decisions Made

### 1. Add to alphaswarm CLI, not alphaswarm CLI

**Decision:** Add bead commands to `alphaswarm` CLI instead of `alphaswarm` CLI.

**Rationale:**
- `alphaswarm` is the VRS-specific entrypoint
- Beads are a VRS feature, not core BSKG functionality
- Keeps concerns separated (VRS skills vs. BSKG graph operations)
- Skills already reference `alphaswarm` commands

**Alternative considered:** Add to `alphaswarm` CLI
- Rejected: Would mix VRS orchestration with core BSKG functionality

### 2. Rich Table Display for Human Output

**Decision:** Use Rich library for formatted table output in `bead list`.

**Rationale:**
- Better readability for interactive use
- Color-coded severity (red for critical, yellow for medium, etc.)
- Consistent with other CLI tools
- Doesn't affect JSON output (separate flag)

**Alternative considered:** Plain text output
- Rejected: Less readable, harder to scan visually

### 3. JSON Output Flag for All Commands

**Decision:** Add `--json` flag to all commands for machine-readable output.

**Rationale:**
- Skills need to parse output programmatically
- Separates human-friendly vs. machine-friendly formats
- Standard pattern in CLI tools
- Enables automation without parsing rich text

**Alternative considered:** Always output JSON
- Rejected: Poor user experience for interactive use

### 4. Storage Helper Function

**Decision:** Create `get_storage()` helper function.

**Rationale:**
- DRY - single place to configure storage location
- Consistent `.beads/` directory usage
- Easy to test (can be mocked)
- Simplifies command implementations

**Alternative considered:** Inline storage creation in each command
- Rejected: Repetitive, harder to change storage location

## Integration Points

### Input Dependencies (from 05.6-06)

- Uses `BeadJSONLStorage` class for all operations
- Relies on `.beads/index.jsonl` storage format
- Depends on hash-based ID generation
- Uses blocker tracking from storage layer

### Output Interfaces (to 05.6-07, 05.6-08)

- Skills can invoke `alphaswarm bead create/update/list/show`
- JSON output enables programmatic parsing
- Commands provide full CRUD operations for beads
- Ready for integration in skill workflows

### Separation from alphaswarm CLI

**Important distinction:**
- `alphaswarm`: VRS-specific (beads, skills, orchestration)
- `alphaswarm`: Core BSKG (build-kg, query, label, etc.)
- Two separate entrypoints, different concerns
- Users choose based on their task

## Lessons Learned

### What Worked Well

1. **Reusing existing patterns** - Following established typer/rich patterns from other commands
2. **Test-first verification** - Tests caught edge cases early
3. **Clear separation** - alphaswarm vs. alphaswarm CLI distinction is clean
4. **JSON flag pattern** - Enables both human and machine use

### What Could Improve

1. **Validation:** Could add stricter validation for status/severity values
2. **Bulk operations:** No support for bulk create/update (not needed yet)
3. **Search:** No search/filter by title or pattern (could add later)

### Technical Debt

None significant. Future enhancements could include:
- Enum validation for status/severity values
- Bulk operations (create multiple beads from file)
- Search/filter by title or notes content
- Export/import beads for sharing
