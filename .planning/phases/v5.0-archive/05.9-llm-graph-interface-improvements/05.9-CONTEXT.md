# Phase 5.9: LLM Graph Interface Improvements - Context

**Gathered:** 2026-01-27
**Status:** Ready for planning
**Depends on:** Phase 5.8 (Repository Cleanup and Hygiene)

<domain>
## Phase Boundary

This phase hardens the *LLM-facing graph interface* so multi-agent reasoning is
reliable, evidence-bound, and path-accurate. It is not about UI or narration.
It is about ensuring the graph, retrieval, and pattern engine never mislead
agents due to missing signals, ambiguous ordering, or silent omissions.

The outcome is a single, validated interface contract that all LLM-facing
surfaces must follow (CLI, SDK, orchestration, and skills). It fixes:
- Path-qualifying semantics (dominance-aware ordering)
- Guard dominance semantics (true vs heuristic guarding)
- Taint completeness (external-return + call-target control)
- Evidence-first outputs with explicit unknowns
- Retrieval omission metadata (cut sets, excluded edges, coverage)
- Taxonomy drift with a registry + migration map
- Fragmented context slicing (router/llm/kg)

**Additional hardening goals:**
- Interprocedural ordering and modifier-aware dominance
- Explicit unknown states when dominance/taint cannot be proven
- Deterministic evidence references tied to build hashes
- Formalized coverage scoring (testable + reproducible)
- Contract-compliance as a schema gate, not a warning
- Contract treated as an ABI (semver + compatibility shim)

</domain>

<context>
## Current Implementation Snapshot (0.5 baseline)

Key modules currently involved:
- Operation detection + ordering:
  - `src/alphaswarm_sol/kg/operations.py`
  - `src/alphaswarm_sol/kg/sequencing.py`
- Pattern matching + query execution:
  - `src/alphaswarm_sol/queries/patterns.py`
  - `src/alphaswarm_sol/queries/executor.py`
- Subgraph extraction and slicing:
  - `src/alphaswarm_sol/kg/subgraph.py`
  - `src/alphaswarm_sol/kg/ppr_subgraph.py`
  - `src/alphaswarm_sol/kg/slicer.py`
  - `src/alphaswarm_sol/routing/router.py` (agent-specific slicing)
  - `src/alphaswarm_sol/llm/slicer.py` (triage slicer)
- Taint/dataflow:
  - `src/alphaswarm_sol/kg/taint.py`
- Context policy:
  - `src/alphaswarm_sol/llm/context_policy.py`

Observed gaps in the current implementation:
- Ordering is CFG-ordered, not path-qualified. `compute_ordering_pairs()` uses
  CFG order only; it cannot distinguish "always before" from "sometimes before".
- Ordering is effectively intra-procedural; modifiers and internal calls are not
  summarized, so CEI-like patterns can be wrong across call boundaries.
- Guard detection relies on name/regex heuristics (e.g., `has_access_gate`,
  `has_reentrancy_guard`) without dominance semantics.
- Guard dominance is not defined for modifiers, inherited modifiers, early
  returns, or revert-based guards.
- Taint only supports input -> state; there is no external-return taint or
  call-target control taint. Availability is not surfaced as unknowns.
- Taint lacks explicit sinks/sanitizers, and aliasing rules are unspecified
  (storage aliasing, delegatecall, external returns).
- Pattern results lack evidence by default. `PatternEngine` returns matches,
  but evidence refs are only in optional explain output.
- Subgraph extraction does not publish cut sets or coverage metrics; omissions
  are invisible to the LLM.
- Ops taxonomy drifts: `SemanticOperation` uses `TRANSFERS_VALUE_OUT` while
  docs/tools use `TRANSFERS_ETH/TOKEN` and edge taxonomy. Spec matching in
  `DomainKnowledgeGraph` expects `operations`, not `semantic_ops`.
- Multiple slicing pipelines exist (router/llm/kg), producing different context
  for different agents and surfaces. There is no canonical output contract.
- Evidence references are not deterministic or tied to a build hash; results are
  not reproducible across builds.
- Debug slicing is not available, making omission diagnosis difficult.
- Contract compliance is not enforced as an always-on gate for all outputs.
</context>

<decisions>
## Implementation Decisions

### 1) Graph Interface Contract v2 (Single Source of Truth)

The contract defines the canonical LLM-facing output schema used by:
- CLI query output
- Orchestration routing contexts
- Agent inputs
- Evidence packets

**Contract highlights:**
- Contract treated as an ABI: semver + compatibility shim + migration rules
- Always include `matched`, `failed`, and `unknown` clause lists
- Always include `evidence_refs` (or `evidence_missing` with reason)
- Always include `omissions` (cut sets, excluded edges, coverage score)
- Schema validated at serialization time (fail fast)
- Evidence refs are deterministic (build hash + node id + span)
- Truth-table style output for each finding (explicit clause matrix)
- Unknowns budget gating (excess unknowns -> insufficient evidence)

### 2) Path-Qualified Ordering (Interprocedural)

Ordering must encode *path semantics*:
- `always_before`: A dominates B in all feasible paths
- `sometimes_before`: A precedes B on at least one path
- `never_before`: A is not before B on any path
- `unknown_before`: dominance cannot be proven (e.g., truncated CFG)

This requires:
- Dominator/post-dominator analysis on CFG nodes
- Association of operations to CFG nodes (not just cfg_order index)
- Modifier and internal-call summaries for interprocedural ordering
- Ordered pairs computed with path qualification

### 3) Guard Dominance Semantics

"Guard present" is not sufficient. We must distinguish:
- **Present guard**: modifier/require exists somewhere
- **Dominating guard**: guard dominates the sink on all paths
- **Bypassable guard**: guard exists but can be bypassed
- **Unknown guard**: dominance cannot be proven

Guard dominance is a required property for any "guarded" or "safe" claim.

### 4) Taint Expansion + Availability Flags

Taint sources must include:
- User inputs (parameters)
- Environment sources (msg.sender, msg.value)
- External return values
- Call target control (user-controlled addresses)
- Oracle values

Taint semantics must define:
- Sinks and sanitizers (explicitly)
- Aliasing rules for storage + memory
- Propagation rules for external returns and delegatecall

Each taint check returns both result and availability:
- `taint_available = true/false`
- `taint_confidence` (0-1)

Missing taint data is marked as **unknown**, not safe.

### 5) Retrieval Omission Ledger

Subgraph and PPR outputs must include:
- `coverage_score`: fraction of relevant graph captured
- `cut_set`: blockers that prevent traversal (e.g., modifiers not traversed)
- `excluded_edges`: explicit edge types or nodes omitted
- `omitted_nodes`: optional list of dropped nodes

`coverage_score` must be formally defined and deterministic:
- `coverage_score = captured_nodes_weight / relevant_nodes_weight`
- `relevant_nodes = PPR_selected ∪ query_matched ∪ dependency_closed`

### 6) Ops Taxonomy Registry

A single registry maps:
- Ops (`SemanticOperation`)
- Edge types (`EdgeType`)
- Pattern tags
- Legacy aliases

Registry produces:
- Migration map (old -> new)
- Compatibility aliases (with warnings)
- Validation rules for patterns/VQL
- Versioned deprecations and sunset policy
- Automated migration tests tied to VulnDocs + external tool outputs (SARIF)

### 7) Unified Context Slicing Pipeline

Replace fragmented slicers with one pipeline:
1) PPR seed selection and relevance scoring
2) Subgraph extraction
3) Property slicing (GraphSlicer)
4) ContextPolicy filtering (data minimization)
5) Omission ledger injection

All agents receive slices from this same pipeline, with role-specific budgets.

**Debug slice mode:**
- Bypasses pruning for diagnosis
- Emits `slice_mode: debug|standard` in omissions metadata

### 8) Contract Compliance as a Gate

Any LLM-facing output that violates the schema fails fast. No warnings, no
partial compliance.
</decisions>

<design>
## Graph Interface Contract v2 (Draft)

```yaml
interface_version: 2
query:
  kind: pattern|logic|flow|edges|fetch
  id: <string>
summary:
  nodes: <int>
  edges: <int>
  findings: <int>
  coverage_score: <float>
  omissions_present: <bool>
  build_hash: <string>
findings:
  - id: <pattern-id>
    matched_clauses: [ ... ]
    failed_clauses: [ ... ]
    unknown_clauses: [ ... ]
    clause_matrix:
      - clause: <string>
        status: matched|failed|unknown
        evidence_refs: [ ... ]
        omissions_refs: [ ... ]
    evidence_refs:
      - { file: "Token.sol", line: 42, node_id: "N-123", snippet_id: "EVD-001" }
    evidence_missing:
      - reason: "taint_dataflow_unavailable"
    omissions:
      cut_set: [ ... ]
      excluded_edges: [ ... ]
      omitted_nodes: [ ... ]
      coverage_score: 0.83
```

Contract enforcement:
- `matched_clauses` implies evidence_refs or evidence_missing
- `unknown_clauses` implies either omission or availability=false
- `coverage_score` must be present for any subgraph output
- Evidence refs are deterministic and resolvable to source
- `clause_matrix` (if present) must align with matched/failed/unknown lists
</design>

- **Pattern drift** → Registry-based migration + validation tests
- **Agent confusion from inconsistent slices** → Single pipeline + schema validation

<testing>
## Test Strategy

1) **Ordering correctness**
   - Unit tests for dominance-based always/sometimes/never/unknown ordering
   - Modifier and internal-call summary tests
   - Regression tests for classic CEI patterns

2) **Evidence enforcement**
   - Fail if findings omit evidence_refs without evidence_missing
   - Evidence refs resolve deterministically to source

3) **Omission ledger**
   - Ensure `coverage_score` is present for all subgraph outputs
   - Ensure cut_set and excluded_edges are populated when pruning occurs

4) **Taxonomy migration**
   - Fail if patterns/VQL use deprecated ops
   - Ensure legacy aliases resolve deterministically

5) **Unified slicing pipeline**
   - Ensure router, CLI, and LLM flows use the same pipeline
   - Ensure role budgets do not change semantic correctness
   - Debug slice mode bypasses pruning and annotates omissions metadata

6) **Taint availability + sanitizers**
   - Missing taint data yields unknowns, not safe
   - Sanitizers and sinks behave consistently across flows

7) **Contract ABI + truth table**
   - Semver + compatibility shim validated on legacy outputs
   - Clause matrix aligns with matched/failed/unknown lists
   - Unknowns budget gating works (excess unknowns -> insufficient evidence)
</testing>
