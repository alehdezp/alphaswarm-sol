## Test Coverage

### Contract Compliance Tests (16)

**CLI Compliance:**
- `test_cli_query_output_has_interface_version` - interface_version field present
- `test_cli_query_output_v2_compliant` - passes v2 schema validation
- `test_cli_pattern_output_includes_clause_matrix` - clause_matrix aligns with clause lists

**SDK Compliance:**
- `test_sdk_pattern_results_v2_compliant` - SDK results pass schema validation
- `test_sdk_subgraph_includes_omissions` - subgraph extraction has omission ledger
- `test_sdk_ppr_subgraph_includes_omissions` - PPR extraction has omission ledger
- `test_sdk_results_have_evidence_or_missing` - evidence_refs or evidence_missing present

**Orchestration Compliance:**
- `test_orchestration_subgraph_includes_omissions` - context slices have omissions
- `test_orchestration_debug_mode_exposes_omissions` - debug mode indicates slice_mode

**Cross-Surface Consistency:**
- `test_subgraph_and_packager_same_omission_schema` - identical omission structure
- `test_evidence_refs_deterministic_across_surfaces` - EVD-xxxxxxxx format stable

**Contract Validation:**
- `test_invalid_output_fails_strict_validation` - GraphInterfaceContractViolation raised
- `test_valid_output_passes_validation` - valid output passes without error
- `test_coverage_score_out_of_range_fails` - coverage > 1.0 fails validation
- `test_excess_unknowns_marks_insufficient_evidence` - unknowns budget gating works

### Coverage Determinism Tests (22)

**Score Determinism:**
- Same graph + same query = same coverage score (5 runs)
- PPR extraction produces identical scores
- Rebuild graph from same source = same coverage

**Build Hash Stability:**
- Deterministic 12-char hex hashes
- Different sources produce different hashes

**Evidence ID Stability:**
- EVD-xxxxxxxx format deterministic
- Same build hash + location = same ID
- Different locations produce different IDs

**Coverage Formula Compliance:**
- Formula matches spec: captured_weight / relevant_weight
- Score always in [0.0, 1.0] range
- Score = 1.0 when no pruning
- Score decreases monotonically with budget

### Omission Ledger Tests (16)

**Silent Omission Regressions (CRITICAL):**
- `test_no_silent_omissions_in_subgraph` - MUST have omissions field
- `test_no_silent_omissions_in_ppr_subgraph` - PPR MUST have omissions
- `test_no_silent_omissions_in_pattern_results` - pattern results MUST have omissions

**Pruned Nodes Recording:**
- Pruned nodes appear in omission ledger
- Depth limit recorded in cut_set
- Budget exceeded recorded in cut_set

**Field Consistency:**
- coverage_score is numeric and in [0, 1]
- cut_set, excluded_edges, omitted_nodes are lists
- slice_mode in ["standard", "debug"]
- Serialization produces correct dict structure

**Coverage Score Monotonicity:**
- More aggressive pruning = lower coverage
- Score is non-decreasing with budget
