**Gathered:** 2026-01-27
**Status:** Ready for planning
**Depends on:** Phase 5.9 (Graph Interface Contract + ordering/taint hardening)

<domain>
## Phase Boundary

This phase integrates *economic and protocol context* into the multi-agent
workflow and enforces integrity rules that prevent logic gaps. The goal is to
make agent reasoning natural and realistic: agents should reason like auditors
by incorporating incentives, value flows, governance controls, and off-chain
assumptions, while still obeying evidence-first constraints.

Scope includes:
- Converting ProtocolContextPack into a graph overlay and retrieval layer
- Linking economic context to functions and findings
- Enforcing workflow integrity rules (evidence gates, unknown resolution)
- Economic risk scoring and exploit plausibility modeling
- Agent prompt contract updates + validation tests

Out of scope:
- Full market data ingestion or live price feeds
- Rewriting VulnDocs content
- UI/reporting improvements

</domain>

<assumptions>

  than assume safety.

<critical_review>
## Critical Review (Reality Checks + Must-Fix)

- Dossier drift: docs and governance change faster than manual updates.
- Heuristic link risk: a single wrong link can steer the entire debate.
- Policy ambiguity: expected policy is often underspecified in real protocols.
- Stage model mismatch: forcing stages can invent constraints that do not exist.
- Context overload: too much context leads to omission or hallucination.
- Integration blind spot: if routing ignores context, it becomes doc-only.
- Risk scoring noise: must never be treated as validation.
- Freshness blind spot: stale assumptions must auto-downgrade confidence.

</critical_review>

<novel_solutions>
## Novel Solutions (Creative, Uncommon, and Practical)

1) **Dossier drift -> "Context TTL + Diff Beacons"**
   - Every fact gets a TTL and an auto-expire rule.
   - A lightweight "diff beacon" compares docs + governance proposals against
     the dossier and marks fields as stale without manual review.
   - Stale facts become "unknown" until refreshed.

2) **Heuristic link risk -> "Two-key links"**
   - A link must be supported by two independent evidence sources:
     (a) semantic ops or code evidence, and (b) dossier or on-chain mapping.
   - If only one key exists, the link is quarantined and hidden from agents.

3) **Policy ambiguity -> "Expectation provenance + hypothesis lanes"**
   - Policy expectations are tagged as "declared", "inferred", or "hypothesis".
   - Only "declared" can trigger misconfig findings.
   - Hypotheses trigger a probe task instead of a finding.

4) **Stage model mismatch -> "Stage opt-in contracts"**
   - Stages apply only if the dossier declares them.
   - If not declared, default to "stage-unbounded" and avoid stage errors.

5) **Context overload -> "Contract passports + slice budgets"**
   - Agents see only the passport + top-2 relevant flows by default.
   - Expansion requires explicit need and a reason code.

6) **Integration blind spot -> "Context-first routing gate"**
   - Router must reject any candidate without an attached passport
     or a reason for missing context.

7) **Risk scoring noise -> "Risk-as-priority only"**
   - Risk is never part of correctness; it only orders debate or review depth.

8) **Freshness blind spot -> "Confidence decay"**
   - Confidence decays over time; stale facts force "unknown".
   - Agents must cite the date of assumptions they rely on.

</novel_solutions>

<misconfig_radar>
## Misconfiguration Radar (Preemptive Traps)

- **Shadow admin**: indirect admin via module or proxy not reflected in docs.
- **Zero-timelock drift**: timelock temporarily set to 0 and never restored.
- **Role leakage**: emergency role granted to non-emergency actors.
- **Keeper drift**: keeper role becomes de facto governance via config writes.
- **Oracle switch**: change in oracle source without updating assumptions.
- **Permit bypass**: EIP-2612 or permit-based paths bypass main checks.
- **Dual custody**: assets controlled by both contract and external vault.

</misconfig_radar>

<logic_bug_triggers>
## Logic Bug Triggers (Realistic Failure Cases)

- **Upgradeable mismatch**: implementation changed without dossier update.
- **Partial pause**: pause disables one path but leaves a secondary path open.
- **Parameter shadowing**: config parameter exists in multiple contracts with
  inconsistent validation, allowing bypass.
- **Invariant dilution**: governance changes reduce safety buffers silently.
- **Auction timing gap**: keeper can front-run liquidation windows.

</logic_bug_triggers>

<logic_vuln_coverage>
## Logic Vulnerability Coverage Map (Targeted Completeness)

Goal: ensure the workflow can reason across *all* economic logic classes.
This is a coverage checklist, not a claim of detection completeness.

Core logic families:
- **Invariant violation**: ratios, caps, buffers, solvency bounds
- **Accounting drift**: rounding, precision loss, fee accounting, debt tracks
- **Parameter confusion**: wrong units, wrong decimals, or mis-ordered args
- **State machine break**: invalid transitions, missing guards, re-entrance by state
- **Sequencing hazards**: order-dependent effects, cross-function timing gaps
- **Liquidation mechanics**: bad close factor, oracle reliance, priority errors
- **Configuration misuse**: unsafe defaults, missing bounds, admin misuse
- **Economic extraction**: fee siphons, governance capture, keeper abuse
- **Oracle reliance**: stale, manipulable, or inconsistent feeds
- **Upgrade governance**: admin bypass, timelock evasion, proxy confusion
- **Vault semantics**: share-price manipulation, deposit/withdraw asymmetry
- **MEV interaction**: sandwichable flows that break invariants

Mapping to VulnDocs:
- logic/*, precision-loss/*, oracle/*, governance/*, vault/*, upgrade/*, mev/*

</logic_vuln_coverage>

<reasoning_playbook>
## Economic Reasoning Playbook (Agent Thinking Behavior)

1) **Identify assets and custody**: what value is at stake and who holds it.
2) **List invariants**: what must never be violated (ratios, caps, solvency).
3) **Trace value flow**: entry -> transformation -> exit points.
4) **Enumerate controls**: which roles can change params or execute actions.
5) **Check timing windows**: delays, auctions, keepers, emergency paths.
6) **Probe assumptions**: oracles, off-chain actors, governance process.
7) **Find misconfig drift**: expected vs actual config and role assignment.
8) **Assess exploit viability**: profitability and control-path feasibility.

Agents must cite evidence for each step or mark as unknown.

</reasoning_playbook>

<protocol_research_workflow>

0) **Scope**: list contracts, deployments, and versions in the application.

2) **Fact extraction**: convert narrative into structured fields with evidence.

<economic_lenses>
## Economic Reasoning Lenses (LLM Scaffolding)

Use multiple lenses so agents reason like auditors, not static analyzers:

1) **Value Flow Lens**
   - Where does value enter/exit? Who holds custody at each hop?
2) **Control Lens**
   - Who can alter parameters, pause, upgrade, or seize funds?
3) **Incentive Lens**
   - Who profits from breaking an invariant? At what cost?
4) **Trust Lens**
   - Which off-chain inputs can move prices, thresholds, or states?
5) **Timing Lens**
   - Where do delays, auctions, or timelocks create windows?
6) **Configuration Lens**
   - What mis-set parameters or role assignments create exploitability?

Each finding should reference at least one lens, or explicitly state "lens missing".

</economic_lenses>

<lifecycle_model>
## Function Lifecycle Stage Model

Define a small stage lattice to reduce ambiguity in agent reasoning:
- init: setup and role assignment
- active: normal operation
- paused: limited actions, often admin-only
- emergency: rescue/withdrawal paths only
- sunset: shutdown or migration phase
- upgrade: governance-driven changes

Each function should declare allowed stages. Stage violations become a
misconfiguration finding if a function is callable outside its declared stage.

</lifecycle_model>

<authorization_misconfig>
## Authorization Misconfiguration Detection (Beyond Static)

Objective: detect policy drift between *expected* economic controls and *actual*
code behavior.

Policy artifacts:
- Access-control graph (roles -> functions -> capabilities)
- Action truth tables (who can do what, under which conditions)
- Expected policy set from ProtocolContextPack and docs

Mismatch classes to flag:
- Value-moving functions lacking required role checks
- Admin-only controls missing timelock or multi-sig assumptions
- Privileged paths that bypass standard modifiers
- Emergency controls accessible by non-emergency roles
- Upgrade paths that skip governance gates

Output: a policy diff with evidence links and confidence.

</authorization_misconfig>

<expectation_provenance>
## Expectation Provenance and Misconfig Logic

Expectation labels:
- **Declared**: explicit in official docs/governance or on-chain config.
- **Inferred**: deduced from code + common patterns + risk heuristics.
- **Hypothesis**: plausible but unsupported; requires probe to confirm.

Rules:
- Misconfiguration findings require **Declared** or **Declared + Code** evidence.
- Inferred expectations can emit **warnings** but not final findings.
- Hypotheses trigger probes and context expansion, not findings.

Evidence fields for each expectation:
- source_id, source_date, source_type (docs, governance, on-chain, audit)
- confidence, scope, and conflict flags

</expectation_provenance>

<decisions>
## Implementation Decisions

### 1) Economic Context Overlay Graph

Represent context pack as overlay nodes/edges:
- Node types: Role, ValueFlow, Incentive, OffchainInput, Governance, Assumption
- Edge types:
  - ROLE_CONTROLS_FUNCTION
  - VALUE_FLOW_THROUGH_FUNCTION
  - OFFCHAIN_INPUT_FEEDS_FUNCTION
  - ASSUMPTION_AFFECTS_FUNCTION
  - GOVERNANCE_CAN_CHANGE_STATE

This overlay is not a replacement for the code KG. It is a linked layer
with retrieval hooks for agent contexts.

### 2) Function-Level Linkage

Every critical function should link to one or more economic context nodes.
A linkage registry stores:
- function_id
- context_node_id(s)
- link_type
- confidence + evidence

Link creation sources:
- ProtocolContextPack (authoritative)
- Heuristic mapping (fallback)
- User overrides (manual mapping)

Linkage invariants:
- Every link must include evidence_refs (explicit mapping or semantic ops)
- Link confidence cannot exceed evidence confidence
- Low-confidence links surface as "unknown" to agents and cannot upgrade
  confidence without corroboration

### 3) Evidence-Gated Workflow Integrity

Rules enforced globally:
- No confidence upgrades without evidence_refs
- Missing economic assumptions => unknown, not safe
- If omission ledger shows missing context, auto request expansion
- If context expansion fails, route to human review

### 4) Economic Risk Scoring

Introduce an economic risk score alongside technical severity.
Suggested components:
- **Value-at-risk** (TVL, asset control surface)
- **Privilege concentration** (single-role admin vs multisig)
- **Off-chain reliance** (oracle/managers)
- **Governance mutability** (upgradeability, timelocks)
- **Incentive misalignment** (profitability vs safety)

Economic risk does **not** override technical severity; it informs prioritization
and exploit plausibility.

### 5) Agent Prompt Contract Updates

Prompts must include:
- Economic assumptions relevant to the function
- Role and governance context
- Off-chain dependencies
- Explicit statement if context is missing or uncertain

Validation tests will fail if agent outputs do not reference these fields
when required.

### 6) Authorization Policy Diff

Define a policy diff layer that compares expected permissions from the context
pack against the access-control graph derived from code. Mismatches are
reported as misconfiguration findings with evidence links.

### 7) Economic Lenses in Agent Reasoning

Agents must tag reasoning with at least one economic lens (value, control,
incentive, trust, timing, configuration) or explicitly state "lens missing".

</decisions>

<contract_passport>
## Contract Passport (Per-Contract Economic Summary)

Each contract gets a compact passport attached to its node:
- Purpose: what economic role this contract serves
- Assets handled: custody, mint/burn, escrow, fees
- Critical actions: deposit, withdraw, liquidation, upgrade, pause
- Roles and controls: who can call what and under which conditions
- Invariants: ratios, caps, or safety constraints this contract enforces
- External dependencies: oracle feeds, relayers, governance, keepers

<linking>
## Context Linking Strategy (Deterministic, Evidence-First)

Primary linkage sources (highest confidence first):
1. **ProtocolContextPack explicit mappings** (authoritative)
2. **Semantic-operation inference** (value flows inferred from ops like
   TRANSFERS_VALUE_OUT, READS_ORACLE, MODIFIES_CRITICAL_STATE)
3. **Structured hints** (function annotations, invariants, governance flags)

Link confidence rules:
- **High**: explicit mapping from context pack
- **Medium**: semantic operation evidence supports linkage
- **Low**: heuristic-only (must surface as unknown to agents)

Link evidence format (stored in linkage registry):
- `source`: pack|semantic|heuristic
- `signals`: list of ops/properties that justify the link
- `confidence`: high|medium|low

</linking>

<economic_model>
## Economic Risk Scoring (Sketch)

Economic risk score is a *secondary priority signal* (0-10) used to help
triage findings and decide which beads deserve deeper verification.

Components (example weights):
- Value-at-risk (0-4): scaled by TVL or asset exposure
- Privilege concentration (0-2): single-role admin vs multisig
- Off-chain reliance (0-2): oracle, relayer, or external committee
- Governance mutability (0-1): upgradeability, no timelock
- Incentive misalignment (0-1): profitable exploit vs low cost

Score aggregation example:
```
economic_risk = 0.4*VAR + 0.2*PRIV + 0.2*OFFCHAIN + 0.1*GOV + 0.1*INCENTIVE
```

Economic risk does not change *technical severity*, but it can:
- Increase verification priority
- Trigger stronger attacker/defender debate
- Require extra evidence before closure

</economic_model>
