  <action>
Create three Claude Code skills following the structure in 07-RESEARCH.md:

1. **track-gap.md** - Record detection gaps systematically:
   ```markdown
   ---
   name: track-gap
   description: Track a detection gap or limitation found during testing
   allowed-tools: Read, Write, Grep
   context: fork
   agent: Explore
   ---
   ```
   Include:
   - Gap identification (vuln class, protocol, what was missed)
   - Root cause analysis (builder, pattern, LLM, knowledge)
   - Severity grading (critical/high/medium/low)
   - File creation in `.planning/testing/gaps/${VULN_CLASS}.md`
   - GAP-SUMMARY.md update with total counts

2. **test-improvement.md** - Validate fixes for known gaps:
   ```markdown
   ---
   name: test-improvement
   description: Test a specific fix or improvement targeting a known gap
   allowed-tools: Bash, Read, Write
   context: fork
   agent: Explore
   ---
   ```
   Include:
   - Load gap context from `.planning/testing/gaps/`
   - Run targeted test (pattern-specific or protocol)
   - Compare before/after from SQLite corpus DB
   - Report outcome (fixed/improved/regression)
   - Side-effect check via full lens tests

3. **check-consistency.md** - Validate test/corpus alignment:
   ```markdown
   ---
   name: check-consistency
   description: Validate test expectations match corpus ground truth
   allowed-tools: Read, Grep, Bash
   context: fork
   agent: Explore
   ---
   ```
   Include:
   - Compare test assertions to ground_truth.yaml
   - Flag mismatches (test expects X, ground truth says Y)
   - Verify all ground truth entries have corresponding tests
   - Report coverage gaps

Each skill should:
- Use `context: fork` for isolated exploration
- Specify minimal `allowed-tools`
- Include concrete examples
- Reference the corpus database schema from 07-RESEARCH.md

Create the gap tracking infrastructure:

1. Create `.planning/testing/` directory structure:
   ```bash
   mkdir -p .planning/testing/gaps
   touch .planning/testing/gaps/.gitkeep
   ```

2. Create **GAP-TEMPLATE.md** with standardized gap format:
   ```markdown
   ## Gap: ${PROTOCOL} - ${FUNCTION}

   **ID:** GAP-XXX
   **Severity:** Critical | High | Medium | Low
   **Root Cause:** Builder | Pattern | LLM | Knowledge
   **Found:** YYYY-MM-DD
   **Status:** Open | In Progress | Fixed | Won't Fix

   ### Expected Detection
   [What should have been flagged - vulnerability class, location, evidence]

   ### Actual Behavior
   [What happened instead - false negative, wrong severity, missing evidence]

   ### Impact
   [How this affects real audits - severity of miss, frequency in real code]

   ### Root Cause Analysis
   [Why the detection failed - specific component, specific limitation]

   ### Fix Strategy
   [How to address - specific, actionable steps]
   - [ ] Step 1
   - [ ] Step 2

   ### Related Gaps
   [Link to similar gaps if pattern emerges]

   ### Resolution
   [Filled when fixed - date, commit, verification]
   ```

3. Create **GAP-SUMMARY.md** with tracking dashboard:
   ```markdown
   # Gap Summary

   **Last Updated:** YYYY-MM-DD
   **Total Gaps:** 0

   ## By Severity

   | Severity | Count | Open | Fixed |
   |----------|-------|------|-------|
   | Critical | 0 | 0 | 0 |
   | High | 0 | 0 | 0 |
   | Medium | 0 | 0 | 0 |
   | Low | 0 | 0 | 0 |

   ## By Root Cause

   | Component | Count | Open | Fixed |
   |-----------|-------|------|-------|
   | Builder | 0 | 0 | 0 |
   | Pattern | 0 | 0 | 0 |
   | LLM | 0 | 0 | 0 |
   | Knowledge | 0 | 0 | 0 |

   ## By Vulnerability Class

   [Auto-populated as gaps are tracked]

   ## Open Gaps (Prioritized)

   [List ordered by severity then recency]

   ## Recently Fixed

   [Last 10 fixed gaps]
   ```

The /vkg:track-gap skill will maintain this structure.
