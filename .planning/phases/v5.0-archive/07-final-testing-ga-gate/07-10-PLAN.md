## Critical Limitations

### LIMIT-001: Delegatecall Pattern Detection

**Severity:** CRITICAL
**Component:** Builder
**Impact:** Cannot detect vulnerabilities involving delegatecall to attacker-controlled targets

**Root Cause:**
Builder.py tracks calls but doesn't fully resolve delegatecall targets when they come from:
- Function parameters
- Storage variables
- External contract state

**Evidence:**
- DVDeFi Truster challenge: FAIL
- Real-world: [X protocol] exploit missed

**Affected Patterns:**
- delegatecall-arbitrary
- proxy-uninitialized

**Fix Strategy:**
1. Extend CallTracker to tag delegatecall separately
2. Add taint tracking for target parameters
3. Create specific Tier B pattern for LLM verification

**Effort:** High (40-60 hours)
**Priority:** P0 - Blocks real-world detection

---

### LIMIT-002: Strict Equality Edge Cases

**Severity:** CRITICAL
**Component:** Pattern Engine
**Impact:** Strict equality checks (===) can create DoS but BSKG doesn't detect

**Root Cause:**
Pattern for strict equality exists but doesn't cover:
- Balance equality (totalAssets == totalSupply)
- State equality creating revert paths
- Donation attacks

**Evidence:**
- DVDeFi Unstoppable: FAIL
- Real-world: Euler donation attack vector

**Affected Patterns:**
- strict-equality-dos

**Fix Strategy:**
1. Expand pattern to check for donation-susceptible equalities
2. Add Tier B reasoning for economic attack vectors
3. Integrate with Protocol Context Pack for invariant checking

**Effort:** Medium (20-30 hours)
**Priority:** P0 - Common attack vector

---

### LIMIT-003: Cross-Contract Callback Detection

**Severity:** CRITICAL
**Component:** Builder + Pattern Engine
**Impact:** Callbacks through external contracts may not be detected

**Root Cause:**
CallTracker has 24 callback patterns but:
- Doesn't track callbacks through intermediary contracts
- Can't resolve callback targets in complex composability
- ERC-777/ERC-721 hooks sometimes missed

**Evidence:**
- DVDeFi Side Entrance: FAIL
- Complex flash loan callbacks: Partial

**Fix Strategy:**
1. Extend callback pattern matching to N-level depth
2. Add inter-contract dataflow analysis
3. Create Tier B pattern for LLM to trace callback chains

**Effort:** High (60-80 hours)
**Priority:** P0 - Core reentrancy detection

---

## High Severity Limitations

### LIMIT-010: Library Call Tracking Incomplete

**Severity:** HIGH
**Component:** Builder
**Impact:** Calls through libraries (Address.functionCall) not fully tracked

**Root Cause:**
Library calls are inlined by Slither but the target resolution doesn't always work.

**Evidence:**
- ~30% of library-based external calls have LOW confidence edges

**Fix Strategy:**
1. Add library-specific call resolution in calls.py
2. Create mapping of common library patterns (OpenZeppelin)
3. Add medium-confidence edges for library wrappers

**Effort:** Medium (15-25 hours)
**Priority:** P1

---

### LIMIT-011: Complex Multi-Step Patterns

**Severity:** HIGH
**Component:** Pattern Engine
**Impact:** Multi-transaction attack patterns not detected

**Root Cause:**
Patterns operate on single-transaction scope. Multi-step attacks require:
- Cross-transaction state tracking
- Temporal ordering analysis
- Economic simulation

**Evidence:**
- ~30% detection on complex real protocols (per CLAUDE.md)

**Fix Strategy:**
1. Add temporal pattern matching
2. Integrate with Protocol Context Pack for expected flows
3. Create Tier B patterns for multi-step reasoning

**Effort:** Very High (80+ hours)
**Priority:** P1 - Major capability gap

---

### LIMIT-012: Proxy Implementation Slot Resolution

**Severity:** HIGH
**Component:** Builder (proxy.py)
**Impact:** Non-standard proxy slots may not be resolved

**Root Cause:**
Proxy resolver handles EIP-1967, UUPS, Diamond, Beacon but:
- Custom slot locations not detected
- Unverified proxies can't be resolved
- Beacon implementation changes not tracked

**Evidence:**
- Adversarial proxy tests: Partial pass

**Fix Strategy:**
1. Add heuristic detection for non-standard slots
2. Integrate with external verification services
3. Flag unresolved proxies for human review

**Effort:** Medium (20-30 hours)
**Priority:** P1

---

## Medium Severity Limitations

### LIMIT-020: Token Budget Constraints

**Severity:** MEDIUM
**Component:** LLM Integration
**Impact:** Large codebases may exceed token budget

**Root Cause:**
8k token limit per call means:
- Large functions may be truncated
- Context may be insufficient
- Multiple calls increase cost

**Evidence:**
- Some large protocols hit budget limits

**Fix Strategy:**
1. Improve graph slicing for better context selection
2. Implement sliding window for large functions
3. Add cost/quality tradeoff configuration

**Effort:** Medium (15-20 hours)
**Priority:** P2

---

### LIMIT-021: Tier B False Positive Rate

**Severity:** MEDIUM
**Component:** Pattern Engine + LLM
**Impact:** Tier B patterns have ~20% false positive rate

**Root Cause:**
LLM reasoning can be:
- Over-cautious (reports uncertain as likely)
- Missing context (can't see full protocol)
- Inconsistent (different results on retry)

**Evidence:**
- Tier B precision: ~80% (vs 95% Tier A)

**Fix Strategy:**
1. Improve prompt engineering for Tier B
2. Add confidence calibration
3. Require additional evidence for Tier B findings

**Effort:** Medium (20-30 hours)
**Priority:** P2

---

### LIMIT-022: Economic Attack Vector Coverage

**Severity:** MEDIUM
**Component:** VulnDocs + Pattern Engine
**Impact:** Some economic attacks not in knowledge base

**Root Cause:**
VulnDocs covers major economic attacks but:
- Novel DeFi primitives create new vectors
- Cross-protocol composability creates unique risks
- Economic modeling not implemented

**Evidence:**
- Some novel exploits not in VulnDocs

**Fix Strategy:**
1. Continuous VulnDocs updates (Exa scanning)
2. Add economic simulation capability
3. Create protocol-specific economic context

**Effort:** Ongoing
**Priority:** P2
