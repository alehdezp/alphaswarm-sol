<decisions>
## Implementation Decisions

### SDK Selection Strategy (REFACTORED 2026-01-21)

**IMPORTANT:** Original API-based approach replaced with multi-model CLI-based execution.

- **Primary SDK:** OpenCode SDK (`@opencode-ai/sdk`) for multi-model access
- **Orchestration Runtime:** Claude Code CLI (subscription-based, no per-token cost)
- **Review Runtime:** Codex CLI (ChatGPT Plus subscription, alternative perspective)
- **Legacy Runtimes:** API-based (anthropic.py, openai_agents.py) kept but deprecated

**Rationale:** API billing is too expensive for extensive agent workflows. Combination of subscriptions + free-tier models provides 75-95% cost savings.

### Multi-Model Architecture

#### Primary Orchestration (Subscription-Based)
- **Claude Code CLI:** Uses `claude` command with --print/--output-format json
  - Session management via --resume
  - Subagent spawning via Task tool
  - Full agentic capabilities (Read, Write, Bash, etc.)
  - Cost: $20/month (Pro) or $100/month (Max)
  - **Use for:** Critical analysis (Attacker, Defender, Verifier), orchestration

- **Codex CLI:** Uses `codex exec` for scripted tasks
  - Best for: review, discussion, double-checks (different perspective)
  - Cost: $20/month (ChatGPT Plus)
  - **Use for:** Reviews, alternative opinions, debate participation

### OpenCode SDK Integration

- **Installation:** `npm install @opencode-ai/sdk`
- **CLI Usage:** `opencode -p "prompt" --model "model-id" -f json -q`
- **Session Management:** Via SDK's `client.session.create()` and `client.session.prompt()`
- **Model Selection:** Dynamic based on task requirements and rankings

**Configuration Pattern:**
```json
{
  "$schema": "https://opencode.ai/config.json",
  "mode": {
    "verify": { "model": "minimax/minimax-m2:free" },
    "summarize": { "model": "minimax/minimax-m2:free" },
    "context": { "model": "x-ai/grok-code-fast-1" },
    "code": { "model": "zhipu/glm-4.7" }
  }
}
```

### Model Ranking/Feedback System (Phase 7 Integration)

- **Purpose:** Track which models perform best for specific task types
- **Data Collected:** Success rate, latency, tokens, quality score, cost
- **Update Algorithm:** Exponential moving average with decay
- **Selection Logic:**
  1. Filter by capability (context size, tool use)
  2. Filter by accuracy threshold (if accuracy_critical)
  3. Sort by priority (latency/quality/cost based on task)
  4. Fallback chain: Free → Cost-effective → Claude Code

- **Phase 7 Exercise:** Run each model against standard test suite to build initial rankings

### Hook System Design

- **Persistence model:** Adapt concepts from [steveyegge/beads](https://github.com/steveyegge/beads) library
  - Study the library's storage and hook patterns
  - Adapt to our existing bead schema
  - Not a direct dependency import
- **Work claiming:** Hybrid approach
  - Orchestrator assigns beads to agent hooks
  - Agents can pull more work from their inbox if items available
- **Priority ordering:** Per PHILOSOPHY.md
  - critical severity > exploitability > tool agreement > recency
- **State persistence:** In bead record (`work_state` dict)
  - Agent can resume exactly where it left off
- **Parallelism:** Configurable agents per role
  - Pool config specifies how many agents per role
  - Beads distributed round-robin
- **Failure handling:** Both requeue and escalate with threshold
  - Requeue bead up to N times (configurable)
  - After threshold, escalate to supervisor
  - Track failure count per bead
- **Events:** No events for now — keep it simple
- **SLA:** No SLA enforcement for now

### Test Generation Approach

- **Generation method:** LLM-generated with pattern/docs assistance
  - LLM writes full test code
  - VulnDocs patterns guide test structure
  - Patterns linked to relevant VulnDocs
  - This is the preferred method
- **VulnDocs integration:** Always retrieve relevant VulnDocs
  - Patterns should be linked to VulnDocs
  - Auto-include for vulnerability class context
- **Test location:** Pool directory (`tests/` folder inside pool)
  - Keeps tests with audit artifacts
  - Good for auditing other projects
- **Confidence integration:** Pass = confirmed
  - Passing exploit test elevates confidence to "confirmed"
  - No human review needed if test demonstrates the vulnerability

### Agent Execution Model

- **Agent lifecycle:** Context-fresh (new instance per bead)
  - Clean context, no memory pollution
  - Per PHILOSOPHY.md preference
- **Stuck work handling:** Log and continue
  - Don't auto-intervene
  - Let pool complete with what it can
  - Visible in logs for post-audit review
- **Verdict conflicts:** Route to debate
  - Conflicting verdicts trigger attacker/defender/verifier debate
  - Per existing debate protocol
- **CLI access:** Full shell access
  - Agents can run any command (forge, slither, etc.)
  - Maximum power for test execution and tool integration

### Token Efficiency & Cost Optimization

**Reference:** See `TOKEN-EFFICIENCY-RESEARCH.md` for full 2026 research findings.

#### Free-Tier Utilization (Priority 1 - CRITICAL)

- **Big Pickle (FREE):** 200K context, 128K output, function calling
  - Use for: Verification, validation, double-checks
- **MiniMax M2 (FREE):** 204K context, optimized for agentic coding
  - Use for: Summarization, simple verification
- **Fallback chain:** Free → Cost-effective → Subscription

#### Prompt Caching (Priority 2 - For API Fallback)

- **Anthropic:** Use `cache_control: {"type": "ephemeral"}` markers
  - 90% cost reduction on cached reads, 85% latency reduction
  - Cache: VulnDocs, tool definitions, target contract code (stable prefix)
  - Dynamic: bead-specific context, agent role, investigation state
- **OpenAI:** Automatic for prompts >1024 tokens
- **Cache architecture:**
  ```
  CACHED (shared)     → VulnDocs + Tools + Contract Source
  DYNAMIC (per-bead)  → Bead context + Agent role + Work state
  ```

#### Model Routing (Priority 1 - MUST IMPLEMENT)

- **Router model:** Use task type + role to select model
- **Role-to-model mapping:**
  | Role | Model | Rationale |
  |------|-------|-----------|
  | Bead triage | Free model | Simple filtering |
  | Defender | Claude Code | Accuracy matters |
  | Attacker | Claude Code | Deep exploit reasoning |
  | Verifier | Claude Code | Critical accuracy |
  | Test Builder | Claude Code | Code generation |
  | Summarizer | Free model | Compression task |
- **Expected savings:** 75-95% via free-tier + subscription mix

#### Context Management (Priority 2)

- **Summarization checkpoints:** Every 5 turns, compress history
- **Selective retention:** Preserve decisions/constraints, discard conversational residue
- **Extended thinking:** Auto-stripped from context (use freely for complex reasoning)
