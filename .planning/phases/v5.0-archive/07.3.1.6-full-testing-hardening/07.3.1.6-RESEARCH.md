### Global Verification Requirements
- **Definition Gate first:** No execution until EI/CTL, Tier C thresholds, workflow contracts, and alignment ledger are complete.
- **Demo claude-code-agent-teams label:** All live runs must use `vrs-demo-{workflow}-{timestamp}` session labels.
- **Proof tokens:** Evidence packs must include proof tokens per `docs/PHILOSOPHY.md` (graph hash, node/edge counts, agent spawn proof, debate proof, detection proof).
- **Graph-first:** BSKG/VQL queries must run before any conclusions or reports.
- **Evidence pack schema:** Use standardized transcript headers and evidence pack layouts from `.planning/testing/guides/guide-evidence.md`.

### Standard Stack
- Execution control: Claude Code + `claude-code-controller` skill for all interactive testing.
- Evidence artifacts: `.vrs/testing/runs/<run_id>/` packs with `transcript.txt`, `report.json`, `commands.log`, `environment.json`, `ground_truth.json`.
- Workflow contracts: `docs/workflows/*` (audit, graph, context, tools, tasks, verify, progress).
- Validation rules: `.planning/testing/rules/canonical/*` + `docs/reference/testing-framework.md`.
- Scenario coverage: `.planning/testing/scenarios/SCENARIO-MANIFEST.yaml`.
- Registry integrity: `scripts/validate_workflow_refs.py` against `src/alphaswarm_sol/skills/registry.yaml` and `src/alphaswarm_sol/agents/catalog.yaml`.

### Architecture Patterns
- Controller to subject to evaluator testing model with dedicated claude-code-agent-teams sessions.
- Evidence-first enforcement where Task lifecycle markers and transcripts are required to accept findings.
- Workflow contracts are the source of truth: docs updated first, skills follow, then claude-code-agent-teams evidence.
- Tiered reasoning: Tier A and Tier B/C flows gated by context and label coverage.
- Alignment ledger maps workflow to skills to subagents to commands to transcripts.

### Do Not Hand-Roll
- claude-code-agent-teams: use `claude-code-controller` and documented patterns, do not invoke raw claude-code-agent-teams.
- Evidence packs: follow existing layout and schema; extend docs, do not invent new locations.
- Workflow references: use `scripts/validate_workflow_refs.py`, do not create ad-hoc mappings.
- Scenario tracking: record all tests in `SCENARIO-MANIFEST.yaml` instead of scattered notes.

### Common Pitfalls
- Running validation in mock or simulated mode.
- Missing TaskCreate or TaskUpdate markers in transcripts.
- Tier C running without context or without label coverage thresholds.
- Evidence packs missing raw transcripts or provenance artifacts.
- Using `/vrs:` instead of `vrs-*` naming.

## Blocking Gate (B1-B25)

### B1 - Installation and first-run validation

#### Reasoning
First-run validation exposes packaging, environment, and CLI drift that cannot be detected in unit tests. Evidence-based workflow validation requires a real claude-code-agent-teams session that proves install, init, and health-check flows are functional in a clean environment.

#### Purpose
Confirm installation and initialization are repeatable, observable, and produce evidence artifacts for auditability.

#### Why It Matters
Every other workflow depends on a working CLI. If install or init fails, any subsequent evidence is untrustworthy.

#### How To Implement
- Run `uv tool install -e .` and validate tool registration.
- Run `alphaswarm init` and capture generated state or config files.
- Run `alphaswarm --help` and `alphaswarm health-check --project . --json`.
- Include negative cases: missing `uv`, missing Python, wrong Python version.
- Capture claude-code-agent-teams transcript and environment metadata for the successful path and failure paths.

#### How To Verify
- Evidence pack contains transcript, `commands.log`, and `environment.json` for each run.
- Transcript includes explicit install, init, help, and health-check markers.
- Failure cases show clear error messaging and recovery guidance.

#### References

#### Purpose

### B3 - Iteration protocol and diagnostics

#### Reasoning
Iterative improvement is a core promise of multi-agent verification. A consistent failure taxonomy and iteration log make it possible to measure progress and avoid repeating mistakes.

#### Purpose
Create a standardized iteration protocol, failure taxonomy, and diagnostic artifacts for repeatable improvement cycles.

#### Why It Matters
Without a common iteration protocol, retries become ad-hoc and regressions are hard to detect or explain.

- Create an iteration log template with before and after deltas and a decision rationale.
- Document transcript diff guidance and per-workflow `claude-code-controller wait_idle` timeout budgets.
- Specify a missing-marker reporter for TaskCreate, TaskUpdate, and tool markers.

#### How To Verify
- Apply the taxonomy to an existing failed run and confirm classification is consistent.
- Ensure iteration log entries include deltas and remediation steps.
- Validate timeout budgets are referenced in workflow docs or testing guides.
