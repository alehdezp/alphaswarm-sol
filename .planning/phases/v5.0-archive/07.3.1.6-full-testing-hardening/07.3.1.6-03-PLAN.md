**Reasoning**
Phase 7.3.1.5 delivered the full-testing orchestrator, but audit workflows still behave like report generators. This plan transforms vrs-audit into a real orchestrator that creates tasks, spawns subagents, shows progress, and blocks invalid runs - the kind of behavior that distinguishes a security assistant from a static analysis tool.

**Purpose**
Lock in audit entrypoint guarantees: explicit task lifecycle markers, gated preflight checks, Tier C rules tied to context requirements, and CLI output contracts that downstream tools can consume.

**Why**
Without enforced orchestration, audit outputs are not reliable enough for benchmarking or continuous validation. These guardrails prevent silent failures, ambiguous states, and unverifiable findings, which are GA blockers for TEST and PHIL requirements.

**How To Implement**
Execution order: update docs first (T0), then align skills/policies (T1), then run claude-code-agent-teams tests (T2-T5).
1. Split `/vrs-audit` into a command router with explicit `init` and `continue` subcommands and handoff markers.
2. Require a preflight gate (settings + tool readiness) before init, and emit progress guidance for each stage (preflight, init, task, verify, report).
3. Run detection in parallel when enabled (tool subagents + pattern mini-orchestrator), while preserving a sequential fallback.
4. Create tasks during detection with `pending_dedup` status; run dedup to merge and emit `[DEDUP_COMPLETE ...]` markers.
5. Add preflight and completion hooks that block missing settings, missing tool initialization, or missing task lifecycle evidence.
6. Standardize settings/state validation (including precedence and recovery) and Tier C gating behavior across `/vrs-status` and `/vrs-resume`, including the context-bypass marker and mock context artifact capture.
7. Require tool initialization in the audit flow, document tool marker expectations, and confirm policy permits required tool commands.
8. Document exit codes and structured YAML outputs so task updates map to a deterministic schema.

## Verification

Verification is outcome-based, not threshold-based:

| Task | Promise | Verification Question |
|------|---------|----------------------|
| T0 | Docs + outputs | Are contracts/markers/exit codes documented and usable? |
| T1 | Skills aligned | Do skills/policies enforce the documented behaviors? |
| T2 | Real orchestration | Could you follow the audit logic? Did it investigate or just assert? |
| T3 | Helpful blocking | Would a new user know what to do after seeing the error? |
| T4 | Context gating | Is it clear which Tier C patterns were skipped and why, with EI/CTL artifacts visible? |
| T5 | Tools inform findings | Did tool output actually inform the investigation? |

**Anti-pattern warning:** Do NOT rely *only* on line counts, durations, or keyword presence. These are necessary anti-fabrication gates but insufficient alone. Always combine them with outcome-based checks that prove the promised capability worked.
