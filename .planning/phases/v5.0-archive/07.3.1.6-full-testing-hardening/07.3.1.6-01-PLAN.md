## Reasoning
This plan is the Definition Gate for 7.3.1.6. It forces explicit, TESTABLE definitions before any claude-code-agent-teams-driven execution. The key insight is that definitions must answer "how do I verify this?" not just "what is this?". Without this, later plans cannot write meaningful tests.

## Purpose
Produce definitions that downstream plans can USE to:
- Write test assertions
- Implement CI gates
- Determine pass/fail from transcripts
- Track which workflows have evidence

## Why
- Testing evidence is only credible when definitions specify MEASURABLE outputs.
- Workflow contracts without OBSERVABLE criteria cannot be tested.
- Alignment ledgers that don't show GAPS are useless for tracking progress.
- Validation scripts that don't catch ERRORS provide false confidence.

## How To Implement

### T1: Document EI/CTL outputs and Tier C gating thresholds
**Goal:** Someone can write a test assertion based on these definitions.

For EI/CTL in `docs/workflows/workflow-context.md`:
- EI must specify: trigger condition, output artifacts, verification method
- CTL must specify: pipeline stage, gate conditions, pass/fail outcomes
- Include example: "EI produces `evidence/economic-context.json` with fields X, Y, Z"

For Tier C in `docs/reference/testing-framework.md` and `.planning/testing/guides/guide-tier-c.md`:
- Every threshold must be: `measure X, if X < Y then fail`
- Label coverage: specify percentage and how to measure
- Simulated context bypass: specify exact marker string

**Verification:** Can you implement a CI check from the definition alone?

### T2: Formalize orchestration markers and workflow contracts
**Goal:** Someone can determine pass/fail using ONLY the transcript and contract.

For workflow contracts in `docs/workflows/workflow-audit.md`:
- Required inputs table (what must exist before starting)
- Expected outputs table (what must exist after completion)
- Success criteria list (observable in transcript)
- Failure criteria list (what causes rejection)

For orchestration markers in `.planning/testing/workflows/workflow-orchestration.md`:
- TaskCreate required fields and timing
- TaskUpdate required fields and state transitions
- Tool markers expected per stage

**Verification:** Given a transcript, can you score pass/fail without guessing?

### T3: Create alignment ledger and readiness evidence table
**Goal:** Gaps are visible at a glance.

For alignment ledger in `07.3.1.6-ALIGNMENT-LEDGER.md`:
- Row per workflow from `docs/workflows/README.md`
- Columns: workflow | skills | subagents | commands | transcript_path | status
- Use placeholder "NEEDED" for missing evidence
- Color-code or mark complete vs incomplete

For readiness table in `07.3.1.6-READINESS-EVIDENCE.md`:
- Row per B1-B18 blocker from SUPER-REPORT
- Columns: blocker_id | description | evidence_path | timestamp | status
- Status: unstarted | in-progress | complete

**Verification:** Can a PM count resolved vs unresolved blockers?

### T4: Validate workflow_refs in registries
**Goal:** Validation catches real errors, not just reports success.

For `scripts/validate_workflow_refs.py`:
- Must FAIL when refs point to nonexistent files
- Must report WHICH ref is broken and WHERE
- Strict mode must enforce additional rules

**Verification:** Break a ref intentionally, does it fail? Restore it, does it pass?

## Verification

1. Downstream plans can implement tests based on EI/CTL definitions (not guess)
