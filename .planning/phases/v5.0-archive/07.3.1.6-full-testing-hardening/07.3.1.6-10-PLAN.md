        - Evidence pack entries for tool outputs or explicit bypass marker

    - name: tool_outputs_influence_reasoning
      verifies: "LLM reasoning actually uses tool outputs, not just ignores them"
      how: |
        1. Run audit with tools enabled

        4. Verify reasoning references specific tool outputs (line numbers, detector names)
      success_criteria: |
        - Reasoning contains at least 2 references to specific tool outputs
        - References include concrete details (line numbers, function names from tools)
        - Reasoning explains WHY tool output supports the finding
        - Not just "Slither found X" but "Slither's reentrancy detector flagged line 18..."
      evidence_required: |
        - Full reasoning section from audit
        - Highlighted passages showing tool output integration
        - Tool raw outputs for comparison

    - name: tool_bypass_is_explicit
      verifies: "Disabling tools is explicit and does not silently skip evidence"
      how: |
        1. Run audit with tools disabled via settings
        2. Verify explicit bypass markers are emitted
        3. Verify findings are marked as reduced evidence or gated accordingly
      success_criteria: |
        - Tool bypass marker appears in transcript
        - Evidence pack records which tools were skipped and why
        - Output indicates reduced evidence due to tool bypass
      evidence_required: |
        - Transcript with bypass marker
        - Evidence pack showing tool bypass metadata

    - name: tool_matrix_is_actionable
      verifies: "The tool maximization matrix enables concrete tool selection decisions"
      how: |

        3. Run a scenario following matrix requirements
        4. Verify all required tools were actually invoked
      success_criteria: |
        - Matrix provides clear tool requirements per scenario type
        - Requirements are targeted (not just "use everything")
        - Following matrix produces complete tool coverage for scenario type
      evidence_required: |
        - Matrix excerpt for test scenario
        - Transcript showing required tools were invoked
        - Explanation of why these specific tools for this scenario

      verifies: "Detector-to-category mapping is explicit and consumable by dedup"

        1. Read docs/reference/tools-adapters.md mapping section

        - Mapping covers core detectors for Slither/Aderyn/Mythril
        - Mapping output is deterministic (no LLM fallback required by default)
        - Dedup workflow references the mapping source
      evidence_required: |
        - Mapping excerpt from tools reference
        - Reference in workflow-dedup or workflow-tools doc
