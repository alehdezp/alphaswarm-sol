## Evidence Requirements
- Any live run must use demo claude-code-agent-teams session label `vrs-demo-{workflow}-{timestamp}`.
- Evidence packs must include proof tokens per `docs/PHILOSOPHY.md`, but required tokens are workflow-specific (install/init may mark graph/agent/debate tokens as N/A; audit workflows must include graph hash, node/edge counts, agent spawn proof, debate proof, detection proof).
- Anti-fabrication thresholds are enforced for B1 transcripts: min line count >= 50, min duration >= 15s, required markers present (transcript header + ALPHASWARM-START/ALPHASWARM-END).
- Graph-first rule enforced: BSKG/VQL queries must run before conclusions for any reasoning step.

## Reasoning
This plan closes the evidence (B2) and iteration (B3) blockers and establishes a real B1 baseline. Without standardized evidence packs, downstream workflow validation cannot be compared or audited. Without proving installation works, no higher-order workflow claims are credible.

## Purpose
Produce:
1. Evidence pack schema that downstream plans USE to capture workflow runs
2. Iteration protocol that helps DIAGNOSE failures (not just report them)
3. Real installation transcript proving the tool WORKS (not just exists)

## Why
- Evidence packs are the proof substrate - inconsistent structure breaks comparability
- Iteration without actionable taxonomy cannot demonstrate progress
- Installation must be proven before higher-order workflows can claim credibility

## How To Implement

### T1: Evidence pack schema and transcript headers
**Goal:** Someone can create an evidence pack for any workflow using only the documentation.

For evidence guide in `.planning/testing/guides/guide-evidence.md`:
- Required files: manifest.json, transcript, report.json, environment.json
- Required fields per file (with types and examples)
- Required proof tokens by workflow type (install/init vs audit) with explicit N/A rules
- Transcript header fields: session_label, workflow, timestamp_utc, settings_hash, commit_hash
- Boundary markers: ALPHASWARM-START, ALPHASWARM-END (exact strings)
- Orchestration markers required in evidence packs for reasoning workflows (TaskCreate/TaskUpdate, subagent spawn markers)
- Graph-first verification field for reasoning workflows (query markers recorded before conclusions)
- Validation command: how to check a pack is complete

For evidence checklist in `.planning/testing/templates/evidence-checklist.md`:
- Per-workflow checklist (what to capture for audit vs install vs tool-run)
- Required vs optional fields

For example pack in `.planning/testing/templates/evidence-pack-example/`:
- manifest.json with all required fields populated
- Example transcript with header and boundary markers
- report.json, environment.json, ground_truth.json
- validator-output.json showing the pack validates

**Verification:** Can a developer create an evidence pack using only the guide?

### T2: Iteration protocol and diagnostics
**Goal:** When a test fails, the guide tells you what to do (not just what failed).

For iteration guide in `.planning/testing/guides/guide-iteration.md`:
- Failure taxonomy with at least 5 categories:
  - missing-markers: required markers not in transcript
  - missing-evidence: required files not in pack
  - duration-anomaly: run was too fast (likely skipped steps)
  - output-mismatch: expected output differs from actual
  - environment-drift: environment changed between runs
- Per-category diagnostic steps (what to check)
- Per-category remediation (what to try)
- Timeout budgets per workflow (how long to wait before concluding failure)

For missing-marker reporter in `.planning/testing/templates/missing-marker-report-example.json`:
- Fields: marker_name, expected_location, search_result, severity, remediation
- Example showing how to use the report

For iteration artifacts:
- iteration-log-example.md: template showing iteration progression
- iteration-diff-example.md: how to compare transcripts between iterations

**Verification:** Given a failing transcript, can you classify the failure and find remediation?

### T3: B1 install + first-run validation (live claude-code-agent-teams)
**Goal:** Prove installation works by running it in a real claude-code-agent-teams session and capturing visible output.

Execute in claude-code-agent-teams (NOT simulated):
```bash
claude-code-controller launch "zsh"
claude-code-controller send "cd /path/to/project" --pane=X
claude-code-controller wait_idle --pane=X
claude-code-controller send "uv tool install -e ." --pane=X
claude-code-controller wait_idle --pane=X --idle-time=15.0
# Verify: output shows "Installed X packages" or similar
claude-code-controller send "uv run alphaswarm init ." --pane=X
claude-code-controller wait_idle --pane=X --idle-time=10.0
# Verify: output shows files created
claude-code-controller send "uv run alphaswarm --help" --pane=X
claude-code-controller wait_idle --pane=X --idle-time=5.0
# Verify: output shows command list
claude-code-controller send "uv run alphaswarm health-check --project . --json" --pane=X
claude-code-controller wait_idle --pane=X --idle-time=10.0
# Verify: JSON output with status fields
claude-code-controller capture --pane=X --output=.vrs/testing/runs/02-T3-b1-install/transcript.txt
```

Include negative cases:
- What happens if uv is not installed? (capture error message)
- What happens if Python is missing? (capture error message)
- What happens if Python version is wrong? (capture error message)

For B1-INSTALL-REPORT.md:
- Summary of each command's output
- Success indicators observed (not exit codes - visible output)
- Negative case results
- Any issues encountered and how resolved

**Verification:** Does the transcript show command OUTPUT, not just that commands ran?

## What Success Looks Like

**T1 Success:** A new developer can create an evidence pack by reading only guide-evidence.md and the example. The validator passes on the example.

**T2 Success:** Given a transcript where TaskCreate is missing, the iteration guide tells you:
- Category: missing-markers
- Diagnostic: search transcript for "TaskCreate", check workflow contract for when it should appear
- Remediation: verify skill is loaded, check for early exit, review workflow stage

**T3 Success:** The transcript shows:
- `uv tool install` output with "Installed X executables" visible
- `alphaswarm init` output showing created files
- `alphaswarm --help` showing available commands
- `alphaswarm health-check` showing JSON with status
- Error cases for missing uv and missing Python via PATH masking, plus wrong Python version via explicit interpreter path
- Transcript metadata shows >= 50 lines, >= 15s duration, and required markers present
