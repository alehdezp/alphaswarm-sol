# Phase 7.3.3: Adversarial Self-Testing (The Gauntlet)

## Purpose

Make the system prove it works by defeating challenges that are IMPOSSIBLE to pass unless the product genuinely works.

---

## The Problem It Solves

Current tests are "positive path" tests:
- They assume things work
- They check outputs exist
- A broken product can produce outputs that look right but are wrong

Adversarial tests are BINARY:
- Either you detect the vulnerability or you don't
- Either you avoid the false positive or you don't
- No partial credit, no "infrastructure exists"

---

## Gauntlet Design Philosophy

```
The best way to prove a system works is to try to break it.
If it survives designed attempts to break it, it probably works.
```

Each gauntlet:
1. Has a specific failure mode it tests
2. Is designed to catch specific weaknesses
3. Has binary pass/fail outcome
4. Cannot be gamed or worked around
5. Reveals root causes when failed

## Gauntlet 1: The Reentrancy Challenge

### Setup
10 contracts, identical structure, different security postures:

| # | Contract | Vulnerable? | Why |
|---|----------|-------------|-----|
| 1 | `Vault_01.sol` | YES | Classic CEI violation |
| 2 | `Vault_02.sol` | NO | Has ReentrancyGuard |
| 3 | `Vault_03.sol` | YES | Guard on wrong function |
| 4 | `Vault_04.sol` | NO | Uses pull pattern |
| 5 | `Vault_05.sol` | YES | Cross-function reentrancy |
| 6 | `Vault_06.sol` | NO | Checks-effects-interactions |
| 7 | `Vault_07.sol` | YES | Read-only reentrancy |
| 8 | `Vault_08.sol` | NO | Uses mutex lock |
| 9 | `Vault_09.sol` | YES | Reentrancy via callback |
| 10 | `Vault_10.sol` | NO | No external calls |

### Challenge Rules
```yaml
scoring:
  correct_vulnerable: +1
  correct_safe: +1
  false_positive: -2  # Penalize crying wolf
  false_negative: -2  # Penalize missing real bugs

pass_threshold: 10/10 (100% accuracy required)

failure_modes:
  - "Flags all as vulnerable" → Fails (5 false positives)
  - "Flags none as vulnerable" → Fails (5 false negatives)
  - "Random guessing" → Expected score: 0
  - "Only finds classic patterns" → Fails (misses cross-function, read-only)
```

### Why It Can't Be Gamed
- Classification is binary and verifiable
- Ground truth is external (we define the contracts)
- Requires understanding different reentrancy types
- Pattern matching alone won't achieve 100%

### What Failure Reveals
| Failure Pattern | Root Cause |
|-----------------|------------|
| Misses cross-function | Pattern doesn't track cross-function calls |
| Misses read-only | Pattern requires writes (wrong signature) |
| False positive on guarded | Guard detection broken |
| False positive on pull | Doesn't understand pull pattern |

## Gauntlet 2: Needle in Haystack

### Setup
51 contracts:
- 50 completely safe (but complex, look suspicious)
- 1 with subtle vulnerability

### Challenge Rules
```yaml
input:
  - 50 safe contracts (various DeFi patterns)
  - 1 vulnerable contract (hidden, not labeled)

output_required:
  - Identify THE vulnerable contract
  - Explain why it's vulnerable
  - Explain why the others are safe

pass_criteria:
  - Correct identification: required
  - Correct explanation: required
  - No false positives on safe contracts: required

scoring:
  identify_correct: PASS
  any_false_positive: FAIL (even with correct identification)
  explain_vulnerability: required for PASS
```

### Challenge Variants

**Variant A: Hidden Reentrancy**
- 50 contracts with external calls but proper guards
- 1 contract with subtle guard bypass

**Variant B: Hidden Access Control**
- 50 contracts with proper auth
- 1 contract with role confusion

**Variant C: Hidden Oracle Issue**
- 50 contracts with proper price handling
- 1 contract with staleness vulnerability

### Why It Can't Be Gamed
- Must correctly identify ONE out of 51
- Any false positive = FAIL
- Random chance: 1/51 = 2% (too low to luck into)
- Explanation requirement prevents lucky guesses

## Gauntlet 3: False Positive Trap

### Setup
Contracts designed to LOOK vulnerable but are ACTUALLY safe.

| # | Contract | Looks Like | Why It's Safe |
|---|----------|------------|---------------|
| 1 | `FPTrap_01.sol` | Reentrancy | Guard in base class |
| 2 | `FPTrap_02.sol` | Reentrancy | Only sends to trusted contract |
| 3 | `FPTrap_03.sol` | Access control | Admin is immutable and verified |
| 4 | `FPTrap_04.sol` | Oracle manipulation | Uses TWAP with sufficient window |
| 5 | `FPTrap_05.sol` | Integer overflow | Solidity 0.8+ with checks |
| 6 | `FPTrap_06.sol` | Flash loan attack | Protocol invariant preserved |
| 7 | `FPTrap_07.sol` | Frontrunning | Commit-reveal pattern used |
| 8 | `FPTrap_08.sol` | Sandwich attack | Slippage protection |
| 9 | `FPTrap_09.sol` | Price manipulation | Multiple oracle sources |
| 10 | `FPTrap_10.sol` | Governance attack | Timelock + multisig |

### Challenge Rules
```yaml
input:
  - 10 contracts that LOOK vulnerable
  - All 10 are actually SAFE

output_required:
  - Flag ZERO vulnerabilities
  - Or explain why each flagged item is actually safe

pass_criteria:
  - 0 false positives: PASS
  - 1+ false positives: FAIL

failure_penalty:
  - Each false positive documented with why it's wrong
```

### Why It Can't Be Gamed
- Only way to pass is to find ZERO vulnerabilities
- A system that over-reports will fail
- Tests precision specifically
- Reveals which guards the system doesn't understand

## Gauntlet 4: Multi-Agent Necessity Test

### Setup
Vulnerabilities that REQUIRE multi-agent debate to correctly classify.

| # | Scenario | Solo Agent Result | Swarm Result | Why |
|---|----------|-------------------|--------------|-----|
| 1 | Complex guard | False positive | True negative | Defender finds guard |
| 2 | Subtle exploit | True negative | True positive | Attacker finds path |
| 3 | Ambiguous code | Wrong | Correct | Verifier weighs evidence |
| 4 | Context-dependent | Uncertain | Confident | Debate resolves ambiguity |
| 5 | Multi-contract | Missed | Found | Cross-agent investigation |

### Challenge Rules
```yaml
mode_a: solo
  - Run AlphaSwarm with --mode solo
  - Record results

mode_b: swarm
  - Run AlphaSwarm with --mode swarm
  - Record results

pass_criteria:
  - swarm mode achieves >= 80% accuracy
  - swarm mode outperforms solo by >= 20% on designed cases
  - This proves multi-agent debate adds value

failure_modes:
  - "Solo same as swarm" → Debate is theater, not functional
  - "Swarm worse than solo" → Debate is broken
```

### Why It Can't Be Gamed
- Compares same system in two modes
- Designed cases are chosen to show debate value
- If debate doesn't add value, can't pass

## Gauntlet 5: Economic Context Challenge

### Setup
Vulnerabilities that ONLY exist in specific economic contexts.

| # | Protocol | Vulnerability | Without Context | With Context |
|---|----------|---------------|-----------------|--------------|
| 1 | Lending | Flash loan attack | Not found | Found |
| 2 | DEX | Sandwich attack | Not found | Found |
| 3 | Vault | MEV extraction | Not found | Found |
| 4 | Bridge | Front-running | Not found | Found |
| 5 | Governance | Vote buying | Not found | Found |

### Challenge Rules
```yaml
pass_criteria:
  - Must detect >= 4/5 context-dependent vulnerabilities
  - Detection must cite economic context (Exa research)
  - Cannot pass without using Exa MCP

verification:
  - Check Exa tool calls in transcript
  - Check context pack references similar exploits
  - Check finding cites economic risk
```

### Why It Can't Be Gamed
- Vulnerabilities are invisible without context
- Context comes from external research
- No hardcoding possible (protocols change)

## Self-Improvement Loop

```
┌─────────────────────────────────────────────────────────────┐
│                   GAUNTLET CYCLE                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. RUN gauntlet (all 5 challenges)                         │
│     ↓                                                       │
│  2. RECORD exact failures                                   │
│     │                                                       │
│     ├── All gauntlets passed? ────────→ PHASE COMPLETE     │
│     │                                                       │
│     └── Any gauntlet failed ────→ 3. ANALYZE failure       │
│                                          ↓                  │
│                                     4. IDENTIFY root cause  │
│                                          ↓                  │
│                                     5. FIX product (not test)│
│                                          ↓                  │
│                                     6. GOTO 1 (re-run all)  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### Failure Analysis Template

```yaml
gauntlet: "Reentrancy Challenge"
failure:
  expected: "Vault_05.sol is vulnerable (cross-function)"
  actual: "Vault_05.sol marked as safe"

investigation:
  - Does pattern detect cross-function reentrancy?
  - Does graph track calls between functions?
  - Did agent investigate cross-function paths?

root_cause: "Pattern only checks single-function CEI violation"

fix_required:
  - Add cross-function reentrancy pattern
  - Update graph builder to track internal call graph
  - Update agent to investigate cross-function paths

verification: "Re-run gauntlet, Vault_05 must be flagged"
```

## Suite Registry

All gauntlets must be registered in a suite registry with:
- suite_id
- corpus manifest path
- scoring config
- expected evidence pack outputs

This ensures repeatable runs and standardized outputs.

## Anti-Perfect Metrics Rule

If any gauntlet suite reports 100% precision and 100% recall, the run is marked **SUSPICIOUS** and must trigger an investigation with an expanded set. This prevents overfitting and silent regressions.

## Needle-in-Haystack Update

Replace "explain all safe contracts" with:
- **No false positives**
- **Explain top-5 most suspicious safe cases**

This makes the challenge binary without forcing exhaustive explanations.
