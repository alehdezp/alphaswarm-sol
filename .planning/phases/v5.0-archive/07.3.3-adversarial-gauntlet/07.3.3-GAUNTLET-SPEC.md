# 07.3.3 Gauntlet Specification

## Overview

The Adversarial Gauntlet is a suite of binary pass/fail challenges designed to prove the AlphaSwarm.sol system works correctly. Unlike standard tests, gauntlet challenges:

- Have **binary outcomes** (pass or fail, no partial credit)
- Use **explicit ground truth** (pre-defined correct answers)
- Require **evidence packs** (proof of execution)
- Allow **no retries without fixes** (must address root cause)

---

## Core Principles

### 1. Binary Scoring

Every challenge has exactly two outcomes:
- **PASS**: Correct answer with valid evidence
- **FAIL**: Wrong answer OR missing evidence

No partial credit. No "close enough." No curves.

### 2. Non-Gameable Design

Challenges are designed to resist gaming:
- Ground truth is external and pre-defined
- Random guessing yields expected score of zero
- Pattern matching alone cannot achieve perfect scores
- Answers require genuine understanding

### 3. Evidence Requirements

Every gauntlet run MUST produce:
- Proof tokens for all 6 stages (graph, context, pattern, agents, debate, report)
- Evidence pack with transcript hashes
- Nonce verification in transcripts
- Duration > 5000ms (no instant/mock runs)

### 4. No Retry Without Fix

If a gauntlet fails:
1. Root cause must be identified
2. Product fix must be implemented
3. Only then can gauntlet be re-run
4. Fix-then-retest cycle continues until pass

## Gauntlet Suites

### Suite 1: Reentrancy Challenge (gauntlet-reentrancy)

**Challenge:** Classify 10 vault contracts as vulnerable or safe.

| Contract | Ground Truth | Vulnerability Type |
|----------|--------------|-------------------|
| Vault_01.sol | VULNERABLE | Classic CEI violation |
| Vault_02.sol | SAFE | Has ReentrancyGuard |
| Vault_03.sol | VULNERABLE | Guard on wrong function |
| Vault_04.sol | SAFE | Uses pull pattern |
| Vault_05.sol | VULNERABLE | Cross-function reentrancy |
| Vault_06.sol | SAFE | Checks-effects-interactions |
| Vault_07.sol | VULNERABLE | Read-only reentrancy |
| Vault_08.sol | SAFE | Uses mutex lock |
| Vault_09.sol | VULNERABLE | Reentrancy via callback |
| Vault_10.sol | SAFE | No external calls |

**Pass Criteria:**
- 10/10 correct classifications
- No false positives
- No false negatives

**Required Gates:** G0, G1, G2, G4

### Suite 2: Access Control & Oracle Challenge (gauntlet-access-oracle)

**Challenge:** Classify 10 contracts for access control and oracle vulnerabilities.

| Contract | Ground Truth | Vulnerability Type |
|----------|--------------|-------------------|
| AccessVault_01.sol | VULNERABLE | Missing onlyOwner |
| AccessVault_02.sol | SAFE | Proper role checks |
| AccessVault_03.sol | VULNERABLE | tx.origin auth |
| AccessVault_04.sol | SAFE | OpenZeppelin Ownable |
| AccessVault_05.sol | VULNERABLE | Ownership manipulation |
| OracleVault_01.sol | VULNERABLE | No staleness check |
| OracleVault_02.sol | SAFE | Proper TWAP |
| OracleVault_03.sol | VULNERABLE | Single oracle source |
| OracleVault_04.sol | SAFE | Multi-oracle aggregation |
| OracleVault_05.sol | VULNERABLE | Missing L2 sequencer grace |

**Pass Criteria:**
- 10/10 correct classifications
- Each finding must cite specific code location
- Oracle findings must reference price manipulation path

**Required Gates:** G0, G1, G2, G4

---

### Suite 3: Mutation Robustness (gauntlet-mutation)

**Challenge:** Detect vulnerabilities across adversarial mutations.

**Mutation Types:**
1. **Rename mutations**: Variables/functions renamed
2. **Reorder mutations**: Statement order changed
3. **Comment mutations**: Misleading comments added
4. **Wrapper mutations**: Extra abstraction layers
5. **Split mutations**: Logic split across functions

**Matrix:**
- 5 base vulnerable contracts
- 5 mutation types each
- 25 total mutated contracts
- 5 safe baseline contracts (negative controls)

**Pass Criteria:**
- Detection rate >= 75% on mutated contracts
- 0 false positives on safe baselines
- Consistent classification across mutation types

**Required Gates:** G0, G1, G2, G4

## Scoring Rules

### Point Values

```yaml
correct_vulnerable: +1   # True positive
correct_safe: +1         # True negative
false_positive: -2       # False alarm (penalized 2x)
false_negative: -2       # Missed vulnerability (penalized 2x)
```

### Rationale

False positives and false negatives are penalized 2x because:
- False positives waste auditor time and erode trust
- False negatives leave real vulnerabilities undetected
- A system that flags everything or nothing will score negative

### Pass Thresholds

| Suite | Minimum Score | Required Accuracy |
|-------|---------------|-------------------|
| gauntlet-reentrancy | 10/10 | 100% |
| gauntlet-access-oracle | 10/10 | 100% |
| gauntlet-mutation | 75%+ detection | 0 FP on safe |

### Anti-Perfect Metrics Rule

If any suite achieves 100% precision AND 100% recall:
1. Mark run as **SUSPICIOUS**
2. Expand test set with additional edge cases
3. Investigate for overfitting or data leakage
4. Re-validate with expanded corpus

## Evidence Pack Requirements

Every gauntlet run MUST produce a valid evidence pack:

```yaml
evidence_pack:
  run_id: "gauntlet-reentrancy-{timestamp}"
  version: "1.0.0"

  proofs:
    stage_graph:
      started_at: "2026-01-31T00:00:00Z"
      completed_at: "2026-01-31T00:01:00Z"
      hash: "sha256:..."
      nonce: "abc123"
    # ... stages 2-6 ...

  transcripts:
    - path: ".vrs/transcripts/gauntlet-run.txt"
      hash: "sha256:..."
      line_count: 150

  artifacts:
    - path: ".vrs/validation/gauntlet-reentrancy-results.json"
      hash: "sha256:..."

  duration_ms: 45000
  mode: "live"  # MUST be "live", not "mock"
```

### Validation Rules

Evidence packs are validated against:
- **G1 (Evidence Integrity):** All proof tokens present, hashes match
- **G2 (Graph Soundness):** BSKG hash valid, operations detected
- **G4 (Mutation Robustness):** Detection thresholds met
- **G6 (Regression Baseline):** No precision/recall drop

## Execution Protocol

### Step 1: Preflight (G0)

```bash
# Verify environment
claude-code-agent-teams -V                    # claude-code-agent-teams available
alphaswarm --version       # CLI version correct
ls tests/gauntlet/         # Corpus present
```

### Step 2: Run Gauntlet

```bash
# Execute gauntlet suite with evidence collection
./scripts/e2e/run_gauntlet_suite.sh --suite gauntlet-reentrancy
```

### Step 3: Validate Evidence

```bash
# Validate evidence pack
python scripts/e2e/validate_evidence_pack.py .vrs/evidence/pack.json

# Validate gates
python scripts/e2e/validate_all_gates.py --gates G0,G1,G2,G4
```

### Step 4: Record Results

Results are recorded in:
- `.vrs/validation/gauntlet-results.json` (machine-readable)
- `.vrs/validation/gauntlet-results.md` (human-readable)
- `.planning/phases/07.3.3-adversarial-gauntlet/07.3.3-GAUNTLET-RESULTS.md` (official record)

## Failure Analysis Protocol

When a gauntlet fails:

### 1. Identify Failure Pattern

```yaml
failure:
  suite: "gauntlet-reentrancy"
  contract: "Vault_05.sol"
  expected: "VULNERABLE (cross-function)"
  actual: "SAFE"
  classification: "FALSE_NEGATIVE"
```

### 2. Root Cause Investigation

- Does pattern detect this vulnerability type?
- Does graph capture required relationships?
- Did agent investigate correct paths?
- Did debate consider all evidence?

### 3. Fix Implementation

Fix MUST address root cause, not symptom:

```yaml
root_cause: "Pattern only checks single-function CEI"
fix:
  - "Add cross-function reentrancy pattern"
  - "Update graph builder for internal call tracking"
  - "Enhance agent investigation prompts"
```

### 4. Verification

- Re-run FULL gauntlet (not just failed case)
- All gates must pass
- No regression on previously passing cases

## Gate Integration

Gauntlet results integrate with existing gates:

| Gate | Gauntlet Relevance |
|------|-------------------|
| G0 (Preflight) | Environment validation before run |
| G1 (Evidence) | Proof tokens required for each run |
| G2 (Graph) | BSKG must be valid for classification |
| G3 (Ground Truth) | External corpus provides ground truth |
| G4 (Mutation) | Mutation gauntlet enforces this gate |
| G5 (Variance) | Multi-run stability across gauntlet reruns |
| G6 (Baseline) | No regression vs locked gauntlet scores |
| G7 (Continuous) | Gauntlet runs in CI/CD pipeline |

## Registry

All gauntlet suites are registered in:
`configs/gauntlet_scoring.yaml`

Registry entry format:
```yaml
suites:
  - suite_id: "gauntlet-{name}"
    manifest: "tests/gauntlet/{name}_manifest.yaml"
    scoring: "configs/gauntlet_scoring.yaml"
    required_gates: [G0, G1, G2, G4]
    evidence_pack_required: true
```

## Exit Criteria

Phase 7.3.3 is COMPLETE when ALL gauntlets pass:

1. **gauntlet-reentrancy:** 10/10 correct
2. **gauntlet-access-oracle:** 10/10 correct
3. **gauntlet-mutation:** >= 75% detection, 0 FP on safe

ALL required gates (G0, G1, G2, G4) must pass for each suite.
