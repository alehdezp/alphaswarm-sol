## Reasoning
The earlier plan lacked stress coverage for git workflow variants and failure injection, which are frequent sources of orchestration bugs. This plan expands the stress matrix to include these dimensions.

## Purpose
- Validate multi-project stability.
- Capture claude-code-agent-teams evidence across stress matrix.
- Ensure recovery runs exist for failures.
- Validate resilience under git workflow variants.

## Why
- Multi-worktree concurrency can reveal hidden orchestration bugs.
- Git state differences can break worktree assumptions.
- Recovery behavior is critical for production reliability.

### T1: Define stress matrix
**Goal:** Specify repo/workflow/worktree combinations and git variants.

### T2: Implement stress runner
**Goal:** Automate claude-code-controller runs and evidence capture.

### T3: Execute stress runs and capture evidence
**Goal:** Produce transcripts and evidence packs for all matrix entries.

### T4: Execute chaos injection cases
**Goal:** Inject failure conditions and validate recovery runs.

### T5: Summarize failures and recovery reruns
**Goal:** Document stability findings.

### T6: Tooling update
**Goal:** Document stress runner in TOOLING.md.
