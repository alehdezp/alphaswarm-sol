# Phase 7.1: Testing Infrastructure & Framework - Research

**Researched:** 2026-01-22
**Domain:** Software testing infrastructure, multi-agent system testing, LLM evaluation
**Confidence:** HIGH

## Summary

Phase 7.1 builds the AlphaSwarm Test Forge — a testing infrastructure for validating multi-agent vulnerability detection systems. This research investigated testing patterns for multi-agent systems, corpus management approaches, model comparison frameworks, and implementation dependencies.

**Key findings:**
1. **Phase 6 dependency is SOFT** — testing can proceed with current `true_vkg` naming and migrate during execution
2. **Existing testing infrastructure is STRONG** — pytest-xdist configured (3.79x speedup), 244 test files, fixtures in place
3. **Multi-agent testing requires layered approach** — component, integration, E2E with deterministic mocking
4. **Corpus management is specialized** — security testing requires manual ground truth, academic ML practices don't directly apply
5. **Model comparison is well-established** — EleutherAI LM Harness, HELM, DeepEval provide patterns
6. **Gap tracking is straightforward** — YAML schema with categorization, similar to issue tracking

**Primary recommendation:** Proceed with Phase 7.1 planning now. Begin implementation with current naming, insert rebrand task (07.1-09) to align with Phase 6 completion.

### Pattern 1: Layered Agent Testing

**What:** Three-tier testing pyramid for multi-agent systems

**When to use:** Testing any system with spawned agents, orchestration, or agent interactions

**Example:**
```python
# LAYER 1: Component tests (isolated agent behavior)
# Source: Pydantic AI testing docs
def test_context_merge_agent_component(mock_llm):
    """Test agent in isolation with deterministic responses."""
    agent = ContextMergeAgent(model=TestModel())

    # Mock LLM returns fixed response
    result = agent.merge_contexts(vuln_class="reentrancy", bundles=[...])

    assert result.status == "complete"
    assert "CEI pattern" in result.reasoning

# LAYER 2: Integration tests (agent interactions)
def test_context_merge_to_bead_flow(temp_pool):
    """Test agent interaction via real beads/storage."""
    coordinator = SubCoordinator()

    # Real bead creation, real skill invocation
    bead = await coordinator.spawn_context_merge(vuln_class="reentrancy")

    assert bead.status == "verified"
    assert len(bead.finding_bead_ids) > 0

# LAYER 3: E2E tests (full pipeline)
def test_full_audit_pipeline(test_contract):
    """Test complete flow from source to verdicts."""
    result = run_audit(contract_path=test_contract)

    assert result.precision >= 0.85
    assert result.recall_critical >= 0.95
```

### Pattern 2: Deterministic Agent Testing

**What:** Use mock models for predictable agent behavior in tests

**When to use:** Unit/component tests where real LLM calls are too slow or non-deterministic

**Example:**
```python
# Source: Pydantic AI TestModel pattern
from pydantic_ai import TestModel, FunctionModel

@pytest.fixture
def deterministic_agent():
    """Agent with fixed responses for testing."""
    test_model = TestModel()
    # Pre-program expected responses
    test_model.add_response(
        "Analyze reentrancy in withdraw()",
        {"finding": "reentrancy-classic", "confidence": 0.95}
    )
    return VulnDiscoveryAgent(model=test_model)

def test_vuln_discovery_deterministic(deterministic_agent):
    """Test agent logic without LLM variability."""
    result = deterministic_agent.analyze(function_id="withdraw")
    assert result["finding"] == "reentrancy-classic"
```

**What:** Manual ground truth with versioned snapshots

**When to use:** Security/vulnerability testing where labels require expert knowledge

**Key insight:** Security testing corpus requires manual curation by experts. Academic ML corpus management (automated labeling, crowd-sourcing) doesn't apply — vulnerabilities require deep semantic understanding.

### Pattern 4: Model Comparison Framework

**What:** Task-type-specific benchmarking with accuracy > cost prioritization

**When to use:** Comparing multiple LLMs for security analysis tasks

**Example:**
```python
# Source: EleutherAI LM Harness pattern
class ModelRankingUpdater:
    """Update rankings based on test corpus performance."""

    def benchmark_model(self, model_id: str, task_type: str, corpus: List[TestCase]):
        """Run model on corpus, calculate metrics."""
        results = []
        for test_case in corpus:
            response = self.run_model(model_id, test_case.prompt)
            results.append(self.evaluate(response, test_case.ground_truth))

        precision = self.calculate_precision(results)
        recall = self.calculate_recall(results)
        f1 = 2 * (precision * recall) / (precision + recall)

        # Update EMA-based ranking (from Phase 5.3)
        self.update_ranking(task_type, model_id, f1_score=f1)
```

**Dimensions to track:**
- Precision, recall, F1 per task type (verify, analyze, detect, summarize)
- Cost per 1000 findings
- Latency per task
- False positive rate (critical for usability)

### Anti-Patterns to Avoid

- **Session-scoped graph fixtures:** Don't help with pytest-xdist, add complexity (Phase 8 finding)
- **Mutable shared state between tests:** Breaks parallel execution
- **Auto-confirming tool findings:** All findings go through verification (security critical)
- **Optimizing for cost before accuracy:** Accuracy must meet GA gates first (≥85% precision)

## Don't Hand-Roll

Problems that look simple but have existing solutions:

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Parallel test execution | Custom multiprocessing | pytest-xdist | 3.79x speedup validated, mature, handles fixtures |
| LLM response mocking | Manual mock classes | TestModel/FunctionModel | Deterministic, pytest-compatible |
| Test corpus versioning | Git-based versioning | SQLite with corpus_version column | Immutable snapshots, query by version |
| Model benchmarking | Custom eval loop | EleutherAI LM Harness or DeepEval | 60+ standard benchmarks, pip installable |
| Gap tracking | Custom issue tracker | YAML files + categorization schema | Simple, git-friendly, grep-able |
| Incremental testing | Custom change detection | pytest-testmon | Coverage-based, already validated |

**Key insight:** Don't build custom test infrastructure. pytest ecosystem + simple data structures (SQLite + YAML) are sufficient. Focus effort on domain-specific testing logic (vulnerability detection accuracy).

## Common Pitfalls

### Pitfall 1: Phase 6 Hard Dependency

**What goes wrong:** Waiting for Phase 6 rebrand before starting Phase 7.1 delays testing infrastructure by weeks

**Why it happens:** Roadmap shows Phase 7.1 depends on Phase 6, context doc uses AlphaSwarm naming throughout

**How to avoid:**
1. Start Phase 7.1 with current `true_vkg` naming
2. Insert rebrand task (07.1-09) to update tests/infrastructure once Phase 6 completes
3. Use search-replace for naming changes (low risk, high automation)

**Warning signs:** Blocking Phase 7.1 planning until Phase 6 is complete

### Pitfall 2: Over-Engineering Corpus Management

**What goes wrong:** Building complex versioning, labeling, or management systems for test corpus

**Why it happens:** ML research has sophisticated corpus infrastructure (HuggingFace datasets, annotation platforms)

**How to avoid:**
1. Use SQLite for structured storage (no DB server needed)
2. Manual ground truth labeling by security experts (no crowd-sourcing)
3. Simple immutable versions (corpus v1, v2, v3)
4. Git for backup/history, not primary versioning

**Warning signs:**
- Investigating annotation platforms (LabelStudio, Prodigy)
- Building API for corpus management
- Considering automated labeling pipelines

### Pitfall 3: Real LLM Calls in Unit Tests

**What goes wrong:** Unit tests become slow, flaky, expensive when calling real LLM APIs

**Why it happens:** Agent code uses LLM calls, easy to forget to mock in tests

**How to avoid:**
1. Use TestModel for deterministic responses in component tests
2. Reserve real LLM calls for integration/E2E tests only
3. Set `ALLOW_MODEL_REQUESTS=False` in test environment
4. Use pytest fixtures to inject mock models

**Warning signs:**
- Test suite takes >10 minutes with xdist
- Tests fail intermittently due to rate limits
- Unexpected OpenAI/Anthropic bills during test runs

### Pitfall 4: Premature Model Comparison

**What goes wrong:** Spending effort benchmarking free models before core accuracy is validated

**Why it happens:** Context doc emphasizes cost optimization and free model evaluation

**How to avoid:**
1. Validate GA accuracy gates FIRST (≥85% precision, ≥95% recall critical)
2. Use Opus/Sonnet baseline for initial testing
3. Add model comparison only after core accuracy proven
4. Phase 7.1 builds infrastructure, Phase 7.2 runs full evaluation

**Warning signs:**
- Testing DeepSeek, MiniMax, Grok before baseline works
- Investigating OpenRouter integration before accuracy validated
- Comparing costs before quality meets targets

### Pitfall 5: Missing Agent Orchestration Validation

**What goes wrong:** Testing individual agents but not orchestrator delegation, concurrency limits, or failure recovery

**Why it happens:** Easier to test agents in isolation than full orchestration

**How to avoid:**
1. Test orchestrator delegation (orchestrators don't do work directly)
2. Validate concurrency limits (max 5 subagents, max 2 sub-orchestrators)
3. Test failure recovery via bead state persistence
4. E2E tests must exercise full agent spawning flow

**Warning signs:**
- Only component tests exist for agents
- No tests for SubCoordinator or MainOrchestrator
- Bead state persistence not tested
- No tests for agent spawning with real skills

# Source: Context doc metrics schema
from dataclasses import dataclass
from typing import Dict, Optional

@dataclass
class CategoryMetrics:
    """Metrics for a specific vulnerability category."""
    precision: float
    recall: float
    f1_score: float
    tp: int
    fp: int
    fn: int

@dataclass
class SegmentMetrics:
    """Metrics for a corpus segment (recent-audits, mutations, adversarial)."""
    precision: float
    recall: float
    contracts_tested: int
    findings_correct: int
    findings_incorrect: int

@dataclass
class TestMetrics:
    """Complete test metrics for a run."""
    # Core accuracy
    precision: float
    recall: float
    f1_score: float

    # Severity-weighted recall
    recall_critical: float
    recall_high: float

    execution_time_ms: int
    contracts_tested: int

### Example 5: Model Comparison Test

```python
# Source: EleutherAI LM Harness pattern
import pytest
from typing import List

class ModelComparison:
    """Compare multiple models on same corpus."""

    def benchmark_models(
        self,
        models: List[str],
        task_type: str,
        corpus_segment: str
    ) -> Dict[str, TestMetrics]:
        """Run all models on corpus, return metrics per model."""
        results = {}
        test_cases = self.load_corpus(corpus_segment)

        for model_id in models:
            metrics = self.run_model_on_corpus(model_id, task_type, test_cases)
            results[model_id] = metrics

        return results

    def calculate_vs_opus_accuracy(
        self,
        model_metrics: TestMetrics,
        opus_metrics: TestMetrics
    ) -> float:
        """Calculate percentage of Opus F1 achieved by model."""
        if opus_metrics.f1_score == 0:
            return 0.0
        return (model_metrics.f1_score / opus_metrics.f1_score) * 100
```

## Open Questions

Things that couldn't be fully resolved:

1. **Agent testing framework choice**
   - What we know: agenttest exists, TestModel/FunctionModel work with pytest
   - What's unclear: Is agenttest worth adding, or is TestModel sufficient?
   - Recommendation: Start with TestModel (Pydantic built-in), add agenttest if gaps found

2. **Model comparison framework choice**
   - What we know: EleutherAI LM Harness (60+ benchmarks), DeepEval (pytest-like), HELM (holistic)
   - What's unclear: Which fits security testing best? Custom needed?
   - Recommendation: Try DeepEval first (pytest-like interface), fallback to custom if insufficient

3. **Corpus size for GA gate**
   - What we know: Context doc mentions "50+ real protocols" in Phase 7.2
   - What's unclear: How many contracts minimum for Phase 7.1 infrastructure validation?
   - Recommendation: Start with 10-20 contracts for infrastructure testing, full corpus in 7.2

4. **Gap tracking tool integration**
   - What we know: YAML schema defined, manual tracking via skills
   - What's unclear: Should gaps integrate with GitHub Issues or stay independent?
   - Recommendation: Stay YAML-based for Phase 7.1, consider GitHub integration if team collaboration needed

5. **Mutation testing depth**
   - What we know: 40% of corpus should be mutations (10x variants per pattern)
   - What's unclear: Semantic preservation validation — how to verify mutation preserves vulnerability?
   - Recommendation: Manual verification for Phase 7.1, consider symbolic execution in future
