# Phase 7.1: Autonomous Testing Framework Synthesis

**Synthesized:** 2026-01-22
**Purpose:** Comprehensive documentation of all discovered testing approaches, frameworks, and patterns for building the "best-in-class" self-improving, brutally honest testing system.

## Executive Summary

This document synthesizes research from 40+ sources to define a comprehensive autonomous testing framework for True VKG. The framework enables:

1. **Fully Autonomous Testing** - Claude Code CLI invocation without human intervention
2. **Three-Level Testing** - Unit (subagents) → Integration (sub-orchestrators) → E2E (full orchestrators)
3. **Dual Invocation Patterns** - CLI subprocess + Direct Task tool spawning
4. **Self-Improving Feedback Loops** - Continuous refinement based on performance data
5. **Chaos Engineering** - Systematic failure injection for robustness testing
6. **Blind/Adversarial Testing** - Real-world simulation without solution leakage
7. **Multi-Model Benchmarking** - Comparative analysis with EMA ranking updates
8. **Trajectory Analysis** - Beyond final answers, evaluating reasoning paths

## Critical Architecture: Three-Level Testing Pyramid

```
                    ┌─────────────────────────────────┐
                    │     Level 3: E2E Tests          │
                    │  Full Orchestrators (vrs:audit) │
                    │  CLI + Direct spawning          │
                    └───────────────┬─────────────────┘
                                    │
                    ┌───────────────┴─────────────────┐
                    │  Level 2: Integration Tests     │
                    │  Sub-Orchestrators (debate,     │
                    │  verification loop, routing)    │
                    └───────────────┬─────────────────┘
                                    │
        ┌───────────────────────────┴───────────────────────────┐
        │              Level 1: Unit Tests                      │
        │  Individual Subagents (attacker, defender, verifier)  │
        │  Pattern detection, VKG queries, tool execution       │
        └───────────────────────────────────────────────────────┘
```

**Bottom-Up Testing Strategy:**
1. First ensure individual subagents work correctly in isolation
2. Then test how subagents communicate in sub-orchestration flows
3. Finally test full end-to-end orchestration with real-world scenarios

## Two Invocation Patterns

### Pattern 1: CLI Subprocess (External Testing)

```python
# For testing Claude Code CLI features, session management, hooks
result = subprocess.run(
    ["claude", "-p", prompt, "--output-format", "json", "--allowedTools", tools],
    capture_output=True, text=True
)
```

**Use Cases:**
- Testing CLI-specific features (hooks, sessions, output formats)
- External/black-box testing of complete workflows
- CI/CD pipeline integration
- Cross-process isolation testing

### Pattern 2: Direct Task Tool Spawning (Internal Testing)

```python
# Within Claude Code session - spawn subagent directly
Task(
    prompt="Analyze {contract} for reentrancy vulnerabilities",
    subagent_type="vkg-attacker",
    model="claude-opus-4",
    description="Test attacker subagent"
)
```

**Use Cases:**
- Testing subagent behavior with full context access
- Unit testing individual agents in isolation
- Faster iteration (no subprocess overhead)
- Direct access to agent internals and trajectory

### Combining Both Patterns

```python
class HybridTestRunner:
    """Combines CLI and direct spawning for comprehensive testing."""

    def test_subagent_direct(self, agent_type, prompt, expected):
        """Level 1: Direct spawn for unit testing."""
        # This runs within a Claude Code session
        result = Task(
            prompt=prompt,
            subagent_type=agent_type,
            model="haiku"  # Faster for unit tests
        )
        return self.evaluate(result, expected)

    def test_orchestrator_cli(self, orchestrator_prompt, expected):
        """Level 3: CLI invocation for E2E testing."""
        # This spawns external process
        result = subprocess.run(
            ["claude", "-p", orchestrator_prompt, "--output-format", "json"],
            capture_output=True
        )
        return self.evaluate(json.loads(result.stdout), expected)
```

```python
class ClaudeCodeRunner:
    def run_analysis(self, prompt, allowed_tools, json_schema):
        cmd = ["claude", "-p", prompt, "--output-format", "json"]
        if allowed_tools:
            cmd.extend(["--allowedTools", ",".join(allowed_tools)])
        if json_schema:
            cmd.extend(["--json-schema", json.dumps(json_schema)])
        result = subprocess.run(cmd, capture_output=True, text=True)

with ThreadPoolExecutor(max_workers=4) as executor:
```

## Part 2: Self-Improving Feedback Loops

### 2.1 AgentDevel: Release Engineering Approach

**Source:** arXiv 2601.04620 (Jan 2026)

Treats agent improvement as **release engineering** rather than in-agent self-modification:
- Agents are shippable artifacts
- Improvement is externalized into regression-aware pipelines
- Non-regression guarantees across versions
- Audit trails for improvement trajectories

**Implementation Pattern:**
```python
class AgentVersionManager:
    """Manage agent versions with regression testing."""

    def propose_improvement(self, agent_id, improvement):
        """Stage improvement for regression testing."""
        candidate = self.create_candidate(agent_id, improvement)
        regression_results = self.run_regression_suite(candidate)

        if regression_results.passes_all():
            return self.promote_candidate(candidate)
        else:
            return self.reject_with_report(candidate, regression_results)
```

### 2.3 Evolver: Meta-Agent Optimization

**Source:** aiXplain Evolver (Oct 2025)

Meta-agent that automatically optimizes other agents:
- Evolutionary optimization without code changes
- Prompt refinement based on performance
- Hyperparameter tuning for tool selection

**VKG Application:**
```python
class AgentEvolver:
    """Meta-agent that optimizes VKG agents."""

    def evolve(self, agent, benchmark_suite, generations=10):
        population = [agent.fork() for _ in range(10)]

        for gen in range(generations):
            scores = [self.evaluate(a, benchmark_suite) for a in population]
            population = self.select_and_mutate(population, scores)

        return max(population, key=lambda a: self.evaluate(a, benchmark_suite))

    def mutate(self, agent):
        """Mutate agent configuration."""
        mutations = [
            self._mutate_system_prompt,
            self._mutate_tool_selection,
            self._mutate_temperature,
            self._mutate_reasoning_style
        ]
        return random.choice(mutations)(agent)
```

## Part 4: Blind & Adversarial Testing

### 4.1 Contamination Prevention

**Source:** LiveBench, DCR Paper, EMNLP 2025

Key problem: Models may have seen benchmark data during training.

**Mitigation Strategies:**
1. **Post-cutoff contracts** - Use audits from after model training cutoff
2. **Paraphrased prompts** - Rephrase detection queries
3. **Dynamic generation** - Generate new test cases programmatically
4. **Blind mode** - Don't reveal expected vulnerability class

### 6.1 Comparative Benchmarking

```python
class ModelBenchmark:
    """Compare models on VKG tasks."""

    MODELS = [
        "claude-opus-4",
        "claude-sonnet-4",
        "claude-haiku-4"
    ]

    TASK_TYPES = [
        "detect",       # Vulnerability detection
        "verify",       # Finding verification
        "construct",    # Exploit construction
        "defend",       # Defense identification
    ]

    def run_comparison(self, corpus, task_type):
        results = {}

        for model in self.MODELS:
            start_time = time.time()

            metrics = self._run_model_on_corpus(model, corpus, task_type)

            results[model] = {
                "precision": metrics.precision,
                "recall": metrics.recall,
                "f1_score": metrics.f1,
                "latency_p50": metrics.latency_p50,
                "latency_p95": metrics.latency_p95,
                "cost_per_contract": metrics.cost,
                "total_time": time.time() - start_time,
            }

        return self._rank_models(results)
```

### 6.2 EMA Ranking Updates

**Source:** 07.1-CONTEXT.md existing design

```python
class ModelRankingUpdater:
    """Update model rankings with exponential moving average."""

    def __init__(self, alpha: float = 0.3):
        self.alpha = alpha

    def update(self, model: str, task_type: str, new_score: float):
        """Update EMA for model on task type."""
        current = self.rankings.get_ema(model, task_type)

        # EMA formula: new_ema = alpha * new_score + (1 - alpha) * current
        new_ema = self.alpha * new_score + (1 - self.alpha) * current

        self.rankings.set_ema(model, task_type, new_ema)

        # Update recommendation
        self._recalculate_recommendations(task_type)
```
