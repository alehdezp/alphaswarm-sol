<domain>
## Phase Boundary

Build the AlphaSwarm Test Forge — a comprehensive testing ecosystem that validates the multi-agent vulnerability detection system. This phase creates testing infrastructure that exercises the full VRS (Vulnerability Reasoning System) pipeline, from BSKG graph building through context-merge agents to final verdicts.

**Core Deliverable:** Reusable testing framework with corpus management, model comparison infrastructure, agent orchestration testing, accuracy measurement, and gap tracking.

**Key Integration Points:**
- **Phase 5.3:** Model ranking infrastructure (`rankings.yaml`), task router, OpenCode SDK
- **Phase 5.4:** VulnDocs with `index.yaml` schema (`reasoning_template`, `semantic_triggers`, `vql_queries`)
- **Phase 5.5:** Agent flow (context-merge → verifier → bead → vuln-discovery), evidence chains in beads
- **Phase 5.6:** Skill organization (`/vrs-*`), orchestrator delegation pattern, health checks
- **Phase 6:** AlphaSwarm rebrand (all new code uses `alphaswarm`, `aswarm`, `vrs-*`, `.vrs/`)

**Does NOT include:** Actual corpus population at scale, shadow audit execution against real protocols, or GA dossier generation (those are Phase 7.2).

**Naming:** All new code uses AlphaSwarm naming (post Phase 6 rebrand). Agents use `vrs-*` prefix, skills use `/vrs-*`, config in `.vrs/`.

<corpus-design>
## Corpus Design

### Ground Truth Philosophy

**Critical Rule:** Use vulnerabilities NOT in LLM training data.

**Balanced Composition (100% total):**
- **30% Recent Audits** — Post-January 2025 audit reports
  - Higher validity (outside training cutoffs)
  - Requires manual curation
  - Source: Code4rena, Sherlock, private audits (with permission)

- **40% Contract Mutations** — Auto-generated variants
  - Variable renaming, function reordering, control flow changes
  - Generate 10x variants per known pattern
  - Volume for statistical significance

- **30% Adversarial Hand-Crafted** — Novel scenarios
  - Custom contracts specifically designed to challenge detection
  - False positive traps (safe code that looks vulnerable)
  - Edge cases (proxies, delegatecall, cross-contract)
  - Obfuscation techniques

### Corpus Structure

```
.vrs/
├── corpus/
│   ├── corpus.db              # SQLite - contracts, findings, ground truth
│   ├── contracts/
│   │   ├── recent-audits/     # Post-2025 audit findings
│   │   ├── mutations/         # Auto-generated variants
│   │   ├── adversarial/       # Hand-crafted challenges
│   │   └── safe/              # Known safe (false positive testing)
│   ├── ground-truth/
│   │   ├── recent-audits.json # Audit findings with severity
│   │   └── mutations.json     # Expected detection for mutations
│   └── benchmarks/
│       ├── quick.yaml         # < 2 min smoke tests
│       ├── standard.yaml      # < 10 min regression
│       └── thorough.yaml      # < 30 min exhaustive
```

### Mutation Generation

**Mutation Types:**
1. **Identifier renaming** — Variable/function names changed
2. **Code reordering** — Statement order permuted (where semantically equivalent)
3. **Pattern variation** — Same vulnerability, different implementation
4. **Structural changes** — Split functions, merge functions, extract modifiers

**Validation:** Each mutation verified to preserve vulnerability (or explicitly mark as safe variant).

### Corpus Versioning

- Immutable snapshots: Each version is point-in-time
- New entries create new snapshot, don't modify existing
- Version tracked in corpus.db metadata

</corpus-design>

<model-comparison>
## Model Comparison Infrastructure

### Philosophy

Test all approaches — Anthropic tiers AND OpenCode SDK models. **Accuracy is paramount** — cost optimization follows quality validation.

### Comparison Dimensions (Weighted)

1. **Quality/Accuracy (Primary - 60% weight)**
   - Precision, recall, F1 per task type
   - False positive rate
   - Detection of critical vs non-critical vulnerabilities

2. **Task-Type Suitability (30% weight)**
   - Which model excels at which specific task
   - Rankings per TaskType (verify, analyze, detect, generate_test)

3. **Cost-Accuracy Curves (10% weight)**
   - Cost per 1000 findings
   - Cost per percentage point of accuracy
   - Identify optimal tradeoff points

### Model Pool

**Claude Baseline (Quality Standard):**
- Opus 4.5 — Maximum quality reference
- Sonnet 4.5 — Balanced baseline
- Haiku 4.5 — Speed/cost baseline

**OpenCode SDK (Via OpenRouter):**
| Model | Cost | Context | Primary Use |
|-------|------|---------|-------------|
| Gemini 3 Pro | $2/M in | 1M+ | Complex reasoning |
| Gemini 3 Flash | $0.50/M in | 1M | Agentic workflows |
| DeepSeek V3.2 | $0.25/M in | 164K | Reasoning |
| Grok Code Fast 1 | **FREE** | 256K | Agentic coding |
| GLM-4.7 Z.AI | $6/mo | 128K | Non-critical |
| MiniMax M2.1 | **FREE** | 204K | Verification |
| Big Pickle | **FREE** | 200K | Validation |

### Quality Thresholds

- **Eligibility:** Model must achieve ≥95% of Claude Opus accuracy for a task type to be eligible
- **Security Tasks:** Default to Claude/OpenAI for Attacker, Defender, Verifier roles
- **Free models:** Evaluated on security tasks for research, but not used in production without explicit approval

### Ranking Integration

- **Scope:** Per task type (not per agent)
- **Task Types:** summarize, analyze, verify, detect, generate_test, mutation, etc.
- **Storage:** `.vrs/rankings/rankings.yaml` (Phase 5.3 infrastructure)
- **Auto-update:** Testing results feed directly into EMA scores
- **Output:** Task-type rankings + cost-accuracy curves for comprehensive analysis

### Testing Cadence

- On-demand only (no continuous background testing)
- Run when: new models added, before major releases, after significant code changes

<metrics>
## Metric Definitions

### TestMetrics Schema

```python
@dataclass
class TestMetrics:
    # Core accuracy
    precision: float           # TP / (TP + FP)
    recall: float              # TP / (TP + FN)
    f1_score: float            # Harmonic mean

    # Severity-weighted recall
    recall_critical: float     # Recall for critical findings
    recall_high: float         # Recall for high severity
    recall_medium: float       # Recall for medium severity
    recall_low: float          # Recall for low severity

    # Performance
    execution_time_ms: int
    contracts_tested: int
    patterns_tested: int

    # Per-category breakdown
    by_category: Dict[str, CategoryMetrics]

    # By corpus segment
    by_segment: Dict[str, SegmentMetrics]  # recent-audits, mutations, adversarial

    # Baseline comparison
    baseline_delta: Optional[MetricsDelta]

    # Cost (tracked, not gated)
    tokens_used: int
    cost_usd: float

    # Model comparison (when applicable)
    model_used: str
    vs_opus_accuracy: Optional[float]  # Percentage of Opus accuracy achieved
```

### GA Gate Targets (Accuracy Focus)

| Metric | Target | Rationale |
|--------|--------|-----------|
| **Precision** | ≥85% | Low false positive rate critical for usability |
| **Recall (Critical)** | ≥95% | Must catch critical vulnerabilities |
| **Recall (High)** | ≥85% | Should catch most high-severity |
| **Recall (Medium/Low)** | ≥70% | Acceptable miss rate for lower severity |
| **Cost** | Track only | No target for GA, informational |

### Segment-Specific Targets

| Corpus Segment | Precision | Recall |
|----------------|-----------|--------|
| Recent Audits | ≥85% | ≥90% (real-world baseline) |
| Mutations | ≥80% | ≥85% (volume validation) |
| Adversarial | ≥70% | ≥75% (stress test, expected harder) |

<gap-tracking>
## Gap Tracking Infrastructure

### Philosophy

**Brutally honest documentation** of all known gaps, failure modes, and root causes.

### Gap Categories

1. **Detection Gaps** — Vulnerabilities we miss
2. **False Positive Patterns** — What triggers incorrect findings
3. **Technical Constraints** — Builder/pattern/VQL limitations
4. **Agent Limitations** — Reasoning failures, context issues
5. **Scalability Issues** — Performance bottlenecks
6. **UX Friction** — CLI/workflow problems

### Gap Schema

```yaml
gap_id: GAP-001
category: detection
severity: high  # critical/high/medium/low
title: "Cross-contract reentrancy via callback"
description: |
  Detection fails when reentrancy occurs through
  external callback to different contract.
root_cause: "CallTracker doesn't follow callback paths across contracts"
affected_component: builder.calls
test_case: tests/adversarial/test_cross_contract_reentrancy.py
status: open  # open/investigating/fixed/wontfix
roadmap_item: BACKLOG-023  # If actionable
research_theme: cross-contract-analysis  # If systemic
discovered_date: 2026-01-22
discovered_by: vrs-gap-finder
```

### Integration

- **Actionable items** → `.planning/BACKLOG.md`
- **Systemic gaps** → Research themes for future milestones
- **Skill-managed** — `/vrs-track-gap` handles formatting and organization
- **Gap inventory** — `.vrs/testing/gaps/` with YAML per gap

</gap-tracking>
