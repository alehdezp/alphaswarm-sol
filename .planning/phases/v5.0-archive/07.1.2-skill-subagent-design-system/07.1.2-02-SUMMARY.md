# Phase 7.1.2 Plan 02: Secure Reviewer Prompts

**One-liner:** Created evidence-first security reviewer with creative attack + adversarial refutation modes enforcing graph-first reasoning via JSON output contract.

---

## Overview

Implemented the `vrs-secure-reviewer` agent that blends creative attack thinking with adversarial skepticism while enforcing strict evidence-first and graph-first reasoning. The agent operates in two distinct modes:

1. **Creative Mode** - Attack discovery and vulnerability brainstorming
2. **Adversarial Mode** - Claim refutation and counterargument construction

Both modes require:
- BSKG query execution BEFORE conclusions
- Evidence anchored to code locations or graph node IDs
- Explicit uncertainty declaration via `known_unknowns`
- JSON output matching `schemas/secure_reviewer_output.json`

## Technical Implementation

### Output Contract Schema

**Core Structure:**
```json
{
  "summary": "string (20-500 chars)",
  "risk_level": "CRITICAL|HIGH|MEDIUM|LOW|INFORMATIONAL|NONE",
  "mode": "creative|adversarial",
  "evidence": [],
  "graph_queries": [],
  "gaps": [],
  "recommendations": [],
  "known_unknowns": {}
}
```

**Evidence Anchoring:**
Every evidence item requires EITHER:
- `code_location` (e.g., "Vault.sol:L45")
- `graph_node_id` (e.g., "Vault.withdraw")
- Both (preferred)

This enforces verifiability and prevents hallucination.

**Graph-First Enforcement:**
- Minimum 1 query in `graph_queries` array
- Each query includes `query`, `rationale`, and optional `results_count`
- Queries must be executed BEFORE conclusions
- Violation: Output fails schema validation

**Uncertainty Tracking:**
```json
{
  "known_unknowns": {
    "has_uncertainty": true,
    "uncertainties": [
      {
        "area": "External contract behavior",
        "reason": "Source not in graph",
        "needs_human_review": true
      }
    ]
  }
}
```

### Mode Behavior

**Creative Mode:**
- Generate attack hypotheses
- Each hypothesis includes: description, attack_path (steps), feasibility, supporting_evidence
- Explore unconventional paths
- Consider MEV, front-running, multi-step attacks
- Output: `creative_hypotheses` array

**Adversarial Mode:**
- Challenge existing claims
- Each refutation includes: claim_being_refuted, refutation, strength, supporting_evidence
- Find protective guards and counter-signals
- Point out evidence gaps
- Output: `refutations` array

**Mode Switching:**
- Creative → Adversarial: After hypotheses generated, for verification
- Adversarial → Creative: After refutations fail (vulnerability confirmed)

### Evidence Types

1. **code_property** - Specific code property (visibility, modifiers)
2. **semantic_operation** - Operation detected (TRANSFERS_VALUE_OUT)
3. **behavioral_signature** - Operation sequence (R:bal->X:out->W:bal)
4. **missing_guard** - Absence of protection (has_reentrancy_guard=false)
5. **operation_sequence** - Ordering of operations
6. **graph_structure** - Graph topology feature
7. **counter_signal** - Evidence against vulnerability (adversarial mode)

### Confidence Scoring

- **0.9-1.0:** Direct graph property confirmed
- **0.7-0.89:** Strong inference from multiple signals
- **0.5-0.69:** Moderate confidence, some uncertainty
- **0.3-0.49:** Weak signal, needs verification
- **0.0-0.29:** Speculative, low confidence

**Risk Level Requirements:**
- CRITICAL: confidence > 0.85
- HIGH: confidence > 0.7
- MEDIUM: confidence > 0.5

## Lessons Learned

### What Went Well
1. **Output contract design** - JSON schema enforces all required behaviors
2. **Mode separation** - Clear distinction between creative and adversarial roles
3. **Evidence anchoring** - Prevents hallucination at contract level
4. **Graph-first enforcement** - Minimum query requirement ensures compliance
5. **Explicit uncertainty** - Forces reviewer to acknowledge limitations

### What Could Improve
1. **Mode switching** - Could add automatic mode detection based on context
2. **Query optimization** - Could provide query templates for common patterns
3. **Evidence templates** - Could standardize evidence format for common types

### Recommendations for Future Plans
1. Test reviewer in real audit scenarios (Phase 7.3)
2. Collect metrics on creative vs adversarial mode effectiveness
3. Consider adding query result caching to reduce repeated queries
4. Document reviewer performance on known vulnerability patterns
