## Registry Structure

### Entry Fields

```yaml
id: skill-id              # Unique identifier (kebab-case)
name: "Skill Name"        # Human-readable name
version: 1.0.0            # Semantic version (major.minor.patch)
status: active            # Lifecycle status
category: orchestration   # Category for organization
description: "..."        # One-line description
location:
  shipped: path/to/shipped.md  # Shipped location (null if dev-only)
  dev: path/to/dev.md           # Dev location (null if shipped-only)
replaces: old-skill       # Skill this replaces (optional)
deprecated_by: new-skill  # Skill deprecating this one (optional)
sunset_date: 2026-06-01   # Removal date (optional, ISO 8601)
migration_notes: "..."    # Migration guidance (optional)
```

### Lifecycle Status Values

| Status | Meaning | Action |
|--------|---------|--------|
| `active` | Production-ready, stable | Use freely |
| `experimental` | Under development, API may change | Use with caution |
| `deprecated` | Marked for removal, replacement available | Migrate to replacement |
| `sunset` | Removed or scheduled for imminent removal | Stop using immediately |

## Deprecation Policy

### Workflow Stages

**1. Announcement (Initial Deprecation)**
- Status → `deprecated`
- `deprecated_by` set to replacement skill
- `migration_notes` provided
- Announced in release notes
- **Minimum duration:** 90 days

**2. Warning Period**
- Warnings in documentation
- CLI warnings when skill loaded
- Integration tests fail if used
- **Minimum duration:** 60 days after announcement

**3. Sunset (Removal)**
- Status → `sunset`
- `sunset_date` set (ISO 8601)
- Final warning period
- **Minimum duration:** 30 days after warning

**4. Removal**
- Skill files deleted
- Registry entry remains with `status: sunset` for historical reference

### Timeline

| Stage | Duration | Status | Cumulative |
|-------|----------|--------|-----------|
| Announcement | 90 days | `deprecated` | 90 days |
| Warning | 60 days | `deprecated` | 150 days |
| Sunset | 30 days | `sunset` | 180 days |
| Removal | — | `sunset` (kept in registry) | — |

**Total deprecation period: Minimum 180 days (6 months)**

## Decisions Made

**1. 47 skills registered across 8 categories**

**Rationale:** Complete coverage of shipped and development skills
**Categories chosen:** Organize by workflow phase (orchestration, investigation, validation, etc.)
**Benefit:** Easy discovery and filtering

**2. Semantic versioning for all skills**

**Rationale:** Industry-standard versioning scheme
**Format:** major.minor.patch
**Benefit:** Clear compatibility expectations, breaking change tracking

**3. Four lifecycle statuses**

**Rationale:** Cover full skill lifecycle from development to removal
**Statuses:** active (production), experimental (dev), deprecated (migrating), sunset (removed)
**Benefit:** Clear signals for skill stability and migration needs

**4. 180-day minimum deprecation period**

**Rationale:** Give users ample time to migrate
**Stages:** 90-day announce + 60-day warn + 30-day sunset
**Alternative considered:** 90-day total (rejected - too short for major migrations)
**Benefit:** Prevents breaking changes, supports gradual migration

**5. Location tracking for shipped vs dev-only**

**Rationale:** Distinguish user-facing from internal skills
**Implementation:** shipped and dev fields in location object
**Dual-location:** 3 skills maintain both (active development while shipped)
**Benefit:** Prevents accidental shipping of internal tools

**6. Registry loader with filtering helpers**

**Rationale:** Common queries should be one-liners
**Helpers:** by_status, by_category, shipped, dev_only, deprecated
**Alternative considered:** Single list() with manual filtering (rejected - verbose)
**Benefit:** Clean API, reduced boilerplate

**7. CLI validation entry points**

**Rationale:** Enable CI/CD integration and manual validation
**Commands:** validate, check-duplicates, list-deprecated, stats
**Alternative considered:** Python-only API (rejected - harder for CI/CD)
**Benefit:** Human-readable output, scriptable validation

## Lessons Learned

### What Went Well

1. **Registry structure** - Clean YAML format, human-readable and editable
2. **JSON schema validation** - Catches structural errors early
3. **Filtering helpers** - Common queries are one-liners
4. **CLI tools** - Human-readable output, scriptable validation
5. **Documentation completeness** - Reference guide covers all use cases
6. **Deprecation policy** - Clear timeline and workflow stages

### What Could Improve

1. **Schema validation** - Currently optional (requires jsonschema), could be mandatory
2. **Version bumping** - Manual process, could be automated
3. **Changelog generation** - No automatic changelog from registry diffs
4. **Migration automation** - No scripts to assist with skill migration
5. **Usage tracking** - No analytics on which skills are most used

### Recommendations for Future Plans

1. **Add automatic changelog generation** - Diff registry and generate CHANGELOG entries
2. **Implement version bump automation** - CLI command to bump version with validation
3. **Add migration scripts** - Automated skill migration assistance
4. **Track skill usage** - Analytics on which skills are invoked most
5. **Auto-generate documentation** - Generate skill tables from registry
6. **CI/CD integration** - Block merges if registry validation fails
