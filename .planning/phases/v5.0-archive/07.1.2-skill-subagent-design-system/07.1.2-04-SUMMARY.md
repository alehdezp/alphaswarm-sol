## Validation Checks

The `validate_catalog()` function checks:

1. **Required fields present:** id, name, role, model_tier, purpose
2. **Role enum valid:** Must be in allowed roles (attacker, defender, verifier, etc.)
3. **Model tier valid:** Must be opus, sonnet, haiku, or adaptive
4. **Context mode valid:** Must be fork or inline
5. **Output contract format:** Must be specified
6. **Schema refs exist:** If schema_ref specified, file must exist
7. **Location specified:** At least one of shipped or dev must be set
8. **Location files exist:** Specified paths must exist

## Decisions Made

**1. Catalog as YAML for human readability**
- **Rationale:** Easy to edit, version control friendly, supports comments
- **Alternative considered:** JSON (more strict but less readable)
- **Trade-off:** YAML parsing slightly slower, but not performance-critical

**2. Separate shipped vs dev-only tracking**
- **Rationale:** Clear distinction between user-facing and internal agents
- **Implementation:** `location.shipped: null` for dev-only agents
- **Benefit:** Prevents accidental shipping of internal tools

**3. Evidence requirements in catalog**
- **Rationale:** Single source of truth for agent behavior constraints
- **Integration:** Validates against skill schema v2
- **Enforcement:** Orchestration layer can check before agent invocation

**4. Output contract references (not inline schemas)**
- **Rationale:** Reusable schemas, maintainable, tooling-friendly
- **Fallback:** Embedded required_fields for simple contracts
- **Example:** secure-reviewer uses external schema, others use embedded

**5. Dataclass-based loader for type safety**
- **Rationale:** IDE autocomplete, type checking, clearer API
- **Alternative considered:** Dict-based (simpler but less safe)
- **Benefit:** Catches catalog errors at load time

**6. Validation checks file existence**
- **Rationale:** Catch broken references early
- **Implementation:** Checks both absolute and relative-to-project paths
- **Limitation:** Only validates at load time, not continuously

## Lessons Learned

### What Went Well

1. **YAML catalog format** - Human-readable, easy to maintain, comment-friendly
2. **Dataclass API** - Type-safe, IDE-friendly, self-documenting
3. **Evidence requirements integration** - Builds on skill schema v2 work
4. **Validation checks** - Catch catalog errors early
5. **Documentation completeness** - Reference guide covers all use cases

### What Could Improve

1. **JSON schema validation** - Could add schema for catalog.yaml itself
2. **Output contract validation** - Could validate agent responses against contracts
3. **Location tracking** - Could track version history per agent
4. **Deprecation support** - No formal deprecation workflow yet
5. **A/B testing** - No support for comparing model tiers

### Recommendations for Future Plans

1. **Add JSON schema for catalog.yaml** - Validate structure, not just content
2. **Implement contract validation** - Runtime validation of agent outputs
3. **Track agent versions** - History of changes per agent
4. **Add deprecation workflow** - Mark agents as deprecated with migration path
5. **Support adaptive routing** - Runtime model selection based on task
