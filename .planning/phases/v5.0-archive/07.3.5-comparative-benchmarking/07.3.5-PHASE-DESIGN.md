## Purpose

Ground-truth the system against known external benchmarks to provide an honest, unbiased assessment of actual capabilities.

## The Problem It Solves

Internal testing can fool itself:
- We define the tests
- We define what "pass" means
- We might unconsciously design tests we can pass

External benchmarks provide reality check:
- Someone else defined the vulnerabilities
- Someone else verified the ground truth
- We can't game it

## Benchmark Categories

### Category 1: Tool Comparison (Slither, Mythril, Aderyn)

Run the same contracts through multiple tools and compare.

```
┌──────────────────────────────────────────────────────────────┐
│                    TOOL COMPARISON                           │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  Same Contracts → Slither    → Findings A                   │
│                → Mythril    → Findings B                   │
│                → Aderyn     → Findings C                   │
│                → AlphaSwarm → Findings D                   │
│                                                              │
│  Compare:                                                    │
│    • What A,B,C find that D misses (our gaps)               │
│    • What D finds that A,B,C miss (our value-add)           │
│    • What all agree on (high confidence)                    │
│    • What D uniquely finds (novel detection)                │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### Metrics to Capture

| Metric | Formula | Why It Matters |
|--------|---------|----------------|
| Coverage Gap | `|A∪B∪C| - |D ∩ (A∪B∪C)|` | What we miss |
| Unique Finds | `|D| - |D ∩ (A∪B∪C)|` | Our differentiation |
| Agreement Rate | `|D ∩ (A∪B∪C)| / |A∪B∪C|` | Confidence level |
| FP Reduction | `FP_slither - FP_alphaswarm` | Noise reduction |

### Category 2: Historical Audit Comparison

Take published audit reports and compare to AlphaSwarm findings.

```yaml
audit_sources:
  - auditor: "Trail of Bits"
    reports:
      - project: "Compound v2"
        url: "https://github.com/trailofbits/publications/..."
        findings: 12 (4 critical, 3 high, 5 medium)

  - auditor: "OpenZeppelin"
    reports:
      - project: "Uniswap v3"
        url: "https://blog.openzeppelin.com/..."
        findings: 8 (1 critical, 2 high, 5 medium)

  - auditor: "Consensys Diligence"
    reports:
      - project: "Aave v2"
        url: "https://consensys.net/diligence/audits/..."
        findings: 15 (2 critical, 4 high, 9 medium)
```

### Comparison Protocol

```
1. Get codebase at pre-audit commit (before fixes)
2. Run AlphaSwarm on pre-audit code
3. Compare AlphaSwarm findings to published report
4. Calculate overlap:
   - Did we find their criticals?
   - Did we find things they missed?
   - Did we flag things they confirmed safe?
```

### Expected Outcomes

| Outcome | Interpretation |
|---------|----------------|
| Find all criticals | Excellent - matching expert auditors |
| Miss some criticals | Gap - need pattern/detection improvement |
| Find things they missed | Value-add - novel detection capability |
| Flag their safe items | False positive - need better guards |

## Self-Improvement Loop

```
┌─────────────────────────────────────────────────────────────┐
│                BENCHMARK CYCLE                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. RUN all benchmark comparisons                           │
│     ↓                                                       │
│  2. COMPUTE metrics (precision, recall, gaps)               │
│     ↓                                                       │
│  3. IDENTIFY coverage gaps and FP hotspots                  │
│     │                                                       │
│     ├── Metrics meet targets? ──────→ PHASE COMPLETE       │
│     │   (P >= 70%, R >= 60%)                                │
│     │                                                       │
│     └── Metrics below targets ────→ 4. PRIORITIZE gaps     │
│                                          ↓                  │
│                                     5. CREATE patterns/fixes │
│                                          ↓                  │
│                                     6. GOTO 1 (re-benchmark)│
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### Improvement Protocol

```yaml
for_each_gap:
  1. Identify root cause:
     - Missing pattern?
     - Pattern too narrow?
     - Agent not investigating?
     - Graph missing property?

  2. Create fix:
     - New pattern YAML
     - Graph builder update
     - Agent prompt improvement
     - VulnDocs addition

  3. Verify fix:
     - Run specific benchmark case
     - Confirm detection
     - Check no new FPs

  4. Re-run full benchmark:
     - Ensure no regressions
     - Track metric improvement
```

- access date
- dataset hash
- category coverage

This satisfies external ground truth requirements and ensures reproducibility.

## Normalization and Dedupe

Before comparing tools, normalize outputs (SARIF) and dedupe with `alphaswarm tools dedupe`. This avoids inflated FP counts and inconsistent baselines.

## Placeholder Audit Sources

Any placeholder URLs must be replaced with real sources or explicitly marked TODO in the manifest. Placeholder entries are non-runnable and fail validation.
