# Phase 5.3: OpenCode SDK Refactor - Research

**Researched:** 2026-01-21
**Domain:** Multi-model LLM execution, CLI-based agent orchestration, model benchmarking
**Confidence:** HIGH

## Summary

Phase 5.3 refactors the existing API-based multi-agent execution (Phase 5.2's AnthropicRuntime and OpenAIAgentsRuntime) to use CLI-based execution via OpenCode SDK, Claude Code CLI, and Codex CLI. The primary goal is cost optimization (75-95% reduction) while maintaining or improving quality through intelligent model routing and benchmarking.

Research validates the planned approach: **OpenCode SDK provides access to 400+ models via OpenRouter and the AI SDK**, making it an excellent choice for multi-model execution. Claude Code CLI's `--print` and `--output-format json` flags enable reliable non-interactive subprocess orchestration. The model benchmarking system should leverage existing LLM evaluation frameworks (DeepEval, RAGAS) for scoring methodology while building VKG-specific ground truth datasets.

**Primary recommendation:** Implement OpenCode SDK as the primary runtime using Python subprocess calls to the `opencode` CLI with JSON output format. Use LiteLLM Router patterns for fallback logic. Remove API-based runtimes entirely as they are deprecated per Phase 5.3 decisions.

### Pattern 1: Subprocess CLI Execution

**What:** Execute CLI tools via asyncio.subprocess for non-blocking, non-interactive operation
**When to use:** All runtime execution (OpenCode, Claude Code, Codex)
**Example:**
```python
# Source: Research findings from Claude Code CLI docs
async def execute_cli(cmd: List[str], timeout: int, cwd: Path) -> Dict[str, Any]:
    """Execute CLI command with JSON output parsing."""
    proc = await asyncio.create_subprocess_exec(
        *cmd,
        stdout=asyncio.subprocess.PIPE,
        stderr=asyncio.subprocess.PIPE,
        cwd=str(cwd),
    )

    stdout, stderr = await asyncio.wait_for(
        proc.communicate(),
        timeout=timeout,
    )

    if proc.returncode != 0:
        raise RuntimeError(f"CLI error: {stderr.decode()}")

    return json.loads(stdout.decode())
```

**What:** Use --print and --output-format flags for automation
**When to use:** ClaudeCodeRuntime.execute()
**Example:**
```python
# Source: Claude Code CLI docs (https://code.claude.com/docs/en/cli-reference)
def build_claude_code_command(
    prompt: str,
    output_format: str = "json",
    model: str = "opus",
    allowed_tools: Optional[List[str]] = None,
) -> List[str]:
    """Build Claude Code CLI command for non-interactive execution."""
    cmd = [
        "claude",
        "-p", prompt,                        # Non-interactive mode
        "--output-format", output_format,    # "json", "text", or "stream-json"
    ]

    if model:
        cmd.extend(["--model", model])

    if allowed_tools:
        cmd.extend(["--allowed-tools", ",".join(allowed_tools)])

    return cmd
```

### Anti-Patterns to Avoid

- **Importing Node.js SDK in Python:** Don't try to use @opencode-ai/sdk from Python; use CLI subprocess instead
- **Blocking subprocess calls:** Always use asyncio.subprocess, never subprocess.run in async code
- **Hardcoded model IDs:** Use configuration-driven model selection from rankings
- **Ignoring rate limits:** Implement exponential backoff and respect 429 responses
- **Large prompts without chunking:** Split large inputs for models with smaller context windows

## Don't Hand-Roll

Problems that look simple but have existing solutions:

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Model fallback logic | Custom retry chains | OpenRouter fallback or LiteLLM Router | Production-tested, handles edge cases |
| LLM evaluation metrics | Custom scoring functions | DeepEval/RAGAS patterns | Standardized, validated metrics |
| JSON output parsing | Manual regex extraction | Built-in --output-format json flags | Reliable, handles edge cases |
| Rate limit handling | Simple retry loops | LiteLLM's built-in rate limiting | Tracks TPM/RPM across models |
| Model cost calculation | Hardcoded price tables | OpenRouter's pricing API or LiteLLM | Always current, per-model accurate |

**Key insight:** CLI tools already solve subprocess communication, output format, and error handling. Don't wrap them in unnecessary abstraction layers.

## Common Pitfalls

### Pitfall 1: Subprocess Hanging on Large Outputs

**What goes wrong:** subprocess.communicate() blocks forever when output exceeds buffer
**Why it happens:** Default pipe buffer is small (~64KB), large LLM responses overflow
**How to avoid:** Use asyncio.wait_for with timeout, stream output for large responses
**Warning signs:** Tests pass but production hangs on complex prompts

### Pitfall 2: Model ID Format Mismatch

**What goes wrong:** "gpt-4" works in OpenAI but fails in OpenRouter
**Why it happens:** Different providers use different ID formats (provider/model vs just model)
**How to avoid:** Always use full provider-prefixed IDs: "openai/gpt-4o", "anthropic/claude-3-5-sonnet"
**Warning signs:** Works with some models but not others

### Pitfall 3: Claude Code CLI TTY Detection

**What goes wrong:** Claude CLI expects TTY and shows interactive UI
**Why it happens:** Some flags require explicit --print for non-interactive mode
**How to avoid:** Always include -p/--print flag for subprocess usage
**Warning signs:** Subprocess hangs waiting for user input

### Pitfall 4: Benchmark Data Leakage

**What goes wrong:** LLMs ace benchmarks because they've seen the test cases in training
**Why it happens:** Famous exploits (DAO hack, etc.) are in training data
**How to avoid:** Include synthetic mutations and hand-crafted novel scenarios per CONTEXT.md
**Warning signs:** Perfect scores on historical exploits, poor scores on fresh scenarios

### Pitfall 5: Rankings Stale After Model Updates

**What goes wrong:** Rankings become inaccurate when models are updated
**Why it happens:** Model versions change (gpt-4o-2024-11-20 vs gpt-4o-2025-01-01)
**How to avoid:** Include model version in rankings, periodic recalibration, timestamp tracking
**Warning signs:** Previously top model starts underperforming

### Pitfall 6: Loop Prevention Failure

**What goes wrong:** Agent enters infinite loop consuming tokens
**Why it happens:** Model repeats same response, output comparison not triggered
**How to avoid:** Implement all three mechanisms per CONTEXT.md: max iterations, output comparison, total token ceiling
**Warning signs:** Single task consumes entire token budget

Verified patterns from official sources:

### OpenCode Runtime Execute

```python
# Source: Research synthesis from OpenCode and OpenRouter docs
class OpenCodeRuntime(AgentRuntime):
    """OpenCode SDK runtime via CLI subprocess."""

    async def execute(
        self,
        config: AgentConfig,
        messages: List[Dict[str, Any]],
        task_type: str = "analyze",
    ) -> AgentResponse:
        """Execute via OpenCode CLI with JSON output."""
        model = self._select_model(task_type, config.role)
        prompt = self._format_messages(config.system_prompt, messages)

        cmd = [
            "opencode",
            "-p", prompt,
            "--model", model,
            "-f", "json",
            "-q",
        ]

        start = time.time()
        result = await self._run_subprocess(cmd, config.timeout_seconds)
        latency_ms = int((time.time() - start) * 1000)

        return AgentResponse(
            content=result.get("content", ""),
            tool_calls=result.get("tool_calls", []),
            input_tokens=result.get("usage", {}).get("input_tokens", 0),
            output_tokens=result.get("usage", {}).get("output_tokens", 0),
            model=model,
            latency_ms=latency_ms,
            cost_usd=self._calculate_cost(model, result.get("usage", {})),
        )
```

### Claude Code Runtime Execute

```python
# Source: Claude Code CLI docs (https://code.claude.com/docs/en/cli-reference)
class ClaudeCodeRuntime(AgentRuntime):
    """Claude Code CLI runtime for critical analysis tasks."""

    async def execute(
        self,
        config: AgentConfig,
        messages: List[Dict[str, Any]],
    ) -> AgentResponse:
        """Execute via Claude Code CLI with JSON output."""
        prompt = self._format_messages(config.system_prompt, messages)

        cmd = [
            "claude",
            "-p", prompt,
            "--output-format", "json",
            "--model", "opus",  # Use opus for critical tasks
        ]

        if config.tools:
            tool_names = [t.get("name", "") for t in config.tools]
            cmd.extend(["--allowed-tools", ",".join(tool_names)])

        result = await self._run_subprocess(cmd, config.timeout_seconds)

        return self._parse_claude_response(result)
```

### Benchmark Executor Pattern

```python
# Source: Research synthesis from LLM evaluation frameworks
@dataclass
class BenchmarkResult:
    """Result of a single benchmark execution."""
    benchmark_id: str
    model_id: str
    category: str
    score: float  # 0.0-1.0
    latency_ms: float
    input_tokens: int
    output_tokens: int
    success: bool
    error_message: Optional[str] = None

async def run_benchmark(
    model_id: str,
    benchmark: BaseBenchmark,
    runtime: OpenCodeRuntime,
) -> BenchmarkResult:
    """Execute single benchmark and evaluate against ground truth."""
    prompt = benchmark.to_prompt()
    start = time.time()

    try:
        response = await runtime.execute_raw(prompt, model_id)
        latency_ms = (time.time() - start) * 1000

        # Evaluate against ground truth
        score = benchmark.evaluate(response)

        return BenchmarkResult(
            benchmark_id=benchmark.task_id,
            model_id=model_id,
            category=benchmark.category.value,
            score=score,
            latency_ms=latency_ms,
            input_tokens=response.input_tokens,
            output_tokens=response.output_tokens,
            success=True,
        )
    except Exception as e:
        return BenchmarkResult(
            benchmark_id=benchmark.task_id,
            model_id=model_id,
            category=benchmark.category.value,
            score=0.0,
            latency_ms=(time.time() - start) * 1000,
            input_tokens=0,
            output_tokens=0,
            success=False,
            error_message=str(e),
        )
```

### Model Ranking EMA Update

```python
# Source: Research synthesis from ranking system patterns
def update_ranking_ema(
    current: ModelRanking,
    feedback: TaskFeedback,
    decay_factor: float = 0.95,
) -> ModelRanking:
    """Update model ranking using exponential moving average."""
    # Exponential moving average preserves recent performance
    # while smoothing over historical data

    new_success_rate = (
        current.success_rate * decay_factor +
        (1.0 if feedback.success else 0.0) * (1 - decay_factor)
    )

    new_latency = (
        current.average_latency_ms * decay_factor +
        feedback.latency_ms * (1 - decay_factor)
    )

    new_quality = (
        current.quality_score * decay_factor +
        feedback.quality_score * (1 - decay_factor)
    )

    return ModelRanking(
        model_id=current.model_id,
        task_type=current.task_type,
        success_rate=new_success_rate,
        average_latency_ms=new_latency,
        quality_score=new_quality,
        sample_count=current.sample_count + 1,
        last_updated=feedback.timestamp,
    )
```

## Open Questions

Things that couldn't be fully resolved:

1. **OpenCode SDK Session Management**
   - What we know: SDK supports `client.session.create()` and `client.session.prompt()`
   - What's unclear: How session state persists across subprocess calls
   - Recommendation: Use CLI mode for simplicity, only use SDK for complex multi-turn if needed

2. **Benchmark Ground Truth Quantity**
   - What we know: CONTEXT.md specifies 25 test cases per category (200 total)
   - What's unclear: Whether 25 is statistically sufficient for all categories
   - Recommendation: Start with 25, add more based on variance analysis

3. **Model Version Tracking**
   - What we know: Models update frequently (gpt-4o-2024-11-20 vs gpt-4o-2025-01-01)
   - What's unclear: How OpenRouter/OpenCode handle version pinning
   - Recommendation: Use explicit model IDs when available, track in rankings

4. **Free Tier Rate Limits**
   - What we know: Free models exist but may have rate limits
   - What's unclear: Specific TPM/RPM limits for each free model
   - Recommendation: Implement rate limit detection with automatic fallback
