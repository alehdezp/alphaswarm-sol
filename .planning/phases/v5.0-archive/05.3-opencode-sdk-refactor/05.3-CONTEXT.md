# Phase 5.3: OpenCode SDK Refactor - Context

**Gathered:** 2026-01-21 (updated)
**Status:** Ready for planning

<domain>
## Phase Boundary

Refactor the multi-agent execution layer to use OpenCode SDK for multi-model access, Claude Code CLI for orchestration, and Codex CLI for alternative perspectives. Implement a **lightweight ranking/feedback system** for runtime model selection optimization.

**Key Goal:** Enable cost-optimized multi-model execution with continuous feedback-driven improvement. Rankings update via EMA after each execution.

**Out of scope:** New vulnerability detection patterns, VulnDocs changes, builder modifications, new agent roles.

**Deferred to Phase 7:** Comprehensive benchmarking system with 200 ground truth test cases, 8 benchmark categories, BenchmarkExecutor infrastructure, and formal model evaluation. Phase 5.3 provides the ranking/feedback foundation that Phase 7 will leverage for full benchmarking.

</domain>

<decisions>
## Implementation Decisions

### Ranking/Feedback System (Phase 5.3 Scope)

- **Feedback collection:** Record success/failure, latency, tokens, quality after each execution
- **EMA-based ranking updates:** Recent feedback weighted more heavily
- **Simple model selection:** Best model per task type based on rankings
- **YAML persistence:** Rankings stored in `.vrs/rankings/rankings.yaml`
- **CLI commands:** View rankings, reset rankings, basic benchmark run

### Benchmark System (DEFERRED to Phase 7)

> **Note:** The comprehensive benchmarking system described below is DEFERRED to Phase 7 (Final Testing). Phase 5.3 implements only the ranking/feedback infrastructure that Phase 7 will build upon.

- **Full 8 benchmark categories:** vuln detection, code reasoning, pattern matching, exploit synthesis, guard detection, summarization, verification, code generation
- **25 ground truth test cases per category** (200 total) — statistically significant
- **Manual benchmark execution only** — developer triggers when needed (simpler, cheaper)
- **Parallel model execution** — models tested simultaneously with simple conflict prevention
- **Smart retry on failure** — refine instructions without giving solution, adapt to model differences
- **Scores only output** — no detailed traces, just numerical scores and latency
- **Both simplified and historical scenarios** — simplified for quick tests, historical snapshots for accuracy validation
- **Token tracking always** — input/output tokens recorded for every benchmark run
- **Hybrid ground truth sourcing** — existing datasets + curated high-impact exploits + synthetic mutations + fresh novel scenarios (to prevent LLM training data leakage)
- **In-repo storage** — ground truth in `.vrs/benchmarks/` or `tests/benchmarks/`

### Novel Test Scenario Generation (Phase 7)

**Critical:** LLMs may have seen famous exploits in training data. Benchmarks must include:
- **Synthetic mutations** — known patterns with changed variable names, structure, context
- **Fresh hand-crafted scenarios** — novel vulnerable contracts never published
- **Hybrid approach** — mutations for volume, hand-crafted for key categories

### Runtime Priorities

- **OpenCode SDK is primary** — implement first, multi-model access via OpenRouter
- **Claude Code CLI as orchestrator** — spawns subagents and skills, not a direct runtime
- **Codex CLI is priority** — implement alongside OpenCode for model diversity
- **Remove API runtimes entirely** — no anthropic.py, openai_agents.py fallback (clean break)
- **Hybrid orchestration** — direct skills for simple queries, Task tool spawns for complex tasks
- **Research latest multi-agent trends** — use Exa MCP to find current best practices
- **CLI primary, SDK for complex** — subprocess calls for simple, SDK for session management
- **Configuration precedence:** CLI flags > env vars > config file
- **Soft timeout limits** — adjust as we learn, focus on loop prevention not strict limits

### Loop Prevention

- **Max iterations** — hard limit on LLM calls per task (e.g., 10 max)
- **Output comparison** — detect repeated outputs 3+ times
- **Total token ceiling** — prevent runaway token usage
- All three mechanisms combined to prevent infinite loops

### Model Selection Strategy

- **All 7 models benchmarked:** Gemini Pro/Flash, DeepSeek V3.2, Grok Code Fast 1, GLM-4.7, MiniMax M2.1, Big Pickle
- **Default model:** Gemini 3 Flash (when no ranking data exists)
- **Periodic recalibration** — re-run benchmarks monthly or when new models added
- **Model pool per task type** — top 3 models, random selection for diversity
- **User override always allowed** — CLI flag `--model` overrides auto-selection
- **Context window filtering** — only consider models with sufficient context for task
- **New models allowed with warning** — can use unbenchmarked models, but warns about unknown quality

### Ranking Persistence

- **YAML format** — human-readable, git-friendly
- **Project-level storage:** `.vrs/rankings/`
- **Version-controlled** — rankings committed to repo for reproducibility
- **Latest only** — single file with current rankings, overwritten each time
- **Structured by task type** — top-level keys are task types, each contains ranked models
- **Full CRUD CLI commands** — view, edit, reset, export rankings
- **Timestamps informational** — show benchmark date, no staleness warnings

### SDK Selection Strategy

**IMPORTANT:** Phase 5.2's API-based approach replaced with multi-model CLI-based execution.

- **Primary SDK:** OpenCode SDK (`@opencode-ai/sdk`) for multi-model access via OpenRouter
- **Orchestration Runtime:** Claude Code CLI (subscription-based, spawns subagents/skills)
- **Review Runtime:** Codex CLI (ChatGPT Plus subscription, alternative perspective)
- **Legacy Runtimes:** API-based (anthropic.py, openai_agents.py) **REMOVED** (not kept as fallback)

### Multi-Model Architecture

#### Primary Orchestration (Subscription-Based)
- **Claude Code CLI:** Uses `claude` command with --print/--output-format json
  - Session management via --resume
  - Subagent spawning via Task tool
  - Full agentic capabilities (Read, Write, Bash, etc.)
  - Cost: $20/month (Pro) or $100/month (Max)
  - **Use for:** Orchestration, spawning specialized agents

- **Codex CLI:** Uses `codex exec` for scripted tasks
  - Best for: review, discussion, double-checks (different perspective)
  - Cost: $20/month (ChatGPT Plus)
  - **Use for:** Reviews, alternative opinions, debate participation

### OpenCode SDK Integration

- **Installation:** `npm install @opencode-ai/sdk`
- **CLI Usage:** `opencode -p "prompt" --model "model-id" -f json -q`
- **Session Management:** Via SDK's `client.session.create()` and `client.session.prompt()`
- **Model Selection:** Dynamic based on task requirements and rankings

**Configuration Pattern:**
```json
{
  "$schema": "https://opencode.ai/config.json",
  "mode": {
    "reasoning": { "model": "google/gemini-3-pro-preview" },
    "agentic": { "model": "google/gemini-3-flash-preview" },
    "reasoning_cheap": { "model": "deepseek/deepseek-v3.2" },
    "code": { "model": "x-ai/grok-code-fast-1" },
    "code_cheap": { "model": "zhipu/glm-4.7" },
    "verify": { "model": "minimax/minimax-m2.1" },
    "validate": { "model": "bigpickle/bigpickle" }
  }
}
```
