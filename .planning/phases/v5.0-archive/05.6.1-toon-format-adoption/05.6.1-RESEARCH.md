# Phase 5.6.1: TOON Format Adoption - Research

**Researched:** 2026-01-23
**Domain:** Data serialization for LLM token optimization
**Confidence:** HIGH

## Summary

TOON (Token-Oriented Object Notation) is a compact, human-readable serialization format specifically designed to reduce token consumption when passing structured data to LLMs. Research confirms 30-60% token reduction for suitable data structures compared to JSON.

The AlphaSwarm.sol knowledge graph is an excellent candidate for TOON conversion because:
1. Graphs contain uniform arrays of nodes and edges with consistent schemas
2. Node properties (50+ security properties) are primarily primitive values (booleans, numbers, strings)
3. Evidence arrays are highly uniform (same 4 fields per entry)
4. Current JSON graphs are large (1MB+, 28k+ lines for test graphs)

**Primary recommendation:** Implement TOON output as default format for knowledge graph JSON, using the `toons` PyPI package (Rust-based, spec v2.0 compliant) with `--format json` fallback flag for legacy compatibility.

### Pattern 1: Layered Serialization
**What:** Keep JSON as internal representation, convert to TOON only at output boundary
**When to use:** All graph output operations
**Example:**
```python
# Source: toons PyPI documentation
import toons
import json

class GraphStore:
    def save(self, graph: KnowledgeGraph, *, format: str = "toon", overwrite: bool = False) -> Path:
        # Internal: always work with dict
        payload = {
            "format": FORMAT_VERSION,
            "saved_at": datetime.now(timezone.utc).isoformat(),
            "graph": graph.to_dict(),
        }

        if format == "toon":
            # Convert to TOON at output boundary
            content = toons.dumps(payload)
            ext = ".toon"
        else:
            content = json.dumps(payload, indent=2)
            ext = ".json"

        target = self.root / f"graph{ext}"
        target.write_text(content, encoding="utf-8")
        return target
```

### Pattern 2: LLM-Optimized Field Ordering
**What:** Order object keys by semantic importance for better LLM parsing
**When to use:** Graph node/edge serialization
**Example:**
```python
# Order fields: identity first, then critical properties, then details
def to_toon_dict(self) -> dict[str, Any]:
    return {
        # Identity (most important for LLM understanding)
        "id": self.id,
        "type": self.type,
        "label": self.label,
        # Security-critical properties next
        "visibility": self.properties.get("visibility"),
        "has_access_control": self.properties.get("has_access_control"),
        "writes_state": self.properties.get("writes_state"),
        # Then remaining properties
        "properties": self.properties,
        # Evidence last (often verbose)
        "evidence": [e.to_dict() for e in self.evidence],
    }
```

### Pattern 3: Tabular Evidence Arrays
**What:** Evidence arrays are perfect for TOON tabular format
**When to use:** Evidence serialization
**Example:**
```python
# Evidence arrays in graph are highly uniform:
# Each has: file, line_start, line_end, detail
# TOON tabular format is ideal:
# evidence[13]{file,line_start,line_end,detail}:
#   ListManagementTest.sol,20,20,null
#   ListManagementTest.sol,20,20,null
# vs JSON:
# "evidence": [{"file": "ListManagementTest.sol", "line_start": 20, ...}, ...]
```

### Pattern 4: Cache Invalidation via mtime
**What:** Use file modification time for cache staleness detection
**When to use:** TOON cache alongside original
**Example:**
```python
def get_toon_path(json_path: Path) -> Path:
    return json_path.with_suffix(".toon")

def is_toon_stale(json_path: Path) -> bool:
    toon_path = get_toon_path(json_path)
    if not toon_path.exists():
        return True
    return json_path.stat().st_mtime > toon_path.stat().st_mtime
```

### Anti-Patterns to Avoid
- **Converting non-LLM JSON:** Don't convert package.json, config files, or internal state
- **Storing both formats:** Per CONTEXT.md, replace JSON graphs with TOON only (not side-by-side)
- **Deep nesting optimization:** TOON excels at flat/tabular; don't try to optimize deeply nested metadata
- **Custom escaping:** Use library; TOON has specific escape rules (only 5 escapes: `\\`, `\"`, `\n`, `\r`, `\t`)

## Don't Hand-Roll

Problems that look simple but have existing solutions:

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| TOON encoding | Custom string builder | `toons.dumps()` | Spec has complex quoting, delimiter, and escaping rules |
| Tabular detection | Manual array inspection | Library handles | Library auto-detects when arrays are tabular-eligible |
| Token counting | Character-based estimate | `tiktoken` | Token != character; need actual tokenizer |
| Format detection | Extension parsing | Check file header | TOON files start with structure, JSON with `{` |

**Key insight:** TOON spec v3.0 has 16+ sections of rules for encoding. The library handles all edge cases (quoted strings, reserved literals, numeric normalization, delimiter scoping).

## Common Pitfalls

### Pitfall 1: Deeply Nested Properties
**What goes wrong:** TOON loses efficiency advantage with deeply nested structures
**Why it happens:** TOON is optimized for flat/tabular data; nesting requires indentation overhead
**How to avoid:** Keep graph properties flat (which they already are - 50+ boolean properties)
**Warning signs:** Properties with nested objects/arrays; token count not significantly reduced

### Pitfall 2: Non-Uniform Arrays
**What goes wrong:** Mixed-type arrays fall back to verbose list format
**Why it happens:** TOON tabular format requires uniform schemas
**How to avoid:** Nodes array has varying property sets by type (Contract vs Function vs StateVariable)
**Warning signs:** Large arrays but no tabular encoding applied

### Pitfall 3: Quoted String Overhead
**What goes wrong:** Strings requiring quotes lose some savings
**Why it happens:** TOON quotes strings with: whitespace, control chars, colons, brackets, delimiters
**How to avoid:** Most graph data is identifiers, booleans, numbers - minimal quoting needed
**Warning signs:** Many quoted strings in output

### Pitfall 4: Evidence Duplication
**What goes wrong:** Same evidence repeated many times bloats output
**Why it happens:** Current graph has redundant evidence entries (same file/line repeated)
**How to avoid:** Consider deduplicating evidence before conversion
**Warning signs:** Evidence arrays with 10+ identical entries (visible in test graph)

### Pitfall 5: API Changes Before 1.0
**What goes wrong:** Code breaks with library updates
**Why it happens:** `toon_format` is beta v0.9.x; `toons` is more stable but still pre-1.0
**How to avoid:** Pin version in pyproject.toml; use simple API (`dumps`/`loads`)
**Warning signs:** Using advanced options or internal APIs

## Code Examples

Verified patterns from official sources:

### Basic Encoding/Decoding
```python
# Source: PyPI toons 0.4.0 documentation
import toons

# Encode Python object to TOON string
data = {"name": "Alice", "age": 30, "tags": ["python", "rust", "toon"]}
toon_str = toons.dumps(data)
# Output:
# name: Alice
# age: 30
# tags[3]: python,rust,toon

# Decode TOON string to Python object
data = toons.loads("name: Alice\nage: 30\ntags[3]: python,rust,toon")
# Returns: {'name': 'Alice', 'age': 30, 'tags': ['python', 'rust', 'toon']}

# File operations (same as json module)
toons.dump(data, file_obj)
data = toons.load(file_obj)
```

### Tabular Array Encoding
```python
# Source: TOON spec v3.0, toons documentation
# Uniform array of objects -> tabular format
nodes = [
    {"id": "node:1", "type": "Function", "label": "transfer"},
    {"id": "node:2", "type": "Function", "label": "approve"},

toon_str = toons.dumps({"nodes": nodes})
# Output:
```

### Graph Store Integration
```python
# Proposed implementation for store.py
from pathlib import Path
import json
import toons
from typing import Literal

FORMAT_VERSION = "alphaswarm-kg-v1"

class GraphStore:
    def save(
        self,
        graph: KnowledgeGraph,
        *,
        format: Literal["toon", "json"] = "toon",
        overwrite: bool = False
    ) -> Path:
        self._ensure_root()
        payload = {
            "format": FORMAT_VERSION,
            "saved_at": datetime.now(timezone.utc).isoformat(),
            "graph": graph.to_dict(),
        }

        ext = ".toon" if format == "toon" else ".json"
        target = self.root / f"graph{ext}"

        if target.exists() and not overwrite:
            timestamp = datetime.now(timezone.utc).strftime("%Y%m%dT%H%M%SZ")
            target = self.root / f"graph-{timestamp}{ext}"

        if format == "toon":
            target.write_text(toons.dumps(payload), encoding="utf-8")
        else:
            target.write_text(json.dumps(payload, indent=2), encoding="utf-8")

        return target

    def load(self, path: Path | None = None) -> KnowledgeGraph:
        if path is None:
            # Try TOON first, fall back to JSON
            toon_path = self.root / "graph.toon"
            json_path = self.root / "graph.json"
            path = toon_path if toon_path.exists() else json_path

        content = path.read_text(encoding="utf-8")
        if path.suffix == ".toon":
            data = toons.loads(content)
        else:
            data = json.loads(content)

        return KnowledgeGraph.from_dict(data.get("graph", {}))
```

## Token Savings Analysis

Based on graph structure analysis:

### Graph Structure Characteristics
- **Nodes array:** Variable schemas per node type (Contract, Function, StateVariable)
  - Not ideal for tabular (different property sets)
  - Still benefits from key/value simplification
- **Edges array:** Uniform schema (id, type, source, target, properties, evidence)
  - Good candidate for tabular
- **Evidence arrays:** Highly uniform (file, line_start, line_end, detail)
  - Excellent candidate for tabular
- **Properties:** Flat structure with 50+ boolean/string/number fields
  - Good for TOON simple key/value

### Expected Savings
| Component | JSON Tokens | TOON Estimate | Savings |
|-----------|-------------|---------------|---------|
| Node array (variable schema) | Baseline | ~20-30% | Modest |
| Edge array (uniform) | Baseline | ~40-50% | Good |
| Evidence arrays (uniform) | Baseline | ~50-60% | Excellent |
| Metadata (nested) | Baseline | ~10-20% | Minimal |
| **Overall estimate** | Baseline | **~35-45%** | Significant |

### Verification Approach
```python
import tiktoken
import toons
import json

def compare_token_counts(graph_data: dict) -> dict:
    enc = tiktoken.get_encoding("cl100k_base")  # GPT-4/Claude tokenizer

    json_str = json.dumps(graph_data, separators=(",", ":"))
    toon_str = toons.dumps(graph_data)

    json_tokens = len(enc.encode(json_str))
    toon_tokens = len(enc.encode(toon_str))

    return {
        "json_chars": len(json_str),
        "toon_chars": len(toon_str),
        "json_tokens": json_tokens,
        "toon_tokens": toon_tokens,
        "token_reduction": (json_tokens - toon_tokens) / json_tokens * 100,
    }
```

## Open Questions

Things that couldn't be fully resolved:

1. **Node Array Tabular Eligibility**
   - What we know: Nodes have different property sets by type (Contract vs Function)
   - What's unclear: Will TOON library detect partial uniformity or treat as mixed array?
   - Recommendation: Test with real graph data; may need to group by node type

2. **Evidence Deduplication**
   - What we know: Test graphs have many duplicate evidence entries
   - What's unclear: Is this intentional (different reasons for same location)?
   - Recommendation: Consider deduplication as separate optimization, independent of TOON

3. **Test Graph Regeneration**
   - What we know: Per CONTEXT.md, "graphs regenerate during testing"
   - What's unclear: Will test suite handle .toon files correctly?
   - Recommendation: Update test graph cache to use TOON format
