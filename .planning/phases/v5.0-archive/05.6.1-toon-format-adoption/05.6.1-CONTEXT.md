# Phase 5.6.1: TOON Format Adoption - Context

**Gathered:** 2026-01-23
**Status:** Ready for planning

<domain>
## Phase Boundary

Convert knowledge graph JSON output to TOON format for optimized LLM consumption. This is a focused optimization targeting the graph data that agents consume - not a wholesale format migration.

**What this phase IS:**
- Graph JSON → TOON conversion for LLM context efficiency
- CLI flag to choose output format (TOON default)
- Caching of TOON conversions

**What this phase is NOT:**
- Converting all YAML files (YAML is already good for LLM)
- Converting config JSON files (don't break anything)
- Rewriting patterns (they need multiline YAML)

</domain>

<decisions>
## Implementation Decisions

### Format Selection Rules
- **YAML files:** Keep as-is - YAML is already good for LLM consumption
- **Pattern files:** Stay YAML - multiline content quality must be preserved
- **Config JSON files:** Stay JSON - don't break dev/shipped configs
- **Graph output JSON:** Convert to TOON - this is the primary target
- **Non-LLM JSON:** Keep as JSON (package.json, tsconfig.json, etc.)

### Migration Strategy
- **Target:** Only knowledge graph JSON output → TOON
- **Approach:** Focused optimization, not wholesale conversion
- **Commit strategy:** Per file type, git-based rollback
- **YAML anchors:** Expand inline if encountered (TOON doesn't support refs)
- **File references:** Claude determines best approach based on file analysis

### Runtime Conversion
- **Both options available:** Output TOON at build time OR convert at agent injection
- **Default format:** TOON (--format toon default, --format json available)
- **Caching:** Yes, cache .toon alongside original, invalidate on source change
- **Staleness detection:** Claude's discretion (mtime or content hash)

### JSON Elimination (Graph Only)
- **Existing .json graphs:** Replace with TOON only (not side-by-side)
- **File extension:** `.toon` (standard)
- **Reader updates:** Not needed - graphs regenerate during testing
- **Backwards compatibility:** --format json flag for legacy needs

### Quality Preservation
- **Verification:** Claude's discretion (round-trip test, schema validation, or spot checks)
- **Token measurement:** Claude's discretion on whether to measure savings
- **LLM readability optimization:** Yes, restructure for better LLM parsing
  - Flatten nested structures where sensible
  - Order keys by importance (id, name, severity first)
  - Standardize property names across node types
  - Claude applies LLM-readability best practices

### Claude's Discretion
- Detection heuristic for files needing YAML vs TOON
- Dependency order for file migrations
- Cache staleness detection approach (mtime vs hash)
- Verification level (round-trip, schema, spot checks)
- Token count measurement value
- Specific LLM readability optimizations

</decisions>

<specifics>
## Specific Ideas

- "Do not format yaml in pattern or content that have multiline strings or instructions as we will lose quality"
- "Replace with toon if not needed and replace all the json or convert it to toon before llm consumption"
- "Make sure that everything consumed by the LLM is yaml if contains multiline and preferably toon if does not contain multiline or comes from json"
- "Only production LLM consumption information that is JSON should be parsed to TOON - mostly this will be the graph"
- The `toons` package (Rust-based, 30-60% faster) is the target library

</specifics>
