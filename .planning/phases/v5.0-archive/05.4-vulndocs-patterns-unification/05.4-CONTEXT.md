# Phase 5.4: VulnDocs-Patterns Unification Framework - Context

**Gathered:** 2026-01-22
**Status:** Ready for planning

<domain>
## Phase Boundary

Build a **framework for unified vulnerability knowledge management** that merges vulndocs and patterns into a single source of truth. This phase creates the structure, templates, validation, and agent skills - NOT the actual migration of existing content.

Key deliverables:
- Unified folder structure at `vulndocs/` root
- Templates and index.yaml schema for vulnerabilities
- Instruction files for maintaining each section
- Agent skills for discovery, research, refinement, and testing
- Validation logic (progressive + minimal quality enforcement)

**Not in scope:** Migrating existing vulndocs/patterns content (separate task after framework is built).

</domain>

<decisions>
## Implementation Decisions

### Folder Structure
- Root location: `vulndocs/` at project root (top-level visibility)
- Patterns embedded inside vulnerability folders (co-located with their vulndoc)
- Nesting depth: Prefer 2 levels (category/subcategory/), allow 3 levels (category/subcategory/variant/) when truly needed
- Similar vulnerabilities stay together even if not mergeable - aids creative discovery of related attack vectors
- File structure per vulnerability folder:
  - `index.yaml` - Machine-readable metadata (required)
  - `overview.md` - Human-readable description
  - `detection.md` - How to detect this vulnerability
  - `verification.md` - How to verify findings
  - `exploits.md` - Real-world exploits (tight, condensed, native code only for exact-match library vulns)
  - `patterns/*.yaml` - All applicable patterns for this vulnerability

### Migration Strategy
- **Framework-first approach**: Build empty structure, templates, validation, and agents first
- Actual migration of existing content is a separate task
- Fresh start for git history - old structure archived in `.planning/archive/`
- No registry file - folder structure IS the source of truth

### Validation
- **Progressive + strict hybrid**: Minimal quality enforced, more files unlock more features
- `index.yaml` required with minimum context fields
- Validation suggests Exa research when content gaps detected
- CI warns on incomplete entries but doesn't block
- Quality gates for patterns: semantic reasoning must make sense

### Agent/Skill Framework

**Model Tier Assignments:**
| Task Type | Model | Rationale |
|-----------|-------|-----------|
| URL filtering/extraction | Haiku 4.5 | Mechanical, low reasoning |
| Generic vulnerability research | Sonnet 4.5 | Good reasoning, cost-effective |
| Novel attack vectors | Opus 4.5 | Complex reasoning required |
| Verification/quality gates | Opus 4.5 | High standards enforcement |
| Exa research coordination | Sonnet 4.5 | Filters info before Opus review |

**Skills (using `vrs:` prefix - Phase 6 rebrand):**

| Skill | Purpose | Model |
|-------|---------|-------|
| `/vrs-discover` | Automated: Exa search for new vulns, compare to coverage, aggregate sources | Sonnet 4.5 + Haiku 4.5 |
| `/vrs-research` | Guided: User provides topic/prompt, skill researches and integrates | Sonnet 4.5 |
| `/vrs-refine` | Improve: Mark draft, investigate failures, generate tests, improve accuracy | Opus 4.5 |
| `/vrs-add-vulnerability` | Add new vulnerability to framework with proper structure | Sonnet 4.5 |
| `/vrs-merge-findings` | Deduplicate and merge similar findings into dense context | Sonnet 4.5 |
| `/vrs-generate-tests` | Create Phase 7-ready test cases from vulndoc reasoning templates | Sonnet 4.5 |
| `/vrs-test-pattern` | Validate patterns against real-world projects, measure FP/FN | Sonnet 4.5 |

**Knowledge Access:**
- Structured retrieval: Agent reads `index.yaml` first, then targeted file reads
- Agents MUST use graph queries, NOT manual code reading
- `.meta/instructions/` folder contains per-section maintenance guidance

### Linkage System
- Pattern-to-vulndoc: **Both implicit and explicit**
  - Implicit: Pattern lives in `vulndocs/category/subcategory/patterns/`
  - Explicit: Pattern YAML has `vulndoc: category/subcategory` field for validation
- Cross-references: No explicit cross-refs - category grouping is sufficient
- Test coverage tracked in `index.yaml` with `test_coverage` field referencing test files

### Phase 7 Test Generation Support
- `index.yaml` includes:
  - `semantic_triggers` - What to look for in graph
  - `vql_queries` - Example VQL queries for detection
  - `graph_patterns` - Graph structure patterns
  - `reasoning_template` - Pseudocode/semantic detection logic for LLM reasoning

**Test Quality Focus (All Three):**
1. **Semantic complexity**: Tests require graph-based reasoning, not name matching
2. **Real-world patterns**: Derived from actual exploits and audits
3. **Adversarial**: Obfuscation, edge cases, false-positive traps

### Graph-First Enforcement
- LLMs MUST use BSKG graph queries for analysis, not manual code reading
- This is critical for consistent semantic reasoning
- Consider separate phase to enforce/fix this if needed

</decisions>

<specifics>
## Specific Ideas

- Vulndocs structure should enhance complex reasoning for detection and verification
- Dense context helps LLM explore all possibilities for different patterns
- Avoid unnecessary duplicates but merge similar findings
- Framework enables future knowledge aggregation, not immediate migration
- Test generation is MORE important than vulndocs themselves
- Tests should challenge LLM reasoning without excessive false positives
- Model assignments optimize cost while maintaining quality where it matters

**Example folder structure:**
```
vulndocs/
├── .meta/templates/           # Skeleton templates for new entries
├── .meta/instructions/        # Per-section maintenance guidance
│   ├── index.md
│   ├── detection.md
│   ├── verification.md
│   └── patterns.md
├── oracle/               # Category
│   ├── overview.md       # Category overview
│   ├── price-manipulation/  # Subcategory (preferred: 2 levels)
│   │   ├── index.yaml
│   │   ├── overview.md
│   │   ├── detection.md
│   │   ├── verification.md
│   │   ├── exploits.md
│   │   └── patterns/
│   │       ├── oracle-001-twap.yaml
│   │       └── oracle-002-spot.yaml
│   └── stale-price/
│       └── ...
├── reentrancy/
│   ├── classic/
│   └── cross-function/
└── access-control/
    └── ...
```

**Example index.yaml schema:**
```yaml
id: oracle-price-manipulation
category: oracle
subcategory: price-manipulation
severity: critical
vulndoc: oracle/price-manipulation  # Explicit for validation

# Phase 7 test generation support
semantic_triggers:
  - READS_EXTERNAL_VALUE
  - READS_ORACLE
  - USES_IN_CALCULATION
vql_queries:
  - "FIND functions WHERE reads_oracle AND uses_external_value_in_calculation"
graph_patterns:
  - external_price_read -> calculation -> state_write
reasoning_template: |
  1. Identify oracle reads (getPrice, latestAnswer, etc.)
  2. Trace value flow to calculations
  3. Check for manipulation windows (single-block, no TWAP)
  4. Verify bounds checking or staleness checks

# Test tracking
test_coverage:
  - tests/test_oracle_lens.py:test_price_manipulation_*
  - tests/adversarial/test_oracle_obfuscation.py

# Pattern references (also embedded in patterns/ folder)
patterns:
  - oracle-001-twap
  - oracle-002-spot
```
