## Decisions Made

### VRS Prefix Selection

Chose `vrs:` (VKG Vulnerability Research System) prefix to:
- Distinguish vulnerability research skills from core BSKG operations (`vkg:`)
- Enable clear namespace separation for future skill categorization
- Align with Phase 6 rebrand decisions

### Skill Invocation Model Documentation

**Critical decision:** Explicitly documented that skills are documentation files, not executable programs.

- Skills ARE prompts that guide agent behavior
- Agents use standard tools (Bash, Read, Write) to execute workflows
- CLI commands invoked via Bash tool (e.g., `uv run alphaswarm vulndocs scaffold`)
- No separate "skill execution engine" exists

Rationale: Prevents confusion about how skills work and ensures proper tool usage.

### Phase 7 Field Population

`/vrs-add-vulnerability` skill populates these index.yaml fields for test generation:

- `semantic_triggers`: BSKG operations indicating vulnerability
- `vql_queries`: Example VQL queries for detection
- `graph_patterns`: Vulnerable vs. safe patterns
- `reasoning_template`: Pseudocode for detection logic

Rationale: These fields enable Phase 7 to generate semantically complex, graph-based tests.

### Exa MCP Integration

Used `mcp__exa-search__web_search_exa` and `mcp__exa-search__get_code_context_exa` per `.planning/config.json` research tools preference.

Rationale: Exa provides better code context and more relevant results than generic web search.

### Haiku Filtering for Cost Efficiency

`/vrs-discover` spawns parallel Haiku tasks for relevance filtering before Sonnet processes results.

Cost analysis:
- 100 search results × Haiku filter = ~$0.01
- Without filtering: 100 × Sonnet = ~$0.50
- Savings: 50x cost reduction

Rationale: Haiku is fast and cheap for mechanical tasks (yes/no filtering), Sonnet only processes relevant results.

### Graph-First Enforcement

Both skills enforce graph-first detection approach:
- `detection.md` must use BSKG operations, not function names
- `reasoning_template` must use semantic operations
- VQL queries must be operation-based, not name-based

Rationale: Maintains consistency with AlphaSwarm.sol philosophy (semantic operations > name heuristics).
