    Create the /vrs-refine skill for iterative pattern improvement.

    **Reference 05.4-05 skill patterns for consistency** - match invocation model,
    tool documentation, and output format from discover.md and add-vulnerability.md.

    Structure the skill file:

    1. Header section:
       - Slash command: vrs:refine
       - Context: fork
       - Model tier: Opus 4.5 (quality-critical)

    2. Purpose:
       - Improve pattern accuracy based on test results
       - Reduce false positives while maintaining recall
       - Update pattern YAML with refined conditions
       - Update test_coverage metrics

    3. Workflow:
       a) Identify target:
          - Pattern ID or vulndoc path
          - Load current pattern YAML
          - Load associated vulndoc

       b) Analyze failures:
          - Parse FP/FN from test results
          - Identify common characteristics of failures
          - Group by failure type

       c) Propose refinements:
          - Adjust tier_a conditions
          - Add tier_c label requirements if applicable
          - Tighten or loosen sequence_order
          - Add false_positive_indicators

       d) Test refinement (via Bash tool):
          - Run refined pattern against test corpus
          - Compare metrics before/after
          - Only proceed if improvement

       e) Update files (via Write tool):
          - Update pattern YAML
          - Update vulndoc index.yaml test_coverage
          - Mark pattern as "draft" if significant changes

    4. Tools (invoked by agent via standard tool calls):
       - Read (patterns, vulndocs, test results)
       - Write (pattern YAML, vulndoc index.yaml)
       - Bash (uv run alphaswarm query, pytest)

       **Note:** CLI commands are invoked via Bash tool, matching 05.4-05 skills.

    5. Refinement strategies:
       - FP: Add exclusion conditions (none: [...])
       - FN: Loosen matching (op: in instead of eq)
       - Context-dependent: Add tier_c label requirements
       - Ordering issues: Adjust sequence_order

    6. Output format:
       ```yaml
       refinement:
         pattern_id: vm-001-classic
         before:
           precision: 0.75
           recall: 0.90
         after:
           precision: 0.88
           recall: 0.87
         changes:
           - added: has_reentrancy_guard = false condition
           - reason: Reduced FP on guarded functions
       ```

    7. Key rules:
       - Never reduce recall below 0.70
       - Always test before committing changes
       - Mark as draft if precision < 0.70
       - Use semantic operations, not function names
  </action>

  <name>Task 2: Create vrs:test-pattern skill</name>
  <files>
    .claude/skills/vrs/test-pattern.md
  </files>
  <action>
    Create the /vrs-test-pattern skill for pattern validation.

    **Reference 05.4-05 skill patterns for consistency.**

    Structure the skill file:

    1. Header section:
       - Slash command: vrs:test-pattern
       - Context: fork
       - Model tier: Sonnet 4.5

    2. Purpose:
       - Validate patterns against real-world projects
       - Measure precision and recall
       - Generate ground truth for new test cases
       - Identify FP/FN for refinement

    3. Workflow:
       a) Select test corpus:
          - DVDeFi challenges
          - Real-world audit reports
          - Benchmark contracts
          - Option: user-specified projects

       b) Build knowledge graphs (via Bash tool):
          - `uv run alphaswarm build-kg {project}`
          - Ensure graphs are fresh

       c) Run pattern (via Bash tool):
          - `uv run alphaswarm query "pattern:{pattern_id}"`
          - Collect matches

       d) Validate matches:
          - For each match, check against ground truth
          - If no ground truth, request manual verification
          - Track TP, FP, FN counts

       e) Calculate metrics:
          - Precision = TP / (TP + FP)
          - Recall = TP / (TP + FN)
          - Update pattern test_coverage

       f) Report:
          - Summary statistics
          - List of FP (for refinement)
          - List of FN (for investigation)
          - Suggested improvements

    4. Tools (invoked by agent via standard tool calls):
       - Bash (alphaswarm build-kg, alphaswarm query)
       - Read (patterns, ground truth files)
       - Write (test results, pattern updates)

       **Note:** CLI commands are invoked via Bash tool, matching 05.4-05 skills.

    5. Ground truth format:
       ```yaml
       # tests/ground_truth/{category}/{subcategory}.yaml
       vulnerable_functions:
         - file: contracts/Vulnerable.sol
           function: withdraw
           vuln_type: reentrancy-classic
       safe_functions:
         - file: contracts/Safe.sol
           function: safeWithdraw
           reason: Uses ReentrancyGuard
       ```

    6. Key rules:
       - Always use BSKG queries (via Bash), not code scanning
       - Record all verification decisions
       - Update test_coverage in vulndoc after testing
       - Flag patterns below 0.70 precision as draft
  </action>
