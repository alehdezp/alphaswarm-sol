Purpose: Enable automated vulnerability knowledge discovery via Exa search and
structured addition of new vulnerabilities to the framework. Skills use vrs:
prefix per Phase 6 rebrand decision.

**Important: Skill Invocation Model**
Skills are documentation files that define agent behavior. When an agent (Claude Code)
loads a skill, it follows the documented workflow and invokes CLI commands via the
Bash tool. Skills do NOT have a separate invocation mechanism - they ARE the prompt
that guides the agent, and the agent uses standard tools (Bash, Read, Write, etc.)
to execute the skill's workflow.

  <action>
    Create the /vrs-discover skill for automated vulnerability discovery.

    Structure the skill file:

    1. Header section:
       - Slash command: vrs:discover
       - Context: fork (separate conversation)
       - Model tier: Sonnet 4.5 (with Haiku 4.5 for filtering)

    2. Purpose:
       - Automated search for new vulnerabilities not in vulndocs/
       - Compare findings against existing coverage
       - Aggregate sources into dense context
       - Suggest new entries or updates

    3. Workflow:
       a) Load current vulndocs coverage:
          - Run `uv run alphaswarm vulndocs list --json` via Bash tool
          - Parse categories and subcategories

       b) Search for new vulnerabilities:
          - Use mcp__exa-search__web_search_exa
          - Queries: "Solidity vulnerability 2025", "smart contract exploit",
            specific category gaps
          - Filter results with Haiku (quick relevance check)

       c) Compare to existing coverage:
          - Check if found vulnerability matches existing category
          - Identify truly new patterns vs. variants

       d) Aggregate and report:
          - Summarize findings
          - Suggest: new category, new subcategory, update existing
          - Output structured suggestions

    4. Tools (invoked by agent, not separate mechanism):
       - Read (vulndocs/, patterns/)
       - Bash (uv run alphaswarm vulndocs *)
       - mcp__exa-search__web_search_exa
       - mcp__exa-search__get_code_context_exa
       - Task (for spawning Haiku filter workers)

       **Note:** Agent invokes CLI commands via Bash tool. Skill documentation
       guides the agent on WHICH commands to run.

    5. Output format:
       ```yaml
       discoveries:
         - type: new_category
           suggested_id: defi-lending
           evidence: [urls]
           priority: high
         - type: new_subcategory
           category: oracle
           suggested_id: chainlink-stale
           evidence: [urls]
           priority: medium
       ```

    6. Key rules:
       - Never add without user confirmation
       - Always check existing coverage first
       - Dense summaries, not verbose narratives

  <files>
    .claude/skills/vrs/add-vulnerability.md
  </files>
  <action>
    Create the /vrs-add-vulnerability skill for structured addition.

    Structure the skill file:

    1. Header section:
       - Slash command: vrs:add-vulnerability
       - Context: fork
       - Model tier: Sonnet 4.5

    2. Purpose:
       - Add new vulnerability to vulndocs framework
       - Ensure proper structure from templates
       - Populate Phase 7 test generation fields
       - Validate before committing

    3. Workflow:
       a) Gather inputs:
          - Category (existing or new)
          - Subcategory name
          - Severity
          - Brief description

       b) Check for existing:
          - Search vulndocs/ for similar entries
          - If duplicate, suggest update instead

       c) Create structure (via Bash tool):
          - Run `uv run alphaswarm vulndocs scaffold {category} {subcategory}` via Bash
          - Verify folder created

       d) Populate content (via Write tool):
          - Edit index.yaml with provided info
          - Add semantic_triggers based on vulnerability type
          - Add reasoning_template
          - Create detection.md with graph-based approach

       e) Validate (via Bash tool):
          - Run `uv run alphaswarm vulndocs validate vulndocs/{category}/{subcategory}` via Bash
          - Fix any errors

       f) Report:
          - Show created files
          - Suggest next steps (add patterns, write tests)

    4. Tools (invoked by agent via standard tool calls):
       - Read (existing vulndocs for comparison)
       - Write (new files)
       - Bash (vulndocs CLI commands - scaffold, validate)
       - Glob (find similar entries)

       **Note:** The skill file documents the workflow. The agent follows this
       workflow and uses its standard tools (Bash, Read, Write) to execute.
       CLI commands are invoked via `Bash` tool calls.

    5. Example invocation:
       ```
       /vrs-add-vulnerability
       Category: oracle
       Subcategory: chainlink-stale-price
       Severity: high
       Description: Chainlink price feeds not checked for staleness
       ```

    6. Key rules:
       - Always use scaffold command first (via Bash)
       - Populate Phase 7 fields (semantic_triggers, reasoning_template)
       - Never create without validation passing
       - Use BSKG operations, not function names
  </action>
