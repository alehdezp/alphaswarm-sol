  <name>Task 3: Create vrs:generate-tests skill</name>
  <files>
    .claude/skills/vrs/generate-tests.md
  </files>
  <action>
    Create the /vrs-generate-tests skill for Phase 7 test case generation.

    **Reference existing vrs: skills for consistency.**

    Structure the skill file:

    1. Header section:
       - Slash command: vrs:generate-tests
       - Context: fork
       - Model tier: Sonnet 4.5

    2. Purpose:
       - Generate test cases from vulndoc reasoning_template
       - Create test contracts for Phase 7 validation
       - Support all three test quality focuses:
         a) Semantic complexity (graph-based reasoning)
         b) Real-world patterns (from exploits)
         c) Adversarial (obfuscation, edge cases)

    3. Workflow:
       a) Load vulndoc:
          - Read index.yaml for vulnerability
          - Extract semantic_triggers, vql_queries, reasoning_template
          - Load related exploits.md

       b) Generate test contracts (via Write tool):
          - Vulnerable variant: Matches pattern
          - Safe variant: Uses mitigations
          - Adversarial variant: Obfuscated but vulnerable
          - Edge case variant: Boundary conditions

       c) Generate test assertions:
          - VQL queries that should match vulnerable
          - VQL queries that should NOT match safe
          - Expected detection outcome per contract

       d) Output test files (via Write tool):
          - tests/contracts/{category}/{subcategory}/
          - test_{subcategory}_vulnerable.sol
          - test_{subcategory}_safe.sol
          - test_{subcategory}_adversarial.sol

       e) Update vulndoc (via Write tool):
          - Add test file references to test_coverage
          - Update validation level

    4. Tools (invoked by agent via standard tool calls):
       - Read (vulndocs, patterns)
       - Write (test contracts, test files)
       - Bash (uv run alphaswarm build-kg, validate)

       **Note:** CLI commands invoked via Bash tool, matching other vrs: skills.

    5. Test quality requirements:
       - Semantic: Must test graph properties, not names
       - Real-world: Derived from documented exploits
       - Adversarial: Include misleading function names

    6. Example output:
       ```solidity
       // tests/contracts/reentrancy/classic/test_classic_vulnerable.sol
       // Test for: reentrancy/classic
       // Expected: Pattern vm-001-classic matches withdraw()
       // Semantic: TRANSFERS_VALUE_OUT before WRITES_USER_BALANCE

       contract VulnerableWithdraw {
           mapping(address => uint) balances;

           function withdraw(uint amount) public {
               require(balances[msg.sender] >= amount);
               (bool success, ) = msg.sender.call{value: amount}("");
               balances[msg.sender] -= amount; // Write after call
           }
       }
       ```

    7. Key rules:
       - Test semantic operations, not function names
       - Include both positive and negative test cases
       - Reference reasoning_template in test comments
       - Track all tests in test_coverage
  </action>
