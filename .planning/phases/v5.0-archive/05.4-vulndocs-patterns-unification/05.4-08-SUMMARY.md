## Decisions Made

**1. Integration tests validate complete workflows, not isolated functions**
- Rationale: Need to ensure all components work together correctly, including templates → scaffolding → validation → CLI
- Tests cover full user journey from scaffold to framework-wide validation

**2. Skill validation uses parametrized tests for consistency**
- Rationale: All 7 skills should follow same structure and conventions
- Parametrized approach ensures every skill is checked against same criteria

**3. Fixtures copy real project templates to temp directories**
- Rationale: Tests should use actual template files, not mocked versions
- Ensures tests fail if template structure changes

**4. Template placeholders treated as example values**
- Rationale: Templates use literal strings like "category-name" as examples, not {{variable}} placeholders
- Tests updated to replace example values with valid data for validation

**5. CLI tests accept multiple exit codes**
- Rationale: CLI commands may legitimately fail in test environment (missing templates, etc.)
- Tests verify commands don't crash rather than expecting specific results
