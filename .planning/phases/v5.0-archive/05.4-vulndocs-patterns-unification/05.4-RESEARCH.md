# Phase 5.4: VulnDocs-Patterns Unification Framework - Research

**Researched:** 2026-01-22
**Domain:** Knowledge management, pattern systems, agent skills
**Confidence:** HIGH

## Summary

This phase creates a **framework** for unified vulnerability knowledge management - the structure, templates, validation, and agent skills. The actual migration of existing content (44 patterns, 17+ vulndocs categories) is explicitly OUT OF SCOPE and deferred to a separate task.

The research reveals that the existing system has TWO parallel knowledge structures:
1. **VulnDocs** in `knowledge/vulndocs/categories/` - 20 categories with nested subcategories, using `index.yaml` + `core-pattern.md` (minimal approach)
2. **Patterns** in `patterns/` - 44 YAML patterns in `core/`, `semantic/`, and `label_patterns/` directories, plus 7 lens files

The unification goal is to move patterns INTO vulnerability folders (co-location) and flatten the structure to `vulndocs/` at project root for better discoverability. The framework-first approach means building empty structure, templates, validation CLI, and agent skills before any content migration.

**Primary recommendation:** Build the framework in 5-6 waves: (1) folder scaffolding with `.meta/templates/` and `.meta/instructions/`, (2) `index.yaml` schema with Phase 7 test generation support, (3) validation CLI with progressive quality enforcement, (4) agent skills using `vrs:` prefix, (5) integration tests, (6) documentation.

### Pattern 1: Progressive Validation
**What:** Minimal quality enforced, more files unlock more features.
**When to use:** During validation of vulnerability folders.
**Example:**
```python
class ValidationLevel(Enum):
    MINIMAL = "minimal"      # Just index.yaml with required fields
    STANDARD = "standard"    # index.yaml + at least one md file
    COMPLETE = "complete"    # All recommended files present
    EXCELLENT = "excellent"  # All files + patterns with test coverage

def validate_vulnerability(path: Path) -> ValidationResult:
    """Progressive validation - minimal requirements, incremental quality."""
    result = ValidationResult(path=path)

    # REQUIRED: index.yaml must exist and have minimum fields
    index_path = path / "index.yaml"
    if not index_path.exists():
        result.errors.append("Missing required index.yaml")
        result.level = None
        return result

    index = yaml.safe_load(index_path.read_text())
    required_fields = ["id", "category", "subcategory", "severity"]
    for field in required_fields:
        if field not in index:
            result.errors.append(f"Missing required field: {field}")

    if result.errors:
        result.level = None
        return result

    # MINIMAL achieved
    result.level = ValidationLevel.MINIMAL

    # Check for STANDARD level
    md_files = list(path.glob("*.md"))
    if md_files:
        result.level = ValidationLevel.STANDARD

    # Check for COMPLETE level
    expected_files = ["overview.md", "detection.md", "verification.md"]
    present = [f for f in expected_files if (path / f).exists()]
    if len(present) >= 3:
        result.level = ValidationLevel.COMPLETE

    # Check for EXCELLENT level
    patterns_dir = path / "patterns"
    if patterns_dir.exists():
        patterns = list(patterns_dir.glob("*.yaml"))
        tested_patterns = [p for p in patterns if _has_test_coverage(p)]
        if tested_patterns:
            result.level = ValidationLevel.EXCELLENT

    return result
```

### Pattern 2: Index Schema with Phase 7 Support
**What:** `index.yaml` schema that supports both LLM analysis AND test generation.
**When to use:** Every vulnerability folder must have this.
**Example:**
```yaml

  - READS_EXTERNAL_VALUE
  - READS_ORACLE
  - USES_IN_CALCULATION

  - "FIND functions WHERE reads_oracle AND uses_external_value_in_calculation"

  - external_price_read -> calculation -> state_write

  1. Identify oracle reads (getPrice, latestAnswer, etc.)
  2. Trace value flow to calculations
  3. Check for manipulation windows (single-block, no TWAP)
  4. Verify bounds checking or staleness checks

  - tests/test_oracle_lens.py:test_price_manipulation_*

  - oracle-001-twap
  - oracle-002-spot
```

### Anti-Patterns to Avoid
- **Registry files:** No `registry.yaml` - folder structure IS the source of truth
- **Deep nesting:** Prefer 2 levels, allow 3 only when truly needed
- **Name-based detection:** Always use semantic operations, not function names
- **Duplicate content:** Patterns embedded in vulndocs, not separate hierarchy
- **Manual code reading:** Agents MUST use BSKG graph queries

## Don't Hand-Roll

Problems that look simple but have existing solutions:

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| YAML validation | Custom parser | pydantic + jsonschema | Edge cases, type coercion |
| Template rendering | String concatenation | jinja2 | Escaping, loops, conditionals |
| CLI commands | argparse | click (existing) | Consistent with project |
| Path handling | String manipulation | pathlib | Cross-platform, safe |
| Schema generation | Manual JSON Schema | pydantic.json_schema() | Automatic from models |

**Key insight:** The project already has all the infrastructure needed. Don't introduce new patterns.

## Common Pitfalls

### Pitfall 1: Over-Specifying Migration Details
**What goes wrong:** Spending time on migration scripts when framework isn't validated.
**Why it happens:** Natural instinct to plan the full journey.
**How to avoid:** Framework-first: build empty structure, validate it works, THEN migrate.
**Warning signs:** Discussing "how to preserve git history" before templates exist.

### Pitfall 2: Registry File Creep
**What goes wrong:** Creating registry.yaml that duplicates folder structure.
**Why it happens:** Seems easier to parse a single file.
**How to avoid:** Folder structure IS the registry. Build discovery from filesystem.
**Warning signs:** Discussions about "syncing registry with folders".

### Pitfall 3: Inconsistent Pattern Location
**What goes wrong:** Some patterns in `patterns/`, some in `vulndocs/category/patterns/`.
**Why it happens:** Partial migration or unclear rules.
**How to avoid:** Framework validation rejects patterns without vulndoc reference.
**Warning signs:** Patterns without `vulndoc:` field in YAML.

### Pitfall 4: Validation Too Strict
**What goes wrong:** CI blocks on incomplete entries, slowing iteration.
**Why it happens:** Desire for quality from day one.
**How to avoid:** Progressive validation: warn on gaps, don't block. Require only `index.yaml`.
**Warning signs:** "All entries must have detection.md" as hard requirement.

### Pitfall 5: Forgetting Graph-First
**What goes wrong:** Skills that read source code instead of using BSKG queries.
**Why it happens:** LLMs naturally prefer reading code.
**How to avoid:** Skills MUST document required graph queries. Validation checks for VQL.
**Warning signs:** Agent prompts that mention "read the source file".

    """Validate vulndocs folder structure."""
    result = validate_framework(Path(path))

    for vuln in result.vulnerabilities:
        status = "OK" if vuln.level else "FAIL"
        click.echo(f"{status}: {vuln.path}")

            for err in vuln.errors:
                click.echo(f"  ERROR: {err}")

        if vuln.suggestions:
