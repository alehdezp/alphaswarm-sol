Created unified vulndocs framework with templates, instructions, and Phase 7 test generation support for vulnerability knowledge management.

## What Was Built

### Objective
Establish the foundation scaffolding for the unified VulnDocs framework at project root. Create empty folder structure with templates and instructions that enable consistent vulnerability documentation creation. Framework-first: no content migration yet.

### Components Delivered

1. **Templates Folder** (`vulndocs/.meta/templates/`)
   - `category.yaml`: Category structure template
   - `subcategory/index.yaml`: Full vulnerability metadata template (88 LOC)
     - Required fields: id, category, subcategory, severity, vulndoc
     - Phase 7 fields: semantic_triggers, vql_queries, graph_patterns, reasoning_template
   - `subcategory/*.md`: 5 markdown templates (overview, detection, verification, exploits)
   - `pattern.yaml`: Pattern template with vulndoc linkage

2. **Instructions Folder** (`vulndocs/.meta/instructions/`)
   - `index.md`: How to write index.yaml with all field explanations (281 LOC)
   - `detection.md`: Graph-first detection guidance (218 LOC)
   - `verification.md`: Verification workflow with graph queries (183 LOC)
   - `exploits.md`: Exploit documentation rules (151 LOC)
   - `patterns.md`: Pattern authoring guide (283 LOC)

3. **README Documentation** (`vulndocs/README.md`)
   - Framework purpose and structure (294 LOC)
   - Quick start guides
   - Validation levels (MINIMAL -> EXCELLENT)
   - Agent skills reference
   - Phase 7 test generation explanation
   - Best practices and design principles

## Key Decisions

### Decision 1: Progressive Validation
**Context:** Need to balance quality with iterative development.
**Choice:** Four-level validation (MINIMAL -> EXCELLENT) with only index.yaml required.
**Rationale:** Allows incremental improvements without blocking progress.

### Decision 2: Co-Located Patterns
**Context:** Patterns were separate from vulnerability docs in old structure.
**Choice:** Embed patterns in `vulndocs/category/subcategory/patterns/`.
**Rationale:** Better discoverability, simpler maintenance, bidirectional linking.

### Decision 3: Graph-First Enforcement
**Context:** Need to ensure agents use BSKG queries consistently.
**Choice:** All instructions emphasize "Use BSKG graph queries, NOT manual code reading."
**Rationale:** Consistent semantic reasoning, no name-based detection.

### Decision 4: Phase 7 Fields in Templates
**Context:** Test generation needs semantic triggers and reasoning templates.
**Choice:** Include semantic_triggers, vql_queries, graph_patterns, reasoning_template in index.yaml.
**Rationale:** Enable automated test generation from vulnerability knowledge.
