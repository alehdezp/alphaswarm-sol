# Phase 7.3.6: Continuous Validation Pipeline (The Living Test)

## Purpose

Ensure the product keeps working over time by establishing perpetual, automated validation that catches regressions immediately.

---

## Mandatory Rules

**CRITICAL: Before executing ANY plan in this phase, load these rules:**

1. **Read `.planning/testing/rules/canonical/RULES-ESSENTIAL.md`** — Condensed actionable rules
2. **Acknowledge rules** — "I have loaded the essential rules and will follow them"

### Blocking Requirements

All continuous validation MUST:

- [ ] Use `claude-code-controller` for ALL live workflow testing
- [ ] Save transcripts to `.vrs/testing/transcripts/`
- [ ] Meet minimum transcript lengths (50+ lines)
- [ ] Have realistic duration (not instant)
- [ ] Use external ground truth baselines
- [ ] Store baselines in `.vrs/validation/baselines/`
- [ ] Use the testing framework docs in `.planning/testing/` (start at `DOC-INDEX.md`)

### CI-Safe vs Live Runs

| Mode | When | Rules |
|------|------|-------|
| CI-safe | Unit tests, static checks | No claude-code-controller needed |
| Live | Workflow tests, E2E | MUST use claude-code-controller |

**CRITICAL:** CI pipeline MUST NOT claim live claude-code-agent-teams runs unless the environment supports them.

### claude-code-controller Testing Pattern (for Live Runs)

```bash
claude-code-controller launch "zsh"                    # 1. ALWAYS launch shell first
claude-code-controller send "cd /path/to/project" --pane=X
claude-code-controller wait_idle --pane=X
claude-code-controller send "claude" --pane=X          # 2. Start Claude Code
claude-code-controller wait_idle --pane=X --idle-time=10.0
claude-code-controller send "/vrs-audit contracts/" --pane=X  # 3. Execute workflow
claude-code-controller wait_idle --pane=X --idle-time=15.0 --timeout=300
claude-code-controller capture --pane=X                # 4. Capture transcript
# Save to .vrs/testing/transcripts/
claude-code-controller send "/exit" --pane=X           # 5. Cleanup
claude-code-controller kill --pane=X
```

## The Problem It Solves

Point-in-time validation has a fatal flaw:
- Test passes on January 30
- Code changes on February 1
- Regression introduced
- No one notices until user reports bug

Continuous validation catches regressions BEFORE they reach users.

## Tier 1: Git Hook Validation (Per-Commit)

### Trigger
Every commit to `main` or PR branches.

### What It Runs
```yaml
quick_validation:
  duration: < 5 minutes
  tests:
    - "Smoke test: alphaswarm --version works"
    - "Unit tests: pytest tests/unit/ -x --fast"
    - "Mini gauntlet: 3 contracts (1 vuln, 2 safe)"
    - "Graph build: Single contract builds successfully"

  pass_criteria:
    - All 4 tests pass
    - No regressions from previous commit

  on_failure:
    - Block commit/merge
    - Show failure details
    - Suggest fix
```

### Implementation

```bash
# .git/hooks/pre-push
#!/bin/bash
set -e

echo "Running AlphaSwarm quick validation..."

# 1. Version check
alphaswarm --version || exit 1

# 2. Fast unit tests
pytest tests/unit/ -x --fast -q || exit 1

# 3. Mini gauntlet
python scripts/mini_gauntlet.py || exit 1

# 4. Graph build test
alphaswarm build-kg tests/fixtures/simple/ || exit 1

echo "✓ Quick validation passed"
```

## Tier 2: Nightly Full Validation

### Trigger
Every night at 2 AM UTC (low-activity time).

### What It Runs
```yaml
nightly_validation:
  duration: 30-60 minutes
  tests:
    - "Full unit test suite"
    - "Full gauntlet (all 5 challenges)"
    - "DVD corpus (all available challenges)"
    - "SmartBugs reentrancy suite"
    - "Integration tests"

  metrics_captured:
    - precision
    - recall
    - f1_score
    - true_positives
    - false_positives
    - false_negatives
    - execution_time

  comparison:
    - Compare to previous night
    - Compare to weekly average
    - Compare to baseline

  alerting:
    regression_threshold: 5%  # Alert if any metric drops > 5%
```

## Tier 4: New Vulnerability Integration

### Trigger
When new CVE published in Web3/DeFi space.

### Protocol
```yaml
new_vulnerability_integration:
  trigger: "New CVE or exploit in Web3 space"
  timeline: "Within 48 hours of disclosure"

  process:
    1. Identify vulnerable code (if available)
    2. Create test fixture
    3. Run AlphaSwarm against it
    4. Evaluate:
       - detected: Add to baseline, celebrate
       - missed: Create pattern, verify, add to baseline

  tracking:
    - CVE ID
    - Date added to corpus
    - Detection status
    - Pattern created (if needed)
    - Verification date
```

## Tier 5: User Feedback Loop

### Trigger
User reports issue (GitHub issue, Discord, email).

### Protocol
```yaml
user_feedback_integration:
  trigger: "User reports bug, FP, or missed detection"

  process:
    1. Triage: Is it valid?
    2. Reproduce: Create minimal test case
    3. Add to corpus: Test fixture for regression
    4. Fix: Pattern/detection improvement
    5. Verify: Test case passes
    6. Regression test: Add to nightly suite

  tracking:
    - Issue ID
    - Reporter
    - Category (FP, missed, bug)
    - Reproduction steps
    - Fix implemented
    - Regression test added
```

### Issue → Test Conversion

```yaml
# Example: User reports false positive

github_issue: "#123 - False positive on safe reentrancy pattern"

reproduction:
  contract: |
    contract SafeVault {
        ReentrancyGuard guard;
        function withdraw() external guard.nonReentrant {
            // Safe: has guard
        }
    }
  expected: "No vulnerability"
  actual: "Flagged as reentrancy"

test_case:
  file: tests/fixtures/user-reported/issue-123-safe-reentrant.sol
  ground_truth: safe
  expected_findings: 0

fix:
  type: "Improved guard detection"
  pr: "#124"
  status: "Merged"

regression_test:
  added_to: nightly_suite
  date: 2026-02-01
```

## Alerting & Escalation

### Alert Levels

| Level | Trigger | Action |
|-------|---------|--------|
| INFO | Metric changed < 2% | Log only |
| WARN | Metric dropped 2-5% | Slack notification |
| ERROR | Metric dropped > 5% | Slack + Email + GitHub issue |
| CRITICAL | Gauntlet failure | Block releases, immediate fix |

### Escalation Protocol

```yaml
regression_detected:
  severity: ERROR (precision dropped 7%)

  immediate_actions:
    - Create GitHub issue with details
    - Notify #alphaswarm-alerts Slack channel
    - Assign to on-call developer

  investigation:
    - Identify commit that introduced regression
    - Analyze root cause
    - Determine if revert needed

  resolution_timeline:
    - CRITICAL: 4 hours
    - ERROR: 24 hours
    - WARN: 1 week
```

## CI-Safe vs Live Runs

Continuous validation is split into:
- **CI-safe checks** (no Claude Code live sessions)
- **Live claude-code-agent-teams runs** on a self-hosted runner with subscription

Pipelines must explicitly state which mode is used. CI must not claim live claude-code-agent-teams runs unless the environment supports them.

## Baseline Manifest

Store baseline artifacts in:
`.vrs/validation/baselines/<suite>.json`

All regression checks must compare against this baseline, and baseline updates must be explicitly recorded.
