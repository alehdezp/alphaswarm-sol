# Phase 5.1: Static Analysis Tool Integration - Research

**Researched:** 2026-01-21
**Domain:** Static analysis tool integration and orchestration
**Confidence:** HIGH

## Summary

Phase 5.1 implements Pillar 6 (Tool Orchestration) by integrating external static analysis tools (Slither, Aderyn, Mythril, Echidna, Halmos, Foundry, Semgrep) into VKG's workflow. The research reveals that BSKG already has foundational infrastructure in place (`orchestration/runner.py`, `orchestration/dedup.py`, `tools/runner.py`) supporting Slither, Aderyn, and BSKG patterns with basic deduplication.

The standard approach is to run tools in parallel, normalize outputs to a common format (SARIF or internal schema), deduplicate findings using location-based + semantic similarity algorithms, and spawn beads for multi-agent verification. Cross-tool deduplication is technically challenging due to vendor-specific output formats and classification systems.

**Primary recommendation:** Build on existing infrastructure by (1) creating a SARIF adapter as the universal normalization layer, (2) implementing tool-specific agent skills with lazy loading, (3) enhancing dedup.py with semantic similarity (embeddings), (4) creating a tool coordinator subagent for smart execution strategy, and (5) extending bead creation to handle tool findings with initial confidence=uncertain.

### Pattern 1: SARIF as Universal Normalization Layer

**What:** Use SARIF 2.1.0 as the intermediate format for all tool outputs. Convert tool-specific JSON/text output to SARIF, then normalize SARIF to VKG's internal finding format.

**When to use:** Always, for all external tools. SARIF provides standardization and future-proofs against tool output format changes.

**Example:**
```python
# Source: BSKG codebase analysis + SARIF spec research
from pathlib import Path
from typing import Dict, Any, List

class SARIFAdapter:
    """Universal adapter: Tool output → SARIF → BSKG Finding"""

    def normalize_to_sarif(self, tool: str, raw_output: str) -> Dict[str, Any]:
        """Convert tool-specific output to SARIF 2.1.0"""
        if tool == "slither":
            return self._slither_to_sarif(raw_output)
        elif tool == "aderyn":
            return self._aderyn_to_sarif(raw_output)
        # ... other tools

    def sarif_to_vkg_findings(self, sarif: Dict[str, Any]) -> List[Dict[str, Any]]:
        """Convert SARIF to BSKG internal finding format"""
        findings = []
        for run in sarif.get("runs", []):
            tool_name = run["tool"]["driver"]["name"]
            for result in run.get("results", []):
                findings.append({
                    "source": tool_name.lower(),
                    "id": result["ruleId"],
                    "severity": self._sarif_level_to_severity(result["level"]),
                    "category": result["ruleId"],
                    "file": self._extract_file(result),
                    "line": self._extract_line(result),
                    "function": self._extract_function(result),
                    "confidence": self._extract_confidence(result),
                    "raw": result
                })
        return findings
```

### Pattern 2: Tool Coordinator for Smart Execution

**What:** A coordinator subagent analyzes the project and decides which tools to run, with what configurations, and in what order.

**When to use:** Every analysis run. The coordinator makes project-specific decisions (e.g., skip fuzzing for view-only contracts, prioritize Mythril for complex arithmetic).

**Example:**
```python
# Source: Phase 5.1 context requirements
from dataclasses import dataclass
from typing import List, Dict, Any

@dataclass
class ToolStrategy:
    """Execution strategy determined by coordinator"""
    tools_to_run: List[str]
    parallel_groups: List[List[str]]  # Tools that can run in parallel
    tool_configs: Dict[str, Dict[str, Any]]
    estimated_time: int  # seconds
    rationale: str

class ToolCoordinator:
    """Analyzes project and decides tool execution strategy"""

    def analyze_and_plan(self, project_path: Path) -> ToolStrategy:
        """Analyze project complexity and return execution strategy"""
        # Analyze project characteristics
        has_complex_math = self._detect_complex_arithmetic(project_path)
        has_value_transfers = self._detect_value_transfers(project_path)
        has_upgradeable = self._detect_proxy_patterns(project_path)
        contract_count = self._count_contracts(project_path)

        tools = ["slither", "aderyn"]  # Always run baseline

        # Add symbolic execution for complex logic
        if has_complex_math or has_upgradeable:
            tools.append("mythril")
            tools.append("halmos")

        # Add fuzzing for value-handling contracts
        if has_value_transfers:
            tools.append("echidna")
            tools.append("foundry")

        # Parallel execution groups (independent tools)
        parallel_groups = [
            ["slither", "aderyn", "semgrep"],  # Fast static analysis
            ["mythril", "halmos"],              # Symbolic execution
            ["echidna", "foundry"]              # Fuzzing
        ]

        return ToolStrategy(
            tools_to_run=tools,
            parallel_groups=parallel_groups,
            tool_configs=self._generate_configs(tools, project_path),
            estimated_time=self._estimate_time(tools, contract_count),
            rationale=self._explain_strategy(tools)
        )
```

### Pattern 3: Semantic Similarity Deduplication

**What:** Use embeddings + cosine similarity to detect duplicate findings across tools, even when locations differ slightly or descriptions vary.

**When to use:** After location-based clustering fails to match findings, or when tools report same issue in different ways.

**Example:**
```python
# Source: Semantic deduplication research (SemHash, NVIDIA NeMo)
from typing import List, Tuple
import hashlib

class SemanticDeduplicator:
    """Enhanced deduplication with semantic similarity"""

    def __init__(self, embedding_model="all-MiniLM-L6-v2"):
        # Lightweight embedding model for fast similarity
        from sentence_transformers import SentenceTransformer
        self.model = SentenceTransformer(embedding_model)

    def deduplicate_with_semantics(
        self,
        findings: List[Dict[str, Any]],
        location_tolerance: int = 5,
        semantic_threshold: float = 0.85
    ) -> List[DeduplicatedFinding]:
        """Two-stage deduplication: location first, then semantic"""

        # Stage 1: Location-based clustering (existing dedup.py logic)
        location_clusters = self._cluster_by_location(findings, location_tolerance)

        # Stage 2: Semantic similarity within fuzzy location range
        final_clusters = []
        for cluster in location_clusters:
            semantic_subclusters = self._cluster_by_semantics(
                cluster, semantic_threshold
            )
            final_clusters.extend(semantic_subclusters)

        return [self._cluster_to_finding(c) for c in final_clusters]

    def _cluster_by_semantics(
        self,
        findings: List[Dict[str, Any]],
        threshold: float
    ) -> List[List[Dict[str, Any]]]:
        """Group findings by semantic similarity of descriptions"""
        # Generate embeddings for finding descriptions
        descriptions = [f.get("title", "") + " " + f.get("description", "")
                       for f in findings]
        embeddings = self.model.encode(descriptions)

        # Compute pairwise cosine similarities
        from sklearn.metrics.pairwise import cosine_similarity
        similarities = cosine_similarity(embeddings)

        # Cluster based on threshold
        clusters = []
        used = set()
        for i, finding in enumerate(findings):
            if i in used:
                continue
            cluster = [finding]
            used.add(i)
            for j in range(i + 1, len(findings)):
                if j not in used and similarities[i][j] >= threshold:
                    cluster.append(findings[j])
                    used.add(j)
            clusters.append(cluster)

        return clusters
```

### Anti-Patterns to Avoid

- **Running all tools always:** Wasteful. Use coordinator to skip irrelevant tools (e.g., fuzzing for view-only contracts).
- **Trusting tool confidence scores directly:** Tools have different calibration. Use reliability-based weighting instead.
- **Auto-confirming on tool agreement:** Per PHILOSOPHY.md, all verdicts require multi-agent verification. Tool agreement → "likely", not "confirmed".
- **Blocking on tool failures:** Tools crash/timeout regularly. Graceful degradation is mandatory.
- **Ignoring tool plugins/extensions:** Semgrep needs Decurity rules, Echidna needs property tests. Check for required additions.

## Don't Hand-Roll

Problems that look simple but have existing solutions:

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| SARIF schema validation | Custom JSON schema validator | `jsonschema` with official SARIF 2.1.0 schema | SARIF is complex (500+ page spec), edge cases matter |
| Semantic similarity | Custom word overlap heuristics | `sentence-transformers` with pre-trained models | Pre-trained models handle synonyms, context, embeddings |
| File hash caching | Manual MD5 tracking | `hashlib.sha256` with file mtime checks | Race conditions, collision handling, partial reads |
| Tool discovery | Hardcoded paths | `shutil.which()` + config registry + env vars | Cross-platform path resolution, user customization |
| Tool output parsing | Regex on text output | Parse JSON/SARIF, fallback to text only if needed | Tools change text format frequently, JSON is stable |
| Confidence calibration | Linear weighted average | Tool-specific reliability scores with Bayesian updates | Tools have different false positive rates per detector |

**Key insight:** Tool orchestration has mature solutions for normalization (SARIF), discovery (PATH + registry), and error handling (graceful degradation). Custom implementations introduce bugs and maintenance burden.

## Common Pitfalls

### Pitfall 1: Cross-Tool Deduplication Overconfidence

**What goes wrong:** Assuming two tools reporting the same category + location means same finding. Different tools use different classification taxonomies (Slither "reentrancy-eth" vs Aderyn "state-change-after-external-call").

**Why it happens:** Vendor-specific categorization, no universal vulnerability taxonomy.

**How to avoid:**
- Use semantic similarity on descriptions, not just category matching
- Maintain explicit tool detector → BSKG pattern mapping
- Default to separate beads if similarity < 0.85

**Warning signs:** High false duplicate rate (>20%), findings grouped that are actually distinct issues.

### Pitfall 2: Tool Installation Assumptions

**What goes wrong:** Failing hard when a tool is not installed, or silently skipping without informing user.

**Why it happens:** Complex installation (cargo for Aderyn, Foundry script, Nix for Echidna), not in PyPI.

**How to avoid:**
- Health check all tools before analysis, report missing tools clearly
- Offer guided installation with recommended method per tool
- Gracefully skip missing tools with warning, continue with available tools
- Provide super-install script (`.vrs/install-tools.sh`) with selectable tools

**Warning signs:** User confusion ("why no Mythril results?"), silent failures, cryptic PATH errors.

### Pitfall 3: Tool Config Drift

**What goes wrong:** Tool runs with suboptimal defaults, misses detectors, or flags too many false positives.

**Why it happens:** Tools have 20+ config options, defaults change between versions.

**How to avoid:**
- Ship VKG-optimized configs per tool in `.vrs/tool-configs/`
- Allow project-level overrides in `.vrs/tools.yaml`
- Document why each config option is set (rationale in comments)
- Validate configs before running (catch typos early)

**Warning signs:** Unexpected tool behavior, missed detections that should be caught, excessive noise.

### Pitfall 4: Blocking on Slow Tools

**What goes wrong:** Analysis hangs for 10+ minutes waiting for Mythril symbolic execution on large contracts.

**Why it happens:** Symbolic execution is exponential in code complexity, timeouts are too generous.

**How to avoid:**
- Set aggressive timeouts per tool (60s Slither, 120s Mythril, 300s Echidna)
- Run slow tools (Mythril, Halmos, Echidna) only when coordinator deems necessary
- Use partial results from timeout (capture stdout before kill)
- Report timeout as "incomplete" not "failure"

**Warning signs:** User complaints about analysis time, hung processes, impatient cancellations.

### Pitfall 5: Ignoring Library Scope

**What goes wrong:** Tools analyze OpenZeppelin, Solmate, PRBMath, generating hundreds of false positives on battle-tested library code.

**Why it happens:** Tools don't distinguish protocol code from dependencies.

**How to avoid:**
- Maintain library allowlist (OpenZeppelin, Solmate, PRBMath + version registry)
- Use path heuristics (`node_modules/`, `lib/`, `.cache/`) to detect dependencies
- Pass `--filter-paths` flags to tools to skip libraries
- Allow user override in `.vrs/tools.yaml` for custom libraries

**Warning signs:** 90%+ findings in library code, user frustration with noise, wasted analysis time.

### File Hash Caching

```python
# Source: Research on file-based caching + BSKG cache patterns
import hashlib
import json
from pathlib import Path
from typing import Optional

class ToolResultCache:
    """Cache tool results by file hash for incremental analysis"""

    def __init__(self, cache_dir: Path):
        self.cache_dir = cache_dir
        self.cache_dir.mkdir(parents=True, exist_ok=True)

    def get_file_hash(self, file_path: Path) -> str:
        """Compute SHA256 hash of file content"""
        sha256 = hashlib.sha256()
        with open(file_path, 'rb') as f:
            for chunk in iter(lambda: f.read(4096), b''):
                sha256.update(chunk)
        return sha256.hexdigest()

    def get_cached_result(
        self,
        tool: str,
        file_path: Path
    ) -> Optional[dict]:
        """Retrieve cached tool result if hash matches"""
        file_hash = self.get_file_hash(file_path)
        cache_key = f"{tool}_{file_hash}.json"
        cache_path = self.cache_dir / cache_key

        if cache_path.exists():
            with open(cache_path) as f:
                cached = json.load(f)
                # Verify mtime hasn't changed (detect manual edits)
                if cached["mtime"] == file_path.stat().st_mtime:
                    return cached["result"]
        return None

    def cache_result(
        self,
        tool: str,
        file_path: Path,
        result: dict
    ) -> None:
        """Cache tool result with file hash and mtime"""
        file_hash = self.get_file_hash(file_path)
        cache_key = f"{tool}_{file_hash}.json"
        cache_path = self.cache_dir / cache_key

        with open(cache_path, 'w') as f:
            json.dump({
                "file_hash": file_hash,
                "mtime": file_path.stat().st_mtime,
                "tool": tool,
                "result": result
            }, f, indent=2)
```
