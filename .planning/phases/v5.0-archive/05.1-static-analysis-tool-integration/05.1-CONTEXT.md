# Phase 5.1: Static Analysis Tool Integration - Context

**Gathered:** 2026-01-21
**Status:** Ready for planning

<domain>
## Phase Boundary

Integrate external static analysis tools (Slither, Mythril, Aderyn, Echidna, Halmos, Foundry, Semgrep) into the BSKG workflow with agent skills, smart deduplication, and verification pipeline integration. This implements Pillar 6 (Tool Orchestration) from PHILOSOPHY.md.

Tools provide findings; BSKG normalizes, deduplicates, and feeds them to agents for verification. No auto-confirmation — all tool findings require multi-agent verification.

</domain>

<decisions>
## Implementation Decisions

### Tool Priority & Coverage
- **Core tools (MUST-HAVE):** All Tier 0/1 — Slither, Aderyn, Mythril, Echidna, Foundry, Semgrep
- **Formal verification:** Include Halmos, defer Certora (licensed)
- **Missing tools:** Offer installation with recommended method, create super script for all dependencies (selectable which to install), gracefully skip if not installed
- **Tool health:** Validate all tools upfront before analysis

### Tool Execution
- **Execution model:** Parallel but smart — analyze project first, decide optimal strategy
- **Tool plugins:** Ensure all tools have complete plugins, extensions, custom rules installed
- **Tool configs:** VKG-managed optimal defaults + user overrides per project
- **Custom detectors:** Ship VKG-specific detectors for each tool (not just defaults)

### Scope Filtering
- **Library handling:** Scope-aware — skip known safe libraries (not protocol code)
- **Library detection:** Allowlist (OpenZeppelin, Solmate, PRBMath) + version-pinned registry + path heuristics (node_modules/, lib/)
- **Caching:** Cache tool results by file hash for incremental analysis

### Tool-Specific Decisions
- **Echidna:** Auto-generate property tests from BSKG patterns + supplement existing project tests
- **Foundry:** Test generation + execution (generate PoCs from findings, run them)
- **Semgrep:** Expand rules significantly + auto-expand when new patterns discovered
- **Rule expansion criteria:** Agnostic to variable names, not pattern-duplicated, good precision regardless of codebase

### Agent Skill Design
- **Skill structure:** Both levels — individual tool skills (/vkg:run-slither, etc.) + unified orchestration skill + specialized subagents
- **Skill loading:** Lazy loaded on demand (not always in context — reduces token pollution)
- **Skill documentation:** Full playbook — invocation + interpretation guide + common false positives + when to use vs skip + limitations + examples
- **Dynamic config:** Agents have full flexibility to modify tool configs during audit
- **Tool selection:** Include decision trees in skills ("why this tool" reasoning)
- **Tool coordinator:** Dedicated coordinator subagent that analyzes project and decides tool strategy
- **Error handling:** Retry with different config, then report failure if still broken
- **Effectiveness tracking:** Log which tools found what, feed back into strategy
- **Auto-update:** Skills and subagents should keep up-to-date with latest tool releases, patterns, configs, flags
- **Skill testing:** Automated test framework to validate skill behavior
- **Tool discovery:** Standard paths + env vars + config registry (.vrs/tools.yaml) + auto-detection

### Deduplication Strategy
- **Identity matching:** Both location-based (file+function+line) AND semantic similarity (vuln class + overlapping code) with configurable thresholds
- **Confidence boost:** Weighted by tool reliability (Slither agreement counts more than Semgrep)
- **Tool disagreement:** Not meaningful — tools have different checks; don't penalize for disagreement
- **Merge approach:** Two-stage — dedupe tools first, then merge with BSKG pattern matches
- **Output format:** BSKG native format optimized for LLM usage (not SARIF for internal processing)
- **Programmatic priority:** Do as much deduplication programmatically as possible; don't rely on LLM interpretation for grouping
- **Tool mapping:** Explicit mapping of tool detector → BSKG pattern (e.g., slither.reentrancy-eth → vkg.reentrancy-classic)

### Finding Verification Flow
- **Initial confidence:** Tool-reliability-based (Slither high-confidence starts as "likely", others "uncertain")
- **Bead spawning:** All findings become beads after deduplication
- **Bead relationships:** Strong relationships between related beads for cascading effects
- **Data format:** Normalized data consistent across all tools, optimized for LLM understanding
- **Triage agent:** Prioritizes beads by severity/exploitability/agreement; knows about similar beads for cascade processing
- **No separate report:** Tool findings feed directly into agent verification workflow

### Batch Dismissal & Cross-Agent Verification
- **Batch dismissal:** Explicit mechanism — agent can dismiss all X-related beads with single reasoning
- **Dismissal logging:** All dismissals logged and challengeable by user or future findings
- **Cross-agent verification:** Required for batch dismissals — 2 of 2 agents must agree
- **Model diversity:** Different models required for cross-agent verification (e.g., Claude + GPT-4)
- **Conflict handling:** If agents disagree on dismissal, beads are NOT dismissed
- **Challenge process:** User can challenge dismissals; original reasoning shown to new verifiers
- **Challenge limit:** One re-challenge per dismissal, then final
- **Learning from mistakes:** If challenge succeeds, dismissal agent learns to not repeat same error

### Claude's Discretion
- Tool coverage matrix (which tool is best for which vuln class)
- Stable fingerprints for tracking same issues across runs
- Coverage report format (detailed attribution vs summary)
- Auto PoC generation strategy (always, on-demand, or based on finding type)
- Skill versioning approach

</decisions>

<specifics>
## Specific Ideas

- "Every tool should have complete plugins and additions — installable via super script"
- "Tools should smartly know where they are located — auto-detection + registry"
- "Skills should not overload agent context — lazy loading is critical"
- "Programmatic deduplication over LLM interpretation wherever possible"
- "When new patterns are discovered, auto-create rules for tools that could detect them if they meet quality bar"
- "Triage agent should speed up process by knowing which beads to verify first for cascade effects"
- "Cross-model verification reduces hallucination risk for batch dismissals"
- "This phase is about feeding LLM agents, not generating reports"

</specifics>
