<domain>
## Phase Boundary

Merge all vulnerability knowledge (patterns/, knowledge/, vulndocs/) into a single unified `vulndocs/` structure. Create hierarchical organization with proper metadata, templates, and instructions. Update all skills, agents, and CLI to use the new structure exclusively. Remove legacy folders after verified migration.

**Format clarification:**
- Pattern files: Stay YAML (multiline content quality preserved)
- Index/metadata files: Stay YAML (good for LLM consumption)
- Graph output: TOON format (handled separately by 5.6.1)
- Skills: Stay markdown

**Parallel execution with 5.6.1:** This phase uses YAML for all vulndocs content. Phase 5.6.1 (graph JSON → TOON) operates on different files. No blocking dependency.

</domain>

<decisions>
## Implementation Decisions

### Hierarchy Design
- Flexible depth: 2 levels default (category/subcategory), 3rd level only when subcategory gets complex
- Every folder gets `index.yaml` with metadata (YAML, not TOON - good for LLM already)
- Templates and instructions in `vulndocs/.meta/` (not .meta/templates/.meta/instructions)
- Cross-references via `related` field in index.yaml when applicable and clear (not confusing to LLM)
- Kebab-case naming: `reentrancy`, `access-control`, `oracle-manipulation`
- Top-level manifest/index for LLM context engineering - granular top-down access
- Severity in metadata only, doesn't affect folder structure

### Grouping Criteria
- Group by same root cause with good semantic relationship and economic context
- 3rd level subfolders: Claude decides case-by-case, prefer when context doesn't help detect other patterns
- Rare variants: Keep in same pattern if underlying issue is same (helps find novel similar vulns)
- Edge cases that don't help LLM thinking process go in separate pattern/folder
- Empty categories: Yes, create placeholder structure for completeness
- Cross-category vulns: Primary location + cross-references (no symlinks or duplicates)
- Uncategorized folder: Try categorization first, use `uncategorized/` only if truly doesn't fit
- Core documentation file: `core-pattern.md` (current convention)
- No hard line limit, prefer concise - skills/agents decide appropriate length
- Lens concept: Claude decides best organization, moving away from lens-centric to logical agentic approach

### Pattern Co-location
- **Patterns stay YAML** (multiline content, quality must be preserved)
- Patterns in nested `patterns/` subfolder: `vulndocs/reentrancy/classic/patterns/`
- Multiple patterns per subcategory: Claude and skills decide based on detection needs
- Tier A/B/C is metadata only, not folder structure
- Label patterns (Tier C) merge into vulndocs structure
- Pattern naming: Descriptive if needed, short for specifics - not redundant

### URL Ingestion Skill
- Auto-categorize and insert (no user confirmation needed)
- Deduplication logic: Claude decides appropriate similarity threshold
- Extract both docs and patterns if source quality is sufficient
- Multiple vulns from one URL: Claude decides split vs group based on content relationship
- Provenance tracking: Separate `provenance.yaml` file per subcategory
- Discovery agents should not follow provenance URLs unless strictly needed
- Low confidence categorization: Create new category or use `uncategorized/`
- Quality gate required: If 2-3 research passes find no more quality info, insert best effort
- Condensation: Claude decides when it helps, must not lose quality or miss vulnerabilities
- Batch processing with parallelization supported
- Exa integration: Claude decides integration approach

### VulnDocs Orchestrator Agent
- Hierarchical architecture: Main orchestrator + specialized worker agents
- Model tiering:
  - Opus: Categorization, quality control decisions
  - Sonnet: Extraction, validation, test creation, mutation prep (Phase 7 integration)
  - Haiku: Filtering large crawl4ai output for Solidity relevance
- Primary skill: `/vrs-vulndocs` with subcommands/actions or intelligent prompt routing
- Subagent capabilities: Category creation, URL processing, batch extraction, research, doc/pattern improvement
- Pattern test creation compatible with Phase 7.1/7.2 testing infrastructure
- Quality flag approach: Best effort insertion with quality level (draft/ready/excellent)
- Supports `--dry-run` / `--explain` flag to show planned actions
- Auto-commit: Yes, atomic commits per vuln/pattern
- No separate activity log (git history sufficient)

### Claude's Discretion
- Index file structure for optimal LLM context engineering
- When to use 3rd folder level vs keep in parent
- Deduplication similarity thresholds
- Pattern extraction from sources based on quality
- Splitting vs grouping multiple vulns from same URL
- Condensation timing and approach
- Exa integration depth
- Pattern naming specifics

</decisions>

<specifics>
## Specific Ideas

- "Context engineering for LLM and agents - exact information needed for specific tasks in structural, granular, top-down way"
- "Edge cases that share underlying issue should stay together - helps find novel similar vulnerabilities via semantic similarity consensus"
- "Skills and subagents responsible for maintaining vulndocs must be configured to match all these decisions"
- "Pattern tests should be realistic, not just-pass tests - make sense in real world"
- "Lower false positive rate for simple patterns, more permissive for complex semantic reasoning vulnerabilities requiring human-like reasoning"
- "Mutation testing techniques from Phase 7.1/7.2 should integrate with pattern creation"
- "Do not format YAML patterns or content with multiline strings to TOON - quality must be preserved"

</specifics>
