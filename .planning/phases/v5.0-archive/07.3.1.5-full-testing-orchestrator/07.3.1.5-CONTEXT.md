# 07.3.1.5 Full Testing Orchestrator Context

## Mission
Deliver a reusable, production-grade, autonomous E2E testing workflow named **/vrs-full-testing** that drives every AlphaSwarm workflow through Claude Code and claude-code-controller, with self-improving loops, debug-mode visibility, and stage-level resume/skip controls.

## Why This Phase Exists
Phase 7.3 built infrastructure but failed to execute real workflows. This phase (07.3.1.5) makes the E2E workflow **reusable**, **fully orchestrated**, and **self-improving**, so the system can validate itself continuously and expose granular debug data when any subworkflow fails. Real-world E2E runs, metrics aggregation, and GA gating are now part of the full-testing runbook and continuous validation pipeline (7.3.6).

## Non-Negotiables
- /vrs-full-testing is the single entry point to run the entire production workflow test suite.
- All tests are executed via Claude Code sessions using claude-code-controller.
  - **Full claude-code-controller documentation:** See `.planning/testing/rules/claude-code-controller-REFERENCE.md` and `.planning/testing/rules/canonical/claude-code-controller-instructions.md`
- The claude-code-agent-teams session name is fixed: `alphaswarm-claude-code-agent-teams-session`.
- Worktrees are mandatory for isolation and stage snapshotting.
- Every skill and subagent shipped to production has a **debug mode** that captures:
  - prompt/inputs
  - tool calls
  - VKG/BSKG queries
  - outputs and intermediate artifacts
- All workflows must be resumable, stoppable, and iteratively improvable without full reruns.

## Required Capabilities
- Super-orchestrator that schedules and monitors subworkflows (command suite, agent suite, context suite, resume matrix).
- Failure-point detection with auto-triage and targeted re-runs.
- Golden transcript capture and diff for regressions.
- Debug-mode enforcement across all skills/agents.
- Structured metrics export (stage timing, precision/recall, token usage, agreement rates).

## Success Indicators
- /vrs-full-testing can run the entire E2E suite in one invocation and produce evidence packs.
- Failures are automatically classified and produce actionable debug artifacts.
- Any stage can be re-run from a snapshot without re-running earlier stages.
