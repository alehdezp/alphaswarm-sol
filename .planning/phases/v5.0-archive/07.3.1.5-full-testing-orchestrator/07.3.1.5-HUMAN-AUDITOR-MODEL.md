# Human Auditor Behavior Model

> **WARNING (2026-02-03):** The implementation files (`tests/e2e/human_auditor_workflows.yaml`, `scripts/e2e/run_human_workflows.sh`) that were created based on this model have been REMOVED. They referenced non-existent CLI commands (`alphaswarm audit --checkpoint`, `--resume`) and were never validated. This document remains as a conceptual model only. Any future implementation must use ONLY existing CLI commands.
>
> **CLI Status:**
> - ✅ EXISTS: `alphaswarm build-kg`, `alphaswarm query`, `alphaswarm context generate`
> - ❌ ASPIRATIONAL: `alphaswarm audit` (removed), `--checkpoint`, `--resume`, `--debug`

This document models how a human security auditor interacts with the AlphaSwarm.sol tooling during a real audit engagement. The model informs test case design for realistic workflow validation.

---

## Core Auditor Behaviors

Human auditors do NOT run straight through an audit pipeline. They exhibit:

1. **Iterative exploration** - Start broad, narrow based on findings
2. **Hypothesis-driven investigation** - Form suspicions, test them
3. **Context switching** - Pivot between vulnerability types
4. **Interruption handling** - Stop/resume across sessions
5. **Debug-driven refinement** - Request additional output when stuck

---

## Behavior Model: Stop and Resume

### B1: Stop After Graph Build

**Pattern:** Auditor builds the knowledge graph, then pauses to review structure before continuing.

```
Session 1:
  $ alphaswarm build-kg contracts/
  [graph built: .vrs/kg/graph.toon]
  $ exit  # Stop here

Session 2 (hours/days later):
  $ alphaswarm query "functions with external calls"
  [continue investigation]
```

**Test Requirements:**
- Graph file must persist across sessions
- Resume from existing graph without rebuild
- Snapshot checkpoint at post-graph stage

**Workflow ID:** `HAW-STOP-GRAPH`

---

### B2: Stop After Pattern Match

**Pattern:** Auditor runs pattern detection, finds candidates, then stops to research externally.

```
Session 1:
  $ alphaswarm build-kg contracts/ --with-labels
  $ alphaswarm query "pattern:reentrancy-classic"
  [3 candidates found]
  $ exit  # Stop to read docs, check similar CVEs

Session 2:
  $ alphaswarm investigate reentrancy-classic --node-id func_0x3a1
  [continue with specific finding]
```

**Test Requirements:**
- Pattern results must be cached/reproducible
- Investigation can target specific node
- Snapshot checkpoint at post-patterns stage

**Workflow ID:** `HAW-STOP-PATTERNS`

---

### B3: Resume With Context Generation

**Pattern:** Auditor resumes after reading protocol documentation and wants to generate context.

```
Session 1:
  $ alphaswarm build-kg contracts/
  $ exit  # Stop to read protocol docs

Session 2:
  $ alphaswarm context generate contracts/ --doc-path docs/PROTOCOL.md
  [context pack generated with roles, oracles, invariants]
  $ alphaswarm audit contracts/ --use-context
```

**Test Requirements:**
- Context generation works on existing graph
- Context pack integrates with prior graph
- Doc-code conflict detection runs on resume

**Workflow ID:** `HAW-RESUME-CONTEXT`

---

## Behavior Model: Pivot Between Investigations

### B4: Pivot to Another Vulnerability Type

**Pattern:** Auditor starts investigating reentrancy, finds access control issue, pivots.

```
$ alphaswarm investigate reentrancy-classic --scope contracts/Vault.sol
[Finding: withdraw() has external call before state update]
[Note: Also found: setAdmin() has no access control]

$ alphaswarm investigate access-control-missing --scope contracts/Vault.sol
[Pivot to access control investigation]
```

**Test Requirements:**
- Mid-investigation pivot must preserve context
- Both investigations recorded in evidence pack
- Pivot reason logged (manual vs. suggested)

**Workflow ID:** `HAW-PIVOT-VULN`

---

### B5: Pivot to Specific Agent

**Pattern:** Auditor wants attacker-only analysis on a specific function.

```
$ alphaswarm audit contracts/
[Full audit running...]
^C  # Cancel

$ alphaswarm investigate --agent attacker --target contracts/Pool.sol:flashLoan
[Single agent, single function analysis]
```

**Test Requirements:**
- Agent selection respected mid-workflow
- Scope narrowing works from any stage
- Partial results preserved on cancel

**Workflow ID:** `HAW-PIVOT-AGENT`

---

## Behavior Model: Debug Mode Requests

### B6: Request Debug Output on Failure

**Pattern:** Finding doesn't make sense, auditor requests debug traces.

```
$ alphaswarm audit contracts/
[Finding: Critical - reentrancy in safeTransfer]

# Auditor: "That function has a guard, this is wrong"

$ alphaswarm investigate reentrancy-classic --target safeTransfer --debug
[Debug output: VKG queries, semantic ops checked, guard detection failed because...]
```

**Test Requirements:**
- Debug mode reveals full reasoning trace
- VKG queries logged with results
- Guard detection logic exposed
- Debug record saved to evidence/debug/

**Workflow ID:** `HAW-DEBUG-FINDING`

---

### B7: Request Debug on No Findings

**Pattern:** Expected vulnerability not found, auditor wants to know why.

```
$ alphaswarm investigate reentrancy-classic --scope contracts/Vulnerable.sol
[No findings]

# Auditor: "This contract has known reentrancy, why wasn't it found?"

$ alphaswarm investigate reentrancy-classic --scope contracts/Vulnerable.sol --debug
[Debug output:
  - Function analyzed: withdraw()
  - Operation sequence: R:bal -> W:bal -> X:out
  - Pattern match: FAIL (write before external call)
  - Note: Solidity 0.8+ with SafeMath, check-effect pattern detected
]
```

**Test Requirements:**
- Debug explains why pattern did NOT match
- False negative reasoning exposed
- Semantic operation sequence visible

**Workflow ID:** `HAW-DEBUG-MISSING`

---

## Behavior Model: Branch on Findings

### B8: Branch Based on Severity

**Pattern:** Auditor routes workflow based on finding severity.

```
$ alphaswarm audit contracts/
[Summary: 1 Critical, 3 High, 5 Medium]

# If Critical found:
$ alphaswarm debate --finding FIND-001-reentrancy
[Full multi-agent debate on critical]

# If only Medium:
$ alphaswarm verify --finding FIND-004-unchecked-return
[Quick verification only]
```

**Test Requirements:**
- Workflow branching based on audit results
- Different depths for different severities
- Branch decision logged in evidence

**Workflow ID:** `HAW-BRANCH-SEVERITY`

---

### B9: Branch to Manual Review

**Pattern:** Tool output inconclusive, auditor switches to manual review mode.

```
$ alphaswarm investigate logic-flaw --scope contracts/Router.sol
[Finding: Possible logic flaw in swap() - Confidence: 0.45]

# Low confidence triggers manual mode

$ alphaswarm query "show Router.swap full-source"
[Returns annotated source with semantic operations]

# Auditor reviews manually, then confirms/rejects
$ alphaswarm finding update FIND-007 --status confirmed --manual-notes "Verified: division by zero on empty pool"
```

**Test Requirements:**
- Low-confidence findings trigger manual review suggestion
- Full source retrieval with annotations
- Manual finding updates recorded

**Workflow ID:** `HAW-BRANCH-MANUAL`

---

## Behavior Model: Failure Recovery

### B10: Stop on Failure and Request Debug

**Pattern:** Pipeline fails, auditor needs diagnostics.

```
$ alphaswarm audit contracts/
[Error: Slither parser failed on contracts/Complex.sol]

$ alphaswarm audit contracts/ --skip-file contracts/Complex.sol --debug
[Debug: Shows which files succeeded, which failed]

# Or request verbose error
$ alphaswarm tools status --verbose
[Shows tool versions, capabilities, known issues]
```

**Test Requirements:**
- Failure produces actionable error message
- Debug mode on failure shows full trace
- Partial results preserved on failure
- Skip mechanism for problematic files

**Workflow ID:** `HAW-FAIL-DEBUG`

## Checkpoint and Snapshot Requirements

### Required Snapshot Points

For human-like workflows to work, these snapshots MUST be capturable and restorable:

| Checkpoint | Stage | Contents |
|------------|-------|----------|
| SNAP-INIT | After setup | Clean worktree, deps installed |
| SNAP-GRAPH | After build-kg | Graph file, completeness report |
| SNAP-LABELS | After labeling | Label overlay, label stats |
| SNAP-CONTEXT | After context gen | Protocol context pack |
| SNAP-PATTERNS | After pattern match | Match results, candidates |
| SNAP-AGENTS | After agent runs | Agent outputs, intermediate findings |
| SNAP-DEBATE | After debate | Debate transcript, verdicts |

### Snapshot Integrity Rules

1. **Checksum verification** - SHA256 hash of snapshot contents
2. **Dependency tracking** - Snapshot knows its prerequisites
3. **Restore validation** - Post-restore check that state is consistent
4. **Orphan detection** - Warn if snapshot references missing files

## Implementation Requirements

### For /vrs-full-testing

1. **Workflow cases must encode each behavior** - See `tests/e2e/human_auditor_workflows.yaml`
2. **Runner must support stop/resume** - Via worktree snapshots
3. **Debug mode must be toggleable** - Per-workflow and per-command
4. **Evidence pack must capture workflow path** - Which branches taken

### For Evidence Validation

1. **All workflows must produce evidence pack** - No silent execution
2. **Snapshot integrity validated on restore** - Checksum match required
3. **Workflow path logged** - Actual execution sequence recorded
4. **Debug records required when requested** - Missing debug = failure

## Success Criteria

A human-auditor workflow test passes when:

- [ ] Workflow executes in correct stage sequence
- [ ] Stop/resume preserves state correctly
- [ ] Pivot operations maintain context
- [ ] Debug requests produce expected output
- [ ] Branch decisions logged with rationale
- [ ] Evidence pack complete for workflow path
- [ ] Snapshot checksums valid
