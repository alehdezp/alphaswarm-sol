# /vrs-full-testing Reporting Specification

This document defines the reporting format for `/vrs-full-testing` results, including dashboard sections, metrics visualization, and validation markers.

---

## Report Sections

### 1. Gate Status Summary

The gate status summary provides an at-a-glance view of all QA gates.

**Format:**
```markdown
## Gate Status Summary

**Overall Status:** PASS | FAIL | WARN
**Evaluated:** 2026-01-30T22:00:00Z
**Evidence Pack:** evidence-pack-run-20260130-abc123

| Gate | Name | Status | Value | Threshold | Blocking |
|------|------|--------|-------|-----------|----------|
| G01 | Stage Completion | PASS | 4/4 | All passed | Yes |
| G02 | Precision | PASS | 0.92 | >= 0.85 | Yes |
| G03 | Recall | PASS | 0.88 | >= 0.80 | Yes |
| G04 | F1 Score | PASS | 0.90 | >= 0.82 | Yes |
| G05 | Debug Coverage | PASS | 45/45 | 100% | Yes |
| G06 | Tier-C Detection | PASS | 3 TPs | >= 1 | Yes |
| G07 | Economic Context | PASS | 8/10 | >= 1 | Yes |
| G08 | Agent Agreement | WARN | 0.89 | >= 0.90 | No |
| G09 | Transcript Validity | PASS | 95/95 | All valid | Yes |
| G10 | Duration Validity | PASS | All > 5s | > 5000ms | Yes |

**Summary:**
- Passed: 9/10 gates
- Warnings: 1 (G08: Agent Agreement)
- Blocking Failures: 0
```

**Required Fields:**
- `ga_gate_status`: Overall status (PASS | FAIL | WARN)
- `evaluated_at`: ISO 8601 timestamp
- `evidence_pack`: Pack identifier
- `gates_passed`: List of passed gate IDs
- `gates_failed`: List of failed gate IDs
- `gates_warned`: List of warned gate IDs
- `blocking_failures`: Count of blocking failures

### 4. Failure Taxonomy Breakdown

Classification and analysis of test failures.

**Format:**
```markdown
## Failure Taxonomy Breakdown

### By Error Category

| Category | Count | Retry Eligible | Examples |
|----------|-------|----------------|----------|
| Infrastructure | 2 | Yes | E-WORKTREE, E-claude-code-agent-teams |
| Marker Validation | 1 | No | E-NO-END |
| Test Failures | 5 | No | E-DETECTION |
| Resource Errors | 2 | Yes | E-TIMEOUT |
| **TOTAL** | **10** | 4 | - |

### Detailed Failure List

| Scenario | Error Code | Category | Message | Retry |
|----------|------------|----------|---------|-------|
| S18 | E-DETECTION | test | FP: Safe access reported as vuln | No |
| S42 | E-TIMEOUT | resource | Debate timed out after 120s | Yes |
| S93 | E-DETECTION | test | Expected timeout not triggered | No |
| ... | ... | ... | ... | ... |

### Failure Patterns

**Most Common:**
1. E-DETECTION (5 occurrences) - Detection mismatches
2. E-TIMEOUT (2 occurrences) - Timeouts during execution

**Actionable Insights:**
- [ ] Review S18: False positive on safe access pattern
- [ ] Increase timeout for debate-heavy scenarios
- [ ] Verify S93 timeout injection is configured correctly
```

**Required Fields:**
- `failures.by_category`: Map of category -> {count, retry_eligible, error_codes}
- `failures.by_error_code`: Map of error_code -> {count, scenarios}
- `failures.detailed_list`: List of {scenario_id, error_code, category, message, retry_eligible}
- `failures.patterns`: List of {error_code, count, description}
- `failures.actionable_insights`: List of recommended actions

### 5. Perfect Metrics Investigation

Automatic flag when metrics appear suspiciously perfect.

**Format:**
```markdown
## Perfect Metrics Investigation

**Status:** TRIGGERED | NOT_TRIGGERED

### Triggered Checks

| Metric | Value | Threshold | Status |
|--------|-------|-----------|--------|
| Precision | 1.00 | != 1.0 | INVESTIGATE |
| Recall | 1.00 | != 1.0 | INVESTIGATE |
| F1 Score | 1.00 | != 1.0 | INVESTIGATE |

### Investigation Notes

Perfect metrics (precision=1.0 OR recall=1.0) trigger investigation:
- Is the corpus too easy?
- Are negative controls included?
- Is ground truth correctly defined?
- Are edge cases covered?

**Automatic Checks:**
- [ ] Negative controls present: 19/19 included
- [ ] Edge cases present: S80-S84 parser edge cases included
- [ ] Failure injection present: S93-S95 included
- [ ] Ground truth source diversity: DVD=12, SmartBugs=10, SWC=12, Audit=15

**Recommendation:** Manual review required for perfect metrics.
```

**Trigger Conditions:**
- `precision == 1.0`
- `recall == 1.0`
- `f1_score == 1.0`
- `agreement_rate == 1.0`

## Validation Markers

The dashboard output must pass validation against these markers:

### Required Sections
- [ ] Gate Status Summary present
- [ ] Scenario Coverage Heatmap present
- [ ] Debug-Mode Compliance present
- [ ] Failure Taxonomy Breakdown present

### Required Fields (Gate Summary)
- [ ] `ga_gate_status` is one of: PASS, FAIL, WARN
- [ ] `evaluated_at` is ISO 8601 timestamp
- [ ] All 10 gates have entries
- [ ] `blocking_failures` is non-negative integer

### Required Fields (Coverage)
- [ ] `overall_percentage` is 0-100
- [ ] `by_category` has entries for all 14 categories
- [ ] `negative_controls` section present if any exist

### Required Fields (Debug)
- [ ] `debug.enabled` is boolean
- [ ] `compliance_rate` is 0-100
- [ ] Agent and skill tables present

### Required Fields (Failures)
- [ ] `failures.by_category` has standard categories
- [ ] `failures.detailed_list` entries have required fields

## Renderer Implementation

The dashboard renderer (`scripts/e2e/render_full_testing_dashboard.py`) must:

1. **Read Evidence Pack:** Load manifest.json, accuracy.json, agreement.json, debug/index.json
2. **Compute Metrics:** Calculate coverage percentages, compliance rates
3. **Detect Perfect Metrics:** Flag precision/recall == 1.0 for investigation
4. **Generate Gate Table:** Format all 10 gates with status
5. **Build Coverage Heatmap:** Aggregate by category and suite
6. **Compile Failures:** Categorize and list all failures
7. **Compare Baseline:** If baseline exists, compute deltas
8. **Output Formats:** Generate both Markdown and JSON dashboards
9. **Validate Output:** Check against markers before writing
