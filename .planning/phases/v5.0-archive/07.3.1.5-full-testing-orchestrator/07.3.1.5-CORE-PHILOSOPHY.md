# Core Testing Philosophy (Condensed)

## Why Real-World, Human-Operated Testing Is Mandatory
- **The product is a human-facing CLI workflow**; correctness depends on interactive behavior, timing, and operator control.
- **Simulations and mocks miss failure modes**: claude-code-agent-teams sessions, CLI prompts, tool calls, and workflow resumes must be exercised exactly as users do.
- **Evidence-first validation** requires real transcripts, timestamps, and artifacts; fabricated outputs invalidate trust.
- **Security tooling must match human auditor reality**: stop/resume, pivots, partial runs, and recovery paths are core to usage.

## Expected Usage Model (Non-Negotiable)
- **Claude Code acts as the human operator** and drives the CLI in a claude-code-agent-teams session.
- **All workflows and subworkflows must be executed via `claude-code-controller`** in `alphaswarm-claude-code-agent-teams-session`.
  - **Full claude-code-controller documentation:** See `.planning/testing/rules/claude-code-controller-REFERENCE.md` and `.planning/testing/rules/canonical/claude-code-controller-instructions.md`
- **Every run must generate an evidence pack** with transcripts, debug artifacts, and timing metadata.
- **Failures trigger a real retry loop** (no mock reruns), with new worktrees and new evidence packs.

## Strict Reasoning Rule
If a behavior cannot be observed through a real claude-code-agent-teams-driven CLI run, it is **not validated**.
