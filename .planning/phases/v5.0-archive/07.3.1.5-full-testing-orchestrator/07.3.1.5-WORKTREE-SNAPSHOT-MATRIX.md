# 07.3.1.5 Worktree Snapshot Matrix

This document defines the required snapshot checkpoints for each scenario category during E2E testing. Snapshots enable resume/skip capabilities and provide evidence for debugging failed runs.

## Snapshot Stages (Canonical List)

| Stage | Description | Artifacts Captured |
|-------|-------------|-------------------|
| `pre-graph` | Before BSKG construction | Source contracts, config, env |
| `post-graph` | After BSKG construction | `.vrs/graph/*.toon`, build logs |
| `post-context` | After protocol context pack | `.vrs/context/*.yaml`, context evidence |
| `post-patterns` | After pattern matching | `.vrs/patterns/*.yaml`, match results |
| `post-agents` | After specialized agent runs | `.vrs/agents/*.json`, agent outputs |
| `post-debate` | After multi-agent debate | `.vrs/debate/*.json`, verdicts |
| `post-report` | After final report generation | `.vrs/reports/*.md`, metrics |

## Snapshot File Naming Convention

```
{worktree_base}/{run_id}/snapshots/{scenario_id}/{stage}.tar.gz
```

Example:
```
/tmp/alphaswarm-tests/run-20260130-abc123/snapshots/S01/pre-graph.tar.gz
/tmp/alphaswarm-tests/run-20260130-abc123/snapshots/S01/post-graph.tar.gz
/tmp/alphaswarm-tests/run-20260130-abc123/snapshots/S01/post-context.tar.gz
...
```

## Validation Rules

1. **Completeness Check:** For each scenario, all required stages must have snapshots.
2. **Ordering Check:** Snapshots must be created in stage order.
3. **Freshness Check:** Snapshot timestamp must be within the run window.
4. **Integrity Check:** Checksum must match recorded value.

---

## Resume Protocol

When resuming from a snapshot:

1. Locate the latest complete snapshot for the target stage.
2. Extract the snapshot to the worktree.
3. Verify artifact integrity.
4. Continue execution from the next stage.

Example resume command:
```bash
./scripts/e2e/full_testing_runner.sh --resume run-20260130-abc123 --scenario S01 --stage post-context
```
