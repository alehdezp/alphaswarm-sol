# 07.3.1.5 Plan Critique (Hard Review)

This document identifies concrete weaknesses in the /vrs-full-testing design and proposes strict improvements with failure conditions.

---

## Critical Weaknesses Identified

### W1: Coverage Underspecification (Severity: CRITICAL)

**Problem:** The plan does not enumerate all production skills and agents, nor map them explicitly to scenarios. This creates risk of silent regression where untested skills/agents break without detection.

**Evidence:**
- Skill registry lists 47 skills across 9 categories
- Agent catalog lists 24 agents (21 shipped, 3 dev-only)
- Scenario matrix (72 scenarios) does not explicitly map to each skill/agent

**Required Fix:**
- SKILL-COVERAGE.md must list EVERY shipped skill and agent
- Each must have at least one scenario ID that exercises it
- Coverage validator must reject runs missing any skill/agent coverage

**Failure Condition:** If any shipped skill or agent has 0 scenario coverage, /vrs-full-testing MUST fail with E011 (INCOMPLETE_SKILL_COVERAGE).

---

### W2: Scenario Diversity Insufficient for Edge Cases (Severity: HIGH)

**Problem:** Current 72 scenarios focus on vulnerability detection but lack:
- Failure injection scenarios (forced timeouts, corrupt graphs, missing files)
- Negative control coverage (safe contracts that MUST NOT trigger findings)
- Parser stress tests (deeply nested, large files, unusual syntax)
- Recovery path validation (resume from each failure state)

**Evidence:**
- Only 2 scenarios (S71, S72) cover resume/recovery
- Only ~15 explicit negative controls (safe baselines)
- Zero explicit failure injection beyond S72

**Required Fix:**
- Add 10+ failure injection scenarios (S73-S82)
- Add 8+ parser edge case scenarios (S62-S69 expanded)
- Add explicit negative controls for each vulnerability category
- Require 60+ unique scenarios minimum (current 72 is acceptable, but diversity gaps remain)

**Failure Condition:** If negative control precision < 95% (false positive rate > 5%), the run MUST fail with E012 (EXCESSIVE_FALSE_POSITIVES).

---

### W3: claude-code-controller Enforcement Not Guaranteed (Severity: CRITICAL)

**Problem:** The plan mentions claude-code-agent-teams but lacks a hard gate that rejects runs not using claude-code-controller. Manual subprocess invocations could bypass isolation and produce unreliable results.

**Full claude-code-controller documentation:** See `.planning/testing/rules/claude-code-controller-REFERENCE.md` and `.planning/testing/rules/canonical/claude-code-controller-instructions.md`

**Evidence:**
- Orchestrator spec states "All test execution MUST use claude-code-controller"
- But no runtime validator enforces this

**Required Fix:**
- All commands must be wrapped with ALPHASWARM-START/END markers
- Evidence pack validator must check:
  - All transcripts have markers
  - No transcript contains subprocess spawn patterns without claude-code-agent-teams context
  - All durations > 5000ms
- Add explicit claude-code-agent-teams session creation verification step

**Failure Condition:** Any transcript missing markers or any duration < 5000ms MUST invalidate the run with E003/E004/E005.

---

### W4: Debug-Mode Enforcement Aspirational (Severity: HIGH)

**Problem:** Debug mode is proposed but not enforced across every production skill and agent. This risks investigating failures without adequate debug artifacts.

**Evidence:**
- DEBUG-MODE-SPEC.md defines schema but no enforcement mechanism
- Skills/agents may silently skip debug output without failing the run

**Required Fix:**
- Every skill invocation must set DEBUG_MODE=true
- Every agent invocation must produce a debug record in evidence/debug/
- Debug record validator must check:
  - Record exists for every skill/agent invoked
  - Required fields present (inputs, tool_calls, outputs)
  - Timestamps are realistic (not zero, not future)

**Failure Condition:** If any invoked skill/agent lacks a debug record, run MUST fail with E013 (MISSING_DEBUG_RECORD).

---

### W5: Golden Transcript Strategy Lacks Strictness (Severity: MEDIUM)

**Problem:** Golden transcript capture is mentioned but lacks:
- Diff tolerance rules (what variance is acceptable?)
- Deterministic capture rules (timestamps, order-sensitive output)
- Regression detection thresholds

**Evidence:**
- No specification of acceptable diff types (timing vs. content)
- No golden transcript baseline management strategy

**Required Fix:**
- Define golden transcript format with redacted timestamps/UUIDs
- Specify diff tolerance: content changes = failure, timing variance = warning
- Require golden transcript baseline for GA release
- Add golden transcript comparison step in evidence pack validation

**Failure Condition:** Content diff from golden transcript MUST fail with E014 (GOLDEN_TRANSCRIPT_MISMATCH).

---

### W6: Worktree Snapshot Strategy Incomplete (Severity: MEDIUM)

**Problem:** Snapshot strategy is defined at suite level but lacks:
- Per-stage snapshot points (pre-graph, post-graph, post-context, etc.)
- Snapshot reuse policy (when can we skip stages?)
- Snapshot integrity verification

**Evidence:**
- Only 4 snapshots (S1-S4 suites)
- No per-scenario or per-stage snapshots
- No checksum verification on restore

**Required Fix:**
- Define 7 snapshot points per scenario: init, graph, labels, context, patterns, agents, debate
- Add checksum verification on all snapshot operations
- Define snapshot reuse policy: reuse allowed only if source files unchanged

**Failure Condition:** Snapshot checksum mismatch on restore MUST fail with E015 (SNAPSHOT_CORRUPTION).

---

### W7: External Ground Truth Sources Not Explicit (Severity: HIGH)

**Problem:** Scenarios reference vulnerabilities but do not explicitly link to external ground truth sources (DVD, SmartBugs, audit reports).

**Evidence:**
- Scenario matrix lists scenarios but ground_truth_source field is missing
- Evidence pack schema has ground-truth/ directory but no source provenance

**Required Fix:**
- Every scenario must have explicit ground_truth_source in SCENARIO-MATRIX.md
- Valid sources: DVD (Damn Vulnerable DeFi), SmartBugs, SWC-Registry, Published Audit
- Evidence pack must log source URL, access date, and hash
- Invalid ground truth source MUST block run start

**Failure Condition:** Scenario with missing/invalid ground_truth_source MUST fail with E007 (GROUND_TRUTH_MISSING).

---

### W8: Self-Improvement Loop Undefined (Severity: MEDIUM)

**Problem:** Plan mentions "self-improving loops" but lacks concrete specification of:
- Failure taxonomy (what failure types exist?)
- Auto-retry conditions (when to retry vs. abort?)
- Test refinement triggers (when to update tests?)

**Evidence:**
- Retry policy exists in ORCHESTRATOR-SPEC but failure taxonomy is implicit
- No auto-refinement loop specification

**Required Fix:**
- Define explicit failure taxonomy:
  - INFRA_FAILURE (claude-code-agent-teams crash, worktree creation fail)
  - TIMEOUT_FAILURE (command exceeded limit)
  - DETECTION_FAILURE (expected vs. actual mismatch)
  - EVIDENCE_FAILURE (missing artifacts)
  - ASSERTION_FAILURE (logic error in test)
- Define retry policy per failure type (see retry_policy in orchestrator spec)
- Define refinement triggers: after 3 consecutive DETECTION_FAILURE on same pattern, flag for human review

**Failure Condition:** After max_total_retries (10) exceeded, MUST abort with E009 and postmortem.

## Non-Negotiable Success Criteria (v2)

All of the following MUST be true for /vrs-full-testing to pass:

| # | Criterion | Validation |
|---|-----------|------------|
| N1 | All 47 shipped skills have at least 1 scenario coverage | SKILL-COVERAGE.md mapping exists, runtime check |
| N2 | All 21 shipped agents have at least 1 scenario coverage | SKILL-COVERAGE.md mapping exists, runtime check |
| N3 | 60+ scenarios executed with valid transcripts | Evidence pack scenario count >= 60 |
| N4 | All transcripts have ALPHASWARM markers | E003/E004 validation |
| N5 | All durations > 5000ms | E005 validation |
| N6 | All skills/agents produce debug records | E013 validation |
| N7 | Negative control precision >= 95% | accuracy.json FP rate check |
| N8 | Overall precision >= 85% | accuracy.json check |
| N9 | Overall recall >= 80% | accuracy.json check |
| N10 | All ground truth sources valid | E007 validation |

## Stop-the-Line Conditions

These conditions trigger immediate abort of /vrs-full-testing:

| Condition | Trigger | Action |
|-----------|---------|--------|
| Repeated infra failure | 3+ consecutive INFRA_FAILURE | Abort, create postmortem |
| Missing ground truth | Any scenario missing ground_truth_source | Block run start |
| claude-code-agent-teams session crash | claude-code-agent-teams session exits unexpectedly | Abort, preserve worktree |
| Evidence pack corruption | Manifest unwritable or unreadable | Abort, preserve artifacts |
| Skill/agent coverage gap | Any shipped skill/agent has 0 scenarios | Block run start |
| Budget exceeded | Token usage exceeds 500k per run | Abort, flag for review |
