### Test Script Features (test_stage_context.sh)

1. **Worktree creation**: Creates isolated worktree from foundry-vault fixture
2. **Graph prerequisite**: Builds knowledge graph first (required for context)
3. **claude-code-controller integration**: Launches Claude Code session via claude-code-controller
4. **Context generation**: Requests protocol context pack generation with Exa search
5. **Validation**: Validates protocol-pack.yaml exists and contains expected fields
6. **Exa search check**: Verifies transcript for Exa/web search invocation

### Validation Script Features (validate_context.py)

- **Protocol type detection**: Checks for vault/custody/defi type
- **Role extraction**: Validates owner/user roles present
- **Structure validation**: Verifies required YAML fields
- **Economic context**: Checks for optional economic model
- **Expected checks**: Validates against context-checks.json ground truth
- **JSON output**: Produces validation results for CI integration

### Expected Checks (context-checks.json)

- Protocol type: vault, custody, storage, defi
- Required roles: owner, user
- Trust assumptions documented
- Optional oracles and similar exploits fields
- Ground truth for validation comparison

## Decisions Made

| Decision | Rationale |
|----------|-----------|
| Graph build as prerequisite | Context generation requires BSKG data for accurate protocol detection |
| Flexible field name matching | Context pack may use different keys (type vs protocol_type) |
| Exa check as SHOULD | Exa may not always be invoked (cached data, offline) |
| Separate validation script | Reusable for CI/CD and other stage tests |
