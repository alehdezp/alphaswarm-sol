**Source:** Damn Vulnerable DeFi - Side Entrance Challenge
**URL:** https://www.damnvulnerabledefi.xyz/challenges/side-entrance/
**Official Solution:** https://github.com/tinchoabbate/damn-vulnerable-defi/blob/master/test/side-entrance/side-entrance.challenge.js

```yaml
# From tests/fixtures/dvd/side-entrance/ground-truth.yaml
contract: SideEntranceLenderPool
file: src/SideEntranceLenderPool.sol
source: damn-vulnerable-defi
challenge_url: https://www.damnvulnerabledefi.xyz/challenges/side-entrance/

vulnerabilities:
  - id: DVD-SE-001
    type: accounting-manipulation
    severity: critical
    function: flashLoan
    description: |
      The flash loan implementation only checks that the pool's ETH balance
      doesn't decrease. However, the deposit() function can be called during
      the flash loan callback, allowing the borrower to:
      1. Borrow X ETH via flashLoan()
      2. In execute() callback, call deposit() with the borrowed X ETH
      3. This credits X to borrower's balance AND returns X to pool
      4. After callback, pool balance is same, check passes
      5. Borrower can now withdraw() the credited X ETH

  - id: DVD-SE-002
    type: reentrancy
    severity: high
    function: withdraw
    description: |
      The withdraw function has potential reentrancy risk with external call.
```

## Execution Script

**File:** `scripts/run_dvd_side_entrance_test.sh`

```bash
#!/bin/bash
# Run Side Entrance audit test

set -e

SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
MAIN_REPO="$(dirname "$SCRIPT_DIR")"
WORKTREE_NAME="dvd-side-entrance"
WORKTREE_PATH="/tmp/vrs-worktrees/$WORKTREE_NAME"
TIMEOUT=600

echo "========================================"
echo " DVD SIDE ENTRANCE TEST"
echo "========================================"

# Step 1: Create worktree
echo "[1/6] Creating worktree from fixture..."
"$SCRIPT_DIR/manage_worktrees.sh" create-from-fixture "$WORKTREE_NAME" "dvd/side-entrance"

# Step 2: Launch Claude Code
echo "[2/6] Launching Claude Code session..."
PANE_ID=$(claude-code-controller launch "cd $WORKTREE_PATH && claude --dangerously-skip-permissions" 2>&1 | grep -oP 'pane_id:\s*\K\S+' || echo "")

if [ -z "$PANE_ID" ]; then
    echo "ERROR: Failed to launch session"
    exit 1
fi

sleep 15

# Step 3: Send audit command
echo "[3/6] Sending /vrs-audit command..."
claude-code-controller send "/vrs-audit contracts/ --mode swarm" --pane="$PANE_ID"

# Step 4: Wait for completion
echo "[4/6] Waiting for completion (timeout: ${TIMEOUT}s)..."
START_TIME=$(date +%s)
claude-code-controller wait_idle --pane="$PANE_ID" --idle-time=60 --timeout="$TIMEOUT" || true
END_TIME=$(date +%s)
DURATION=$((END_TIME - START_TIME))

# Step 5: Capture and validate
echo "[5/6] Capturing output..."
claude-code-controller capture --pane="$PANE_ID" --lines=5000 > "$WORKTREE_PATH/audit-transcript.txt"

echo "[6/6] Validating..."
"$SCRIPT_DIR/manage_worktrees.sh" validate "$WORKTREE_NAME" all

# Compare to ground truth
uv run python "$SCRIPT_DIR/compare_findings.py" \
    --worktree "$WORKTREE_PATH" \
    --output "$WORKTREE_PATH/metrics.json"

# Display results
echo ""
echo "========================================"
echo " RESULTS"
echo "========================================"
echo "Duration: ${DURATION}s"
cat "$WORKTREE_PATH/metrics.json"

# Cleanup
claude-code-controller kill --pane="$PANE_ID" 2>/dev/null || true

# Check success (at least critical vulnerability detected)
TP=$(jq -r '.true_positives' "$WORKTREE_PATH/metrics.json")
if [ "$TP" -ge 1 ]; then
    echo ""
    echo "SUCCESS: Critical vulnerability detected (TP = $TP)"
    exit 0
else
    echo ""
    echo "FAILURE: Critical vulnerability missed (TP = $TP)"
    exit 1
fi
```

## Notes

1. **Ground Truth Integrity:** The vulnerability descriptions come directly from Damn Vulnerable DeFi, not from our tool's detection. This ensures we're measuring real detection capability.

2. **Complexity of DVD-SE-001:** This is a complex accounting vulnerability that requires understanding:
   - Flash loan mechanics (borrow, callback, repay)
   - How `deposit()` credits balance
   - How `flashLoan()` only checks total balance, not accounting
   - The interaction between these functions during callback

3. **Detection Challenge:** This vulnerability may be harder to detect than reentrancy because:
   - No direct external call before state update
   - Requires understanding cross-function interaction
   - Needs to recognize that deposit() can be called during callback
   - May require Tier B/C pattern matching or agent reasoning

4. **Expected Detection Path:**
   - Graph should show `flashLoan` calls external (receiver.execute)
   - Graph should show `deposit` writes balance
   - Logic specialist should identify balance manipulation potential
   - Attacker agent should construct the exploit path
   - Verifier should confirm based on lack of callback isolation

5. **Multi-Agent Debate:** In swarm mode:
   - Attacker: Constructs exploit (borrow -> deposit in callback -> withdraw)
   - Defender: Looks for callback isolation (none exists)
   - Verifier: Confirms critical - no isolation between flash loan and deposit

6. **Flexible Workflow Reference:** See `FLEXIBLE-WORKFLOW-DESIGN.md` for all available stage control, pattern control, and workflow control commands.
