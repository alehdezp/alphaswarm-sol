## Evidence Requirements
- Dedicated demo label `vrs-demo-{workflow}-{timestamp}` for each scenario.
- Minimum transcript thresholds (RULES-ESSENTIAL F1/F3).
- Evidence packs include proof tokens and stage markers.

## Reasoning
The original plan risked testing each workflow on a single repo, which would under-sample variability. This plan enforces multi-repo coverage per workflow and adds a mini-stress subset to raise test size without breaking the 3-scenario requirement.

## Purpose
- Execute workflow tests on synthetic repos.
- Capture transcripts and evidence for auditability.
- Increase test size via mini-stress subset.
- Update alignment ledger for traceability.

## Why
- Workflow flexibility and branching must be demonstrated with real transcripts.
- Evidence packs prove orchestration actually occurs.
- Multi-repo coverage reduces false confidence.

## How To Implement

### T1: Build scenario manifest
**Goal:** Encode 3 scenarios per workflow with repo and checkpoint revision mapping.

### T2: Implement claude-code-agent-teams runner for scenario matrix
**Goal:** Automate claude-code-controller execution with transcript capture using jj workspaces.

### T3: Execute scenario matrix and collect evidence
**Goal:** Produce transcripts and evidence packs for each scenario.

### T4: Execute mini-stress subset
**Goal:** Run expanded subset (>=5 repos x 5 workflows) using parallel jj workspaces.

### T5: Evaluate results and update alignment ledger
**Goal:** Publish validation report and evidence references.

### T6: Tooling update
**Goal:** Document scenario matrix runner in TOOLING.md.
