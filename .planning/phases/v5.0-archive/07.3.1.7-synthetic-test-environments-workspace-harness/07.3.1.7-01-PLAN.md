## Reasoning
The original plan focused only on compilation, which is insufficient. A repo can compile but still fail to build a usable graph or lack meaningful vulnerability diversity. This plan adds quality gates and graph viability checks to ensure the dataset is actually fit for workflow validation.

## Purpose
- Produce a consistent synthetic dataset.
- Guarantee each repo is buildable, graph-buildable, and structurally valid.
- Enforce minimum vulnerability diversity and decoy coverage.

## Why
- Repos without checkpoints cannot support rollback testing.
- Unbuildable repos waste claude-code-agent-teams cycles and invalidate workflow validation.
- Graph-build failures would undermine graph-first workflows.
- Diversity and decoys are required to test false positives and robustness.
