# Phase 07.1.4 Plan 01: Adapter Interface Foundation Summary

**One-liner:** Created OrchestratorAdapter base class with capability detection, handoff semantics, and evidence preservation for external framework interoperability.

## What Was Built

This plan established the adapter interface foundation for running AlphaSwarm VRS workflows across external orchestration frameworks (Agents SDK, LangGraph, AutoGen, CrewAI).

### Core Components

1. **OrchestratorAdapter Base Class** (`base.py`, ~420 LOC)
   - Abstract base class with execute_agent, handoff, spawn_with_trace methods
   - AdapterConfig with capability detection and trace propagation modes
   - HandoffContext for agent-to-agent transfers with evidence preservation
   - TraceContext for distributed tracing across framework boundaries
   - HandoffResult for validation of successful handoffs
   - Evidence preservation validation methods (preserve_evidence, validate_evidence_preserved)

2. **Capability Matrix** (`capability.py`, ~280 LOC)
   - AdapterCapability enum with 14 capability types (tool, memory, trace, VRS features)
   - CapabilityMatrix class with supports/supports_all/supports_any/missing_for methods
   - ADAPTER_CAPABILITIES constant defining 5 framework capabilities
   - get_capability_matrix() helper for runtime adapter selection
   - check_evidence_requirements() for evidence preservation warnings

3. **Reference Documentation** (`adapter-capabilities.md`, ~230 lines)
   - Overview of adapter system and interoperability goals
   - Capability definitions table
   - Adapter comparison matrix (agents-sdk, langgraph, autogen, crewai, beads-gastown)
   - Evidence preservation mechanisms and validation examples
   - Trace propagation modes (header, context, none)
   - Usage examples for capability checking and agent execution
   - Agentwise/Agentrooms integration notes for future multi-agent marketplace work

### Evidence Preservation

Evidence contracts (VulnerabilityBead) remain unchanged during framework transitions through:
- **Evidence snapshots** captured in HandoffContext before transfer
- **Validation methods** that compare critical fields (bead ID, vulnerability class, severity, operations)
- **Evidence modes** (inline, bead, external) for different framework requirements

Adapters lacking BEAD_REPLAY or GRAPH_FIRST capabilities receive warnings from `check_evidence_requirements()`.

## Decisions Made

### D-07.1.4-01-01: Adapter base class with abstract methods

**Context:** Need unified interface for external orchestration frameworks.

**Decision:** Use abstract base class (ABC) with abstractmethod decorators for execute_agent, handoff, spawn_with_trace, export_trace, import_trace methods.

**Rationale:**
- Enforces implementation contract at class definition time
- Clear separation between adapter-agnostic logic (evidence preservation) and adapter-specific logic (SDK calls)
- IDE/LSP support for abstract method detection

**Alternatives considered:**
- Protocol-based interface (typing.Protocol) - Less strict, no runtime enforcement
- Duck typing - No type safety, harder to discover required methods

**Impact:** All adapter implementations (Phase 07.1.4-02 to 07.1.4-05) must implement these methods.

### D-07.1.4-01-02: Evidence preservation through snapshots

**Context:** VulnerabilityBead evidence contracts must survive framework boundaries unchanged.

**Decision:** Capture evidence snapshot in HandoffContext before handoff, validate after handoff with `validate_evidence_preserved()`.

**Rationale:**
- Snapshot provides reference point for validation
- Critical fields (bead ID, vulnerability class, severity, operations) are compared
- Adapters can implement additional validation for framework-specific concerns

**Alternatives considered:**
- Deep copy comparison - More expensive, harder to debug differences
- Hash-based validation - Sensitive to field ordering, opaque errors

**Impact:** All handoffs must call `preserve_evidence()` before transfer and validate after.

### D-07.1.4-01-03: Capability-based adapter selection

**Context:** Different frameworks have different feature sets (tool execution, memory persistence, trace propagation).

**Decision:** Define AdapterCapability enum with 14 capability types and CapabilityMatrix for runtime detection.

**Rationale:**
- Enables feature-based adapter selection (e.g., "need GUARDRAILS + COST_TRACKING")
- Explicit capability requirements in workflow definitions
- Warnings for missing evidence-critical capabilities (BEAD_REPLAY, GRAPH_FIRST)

**Alternatives considered:**
- Static adapter configuration - Less flexible, no runtime checks
- Runtime feature probing - Expensive, non-deterministic

**Impact:** Workflows can declare required capabilities and fail fast if adapter doesn't support them.
