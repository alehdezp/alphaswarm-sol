# Phase 07.1.4 Plan 05: Beads/Gas Town and Claude Code Adapters Summary

**One-liner:** Git-backed orchestration adapter with bead replay and local Claude Code CLI adapter with unified adapter registry for capability-based discovery

## What Was Built

This plan implemented git-backed orchestration with the Beads/Gas Town adapter, local CLI execution with the Claude Code adapter, and a unified adapter registry for capability-based adapter discovery.

### Core Components

**1. BeadsGasTownAdapter (beads_gastown.py, 579 LOC)**
- `BeadsGasTownConfig`: Git-specific configuration with branch prefix, auto-commit, worktree settings
- `GitBackedBead`: Version-controlled bead storage with:
  - `save()`: Commit bead state to git and return commit hash
  - `load(commit_hash)`: Load bead from specific commit
  - `get_history()`: Get commit history for bead
  - `replay_to(commit_hash)`: Replay bead to specific state
- `BeadsGasTownAdapter`: Git-backed orchestration with:
  - Git branch per bead investigation (bead/{bead_id})
  - Deterministic commit histories for audit trail
  - Replay capability via commit hash
  - Worktree integration for parallel execution
  - Evidence preservation through git commits
  - Capabilities: BEAD_REPLAY, GRAPH_FIRST, MEMORY_PERSISTENT

**2. ClaudeCodeAdapter (claude_code.py, 477 LOC)**
- `ClaudeCodeConfig`: CLI-specific configuration with claude path, model, timeout
- `ClaudeCodeAdapter`: Local CLI execution with:
  - Subprocess execution of claude CLI
  - Worktree isolation for parallel agents
  - System prompt injection from agent configs
  - Trace propagation via environment variables
  - Bead context written to worktree YAML
  - GRAPH_FIRST enforcement via CLAUDE.md
  - Capabilities: TOOL_EXECUTION, GRAPH_FIRST

**3. AdapterRegistry (registry.py, 345 LOC)**
- `AdapterRegistry` class with:
  - `register(name, adapter_class)`: Register adapter class
  - `get(name, config)`: Get adapter instance (cached or new)
  - `list_adapters()`: List all registered adapters
  - `get_capability_comparison()`: Generate capability matrix
  - `find_adapter_for_capabilities(required)`: Capability-based search
  - `clear_cache()`: Clear instance cache
- Module-level singleton `_registry` with convenience functions:
  - `get_adapter(name, config)`
  - `register_adapter(name, adapter_class)`
  - `list_adapters()`
  - `find_adapters_with_capabilities(required)`
- Auto-registration of 7 built-in adapters:
  - agents-sdk, codex-mcp, beads-gastown, claude-code
  - langgraph (if available), autogen (if available), crewai (if available)

**4. Unit Tests (test_beads_gastown.py, 335 LOC)**
- 14 comprehensive tests covering:
  - GitBackedBead save/load/history/replay operations
  - BeadsGasTownAdapter initialization and handoff
  - Evidence preservation via git commits
  - AdapterRegistry registration and retrieval
  - Capability-based adapter search
  - Capability comparison matrix generation
- Pytest fixtures for temp git repo and sample bead
- Git CLI availability checks with skipif markers

## Technical Decisions Made

### 1. Git-Backed Bead State Management
**Decision:** Use git branches with commit tracking for bead state

**Context:** Need deterministic bead history with replay capability

**Impact:**
- ✅ Full audit trail of all state changes
- ✅ Deterministic replay to any commit
- ✅ Git-friendly diffs for human review
- ✅ Parallel execution via worktrees
- ⚠️ Requires git CLI available

**Alternatives considered:**
- Database storage: Less transparent, requires migration
- File-based versioning: Manual implementation, no audit trail

### 2. CLI Subprocess Execution for Claude Code
**Decision:** Execute claude CLI as subprocess with worktree isolation

**Context:** Need local execution without SDK dependencies

**Impact:**
- ✅ No SDK version coupling
- ✅ Works with any claude CLI version
- ✅ Worktree isolation for parallel agents
- ⚠️ Slower than direct API calls
- ⚠️ Output parsing required

**Alternatives considered:**
- HTTP API calls: More complex authentication
- Direct SDK integration: Version coupling, dependency bloat

### 3. Registry Singleton with Instance Caching
**Decision:** Module-level singleton registry with lazy instance creation and caching

**Context:** Need global adapter access across codebase

**Impact:**
- ✅ Simple access pattern (get_adapter)
- ✅ Performance via instance caching
- ✅ No configuration duplication
- ⚠️ Global state (testing requires cache clear)

**Alternatives considered:**
- Factory pattern: More verbose, no caching
- Dependency injection: Requires framework, complex setup

## Known Limitations

1. **Git CLI Dependency:** BeadsGasTownAdapter requires git CLI available
   - Tests use `pytest.mark.skipif` for git availability checks
   - Consider git library (pygit2, GitPython) for future versions

2. **Claude CLI Dependency:** ClaudeCodeAdapter requires claude CLI installed
   - Graceful failure with clear error message if not available
   - No runtime check during import (only during instantiation)

3. **Placeholder Agent Execution:** Both adapters have placeholder execute_agent implementations
   - Full implementation requires agent executor integration
   - Consider adding in Phase 07.3 for GA validation

4. **No Concurrent Git Operations:** BeadsGasTownAdapter doesn't handle concurrent git writes
   - Worktrees isolate file writes but not git metadata writes
   - Consider git locking for production use

## Lessons Learned

1. **Git as State Backend:** Git commits provide excellent audit trail and replay capability
   - Append-only nature similar to event sourcing
   - Human-readable diffs for debugging
   - Deterministic history for test reproducibility

2. **Registry Singleton Pattern:** Module-level singleton provides clean API
   - Auto-registration on import simplifies usage
   - Instance caching improves performance
   - Clear cache method essential for testing

3. **Capability-Based Discovery:** Feature matrix enables declarative adapter selection
   - Workflows can specify required capabilities
   - Registry finds compatible adapters automatically
   - Better than manual adapter selection

4. **CLI Subprocess Pattern:** Subprocess execution decouples from SDK versions
   - Works with any CLI-compatible tool
   - Output parsing adds complexity but provides flexibility
   - Environment variable trace propagation simpler than headers
