## Summary

Phase 5.5 implements a **context-merge orchestration pattern** where specialized agents receive vulnerability-specific reasoning templates merged with protocol economic context. This research identifies the standard stack and architectural patterns for implementing fresh-context agent spawning, quality verification loops, and bead-based task tracking.

**Core approach:** Template-based injection (vulndoc reasoning + protocol context) → Quality verification → Bead creation → Vuln-discovery with full evidence chain.

**Key findings:**
- Multi-agent orchestration in 2026 follows **hierarchical delegation** patterns with supervisor-coordinator-worker layers
- Claude Code CLI subagents **cannot spawn nested subagents** (architectural limitation requires main orchestrator coordination)
- **Beads pattern** (steveyegge/beads) provides Git-backed DAG task management ideal for multi-agent workflows
- Context injection uses **system prompt for methodology** + **user prompt for protocol data** to optimize reasoning
- Token efficiency achieved via **prompt caching** (90% reduction), **model routing** (64% cost savings), and **context compression**

**Primary recommendation:** Implement Sub-coordinator pattern where Opus-4.5 orchestrator spawns parallel Sonnet-4.5 context-merge agents, each creating verified beads for downstream vuln-discovery agents. Use Template+Protocol context layering with quality verification loops.

### Pattern 1: Hierarchical Delegation with Verification Loop
**What:** Main orchestrator delegates context preparation to sub-coordinator, which spawns parallel context-merge agents with quality verification before vuln-discovery.

**When to use:** When context preparation is complex and failure-prone, requiring quality gates before downstream work.

**Example:**
```python
# Source: Multi-agent orchestration patterns 2026
# https://learn.microsoft.com/en-us/azure/architecture/ai-ml/guide/ai-agent-design-patterns

class SubCoordinator:
    """Spawns parallel context-merge agents with verification loop."""

    def coordinate_context_merge(self, protocol_pack, vuln_classes):
        """Parallel context-merge with quality verification."""
        tasks = []
        for vuln_class in vuln_classes:
            # Extract vulndoc reasoning template
            vulndoc = self.load_vulndoc(vuln_class)

            # Spawn context-merge agent (Sonnet 4.5)
            task = self.spawn_context_merge(
                protocol_pack=protocol_pack,
                vulndoc=vulndoc,
                vuln_class=vuln_class
            )
            tasks.append(task)

        # Wait for all context-merge agents
        merged_contexts = await asyncio.gather(*tasks)

        # Verify each merged context (Sonnet 4.5)
        verified_beads = []
        for merged_ctx in merged_contexts:
            verification = self.verify_context_merge(merged_ctx)

            if verification.passed:
                # Create bead via skill
                bead = self.create_bead_via_skill(
                    skill="vrs:create-bead-context-merge",
                    context=merged_ctx
                )
                verified_beads.append(bead)
            else:
                # Loop back to context-merge with errors
                self.retry_context_merge(
                    merged_ctx,
                    errors=verification.errors
                )

        return verified_beads
```

### Pattern 2: Template-Based Context Injection
**What:** System prompt contains reasoning methodology (HOW to think), user prompt contains protocol data (WHAT to analyze).

**When to use:** Always for vuln-discovery agents - separates methodology from data for optimal reasoning.

**Example:**
```python
# Source: LLM system vs user prompt best practices
# https://www.nebuly.com/blog/llm-system-prompt-vs-user-prompt

def build_vuln_discovery_prompt(vulndoc, protocol_pack, target_contracts):
    """Build prompts with methodology/data separation."""

    # System prompt = Reasoning template (HOW to think)
    system_prompt = f"""
You are a security researcher investigating {vulndoc.category}/{vulndoc.subcategory}.

# Detection Methodology
{vulndoc.reasoning_template}

# Anti-Patterns (NOT vulnerable if...)
{vulndoc.anti_patterns}

# Semantic Operations to Query
{vulndoc.semantic_triggers}

Use BSKG queries to analyze code behavior, not manual reading.
Flag uncertain cases with confidence=uncertain.
"""

    # User prompt = Protocol context (WHAT to analyze)
    user_prompt = f"""
# Protocol Context
{protocol_pack.roles}
{protocol_pack.assumptions}
{protocol_pack.risk_categories}

# Target Contracts
{target_contracts}

# Task
Investigate {vulndoc.subcategory} vulnerabilities in the target contracts.
Create finding beads via /vrs-create-bead-finding for each discovery.
"""

    return system_prompt, user_prompt
```

### Pattern 4: Quality Verification Loop
**What:** Context-merge verifier checks completeness and quality, sending errors back to merge agent if quality fails.

**When to use:** Before creating beads from complex agent outputs to prevent low-quality context pollution.

**Example:**
```python
# Source: Agent orchestration verification patterns
# https://www.getdynamiq.ai/post/agent-orchestration-patterns-in-multi-agent-systems

class ContextMergeVerifier:
    """Verifies context-merge output quality."""

    def verify(self, merged_context):
        """Run quality checks on merged context."""
        errors = []

        # Check 1: Reasoning template present and semantic
        if not merged_context.reasoning_template:
            errors.append("Missing reasoning_template")
        elif "SELECT" in merged_context.reasoning_template:
            errors.append("Template contains VQL, should be semantic")

        # Check 2: Protocol context present
        if not merged_context.protocol_context:
            errors.append("Missing protocol_context")

        # Check 3: Semantic triggers valid
        valid_operations = load_semantic_operations()
        for trigger in merged_context.semantic_triggers:
            if trigger not in valid_operations:
                errors.append(f"Invalid semantic trigger: {trigger}")

        # Check 4: No contradictions
        if self.detect_contradictions(merged_context):
            errors.append("Contradictory reasoning steps detected")

        # Return verification result
        if errors:
            return VerificationResult(
                passed=False,
                errors=errors,
                action="retry_with_errors"
            )
        else:
            return VerificationResult(
                passed=True,
                action="create_bead"
            )
```

### Anti-Patterns to Avoid
- **Nested subagent spawning:** Claude Code subagents cannot spawn subagents. Sub-coordinator must live in main context.
- **Shared mutable state:** Beads must be self-contained. No shared global state between agents.
- **VQL in reasoning templates:** Templates should describe semantic methodology, not SQL-like queries. Agents formulate queries themselves.
- **Unbounded token context:** Context trimming must drop exploits/examples first, preserve reasoning template and protocol context.
- **Direct LLM API billing:** Use subscription-based CLIs (Claude Code, Codex) for main workflows to avoid per-token costs.

## Don't Hand-Roll

Problems that look simple but have existing solutions:

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Task persistence | Custom JSON task queue | Beads pattern (Git-backed YAML) | Git provides free versioning, merging, conflict resolution, and distributed coordination |
| Multi-model routing | Custom provider abstraction | OpenCode SDK | Already supports 75+ providers with unified API and context preservation |

**Key insight:** Multi-agent orchestration in 2026 has mature tooling. Focus on composition patterns (hierarchical delegation, verification loops, context layering) rather than building infrastructure.

## Common Pitfalls

### Pitfall 1: Nested Subagent Spawning
**What goes wrong:** Attempting to spawn subagents from subagents in Claude Code fails silently or produces non-interactive instances.

**Why it happens:** Claude Code architectural limitation - only main context can spawn Task tool subagents.

**How to avoid:**
- Keep Sub-coordinator in main orchestrator context
- Sub-coordinator spawns parallel context-merge agents directly
- Never attempt to spawn from within a subagent

**Warning signs:**
- Subagent logs show "Task tool not available"
- Spawned agents lack context sharing
- Inconsistent behavior across runs

### Pitfall 2: System Prompt Overload
**What goes wrong:** Putting both methodology AND protocol data in system prompt creates confusion and reduces reasoning quality.

**Why it happens:** Misunderstanding of system vs user prompt separation.

**How to avoid:**
- System prompt = Reasoning template (methodology, anti-patterns, semantic triggers)
- User prompt = Protocol pack (roles, assumptions, risk categories, target contracts)
- This separation optimizes LLM reasoning by clearly distinguishing HOW from WHAT

**Warning signs:**
- Agent outputs mix methodology with data
- Inconsistent reasoning across similar tasks
- Token budget exhausted by repeated protocol data in system prompt

### Pitfall 3: Context Merge Without Verification
**What goes wrong:** Low-quality merged context creates garbage-in-garbage-out for vuln-discovery agents.

**Why it happens:** Skipping verification step to "move faster" or assuming LLM output is always correct.

**How to avoid:**
- Always verify context-merge output before bead creation
- Check: reasoning template complete, protocol context present, semantic triggers valid, no contradictions
- Loop back to context-merge with specific errors if verification fails

**Warning signs:**
- Vuln-discovery agents report "unclear instructions"
- False positives due to contradictory reasoning
- Beads lack required fields or have malformed data

### Pitfall 4: Token Budget Mismanagement
**What goes wrong:** Context exceeds model limits or costs spiral due to uncached repetition.

**Why it happens:** Not leveraging prompt caching or trimming context incorrectly.

**How to avoid:**
- Use Anthropic prompt caching for stable prefixes (VulnDocs, tool definitions, contract source)
- Trim order when needed: Drop exploits/examples FIRST, keep reasoning template and protocol context LAST
- No hard token limit, but keep context dense and focused

**Warning signs:**
- Repeated context in every request (no caching)
- Model context window errors
- High API costs despite using free-tier models

### Pitfall 5: Bead Structure Inconsistency
**What goes wrong:** Different agents create beads with incompatible schemas, breaking handoffs.

**Why it happens:** No standardized bead creation mechanism.

**How to avoid:**
- ALL bead creation goes through skills (`/vrs-create-bead-context-merge`, `/vrs-create-bead-finding`)
- Skills enforce schema validation
- Include full evidence chain inline (no external references)

**Warning signs:**
- Beads missing required fields
- Downstream agents cannot parse beads
- Evidence links broken or missing

### Context Merger Implementation
```python
# Source: Phase 5.4 VulnDocs framework + Phase 3 Protocol context
# Combines vulndoc reasoning template with protocol economic context

from dataclasses import dataclass
from typing import Dict, List

@dataclass
class MergedContext:
    """Context bundle for vuln-discovery agent."""
    vulnerability_class: str
    reasoning_template: str
    anti_patterns: List[str]
    semantic_triggers: List[str]
    protocol_context: Dict
    target_contracts: List[str]

class ContextMerger:
    """Merges vulndoc reasoning with protocol pack."""

    def merge(self, vulndoc_path: str, protocol_pack: Dict) -> MergedContext:
        """
        Merge vulndoc reasoning template with protocol context.

        Template-based injection pattern:
        - System prompt = vulndoc's reasoning_template (HOW to think)
        - User prompt = protocol pack (WHAT to analyze)
        """
        # Load vulndoc index.yaml
        vulndoc = self.load_vulndoc_index(vulndoc_path)

        # Extract reasoning components
        reasoning_template = vulndoc.get('reasoning_template', '')
        anti_patterns = self.extract_anti_patterns(vulndoc)
        semantic_triggers = vulndoc.get('semantic_triggers', [])

        # Extract protocol context
        protocol_context = {
            'roles': protocol_pack.get('roles', []),
            'assumptions': protocol_pack.get('assumptions', []),
            'risk_categories': self.extract_relevant_risks(
                protocol_pack,
                vulndoc['category']
            ),
            'accepted_risks': protocol_pack.get('accepted_risks', [])
        }

        # Build target contract list
        target_contracts = protocol_pack.get('scope', {}).get('contracts', [])

        return MergedContext(
            vulnerability_class=f"{vulndoc['category']}/{vulndoc['subcategory']}",
            reasoning_template=reasoning_template,
            anti_patterns=anti_patterns,
            semantic_triggers=semantic_triggers,
            protocol_context=protocol_context,
            target_contracts=target_contracts
        )

    def extract_relevant_risks(self, protocol_pack: Dict, category: str) -> Dict:
        """Extract risk categories relevant to vulnerability class."""
        risk_mapping = {
            'reentrancy': ['liquidity_risks', 'access_risks'],
            'oracle': ['oracle_risks', 'timing_risks'],
            'access-control': ['access_risks', 'governance_risks'],
            'upgrade': ['upgrade_risks', 'governance_risks']
        }

        relevant_categories = risk_mapping.get(category, [])
        risks = {}

        for cat in relevant_categories:
            if cat in protocol_pack.get('risk_categories', {}):
                risks[cat] = protocol_pack['risk_categories'][cat]

        return risks
```

### Sub-Coordinator with Parallel Spawning
```python
# Source: Hierarchical orchestration patterns
# https://developers.googleblog.com/developers-guide-to-multi-agent-patterns-in-adk/

import asyncio
from typing import List

class SubCoordinator:
    """
    Coordinates parallel context-merge agents with verification loop.
    Lives in main orchestrator context (Opus 4.5).
    """

    async def coordinate_context_preparation(
        self,
        protocol_pack: Dict,
        vuln_classes: List[str]
    ) -> List[str]:
        """
        Spawn parallel context-merge agents for each vulnerability class.
        Returns list of verified bead IDs.
        """
        # Spawn context-merge agents in parallel (Sonnet 4.5)
        merge_tasks = [
            self.spawn_context_merge_agent(protocol_pack, vuln_class)
            for vuln_class in vuln_classes
        ]

        # Wait for all context-merge agents
        merged_contexts = await asyncio.gather(*merge_tasks)

        # Verify each merged context (Sonnet 4.5)
        verified_beads = []
        for merged_ctx in merged_contexts:
            bead_id = await self.verify_and_create_bead(merged_ctx)
            if bead_id:
                verified_beads.append(bead_id)

        return verified_beads

    async def spawn_context_merge_agent(
        self,
        protocol_pack: Dict,
        vuln_class: str
    ) -> Dict:
        """Spawn single context-merge agent (Sonnet 4.5)."""
        # Load vulndoc for this class
        vulndoc_path = f"vulndocs/{vuln_class}"

        # Build agent prompt
        prompt = f"""
You are a context-merge specialist for {vuln_class} vulnerabilities.

# Task
Merge the vulnerability detection methodology with protocol context:
1. Extract reasoning template from vulndoc
2. Identify anti-patterns (NOT vulnerable if...)
3. Map semantic triggers to protocol risk categories
4. Combine into cohesive investigation context

# Vulndoc Location
{vulndoc_path}

# Protocol Context
{protocol_pack}

Output merged context as YAML with:
- reasoning_template (semantic, not VQL)
- anti_patterns (list)
- semantic_triggers (list)
- protocol_context (dict)
"""

        # Spawn via OpenCode SDK (Sonnet 4.5)
        result = await self.opencode_client.execute(
            model="anthropic/claude-sonnet-4",
            prompt=prompt,
            output_format="yaml"
        )

        return result

    async def verify_and_create_bead(self, merged_ctx: Dict) -> str:
        """
        Verify merged context and create bead via skill.
        Loops back with errors if verification fails.
        """
        max_retries = 3
        retry_count = 0

        while retry_count < max_retries:
            # Verify quality (Sonnet 4.5)
            verification = await self.spawn_verifier_agent(merged_ctx)

            if verification['passed']:
                # Create bead via skill
                bead_id = await self.create_bead_via_skill(
                    skill="/vrs-create-bead-context-merge",
                    context=merged_ctx
                )
                return bead_id
            else:
                # Retry with specific errors
                merged_ctx = await self.retry_context_merge(
                    merged_ctx,
                    errors=verification['errors']
                )
                retry_count += 1

        # Failed after retries
        self.log_failure(merged_ctx, "verification_failed")
        return None
```

### Vuln-Discovery Agent with Evidence Chain
```python
# Source: Phase 5.5 agent execution architecture
# Full evidence chain stored inline in bead

class VulnDiscoveryAgent:
    """
    Investigates specific vulnerability class with full evidence chain.
    Model selected by task router (Opus for critical, Sonnet for standard).
    """

    def investigate(self, context_bead_id: str) -> List[str]:
        """
        Investigate vulnerability class, create finding beads.
        Returns list of finding bead IDs.
        """
        # Load context bead
        context = self.load_bead(context_bead_id)

        # Build prompts with methodology/data separation
        system_prompt = self.build_system_prompt(context)
        user_prompt = self.build_user_prompt(context)

        # Execute investigation
        findings = self.execute_with_vkg_queries(system_prompt, user_prompt)

        # Create finding beads
        finding_bead_ids = []
        for finding in findings:
            bead_id = self.create_finding_bead(finding)
            finding_bead_ids.append(bead_id)

        return finding_bead_ids

    def build_system_prompt(self, context: Dict) -> str:
        """System prompt = HOW to think (methodology)."""
        return f"""
You are investigating {context['vulnerability_class']} vulnerabilities.

# Detection Methodology
{context['reasoning_template']}

# Anti-Patterns (NOT vulnerable if...)
{'\n'.join(f'- {ap}' for ap in context['anti_patterns'])}

# Tools Available
- /vrs-vql-help - Get VQL syntax help
- /vrs-create-bead-finding - Create finding bead

# Instructions
1. Use BSKG queries to analyze code behavior (NOT manual reading)
2. Follow reasoning template methodology
3. Check anti-patterns to avoid false positives
4. Flag uncertain cases with confidence=uncertain
5. Create finding bead for each discovery
6. Include FULL evidence chain inline in bead
"""

    def build_user_prompt(self, context: Dict) -> str:
        """User prompt = WHAT to analyze (protocol data)."""
        protocol = context['protocol_context']
        return f"""
# Protocol Context
## Roles
{self._format_roles(protocol['roles'])}

## Risk Categories
{self._format_risks(protocol['risk_categories'])}

## Accepted Risks
{self._format_accepted_risks(protocol.get('accepted_risks', []))}

# Target Contracts
{'\n'.join(f'- {c}' for c in context['target_contracts'])}

# Task
Investigate {context['vulnerability_class']} in the target contracts.
Use BSKG queries to identify vulnerable patterns.
Create finding beads with full evidence chains.
"""

    def create_finding_bead(self, finding: Dict) -> str:
        """Create finding bead via skill with full evidence chain."""
        # Build full evidence chain (stored INLINE in bead)
        evidence_chain = {
            'code_locations': finding['locations'],
            'vulndoc_reference': finding['vuln_class'],
            'reasoning_steps': finding['reasoning'],
            'vql_queries_used': finding['queries'],
            'protocol_context_applied': finding['protocol_context'],
            'graph_properties': finding['properties'],
            'semantic_operations': finding['operations'],
            'behavioral_signature': finding.get('signature')
        }

        # Create bead via skill (Sonnet 4.5)
        bead_id = self.call_skill(
            skill="/vrs-create-bead-finding",
            finding_data={
                'vulnerability_class': finding['vuln_class'],
                'severity': finding['severity'],
                'confidence': finding['confidence'],
                'description': finding['description'],
                'evidence_chain': evidence_chain,
                'next_action': 'verify_with_test'
            }
        )

        return bead_id
```

def build_cached_prompt(vulndoc: Dict, protocol_pack: Dict, bead_context: Dict):
    """
    Build prompt with caching for 90% cost reduction.
    Stable prefix cached, dynamic suffix per-bead.
    """
    # CACHED PREFIX (shared across all agents in this audit)
    cached_messages = [
        {
            "role": "user",
            "content": [
                # VulnDocs knowledge (stable)
                {
                    "type": "text",
                    "text": f"# Vulnerability Knowledge\n{vulndoc['reasoning_template']}",
                    "cache_control": {"type": "ephemeral"}
                },
                # Protocol pack (stable per audit)
                {
                    "type": "text",
                    "text": f"# Protocol Context\n{yaml.dump(protocol_pack)}",
                    "cache_control": {"type": "ephemeral"}
                },
                # Target contract source (stable)
                {
                    "type": "text",
                    "text": f"# Contract Source\n{load_contract_source()}",
                    "cache_control": {"type": "ephemeral"}
                }
            ]
        }
    ]

    # DYNAMIC SUFFIX (per-bead, not cached)
    dynamic_message = {
        "role": "user",
        "content": f"# Bead Context\n{yaml.dump(bead_context)}"
    }

    return cached_messages + [dynamic_message]

| Markdown task plans | Structured YAML beads with DAG | 2026 | Better for agent handoffs, no context loss |

**Deprecated/outdated:**
- **API-based agent runtimes:** Too expensive for extensive multi-agent workflows ($2-12 per 1M tokens). Use subscription CLIs.
- **Nested subagent spawning:** Claude Code limitation discovered 2026. Use main-context Sub-coordinator.
- **System prompt overload:** Mixing methodology + data in system prompt reduces reasoning quality. Separate into system/user prompts.
- **VQL in reasoning templates:** LLMs reason better with semantic methodology than SQL-like query examples.

## Open Questions

Things that couldn't be fully resolved:

1. **Optimal context-merge batch size**
   - What we know: Parallel spawning faster, but too many risks context pollution
   - What's unclear: Sweet spot between 5-20 parallel agents per sub-coordinator
   - Recommendation: Start with 10, monitor quality/latency tradeoff

2. **Verification loop retry threshold**
   - What we know: Max 3 retries prevents infinite loops
   - What's unclear: Should some error types abort immediately vs retry?
   - Recommendation: Abort on schema errors, retry on quality/completeness

3. **Token budget for merged context**
   - What we know: No hard limit, dense and compact preferred
   - What's unclear: Practical guideline for reasoning template + protocol pack
   - Recommendation: Target 2-4k tokens total, drop exploits if >6k

4. **Model escalation criteria**
   - What we know: Task router decides Opus vs Sonnet
   - What's unclear: Exact criteria for "critical" vs "standard" tasks
   - Recommendation: Critical = high severity OR novel pattern, standard = known patterns

5. **Bead storage location**
   - What we know: Git-backed YAML in `.vrs/beads/`
   - What's unclear: Organizational structure for 100+ beads per audit
   - Recommendation: Flat structure with hash IDs, use `type:` field for filtering
