# Phase 5.5: Agent Execution & Context Enhancement - Context

**Gathered:** 2026-01-22
**Status:** Ready for planning

<domain>
## Phase Boundary

Improve accuracy and execution of spawned subagents for vulnerability testing by enhancing context gathering with real-world economic/security knowledge. Build the agent orchestration flow for context-merge → verification → bead creation → vuln-discovery.

Key deliverables:
- Context extraction protocol (protocol pack + vulndoc merge)
- Sub-coordinator and context-merge agent architecture
- Context-merge verifier with quality loop
- Bead-based task tracking for all context-merge and finding outputs
- Skills for bead creation and VQL assistance
- Vuln-discovery agent with full evidence chain output

**Builds on Phase 5.4:** Uses unified vulndocs structure (index.yaml, reasoning_template, semantic_triggers)

</domain>

<decisions>
## Implementation Decisions

### Context Architecture

**Template-based injection pattern:**
- **System prompt** = Vulndoc's `reasoning_template` (HOW to think)
  - Step-by-step detection methodology
  - Anti-patterns section ("NOT vulnerable if...")
  - Semantic, not VQL (agents formulate queries themselves)
- **User context** = Protocol pack (WHAT to analyze)
  - Risk profile categories
  - Roles, assumptions, accepted risks
  - Target contracts scope

**Context layering:** Protocol + Vulndoc merged upfront → Agent queries graph for evidence

**Token budget:** No hard limit, but keep context compact and dense. When trimming needed: drop exploits/examples first, keep reasoning template and protocol context.

**Spawn scope:** One vulnerability class per agent spawn. Fresh context, focused reasoning.

### Agent Flow Architecture

```
MAIN ORCHESTRATOR (Opus 4.5)
         │
         ▼
SUB-COORDINATOR (Opus 4.5)
  - Has minimal base protocol context
  - Spawns context-merge agents in PARALLEL
  - For each vuln class to test
         │
         ├──────────────┬──────────────┐
         ▼              ▼              ▼
Context-Merge    Context-Merge    Context-Merge
 (Sonnet 4.5)    (Sonnet 4.5)    (Sonnet 4.5)
         │              │              │
         ▼              ▼              ▼
Context-Merge    Context-Merge    Context-Merge
  VERIFIER        VERIFIER         VERIFIER
 (Sonnet 4.5)    (Sonnet 4.5)    (Sonnet 4.5)
   │                  │                │
   │ IF FAIL: Report errors, back to merge
   │ IF PASS: Invoke /vrs-create-bead-context-merge
   │
   ▼
BEADS (organized in pools)
  - Each bead = one vuln class to investigate
  - Contains merged context (protocol + vulndoc)
         │
         ▼
MAIN ORCHESTRATOR (continues)
  - Reads beads
  - Spawns vuln-discovery agents
         │
         ▼
Vuln-Discovery Agents (model via task router)
  - Can create sub-beads for findings
  - Uses /vrs-create-bead-finding
```

### Model Assignments

| Role | Model | Rationale |
|------|-------|-----------|
| Main Orchestrator | Opus 4.5 | Complex coordination decisions |
| Sub-coordinator | Opus 4.5 | Guides parallel context-merge |
| Context-merge agent | Sonnet 4.5 | Good reasoning, cost-effective |
| Context-merge verifier | Sonnet 4.5 | Quality checks, not deep reasoning |
| Vuln-discovery agent | Task router decides | Critical tasks get Opus, standard get Sonnet |

**Critical rule:** Only Orchestrator and Sub-coordinator use Opus. All other subagents use Sonnet unless task router escalates.

### Economic Context (Protocol Pack)

**Risk categories (expanded coverage):**
- `oracle_risks` - Oracle dependencies, price feed exposure
- `liquidity_risks` - Flash loan exposure, liquidity depth
- `access_risks` - Privilege escalation, admin key risks
- `upgrade_risks` - Proxy patterns, implementation changes
- `integration_risks` - Cross-protocol dependencies
- `timing_risks` - MEV, frontrunning exposure
- `economic_risks` - Incentive misalignment, game theory
- `governance_risks` - Voting manipulation, timelock bypass

**Format:** Each category has `present: bool` + `notes: string`. No severity scores - agent interprets.

**Data source:** Auto-generated during Phase 3 protocol pack creation (code analysis + web research)

**Conservative default:** Unknown exposure = assume present. Better false positives than missed vulns.

**NOT included:** TVL (doesn't help find vulnerabilities). Treat every protocol as high-value.

### Reasoning Pattern Application

**Template depth:** Adaptive by complexity
- Simple vulns: 4-6 high-level steps
- Complex vulns (cross-contract, MEV): 10-15 detailed steps with edge cases

**Anti-patterns:** Explicitly included in template - "NOT vulnerable if..." section to reduce false positives

**Query formulation:** Templates use semantic language, not VQL. Agents use `/vrs-vql-help` skill when needed to formulate proper queries.

**Finding criteria:** Agent judgment with template guidance
- Template provides methodology (what to look for)
- Agent applies judgment using protocol context
- Conservative: flag uncertain cases for human review

**Uncertainty handling:** BOTH explicit instruction AND confidence-based routing
- System prompt tells agent to flag uncertain cases
- Orchestrator routes low-confidence findings to human

### Skills and Subagents

**Skills for this phase:**

| Skill | Purpose | Model |
|-------|---------|-------|
| `/vrs-create-bead-context-merge` | Create bead from verified context-merge output | Sonnet 4.5 |
| `/vrs-create-bead-finding` | Create finding bead from vuln-discovery output | Sonnet 4.5 |
| `/vrs-vql-help` | On-demand VQL syntax/examples for agents | Haiku 4.5 |

**Bead management:** Follow steveyegge/beads pattern for task management skills. Agents can create tasks and subtasks.

**Deletion protection:** Sonnet agents CANNOT delete beads/tasks without Opus confirmation.

**Core + optional skills:** All vuln-discovery agents have core skills (`/vrs-vql-help`, `/vrs-create-bead-finding`). Context bundle can add vuln-specific skills if they exist.

### Agent Output Format

**Format:** Bead-compatible YAML. All findings created via `/vrs-create-bead-finding` skill.

**Evidence chain:** Full chain stored INLINE in bead:
- Code locations (file:line references)
- Vulndoc reference that guided analysis
- Reasoning steps taken
- VQL queries used
- Protocol context applied

**Confidence:** Enum + reasoning
- Level: confirmed/likely/uncertain/rejected (matches Phase 4 VerdictConfidence)
- Reasoning: free-text explanation of why that confidence level

### Context Bundle Structure

**Implicit sections by depth:**
- `quick_checks` - Entry point queries, pre-run results (start fast)
- `investigation_areas` - Deep dive needed, requires creative queries
- `open_questions` - Areas for creative exploration

**Structured pointers in references:**
- `detection_logic_at` - Path to detection.md
- `verification_at` - Path to verification.md
- `similar_exploits_at` - Path to exploits.md
- `vql_queries` - Suggested queries to start with

**VQL handling:** Hybrid approach
- Simple queries pre-run, results cached (entry points)
- Complex/target-specific queries left for vuln-discovery agent
- Emphasis: pre-run queries are just starting points, agent must do exhaustive creative analysis

### Naming Convention

**Use `vrs` and `.vrs/` everywhere** - `vkg` naming deprecated in Phase 6.

<specifics>
## Specific Ideas

- Orchestrator delegates ALL work to subagents - never performs tasks directly
- Sub-coordinator ends job after context-merge beads are created (doesn't spawn vuln-discovery)
- Context-merge verifier loops back to merge agent with specific errors if quality fails
- Vuln-discovery agents can create sub-beads for findings they discover
- Every bead creation goes through a skill (`/vrs-create-bead-context-merge` or `/vrs-create-bead-finding`)
- Beads are the single source of truth for task tracking (no separate folder artifacts)
- Full evidence chain inline in beads - self-contained, auditable

**Key insight from discussion:**
> "The orchestrator should delegate as much work as possible. We should use skills and subagents as much as possible for every task of this project."

This principle applies to all future planning in `.planning/`.

</specifics>
