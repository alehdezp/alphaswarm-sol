# Phase 5.5 Plan 02: Context Merger & Verifier Summary

**One-liner:** Context merger combines vulndoc + protocol pack with conservative defaults; verifier validates completeness with quality scoring and actionable retry feedback.

## What Was Built

### ContextMerger (`src/true_vkg/agents/context/merger.py`)

Orchestrates merging of multiple context sources into unified bundles:
- **Base extraction:** Vulndoc (reasoning template, semantic triggers) + protocol pack (risk profile)
- **Additional context:** Merges extra VQL queries, graph patterns, reasoning notes
- **Conservative defaults:** Unknown risks → present=True (better false positives than misses)
- **Token budget:** Target 4k, max 6k, trims queries/patterns if exceeded
- **Error handling:** Graceful failure with detailed error messages

**Key Methods:**
- `merge()` - Main orchestration: extract → merge → apply defaults → check budget
- `_merge_additional()` - Append additional context to bundle
- `_apply_conservative_defaults()` - Set unknown risks to present

**Result:** `MergeResult` dataclass with success status, bundle, errors, warnings, token count, trimming flag, and sources used.

### ContextVerifier (`src/true_vkg/agents/context/verifier.py`)

Validates merged context quality before bead creation:
- **Required fields:** vulnerability_class, reasoning_template, semantic_triggers, risk_profile, target_scope
- **Quality checks:** Template length (≥100 chars), step structure (numbered steps), valid operations, risk completeness
- **Scoring:** Quality score 0.0-1.0 (errors -0.25 each, warnings -0.05 each)
- **Retry feedback:** Formatted feedback for context-merge agent retry loop

**Key Methods:**
- `verify()` - Main validation: check fields → quality checks → scoring → feedback
- `_has_step_structure()` - Detect numbered steps in template
- `_check_valid_operations()` - Validate semantic operation names
- `_count_unknown_risks()` - Count risk categories with unknown confidence
- `_calculate_quality_score()` - Compute 0.0-1.0 quality metric
- `_format_retry_feedback()` - Generate actionable retry feedback

**Result:** `VerificationResult` dataclass with valid flag, errors, warnings, quality score, and optional feedback.

## Implementation Decisions

### 1. Conservative Defaults for Unknown Risks

**Decision:** When risk confidence is "unknown", set `present=True`.

**Rationale:**
- Better to investigate false positives than miss real vulnerabilities
- Aligns with security-first philosophy
- Agent can reject findings if not applicable

**Impact:** Increases false positive rate but ensures comprehensive analysis coverage.

### 2. Error vs. Warning Distinction

**Decision:** Errors block verification (valid=False), warnings reduce quality score only.

**Rationale:**
- Enables smart retry loops: fix errors first, improve quality after
- Allows agents to proceed with warnings if acceptable
- Matches human review patterns (critical vs. nice-to-have)

**Implementation:**
- Errors: Missing required fields, too-short template, empty triggers
- Warnings: Invalid operations, no step structure, many unknown risks

### 3. Token Budget Trimming Order

**Decision:** Trim graph_patterns and vql_queries beyond first 3 each. Never trim reasoning_template or risk_profile.

**Rationale:**
- Reasoning template is HOW to think (critical)
- Risk profile is WHAT to analyze (critical)
- Queries/patterns are hints (helpful but not essential)
- Agent can formulate own queries using `/vrs-vql-help`

**Implementation:** `_trim_to_budget()` in extractor, called by merger when token count > 6000.

## Technical Insights

### 1. Quality Loop Pattern

The verifier implements a quality loop that enables iterative improvement:
1. Merge context sources
2. Verify completeness and quality
3. If invalid: return specific feedback
4. Agent retries with feedback
5. Loop until valid or max retries (3)

This pattern is reusable for other agent workflows requiring quality assurance.

### 2. Conservative Defaults Philosophy

"Unknown = assume present" is a key security pattern. When unsure about risk exposure, agents should investigate rather than skip. This philosophy should extend to:
- Pattern matching (uncertain = flag for review)
- Evidence gathering (partial evidence = investigate further)
- Finding confidence (uncertain = route to human)

### 3. Token Budget as Quality Signal

Token count > 6000 indicates overly complex context. Trimming is a signal to:
- Simplify reasoning template
- Reduce number of starting queries
- Focus on most relevant patterns

Future work: Analyze which vulndocs consistently exceed budget and refine their templates.
