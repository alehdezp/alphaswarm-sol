## Decisions Made

**1. Runtime imports to avoid circular dependency**
- **Rationale:** Beads package imports from agents.context (ContextBundle), agents.context imports from beads (ContextMergeBead) would create circular import
- **Solution:** Use TYPE_CHECKING for type hints, runtime imports in factory methods
- **Impact:** Clean type hints without circular dependency issues

**2. Inline context bundle storage**
- **Rationale:** Per 05.5-CONTEXT.md, full evidence chain should be embedded in bead
- **Solution:** ContextBundle stored as nested dataclass in bead.context_bundle
- **Impact:** Self-contained beads, no external reference resolution needed

**3. Deterministic bead ID generation**
- **Rationale:** Same inputs should produce same ID for idempotency
- **Solution:** Hash of vuln_class + protocol_name + timestamp with CTX- prefix
- **Impact:** Predictable IDs, easy to verify bead creation, collision-resistant

**4. Pool-based directory organization**
- **Rationale:** Beads need to be organized by audit pool for filtering
- **Solution:** .vrs/beads/{pool_id}/ structure with unassigned/ fallback
- **Impact:** Easy to list pending beads by pool, clear organization

**5. Finding bead association tracking**
- **Rationale:** Context beads spawn finding beads, need to track downstream work
- **Solution:** finding_bead_ids list with add_finding_bead() method
- **Impact:** Can trace which findings came from which context bundle
