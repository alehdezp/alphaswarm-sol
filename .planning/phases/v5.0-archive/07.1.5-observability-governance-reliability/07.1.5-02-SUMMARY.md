# Phase 07.1.5 Plan 02: Structured Audit Logging & Evidence Lineage Summary

**One-liner:** Structured audit logging with evidence provenance chains using structlog and lineage tracking

## What Was Built

### 1. Structured Audit Logger (audit.py)
- **AuditLogger class** with structlog integration
  - JSON processor with ISO timestamps
  - Context binding for trace correlation
  - Category-specific logging methods
  - Automatic severity assignment

- **AuditCategory enum** for filtering:
  - VERDICT_ASSIGNMENT
  - CONFIDENCE_UPGRADE
  - EVIDENCE_USAGE
  - POLICY_VIOLATION
  - POOL_LIFECYCLE
  - COST_TRACKING
  - TOOL_EXECUTION
  - HANDOFF

- **Logging methods:**
  - `log_verdict()` - Verdict assignments with evidence refs
  - `log_confidence_upgrade()` - Confidence changes with justification
  - `log_evidence_usage()` - Evidence usage for lineage tracking
  - `log_policy_violation()` - Policy violations with severity/action
  - `log_pool_event()` - Pool lifecycle events

### 2. Evidence Lineage Tracking (lineage.py)
- **LineageTracker class** for provenance tracking
  - Create lineage from source (BSKG, tool, manual)
  - Add transformation steps (confidence upgrades, derivations)
  - Add verification steps (agent verifications)
  - Query by source to find derived evidence
  - Build complete lineage chains for audit

- **SourceType enum:**
  - BSKG - Knowledge graph nodes
  - TOOL - External tool findings
  - MANUAL - Manual user input
  - DERIVED - Derived from other evidence

- **Data structures:**
  - LineageStep - Single operation in chain
  - EvidenceLineage - Complete provenance from origin
  - Serialization/deserialization support

### 3. JSON Schema (audit_schema.json)
- Complete schema for audit log entries
- Category-specific required/optional fields
- Conditional validation using allOf/if/then
- Supports all 8 audit categories
- Evidence lineage field definitions

### 4. Comprehensive Tests (test_audit_lineage.py)
- AuditLogger initialization and method tests
- LineageTracker creation and query tests
- Transformation and verification tests
- Serialization roundtrip tests
- Error handling tests

## Key Design Decisions

### Decision 1: Structlog with JSON Processor
**Choice:** Use structlog with JSON processor for all audit logs

**Why:**
- Industry standard for structured logging
- Native support for context binding (trace_id, pool_id)
- JSON output is machine-parsable for dashboards
- Processor chains enable custom formatting

**Tradeoffs:**
- Adds dependency (but already in project)
- Requires configuration setup
- More complex than stdlib logging

**Impact:** All audit logs are queryable, parsable, and correlate with traces

### Decision 2: In-Memory Lineage Tracking
**Choice:** Track lineage in-memory with optional persistence

**Why:**
- Fast queries for recent evidence
- No database setup required
- Sufficient for single-instance deployments
- Optional disk persistence for long-term audit

**Tradeoffs:**
- Lost on process restart (unless persisted)
- Not suitable for distributed deployments
- Limited to memory capacity

**Impact:** Simple lineage queries work out-of-box, advanced cases can add persistence

### Decision 3: Enum-Based Audit Categories
**Choice:** Use AuditCategory enum instead of free-form strings

**Why:**
- Type safety at development time
- IDE autocomplete for valid categories
- Prevents typos in production
- Easy to extend with new categories

**Tradeoffs:**
- Requires updating enum for new categories
- Can't add categories at runtime
- More verbose than magic strings

**Impact:** Fewer bugs from category typos, easier filtering in dashboards

## Usage Examples

### Audit Logging
```python
from alphaswarm_sol.observability import AuditLogger

logger = AuditLogger()

# Log verdict assignment
logger.log_verdict(
    pool_id="pool-123",
    bead_id="VKG-042",
    verdict="vulnerable",
    confidence="LIKELY",
    evidence_refs=["ev-001", "ev-002"],
    agent_type="vrs-attacker",
    trace_id="trace_abc123"
)

# Log policy violation
logger.log_policy_violation(
    pool_id="pool-123",
    policy_id="cost_budget.hard_limit",
    violation_type="budget_exceeded",
    actor="vrs-attacker",
    severity="critical",
    suggested_action="block",
    details={"cost_usd": 12.5, "limit_usd": 10.0},
    trace_id="trace_abc123"
)
```

### Evidence Lineage Tracking
```python
from alphaswarm_sol.observability import LineageTracker, SourceType

tracker = LineageTracker()

# Create lineage for BSKG-sourced evidence
lineage = tracker.create_lineage(
    evidence_id="ev-001",
    source_type=SourceType.BSKG,
    source_id="node_func_vault_withdraw_123",
    extracting_agent="vrs-attacker"
)

# Add transformation step
tracker.add_transformation(
    evidence_id="ev-001",
    transform_type="confidence_upgrade",
    transforming_agent="vrs-verifier",
    metadata={"from": "POSSIBLE", "to": "LIKELY"}
)

# Add verification step
tracker.add_verification(
    evidence_id="ev-001",
    verifying_agent="vrs-verifier",
    verification_result="confirmed"
)

# Query all evidence from BSKG node
derived = tracker.query_by_source(
    SourceType.BSKG,
    "node_func_vault_withdraw_123"
)

# Build complete lineage chain
chain = tracker.build_lineage_chain("ev-001")
```
