# Phase 07.1.5 Plan 03: Policy Enforcement Summary

**One-liner:** Runtime policy enforcement with cost budgets, tool access gates, evidence integrity checks, and audit trail integration

## What Was Built

### 1. Policy Definitions (policies.py)
- **PolicyAction enum** for enforcement actions:
  - ALLOW - Operation proceeds
  - BLOCK - Operation blocked, error raised
  - SANITIZE - Operation modified to comply
  - ALERT - Warning logged, operation proceeds

- **Policy dataclasses**:
  - `CostPolicy` - Hard/soft limits for budget enforcement
  - `ToolAccessPolicy` - Role restrictions and forbidden tools
  - `EvidencePolicy` - Evidence integrity requirements
  - `ModelUsagePolicy` - Tier restrictions per agent type

- **PolicyViolation dataclass** - Structured violation records with severity, action, details

- **load_policies()** - YAML configuration parser

### 2. Governance Configuration (governance_policies.yaml)
```yaml
policies:
  cost_budget:
    hard_limit_usd: 10.0    # Blocks if exceeded
    soft_limit_usd: 8.0     # Alerts if exceeded
    block_on_exceed: true

  tool_access:
    role_restrictions:
      vrs-attacker: [slither, mythril, echidna, bskg_query]
      vrs-defender: [slither, aderyn, halmos, bskg_query]
      vrs-verifier: [bskg_query, semgrep]
    forbidden_tools: [shell_exec, file_write]

  evidence_integrity:
    require_evidence_refs: true
    min_evidence_count: 1

  model_usage:
    tier_restrictions:
      vrs-attacker: [opus, sonnet]
      cost-governor: [haiku]
```

### 3. Validators (validators.py)
Pure validator functions that return `Optional[PolicyViolation]`:

- **validate_cost_budget** - Check spend against hard/soft limits
  - Returns BLOCK violation if exceeds hard limit
  - Returns ALERT violation if exceeds soft limit
  - Respects `block_on_exceed` flag

- **validate_tool_access** - Check tool usage against role restrictions
  - Checks forbidden_tools list (BLOCK if found)
  - Checks role_restrictions mapping (BLOCK if unauthorized)
  - No restrictions if agent type not in mapping

- **validate_evidence_integrity** - Check evidence refs for confidence upgrades
  - Only enforces for upgrades (confidence_from != None)
  - Requires min_evidence_count evidence refs
  - Returns BLOCK violation if insufficient

- **validate_model_usage** - Check model tier against agent restrictions
  - Maps model to tier (opus/sonnet/haiku)
  - Checks tier_restrictions per agent type
  - Supports explicit allowed_models override

### 4. Policy Enforcer (enforcer.py)
Main enforcement class with integration points:

- **PolicyEnforcer class**:
  - `__init__(policies_path, cost_ledger, audit_logger)` - Loads policies
  - `check_input_policy()` - Validates before operation (cost, tool, model)
  - `check_output_policy()` - Validates after operation (evidence)
  - `handle_violation()` - Logs to audit trail, raises if BLOCK

- **enforce_policy decorator** - Transparent enforcement for handlers:
  - Extracts kwargs (pool_id, agent_type, tool_name, model, etc.)
  - Calls check_input_policy before function execution
  - Calls check_output_policy after if result has evidence
  - Raises PolicyViolationError if BLOCK action

### 5. Integration Points
- **AuditLogger** - All violations logged via `log_policy_violation()`
- **CostLedger** - Budget checks use `ledger.total_cost` property
- **OpenTelemetry** - trace_id passed through for correlation

## Key Design Decisions

### Decision 1: Validator Pattern
**Choice:** Separate validators.py module with pure functions returning Optional[PolicyViolation]

**Why:**
- Testability: Easy to unit test each validator in isolation
- Composability: Validators can be called independently or combined
- Clear separation: Policy logic separate from enforcement logic
- Reusability: Validators can be used outside PolicyEnforcer

**Tradeoffs:**
- More files/modules vs all-in-one enforcer
- Slightly more verbose vs inline validation

**Impact:** Clean architecture, easy to add new validators

### Decision 2: Hard/Soft Limits for Cost
**Choice:** Separate hard_limit_usd (BLOCK) and soft_limit_usd (ALERT) thresholds

**Why:**
- Graceful degradation: Warnings before hard failures
- User experience: Teams can see costs approaching limit
- Flexibility: Some orgs want alerts only, others want hard stops
- Production safety: Soft limit = 80% hard limit prevents abrupt failures

**Tradeoffs:**
- More configuration vs single limit
- Requires monitoring soft limit alerts

**Impact:** Better production experience, fewer surprise failures

### Decision 3: Decorator Pattern for Enforcement
**Choice:** enforce_policy decorator wraps handler functions

**Why:**
- Non-intrusive: Handlers don't need to call enforcer explicitly
- Declarative: Clear which handlers have enforcement
- Easy to toggle: Add/remove decorator to enable/disable
- Consistent: Same pattern across all handlers

**Tradeoffs:**
- Implicit behavior vs explicit calls
- Requires specific kwarg names (pool_id, agent_type, etc.)

**Impact:** Clean handler code, transparent enforcement

### Decision 4: PolicyAction Enum
**Choice:** ALLOW, BLOCK, SANITIZE, ALERT enum instead of boolean

**Why:**
- Type safety: IDE autocomplete, no typos
- Extensible: Can add new actions (e.g., THROTTLE, DEFER)
- Semantic: Clear intent vs boolean "allowed"
- Future-proof: SANITIZE action reserved for data redaction

**Tradeoffs:**
- More verbose than boolean
- All actions not implemented yet (SANITIZE unused)

**Impact:** Flexible policy system ready for future features

## Usage Examples

### Direct PolicyEnforcer Usage
```python
from alphaswarm_sol.governance import PolicyEnforcer
from alphaswarm_sol.metrics.cost_ledger import CostLedger
from alphaswarm_sol.observability import AuditLogger
from pathlib import Path

# Initialize
enforcer = PolicyEnforcer(
    policies_path=Path("configs/governance_policies.yaml"),
    cost_ledger=ledger,
    audit_logger=logger
)

# Check input policy before operation
violation = enforcer.check_input_policy(
    pool_id="pool-123",
    agent_type="vrs-attacker",
    tool_name="slither",
    model="claude-3-5-sonnet",
    requested_cost=0.05,
    trace_id="trace_abc"
)
# Raises PolicyViolationError if violation.action == BLOCK

# Check output policy after operation
violation = enforcer.check_output_policy(
    bead_id="VKG-042",
    pool_id="pool-123",
    agent_type="vrs-attacker",
    evidence_refs=["ev-001", "ev-002"],
    confidence_from="POSSIBLE",
    confidence_to="LIKELY",
    trace_id="trace_abc"
)
```
