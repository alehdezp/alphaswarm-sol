<innovation_charter>
## Innovation Charter (All 6 Mandatory)

These innovations are intentionally bold. They may introduce complexity, but they target the single biggest risk before GA: **false positives and weak evidence chains**.

### I1: Counterfactual Corpus Generation (Near-Miss Factory)
**Idea:** Generate controlled near-miss variants of real contracts to expose brittle patterns.

**Mechanism:**
- For each vulnerable contract, synthesize 3-5 counterfactual variants:
  1) **Guard inversion**: flip `require()` or modifier order to create safe or unsafe variants.
  2) **Ordering swap**: swap CEI order (read/write/call) while preserving logic.
  3) **Window tweak**: change time windows (TWAP, grace period, timelock delay).
  4) **Helper delegation**: move checks into internal helpers to test detection depth.
  5) **L1/L2 branch**: conditionalize checks for chain ID or sequencer feed.

**Storage:** `.vrs/corpus/contracts/adversarial/counterfactual/` with metadata linking to base contract.

**Acceptance rule:** A counterfactual must differ by a single semantic edit and preserve surrounding behavior; BSKG property diff must reflect the intended change.

### I2: Tier C Shadow Mode (Dual Labeler Consensus)
**Idea:** Tier C findings run in shadow mode until *two independent labeling passes* agree.

**Mechanism:**
- Labeler A and B run with different prompt slices / seed order.
- **Consensus threshold: >= 70% label overlap** (relaxed from 80%) or explicit agreement on required labels.
- If disagreement: pattern downgrades to **unknown** (never "safe").

**Output:** `.vrs/testing/tier_c_shadow.yaml` logs disagreements and root causes.

### I3: Adversarial Label Mutation
**Idea:** Test Tier C robustness by injecting wrong or inverted labels.

**Mechanism:**
- Inject ~10% random label flips per test run.
- Swap label with negation pair where possible (e.g., `access_control.owner_only` -> `access_control.no_restriction`).
- Measure false positive amplification.

**Fail condition:** If FP rate increases > 2x, pattern is too label-fragile and must be redesigned.

### I4: Evidence Debt Ledger
**Idea:** Track patterns that rely on weak evidence and block promotion until debt is paid.

**Mechanism:**
- Each finding must link to evidence IDs.
- If >25% of a pattern's findings lack high-confidence evidence IDs → **evidence debt**.
- Evidence debt gets logged and blocks promotion to "ready/excellent".

**Output:** `.vrs/corpus/metadata/evidence_debt.yaml`.

### I5: Protocol Context TTL + Decay
**Idea:** Economic context must not go stale. Stale context must decay to unknown.

**Mechanism:**
- Every economic-context field gets a **30-day TTL** and `last_verified`.
- If expired: action determined by Claude for optimal detection.

**Output:** `.vrs/context/context_ttl.yaml` + `unknown` downgrade log.

### I6: Exploit Plausibility Scoring (Priority, Not Truth)
**Idea:** Add lightweight economic plausibility scoring to prioritize findings.

**Mechanism:**
- Score factors: liquidity depth, capital requirement, time window length, access barrier.
- Score **never affects correctness**, only ordering and review priority.

**Output:** `plausibility_score` on findings, stored in corpus metadata for analytics.

</innovation_charter>

<exa_key_improvements>
## Key Improvement Areas (Super Creative + Risky, Research-Backed)

- **Tier-C Stability Gate (probabilistic):** run each Tier-C pattern **N=50** times and require **stability >= 0.85** (same findings/labels). Instability is a signal; auto-route to **unknown**.
  Source: https://bssw.io/items/testing-non-deterministic-research-software
- **Metamorphic Pattern Testing (oracle-free):** define metamorphic relations (rename, refactor, helper-move) and require invariance across runs.
  Source: https://en.wikipedia.org/wiki/Metamorphic_testing
- **Evidence-Debt Quarantine:** if a pattern produces findings without strong evidence IDs, quarantine it even if tests pass.
  Source: https://discovery.ucl.ac.uk/1471263/
- **Mutation-Testing for Patterns:** introduce semantic-preserving and semantic-breaking mutations; use mutation score as test-suite strength.
  Source: https://frameworks.securityalliance.org/security-testing/mutation-testing
- **Exa-sourced "Real-World Negative Controls":** use Exa to find audited protocols with explicit "no issue" or "mitigated" outcomes and treat as hard negatives.
- **High-Impact Coverage Anchors:** align corpus coverage to OWASP Smart Contract Top 10 (2025).
  Source: https://owasp.org/www-project-smart-contract-top-10/
- **Economic-Context TTL:** stale assumptions degrade to "unknown," never "safe," especially on L2 oracle/sequencer risk.
  Source: https://docs.chain.link/data-feeds/l2-sequencer-uptime-feeds

</exa_key_improvements>

<tier_c_testing>
## Tier C Testing Plan (Label-Dependent Patterns)

Tier C patterns are the **most fragile** and **most important** to test. The test harness must include:

1. **Label ablation tests**
   - Run Tier A/B only, then Tier A/B/C with labels.
   - Confirm Tier C adds *only* expected findings.

2. **Counterfactual label injection**
   - Inject incorrect labels to ensure Tier C does NOT produce false positives.

3. **Label confidence thresholds**
   - Verify `low` confidence labels do not trigger high-severity Tier C matches.

4. **Context filtering sanity**
   - Ensure labels filtered by context still preserve required signals.

5. **Label drift tests**
   - Randomly drop 10–20% of labels and confirm Tier C falls back to "unknown" rather than "safe".

6. **Shadow mode (I2)**
   - Dual labeler consensus at 70% overlap threshold.
   - Disagreements logged and analyzed.

7. **Mutation stress (I3)**
   - 10% label flips per run.
   - FP amplification must stay < 2x.

**Deliverable:** A dedicated Tier C test harness with baseline/label/ablation runs and metrics.

</tier_c_testing>

<fp_reduction>
## False Positive Reduction Strategy

**Hard-negative suites** (must-have):
- Safe but complex protocols (multiple guards, delegates, helper checks).
- L1/L2 conditional logic where checks are valid only on certain chains.
- Indirect/utility functions that look risky but are not externally callable.

**Mutation testing:**
- Rename variables/functions (should not change detection).
- Reorder effects/calls within safe boundaries.
- Inject dummy external calls to test op-order resilience.

**Metrics:**
- Precision per pattern >= 90% for "excellent" tier.
- False positives must be recorded with a *root-cause tag* (missing property, label error, op ordering error).

</fp_reduction>
