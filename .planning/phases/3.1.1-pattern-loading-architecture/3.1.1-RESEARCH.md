# Phase 3.1.1: Pattern Loading Architecture Fix - Research

**Researched:** 2026-02-11
**Domain:** Python pattern loading, pytest fixtures, test architecture
**Confidence:** HIGH

## Summary

The pattern loading failure is fully understood and straightforward to fix. The root cause: `PatternStore.load()` silently returns `[]` when given a nonexistent path, and the entire codebase was pointing to `Path("patterns")` (a directory that no longer exists -- it was renamed to `vulndocs/`). An emergency `sed` fix changed most occurrences but missed `test_schema_snapshot.py` (16 occurrences still passing `Path("patterns")` to `build_schema_snapshot()`).

The architecture fix is simple: (1) make `PatternStore` fail loudly on nonexistent paths, (2) create a single canonical function for pattern discovery, (3) replace all 80+ inline `PatternStore.load_vulndocs_patterns(Path("vulndocs"))` calls in tests with a shared fixture, and (4) triage the remaining 333 failures into FIX/XFAIL/DELETE categories.

**Primary recommendation:** Create `get_default_patterns() -> list[PatternDefinition]` as a module-level convenience, a `@pytest.fixture` wrapper for tests, and make `PatternStore.load()` raise on missing paths.

## Architecture Patterns

### Current State (What Broke)

```
vulndocs/                          # WHERE patterns actually live
  reentrancy/
    classic/
      patterns/                    # Pattern YAML files live here
        classic-reentrancy.yaml
      index.yaml                   # Category metadata (NO 'id' field for PatternStore._parse)
    index.yaml                     # Top-level category index
  ...

Path("patterns")                   # DOES NOT EXIST (old name)
```

**Two incompatible loading APIs:**
- `PatternStore(dir).load()` -- Recursively loads ALL yaml/json. CRASHES on `index.yaml` (missing `id` field).
- `PatternStore.load_vulndocs_patterns(dir)` -- Only loads `**/patterns/*.yml` and `**/patterns/*.yaml`. Safe for vulndocs structure.

### Pattern 1: Single Canonical Loader

**What:** One function that encapsulates "find and load all patterns from the vulndocs directory."
**When to use:** Every production call site and every test.

```python
# In src/alphaswarm_sol/queries/patterns.py

_DEFAULT_VULNDOCS = Path("vulndocs")

def get_patterns(pattern_dir: Path | None = None) -> list[PatternDefinition]:
    """Load patterns from vulndocs directory. Raises if path missing."""
    pdir = pattern_dir or _DEFAULT_VULNDOCS
    if not pdir.exists():
        raise FileNotFoundError(
            f"Pattern directory '{pdir}' does not exist. "
            f"Expected vulndocs/ in project root."
        )
    patterns = PatternStore.load_vulndocs_patterns(pdir)
    if not patterns:
        raise RuntimeError(
            f"No patterns found in '{pdir}'. "
            f"Expected **/patterns/*.yaml files."
        )
    return patterns
```

### Pattern 2: Shared Test Fixture (conftest.py)

**What:** A `conftest.py` fixture or module-level cached loader that all tests use.
**When to use:** All pattern-loading tests.

Note: Most test files use `unittest.TestCase`, which cannot use pytest fixtures directly. Two options:

**Option A: Module-level cached loader (recommended for unittest.TestCase)**
```python
# tests/pattern_loader.py (new shared module)
from functools import lru_cache
from pathlib import Path
from alphaswarm_sol.queries.patterns import PatternStore, PatternDefinition

@lru_cache(maxsize=1)
def load_all_patterns() -> tuple[PatternDefinition, ...]:
    """Load all vulndocs patterns with caching. Returns tuple for hashability."""
    patterns = PatternStore.load_vulndocs_patterns(Path("vulndocs"))
    assert len(patterns) >= 400, f"Expected 400+ patterns, got {len(patterns)}"
    return tuple(patterns)
```

Then in each test file's `setUp()`:
```python
def setUp(self):
    self.patterns = list(load_all_patterns())
    self.engine = PatternEngine()
```

**Option B: conftest.py session fixture (for pytest-style tests)**
```python
# tests/conftest.py
@pytest.fixture(scope="session")
def all_patterns():
    patterns = PatternStore.load_vulndocs_patterns(Path("vulndocs"))
    assert len(patterns) >= 400
    return patterns
```

**Recommendation:** Option A. The codebase is 100% `unittest.TestCase`. Converting to pytest fixtures would require changing every test class -- not worth it for this phase.

### Pattern 3: Loud Failure on Missing Paths

**What:** `PatternStore.load()` and `load_vulndocs_patterns()` raise instead of returning `[]` silently.
**When to use:** Always -- this is the fix that prevents the same class of bug.

```python
def load(self) -> list[PatternDefinition]:
    if not self.root.exists():
        raise FileNotFoundError(f"Pattern directory not found: {self.root}")
    # ... rest of implementation
```

For `load_vulndocs_patterns()`:
```python
@classmethod
def load_vulndocs_patterns(cls, vulndocs_root: Path) -> list[PatternDefinition]:
    if not vulndocs_root.exists():
        raise FileNotFoundError(f"VulnDocs directory not found: {vulndocs_root}")
    # ... rest of implementation
```

**Backward compatibility concern:** `PatternEngine._load_patterns()` currently returns `[]` for missing dirs. This is used in production code where patterns might genuinely not be available (e.g., `PatternEngine(pattern_dir=None)`). Solution: keep `_load_patterns()` as a lenient wrapper, but make the direct `PatternStore` methods strict.

### Anti-Patterns to Avoid
- **Inline `PatternStore.load_vulndocs_patterns(Path("vulndocs"))` in every test** -- This is the current state. 80+ duplicate calls. One path change breaks everything.
- **Using `PatternStore(dir).load()` on vulndocs directory** -- Will crash on `index.yaml` files that lack the `id` field.
- **Silent empty returns** -- The root cause of 891 tests failing unnoticed.
- **Passing `pattern_dir=Path("patterns")` to functions** -- Still present in `test_schema_snapshot.py` (16 occurrences missed by sed fix).

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Pattern caching | Custom cache dict | `functools.lru_cache` | Already used in `graph_cache.py`, proven pattern |
| Path resolution | Complex env-var lookup | Simple `Path("vulndocs")` with existence check | YAGNI -- the path is always relative to project root |
| Test deduplication | Metaprogramming/test generators | Shared `setUp()` calling cached loader | Keep it simple |

## Common Pitfalls

### Pitfall 1: test_schema_snapshot.py Still Broken
**What goes wrong:** 16+ occurrences of `pattern_dir=Path("patterns")` were NOT caught by the sed fix.
**Why it happens:** The sed command targeted `PatternStore(Path("patterns")).load()` but `test_schema_snapshot.py` passes `Path("patterns")` as an argument to `build_schema_snapshot()`.
**How to avoid:** Search for ALL occurrences of `"patterns"` string in test files, not just the specific `PatternStore(...)` pattern.
**Files affected:** `tests/test_schema_snapshot.py` (16 occurrences)

### Pitfall 2: PatternStore.load() vs load_vulndocs_patterns() Semantic Mismatch
**What goes wrong:** `load()` crashes on vulndocs `index.yaml` files; `load_vulndocs_patterns()` only finds files under `**/patterns/` subdirectories.
**Why it happens:** Two APIs evolved independently. `load()` is the old flat-directory loader.
**How to avoid:** Deprecate `load()` as public API. All external callers should use `load_vulndocs_patterns()` or the new `get_patterns()` wrapper.

### Pitfall 3: unittest.TestCase Cannot Use pytest Fixtures Directly
**What goes wrong:** If you add a `@pytest.fixture` in conftest.py for pattern loading, none of the 21 test files can use it because they all use `unittest.TestCase`.
**Why it happens:** `unittest.TestCase.setUp()` doesn't support pytest fixture injection.
**How to avoid:** Use a plain Python function with `@lru_cache` that tests call from `setUp()`.

### Pitfall 4: Changing PatternStore.load() Breaks PatternEngine._load_patterns()
**What goes wrong:** `PatternEngine._load_patterns()` currently falls back to `PatternStore(pdir).load()` when the directory is not vulndocs-structured. Making `load()` strict would break this fallback.
**Why it happens:** The fallback returns `[]` when `pdir.exists()` is False.
**How to avoid:** `PatternEngine._load_patterns()` already has its own existence check at line 540-541. The strict `PatternStore.load()` only triggers when path doesn't exist, so `_load_patterns()` catches it first. But test this.

### Pitfall 5: Remaining 333 Failures Need Categorization, Not Bulk Fixes
**What goes wrong:** Attempting to "fix" all 333 remaining failures introduces wrong test expectations.
**Why it happens:** Some tests reference pattern IDs from a deprecated numbering scheme (ext-001..ext-072), some test code that was deleted, some have genuine detection gaps.
**How to avoid:** Categorize each failure before deciding action: FIX (infrastructure), XFAIL (known gap with tracking), DELETE (dead code), SKIP (external dep).

## Open Questions

1. **Should `PatternStore.load()` be fully deprecated?**
   - What we know: All production callers now use `load_vulndocs_patterns()`. The only remaining use of `.load()` is as a fallback in `PatternEngine._load_patterns()`.
   - What's unclear: Are there any external consumers of the `load()` API?
   - Recommendation: Add `@deprecated` warning but keep it functional for now. Make it strict (raise on missing path).

2. **What is the correct action for each of the 333 remaining failures?**
   - What we know: ~120 are pattern quality gaps, ~50 are deprecated IDs, ~72 are stale code
   - What's unclear: Exact breakdown per file until someone runs targeted test subsets
   - Recommendation: Run each affected test file individually and categorize. This is triage work, not research.

3. **Should the 16 `test_schema_snapshot.py` tests pass `pattern_dir=Path("vulndocs")` or just omit it?**
   - What we know: `build_schema_snapshot()` defaults to `Path("vulndocs")` when `pattern_dir` is None
   - Recommendation: Remove `pattern_dir=Path("patterns")` entirely -- let it default. The tests were testing schema snapshot functionality, not pattern directory resolution.
