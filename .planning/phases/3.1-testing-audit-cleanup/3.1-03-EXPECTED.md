# Expected Results: 3.1-03 Shipping Manifest + Audits

---

## Philosophy

This plan exists to create the authoritative inventory of shipped skills and agents (MANIFEST.yaml), audit the evaluator infrastructure for 3.1c readiness, assess the workflow_harness for 3.1b, clean settings.json, and produce sample capability contracts. It serves the phase goal by providing the single source of truth that all downstream phases (3.1b, 3.1c, 3.2) reference for "what ships" and "what evaluator APIs exist." This plan explicitly does NOT create evaluation contracts at scale (3.1c-06 does that), does NOT build the evaluation framework (3.1-05 and 3.1c do that), and does NOT fix identified evaluator gaps (those are assigned to specific 3.1b/3.1c plans). Its scope is strictly: inventory, audit, sample, clean.

## Implementation Reasoning

### Decision 1: Filesystem-derived manifest with count verification

- **Selected:** Scan `shipping/skills/` and `shipping/agents/` directories, extract metadata from each .md file, then verify counts match
- **Why:** A hardcoded manifest diverges from reality over time. Filesystem-derived with count checks ensures the manifest is always accurate. If the expected 30/21 counts differ, use actual counts.
- **Anti-patterns:** Do NOT hardcode 30 skills and 21 agents without verifying. Do NOT skip the count verification step. Do NOT include non-.md files in the count.

### Decision 2: Substantive audits with identified gaps, not cursory summaries

- **Selected:** Read each evaluator and harness file, document actual API methods/signatures, and identify concrete gaps or caveats
- **Why:** A cursory "looks fine" audit has zero value for 3.1c planning. The audit must surface API limitations, missing extension points, and edge cases that 3.1c will encounter. A 12-file harness is statistically unlikely to be perfect.
- **Anti-patterns:** Do NOT write "all good, no issues" without evidence. Do NOT skip reading the actual source code. Do NOT document methods that do not exist in the source.

### Decision 3: Sample contracts as pattern templates, not production contracts

- **Selected:** Create 3 representative sample contracts (skill, agent, orchestrator) to demonstrate the schema pattern
- **Why:** 3.1c-06 will create 24 evaluation contracts. These 3 samples show the format, the level of detail expected, and how capability_checks connect to real behaviors. They are explicitly templates for 3.1c to scale.
- **Anti-patterns:** Do NOT try to create all 24 contracts here. Do NOT create samples with < 3 capability_checks (too shallow to be useful as templates). Do NOT use fake skill/agent IDs.

## Real vs Placeholder Criteria

| Artifact | Real Indicators | Placeholder Indicators |
|----------|----------------|----------------------|
| `MANIFEST.yaml` | Each skill/agent entry has id, name, path, category, tier/model_tier; paths point to existing files; counts verified against `ls` | Empty entries; paths to non-existent files; count mismatches; no category/tier metadata |
| `evaluator-audit.md` | Lists actual method signatures from source code (e.g., `evaluate(transcript: str, config: dict) -> EvalResult`); identifies concrete gaps (e.g., "no contract-aware mode"); references line numbers | Generic descriptions without signatures; no gaps identified; references methods that do not exist |
| `harness-audit.md` | Per-file assessment with purpose, status, and specific findings; at least one actionable gap assigned to a 3.1b plan | Single paragraph summary; no per-file breakdown; "no issues found" |
| Sample contracts (.json) | Valid JSON; `workflow_id` uses real skill/agent IDs; `capability_checks` describe observable behaviors specific to that skill/agent; >= 3 checks each | Invalid JSON; generic checks not specific to the skill/agent; < 3 checks; placeholder IDs |
| `settings.json` (cleaned) | Valid JSON; hooks reference existing infrastructure; no entries for tmux/legacy runners | Unchanged from before; still has orphaned entries; invalid JSON |

## Acceptable Deviations

### OK (preserves intent)
- Skill count differs from expected 30 (use actual filesystem count)
- Agent count differs from expected 21 (use actual filesystem count)
- Additional metadata fields in MANIFEST.yaml beyond what plan specifies (enrichment is fine)
- Evaluator audit identifies more than one gap (more gaps = more useful)
- Sample contracts have > 3 capability_checks (more thorough is better)
- Sample contract file locations differ from plan (e.g., `.vrs/testing/contracts/samples/` instead of `tests/workflow_harness/contracts/samples/`) as long as consistently referenced

### DRIFT (misalignment -- needs correction)
- MANIFEST counts do not match filesystem (the whole point is accuracy)
- Evaluator audit does not document public method signatures (too shallow)
- Harness audit has no gaps identified (unrealistic for 12-file codebase)
- Sample contracts have < 3 capability_checks
- settings.json left with orphaned hooks

### STOP (fundamental failure -- needs replanning)
- `shipping/skills/` or `shipping/agents/` directories do not exist (structural problem)
- WorkflowEvaluator or TrajectoryEvaluator source files do not exist (kept runners are missing)
- MANIFEST.yaml is invalid YAML
- settings.json is invalid JSON after cleaning
