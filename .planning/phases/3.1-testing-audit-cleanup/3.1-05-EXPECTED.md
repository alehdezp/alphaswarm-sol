# Expected Results: 3.1-05 Evaluation Contract Foundation

---

## Philosophy

This plan exists to define the evaluation contract schema, create 3 sample contracts, and extend the existing assertions library with contract-aware helpers. It serves the phase goal by building the foundation that 3.1c-06 will scale from 3 contracts to 24. Without a schema and samples, 3.1c-06 invents the format from scratch; with them, it becomes "scale the pattern." This plan explicitly does NOT create all 24 evaluation contracts (3.1c-06), does NOT build standalone enforcement (pytest_rules_plugin.py, transcript_validator.py -- those are 3.1c), does NOT implement LLM-based grading (3.1c's responsibility), and does NOT create a parallel enforcement stack. Its scope is strictly: schema, 3 samples, 2 assertion helpers in the existing file, and tests.

## Implementation Reasoning

### Decision 1: JSON Schema Draft 7 for evaluation contracts

- **Selected:** JSON Schema (Draft 7) as the validation format, using the already-installed jsonschema 4.25.1
- **Why:** JSON Schema is the standard for validating JSON structures. Draft 7 is well-supported by jsonschema 4.x. The schema enables both human authoring (readable) and machine validation (automated). jsonschema is already a dependency (4.25.1 installed).
- **Anti-patterns:** Do NOT use a custom validation library. Do NOT skip the `required` field definitions. Do NOT use Draft 2020-12 features that jsonschema 4.x may not fully support.

### Decision 2: Extend existing assertions.py, not create parallel module

- **Selected:** Add `assert_matches_contract` and `assert_reasoning_dimensions_covered` directly to `tests/workflow_harness/lib/assertions.py`
- **Why:** The plan explicitly says "Extend existing assertions.py, not parallel enforcement." Creating a new module (e.g., `contract_assertions.py`) fragments the assertion library and creates import confusion. The existing file already has 7 assertion categories and established patterns.
- **Anti-patterns:** Do NOT create `tests/workflow_harness/lib/contract_assertions.py` or similar. Do NOT create `src/alphaswarm_sol/testing/enforcement/`. Do NOT create pytest plugins for enforcement here (that is 3.1c).

### Decision 3: Code-graded vs model-graded split in assert_matches_contract

- **Selected:** `assert_matches_contract` handles `grader_type: "code"` checks programmatically and marks `grader_type: "model"` checks as "requires_model_grading" (skip or tag)
- **Why:** LLM-based grading is 3.1c's responsibility. This plan provides the code-graded foundation. The split is explicit so 3.1c knows exactly what to build on.
- **Anti-patterns:** Do NOT implement LLM calls in assertions.py. Do NOT silently skip model-graded checks without indicating they were skipped. Do NOT fail on model-graded checks.

### Decision 4: Keyword-based reasoning dimension coverage (not LLM)

- **Selected:** `assert_reasoning_dimensions_covered` uses keyword/pattern matching to check if transcript mentions each dimension
- **Why:** LLM-powered reasoning assessment belongs in 3.1c. This helper provides a fast, deterministic baseline that 3.1c can later replace or enhance with model grading.
- **Anti-patterns:** Do NOT make API calls to language models. Do NOT require environment variables for model access. Keep it deterministic and fast.

## Real vs Placeholder Criteria

| Artifact | Real Indicators | Placeholder Indicators |
|----------|----------------|----------------------|
| `evaluation_contract.schema.json` | Has `$schema` referencing Draft 7; `required` array with 5+ fields; `properties` with type constraints; `capability_checks` items have their own required subfields; enum constraints on category and grader_type | Missing `required` array; no type constraints; accepts any JSON; no nested object validation |
| Sample contracts (.json) | `workflow_id` matches real skill/agent IDs from MANIFEST; `capability_checks` describe specific observable behaviors (not generic); `rule_refs` use IDs from 3.1-04; each check has distinct id/description | Generic workflow_ids; copy-pasted checks across all 3; rule_refs with made-up IDs; < 3 checks |
| `assert_matches_contract` | Iterates capability_checks; distinguishes code vs model grader_type; raises AssertionError with descriptive message on mismatch; handles edge cases (empty result, missing keys) | Returns True unconditionally; no iteration over checks; no grader_type distinction; no error messages |
| `assert_reasoning_dimensions_covered` | Has keyword map for known dimensions (graph_utilization, evidence_quality, etc.); searches transcript for each; raises with dimension name | Returns True unconditionally; no keyword matching; hardcoded pass |
| `test_evaluation_contracts.py` | Tests schema self-validation; tests 3 contracts pass; tests 3+ invalid cases fail; tests both assertion helpers with pass AND fail cases; tests existing helpers still work | Only happy path tests; no invalid contract tests; no assertion fail cases; doesn't test existing helpers |

## Acceptable Deviations

### OK (preserves intent)
- Schema has additional optional fields beyond what plan specifies (e.g., `version`, `description`)
- Sample contracts have > 3 capability_checks (more thorough)
- Assertion helpers include additional utility functions beyond the 2 specified
- Test file has > 150 lines (more test cases is better)
- Keyword maps for reasoning dimensions include synonyms or related terms
- Sample contract locations at `.vrs/testing/contracts/samples/` (plan specifies this, but 3.1-03 also creates samples at `tests/workflow_harness/contracts/samples/`)

### DRIFT (misalignment -- needs correction)
- Schema missing required fields (workflow_id, category, rule_refs, capability_checks, grader_type)
- Assertion helpers created in a new file instead of extending assertions.py
- No invalid-contract rejection tests (only happy path)
- rule_refs in samples do not match 3.1-04 IDs
- Existing 7 assertion categories broken by additions

### STOP (fundamental failure -- needs replanning)
- jsonschema not importable (dependency missing)
- Schema does not validate the sample contracts it ships with
- Standalone enforcement files created (pytest_rules_plugin.py, transcript_validator.py)
- LLM calls made from assertion helpers (scope violation into 3.1c territory)
