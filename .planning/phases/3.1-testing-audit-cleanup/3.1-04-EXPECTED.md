# Expected Results: 3.1-04 Rules Rewrite

---

## Philosophy

This plan exists to rewrite RULES-ESSENTIAL.md and VALIDATION-RULES.md so they target the actual execution model (Agent Teams + claude-code-controller) instead of legacy infrastructure (tmux, jujutsu-workspace). It serves the phase goal by producing testing rules with stable machine-readable IDs that 3.1-05's evaluation contracts can reference. This plan explicitly does NOT build evaluation contracts (3.1-05), does NOT create enforcement code (3.1c), and does NOT delete source code or test files. Its scope is strictly: archive originals, consolidate ~25 rules into ~15 categories with stable IDs, map each to an enforcement mechanism, and preserve all anti-fabrication substance.

## Implementation Reasoning

### Decision 1: Consolidate ~25 rules into ~15 categories via merge groups

- **Selected:** Merge related rules by theme (F1+F2+F1b -> TRANSCRIPT-AUTH, C1+C2+C3 -> METRICS-REALISM, etc.) while preserving all substance
- **Why:** 25 granular rules create redundancy and confusion. Consolidation into thematic categories makes rules easier to reference, enforce, and maintain. The merge groups are designed so each category maps cleanly to one enforcement mechanism.
- **Anti-patterns:** Do NOT silently drop original rule content during merge. Do NOT create categories so broad they lose specificity. Do NOT go below 12 categories (loses important distinctions).

### Decision 2: Stable machine-readable IDs (UPPERCASE-HYPHENATED)

- **Selected:** Each category gets a unique ID like `EXEC-INTEGRITY`, `TRANSCRIPT-AUTH`, `GROUND-TRUTH` using uppercase-hyphenated format
- **Why:** Evaluation contracts in 3.1-05 need to reference rules via `rule_refs: ["EXEC-INTEGRITY", "GROUND-TRUTH"]`. IDs must be stable (not change between edits), machine-readable (parseable by code), and descriptive (human-understandable).
- **Anti-patterns:** Do NOT use numeric IDs (fragile to reordering). Do NOT use spaces or special characters. Do NOT change IDs after this plan (they become a shared namespace).

### Decision 3: Enforcement mapping as first-class section

- **Selected:** Every rule category explicitly states its enforcement mechanism (hook type, controller check, or code-based grader)
- **Why:** Rules without enforcement mechanisms are aspirational documentation. The enforcement mapping makes the rules actionable and tells 3.1c exactly what infrastructure to build for each rule.
- **Anti-patterns:** Do NOT leave any category without an enforcement mechanism. Do NOT invent enforcement mechanisms that do not exist in the Agent Teams / controller model.

### Decision 4: Content preservation with word-count validation

- **Selected:** Each merged category must contain >= 60% of the sum of original rule word counts
- **Why:** Merging is about consolidation, not deletion. The word-count threshold ensures substance is preserved, not just rule titles.
- **Anti-patterns:** Do NOT reduce a 500-word rule set to a 50-word category. Do NOT preserve only titles while dropping definitions and examples.

## Acceptable Deviations

### OK (preserves intent)
- Exact category count differs from 15 (range 12-18 is acceptable)
- Category names differ from examples (e.g., `EXECUTION-PROOF` instead of `EXEC-INTEGRITY`) as long as they are stable, uppercase-hyphenated, and descriptive
- Additional categories beyond the 5 core merge groups (more granularity where warranted)
- Enforcement mechanisms include types not listed in the plan (e.g., "log-based validation")
- Word count of merged categories exceeds 100% of originals (expansion is fine)

### DRIFT (misalignment -- needs correction)
- Rule IDs use different format than UPPERCASE-HYPHENATED (breaks machine parsing)
- Anti-fabrication rules (nonce, hash, duration, evidence) missing from merged output
- No enforcement mapping table (rules without mechanisms)
- RULES-ESSENTIAL.md and VALIDATION-RULES.md use different ID namespaces
- Category count < 12 or > 18

### STOP (fundamental failure -- needs replanning)
- Legacy infrastructure terms (tmux, jujutsu) remain in rewritten rules
- Original rule substance silently dropped (merged category is trivially shorter than originals)
- Archives are not byte-identical to originals (rollback compromised)
- Rules reference execution model that does not exist (neither legacy nor Agent Teams)
