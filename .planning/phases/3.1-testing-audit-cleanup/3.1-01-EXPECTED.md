## Philosophy

This plan exists to safely remove ~15K LOC of dead testing code by first building a rigorous import graph that proves each file is truly unused. It serves the phase goal of removing dead testing infrastructure by establishing the evidentiary foundation (import graph) before any destructive action. This plan explicitly does NOT touch legacy config/registry cleanup (that is 3.1-02), does NOT rewrite testing rules (3.1-04), and does NOT delete test files in `tests/` (that is 3.1-06). Its scope is strictly: analyze `src/alphaswarm_sol/testing/`, delete dead source code, and rewrite `__init__.py`.

## Implementation Reasoning

### Decision 1: AST-based import analysis with barrel re-export exclusion

- **Selected:** Custom `ast` module script that parses all Python files and builds a symbol-level caller map, explicitly excluding `testing/__init__.py` barrel re-exports from caller counts
- **Why:** The 456-line `__init__.py` re-exports ~150 symbols, making naive `grep` or IDE "find usages" analysis show everything as live. AST parsing provides deterministic, reproducible results. Excluding `__init__.py` barrel imports is the key insight that unlocks safe deletion.
- **Anti-patterns:** Do NOT use `grep` for import analysis (misses aliased imports, catches comments). Do NOT count `__init__.py` re-exports as "real callers." Do NOT use IDE find-references (non-reproducible).

### Decision 2: Four-category classification (TRULY_DEAD / TEST_ONLY / PRODUCTION_DEPENDENT / TRANSITIVE)

- **Selected:** Each file gets one of four categories based on its callers
- **Why:** Binary alive/dead is insufficient. TEST_ONLY files are alive but only for testing. TRANSITIVE files may appear alive but only because other dead files import them. The four categories capture the full dependency reality.
- **Anti-patterns:** Do NOT classify a file as PRODUCTION_DEPENDENT just because `__init__.py` imports it. Do NOT assume TRANSITIVE files are safe to keep without checking if their callers are all dead.

### Decision 3: Delete-after-evidence, not delete-then-verify

- **Selected:** Build complete import graph first, review it, then delete only files classified TRULY_DEAD
- **Why:** Reversing the order (delete optimistically, fix errors) risks cascading breakage. Evidence-first ensures every deletion is justified.
- **Anti-patterns:** Do NOT delete files based on "seems unused" intuition. Do NOT skip running the import graph before deletion.

## Acceptable Deviations

### OK (preserves intent)
- Exact LOC deleted differs from ~15K estimate (actual count from import graph is authoritative)
- `__init__.py` has 25-45 lines instead of exactly 20-30 (as long as < 50 and exports only verified symbols)
- Additional symbols beyond the 8 CLI-consumed ones appear as PRODUCTION_DEPENDENT (import graph may discover more)
- `harness/` directory kept if import graph shows it has live callers
- Import graph script uses additional heuristics beyond pure AST (e.g., dynamic import detection)

### DRIFT (misalignment -- needs correction)
- Files deleted that the import graph classifies as PRODUCTION_DEPENDENT or TEST_ONLY
- `__init__.py` still > 100 lines after rewrite
- Import graph does not exclude `__init__.py` barrel re-exports (the core insight is missing)
- Any of the 8 CLI-consumed symbols missing from rewritten `__init__.py`

### STOP (fundamental failure -- needs replanning)
- CLI commands fail after deletion (`alphaswarm --help` or `build-kg` broken)
- pytest discovers > 5 new failures (cascading breakage from wrong deletions)
- Import graph script does not run or produces invalid JSON
- Files deleted without import graph evidence (deletion before analysis)
