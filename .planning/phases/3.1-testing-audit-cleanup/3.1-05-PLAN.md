1. **Create `.vrs/testing/schemas/evaluation_contract.schema.json`** (JSON Schema Draft 7):
   Required fields (from context.md locked decision):
   - `workflow_id` (string, required): Unique identifier like "skill-vrs-audit" or "agent-vrs-attacker"
   - `category` (string, enum: ["skill", "agent", "orchestrator"], required)
   - `rule_refs` (array of strings, required, minItems: 1): References to stable rule IDs from 3.1-04 (e.g., "EXEC-INTEGRITY", "GROUND-TRUTH")
   - `capability_checks` (array of objects, required, minItems: 1): Each check has:
     - `id` (string, required): Machine-readable check ID
     - `description` (string, required): What to verify
     - `expected_behavior` (string, required): Observable outcome
     - `grader_type` (string, enum: ["code", "model"], required): How to evaluate
   - `reasoning_dimensions` (array of strings): Which trajectory dimensions to assess (e.g., "graph_utilization", "evidence_quality")
   - `evidence_requirements` (array of objects): Each requirement has:
     - `type` (string, required): "transcript", "graph_nodes", "code_locations", "tool_output"
     - `description` (string, required)
   - `grader_type` (string, enum: ["code", "model", "hybrid"], required): Overall grading approach
   - `metadata` (object): Optional additional context

   The schema MUST validate that:
   - `capability_checks` is non-empty
   - `rule_refs` contains only valid rule ID patterns (uppercase + hyphens)
   - Each capability check has all required subfields

2. **Create 3 sample evaluation contracts:**

   **`skill-audit.json`** (for /vrs-audit skill):
   ```json
   {
     "workflow_id": "skill-vrs-audit",
     "category": "skill",
     "rule_refs": ["EXEC-INTEGRITY", "GROUND-TRUTH", "REPORT-INTEGRITY"],
     "capability_checks": [
       {
         "id": "builds-kg-before-analysis",
         "description": "Audit builds knowledge graph before any vulnerability analysis",
         "expected_behavior": "build-kg CLI command appears in transcript before any finding is reported",
         "grader_type": "code"
       },
       {
         "id": "produces-evidence-findings",
         "description": "Each finding includes graph node IDs and code locations",
         "expected_behavior": "Every finding object has non-empty evidence_nodes[] and code_locations[]",
         "grader_type": "code"
       },
       {
         "id": "uses-vulndocs-patterns",
         "description": "Analysis references VulnDocs patterns for known vulnerability types",
         "expected_behavior": "At least one finding references a vulndoc pattern ID",
         "grader_type": "model"
       }
     ],
     "reasoning_dimensions": ["graph_utilization", "evidence_quality", "pattern_coverage"],
     "evidence_requirements": [
       {"type": "transcript", "description": "Full Claude Code session transcript"},
       {"type": "graph_nodes", "description": "BSKG node IDs referenced in findings"},
       {"type": "tool_output", "description": "build-kg and query command outputs"}
     ],
     "grader_type": "hybrid"
   }
   ```

   **`agent-attacker.json`** (for vrs-attacker agent): At least 3 capability checks covering attack vector identification, exploit path construction, and evidence citation with graph node IDs.

   **`orchestrator-debate.json`** (for debate flow): At least 3 capability checks covering agent spawning, verdict collection, and verifier arbitration.

   All sample contracts must use real skill/agent IDs and reference rule IDs from 3.1-04.

2. **Extend assertions.py** with 2 new helpers (do NOT create a separate file):

   **`assert_matches_contract(result: dict, contract: dict) -> None`:**
   - Takes a workflow result dict and an evaluation contract dict
   - For each `capability_check` in the contract:
     - Verify the result contains evidence relevant to the check
     - For `grader_type: "code"`: check programmatically (e.g., key exists, non-empty)
     - For `grader_type: "model"`: skip (model grading is 3.1c's responsibility) or mark as "requires_model_grading"
   - Raises `AssertionError` with clear message on first mismatch
   - Returns None on success

   **`assert_reasoning_dimensions_covered(transcript: str, contract: dict) -> None`:**
   - Takes a workflow transcript string and an evaluation contract dict
   - For each `reasoning_dimension` in the contract:
     - Verify the transcript contains evidence of that dimension being exercised
     - Use keyword/pattern matching (not LLM -- LLM grading is 3.1c's job)
   - Raises `AssertionError` with dimension name on first uncovered dimension
   - Returns None on success

   **Integration rules:**
   - Add to the SAME file (assertions.py), not a new module
   - Follow existing code style and patterns
   - Import jsonschema at the top (already available: jsonschema 4.25.1)
   - Existing 7 assertion categories must continue to work unchanged

3. **Create `tests/test_evaluation_contracts.py`** (~150 lines):
   - Test schema self-validation: `jsonschema.Draft7Validator.check_schema(schema)`
   - Test all 3 sample contracts validate against schema
   - Test invalid contracts are rejected:
     - Missing `workflow_id` -> ValidationError
     - Empty `capability_checks` -> ValidationError
     - Invalid `grader_type` -> ValidationError
   - Test `assert_matches_contract`:
     - Pass case: result with all required evidence -> no error
     - Fail case: result missing evidence for a code-graded check -> AssertionError
   - Test `assert_reasoning_dimensions_covered`:
     - Pass case: transcript containing keywords for all dimensions -> no error
     - Fail case: transcript missing a dimension -> AssertionError
   - Test existing assertion categories still work (import and call at least 2 existing helpers)
