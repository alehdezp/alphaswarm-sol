## Philosophy

This plan exists to run all 7 final verification checks as a single composite gate, proving that Phase 3.1 achieved its goal: dead code removed, legacy references gone, shipping manifest created, rules rewritten, evaluation contract foundation built, tests clean, pipeline working. It serves the phase goal by being the final gatekeeper -- individual plan gates can pass while the overall system is inconsistent; this composite check validates the cumulative effect. This plan explicitly does NOT fix issues discovered (those require re-executing prior plans), does NOT create new infrastructure, and does NOT skip checks or fabricate results. Its scope is strictly: run 7 checks, record real results, and mark the phase complete only if 7/7 pass.

## Implementation Reasoning

### Decision 1: Sequential execution of all 7 checks with individual timing

- **Selected:** Run each check sequentially, capturing start/end timestamps and stdout/stderr for each
- **Why:** Sequential execution ensures each check runs in a clean state. Individual timing prevents a fast check from hiding a slow/failing one. Timestamps prove real execution (fabricated reports have identical timestamps).
- **Anti-patterns:** Do NOT run checks in parallel (may interact). Do NOT use cached results from prior plan gates (re-execute fresh). Do NOT report timestamps without actually timing.

### Decision 2: Anti-fabrication enforcement (> 30 second duration, real stdout)

- **Selected:** Total duration must exceed 30 seconds; each check's `stdout_excerpt` must contain actual command output
- **Why:** A real pytest run + build-kg + grep sweep takes > 30 seconds. If total_duration_ms < 30000, the checks were fabricated (copy-pasted from prior reports). Real stdout excerpts contain actual paths, counts, and output that vary per execution.
- **Anti-patterns:** Do NOT copy stdout from prior gate reports. Do NOT template the output. Do NOT fabricate timestamps.

### Decision 3: Conditional STATE.md update (only on 7/7 pass)

- **Selected:** Update STATE.md to mark Phase 3.1 DONE only if all 7 checks pass. If any fails, document the failure and do NOT mark complete.
- **Why:** Premature completion marking unblocks downstream phases that depend on Phase 3.1 guarantees. A 6/7 pass is not "basically done" -- the failing check may indicate a systematic issue.
- **Anti-patterns:** Do NOT mark phase complete on 6/7 pass. Do NOT silently skip a failing check. Do NOT modify STATE.md without all checks passing.

### Decision 4: Composite gate references all prior gate reports

- **Selected:** The machine gate report (3.1-07.json) references all prior gate reports (3.1-01 through 3.1-06)
- **Why:** The composite gate is the apex of the gate chain. It must reference its dependencies so a reviewer can trace any check back to the plan that produced the underlying artifact.
- **Anti-patterns:** Do NOT create the composite gate in isolation (it must reference prior gates). Do NOT re-verify things that prior gates already verified in detail (run the check, reference the detailed gate).

## Acceptable Deviations

### OK (preserves intent)
- Total duration significantly > 30 seconds (60-180 seconds is normal for full test + build-kg)
- Additional checks beyond the 7 specified (supplementary verification)
- Check order differs from 1-7 (as long as all 7 are present)
- stdout_excerpt is truncated for very long outputs (as long as it shows real content)
- STATE.md update includes additional notes about phase accomplishments

### DRIFT (misalignment -- needs correction)
- Any check skipped or not actually executed
- Total duration < 30 seconds (fabrication indicator)
- stdout_excerpt contains template text instead of actual output
- Phase marked DONE but one or more checks show "fail"
- Verification report has fewer than 7 entries

### STOP (fundamental failure -- needs replanning)
- Multiple checks fail (indicates prior plans did not achieve their goals)
- Check 2 (tests pass) fails (systemic breakage)
- Check 3 (pipeline works) fails (product broken)
- Verification report not created (no evidence of checks running)
- STATE.md marked DONE without running checks
