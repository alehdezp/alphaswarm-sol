# Expected Results: 3.1b-06b Seed Corpus — 15-20 Committed Adversarial Projects

---

## Fundamental Deliverable

**This plan commits the initial hostile corpus: adversarial Solidity projects designed to BREAK the engine.** Each project embeds multiple security vulnerabilities in realistic business logic with creative obfuscation. These are Jujutsu fixtures — immutable baseline projects that every future test run references.

This corpus is the ground truth substrate that Phase 3.1c needs to evaluate skill, agent, and orchestrator capabilities. Without it, 3.1c has nothing to test against.

---

## Alignment Criteria

### 1. Sufficient Corpus Size

**The question:** Are there enough projects to exercise the breadth of our detection capabilities while remaining manageable?

**Aligned if:** The corpus has enough projects (ballpark 15-20) to cover the major vulnerability categories, adversarial types, and include false-positive controls — without scope creep into dozens of redundant projects.

**Misaligned if:** Too few projects (under ~12) to achieve meaningful coverage, or too many (30+) indicating scope creep. The sweet spot is enough to cover the testing matrix without waste.

### 2. Multi-Vulnerability Density Per Project

**The question:** Does each project test multiple detection capabilities in one shot — like a real codebase with multiple issues?

**Aligned if:** Non-FP-control projects contain multiple distinct vulnerability patterns (different pattern IDs, not the same pattern repeated). This tests realistic multi-issue codebases and is more efficient than single-pattern projects.

**Misaligned if:** Projects have only 1-2 vulnerabilities (defeats the multi-pattern purpose), or "distinct" means the same pattern in different functions (not really distinct).

### 3. All Three Adversarial Categories Represented

**The question:** Does the corpus exercise name obfuscation (A), protocol complexity (B), AND false-positive resistance (C)?

**Aligned if:** Multiple projects exist for each adversarial category. The corpus isn't skewed toward one type of testing.

**Misaligned if:** An entire adversarial category is missing, or only one project represents a category.

### 4. False-Positive Controls Exist

**The question:** Are there projects with zero real vulnerabilities — safe code that LOOKS dangerous?

**Aligned if:** Multiple FP control projects exist with complex-looking but actually safe patterns. These are essential for precision measurement in 3.1c.

**Misaligned if:** No FP controls (precision measurement impossible), or FP controls are trivially safe (no adversarial pressure on the detector).

### 5. Meaningful Pattern Coverage Across Corpus

**The question:** Does the corpus exercise a broad range of our vulnerability pattern catalog, not just a handful?

**Aligned if:** When aggregating all ground truth across projects, a substantial portion of our pattern catalog is covered, distributed across major vulnerability categories (access-control, reentrancy, oracle, dos, logic).

**Misaligned if:** All projects use the same small set of patterns, or entire vulnerability categories are unrepresented.

### 6. Projects Compile and Have Valid Ground Truth

**The question:** Is every project actually usable — compiles with solc and has accurate ground truth?

**Aligned if:** Both vulnerable and safe variants compile. Ground truth references actual function names and line ranges in the generated code (not placeholders). Each finding has an expected reasoning chain.

**Misaligned if:** Projects don't compile, ground truth has "TODO" placeholders, function names don't match actual code, or line ranges are wrong.

### 7. Obfuscation Is Non-Trivial

**The question:** Would the vulnerability type be non-obvious from names alone in obfuscated projects?

**Aligned if:** Function/variable names in Category A/B projects don't reveal the vulnerability type. A human reviewer can't immediately guess the vulnerability class from names.

**Misaligned if:** Functions named `vulnerableWithdraw()`, `unsafeTransfer()`, or variables named `exploit_here`.

### 8. Projects Are Jujutsu Fixtures

**The question:** Is every project initialized as a Jujutsu repository for workspace isolation?

**Aligned if:** Every project directory contains a `.jj/` subdirectory with committed state. Workspaces can be added and forgotten.

**Misaligned if:** No `.jj/` directories, or only `.git/` exists.

### 9. Projects Look Like Real Codebases

**The question:** Do projects have realistic structure — multiple contracts, interfaces, libraries, imports?

**Aligned if:** Projects have multi-file structures with realistic DeFi/governance/token patterns, imports between contracts, interfaces.

**Misaligned if:** Single-file toy contracts with no business context.

## Drift Detection

### OK (preserves intent)
- Slightly different project count within reasonable range
- Different project names than suggested
- Additional fields in ground truth beyond the required schema
- Safe variants organized differently (same dir vs separate dir)
- Different solc versions as long as all 0.8.x

### DRIFT (needs correction)
- Significantly fewer projects than planned (below ~12)
- Only 1-4 vulnerabilities per project (below multi-pattern threshold)
- A major adversarial category entirely missing
- Fewer than 2 FP control projects
- Ground truth with placeholder function names or "TODO" entries
- Projects that don't compile
- Missing Jujutsu initialization

### STOP (fundamental failure)
- Corpus has fewer than ~8 projects (insufficient for any meaningful testing)
- All projects have single vulnerabilities (defeats the entire corpus strategy)
- Zero false-positive controls
- Projects don't compile at all (generation system broken)
- No ground truth files
- No Jujutsu initialization (workspace isolation model fails)

## Reviewer Guidance

Focus your review on:
1. **Does the corpus cover the testing matrix?** Categories, adversarial types, FP controls — breadth over depth.
2. **Are projects realistic?** Multi-file, business logic, not toy examples.
3. **Is ground truth accurate?** Spot-check: do function names and lines match actual code?
4. **Do projects compile?** Run build verification on a sample.
5. **Is obfuscation real?** Can't guess vulnerability type from names alone.

Do NOT focus on: exact project counts, exact pattern count thresholds, specific YAML field names in ground truth, exact number of contracts per project, or specific project naming conventions.
