**Part B — Adversarial Obfuscation Taxonomy (`adversarial-taxonomy.md`):**

Document 3 adversarial categories:

**Category A: Name Obfuscation** (tests semantic operation detection vs name shortcuts)
- Rename functions (withdraw -> processRequest)
- Misleading names (safeTransfer that isn't safe)
- Dead code red herrings (safe-looking functions that are never called)
- Variable aliasing (same value through multiple variable names)
- Parameter reordering (swap expected parameter positions)
- At least 5 techniques with Solidity code examples

**Category B: Protocol Complexity** (tests cross-contract + temporal reasoning)
- Multi-contract split vulnerabilities
- Proxy/delegate patterns hiding logic
- State machine violations across transactions
- Inherited vulnerability across contract hierarchy
- Library-mediated exploits (vulnerable library, safe-looking caller)
- At least 5 techniques with Solidity code examples

**Category C: Honeypot Inversions** (tests false positive resistance)
- Safe code with dangerous names (function called `withdraw` but properly guarded)
- Complex but correct access control (multiple checks that look suspicious)
- Defense-in-depth that looks vulnerable (reentrancy guard + CEI + access control)
- Unreachable vulnerable code paths (dead code that contains real patterns)
- Deliberately misleading comments (comments say "unsafe" but code is safe)
- At least 5 techniques with Solidity code examples

Each technique: difficulty rating, Solidity example, compilation requirements.

6. Create ground truth with expected_reasoning_chain per pattern
7. Generate safe variant (same structure, vulnerabilities removed)
8. Compile both variants with solc and validate
