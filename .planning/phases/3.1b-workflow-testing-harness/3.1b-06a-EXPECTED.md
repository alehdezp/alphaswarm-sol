# Expected Results: 3.1b-06a Corpus Generation System — Guidelines + Opus Agent

---

## Fundamental Deliverable

**This plan builds the complete adversarial corpus generation system** — both the "recipe book" (guidelines, pattern catalog, taxonomy) AND the "chef" (Opus-powered Claude Code subagent that generates hostile Solidity projects).

The goal is NOT to test the happy path. The goal is to BUILD PROJECTS THAT BREAK THE ENGINE — obfuscated names, multi-pattern contracts, misleading structure, realistic project layouts.

---

## Alignment Criteria

### 1. Pattern Catalog Covers the Vulnerability Knowledge Base

**The question:** Does the catalog make our vulnerability patterns available as generation-ready specifications?

**Aligned if:** The catalog covers the breadth of active vulndocs patterns, each with behavioral specifications (semantic operations, ordering constraints, tier) that enable the generation agent to produce realistic vulnerable code. Generated programmatically from vulndocs to prevent drift.

**Misaligned if:** Catalog is a small hand-written subset, or patterns lack behavioral specs (just names and IDs), or the catalog has no connection to actual vulndocs files.

### 2. Adversarial Taxonomy Covers Three Distinct Failure Modes

**The question:** Does the taxonomy define adversarial techniques that test fundamentally different capabilities?

**Aligned if:** Three categories exist:
- **Name obfuscation** — tests whether detection works on semantic operations, not function names
- **Protocol complexity** — tests cross-contract and temporal reasoning
- **Honeypot inversions** — tests false-positive resistance (safe code that looks dangerous)

Each category has multiple concrete techniques with examples.

**Misaligned if:** Only one or two categories, or categories are generic ("hard" vs "easy") without targeting specific failure modes, or techniques lack concrete Solidity examples.

### 3. Combination Rules Enable Multi-Pattern Projects

**The question:** Can the generation agent produce projects with 5-10 distinct vulnerabilities that realistically coexist?

**Aligned if:** Rules define which patterns are compatible, which interfere with each other, and target complexity levels for multi-pattern projects. The 5-10 vulnerability minimum per project is documented as a requirement.

**Misaligned if:** No combination guidance exists (agent generates single-pattern projects), or combinations are random without compatibility analysis.

### 4. Generation Agent Produces Realistic, Compilable Projects

**The question:** Does the Opus-powered subagent produce complete Solidity projects — not toy contracts — from pattern specifications?

**Aligned if:** The agent:
- Uses Opus model for maximum creative reasoning
- Reads the guidelines (catalog, taxonomy, rules, pipeline)
- Produces multi-file projects with realistic business logic
- Generates both vulnerable and safe variants
- Runs solc compilation verification
- Creates ground truth with expected reasoning chains
- Initializes Jujutsu workspace

**Misaligned if:** Agent produces single-file toy contracts, doesn't reference the guidelines, doesn't verify compilation, or produces no safe variants.

### 5. Training Data Contamination Prevention

**The question:** Does the system ensure generated projects are NOVEL — not copies of known vulnerable contracts?

**Aligned if:** The agent instructions explicitly prohibit Ethernaut, Damn Vulnerable DeFi, SWC test cases, and other well-known vulnerable contract datasets. Generation must produce original business logic.

**Misaligned if:** No contamination protocol exists, or generated projects are recognizable as existing challenge contracts.

### 6. Programmatic Invocation Works

**The question:** Can the generation agent be invoked programmatically with different configurations?

**Aligned if:** An orchestration script accepts pattern selection, adversarial category, complexity tier, and output location as parameters, and spawns the agent via CLI.

**Misaligned if:** No script exists (agent can only be invoked manually), or the script has hardcoded generation parameters.

### 7. At Least One Test Generation Succeeds

**The question:** Has the system actually produced a working project?

**Aligned if:** At least one generated project exists with compilable contracts, valid ground truth, and obfuscated patterns. This is proof the system works, not just that files exist.

**Misaligned if:** No test generation was attempted, or the generated output doesn't compile.

## Drift Detection

### OK (preserves intent)
- Different catalog structure as long as all patterns are accessible
- More adversarial techniques than the minimum
- More combination rules than specified
- Pipeline with more or fewer steps if the substance is equivalent
- Agent produces additional files beyond the minimum set

### DRIFT (needs correction)
- Catalog covers significantly fewer patterns than available in vulndocs
- Fewer than 3 adversarial categories or sparse techniques per category
- Agent uses a weaker model than Opus (insufficient creativity)
- Agent doesn't read the guidelines (hallucinates patterns)
- Projects have fewer than 5 distinct vulnerability patterns each
- No safe variants or no compilation check
- No training data contamination prevention

### STOP (fundamental failure)
- No pattern catalog (agent has nothing to generate from)
- No adversarial taxonomy (no obfuscation strategy)
- No generation agent (can't produce projects)
- Agent produces static templates, not novel business logic
- Generated projects don't compile at all
- No orchestration script (can't invoke programmatically)

## Reviewer Guidance

Focus your review on:
1. **Does the catalog actually cover our patterns?** Not a tiny hand-written sample — generated from vulndocs.
2. **Are the three adversarial categories distinct?** Each should test a different detection capability.
3. **Does the agent produce REALISTIC projects?** Multi-file, business logic, not `contract Vulnerable { }`.
4. **Does at least one generation succeed end-to-end?** Compilable, ground truth valid, obfuscated.

Do NOT focus on: exact pattern counts, specific YAML field names, number of techniques per category, exact argument names in the orchestration script, or whether the pipeline has exactly 8 steps.
