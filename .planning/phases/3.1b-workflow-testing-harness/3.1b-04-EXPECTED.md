# Expected Results: 3.1b-04 Agent Teams Interactive Framework + Jujutsu Workspace Isolation

---

## Fundamental Deliverable

**This plan builds the primary testing mechanism for Phase 3.1c** — a team lifecycle management API with Jujutsu-based workspace isolation. It wraps Claude Code's native Agent Teams concepts into a Python test harness and adds snapshot-capable, isolated workspaces for repeatable test runs.

**Critical distinction:** This builds the data model and lifecycle patterns for test orchestration. The actual `claude` subprocess integration is proven in 3.1b-07 (smoke test). This plan delivers the SHAPE of the API; the smoke test proves it works with real Claude Code.

---

## Alignment Criteria

### 1. Team Lifecycle Management

**The question:** Does a manager class provide the full team lifecycle: create team, spawn teammates, exchange messages, monitor, shutdown, and cleanup?

**Aligned if:** A `TeamManager` (or equivalent) provides create/spawn/message/shutdown/cleanup methods, tracks active teams, and ensures no orphan resources after cleanup.

**Misaligned if:** Only partial lifecycle (e.g., can create but not cleanup), or cleanup is manual rather than automatic (e.g., no context manager).

### 2. Full Message Content Capture

**The question:** When messages are exchanged between agents, is the FULL message body captured — not just metadata?

**Aligned if:** Message capture stores the actual content text, sender, recipient, timestamp, and message type. 3.1c interactive debrief can analyze what agents actually said.

**Misaligned if:** Only metadata stubs are stored (message ID, timestamp) without the actual content. This would make 3.1c debrief impossible.

### 3. Team Observation Integration

**The question:** Can the team manager produce a `TeamObservation` that links per-agent transcripts to team-level events?

**Aligned if:** A method builds the observation model (from 3.1b-02) by reading agent transcript files and correlating them with captured messages.

**Misaligned if:** Returns disconnected data (transcripts without message correlation), or returns raw dicts instead of using the 3.1b-02 data model.

### 4. Jujutsu Workspace Isolation

**The question:** Can each test run get its own isolated workspace that doesn't contaminate the committed state?

**Aligned if:** `WorkspaceManager` provides create/forget/rollback/snapshot methods using Jujutsu (`jj`) commands. Checks for `jj` availability before running commands. Raises clear errors when `jj` is missing. `forget_workspace` is idempotent.

**Misaligned if:** No workspace isolation exists, or `jj` commands are called without availability checks, or workspace cleanup is fragile (errors on already-forgotten workspaces).

### 5. Sandbox Support

**The question:** Can production skills/agents be copied into a test workspace so that team tests use real shipped assets?

**Aligned if:** Setup copies `.claude/skills/` and `.claude/agents/` into the workspace, and teardown restores the workspace to its pre-test state.

**Misaligned if:** No sandbox support (teams can't access skills), or teardown doesn't clean up (sandbox files persist after tests).

### 6. Backward Compatibility

**The question:** Do all existing harness tests still pass after these extensions?

**Aligned if:** Extensions to `WorkspaceManager` don't break existing API. All existing tests pass unchanged.

**Misaligned if:** Existing tests fail, or existing methods were replaced rather than extended.

## Drift Detection

### OK (preserves intent)
- Different method names than plan pseudocode
- Additional helper methods
- More granular error handling
- Different subprocess invocation style
- Different defaults for team naming

### DRIFT (needs correction)
- Implementing real `claude` subprocess calls in TeamManager (that's 3.1b-07's job)
- Message capture storing only metadata without full content
- Jujutsu methods not checking for `jj` availability
- `forget_workspace()` raising errors instead of being idempotent
- Tests running real `jj` or `claude` subprocesses instead of mocking

### STOP (fundamental failure)
- No message capture implemented
- No Jujutsu workspace methods
- No TeamObservation integration with 3.1b-02
- Context manager doesn't clean up (orphan teams persist)
- No sandbox support
- Existing harness tests fail

## Reviewer Guidance

Focus your review on:
1. **Does the full team lifecycle work?** Create → spawn → message → observe → shutdown → cleanup — all stages covered?
2. **Are messages captured with FULL content?** Not just metadata — the actual text the agents exchanged.
3. **Does workspace isolation actually use Jujutsu?** Not just filesystem copies — proper `jj workspace add` for isolation.
4. **Is cleanup reliable?** Context manager ensures no orphan resources even on exceptions.

Do NOT focus on: exact method counts, specific parameter types, subprocess command syntax details, or matching plan pseudocode.
