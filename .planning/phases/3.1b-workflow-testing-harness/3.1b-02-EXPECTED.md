# Expected Results: 3.1b-02 TranscriptParser Extensions + OutputCollector

---

## Fundamental Deliverable

**This plan extends the existing TranscriptParser with BSKG query extraction and builds the core observation/aggregation data model.** It is the data foundation that 4 blocking 3.1c consumers depend on: observation parsing (3.1c-03), graph value scoring (3.1c-04), evaluation execution (3.1c-08), and orchestrator flows (3.1c-11).

The absolute constraint: **All existing parser tests MUST still pass.** This is extension-only work.

---

## Alignment Criteria

### 1. BSKG Query Extraction

**The question:** Can the parser identify and extract BSKG-specific queries (alphaswarm commands) from transcript Bash tool calls, distinguishing them from non-BSKG Bash calls?

**Aligned if:** A dedicated extraction method returns structured query objects that classify query types (build-kg, query, pattern-query, etc.), extract the query text, and capture result snippets. Non-alphaswarm Bash calls are filtered out.

**Misaligned if:** All Bash calls are treated as BSKG queries (noise), or BSKG extraction doesn't exist, or the method returns unstructured dicts instead of typed objects.

### 2. Graph Citation Rate

**The question:** Can 3.1c-04 get a normalized metric (0.0-1.0) for how often the agent cited BSKG query results in its conclusions?

**Aligned if:** A method returns a float between 0.0 and 1.0 based on a documented heuristic for citation detection. Returns 0.0 (not crash) when there are no queries.

**Misaligned if:** Returns None, int, or crashes on empty queries. Or the heuristic is undocumented, making it impossible for 3.1c to understand what the metric measures.

### 3. Backward-Compatible ToolCall Extension

**The question:** Can timing fields be added to ToolCall without breaking any existing code that constructs ToolCall objects?

**Aligned if:** New fields (timestamp, duration) are optional with defaults so existing callers work unchanged. All 20+ existing parser tests pass without modification.

**Misaligned if:** New fields are required (breaks existing constructors), or a new class replaces ToolCall (breaks all downstream code).

### 4. Team Observation Model

**The question:** Does a data model exist that links per-agent transcripts to team-level events, enabling 3.1c-11 to analyze cross-agent behavior?

**Aligned if:** A `TeamObservation` (or equivalent) model connects individual agent observations to a shared event stream, with methods for looking up agents by role and tracing evidence flow between agents. Methods return empty results gracefully when data is missing.

**Misaligned if:** No cross-agent model exists, or agent data and events are disconnected, or methods raise exceptions on missing data instead of returning empty/None.

### 5. Output Collection

**The question:** Can a single aggregated view of a scenario run be produced, containing tool sequences, BSKG queries, timing, cost, and failure notes?

**Aligned if:** An `OutputCollector` (or equivalent) takes a parsed transcript and produces a collected output with all the fields 3.1c-08 needs, plus a human-readable summary method.

**Misaligned if:** No aggregation exists, or the collector just passes through None values without actually extracting data from the transcript.

## Drift Detection

### OK (preserves intent)
- Different naming in extraction logic
- Additional fields beyond what's specified on any data model
- More sophisticated citation heuristic than simple substring matching
- Different file organization (multiple modules vs single)
- Additional helper methods on any data model

### DRIFT (needs correction)
- New ToolCall fields are required instead of optional (breaks existing callers)
- `get_bskg_queries()` returns None instead of empty list (breaks iteration)
- `graph_citation_rate()` returns int or None instead of float
- OutputCollector doesn't actually extract data from transcript (just wraps None)
- Any existing parser test fails

### STOP (fundamental failure)
- BSKG query extraction doesn't exist (3.1c-04 has no data)
- TeamObservation doesn't link to event stream (3.1c-11 can't correlate)
- OutputCollector doesn't produce a unified model (3.1c-08 has nothing)
- TranscriptParser replaced instead of extended (all downstream breaks)

## Reviewer Guidance

Focus your review on:
1. **Does BSKG extraction actually distinguish alphaswarm commands from general Bash?** Test with mixed transcripts.
2. **Are ALL existing tests untouched and passing?** This is the #1 constraint.
3. **Can downstream consumers iterate over results without crashes?** Empty lists, not None. Floats, not ints.
4. **Does the team observation model actually connect agents to events?** Not just a container of disconnected data.

Do NOT focus on: exact field counts, specific dataclass definitions, implementation of the citation heuristic algorithm, or matching pseudocode from the plan.
