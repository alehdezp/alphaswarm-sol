### Pattern 1: Companion NDJSON Bridge
**What:** Python bridge wrapping Companion's WebSocket NDJSON protocol for programmatic Claude Code session control.
**When to use:** Secondary automation after interactive evaluation proves what "good" looks like.

The bridge should follow the Companion's own architecture: WebSocket for real-time NDJSON streaming, REST for session lifecycle management.

**Key protocol details (from WEBSOCKET_PROTOCOL_REVERSED.md):**
```python
# Source: .planning/research/companion/WEBSOCKET_PROTOCOL_REVERSED.md

# 1. Launch Claude Code in WebSocket mode:
# claude --sdk-url ws://localhost:3456/ws/cli/SESSION_ID \
#        --print --output-format stream-json --input-format stream-json \
#        --verbose -p "placeholder"

# 2. Connection lifecycle:
#    Server <- CLI: system/init (tools, model, session_id, version)
#    Server -> CLI: user message (first prompt)
#    Server <- CLI: assistant messages, control_requests
#    Server -> CLI: control_responses (tool approval)
#    Server <- CLI: result (query complete)

# 3. NDJSON message format (each line is one JSON object + \n):
import json

def send_user_message(ws, content: str, session_id: str) -> None:
    msg = json.dumps({
        "type": "user",
        "message": {"role": "user", "content": content},
        "parent_tool_use_id": None,
        "session_id": session_id,
    }) + "\n"
    ws.send(msg)

def send_tool_approval(ws, request_id: str, tool_input: dict) -> None:
    msg = json.dumps({
        "type": "control_response",
        "response": {
            "subtype": "success",
            "request_id": request_id,
            "response": {
                "behavior": "allow",
                "updatedInput": tool_input,
            },
        },
    }) + "\n"
    ws.send(msg)
```

**Important: `bypassPermissions` mode means `can_use_tool` requests are NEVER sent over WebSocket** -- everything is auto-approved locally. This is the correct mode for unattended regression testing.

### Pattern 4: Anthropic Eval Framework (Tasks + Trials + Graders)
**What:** Evaluation structure from Anthropic's "Demystifying evals for AI agents" blog.
**When to use:** Scenario DSL design (3.1b-05) and corpus creation (3.1b-06).

```yaml
# Source: https://www.anthropic.com/engineering/demystifying-evals-for-ai-agents

# Core concepts:
# - Task: defined inputs + success criteria
# - Trial: one stochastic run of a task
# - Transcript/Trace: complete record (outputs, tool calls, reasoning, intermediate results)
# - Outcome: final environment state (what actually happened vs what agent said)
# - Grader: evaluates transcript/outcome against criteria

# Three grader types:
# 1. Code-based: string match, binary tests, schema validation, outcome verification
#    Pros: fast, reproducible, objective
#    Cons: brittle to valid variations, limited nuance
# 2. Model-based: rubric scoring, NL assertions, pairwise comparison
#    Pros: flexible, handles open-ended outputs
#    Cons: non-deterministic, higher cost, needs calibration
# 3. Human: expert review, gold-standard quality
#    Pros: highest quality
#    Cons: expensive, slow

# Key metrics:
# pass@k = P(at least 1 success in k trials) -- capability measure
# pass^k = P(all k trials succeed) = p^k -- reliability measure
# Example: p=0.75, k=3 -> pass@3 ~98%, pass^3 ~42%

# Implementation guidance:
# - Start with 20-50 tasks from real failures
# - Ensure 2 domain experts would agree on pass/fail
# - Design balanced sets: test when behavior SHOULD and SHOULDN'T occur
# - Isolated clean environments, no state leakage between trials
# - Read transcripts to validate graders measure what matters
```

### Anti-Patterns to Avoid
- **Always-on hooks:** Hooks add overhead and noise. Enable selectively per scenario based on what you're investigating.
- **Mocking Companion/CLI in tests:** The whole point of 3.1b is real infrastructure. Assert real HTTP status codes, real WebSocket connections.
- **Hardcoded 100 scenarios:** 10 curated + dynamic generation guidelines. The guidelines ARE the scalable solution.
- **Automated grading before interactive understanding:** Claude Code evaluates interactively first. Only automate known-good regressions.
- **Single-trial conclusions:** Agent behavior is stochastic. Minimum 3 trials per scenario (pass@k / pass^k).

## Common Pitfalls

### Pitfall 1: Companion REST vs WebSocket Confusion
**What goes wrong:** Companion has a REST API for session management AND a WebSocket bridge for NDJSON streaming. Mixing them up or trying to use REST for real-time interaction.
**Why it happens:** Two protocols serve different purposes but share the same port (3456).
**How to avoid:** REST for session CRUD (create, list, delete). WebSocket for real-time session interaction (prompts, tool approvals, streaming responses).
**Warning signs:** Polling REST endpoints for response data instead of using WebSocket streaming.

### Pitfall 2: bypassPermissions Eliminates can_use_tool Events
**What goes wrong:** Using `bypassPermissions` mode and expecting to see `can_use_tool` control requests over WebSocket.
**Why it happens:** `bypassPermissions` auto-approves locally -- permission requests never reach the wire.
**How to avoid:** If you need to see/approve tool calls programmatically, use `default` mode. For unattended regression, use `bypassPermissions`.
**Warning signs:** WebSocket handler for `can_use_tool` never fires in `bypassPermissions` mode.

### Pitfall 3: Hook Exit Code 2 Not Working for PostToolUse
**What goes wrong:** Using exit code 2 in a PostToolUse hook expecting to block the tool call.
**Why it happens:** PostToolUse fires AFTER the tool already ran. Exit 2 only shows stderr to Claude, doesn't undo the action.
**How to avoid:** Check the "Can block?" column per event. Only PreToolUse, PermissionRequest, UserPromptSubmit, Stop, SubagentStop, TeammateIdle, and TaskCompleted support blocking via exit 2.
**Warning signs:** PostToolUse hook stderr appearing as Claude context but no actual blocking.

### Pitfall 4: Agent Teams Feature Flag Not Set
**What goes wrong:** Trying to use TeamCreate/SendMessage without enabling the experimental flag.
**Why it happens:** Agent Teams are behind `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`.
**How to avoid:** Add to settings.json: `{"env": {"CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"}}`. Verify in settings before team tests.
**Warning signs:** "TeamCreate" tool not available, Claude proposes subagents instead of teams.

### Pitfall 5: Transcript Path Encoding
**What goes wrong:** Transcript paths use URL-encoded project directories. `~/.claude/projects/-Volumes-ex_ssd-home-projects-...` not literal paths.
**Why it happens:** Claude Code encodes the project path into the directory name under `~/.claude/projects/`.
**How to avoid:** Use the `transcript_path` field from hook input (it gives the real path) or read from session metadata.
**Warning signs:** `FileNotFoundError` when constructing transcript paths manually.

### Pitfall 6: settings.json Snapshot Behavior
**What goes wrong:** Modifying hooks in settings.json after session starts, expecting immediate effect.
**Why it happens:** Claude Code snapshots hooks at session startup. Mid-session changes require review in `/hooks` menu.
**How to avoid:** Install hooks BEFORE spawning Claude Code sessions. WorkspaceManager.install_hooks() must run before any CLI invocation.
**Warning signs:** Hook changes not taking effect, no warning about modified hooks.

### Pitfall 7: NDJSON Message Ordering Assumptions
**What goes wrong:** Assuming messages arrive in a strict request-response pattern.
**Why it happens:** Multiple concurrent tool calls, streaming events, and keep_alive messages interleave.
**How to avoid:** Route messages by `type` field, correlate control_request/control_response by `request_id`, filter out `keep_alive`.
**Warning signs:** Missing responses, out-of-order processing, keep_alive messages disrupting parsing.

### CompanionBridge Pattern
```python
# Recommended implementation pattern for companion_bridge.py

import json
import threading
import time
from typing import Any, Callable
import requests
import websocket  # websocket-client library

class CompanionBridge:
    """Python bridge to Companion REST + WebSocket for automated testing."""

    def __init__(self, base_url: str = "http://localhost:3456"):
        self.base_url = base_url
        self.ws_url = base_url.replace("http://", "ws://").replace("https://", "wss://")
        self._session = requests.Session()

    # --- REST API ---
    def list_sessions(self) -> list[dict]:
        resp = self._session.get(f"{self.base_url}/api/sessions")
        resp.raise_for_status()
        return resp.json()

    def create_session(self, cwd: str, model: str = "sonnet",
                       permission_mode: str = "bypassPermissions") -> dict:
        resp = self._session.post(f"{self.base_url}/api/sessions", json={
            "cwd": cwd,
            "model": model,
            "permissionMode": permission_mode,
        })
        resp.raise_for_status()
        return resp.json()

    def delete_session(self, session_id: str) -> None:
        resp = self._session.delete(f"{self.base_url}/api/sessions/{session_id}")
        resp.raise_for_status()

    # --- WebSocket NDJSON ---
    def connect_ws(self, session_id: str,
                   on_message: Callable[[dict], None] | None = None) -> "WSConnection":
        """Connect to a session's WebSocket for NDJSON streaming."""
        url = f"{self.ws_url}/ws/browser/{session_id}"
        return WSConnection(url, on_message)


class WSConnection:
    """WebSocket connection with NDJSON message handling."""

    def __init__(self, url: str, on_message: Callable[[dict], None] | None = None):
        self.url = url
        self._on_message = on_message
        self._messages: list[dict] = []
        self._ws: websocket.WebSocketApp | None = None
        self._thread: threading.Thread | None = None

    def connect(self) -> None:
        self._ws = websocket.WebSocketApp(
            self.url,
            on_message=self._handle_message,
            on_error=self._handle_error,
            on_close=self._handle_close,
        )
        self._thread = threading.Thread(target=self._ws.run_forever, daemon=True)
        self._thread.start()

    def _handle_message(self, ws: Any, data: str) -> None:
        for line in data.strip().split("\n"):
            if not line:
                continue
            try:
                msg = json.loads(line)
                self._messages.append(msg)
                if self._on_message:
                    self._on_message(msg)
            except json.JSONDecodeError:
                pass

    def send_prompt(self, content: str) -> None:
        if self._ws:
            msg = json.dumps({
                "type": "user",
                "message": {"role": "user", "content": content},
                "parent_tool_use_id": None,
                "session_id": "",
            })
            self._ws.send(msg)

    def close(self) -> None:
        if self._ws:
            self._ws.close()

    @property
    def messages(self) -> list[dict]:
        return list(self._messages)
```

### Hook Registration Extension
```python
# Source: Existing workspace.py + official hooks reference

def install_hooks(workspace: Path, hook_configs: list[dict] | None = None) -> None:
    """Install hooks into workspace settings.json.

    Args:
        workspace: Path to workspace directory
        hook_configs: Optional list of hook config dicts, each with:
            - event_name: str (e.g., "SubagentStop", "PreToolUse")
            - command: str (shell command to run)
            - matcher: str | None (regex filter)
            - timeout: int (seconds, default 600)
            - async_: bool (run in background, default False)
    """
    settings_path = workspace / ".claude" / "settings.json"
    settings = _read_json(settings_path) if settings_path.exists() else {}
    hooks = settings.setdefault("hooks", {})

    # Always install the default log_session.py
    _ensure_hook(hooks, "SubagentStop", "python3 .claude/hooks/log_session.py")
    _ensure_hook(hooks, "Stop", "python3 .claude/hooks/log_session.py")

    # Install additional scenario-specific hooks
    if hook_configs:
        for cfg in hook_configs:
            event = cfg["event_name"]
            command = cfg["command"]
            matcher = cfg.get("matcher")
            timeout = cfg.get("timeout", 600)

            hook_entry = {"type": "command", "command": command, "timeout": timeout}
            if cfg.get("async_"):
                hook_entry["async"] = True

            event_hooks = hooks.get(event, [])
            if not isinstance(event_hooks, list):
                event_hooks = []

            # Find or create matcher group
            group = None
            for g in event_hooks:
                if isinstance(g, dict) and g.get("matcher") == matcher:
                    group = g
                    break
            if group is None:
                group = {"hooks": []}
                if matcher:
                    group["matcher"] = matcher
                event_hooks.append(group)

            # Deduplicate within matcher group
            existing_cmds = {h.get("command") for h in group.get("hooks", []) if isinstance(h, dict)}
            if command not in existing_cmds:
                group["hooks"].append(hook_entry)

            hooks[event] = event_hooks

    # Create observations directory
    obs_dir = workspace / ".vrs" / "observations"
    obs_dir.mkdir(parents=True, exist_ok=True)

    with open(settings_path, "w") as f:
        json.dump(settings, f, indent=2)
```

## Open Questions

1. **Agent Teams Debrief via SendMessage** -- **RESOLVED (2026-02-12)**
   - What we know: Teammates go idle after task completion. SendMessage delivers to inbox. Messages injected as user messages into conversation.
   - **RESOLVED:** Agent Teams Deep Research (RQ-1/RQ-2) confirmed: idle agents stay alive as processes in wait state, SendMessage DOES wake them, context is preserved in memory (no compaction unless >80% window used). Three viable debrief approaches documented: (A) TeammateIdle hook with exit code 2, (B) SubagentStop command hook (prompt hooks BROKEN per bug #20221), (C) SendMessage to idle agent (risk: no delivery confirmation for dead agents). See `AGENT-TEAMS-DEEP-RESEARCH.md` and GAP-01-02 resolution.
   - Remaining risk: Cannot distinguish idle from dead agent (SendMessage returns success:true regardless).

2. **Companion REST API Exact Endpoints**
   - What we know: Session CRUD at `/api/sessions`, WebSocket at `/ws/cli/:id` and `/ws/browser/:id`. Git, filesystem, environment management endpoints exist.
   - What's unclear: Full REST API schema isn't documented beyond README. The TypeScript source code (`web/server/routes.ts`) would have exact signatures.
   - Recommendation: Start with session CRUD + WebSocket (sufficient for 3.1b-01). Read `routes.ts` during implementation for additional endpoints.

3. **Hook Input Schema for TeammateIdle/TaskCompleted**
   - What we know: Both receive `teammate_name` and `team_name` (documented). Exit code 2 sends stderr as feedback.
   - What's unclear: Whether `task_id` or task details are included in TeammateIdle input, or only in TaskCompleted.
   - Recommendation: Verify experimentally during 3.1b-03 iteration 3. The official docs show the input schemas clearly.

4. **Solidity Compiler for Test Corpus**
   - What we know: Need `solc` for `build-verification.sh` in each scenario.
   - What's unclear: Which Solidity version(s) the 10 curated contracts should target. Different versions have different features.
