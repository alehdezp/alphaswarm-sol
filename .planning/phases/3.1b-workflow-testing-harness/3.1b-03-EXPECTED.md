# Expected Results: 3.1b-03 Hook Infrastructure (DONE — Verification Only)

---

## Fundamental Deliverable

**This plan verifies that the existing hook infrastructure is 3.1c-ready.** The implementation is already complete from prior iterations. This verification confirms that 3.1c-02 can add 5+ new hooks (PreToolUse, PostToolUse, SubagentStart, etc.) without modifying the core registration architecture.

**The real test:** If 3.1c-02 starts tomorrow, will they be able to call `install_hooks()` with new hook configurations and have them work immediately — no code changes needed?

---

## Alignment Criteria

### 1. Hook Registration Is Extensible (Not Hardcoded)

**The question:** Can new hooks for arbitrary event types be registered without modifying the registration function's source code?

**Aligned if:** `install_hooks()` (or equivalent) treats event types as data passed in by callers, not as a hardcoded list. Adding PreToolUse or any new event is purely a caller-side change.

**Misaligned if:** The function has a hardcoded list of events (SubagentStop, Stop) and adding new events requires editing the function body.

### 2. Hook Deduplication Works

**The question:** Does calling the registration function multiple times with the same hook produce the same result as calling it once?

**Aligned if:** Duplicate detection prevents the same command from appearing multiple times for the same event. Tests demonstrate idempotency.

**Misaligned if:** Repeated calls create duplicate entries in settings.json.

### 3. Multiple Different Hooks Per Event

**The question:** Can multiple distinct hook scripts be registered for the same event type?

**Aligned if:** The settings format supports a list of hooks per event, and registration appends new hooks without replacing existing ones.

**Misaligned if:** Only one hook per event type is supported, or new registrations overwrite previous ones.

### 4. Observations Directory Convention

**The question:** Is there a stable, documented directory where hooks write their output?

**Aligned if:** `.vrs/observations/` is created during workspace setup and is the convention for hook JSONL output.

**Misaligned if:** Hooks write to arbitrary locations, or the directory isn't created until a hook first writes to it.

### 5. Existing Tests Still Pass

**The question:** Has the existing test suite maintained full green status?

**Aligned if:** All harness tests pass with no failures, no newly added xfail markers, no commented-out tests.

**Misaligned if:** Tests were disabled, marked xfail, or modified to pass around regressions.

## Drift Detection

### OK (preserves intent)
- Adding helper methods for debugging (e.g., listing registered hooks)
- Improving error messages or validation
- Adding docstring examples for 3.1c usage
- Increasing default hook timeout if testing shows it's needed

### DRIFT (needs correction)
- Changing the settings.json format (breaks existing tests and 3.1c assumptions)
- Removing the extensibility parameter or changing its signature
- Removing deduplication logic
- Changing the observations directory location

### STOP (needs replanning)
- Replacing the hook API with a different architecture (violates extend-never-replace)
- Introducing external dependencies for hook registration (over-engineering)
- Adding features beyond what 3.1c explicitly needs (feature creep)

## Reviewer Guidance

This is a **verification-only** plan. Focus on:
1. **Is the hook API extensible?** Can you add a new event type from outside the function?
2. **Is registration idempotent?** Call it twice, get same result?
3. **Do all existing tests pass?** Zero regressions.
4. **If gaps are found, were fixes minimal?** No refactoring, no "while we're here" improvements.

Do NOT focus on: exact test counts, specific test function names, exact settings.json structure details, or line-by-line code review. The implementation is DONE — this is about verifying 3.1c readiness, not auditing code quality.
