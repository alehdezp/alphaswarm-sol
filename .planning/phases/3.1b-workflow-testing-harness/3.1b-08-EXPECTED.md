# Expected Results: 3.1b-08 API Contract Design Review

---

## Fundamental Deliverable

**This plan front-loads the 3.1c integration contract design to prevent backtracking during implementation.** It produces explicit Python type signatures for every 3.1c-facing API BEFORE implementation begins. If any contract is underspecified, 3.1c planning will expose the gap immediately — not weeks later during execution.

This plan delivers specifications, not working code. It is Wave 0 (first executed) because all subsequent plans implement against these contracts.

---

## Alignment Criteria

### 1. Every 3.1c Plan Maps to Specific API Contracts

**The question:** Can a 3.1c implementer look up exactly which 3.1b APIs they need, with no ambiguity?

**Aligned if:** A dependency matrix maps each 3.1c plan (all of them) to specific API surfaces — named types, methods, and dataclasses. No plan is left with "TBD" or "see context" entries.

**Misaligned if:** Any 3.1c plan has no mapping, or mappings use vague phrases like "needs parser stuff" instead of naming specific types and methods.

### 2. Contracts Use Python Type Signatures, Not Prose

**The question:** Are API contracts unambiguous enough that two implementers would produce compatible code?

**Aligned if:** Contracts show actual Python dataclass/Pydantic definitions, method signatures with return types, and parameter types including optional/required status. An implementer can copy the signature into code.

**Misaligned if:** Contracts use prose like "TeamObservation contains agent data" without specifying whether agents are a dict, list, or custom class. Or return types say "returns observation data" instead of naming the exact type.

### 3. Backward Compatibility Constraints Are Explicit

**The question:** Do contracts clearly state what MUST NOT change in existing code?

**Aligned if:** For every file that extends existing code, the contract specifies backward compatibility constraints — e.g., new fields must be optional to avoid breaking existing constructors. Lists existing methods whose signatures must not change.

**Misaligned if:** No backward compatibility notes, leaving implementers to accidentally add required fields or change existing method signatures.

### 4. Parse vs Execute Boundary Is Clear

**The question:** For fields shared between 3.1b and 3.1c, is it clear who parses and who executes?

**Aligned if:** Each shared field (evaluation, post_run_hooks, evaluation_guidance) explicitly states: "3.1b parses and stores this; 3.1c executes/interprets it."

**Misaligned if:** The boundary is ambiguous, leading to both phases implementing it or neither.

### 5. Test Stubs Verify Import Compatibility

**The question:** Are contracts import-compatible with the actual Python module structure?

**Aligned if:** Test stubs import the specified types from the specified paths. Stubs are marked xfail (expected to fail until implementation). If a stub ERRORs (import fails), it means the contract doesn't match the codebase structure.

**Misaligned if:** No stubs exist, or stubs are just `pass` with TODO comments, or import paths don't match actual module structure.

### 6. Failure Modes Are Specified

**The question:** Can an implementer determine what should happen in edge cases (empty data, missing agents, no queries)?

**Aligned if:** Contracts specify behavior for common failure modes — e.g., "returns None when agent type not found" vs "raises KeyError." An implementer doesn't have to guess.

**Misaligned if:** Only the happy path is specified, leaving implementers to invent error handling that may be incompatible with 3.1c expectations.

## Drift Detection

### OK (preserves intent)
- Different field naming conventions if contracts are updated to match
- Additional helper methods not in the original contract
- More detailed examples than minimal
- Using TypedDict or Pydantic instead of dataclass if it improves clarity

### DRIFT (needs correction)
- Changing method signatures beyond parameter additions
- Changing data structures (dict → list, etc.) without updating the contract
- Removing specified fields from contracts
- Changing the parse/execute boundary split

### STOP (fundamental failure)
- Skipping contracts for any 3.1c plan (that plan's implementer will guess)
- Breaking backward compatibility constraints
- Test stubs that can't even import (structural mismatch)
- Ambiguous types in contracts ("returns data" instead of typed signatures)

## Reviewer Guidance

Focus your review on:
1. **Is every 3.1c plan covered in the dependency matrix?** No gaps — every plan knows what APIs it needs.
2. **Are types concrete, not vague?** `-> list[BSKGQuery]` not "returns queries."
3. **Are backward compatibility constraints explicit?** "ToolCall new fields must be optional."
4. **Can an implementer work from these contracts alone?** No ambiguity in edge case behavior.

Do NOT focus on: exact line counts, specific file organization within the contracts directory, number of types imported in test stubs, or whether the README has a specific table format.
