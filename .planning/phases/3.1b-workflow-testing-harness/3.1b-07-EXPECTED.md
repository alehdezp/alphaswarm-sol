# Expected Results: 3.1b-07 Full Interactive Smoke Test

---

## Fundamental Deliverable

**This plan validates that all Phase 3.1b infrastructure components work together end-to-end.** It proves the complete data flow: scenario loading → CLI execution → transcript parsing → BSKG extraction → output collection → grading → corpus validation → workspace isolation.

This is the "prove it works, then automate it" step — the LAST gate before 3.1c can build on top of the infrastructure. It has two parts: automated synthetic tests (CI-safe) and a human verification checkpoint (real CLI).

---

## Alignment Criteria

### 1. Automated Tests Cover All Infrastructure Components

**The question:** Do synthetic tests exercise every major 3.1b component without requiring external dependencies (Claude CLI, Companion, jj)?

**Aligned if:** Tests cover:
- Scenario loading (YAML → parsed model with evaluation fields)
- Transcript parsing with BSKG extraction
- Output collection into aggregated model
- Code grading (both passing and failing cases)
- Jujutsu workspace methods (mocked subprocess)
- Corpus project validity (ground truth structure)
- Full synthetic pipeline (scenario → parse → collect → grade)

All tests pass in CI without Claude CLI, Companion, or Jujutsu installed.

**Misaligned if:** Tests require live Claude CLI to pass, or major components are skipped, or tests only check that classes exist (no behavioral verification).

### 2. Scenario File Exercises Full DSL

**The question:** Does a smoke test scenario YAML use all the new DSL fields (evaluation guidance, graders, steps)?

**Aligned if:** A health-check or similar simple scenario exists that exercises the full schema including evaluation fields — proving the DSL parser works end-to-end.

**Misaligned if:** No scenario file exists, or the file only uses pre-existing fields (doesn't test the new DSL additions).

### 3. Human Verification Checkpoint Documents Real Execution

**The question:** Has someone actually run the infrastructure with real tools and documented what happened?

**Aligned if:** A checkpoint document exists showing real CLI execution output, real Jujutsu workspace lifecycle, real corpus compilation, and real hook verification — with actual command outputs, not just "it worked."

**Misaligned if:** Checkpoint doesn't exist, says "skipped" or "not tested," or doesn't include actual command outputs as evidence.

### 4. Existing Tests Unbroken

**The question:** Did adding smoke tests break any existing harness tests?

**Aligned if:** All previously-passing harness tests still pass.

**Misaligned if:** Any existing test fails after adding the smoke test file.

### 5. Full Pipeline Data Flow Proven

**The question:** Can data flow through the entire pipeline — from scenario definition to grading result?

**Aligned if:** At least one test assembles synthetic data and passes it through: scenario → transcript → parse → collect → grade → result. This proves the components connect, not just that they exist individually.

**Misaligned if:** Components are tested in isolation only, with no integration test showing the full flow.

## Drift Detection

### OK (preserves intent)
- More tests than minimum (additional coverage is good)
- Different test names than suggested
- Different corpus project used for validation
- Different scenario topic (doesn't have to be health-check specifically)
- Additional assertions beyond what's specified

### DRIFT (needs correction)
- Tests require real Claude CLI to pass (violates CI-safe requirement)
- Tests require Companion (optional Wave 4)
- Missing human checkpoint record
- Checkpoint exists but has no evidence (no command outputs)
- Tests use real `jj` commands instead of mocking subprocess

### STOP (fundamental failure)
- No synthetic tests exist (only human checkpoint — no automation)
- No human checkpoint executed (only synthetic tests — no real validation)
- Smoke test tests multi-agent behavior or reasoning evaluation (scope creep into 3.1c)
- Existing harness tests break after adding smoke tests
- No corpus project compiles or has valid ground truth

## Reviewer Guidance

Focus your review on:
1. **Does the full pipeline flow work?** Not just individual components — does data flow end-to-end?
2. **Can tests run in CI?** No external dependencies required for automated tests.
3. **Does the human checkpoint have EVIDENCE?** Actual command outputs, not just assertions.
4. **Are existing tests unbroken?** Zero regressions.

Do NOT focus on: exact number of test functions, specific test function names, exact scenario YAML structure, specific JSONL format for synthetic transcripts, or whether the implementation matches plan pseudocode line-by-line.
