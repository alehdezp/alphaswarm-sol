# Expected Results: 3.1b-05 Scenario DSL + EvaluationGuidance

---

## Fundamental Deliverable

**This plan extends the scenario YAML format with evaluation hooks and grader implementations.** It creates the scenario definition contract that connects test infrastructure (3.1b-02, 3.1b-04) to evaluation logic (3.1c).

The real promise: A scenario YAML can specify WHAT to evaluate (via evaluation fields) and HOW to grade outcomes — both deterministic (code graders) and nuanced (model graders). The fields are parsed and preserved; actual evaluation execution happens in 3.1c.

**Critical boundary:** This plan provides DATA MODEL and GRADING PRIMITIVES. Phase 3.1c provides the INTELLIGENCE that uses them.

---

## Alignment Criteria

### 1. Scenario Schema Extended for Evaluation

**The question:** Can a scenario YAML now include evaluation configuration, evaluation guidance, graders, steps, hooks, and trial count — all optional so existing scenarios still work?

**Aligned if:** New fields are accepted and parsed without breaking existing scenarios. Old scenarios without these fields load normally.

**Misaligned if:** New fields are required (breaks existing scenarios), or the fields exist in the schema but aren't actually parsed into the scenario model.

### 2. Parse-But-Don't-Execute Boundary

**The question:** Does the scenario loader parse evaluation and hook fields WITHOUT trying to execute them?

**Aligned if:** Fields like `evaluation`, `post_run_hooks`, and `evaluation_guidance` are preserved in the loaded scenario object. No attempt is made to run hook scripts, validate evaluation contract references, or execute grading logic during loading.

**Misaligned if:** The loader tries to execute hooks, validate that evaluation contracts exist (they don't yet — 3.1c creates them), or import 3.1c code during loading.

### 3. Code Graders Work for Deterministic Checks

**The question:** Can deterministic grading (string matching, regex, schema validation, tool usage verification) produce structured pass/fail results?

**Aligned if:** Code graders accept collected output (from 3.1b-02), check against expectations, and return typed result objects with pass/fail status, score, and reason. Each grading method does what its name says — string match checks strings, regex checks patterns, tool usage checks tool sequences.

**Misaligned if:** Graders return bare booleans instead of structured results, or don't actually inspect the output (return hardcoded values), or use LLM calls for deterministic checks.

### 4. Model Grader Uses Claude CLI

**The question:** Does the model-based grader invoke Claude Code via CLI subprocess, consistent with the project's execution model?

**Aligned if:** Model grading shells out to `claude --print -p` (or equivalent CLI invocation), constructs a prompt from evaluation instructions and output summary, and parses the response.

**Misaligned if:** Model grading calls the Anthropic API directly, uses the SDK, or uses the Companion bridge (Wave 4, optional).

### 5. JSON Schema Validates Both Old and New Fields

**The question:** Does a JSON Schema exist that validates scenario YAML including both pre-existing fields and all new additions?

**Aligned if:** Schema covers all fields (existing + new). Validation produces warnings on failure, not hard errors (avoiding breakage during rapid iteration).

**Misaligned if:** Schema only validates new fields (breaks existing scenarios), or validation raises blocking errors, or no schema exists.

### 6. Backward Compatibility

**The question:** Do all existing ScenarioConfig and ScenarioLoader tests still pass?

**Aligned if:** Zero regressions. Existing scenarios load and work exactly as before.

**Misaligned if:** Any existing test fails after the changes.

## Drift Detection

### OK (preserves intent)
- Additional grading methods beyond the core set
- Additional fields on result objects for extensibility
- Different constructor parameters on model grader
- More tests than specified
- Different Pydantic model organization

### DRIFT (needs correction)
- Code graders calling LLMs (wrong grader type)
- Model grader making direct API calls instead of CLI subprocess
- Scenario loader executing hooks during load (execution is 3.1c's job)
- Graders returning bare bools instead of structured result objects
- Schema validation blocking scenario loading on failure

### STOP (fundamental failure)
- No grader implementations at all
- Scenario schema missing evaluation/guidance fields entirely
- Graders that require 3.1c components (circular dependency)
- evaluation or post_run_hooks being executed in 3.1b (wrong phase)
- Existing scenario tests broken

## Cross-Plan Dependencies

| Plan | What's Exchanged | Impact If Missing |
|------|------------------|-------------------|
| 3.1b-02 | Collected output as grader input | Graders have nothing to grade against |
| 3.1b-04 | ScenarioConfig modes/team_config fields | Team scenarios can't be defined |
| 3.1c-06 | Evaluation contract reference (forward reference) | 3.1c can't route to evaluators |
| 3.1c-08 | Evaluation guidance with reasoning questions, flags | 3.1c can't configure evaluation pipeline |

**Note:** The `evaluation.contract` field is a forward reference — it names contracts that don't exist yet (3.1c builds them). This plan must accept any string without validation.

## Reviewer Guidance

Focus your review on:
1. **Does the parse/don't-execute boundary hold?** Loading a scenario with evaluation fields should NOT trigger any execution.
2. **Do code graders actually inspect output?** Not hardcoded results — actually check strings, patterns, tool sequences.
3. **Are existing scenarios untouched?** Zero regressions on pre-existing tests.
4. **Does model grading use CLI, not API?** Consistent with the project's execution model.

Do NOT focus on: exact field counts on Pydantic models, specific grading method signatures, number of tests, JSON Schema line counts, or specific Literal type constraints.
