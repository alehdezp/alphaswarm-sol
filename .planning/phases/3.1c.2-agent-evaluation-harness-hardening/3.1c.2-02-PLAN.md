# Plan 02: JSONL Transcript Verification + CLIAttemptState + Reasoning Timeline

<objective>
Create the CLIAttemptState enum, wire it as check 13 in agent_execution_validator.py, and build the reasoning timeline extractor â€” the single highest-leverage addition for 3.1c.3 (the intelligence bootstrap phase).

**CLIAttemptState + Check 13 (forensic detection):** Plan 12 Batch 1's critical failure was agents using `python -c "from alphaswarm.kg import ..."` instead of `uv run alphaswarm query`. The delegate_guard from Plan 01 blocks this at runtime, but check 13 provides post-session forensic verification by analyzing JSONL transcripts. Both layers are needed: prevention (Plan 01) + detection (this plan).

**Reasoning Timeline (intelligence seed):** The testing philosophy says "each move is scored independently." Currently there is no code extracting these moves from transcripts. The reasoning timeline is the missing bridge between raw JSONL transcripts and structured per-move reasoning assessment. Without it, 3.1c.3-02 (reasoning decomposer) must build extraction AND scoring from scratch. With it, 3.1c.3-02 only needs the scorer â€” the extraction is done. This is the "one thing" from the creative-synthesis review that transforms 3.1c.2 from pure enforcement into enforcement + intelligence seed.

**Namespaced Observation Schema (contract definition):** This plan also DEFINES the observation schema v1 that downstream consumers (3.1c.3 intelligence modules) will use. The schema establishes namespaced sections (identity, execution, graph, queries, reasoning, integrity) so each intelligence module reads ONE section without key collisions. Plan 04 (merged) IMPLEMENTS the export; this plan creates the extraction code for the `reasoning` section.

Output: CLIAttemptState module, reasoning timeline extractor, check 13 in validator, namespaced observation schema definition, 14 unit tests with synthetic transcripts.

### Namespaced Observation Schema v1

This plan DEFINES the schema; Plan 04 (merged) IMPLEMENTS the export. The `reasoning` section is what this plan creates the extraction code for. The `_enrichment` section is populated in Plan 04 (merged).

```yaml
observation_schema_v1:
  identity:
    session_id: str          # e.g., "plan12-retry-2026-03-02-001"
    observation_id: str      # e.g., "obs-cal01-attacker-001"
    contract_id: str         # e.g., "cal-01"
    agent_id: str            # e.g., "attacker-001"
    agent_type: str          # e.g., "vrs-attacker"
    workflow_id: str          # e.g., "evaluation-calibration" (F1: cross-workflow learning)
    batch_id: str            # e.g., "plan12-retry"

  execution:
    started_at: str          # ISO 8601
    completed_at: str        # ISO 8601
    duration_s: float
    cli_attempt_state: str   # CLIAttemptState enum value
    transcript_path: str     # path to raw JSONL
    isolation_mode: str      # "worktree" | "none"

  graph:
    build_identity: str      # graph hash
    graph_path: str          # absolute path to .toon
    nodes: int
    edges: int
    build_source: str        # "ground_truth" | "agent_built"

  queries:
    count: int
    successful: int
    empty_result: int
    queries: list            # [{index, command, result_count, timestamp}]

  reasoning:
    timeline: list           # list[ReasoningEvent] â€” THIS PLAN creates the extractor
    tool_sequence: list      # list[{tool, subtype, index, ts}] â€” THIS PLAN creates the extractor

  integrity:
    validator_version: str   # e.g., "13-checks"
    verdict: str             # "PASS" | "DEGRADED" | "FAIL"
    critical_violations: int
    warning_violations: int

  debrief:
    layer_used: str
    confidence: str
    answers: list

  _enrichment:
    version: str             # e.g., "3.1c.2"
    enriched_at: str         # ISO 8601
    timing_profile: dict     # {total_duration_s, build_kg_duration_s, ...}
    query_progression_score: float  # 0-100
```

**Design principles:**
1. **Namespaced sections, not flat keys.** Each intelligence module reads ONE section. Prevents key collisions as modules grow.
2. **`identity` block required, everything else optional.** Modules check for their section's existence and degrade gracefully.
3. **`_enrichment` is additive-only, versioned.** Written ONCE after capture, never modified. Version field enables consumers to know which enrichment pipeline produced the data.
4. **`workflow_id` is explicit, not inferred.** Critical for cross-workflow learning (3.1c.3 module 6). Without it, observations keyed by `contract_id + agent` lose workflow context.

    **1. ReasoningEvent dataclass + extract_reasoning_timeline()**

    ```python
    @dataclass
    class ReasoningEvent:
        timestamp: str | None
        move_type: Literal[
            "hypothesis", "query_formulation", "query_execution",
            "result_interpretation", "evidence_integration", "conclusion"
        ]
        content_snippet: str  # first 200 chars of the relevant text
        tool_call_id: str | None
        references_prior_event: int | None  # index of the event this builds on

    def extract_reasoning_timeline(transcript_path: Path) -> list[ReasoningEvent]:
        """Heuristic classifier decomposing JSONL into ReasoningEvent sequences.

        Classification rules:
        - Text before first tool call -> hypothesis
        - Bash commands with 'alphaswarm query' -> query_formulation + query_execution
        - Text immediately after tool_result -> result_interpretation
        - Text referencing multiple prior results -> evidence_integration
        - Final structured output / last assistant message -> conclusion

        Accuracy: ~70% (heuristic). 3.1c.3-02 can refine with LLM-based classification.
        """
    ```

    The function:
    - Parses the JSONL transcript line by line
    - Uses `TranscriptParser` for structured tool call extraction where possible
    - Classifies each meaningful segment into one of 6 move types
    - Returns empty list for None/missing/empty transcripts (graceful degradation)
    - Truncates `content_snippet` to 200 characters
    - Sets `references_prior_event` when a result_interpretation or evidence_integration
      event follows directly after a query_execution event (index back-reference)

    **2. extract_tool_sequence()**

    ```python
    def extract_tool_sequence(transcript_path: Path) -> list[dict]:
        """Extract ordered tool type sequence from transcript.

        Returns: [{tool: str, subtype: str, index: int, ts: str | None}]

        Subtypes:
        - Bash + "build-kg" -> "build-kg"
        - Bash + "query" -> "query"
        - Read -> "read-file"
        - Other -> "other"
        """
    ```

    The function:
    - Uses `TranscriptParser(transcript_path).get_tool_calls()`
    - Classifies each tool call's subtype based on tool name and command content
    - Returns empty list for None/missing/empty transcripts
    - Preserves ordering (index field matches position in transcript)
    - Includes timestamp when available from the tool call

    **Why this matters for 3.1c.3:**
    - The reasoning timeline populates the `reasoning.timeline` section of the observation schema
    - The tool sequence populates the `reasoning.tool_sequence` section
    - 3.1c.3-02 (reasoning decomposer) needs per-move transcript segments to score
    - 3.1c.3-06 (behavioral fingerprinter) needs tool sequences as core input
    - 3.1c.3-10 (recommendation engine) needs structured reasoning traces to map failures to specific reasoning steps

    **Target size:** ~150 lines total. This is a heuristic extraction module, not an LLM pipeline.

    DC-2 enforcement: Do NOT import from `alphaswarm_sol.kg` or `alphaswarm_sol.vulndocs`.
    Use `TranscriptParser` from `tests.workflow_harness.lib.transcript_parser`.
