# Expected Results: Plan 02 — JSONL Transcript Verification + CLIAttemptState + Reasoning Timeline

## Fundamental Deliverable

This plan delivers the CLIAttemptState enum with transcript-based forensic verification (check 13), a heuristic reasoning timeline extractor, and the namespaced observation schema v1 definition so that the evaluation harness can detect post-session whether agents bypassed the CLI (Python import fallback — FM-3) and so that downstream 3.1c.3 intelligence modules have structured reasoning data to work with instead of raw JSONL transcripts. Without it, Plan 04 cannot wire integrity checks into the evaluation runner because check 13 does not exist, Plan 05 (Batch 1 retry) cannot forensically verify CLI usage, and 3.1c.3-02 (reasoning decomposer) must build both extraction AND scoring from scratch instead of just the scorer. Success means synthetic JSONL transcripts with `uv run alphaswarm query` calls produce `ATTEMPTED_SUCCESS`, transcripts without them produce `NOT_ATTEMPTED` (critical violation), reasoning timeline extraction classifies transcript segments into 6 move types, and the observation schema defines namespaced sections that downstream modules can consume without key collisions. Failure means transcript analysis cannot distinguish agents that used the CLI from agents that bypassed it, or reasoning events are not extractable from transcripts, or the observation schema is flat/unnamespaced — any of which forces rework in Plans 04, 05, and the entire 3.1c.3 phase.

### [must_have] CLIAttemptState enum distinguishes 4 agent behaviors

**Source:** must_haves.truths[0] — "CLIAttemptState enum distinguishes: agent used CLI successfully, tried but failed, never tried, or transcript unavailable"

**The question:** Does `cli_attempt_state.py` define a `CLIAttemptState` enum with exactly 4 states that map to distinct agent CLI behaviors, with a `compute_cli_attempt_state()` function that produces the correct state from a transcript path?

**Aligned if:** `src/alphaswarm_sol/testing/evaluation/cli_attempt_state.py` contains a `CLIAttemptState(str, Enum)` with values `ATTEMPTED_SUCCESS`, `ATTEMPTED_FAILED`, `NOT_ATTEMPTED`, and `TRANSCRIPT_UNAVAILABLE`. The `compute_cli_attempt_state(transcript_path: Path | None)` function returns `TRANSCRIPT_UNAVAILABLE` for None/missing/empty paths, `NOT_ATTEMPTED` when no alphaswarm CLI calls appear in the transcript, `ATTEMPTED_SUCCESS` when at least one query returns results, and `ATTEMPTED_FAILED` when all queries fail. Mixed results (some succeed, some fail) return `ATTEMPTED_SUCCESS`.

**Misaligned if:** The enum has fewer than 4 states (e.g., only SUCCESS/FAIL binary). Or `compute_cli_attempt_state` does not handle None/empty transcript paths gracefully. Or mixed results return `ATTEMPTED_FAILED` instead of `ATTEMPTED_SUCCESS`. Or the function does not actually parse JSONL transcripts (returns hardcoded values).

## Drift Detection

### OK: ReasoningEvent uses slightly different field names or adds fields

**Scenario:** The executor implements `ReasoningEvent` with `move` instead of `move_type`, or adds an `agent_id` field, or uses `text_preview` instead of `content_snippet`.

**Why OK:** The heuristic extractor is explicitly ~70% accurate and will be refined by 3.1c.3-02. Field naming is an implementation detail as long as the 6 move types are present, content is truncated, and the dataclass is importable by downstream consumers.

**Action:** Verify the 6 move types exist, content truncation is present, and the dataclass can be imported. Minor naming differences are acceptable.

### OK: Test count is 12-16 instead of exactly 14

**Scenario:** The executor creates 12 tests instead of 14 (combining some edge cases), or 16 tests (splitting the mixed-results test into separate scenarios).

**Why OK:** The plan specifies 14 as a target, not a hard constraint. What matters is coverage of all 4 CLIAttemptState paths, check 13 severity verification, reasoning timeline classification, and tool sequence extraction — not the exact count.

**Action:** Verify all must_have truths are tested. If any of the 4 CLIAttemptState states lacks a test, or check 13 severity is untested, that is DRIFT regardless of total count.

### DRIFT: reasoning_timeline.py requires LLM calls for classification

**Scenario:** The executor implements `extract_reasoning_timeline()` using Claude API calls to classify each transcript segment, making it an LLM pipeline rather than a heuristic extractor.

**Why DRIFT:** The plan explicitly states "heuristic classification" with "~70% accuracy" and "~150 lines total. This is a heuristic extraction module, not an LLM pipeline." An LLM-based classifier would be expensive, non-deterministic, require API keys in test environments, and exceed the intended scope. 3.1c.3-02 is the plan that adds LLM-based refinement.

**Action:** Request replacement with heuristic-only implementation. The function should use pattern matching on tool names and message positions, not LLM inference. If the executor argues for LLM accuracy, point to the plan's explicit "heuristic" designation and the 3.1c.3-02 division of labor.

### DRIFT: Check 13 exists but is not wired into validate_observation_file()

**Scenario:** The executor creates `_check_cli_attempt_state()` as a standalone function but does not add the call into `validate_observation_file()`. The function exists, the tests test it directly, but the validator pipeline never invokes it.

**Why DRIFT:** The must_have states "Check 13 in agent_execution_validator produces critical violation." This means check 13 must fire when `validate_observation_file()` is called — not just exist as dead code. Plan 04 depends on the validator pipeline including check 13 when it wires auto-REJECT.

**Action:** Request wiring. The `_check_cli_attempt_state()` call must appear in `validate_observation_file()` after the existing checks. Verify by reading the function body.

### STOP: CLIAttemptState does not actually parse JSONL transcripts

**Scenario:** The executor creates the `CLIAttemptState` enum and `compute_cli_attempt_state()` but the function returns hardcoded values based on whether the file exists (TRANSCRIPT_UNAVAILABLE for missing, ATTEMPTED_SUCCESS for any existing file), without actually parsing the JSONL content to look for alphaswarm CLI commands.

**Why STOP:** This defeats the entire purpose of forensic detection. The plan exists specifically because agents in Plan 12 Batch 1 had transcripts (files existed) but never used the CLI (content showed Python imports). A file-existence check cannot distinguish an agent that used CLI from one that used Python imports. Check 13 would always return SUCCESS for any agent that ran, making it useless for FM-3 detection.

**Action:** Escalate. The function MUST parse JSONL content via `TranscriptParser.get_bskg_queries()` to determine whether alphaswarm CLI commands were actually executed. File existence is necessary but not sufficient.

## Reviewer Guidance

1. **Verify `compute_cli_attempt_state` actually parses JSONL content, not just file existence.** This is the single most important check. The entire plan exists to distinguish "agent used CLI" from "agent bypassed CLI." If the function only checks whether a file exists, check 13 is useless for FM-3 detection. Read the function body and confirm it calls `TranscriptParser.get_bskg_queries()`.

2. **Confirm check 13 is wired into `validate_observation_file()`, not just standalone.** The function `_check_cli_attempt_state` must be called from within `validate_observation_file()` so it fires during `validate_batch()`. A standalone function that only tests call directly is dead code from Plan 04's perspective.

3. **Verify severity mapping: NOT_ATTEMPTED=critical, ATTEMPTED_FAILED=warning, TRANSCRIPT_UNAVAILABLE=warning.** The severity distinction between NOT_ATTEMPTED (agent chose to bypass CLI — FM-3) and ATTEMPTED_FAILED (CLI is broken — FM-1) is critical for Plan 04's auto-REJECT logic. Critical means auto-reject; warning means investigate. Swapping these severities would either mask real bypass attempts or reject agents for infrastructure bugs.

4. **Check that `extract_reasoning_timeline()` is heuristic-only with no LLM dependency.** The function should be deterministic and fast (~150 lines). If it imports any LLM client or makes API calls, it has exceeded scope. 3.1c.3-02 owns the LLM-based classification layer.

5. **Verify both modules use `TranscriptParser` imports, not custom JSONL parsing.** Decision D-2 mandates TranscriptParser as the primary parsing mechanism. Duplicate parsing logic would diverge from the canonical parser when transcript format changes.
