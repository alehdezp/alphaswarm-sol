# Expected Results: Plan 05 — Plan 12 Batch 1 Retry with Hardened Harness

**Plan:** 3.1c.2-05
**Phase:** 3.1c.2-agent-evaluation-harness-hardening
**Generated:** 2026-03-02

---

## Fundamental Deliverable

This plan delivers a validated end-to-end re-run of the Plan 12 Batch 1 calibration session with all enforcement mechanisms from Plans 01-04 active, so that the entire 3.1c.2 phase can prove its hardening actually prevents the agent bypass failures (Python imports, context leakage, fabricated findings) that invalidated the original batch. Without it, the phase has no empirical proof that its guard config, CLIAttemptState detection, ground truth comparison, and auto-reject pipeline work together under real Agent Teams conditions — leaving 3.1c.3 (Evaluation Intelligence Bootstrap) and 3.1f (Proven Loop Closure) building on unverified infrastructure. Success means the integrity validator returns VERDICT: PASS with 0 critical violations, every agent achieves CLIAttemptState ATTEMPTED_SUCCESS using only CLI tools, observations contain enrichment blocks ready for intelligence module consumption, agent findings are cross-referenced against ground truth expected findings, and the batch-level quality gate passes. Failure means agents still bypass enforcement (proving the harness is incomplete), or the integrity check returns FAIL/DEGRADED across multiple contracts (proving the enforcement stack has gaps), or enrichment data is missing (blocking 3.1c.3 modules from consuming these observations).

## Alignment Criteria

### [must_have] Hardened harness prevents Python imports
**Source:** must_haves.truths[0]
**The question:** Does the hardened harness actually prevent agents from using Python imports during the retry session?
**Aligned if:** Reviewing the JSONL transcripts for each agent in `.vrs/observations/plan12-retry/` shows zero Bash tool calls containing `python`, `python3`, `import `, or `from alphaswarm`. The delegate_guard blocked-call log (if `DELEGATE_GUARD_LOG` was set) shows attempted-and-blocked calls with exit code 2. No observation JSON contains evidence of raw `KnowledgeGraph` Python class usage.
**Misaligned if:** Any agent transcript contains `Bash(python -c "...")` or `Bash(python3 ...)` commands that executed successfully (exit 0). An observation references Python module imports as the analysis method. The guard was not activated (env vars `_GUARD_PROFILE=strict` and `DELEGATE_GUARD_CONFIG` were not set before Agent Teams spawn).

### [must_have] All agents use CLI and produce non-empty results
**Source:** must_haves.truths[1]
**The question:** Did all spawned agents use `build-kg` and/or `query` CLI commands and receive non-empty results?
**Aligned if:** Each observation JSON in `.vrs/observations/plan12-retry/` contains a `queries` section (or top-level fields) showing `queries_executed >= 1` with at least some returning non-zero result counts. At least 3 of the spawned agents (the plan specifies >= 3 of 5) produce non-empty query results. Agent transcripts show `uv run alphaswarm build-kg` and/or `uv run alphaswarm query` calls with substantive output.
**Misaligned if:** Agents have zero queries or all queries returned empty results. Fewer than 3 agents produce usable output. Agents show CLI calls that all errored or timed out.

## Drift Detection

### OK: Fewer than 5 agents spawned but all 4 contracts covered
**Scenario:** The executor spawns 4 agents (one per calibration contract) instead of the suggested 3-5, with no defender on cal-01. All 4 contracts are analyzed, all agents pass integrity, but only 4 observation files are produced instead of 5.
**Why OK:** The plan specifies "3-5 agents" with ">= 3 agents produce non-empty results" as the threshold. Four agents covering all contracts satisfies both the quantity floor and the contract coverage requirement. The defender on cal-01 is listed as "at minimum" but the primary signal is contract coverage and CLI usage.
**Action:** Verify all 4 contracts have observations, all pass integrity, and >= 3 agents have non-empty results.

### OK: query_progression_score is 0.0 for all observations
**Scenario:** The enrichment blocks are present with populated `timing_profile` and `tool_sequence`, but `query_progression_score` is 0.0 for every observation because the scoring heuristic in `reasoning_timeline.py` requires sufficient transcript data that the short calibration sessions do not provide.
**Why OK:** The plan notes this as "if transcript available" and the score is a 0.0-1.0 measure of query sophistication. For short calibration sessions with 1-3 queries, a score of 0.0 is a legitimate data point, not a missing field. The key requirement is that the field EXISTS and is populated (not omitted).
**Action:** Verify the field is present as a float, not absent or null. Accept 0.0 as valid data.

### DRIFT: Integrity PASS but enrichment blocks missing from observations
**Scenario:** The integrity validator returns VERDICT: PASS with 0 critical violations. All agents used CLI correctly. But observation JSONs lack the `_enrichment` block — they contain findings and integrity data but no timing, tool sequence, or query progression data.
**Why DRIFT:** The integrity check (truths[2]) is satisfied, but truths[6] (enrichment blocks) is violated. Without enrichment data, these observations cannot serve as input for 3.1c.3 intelligence modules (tier_manager, coverage_radar, behavioral_fingerprinter). The retry produces valid calibration results but fails to deliver the intelligence-bootstrap prerequisite data that is half the plan's purpose.
**Action:** Request correction. The enrichment capture must be wired into the observation pipeline. Check whether `_enrichment.json` sidecar files exist or whether the fields need to be embedded in the observation JSON directly.

### DRIFT: Finding overlap not computed or _BATCH_QUALITY.json missing
**Scenario:** The retry passes integrity, observations are enriched, but Task 3 was not completed — `_BATCH_QUALITY.json` does not exist, and no finding-overlap comparison was performed against expected_findings.json.
**Why DRIFT:** truths[9] requires agent findings to be "cross-referenced against expected_findings.json per contract." The plan explicitly specifies that low overlap is informational (not a gate), but the computation must still happen. Without it, there is no baseline data for 3.1c.3's coverage_radar or semantic accuracy evaluation.
**Action:** Request Task 3 completion. The batch quality summary and finding overlap analysis are required artifacts even if the overlap scores are low.

### STOP: Agents used Python imports despite guard being configured
**Scenario:** The preflight checks all passed, env vars were set, but during the actual Agent Teams session, agents successfully executed `python -c "from alphaswarm.kg import KnowledgeGraph"` — the guard did not fire for teammates running in worktree isolation.
**Why STOP:** This is the exact Plan 12 Batch 1 failure mode that the entire 3.1c.2 phase exists to prevent. If the guard does not propagate to Agent Teams teammates despite passing the orchestrator-level preflight, then the enforcement stack has a fundamental gap in env-var propagation to child processes. Plans 01-04 are invalidated — not just incomplete, but fundamentally broken for the Agent Teams use case.
**Action:** Escalate immediately. Investigate whether `_GUARD_PROFILE` and `DELEGATE_GUARD_CONFIG` env vars propagate to worktree-isolated teammates. Check Claude Code version compatibility. Do not retry without root-causing the propagation failure. This may require changes to Plans 01 or 04, not just Plan 05.

## Reviewer Guidance

**Priorities:**

1. **Verify VERDICT: PASS with 0 critical violations in `_INTEGRITY_REPORT.json`.** This is the single binary gate for the entire phase. Read the file, confirm `verdict` is `"PASS"`, count critical violations. If this fails, everything else is moot — the hardening did not work.

2. **Confirm no Python import Bash commands in any agent transcript.** This is the forensic proof that the delegate_guard enforcement actually worked at the Agent Teams level. Search transcripts for `python`, `import`, `from alphaswarm` in Bash tool_use entries. Zero matches is the hard gate. This distinguishes "guard was active" from "guard was configured but not propagated to teammates."

3. **Verify enrichment data is present and populated in all observations.** The plan serves dual purposes: enforcement validation AND intelligence-bootstrap data capture. Check that `_enrichment` blocks (or sidecar files) contain non-empty `timing_profile`, `tool_sequence`, and `query_progression_score`. Empty/default enrichment means half the plan's value is lost.

4. **Verify `_BATCH_QUALITY.json` exists with finding-overlap results.** Task 3 is the post-session analysis that produces the batch quality summary and cross-references agent findings against ground truth. Its absence means the semantic accuracy dimension was never evaluated.

5. **Confirm Agent Teams were spawned with proper isolation.** Check that the execution protocol was followed: `isolation: "worktree"` for all teammates, zero project context provided, tools restricted to `build-kg`, `query`, and `Read` (contracts only). If agents had full project context, findings are tainted by knowledge of expected answers — the evaluation is not ecologically valid.
