### Gaps Summary

**Gap 1: Plan 12 Batch 1 Retry — raw validator run was FAIL**

The actual Agent Teams session that produced `.vrs/observations/plan12-retry/` triggered a raw FAIL verdict (5 critical violations):

- 4 `knowledge_leakage` violations: Pattern IDs in agent findings (e.g., `statemachine-reentrancy-guard-missing`, `auth-001`) were flagged as project context leakage. Root cause: the LEAKED_TERMS list was authored before CLI output format was finalized — the CLI itself includes pattern IDs in query output, so agents reporting CLI output were falsely flagged. This was a validator bug, not agent behavior.
- 1 `no_worktrees_found` violation: Worktrees were cleaned up after agent shutdown but before validation ran. The observation files self-report `isolation: "worktree"` which the updated validator accepts.

**Resolution applied:** The validator's LEAKED_TERMS was updated to exclude pattern IDs. The validator was re-run on the same observation files, producing PASS. The failure-analysis.md documents this.

**Residual gap:** The original session's FAIL was resolved by fixing the validator after the fact. An observer cannot confirm that a fresh session would produce PASS without the validator fix. Whether this constitutes meeting the exit gate depends on whether "VERDICT: PASS" means "the final updated validator passes" or "the original session validated cleanly."

**Gap 2: No JSONL transcripts — CLIAttemptState is self-reported**

Agent Teams teammates do not produce JSONL transcripts accessible to the validator in the current architecture. Check 13 finds TRANSCRIPT_UNAVAILABLE for all 4 agents (downgraded to `info` severity due to the `isolation: "worktree"` field). The `cli_attempt_state: "attempted_success"` in each observation file was written by the orchestrator based on observing agent output, not extracted from JSONL by `compute_cli_attempt_state()`.

**Impact on exit gate:** The ROADMAP exit gate states "CLIAttemptState == ATTEMPTED_SUCCESS for all CLI calls." This is met by the self-reported field but NOT by the forensic transcript-based check. The delegate_guard blocking was in place, but no delegate_guard blocked-call log exists as evidence that Python imports were attempted and blocked.

**Severity:** These are architectural gaps that the failure-analysis.md explicitly flags for 3.1c.3 remediation. The underlying agent behavior (using CLI, not Python imports) is consistent with what the orchestrator observed. But forensic confirmation is absent.

**Recommendation:** Either (a) accept orchestrator-attested CLIAttemptState as sufficient for this phase, or (b) run one more targeted session with DELEGATE_GUARD_LOG set to capture blocking evidence.
