# Expected Results: Plan 03 — Ground Truth Graph Generation + Expected Findings

**Plan:** 3.1c.2-03
**Phase:** 3.1c.2-agent-evaluation-harness-hardening
**Generated:** 2026-03-02

---

## Fundamental Deliverable

This plan delivers pre-generated ground truth graph stats (with semantic metadata) and hand-authored expected vulnerability findings for all 4 calibration contracts, so that the agent_execution_validator can compare agent-reported stats against actual build output and downstream 3.1c.3 intelligence modules (tier_manager, coverage_radar, semantic accuracy evaluation) can measure whether agents found the RIGHT bugs — not just whether they used the right tools. Without it, `_check_graph_stats_against_ground_truth()` in the validator silently skips comparison checks (it returns an empty violations list when `gt_path` does not exist), the entire evaluation framework can only measure process compliance ("did agent use CLI?") but not outcome accuracy ("did agent find the reentrancy bug?"), and Plan 05 (Batch 1 retry) cannot validate semantic finding overlap. Success means all 4 `stats.json` files exist with correct `nodes`/`edges` field names matching the validator interface plus new semantic metadata fields, all 4 `expected_findings.json` files exist with hand-curated vulnerability ground truth, persistent queryable graph.toon files are available, and a single repeatable script can regenerate everything. Failure means stats.json files are missing or use wrong field names (e.g., `node_count` instead of `nodes` — breaking the validator's `actual_stats.get('nodes', 0)` call), expected_findings are auto-generated stubs rather than hand-authored vulnerability data, or ground truth exists but the generation script is absent — making regeneration after contract changes a manual process.

## Alignment Criteria

### [must_have] stats.json exists for all 4 calibration contracts
**Source:** must_haves.truths[0]
**The question:** Do `.vrs/ground-truth/cal-01/stats.json` through `.vrs/ground-truth/cal-04/stats.json` all exist and contain valid JSON?
**Aligned if:** All 4 files exist at the paths specified in the plan (`3.1c.2-03-PLAN.md:19-22`). Each is parseable JSON. Each contains at minimum: `contract`, `contract_id`, `nodes`, `edges`. Running `ls .vrs/ground-truth/cal-{01,02,03,04}/stats.json` returns exactly 4 files. Additionally, `.vrs/ground-truth/cal-{01,02,03,04}-graph-stats.json` exist (the validator-compatible path format from `agent_execution_validator.py:640`: `gt_path = ground_truth_dir / f"{contract_id}-graph-stats.json"`).
**Misaligned if:** Any of the 4 files is missing. Files exist but are empty or not valid JSON. The `{cal-id}-graph-stats.json` sibling files are absent (breaking the validator path convention). Files exist but only for some contracts (e.g., cal-01 and cal-02 but not cal-03 and cal-04).

### [must_have] stats.json includes semantic metadata fields
**Source:** must_haves.truths[1]
**The question:** Does each stats.json include the 5 semantic fields: `expected_vulnerability_class`, `semantic_fingerprint`, `function_names[]`, `has_external_calls`, `has_state_modification_after_call`?
**Aligned if:** For every stats.json, `jq '.expected_vulnerability_class, .semantic_fingerprint, .function_names, .has_external_calls, .has_state_modification_after_call'` returns non-null values. `expected_vulnerability_class` is a string (e.g., `"reentrancy"`, `"access-control"`, `"oracle-manipulation"`). `function_names` is a non-empty array of strings. `has_external_calls` and `has_state_modification_after_call` are booleans. The semantic metadata matches the contract's actual characteristics (e.g., cal-01 ReentrancyClassic has `has_state_modification_after_call: true` per the plan's CALIBRATION_CONTRACTS mapping at `3.1c.2-03-PLAN.md:151-188`).
**Misaligned if:** Semantic fields are absent. Fields exist but have null/empty values. `function_names` is empty or not an array. Boolean fields are strings instead of actual booleans. Semantic metadata is generic placeholder text rather than contract-specific.

### [must_have] stats.json uses "nodes" and "edges" field names
**Source:** must_haves.truths[2]
**The question:** Does each stats.json use the exact field names `"nodes"` and `"edges"` (not `"node_count"` or `"edge_count"`) to match the validator's expectations?
**Aligned if:** `jq '.nodes' .vrs/ground-truth/cal-01/stats.json` returns a number (not `null`). The validator code at `agent_execution_validator.py:646` reads `actual_stats.get("nodes", 0)` — the ground truth JSON key must be `"nodes"` not `"node_count"`. Same applies to `"edges"`. This can be verified by: `python -c "import json; d=json.load(open('.vrs/ground-truth/cal-01/stats.json')); assert 'nodes' in d and 'edges' in d"`.
**Misaligned if:** The field is named `"node_count"`, `"nodeCount"`, or any variant other than `"nodes"`. The validator would silently read 0 from `actual_stats.get("nodes", 0)` and all deviation checks would be against 0 — masking real mismatches.

### [must_have] Node counts are positive integers from real builds
**Source:** must_haves.truths[3]
**The question:** Does `jq .nodes` on each stats.json return a positive integer, and are these numbers derived from actual `build-kg` execution rather than fabricated?
**Aligned if:** For each of the 4 stats.json files, `.nodes` is a positive integer (> 0) and `.edges` is a positive integer (> 0). The `scripts/generate_ground_truth.py` contains subprocess calls to `build-kg` that parse the actual stdout output (regex match for `nodes=(\d+)` and `edges=(\d+)` per `3.1c.2-03-PLAN.md:193-194`). The graph_path points to a real `.toon` file that was produced by that build.
**Misaligned if:** Any stats.json has `nodes: 0` or `edges: 0` or negative values. The script hardcodes node/edge counts instead of parsing from build-kg output. The numbers are suspiciously round (e.g., `nodes: 100, edges: 200`) suggesting they were invented rather than measured.

The file exists but is not a valid TOON graph (query returns errors). No persistent copy was made — only a temp build artifact remains.

## Drift Detection

### OK: Script uses tempfile instead of --out for build directory
**Scenario:** The executor implements the build script using Python's `tempfile.mkdtemp()` for the build output directory instead of passing `--out {specific_path}` to build-kg. The graph.toon is then copied from the temp directory to the persistent location.
**Why OK:** The plan specifies using a temp directory for builds (`3.1c.2-03-PLAN.md:191`). Whether this is managed by Python's tempfile module or a manually-created temp directory is an implementation detail. The critical outcome is that the persistent copy at `.vrs/ground-truth/{cal-id}/graph.toon` exists and the stats.json `graph_path` points to it.
**Action:** Verify the persistent graph.toon copy exists and is queryable. Accept the variation.

### OK: Expected findings include additional fields beyond the required 5
**Scenario:** The executor adds extra fields to expected_findings.json entries, such as `confidence`, `pattern_id`, `cwe_id`, or `evidence_patterns`, beyond the required 5 fields (vulnerability_class, severity, function_name, description, key_indicators).
**Why OK:** The plan defines minimum required fields. Additional fields do not break consumers — JSON readers use `.get()` and ignore unknown keys. Extra metadata may be useful for 3.1c.3 intelligence modules.
**Action:** Verify the 5 required fields are present. Accept additional fields as enrichment.

### DRIFT: Stats.json generated but validator-compatible sibling missing
**Scenario:** The script generates `.vrs/ground-truth/cal-01/stats.json` but does NOT generate `.vrs/ground-truth/cal-01-graph-stats.json`. The per-directory stats file exists but the flat-namespace sibling that the validator expects is absent.
**Why DRIFT:** The validator at `agent_execution_validator.py:640` constructs the path as `ground_truth_dir / f"{contract_id}-graph-stats.json"` — a flat file in the ground truth root directory, not inside a subdirectory. Without this file, the validator's `_check_graph_stats_against_ground_truth` silently returns empty violations (the `if not gt_path.exists(): continue` at line 641-642 skips the contract). The ground truth exists but is invisible to the validator.
**Action:** Request correction. The script must produce both `{cal-id}/stats.json` (for human/module consumption) and `{cal-id}-graph-stats.json` (for validator consumption). Task 1 step 3b at `3.1c.2-03-PLAN.md:219` explicitly specifies this dual output.

### DRIFT: Expected findings are auto-generated rather than hand-authored
**Scenario:** The generation script auto-generates expected_findings.json by analyzing the .sol files programmatically (e.g., regex scanning for `call.value`, `tx.origin` patterns) instead of hand-authoring the vulnerability descriptions.
**Why DRIFT:** The plan explicitly states these should be "hand-authored" (`3.1c.2-03-PLAN.md:23-26`, `3.1c.2-03-PLAN.md:320`). Auto-generated findings risk being wrong — a regex scanner cannot reliably determine vulnerability severity, explain the vulnerability mechanism, or provide meaningful key_indicators. For calibration contracts with KNOWN vulnerabilities by design, the ground truth should be human-curated to serve as an authoritative reference.
**Action:** Request the executor verify the findings are substantive and accurate. If auto-generated, verify each entry against the actual contract source. The key test: does the `description` field explain WHY this is a vulnerability, not just WHAT pattern was detected?

### STOP: Generation script does not call build-kg
**Scenario:** The script creates stats.json files with plausible-looking numbers (e.g., `nodes: 12, edges: 17`) but never actually calls `uv run alphaswarm build-kg`. The node/edge counts are hardcoded or copied from Plan 12 observation data.
**Why STOP:** This reproduces the exact fabrication problem the entire phase exists to prevent. If ground truth stats are not from actual builds, they cannot serve as comparison baselines. The validator would compare agent-reported stats against fake baselines — rendering the entire check meaningless. Plan 05 would "pass" integrity checks against fabricated ground truth, proving nothing about agent behavior.
**Action:** Escalate. Verify the script contains `subprocess.run` calls with `build-kg` in the command. Check that the graph.toon files at the `graph_path` locations are real TOON-format files (not empty or text stubs). If stats were fabricated, all downstream integrity checks are invalidated.

## Reviewer Guidance

**Priorities:**

1. **Verify `"nodes"` and `"edges"` field names in stats.json.** This is the single highest-risk failure mode. The validator uses `actual_stats.get("nodes", 0)` at `agent_execution_validator.py:646`. If the ground truth uses `"node_count"` or any variant, the validator silently defaults to 0, and ALL stat comparison checks become meaningless. Run: `python -c "import json; [print(k, json.load(open(f'.vrs/ground-truth/cal-0{i}/stats.json')).get('nodes')) for i,k in enumerate(['cal-01','cal-02','cal-03','cal-04'],1)]"`

2. **Confirm both output formats exist.** Each contract needs TWO files: `{cal-id}/stats.json` (subdirectory format for human/module consumption) AND `{cal-id}-graph-stats.json` (flat format for validator consumption at `ground_truth_dir / f"{contract_id}-graph-stats.json"`). Missing the flat sibling makes the validator's ground truth check a no-op.

3. **Verify graph.toon files are real and queryable.** The persistent graph copies at `.vrs/ground-truth/{cal-id}/graph.toon` must be actual TOON files from build-kg output. Quick check: `uv run alphaswarm query "functions" --graph .vrs/ground-truth/cal-01/graph.toon --compact` must return results. If this fails, the ground truth is not usable for pre-built graph evaluation.

4. **Check that expected_findings.json content is substantive, not templated.** Each contract's expected findings should describe its specific known vulnerability with accurate function names and meaningful key_indicators. Verify function names match actual functions in `tests/contracts/{Contract}.sol`. The `description` field should explain the vulnerability mechanism, not just name the class.

5. **Confirm the generation script calls build-kg via subprocess.** The ground truth's value depends on numbers coming from real builds. Open `scripts/generate_ground_truth.py` and verify it contains subprocess calls to `uv run alphaswarm build-kg` with regex parsing of stdout for node/edge counts. Hardcoded numbers defeat the purpose.
