# Expected Results — 3.1c.2-04: Post-Pipeline Integration: Integrity + Debrief + Enrichment

## Fundamental Deliverable

This plan delivers integrity check wiring, debrief artifact persistence (Stage 9), score tainting, DEGRADED escalation, stage timing telemetry, session correlation IDs, failure_mode classification, and observation enrichment into evaluation_runner.py so that every evaluation session automatically validates agent execution integrity and persists debrief artifacts for downstream consumption by 3.1c.3 intelligence modules and 3.1f regression detection. Without it, integrity failures are silent (polluting baselines with fabricated or CLI-bypassing results), debrief data is lost after sessions end, and downstream phases (3.1c.3, 3.1f) must retrofit timing, tainting, and correlation signals that this plan provides as zero-cost additions to the pipeline. Success means the evaluation runner auto-REJECTs on FAIL verdicts with rejection.json + tainted metadata, persists debrief.json artifacts via Stage 9, records per-stage timing in health.stage_durations, includes session_correlation_id in metadata, classifies IntegrityViolations with failure_mode codes, escalates DEGRADED with 3+ warnings to FAIL, and writes _enrichment.json to observation directories — all verified by 13 passing integration tests. Failure means integrity checks exist as standalone code (Plan 02) but are never invoked during pipeline execution, debrief data computed in Stage 6 is consumed but never persisted, or tainted results silently enter baselines because the rejection gate was never wired.

**Aligned if:**

- `baseline_update_status` is set to something other than "rejected" on FAIL (e.g., "failed", "error")

## Drift Detection

### OK: Stage ordering differs slightly from spec

**Scenario:** The executor implements integrity_check and run_debrief as the last two stages but labels them as Stage 9 and Stage 10 instead of Stage 8.5 and Stage 9. The ordering is correct (both after store_result, integrity before debrief), but the numbering convention differs.

**Why OK:** The plan's intent is about ordering (integrity after store_result, debrief after integrity) and non-blocking behavior (debrief runs even after FAIL). Stage numbering is cosmetic. The PIPELINE_STAGES list and stages_completed values are what downstream consumers use — not numeric stage labels.

**Action:** Verify the ordering is correct: `PIPELINE_STAGES.index('integrity_check') > PIPELINE_STAGES.index('store_result')` and `PIPELINE_STAGES.index('run_debrief') > PIPELINE_STAGES.index('integrity_check')`.

---

### OK: Intelligence signals extraction uses different keyword matching

**Scenario:** The executor implements `_extract_intelligence_signals()` with slightly different keyword patterns than specified (e.g., matching "informative query" instead of "most informative"), or adds additional keywords beyond the three specified.

**Why OK:** The intelligence signals function is a heuristic helper targeting 3.1c.3 modules. The exact keyword patterns depend on the debrief questions actually asked in Stage 6. As long as the function extracts signal keys that 3.1c.3 can consume and degrades gracefully when questions do not match, the plan's intent is preserved.

**Action:** Verify the function returns a dict with meaningful keys when debrief answers contain intelligence-related content, and returns an empty dict when no intelligence questions were asked.

---

### DRIFT: Integrity check is gated behind a configuration flag defaulting to disabled

**Scenario:** The executor adds the integrity check but wraps it in `if self._run_integrity_check:` with a default of `False`. The integration tests set it to `True` so they pass, but normal pipeline runs skip integrity checking entirely.

**Why DRIFT:** CONTEXT.md decision D-5 mandates "integrity check auto-runs after every Agent Teams evaluation session." A disabled-by-default flag directly contradicts this decision. Plan 05 (the batch retry) depends on integrity running automatically — if the flag defaults to off, the retry will produce no integrity report.

**Action:** Request removal of the opt-out flag, or require it to default to `True`. The integrity check must be unconditional when an observation directory exists.

---

### DRIFT: Stage 9 re-invokes run_debrief() instead of persisting the existing debrief object

**Scenario:** The executor adds Stage 9 but implements it as a second call to `run_debrief()` from debrief_protocol.py, instead of persisting the debrief object already computed in Stage 6. Tests pass because the second call also produces valid data.

**Why DRIFT:** The plan explicitly states "Stage 9 is PERSISTENCE only: it takes the debrief object already computed in Stage 6 and writes it as a JSON artifact." Re-invoking run_debrief() wastes an LLM call (in non-simulated mode), may produce different results than what the reasoning evaluator received in Stage 6, and contradicts the plan's design constraint about keeping the Stage 6 call unmodified.

**Action:** Verify that Stage 9 serializes the `debrief` variable from Stage 6, not a freshly computed debrief. Check that there is exactly ONE `run_debrief()` call in the file (in Stage 6), not two.

---

### STOP: validate_batch() is never called inside evaluation_runner.py

**Scenario:** The executor adds PIPELINE_STAGES entries for "integrity_check" and "run_debrief", adds timing infrastructure, adds session_correlation_id, creates 13 integration tests — but the actual `validate_batch()` call is missing from the run() method. The integrity_check stage exists in the list but never executes. Tests pass because they mock the integrity stage.

**Why STOP:** The entire purpose of this plan is to WIRE the standalone validator (Plan 02) into the pipeline. Without the actual validate_batch() call, integrity checking is theatrical — the stage name appears in lists but no validation occurs. Results flow to baselines unchecked. Plan 05 (batch retry) will report PASS because the validator was never invoked, giving false confidence that the hardened harness works. This is the pre-plan state with extra metadata fields.

**Action:** Reject. Do not accept. Verify: `grep "validate_batch" tests/workflow_harness/lib/evaluation_runner.py` must return at least one match showing an actual function call (not just an import or comment).

## Reviewer Guidance

1. **Verify validate_batch() is actually called inside evaluation_runner.py run() method.** This is the primary failure risk. The plan's entire purpose is wiring — if the call is missing, all other additions (timing, tainting, correlation IDs) are cosmetic metadata on an unchecked pipeline. Run: `grep "validate_batch" tests/workflow_harness/lib/evaluation_runner.py` and confirm it shows a function call, not just an import.

2. **Verify Stage 9 persists the Stage 6 debrief object, not a re-invocation.** Check that there is exactly one `run_debrief()` call in evaluation_runner.py (in the Stage 6 block). Stage 9 should serialize the existing `debrief` variable. Two `run_debrief()` calls means a redundant LLM invocation and potential result inconsistency.

3. **Verify the DEGRADED escalation boundary test covers both sides (2 warnings and 3 warnings).** A single test for "3+ warnings escalates" is insufficient — the boundary at exactly 2 warnings (should NOT escalate) must also be verified. Look for a test function named something like `test_degraded_escalation_threshold` that tests both cases.

4. **Verify PipelineHealth.stage_durations field was added to models.py.** The plan requires a non-breaking addition to the Pydantic model. If this field is missing, `health.stage_durations = stage_durations` will raise an AttributeError at runtime. Run: `grep "stage_durations" src/alphaswarm_sol/testing/evaluation/models.py`.

5. **Verify integrity check does not prevent Stage 9 from running.** An integrity FAIL must not short-circuit the pipeline. Check that the integrity block is wrapped in try/except and that the debrief block runs unconditionally after it. An integration test should verify both rejection.json AND "run_debrief" in stages_completed for the same FAIL scenario.
