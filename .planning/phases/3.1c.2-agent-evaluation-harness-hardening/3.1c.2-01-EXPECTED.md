# Expected Results: Plan 01 — Evaluation delegate_guard Config + Env-Var Gating

**Plan:** 3.1c.2-01
**Phase:** 3.1c.2-agent-evaluation-harness-hardening
**Generated:** 2026-03-02

## Fundamental Deliverable

This plan delivers the evaluation-specific delegate_guard configuration (`delegate_guard_config_eval.yaml`) and env-var gated config loading in `delegate_guard.py`, so that evaluation agents are blocked at the tool-call level (exit code 2) from using Python imports, reading project context, or bypassing the CLI — replacing the prompt-based restrictions that Plan 12 Batch 1 proved are unenforceable. Without it, the entire 3.1c.2 hardening phase cannot proceed: Plan 02 (JSONL transcript verification) depends on the guard enforcing tool restrictions to validate against, Plan 04 (pipeline integration) wires the guard into the evaluation runner, and Plan 05 (Batch 1 retry) cannot produce trustworthy results because agents will bypass the CLI exactly as they did before. Success means `python -c "import alphaswarm"` returns exit code 2 (blocked by prefix-match separation of `allowed_file_reads` from `allowed_bash_commands`), while `uv run alphaswarm build-kg contracts/X.sol` returns exit code 0 (allowed by prefix match), and 9 canary tests prove both paths. Failure means the CRITICAL substring bypass (Gap 2.7) remains open — `allowed_reads: ["alphaswarm"]` still lets `Bash(python -c "import alphaswarm")` through — or the config exists but `delegate_guard.py` does not read it because env-var gating was not implemented.

## Alignment Criteria

### [must_have] Separated allowed_file_reads vs allowed_bash_commands closes the bypass
**Source:** must_haves.truths[0]
**The question:** Does the eval config use SEPARATE permission lists — `allowed_file_reads` for Read/Grep path exceptions and `allowed_bash_commands` for Bash prefix-match exceptions — instead of a single `allowed_reads` substring match?
**Aligned if:** `delegate_guard_config_eval.yaml` contains both `allowed_file_reads` and `allowed_bash_commands` as top-level keys. `delegate_guard.py` `main()` dispatches to different match logic per tool type: for Bash, it checks whether `tool_input["command"]` starts with an `allowed_bash_commands` entry (prefix match); for Read/Grep, it checks whether the file path contains an `allowed_file_reads` entry (substring match on the path parameter, not on `json.dumps(tool_input)`). The old single `allowed_reads` escape hatch that checked `json.dumps(tool_input)` as a whole is no longer the active code path when the eval config is loaded.
**Misaligned if:** The config still uses a single `allowed_reads` key. The code has `allowed_file_reads` and `allowed_bash_commands` keys but applies the same substring-on-json.dumps logic to both. The prefix match checks `input_str` (serialized JSON) instead of the raw `command` string.

### [must_have] Env-var gating loads eval config via _GUARD_PROFILE=strict
**Source:** must_haves.truths[1]
**The question:** Does `delegate_guard.py` `_load_config()` check `_GUARD_PROFILE=strict` and `DELEGATE_GUARD_CONFIG` env vars before falling through to the existing search-path logic?
**Aligned if:** `_load_config()` has an early-return path: when `os.environ.get("_GUARD_PROFILE") == "strict"`, it reads `DELEGATE_GUARD_CONFIG` and returns the parsed YAML. When the env var is set but the config path is missing/invalid, it calls `sys.exit(2)` (fail-fast — blocks ALL tool calls). When `_GUARD_PROFILE` is not set, the existing search-path logic runs unchanged.
**Misaligned if:** The function checks a different env var name (e.g., `ALPHASWARM_EVAL_MODE`). The function does not fail-fast on missing config — it silently falls through to dev config. The env-var check is after the search-path logic instead of before it.

### [must_have] Blocked tools include Glob, Grep, Edit with Read/Grep exceptions
**Source:** must_haves.truths[2]
**The question:** Does the eval config block Glob, Grep, and Edit in `blocked_tools`, with `allowed_file_reads` providing an exception path for Read and Grep only?
**Aligned if:** `delegate_guard_config_eval.yaml` has `blocked_tools` containing at minimum: Read, Write, Glob, Edit, Grep. In `delegate_guard.py`, when `tool_name` is Read or Grep and a blocked pattern matches, the code checks `allowed_file_reads` against the tool's file path parameter before blocking. When `tool_name` is Glob or Edit, no exception path exists — they are fully blocked.
**Misaligned if:** Glob, Grep, or Edit are missing from `blocked_tools`. All blocked tools share the same `allowed_file_reads` exception path (Glob getting an exception would defeat tool restriction). `blocked_tools` is present but `main()` does not check it.

### [must_have] Blocked patterns cover shell escape vectors
**Source:** must_haves.truths[3]
**The question:** Does the eval config's `blocked_patterns` list include git, base64, eval, exec, perl, ruby, and node?
**Aligned if:** `delegate_guard_config_eval.yaml` `blocked_patterns` contains at minimum: `"git "`, `"base64"`, `"eval "`, `"exec "`, `"perl "`, `"ruby "`, `"node "`. These block alternative interpreters, encoding escapes, and version control access that agents used as bypass vectors.
**Misaligned if:** Any of these patterns are missing. Patterns are present but without the correct spacing (e.g., `"git"` without space would block `"digital"` — though this is a judgment call on specificity).

### [must_have] python3 pattern catches python3.12
**Source:** must_haves.truths[4]
**The question:** Does the `blocked_patterns` entry for python3 omit a trailing space, ensuring it matches `python3`, `python3.12`, `python3.11`, etc.?
**Aligned if:** `delegate_guard_config_eval.yaml` contains `"python3"` (no trailing space) in `blocked_patterns`. A separate `"python "` (with trailing space) handles the base `python` interpreter. Together they cover all Python interpreter invocations. Test 2 in the canary suite specifically sends `python3.12 -c 'import os'` and asserts exit code 2.
**Misaligned if:** The config uses `"python3 "` (with trailing space), which would miss `python3.12`. Only `"python "` is present, missing `python3.X` entirely. The canary test for `python3.12` is absent or does not assert exit code 2.

### [must_have] Blocked-call logging provides forensic evidence
**Source:** must_haves.truths[5]
**The question:** Does `delegate_guard.py` write JSON records to the `DELEGATE_GUARD_LOG` env var path when calls are blocked?
**Aligned if:** A `_log_blocked()` function exists in `delegate_guard.py` that opens the file at `DELEGATE_GUARD_LOG`, appends a JSON record with at minimum: timestamp, tool_name, blocked_pattern, and a preview of tool_input. The function is called on every blocked-call path. A corresponding `_log_allowed()` function records exception-path hits. Both are best-effort (catch `OSError`, do not affect the block/allow decision). When `DELEGATE_GUARD_LOG` is not set, no logging occurs.
**Misaligned if:** No logging function exists. Logging is present but writes to a hardcoded path instead of the env var. Logging errors cause the guard to fail (blocking is affected by log failures). Only blocked calls are logged — no audit trail for exception-path hits.

## Drift Detection

### OK: Blocked patterns use slightly different spacing conventions
**Scenario:** The executor uses `"git "` (with trailing space) instead of `"git"` in blocked_patterns, or adds additional patterns beyond the minimum set (e.g., `"wget"`, `"curl"`, `"pip "`).
**Why OK:** The plan specifies minimum required patterns but grants discretion on "specific blocked_patterns entries beyond the minimum required set" (CONTEXT.md Claude's Discretion). Additional patterns strengthen enforcement without changing the plan's intent. Minor spacing variations are acceptable as long as the intended commands are still caught.
**Action:** Verify the minimum required patterns are present. Accept additional patterns as defense-in-depth.

### OK: _log_blocked writes additional fields beyond the minimum
**Scenario:** The executor's `_log_blocked()` includes additional fields like `pid`, `cwd`, `guard_config_path`, or `exit_code` beyond the minimum (timestamp, tool_name, blocked_pattern, tool_input_preview).
**Why OK:** The plan specifies the minimum record fields. Additional forensic data improves debugging without changing the logging contract.
**Action:** Verify minimum fields are present. Accept additional fields.

### DRIFT: Prefix match for allowed_bash_commands uses substring instead of startswith
**Scenario:** The executor implements `allowed_bash_commands` using `if allowed_cmd in command` (substring) instead of `if command.startswith(allowed_cmd)` (prefix). This means `echo "uv run alphaswarm" && python -c "import os"` would pass because `"uv run alphaswarm"` appears as a substring.
**Why DRIFT:** The entire plan exists to close the substring bypass (Gap 2.7). Using substring match for the Bash exception reintroduces a command-injection vector. An agent could prepend or embed the allowed prefix within a larger malicious command. The plan explicitly specifies "prefix match on the command string" in the objective and Task 2 action.
**Action:** Request correction to `command.startswith(allowed_cmd)`. This is the core security mechanism of the plan — substring match here is a regression, not a style choice.

### DRIFT: Eval config loaded but main() still uses old single-list logic
**Scenario:** `_load_config()` correctly reads the eval YAML with `allowed_file_reads` and `allowed_bash_commands`. But `main()` still uses `for allowed in allowed_reads: if allowed in input_str` — the old single-list substring check. The new config keys are loaded but never consumed.
**Why DRIFT:** The config separation is cosmetic only. The runtime behavior is identical to the old vulnerable code. The bypass test (Test 9) would fail, but if the test was written to match the broken behavior, this could go undetected.
**Action:** Request correction. Verify that `main()` dispatches to tool-specific match logic. The bypass test (Test 9) must assert exit code 2 for `python -c "import alphaswarm"`.

### STOP: Eval config missing or delegate_guard ignores env vars entirely
**Scenario:** `delegate_guard_config_eval.yaml` does not exist, or `delegate_guard.py` was not modified to check env vars. The existing dev config (`delegate_guard_config.yaml` with `blocked_patterns: [".sol"]`) is the only config available. All evaluation sessions will use the dev config, which provides zero protection against Python imports.
**Why STOP:** The plan's fundamental purpose is creating the enforcement primitive for the evaluation harness. Without the eval config or the env-var gating, there is no enforcement mechanism. Every downstream plan (02, 03, 04, 05) depends on this plan's output. The Plan 12 Batch 1 failure will be reproduced exactly because the same `.sol`-only config runs.
**Action:** Escalate immediately. No downstream plans can proceed. The entire 3.1c.2 phase is blocked.

## Reviewer Guidance

**Priorities:**

1. **Verify the Gap 2.7 bypass is closed.** This is the single highest-priority check. In `delegate_guard.py` `main()`, confirm that Bash tool handling uses `command.startswith(allowed_cmd)` (prefix match on the command string itself), NOT `allowed in input_str` (substring match on serialized JSON). Then confirm bypass test (Test 9) sends `python -c "import alphaswarm"` and asserts exit code 2. This is the plan's reason for existing.

2. **Confirm the two-list dispatch in main().** The config separation is only meaningful if `main()` actually dispatches differently: `allowed_bash_commands` for Bash (prefix match on command), `allowed_file_reads` for Read/Grep (substring match on file path). Check that these are two separate code paths, not a single loop over a merged list.

3. **Verify env-var gating with fail-fast.** In `_load_config()`, confirm the early-return path: `_GUARD_PROFILE=strict` + valid `DELEGATE_GUARD_CONFIG` returns eval config; `_GUARD_PROFILE=strict` + missing config calls `sys.exit(2)`. The fail-fast is a safety net — without it, a misconfigured evaluation session silently falls through to the permissive dev config.

4. **Check that canary test 9 is not a tautology.** Test 9 must send a command that would have PASSED under the old `allowed_reads: ["alphaswarm"]` logic but now FAILS under the new separated logic. Verify the test input contains "alphaswarm" as a substring (triggering the old escape hatch) while the command does NOT start with "uv run alphaswarm" (blocked under the new logic).

5. **Confirm pre-change snapshot timing.** Verify `.vrs/snapshots/3.1c.2/pre-change-sha.txt` contains a SHA that predates any modifications to `delegate_guard.py`. The SHA should correspond to a commit that does NOT include `delegate_guard_config_eval.yaml`.
