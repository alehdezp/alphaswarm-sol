## Base Assumptions

1. The CLI is the PRIMARY interface for agent evaluation — agents MUST use `alphaswarm build-kg` and `alphaswarm query`, not Python imports -- validated by Plan 12 Batch 1 failure where agents fell back to `from alphaswarm_sol.kg import KnowledgeGraph`
2. vulndocs/ (466 patterns) must be accessible from any working directory. For editable installs (development, Jujutsu workspaces): `__file__`-relative paths are valid because Jujutsu workspaces create real file copies (not symlinks) and editable installs resolve `__file__` to the main source tree. For non-editable pip installs: vulndocs must be declared in `package_data` (using `**` recursive glob, not single-level `*`) and accessed via `importlib.resources.files()`. Both resolution modes must be supported. Consumers of `vulndocs_read_path()` MUST NOT cast the result to `pathlib.Path` directly — they must use `Traversable` methods (`.iterdir()`, `.open()`, `.joinpath()`) or `importlib.resources.as_file()` context manager for operations requiring real filesystem paths. All vulndocs consumers (not just `_DEFAULT_VULNDOCS`) must use absolute paths anchored to the resolved root — no bare relative `Path()` constructions downstream of the entry point. Test code must also use the same resolution helpers, not hardcoded `Path("vulndocs")`.
3. Per-contract graph isolation uses a canonical contract identity specification: `key = sha256(os.path.realpath(contract_path).encode('utf-8')).hexdigest()[:12]`. For multi-file compilations (Slither treats compilation unit as atomic), identity = SHA-256 of sorted absolute realpaths of CLI-specified `.sol` inputs only — compiled dependency paths from Slitter's source_mapping are excluded from identity using `source_mapping.is_dependency` flag as primary filter, project-root prefix as fallback for raw solc. Single-file contracts degenerate to per-file hashing. `os.path.realpath()` (not `os.path.abspath()`) is required to canonicalize across Jujutsu workspace symlinks. Path strings are UTF-8 encoded, trailing slashes stripped, before hashing. All callers (build-kg, query, --graph resolution) must use a single shared `contract_identity()` utility.
4. The fix must be backward-compatible: existing `alphaswarm build-kg .` at project root still works. For query, if hash-based subdirectory does not exist, fall back to flat `graph.toon` (root flat file) with a deprecation warning. This prevents silent zero-result failures without requiring a migration step. All callers of `GraphStore.save()` and `GraphStore.load()` must be audited and updated to use hash-based paths or the new API before Plans 01-02 merge.
5. Testing uses REAL Agent Teams in Jujutsu-isolated workspaces — no pytest-only validation of CLI fixes. Prerequisites: (a) jujutsu must be installed (`jj --version` succeeds), (b) the repository must be a jujutsu co-located repo (`jj status` succeeds from repo root), (c) if jujutsu is not available, the fallback is `git worktree add` with reduced isolation (shared index). The preflight check must verify workspace infrastructure before any agent spawning.
6. Agent failures during testing are interrogated using a minimum viable protocol: (a) "What command did you run and what was the exact output?", (b) "What did you expect vs what happened?", (c) "What would you try next given that output?" Responses are stored in `.vrs/observations/plan04/debug-<session-id>-<attempt>.json`. Taxonomy and retry limits emerge from real Plan 04 sessions, not pre-designed.
7. All remaining pending phases (3.1c.2, 3.1f, 3.2, 4, 4.1, 6, 7, 5, 8) must be updated to mandate testing framework usage -- tiered by phase maturity: near-term phases (3.1c.2, 3.1f, 3.2) get full CONTEXT.md updates with specific testing gates; far-future phases (4+) get a ROADMAP.md template that any future CONTEXT.md must include when first authored. No pre-created CONTEXT.md files for undefined phases.
8. The testing framework is a living tool that improves alongside what it tests. Using the testing framework means running it AND recording its outputs as persistent artifacts. A phase that runs Agent Teams but discards transcripts has NOT satisfied the testing mandate. Each phase must produce at least one `.vrs/observations/{phase-id}/` artifact and a post-phase note in `.vrs/evaluations/progress.json`.
9. When BOTH a package-installed vulndocs tree AND an editable-install vulndocs tree exist simultaneously in the same environment, a `VulndocsPathConflict` error MUST be raised at module import time rather than silently using whichever Python's import machinery finds first.

- Jujutsu workspaces create real file copies, not symlinks (verified with jj 0.38.0).

### Prior Art / Related Systems

- `importlib.resources.files()` — Python 3.9+ standard for package data access from any cwd (read-only; returns `Traversable` objects that may not be writable). Consumers MUST use `Traversable` API (`.iterdir()`, `.open()`, `.joinpath()`) or wrap with `importlib.resources.as_file()` for operations requiring `os.stat()`, `glob()`, or YAML loading from path.
- `Path(__file__).resolve().parent` — simpler pattern for `__file__`-relative resolution, works for installed packages in editable mode
- Hash-based subdirectory storage — standard pattern for per-entity isolation (used by Jenkins workspaces, package caches)
- Jujutsu `jj workspace add` — established pattern for parallel isolated sessions sharing repo history
- `os.path.realpath()` vs `os.path.abspath()` — realpath follows symlinks to canonical path, abspath only makes relative absolute from cwd. Realpath is required for stable hashing across Jujutsu workspaces.

## Mini-Plans

### Plan 01: Absolute vulndocs Resolution [HIGH confidence]
**Intent:** Make `get_patterns()` resolve vulndocs via package-aware resolution instead of cwd-relative paths
**What it delivers:**
- `_DEFAULT_VULNDOCS` resolves to the actual vulndocs directory regardless of where CLI is invoked from
- Dual-mode resolution: `vulndocs_read_path()` using `importlib.resources.files()` for detection/query consumers; `vulndocs_write_path()` using `__file__`-relative (editable) or configurable `ALPHASWARM_VULNDOCS_DIR` env var (installed) for write operations (`/vrs:add-vulnerability`, `/vrs:refine`, `vulndocs validate --fix`)
- `VulndocsPathConflict` guard: detects when BOTH a package-installed AND editable-install vulndocs tree exist simultaneously, both containing `.yaml` files and resolving to different paths. Raised at module import time (not per-call), suppressible via `ALPHASWARM_SKIP_VULNDOCS_CONFLICT_CHECK=1` for CI environments that intentionally test both paths
- Traversable-safe API contract: consumers of `vulndocs_read_path()` MUST NOT cast to `pathlib.Path` directly. They must use `Traversable` methods (`.iterdir()`, `.open()`, `.joinpath()`) or `importlib.resources.as_file()` context manager for operations requiring `os.stat()`, `glob()`, or YAML loading from path. Done criterion: category traversal and pattern loader code passes a `Traversable` mock that raises `TypeError` on `__fspath__`.
- All downstream vulndocs consumers (category traversal, pattern loaders, sub-path constructions) verified to use absolute paths anchored to the resolved root — zero bare relative `Path()` constructions outside of `__file__`-anchored roots
- `pyproject.toml` includes vulndocs in `package_data` using `["vulndocs/**"]` recursive glob (not single-level `*`). Done criterion: after building a wheel and installing into a clean venv, `importlib.resources.files("alphaswarm_sol").joinpath("vulndocs/reentrancy").iterdir()` returns at least one `.yaml` file
- Test code uses the same resolution helpers, not hardcoded `Path("vulndocs")`
**What problem it solves:** Agents in Jujutsu workspaces get PatternDirectoryNotFoundError because `Path("vulndocs")` doesn't exist in their cwd. Non-editable installs also fail because vulndocs is absent from `site-packages` without `package_data` declaration. Partial-upgrade environments silently use stale pattern data from the wrong vulndocs root.
**Confidence:** HIGH — well-understood patterns; Jujutsu workspaces confirmed safe for `__file__`-relative (GAP-01 research); `importlib.resources` is stdlib

### Plan 02: Per-contract Graph Storage [HIGH confidence]
**Intent:** Store each contract's graph in a hash-based subdirectory using a canonical contract identity to prevent cross-contamination
**What it delivers:**
- Canonical contract identity specification: `contract_identity(paths: List[Path]) -> str` utility in a shared module, computing `sha256(sorted_realpaths.encode('utf-8')).hexdigest()[:12]`. Identity hash includes only CLI-specified `.sol` inputs — compiled dependency paths from Slither's source_mapping are excluded using `source_mapping.is_dependency` flag as primary filter, project-root prefix as fallback for raw solc.
- Project root resolution: `_resolve_project_root(inputs: List[Path]) -> Path` using `git rev-parse --show-toplevel` from `input[0].parent`. On `CalledProcessError` (non-git directories, bare repos, submodules): log `[warning] not a git repo, using cwd as project_root` to stderr, set `project_root = os.getcwd()`, store `project_root_type: "git_toplevel" | "fallback_cwd"` in meta.json for diagnostics.
- `GraphStore.save()` writes to `.vrs/graphs/{identity}/graph.toon` using atomic write semantics: write to `{hash_dir}/graph.toon.tmp`, then `os.replace()` to atomically rename. Default behavior: skip-if-exists (idempotent re-runs) with INFO log "Graph already exists for {stem}, skipping. Use --force to rebuild." `--force` flag propagates `force=True` to overwrite.
- `--check-fresh` flag: compute SHA256 of current contract files (same algorithm as graph_hash in meta.json), compare to stored `meta.json.graph_hash`. Emit to stderr: `[graph-cache] skipped: <stem> (hash match)` if fresh, or `[WARNING] cache mismatch: <stem> has changed on disk` if stale. Exit code 0 on fresh, 42 on stale for programmatic detection.
- `GraphStore.load()` raises `CorruptGraphError` if `.tmp` exists without `.toon` (partial write from killed build).
- For multi-file compilations (Slither compilation units): identity = SHA-256 of sorted absolute realpaths of CLI-specified files, with assertion that all included paths are under project root (WARNING for excluded dependency paths). Single-file contracts degenerate to per-file hashing.
- Backward compatibility: query falls back to flat `graph.toon` (root flat file) with deprecation warning if hash-based subdirectory doesn't exist
- Complete read-path migration: all callers of `GraphStore.save()` and `GraphStore.load()` (and any direct `.vrs/graphs/` path references in `kg/`, `tools/`, `orchestration/`) audited and updated. Done criterion: the read-path inventory at `.planning/phases/3.1c.1-cli-graph-isolation-hardening/read-path-inventory.md` lists every callsite; each callsite is marked MIGRATED or VERIFIED-USES-GRAPHSTORE. Additionally, broader grep for `'graphs'` with human review of each match catches dynamic path construction patterns.
- Pre-implementation read-path inventory at `.planning/phases/3.1c.1-cli-graph-isolation-hardening/read-path-inventory.md` documenting every callsite with (file, line, current assumption, owning plan)
**What problem it solves:** Sequential builds overwrite each other; Agent A's graph disappears when Agent B builds. Directory vs file scope ambiguity resolved by using compilation unit as atomic identity. Stale cached graphs silently return wrong results after contract changes.
**Confidence:** HIGH — standard storage pattern; Slither compilation model confirmed (GAP-02 research); dependency filtering confirmed (GAP-03 research); canonical identity spec eliminates hash divergence

### Plan 03: Query Routing Fix + --graph Flag [HIGH confidence]
**Intent:** Fix silent query degradation and add explicit graph targeting with agent-actionable errors
**What it delivers:**
- `alphaswarm query` errors when multiple graphs exist and `--graph` is not specified; only auto-selects when exactly one graph is present (no mtime-based auto-discovery — prevents silent wrong-graph selection under concurrency). Auto-select applies only when `graph.toon` exists and `.tmp` is absent.
- `--graph` flag accepts EITHER: (a) a graph subdirectory path (`.vrs/graphs/<hash>/`) used directly, OR (b) a contract filename stem (e.g., `Token`) matched against graph metadata (`source_contract` field in per-graph `meta.json`). Stem-based lookup is path-agnostic and survives worktree moves. If multiple graphs share a stem (e.g., `contracts/v1/Token.sol` and `contracts/v2/Token.sol`), error with exit code 1: `Error: ambiguous --graph 'Token' matches multiple graphs — use full path: contracts/Token.sol, legacy/Token.sol`. Available paths read from `meta.json.contract_paths`. If stem does not match any graph: exit 1 with `Error: no graph found for '--graph Token' — available graphs: Vault, MasterChef. Run 'alphaswarm build-kg contracts/Token.sol' to build.` Available stems derived from `meta.json` in all `.vrs/graphs/` subdirs (shared listing utility).
- CLI I/O Channel Contract: stdout = query result rows only (plus the `# result:` header line). stderr = all errors, warnings, and informational messages including `Available:` lists. Show all available stems without truncation; if count exceeds 20, add "... and N more" note.
- Structured result header: first stdout line for every successful query is `# result: graph_nodes={N} matches={M}` (even for M=0). This is always present when graph loaded successfully. M=0 on a valid graph is NOT an error — agents must use the header line to confirm graph loaded. Query stdout contract: first line always matches `^# result: graph_nodes=\d+ matches=\d+$`, followed by JSON array (one object per line). No other lines begin with `#`. This is the authoritative parsing rule for all consumers.
- Exit code semantics: 0 = query executed (M may be 0); 1 = routing/argument error (no graph found, ambiguous stem, non-existent stem); 2 = graph load failure (file corrupt, missing, `.tmp` present without `.toon`)
- Both error paths exit with code 1 (not 0) for machine-parseable failure detection
- Query router must validate `meta.json` `schema_version` at load time; fail fast with `[error] meta.json schema version mismatch` if version does not match expected
**What problem it solves:** Queries silently return 0 results instead of error; no way to target a specific contract's graph; mtime auto-discovery silently returns wrong graph under concurrent agent builds; agents cannot distinguish zero-results from broken-graph; no parsing contract for consumers piping stdout
**Confidence:** HIGH — builds on Plan 01 + 02 deliverables
**Depends on:** Plan 01, Plan 02 (hard sequential dependency — Plan 03 is meaningless without working vulndocs resolution and per-contract graph subdirs)

### Plan 04: CLI Isolation Smoke Tests with Agent Teams [MEDIUM confidence]
**Intent:** Validate CLI fixes using REAL Agent Teams in Jujutsu-isolated workspaces with defined pass/fail criteria and diagnostic output
**Scope boundary:** This plan performs BINARY smoke tests (CLI ran? Results non-empty? Correct graph?). It does NOT perform agent debriefs or structured post-session interviews — that is Phase 3.1c.2 Plan 05's responsibility. **What it delivers:**
- Framework pre-validation step before any agent spawning: (a) WorkspaceManager: create workspace, verify directory, tear down; (b) TeamManager: create team with echo agent, verify parseable observation; (c) EvaluationRunner: run pipeline on synthetic transcript. Only proceed to agent spawning if all three pass. This disambiguates "CLI is broken" from "WorkspaceManager is broken."

Anti-patterns to flag: agent reads .py source directly, constructs KnowledgeGraph in Python, or says "I will use Python to query"
- Smoke test definition: minimum (a) 2 distinct test contracts with at least one discriminating pattern each (a pattern present in X but absent in Y, and vice versa), (b) 1 `build-kg` per contract from Jujutsu workspace, (c) 3 CLI queries per contract targeting known patterns, (d) 1 concurrent build scenario (two agents building different contracts simultaneously) to validate graph isolation, (e) 1 concurrent same-contract scenario (two `build-kg` calls on same contract simultaneously) to validate atomic write integrity. - Concurrent isolation verification: ground-truth established first by sequential builds — run `query "<x_only_pattern>" --graph X` and `query "<y_only_pattern>" --graph Y` to identify discriminating patterns. - Concurrent same-contract scenario: launch two `build-kg <same-contract>` simultaneously from different workspaces; assert `meta.json` is valid JSON with correct fields (not partial or zero-byte). This tests whether Plan 02 delivered atomic writes without assuming it.
- meta.json schema validation test: load `meta.json`, assert all required fields present and types correct per D-meta schema, assert `schema_version` matches expected value
- Staleness detection test: build contract, modify source file, run `build-kg --check-fresh`, assert exit code 42 and stale warning emitted
- Non-git directory test: create temp dir (not a git repo), run `build-kg`, assert exit 0, warning logged, `meta.json` contains `project_root_type: "fallback_cwd"`
- Query output parsing test: run query, capture stdout, verify first line matches `^# result:` regex, verify subsequent lines are valid JSON or empty, verify no line except first begins with `#`
- Minimum viable interrogation protocol: 3 structured questions on failure, responses stored in `.vrs/observations/plan04/debug-<session-id>-<attempt>.json`. If agent cannot answer "what command did you run," escalate to pre-validation (infrastructure broken).
- Diagnostic output contract: agents report failures with self-contained JSON schema: `{session_id, attempt, timestamp, contract_name, command_run, exit_code, stdout_excerpt, stderr_excerpt, root_cause_hypothesis}`. ObservationRecord compatibility is NOT a Plan 04 concern — Phase 3.1c.3 will define transforms if needed. Agents diagnose; orchestrator decides remediation.
**What problem it solves:** Proves the CLI actually works from isolated contexts; discovers unknown CLI issues through agent behavior; validates the testing framework itself; catches cross-contamination regressions with falsifiable tests
**Phase 6 integration:** This plan IS the testing framework being used — it validates both the CLI fixes AND the interactive testing approach for all downstream phases
**Confidence:** MEDIUM — CLI fixes are testable; interactive debugging protocol is now scaffolded (interrogation protocol + diagnostic contract defined) but exact failure patterns will emerge from sessions

## Implementation Decisions

### vulndocs Resolution (D-1) — DECIDED (refined)
Dual-mode resolution for vulndocs:
- **Read path:** `importlib.resources.files("alphaswarm_sol").joinpath("vulndocs")` for detection/query consumers (layout-agnostic, works across editable and non-editable installs, handles zip-safe packages). Returns a `Traversable` — consumers MUST NOT cast to `pathlib.Path` directly. Use `Traversable` methods (`.iterdir()`, `.open()`, `.joinpath()`) or `importlib.resources.as_file()` context manager for operations requiring `os.stat()`, `glob()`, or YAML loading from path.
- **Write path:** `Path(__file__).resolve().parent / ...` for editable installs; `ALPHASWARM_VULNDOCS_DIR` env var override for installed packages. Write operations (`/vrs:add-vulnerability`, `/vrs:refine`, `vulndocs validate --fix`) need writable paths.
- **Conflict detection:** `_assert_single_vulndocs_root()` detects when both package-installed and editable vulndocs trees exist with `.yaml` files at different resolved paths. Raised at module import time (not per-call). Suppressible via `ALPHASWARM_SKIP_VULNDOCS_CONFLICT_CHECK=1`.
- `pyproject.toml` must declare vulndocs in `package_data` using `["vulndocs/**"]` recursive glob — not single-level `*` which silently omits nested subdirectories in some setuptools versions.
- All vulndocs consumers (category traversal, pattern loaders, sub-path constructions) must be audited to ensure zero bare relative `Path()` constructions downstream of the resolved root. Test code must use the same resolution helpers.

### ALPHASWARM_VULNDOCS_DIR Contract (D-1a) — DECIDED (new)
When `ALPHASWARM_VULNDOCS_DIR` is set, it overrides BOTH `vulndocs_read_path()` and `vulndocs_write_path()`. The package-bundled vulndocs are completely bypassed.
- **Validation:** On first call to either path function. If path does not exist: raises `VulndocsConfigError('ALPHASWARM_VULNDOCS_DIR={path} does not exist')`. If path exists but is not a directory: raises `VulndocsConfigError('... is not a directory')`. If directory exists and is empty of YAML files: logs WARNING but does not raise — empty override is a valid bootstrap state.

### Contract Identity Specification (D-2a) — DECIDED (refined)
A single canonical contract identity specification used by all components:
- **Canonical form:** `os.path.realpath()` of the primary .sol file (or sorted realpaths for multi-file compilations)
- **Encoding:** UTF-8 normalized, trailing slashes stripped
- **Hash algorithm:** SHA-256 of canonical form, truncated to 12 hex chars
- **Multi-file (compilation unit):** Slither treats compilation unit as atomic. Identity = SHA-256(sorted absolute realpaths of CLI-specified `.sol` inputs only). Compiled dependency paths from Slither's `source_mapping` are excluded using `source_mapping.is_dependency` flag as primary filter, project-root prefix as fallback for raw solc. Single-file contracts degenerate to per-file hashing.
- **Project root resolution:** `git rev-parse --show-toplevel` from `input[0].parent`. On failure: log warning, use `os.getcwd()` as project_root, store `project_root_type` in meta.json.
- **Implementation:** Single shared utility `contract_identity(paths: List[Path]) -> str` in a shared module that all callers import (build-kg, query, --graph resolution). All included paths must be under project root; WARNING for excluded dependency paths.
- **Rationale:** IMP-06, IMP-07, IMP-10, IMP-13 were four symptoms of the same missing abstraction. A single spec prevents incompatible identity computations across build, query, and --graph commands.

### Canonical meta.json Schema (D-meta) — DECIDED (new)
Authoritative cross-plan interface contract between Plan 02 (producer) and Plan 03 (consumer):
```json
{
  "schema_version": 1,
  "built_at": "<ISO-8601>",
  "graph_hash": "<sha256 hex of contract files, repo-relative inputs only>",
  "contract_paths": ["<repo-relative path>", ...],
  "stem": "<last path component without extension>",
  "source_contract": "<filename stem for --graph lookup>",
  "identity": "<12-char hex>",
  "slither_version": "<version>",
  "project_root_type": "git_toplevel | fallback_cwd"
}
```
Validation rules: all fields mandatory, string lengths bounded, no extra fields allowed. Plan 03 must validate `schema_version` at load time; fail fast with `[error] meta.json schema version mismatch` if version does not match expected. Plan 04 smoke tests must validate full schema.

### Per-Contract Graph Storage (D-2b) — DECIDED (refined)
Hash-based subdirectories: `.vrs/graphs/{contract_identity}/graph.toon`.
- Each graph subdirectory includes `meta.json` per D-meta schema.
- Atomic writes: `GraphStore.save()` writes to `{hash_dir}/graph.toon.tmp`, then `os.replace()`. Default: skip-if-exists. `--force` flag overwrites.
- `--check-fresh` flag: computes current contract file hash, compares to `meta.json.graph_hash`. Exit code 0 if fresh, 42 if stale.
- **Backward compatibility:** If hash-based subdirectory does not exist, fall back to flat `graph.toon` with deprecation warning. This prevents silent zero-result failures for pre-upgrade operators.
- All callers of `GraphStore.save()` and `GraphStore.load()` must be audited and updated before Plans 01-02 merge.

### Query Graph Targeting (D-3) — DECIDED (refined)
- **Default behavior:** Error when multiple graphs exist and `--graph` is not specified. Only auto-select when exactly one graph is present. No mtime-based auto-discovery — prevents silent wrong-graph selection under concurrent agent builds. Auto-select only when `graph.toon` exists and `.tmp` is absent.
- **--graph flag:** Accepts EITHER (a) a graph subdirectory path (`.vrs/graphs/<hash>/`) used directly, OR (b) a contract filename stem (e.g., `Token`) matched against `meta.json` `source_contract` field. Stem-based lookup is path-agnostic.
- **Stem disambiguation:** If multiple graphs share a stem, error with exit code 1 listing full paths from `meta.json.contract_paths`. If stem matches nothing, error with exit code 1 listing available graphs with build suggestion.
- **Error messages:** Agent-actionable with embedded recovery commands and exit code 1 for machine-parseable failure detection.
- **Query stdout parsing contract:** First line always matches `^# result: graph_nodes=\d+ matches=\d+$`, followed by JSON array (one object per line). No other lines begin with `#`. stdout = data only; stderr = all errors/warnings/info. This is the authoritative parsing rule for all consumers.

### Backward Compatibility (D-4) — DECIDED
Existing `alphaswarm build-kg .` at project root must still work. The `--graph` flag is optional. Backward-compatible read fallback to flat graph files with deprecation warning ensures existing workflows don't break.

### Testing Approach — DECIDED (refined)
