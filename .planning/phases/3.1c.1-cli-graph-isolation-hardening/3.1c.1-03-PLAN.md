# Plan 03: Query Routing Fix + --graph Flag

**Goal:** Fix silent query degradation to raw node search (returning empty/meaningless results) and add explicit graph targeting via `--graph` flag with error-when-ambiguous default, agent-actionable error messages, and a machine-parseable stdout contract.

**Trigger:** Plan 3.1c-12 Batch 1 calibration FAILED because agents' queries silently returned 0 results (query router fell back to "nodes" when patterns were empty), giving agents no useful information and no indication of what went wrong.

**Context decisions:** D-3 (query graph targeting), D-meta (meta.json schema consumed by --graph lookup). All DECIDED and locked in CONTEXT.md.

**Hard dependency:** Plans 01 and 02 MUST be fully merged before this plan begins execution. Plan 03 requires working vulndocs resolution (Plan 01) for pattern loading, and hash-based graph subdirectories with meta.json (Plan 02) for --graph stem lookup. Without these, this plan is meaningless.

## Tasks

<task id="1">
  <name>Task 1: Fix Silent Query Degradation in intent.py</name>
  <objective>Eliminate the silent fallback from pattern-based query to raw "nodes" query when patterns fail to load or match, replacing it with explicit error messages.</objective>
  <what>
    Modify `src/alphaswarm_sol/queries/intent.py`:

    **Current behavior (broken):**
    - L226: When `patterns` is empty (0 matching rules), `query_kind` defaults to "nodes"
    - This silently returns raw graph nodes instead of vulnerability pattern matches
    - Agents get data but it's meaningless for security analysis

    **Fixed behavior:**
    - When patterns fail to load (vulndocs resolution error): error with exit code 1, message: `Error: could not load vulnerability patterns. Run 'alphaswarm build-kg' from project root or set ALPHASWARM_VULNDOCS_DIR.`
    - When patterns load but 0 match the query: report `# result: graph_nodes={N} matches=0` on stdout (this is a valid result, not an error). The agent can see the graph loaded but the query didn't match any patterns.
    - NEVER silently fall back to "nodes" query kind. The "nodes" fallback is removed entirely for the `query` CLI command. If the query cannot be interpreted as a pattern query, error with guidance.

    Preserve backward compatibility for programmatic callers that explicitly request "nodes" query kind through the Python API.
  </what>
  <done_criteria>
    - Zero silent degradation to "nodes" when query is sent through CLI `alphaswarm query`
    - Pattern load failures produce agent-actionable error on stderr + exit code 1
    - Valid query with 0 pattern matches produces `# result: graph_nodes={N} matches=0` on stdout + exit code 0
    - Programmatic "nodes" query kind still works through Python API (backward compat)
  </done_criteria>
  <verify>
    - Run query with valid graph + no-match pattern: verify stdout has `# result:` header with matches=0, exit code 0
    - Run query with missing vulndocs: verify stderr has actionable error, exit code 1
    - Run query through Python API with explicit kind="nodes": verify still returns raw nodes
  </verify>
</task>

<task id="2">
  <name>Task 2: Implement Error-When-Ambiguous Graph Resolution</name>
  <objective>Make `alphaswarm query` refuse to guess when multiple graphs exist, erroring with available graph list instead of silently picking the wrong one.</objective>
  <what>
    Add graph resolution logic to `src/alphaswarm_sol/cli/main.py` for the `query` command:

    **Graph discovery:**
    - Scan `.vrs/graphs/` for subdirectories containing `graph.toon` (without `.tmp`)
    - If 0 valid graphs: error on stderr `Error: no graphs found. Run 'alphaswarm build-kg <path>' first.` Exit code 2.
    - If 1 valid graph: auto-select it (the only unambiguous case)
    - If 2+ valid graphs and `--graph` not specified: error on stderr `Error: {N} graphs found. Specify --graph <stem>.` followed by `Available: {comma-separated stems}`. Exit code 1.

    **Shared graph listing utility:** Add `list_available_graphs(root: Path) -> list[GraphInfo]` to `kg/store.py` (or a new `kg/discovery.py`). Each `GraphInfo` has `identity`, `stem`, `source_contract`, `dir_path`, `meta`. This utility is shared by graph resolution and --graph flag.

    Apply same resolution logic to `suggest`, `schema_snapshot`, `detect`, and other CLI commands that previously hardcoded `Path(".vrs/graphs/graph.json")`.
  </what>
  <done_criteria>
    - 0 graphs: stderr error + exit code 2
    - 1 graph: auto-selected, query runs
    - 2+ graphs without --graph: stderr error with available stems + exit code 1
    - All CLI commands using graph resolution updated (not just `query`)
    - Shared listing utility exists and is reusable
  </done_criteria>
  <verify>
    Integration tests:
    - Build 0 graphs, run query: verify exit 2 and "no graphs found" message
    - Build 1 graph, run query: verify auto-select and successful result
    - Build 2 graphs, run query without --graph: verify exit 1 and "Available:" list
  </verify>
</task>

<task id="3">
  <name>Task 3: Implement --graph Flag with Stem-Based Lookup</name>
  <objective>Add `--graph` flag to `alphaswarm query` (and other graph-consuming commands) supporting both direct path and stem-based lookup against meta.json.</objective>
  <what>
    Add `--graph` option to CLI commands in `src/alphaswarm_sol/cli/main.py`:

    **`--graph` accepts EITHER:**
    - (a) A graph subdirectory path (`.vrs/graphs/{hash}/`) -- used directly
    - (b) A contract filename stem (e.g., `Token`) -- matched against `meta.json` `source_contract` field in all graph subdirectories

    **Stem-based lookup:**
    - Load `meta.json` from each `.vrs/graphs/{hash}/` subdirectory
    - Match `--graph Token` against `source_contract` field
    - Validate `schema_version` at load time; fail fast with `[error] meta.json schema version mismatch` if version != expected
    - If exactly 1 match: use that graph
    - If multiple matches (stem collision): exit 1 with `Error: ambiguous --graph 'Token' matches multiple graphs -- use full path: contracts/Token.sol, legacy/Token.sol` (paths from `meta.json.contract_paths`)
    - If 0 matches: exit 1 with `Error: no graph found for '--graph Token' -- available graphs: Vault, MasterChef. Run 'alphaswarm build-kg contracts/Token.sol' to build.` (stems from all meta.json files)

    **Show all available stems** without truncation; if count exceeds 20, add "... and N more" note.

    **Apply --graph to:** query, suggest, schema-snapshot, detect, label, beads, and any other graph-consuming commands.
  </what>
  <done_criteria>
    - `--graph` flag works with both direct path and stem
    - Stem lookup matches against `meta.json` `source_contract` field
    - `schema_version` validated at load time
    - Ambiguous stem: error with full paths from meta.json
    - Non-existent stem: error with available graphs and build suggestion
    - All graph-consuming CLI commands support --graph
  </done_criteria>
  <verify>
    Integration tests:
    - `--graph {hash-dir}`: direct path, verify graph loaded
    - `--graph Token` (unique stem): verify correct graph loaded
    - `--graph Token` (ambiguous): verify exit 1 with disambiguation paths
    - `--graph NonExistent`: verify exit 1 with available graph list + build suggestion
    - meta.json with wrong schema_version: verify fail-fast error
  </verify>
</task>

<task id="4">
  <name>Task 4: Implement Structured Query Output Contract</name>
  <objective>Make query stdout machine-parseable per the CLI I/O channel contract from CONTEXT.md D-3.</objective>
  <what>
    Implement the structured output contract for `alphaswarm query`:

    **stdout contract:**
    - First line: `# result: graph_nodes={N} matches={M}` (always present when graph loaded successfully)
    - Subsequent lines: JSON objects (one per line), or empty if M=0
    - No other lines begin with `#`
    - This is the authoritative parsing rule for all consumers

    **stderr contract:**
    - All errors, warnings, informational messages go to stderr
    - `Available:` lists go to stderr
    - Diagnostic messages from graph loading go to stderr

    **Exit code semantics:**
    - 0 = query executed successfully (M may be 0 -- this is valid, not an error)
    - 1 = routing/argument error (no graph found, ambiguous stem, non-existent stem)
    - 2 = graph load failure (file corrupt, missing, `.tmp` present without `.toon`)
  </what>
  <done_criteria>
    - First stdout line always matches `^# result: graph_nodes=\d+ matches=\d+$` on successful query
    - Subsequent stdout lines are valid JSON objects or empty
    - No stdout line except the first begins with `#`
    - All errors and warnings on stderr
    - Exit codes follow 0/1/2 semantics
  </done_criteria>
  <verify>
    Integration tests:
    - Run query, capture stdout: first line matches regex `^# result: graph_nodes=\d+ matches=\d+$`
    - Run query, capture stdout: all non-header lines parse as valid JSON
    - Run query with error condition: stderr has error, stdout is empty, exit code 1 or 2
    - Run query with 0 matches: exit code 0, stdout has `matches=0`
  </verify>
</task>

## must_haves

Derived from phase goal using goal-backward methodology:

1. **No silent degradation**: Query NEVER falls back to "nodes" when invoked through CLI. Zero-match is reported explicitly as `matches=0`, not as raw node data.
2. **Error-when-ambiguous**: Multiple graphs without --graph flag produces error with available stems. No mtime-based auto-discovery (prevents silent wrong-graph selection under concurrency).
3. **--graph flag works**: Both direct path and stem-based lookup. Stem resolved against meta.json `source_contract` field (from Plan 02).
4. **Agent-actionable errors**: Every error message includes a recovery command. Agents can parse exit codes to distinguish routing errors (1) from graph errors (2) from success (0).
5. **Structured stdout contract**: First line always `# result: graph_nodes={N} matches={M}`. This is the authoritative parsing rule for all consumers (Plan 04 agents, future CI pipelines).
6. **Exit code semantics**: 0/1/2 consistently applied across all graph-consuming CLI commands.
