# Expected Results — Plan 3.1c.1-05a

**Phase:** 3.1c.1-cli-graph-isolation-hardening
**Generated:** 2026-03-01

## Fundamental Deliverable

This plan delivers a phase-type taxonomy and testing gate template that makes the cross-phase testing mandate enforceable rather than aspirational, so that Plan 05b can roll the mandate out to real phases (3.1c.2, 3.1f, 3.2) with concrete, type-specific acceptance bars. Without it, the testing mandate applies identical requirements to fundamentally different phase types — demanding Agent Teams from a documentation phase or snapshot diffs from a pure evaluation phase — producing either meaningless compliance or impossible requirements. Success means four phase types (CODE, EVALUATION, FRAMEWORK, SYNTHESIS) have measurable, non-vague testing requirements; the mixed-type protocol (CODE+FRAMEWORK) produces a non-contradictory union verified against 3.1c.1; the testing gate template is copy-paste ready with mechanical snapshot protocol instructions; and TESTING-FRAMEWORK.md makes the taxonomy discoverable. Failure means Plan 05b has no concrete design to roll out, phase planners cannot determine what testing they need, and the 3.1c.2 bootstrap exception remains undefined — leaving FRAMEWORK-type phases with no actionable acceptance validation path.

## Drift Detection

### OK — Testing requirements per type differ slightly from CONTEXT.md wording
**Scenario:** The executor used different phrasing for a type's requirements — e.g., EVALUATION says "independent dual evaluator scoring with disagreement threshold" instead of "dual-Opus evaluator" — because the taxonomy document aims for model-agnostic language.
**Why OK:** The must_haves require measurable testing requirements per type, not verbatim reproduction of CONTEXT.md wording. "Independent dual evaluator scoring" is equivalent to "dual-Opus evaluator" in substance. The taxonomy is a canonical reference that may refine language for clarity.
**Action:** Verify the requirement is still measurable and names a specific validation mechanism. If the rewording preserves the same testable behavior (two independent evaluations, disagreement detection), classification is correct.

### OK — Template includes additional optional fields beyond the CONTEXT.md specification
**Scenario:** The testing gate template includes an optional "Regression baseline" field and a "Phase dependencies" field that were not in the original CONTEXT.md specification, because the executor thought phase planners would find them useful.
**Why OK:** The must_haves require the template to be copy-paste ready and include all fields from the CONTEXT.md mandate. Additive optional fields do not violate any must_have — they extend without contradicting. The template remains usable with or without the extra fields.
**Action:** Verify the additional fields are clearly marked as optional and do not change the meaning of the required fields. Confirm all CONTEXT.md-mandated fields are still present.

### DRIFT — Snapshot Protocol uses prose instructions instead of exact commands
**Scenario:** The executor wrote the Snapshot Protocol as "Record your jj change ID before starting modifications. Store it in the snapshots directory. To restore, use jj to checkout the recorded state." The instructions are clear in intent but contain no copy-paste commands.
**Why DRIFT:** must_haves.truths[3] explicitly requires "exact commands, exact file paths, exact verification steps." The CONTEXT.md decision specifies concrete commands (`jj log -r @ --no-graph -T change_id`) and exact paths (`.vrs/snapshots/{phase}/pre-change-id.txt`). Prose instructions require interpretation — the executor must figure out the right jj flags and directory structure. This defeats the "mechanical" requirement.
**Action:** Request correction. The Snapshot Protocol section must include the exact jj command, the exact file path with `{phase}` placeholder, the exact restoration command, and the exact verification step. Compare against CONTEXT.md Task 2 specification for the required content.

### DRIFT — Taxonomy omits the intelligence_contribution metadata field
**Scenario:** The executor defined all four types with testing requirements but did not include the `intelligence_contribution` optional field, reasoning that it is metadata annotation only and not an enforcement gate.
**Why DRIFT:** While `intelligence_contribution` is explicitly described as optional and non-enforcement, the plan's Task 1 done criteria explicitly lists "intelligence_contribution field documented" as a completion requirement. The field enables Phase 3.1c.3 to trace data lineage from phases to Tier 2 intelligence modules. Omitting it breaks that traceability contract even though it is not a testing gate.
**Action:** Request addition of the `intelligence_contribution` field with its description as an optional metadata annotation. The field's presence matters for downstream Phase 3.1c.3 even if it is not enforced.

### STOP — Taxonomy is a summary table without measurable requirements
**Scenario:** The executor created `PHASE-TYPE-TAXONOMY.md` as a 4-row table with columns "Type | Description | Testing" where the Testing column contains entries like "Standard testing" (CODE), "Evaluation-focused testing" (EVALUATION), "Infrastructure testing" (FRAMEWORK), and "Review-based testing" (SYNTHESIS). No specific validators, check counts, or artifact requirements are named.
**Why STOP:** The entire purpose of this plan is to make the testing mandate enforceable by defining TYPE-SPECIFIC, MEASURABLE requirements. A taxonomy with vague testing descriptions is strictly worse than no taxonomy — it gives the illusion of compliance while providing no actionable acceptance criteria. Plan 05b would roll out a meaningless mandate. Phase planners filling in the testing gate template would have no way to determine what specific validation their phase needs. The fundamental deliverable — discriminating testing requirements per phase type — was not delivered.
**Action:** Escalate. Do not proceed to Plan 05b. The taxonomy must name specific validators (AgentExecutionValidator, dual-Opus evaluator), specific check counts or artifact requirements (12 checks, at least 1 red-path session), and specific mechanisms (snapshot diff, human review checklist) per type. Verify by reading the taxonomy and asking: "Could two different reviewers independently determine whether a CODE phase satisfies its testing requirements?" If not, the requirements are not measurable.

## Reviewer Guidance

**Priorities:**

1. **Measurability is the single most important quality.** Every type's testing requirements must name specific tools, validators, or check counts. Read each type's requirements and ask: "Could I write a done criterion from this?" If the answer is "only by adding interpretation," the requirement is too vague. This is the primary failure mode for governance design documents.

2. **Snapshot Protocol mechanicality determines downstream usability.** The protocol will be copy-pasted into phase CONTEXT.md files by Plan 05b and executed by phase implementors who may not understand jj internals. Verify the exact commands are present by checking for `jj log -r @` and `.vrs/snapshots/{phase}/pre-change-id.txt` in the template. Prose instructions that require jj knowledge to interpret are a drift signal.

3. **Mixed-type union verification for CODE+FRAMEWORK.** The plan explicitly calls out 3.1c.1 as the worked example. Read the taxonomy's CODE requirements and FRAMEWORK requirements, list their union, and verify no contradiction. The most likely contradiction: one type implying isolation (FRAMEWORK snapshot in separate workspace) while another implies integration (CODE Agent Teams in the main workspace). Both should be satisfiable independently.

4. **TESTING-FRAMEWORK.md modification scope.** The plan requires adding a reference pointer section and explicitly prohibits modifying existing content. Verify the existing sections (Agent Team Spawning Protocol, etc.) are unchanged. This is a low-probability but high-impact failure — if existing framework content is accidentally modified, it could break Plan 04 or other consumers.

5. **Template completeness against CONTEXT.md mandate.** Cross-reference the template fields against the CONTEXT.md "Cross-Phase Testing Mandate" decision. Every element of the mandate (phase type, enforcement mechanism, framework contribution, snapshot protocol, bootstrap exception reference) should have a corresponding template field or section. Missing fields mean Plan 05b will produce incomplete testing gates.
