# Plan 01: Absolute vulndocs Resolution

**Goal:** Make vulndocs resolution work from ANY working directory (Jujutsu workspaces, non-project paths, wheel installs) by replacing all cwd-relative `Path("vulndocs")` constructions with centralized, package-aware resolution functions.

**Trigger:** Plan 3.1c-12 Batch 1 calibration FAILED because agents in Jujutsu workspaces got `PatternDirectoryNotFoundError` from `Path("vulndocs")` resolving relative to cwd.

**Context decisions:** D-1 (vulndocs resolution), D-1a (ALPHASWARM_VULNDOCS_DIR contract), D-2c (read-path inventory). All DECIDED and locked in CONTEXT.md.

<task id="2" type="research" decision_gate="true">
  <name>Task 2: Validate vulndocs Namespace Collision</name>
  <objective>Determine whether `importlib.resources.files("alphaswarm_sol").joinpath("vulndocs")` collides with the existing `alphaswarm_sol.vulndocs` Python subpackage when vulndocs YAML data is bundled via `force-include` in pyproject.toml.</objective>
  <method>mini-poc</method>
  <what>
    Build a wheel with vulndocs YAML data force-included at `alphaswarm_sol/vulndocs` (same as existing subpackage).
    Install into a clean venv.
    Test: `python -c "import importlib.resources; print(list(importlib.resources.files('alphaswarm_sol').joinpath('vulndocs').iterdir())[:3])"`
    Test: `python -c "from alphaswarm_sol.vulndocs import ingestion; print(ingestion)"` (verify Python module still importable).
    If collision: try alternate path `alphaswarm_sol/_vulndocs_data` and re-test.
  </what>
  <output>.planning/phases/3.1c.1-cli-graph-isolation-hardening/research/vulndocs-namespace-collision.md</output>
  <decision_gate>
    <branch finding="no-collision">Continue Tasks 3-6 as planned using `importlib.resources.files("alphaswarm_sol").joinpath("vulndocs")`</branch>
    <branch finding="collision-use-alternate-path">Adapt Tasks 3-6 to use `_vulndocs_data` as the bundled data path. `vulndocs_read_path()` calls `.joinpath("_vulndocs_data")` instead of `.joinpath("vulndocs")`</branch>
    <branch finding="unfeasible">importlib.resources cannot resolve bundled data alongside a Python subpackage with the same name in any configuration. Accept reality: use `__file__`-relative resolution for ALL cases (editable and wheel), add a runtime check that the path exists. Document the limitation.</branch>
    <branch finding="phase-breaking">The packaging system cannot bundle vulndocs YAML data into wheels at all (hatch/setuptools limitation). Hard stop, surface to user.</branch>
  </decision_gate>
  <verify>Research document exists with wheel build evidence, import test results, and clear finding (A/B/unfeasible/phase-breaking).</verify>
</task>

<task id="3">
  <name>Task 3: Implement vulndocs Resolution Module</name>
  <objective>Create `src/alphaswarm_sol/vulndocs/resolution.py` with `vulndocs_read_path()`, `vulndocs_write_path()`, `_assert_single_vulndocs_root()`, and the `VulndocsPathConflict`/`VulndocsConfigError` exception classes.</objective>
  <what>
    Implement per CONTEXT.md D-1 and D-1a specifications:

    **`vulndocs_read_path() -> Traversable`:**
    - Check `ALPHASWARM_VULNDOCS_DIR` env var first (with validation per D-1a)
    - Otherwise: `importlib.resources.files("alphaswarm_sol").joinpath("vulndocs")` (or `_vulndocs_data` if Task 2 finding = collision)
    - Return type is `Traversable`, NOT `Path`

    **`vulndocs_write_path() -> Path`:**
    - Check `ALPHASWARM_VULNDOCS_DIR` env var first (with validation per D-1a)
    - Otherwise: `Path(__file__).resolve().parent.parent / "vulndocs"` (for editable installs)

    **`_assert_single_vulndocs_root()`:**
    - Called at module import time
    - Detects dual-root: both `importlib.resources.files()` AND `__file__`-relative resolve to different paths with `.yaml` files
    - Raises `VulndocsPathConflict` (suppressible via `ALPHASWARM_SKIP_VULNDOCS_CONFLICT_CHECK=1`)

    **`ALPHASWARM_VULNDOCS_DIR` contract (D-1a):**
    - Overrides BOTH read and write paths
    - Validation: not-exists raises `VulndocsConfigError`, not-a-directory raises `VulndocsConfigError`, empty-of-YAMLs logs WARNING

    Update `pyproject.toml` to include vulndocs in `[tool.hatch.build.targets.wheel.force-include]` using the path determined by Task 2.
  </what>
  <done_criteria>
    - `vulndocs_read_path()` returns a `Traversable` that navigates vulndocs categories
    - `vulndocs_write_path()` returns a writable `Path` for editable installs
    - `_assert_single_vulndocs_root()` raises `VulndocsPathConflict` when dual roots exist
    - `ALPHASWARM_VULNDOCS_DIR` overrides both functions and validates per D-1a
    - `pyproject.toml` includes vulndocs data in wheel build
  </done_criteria>
  <verify>
    Unit tests pass for all four code paths:
    1. `vulndocs_read_path()` from non-project cwd returns valid Traversable
    2. `vulndocs_write_path()` returns writable Path under editable install
    3. `_assert_single_vulndocs_root()` raises VulndocsPathConflict in simulated dual-root
    4. `ALPHASWARM_VULNDOCS_DIR=/tmp/custom` overrides both functions
  </verify>
</task>

## must_haves

Derived from phase goal using goal-backward methodology:

1. **vulndocs resolves from any cwd**: `vulndocs_read_path()` returns valid Traversable from /tmp, from Jujutsu workspace, from project root. This is the core fix for the Plan 12 Batch 1 failure.
2. **Zero bare `Path("vulndocs")` in production code**: All 15+ callsites from RESEARCH.md migrated. Verified by grep returning zero matches.
3. **Traversable API contract enforced**: No consumer casts `vulndocs_read_path()` result to `Path` directly. Verified by mock test.
4. **Dual-root conflict detected**: When both editable and wheel vulndocs exist at different paths, `VulndocsPathConflict` is raised at import time.
5. **Wheel build includes vulndocs**: After building a wheel and installing, `importlib.resources.files("alphaswarm_sol").joinpath("vulndocs").iterdir()` returns YAML files. (Exact joinpath argument may differ per Task 2 finding.)
6. **Read-path inventory exists and is complete**: Every callsite documented, zero unaudited matches remain.
7. **Existing tests still pass**: No regressions in pattern loading, query execution, or vulndocs validation.
