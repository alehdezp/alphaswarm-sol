# Expected Results: Plan 02 — Per-contract Graph Storage

## Fundamental Deliverable

This plan delivers per-contract graph isolation via hash-based subdirectories, a canonical contract identity utility, and atomic write semantics so that concurrent agent builds no longer cross-contaminate each other's graphs — the root cause of the Plan 3.1c-12 Batch 1 calibration failure. Without it, every Agent Teams evaluation session will continue producing invalid results because sequential `build-kg` calls overwrite a single flat `graph.toon` file, and the downstream query routing fix (Plan 03) cannot function because it depends on meta.json sidecars that only this plan produces. Success means two agents building graphs for different contracts simultaneously find their graphs in separate `{identity}/graph.toon` directories with valid meta.json, while pre-upgrade operators experience zero workflow breakage. Failure means either graphs still collide in a single flat file, or the identity function diverges across callers, or meta.json is absent or schema-invalid — any of which blocks Plan 03, Plan 04, and the entire 3.1c.1 phase.

## Alignment Criteria

### [must_have] Per-contract graph isolation

**Source:** must_haves.truths[0] — "Per-contract graph isolation: each contract stored in own {identity} subdirectory, sequential builds never overwrite"

**The question:** Does `GraphStore.save()` produce isolated per-contract subdirectories such that two different contracts produce two different directories, and sequential saves never overwrite?

**Aligned if:** `GraphStore.save()` writes to `{root}/{identity}/graph.toon` where `identity` is a 12-hex-char hash derived from the contract path. A test or grep of `store.py` shows the `save()` method computing a target directory from the identity parameter and creating it with `mkdir(parents=True, exist_ok=True)`. Two calls with different identity values produce files in two different subdirectories.

**Misaligned if:** `GraphStore.save()` still writes to a flat `{root}/graph.toon` or `{root}/graph.json`. Or identity is accepted but not used in the directory path. Or the second save with a different identity overwrites the first.

### [must_have] Canonical contract identity — deterministic, shared utility

**Source:** must_haves.truths[1] — "Canonical contract identity: contract_identity() is deterministic, order-independent, symlink-resolving, single shared utility"

**The question:** Does `kg/identity.py` exist with a `contract_identity()` function that is deterministic, order-independent, uses `os.path.realpath()`, strips trailing slashes, and is the sole implementation used by all callers?

**Aligned if:** `src/alphaswarm_sol/kg/identity.py` exists and exports `contract_identity(paths: list[Path]) -> str`. The function body contains `os.path.realpath()` (not `os.path.abspath()`), sorts the resolved paths, strips trailing slashes, joins with a separator, hashes with SHA-256, and truncates to 12 hex chars. A grep for `contract_identity` across the codebase shows all callers importing from `kg.identity` — no parallel implementations elsewhere.

**Misaligned if:** The function uses `os.path.abspath()` instead of `os.path.realpath()`. Or path ordering affects the output. Or the hash length differs from 12 hex chars. Or multiple `contract_identity` implementations exist in different files. Or any caller computes its own identity hash inline instead of using the shared utility.

### [must_have] Atomic writes prevent corruption

**Source:** must_haves.truths[2] — "Atomic writes: GraphStore.save() uses tmp+rename, no corrupt partial files on crash"

**The question:** Does `GraphStore.save()` write via a temporary file followed by `os.replace()`, ensuring no partial `graph.toon` or `meta.json` remains on crash?

**Aligned if:** `store.py` contains a pattern where data is written to `graph.toon.tmp` (or similar temp suffix) and then `os.replace()` (or `os.rename()`) atomically moves it to `graph.toon`. Same pattern applies to `meta.json`. No intermediate state where `graph.toon` contains partial content is possible.

**Misaligned if:** `store.py` opens `graph.toon` directly and writes to it. Or uses `shutil.copy` or `Path.write_text` to the final path without an intermediate temp file. Or writes to temp but uses `shutil.move` instead of `os.replace()` (which is not atomic on cross-filesystem moves).

### [must_have] meta.json sidecar follows D-meta schema

**Source:** must_haves.truths[3] — "meta.json sidecar per D-meta schema: every graph directory has valid meta.json with all mandatory fields"

**The question:** Does every graph subdirectory contain a `meta.json` file with all fields defined in decision D-meta, validated by a Pydantic model?

**Aligned if:** A `GraphMetadata` Pydantic model exists (in `store.py` or `identity.py`) with fields: `schema_version` (int, value 1), `built_at` (ISO-8601 string), `graph_hash` (hex string), `contract_paths` (list of strings), `stem` (string), `source_contract` (string), `identity` (12 hex chars), `slither_version` (string), `project_root_type` (string). The model is used in `GraphStore.save()` to validate and serialize meta.json. Extra fields are rejected (`model_config` has `extra = "forbid"` or equivalent).

**Misaligned if:** meta.json is written as a plain dict without Pydantic validation. Or any D-meta field is missing from the model. Or `schema_version` is absent or not an integer. Or `source_contract` is missing (Plan 03 depends on this for stem-based lookup). Or extra fields are silently accepted.

### [must_have] Backward-compatible flat graph fallback

**Source:** must_haves.truths[4] — "Backward compatibility: pre-upgrade flat graph.toon files still load with deprecation warning"

**The question:** Does `GraphStore.load()` fall back to a flat `graph.toon` at the root when no hash-based subdirectory exists, emitting a deprecation warning?

**Aligned if:** `GraphStore.load()` has a code path where, if the identity-based subdirectory does not exist, it checks for `self.root / "graph.toon"` (flat file). If found, it loads from the flat file and emits a deprecation warning (via `warnings.warn` or `logging.warning`). If neither exists, it raises an appropriate error.

**Misaligned if:** `GraphStore.load()` only looks in hash-based subdirectories and raises an error when the flat file exists but the subdirectory does not. Or the flat file is loaded silently without any deprecation warning.

### [must_have] Skip-if-exists and --force semantics

**Source:** must_haves.truths[5] — "Skip-if-exists + --force: default idempotent, explicit --force to rebuild"

**The question:** Does `GraphStore.save()` skip writing when the target already exists (default), and overwrite only when `force=True` is passed?

**Aligned if:** `GraphStore.save()` accepts a `force: bool` parameter (default `False`). When `force=False` and the target file exists, it returns the existing path without modifying the file (log message emitted). When `force=True`, it overwrites. The `--force` CLI flag in `build-kg` is propagated to `GraphStore.save(force=True)`.

**Misaligned if:** `GraphStore.save()` always overwrites regardless of `force`. Or the `force` parameter exists but is never connected to the CLI `--force` flag. Or skip-if-exists checks the directory but not the actual graph file.

### [must_have] --check-fresh staleness detection with exit code 42

**Source:** must_haves.truths[6] — "--check-fresh staleness detection: exit 42 on stale for programmatic detection"

**The question:** Does `build-kg --check-fresh` compare the current contract file hash to the stored `meta.json.graph_hash` and exit with code 42 on stale?

**Aligned if:** The `build-kg` CLI command accepts `--check-fresh`. When invoked, it computes the current hash of the contract files and compares to `graph_hash` in the stored meta.json. If they match: exit 0, stderr message like `[graph-cache] skipped: <stem> (hash match)`. If they differ: exit 42, stderr warning about staleness. A grep for `exit(42)` or `sys.exit(42)` or `raise SystemExit(42)` in the CLI code confirms the exit code.

**Misaligned if:** `--check-fresh` always exits 0. Or exits with code 1 on stale instead of 42. Or the flag is accepted but never implemented. Or the hash comparison uses a different algorithm than what `GraphStore.save()` stores in `graph_hash`.

## Drift Detection

### OK: GraphMetadata model uses field aliases or slightly different field names

**Scenario:** The executor implements `GraphMetadata` with Pydantic `Field(alias=...)` for JSON serialization, so the Python field name is `built_at_ts` but serializes as `built_at` in the JSON. Or uses `graph_content_hash` internally but writes `graph_hash` to disk.

**Why OK:** The on-disk contract matters, not the Python attribute names. As long as `meta.json` on disk contains all D-meta fields with correct names and types, Plan 03 can consume it.

**Action:** Verify the on-disk meta.json file matches D-meta schema. Python attribute names are an implementation detail.

### OK: identity.py includes additional utility functions beyond the three specified

**Scenario:** The executor adds `validate_identity(s: str) -> bool` or `identity_dir(root: Path, identity: str) -> Path` helper functions to `identity.py` beyond `contract_identity()`, `_resolve_project_root()`, and `filter_dependency_paths()`.

**Why OK:** Additional utilities that support the core functions do not violate any must_have. The three specified functions are the minimum contract.

**Action:** Verify the three required functions exist and work correctly. Additional helpers are fine.

### DRIFT: CorruptGraphError not implemented or not raised

**Scenario:** The executor implements `GraphStore.load()` but does not check for the `.tmp` file without corresponding `.toon` file, or raises a generic `FileNotFoundError` instead of a specific `CorruptGraphError`.

**Why DRIFT:** Corrupt graph detection is explicitly specified in Task 2 done criteria. Without it, agents may silently load partial graphs from interrupted builds, producing wrong results that look correct.

**Action:** Request implementation of the `.tmp`-without-`.toon` detection check and a specific error type. The error need not be named exactly `CorruptGraphError` but must be distinct from "file not found."

### DRIFT: --check-fresh implemented but uses different hash algorithm than graph_hash

**Scenario:** The executor implements `--check-fresh` using MD5 or file mtime comparison, while `GraphStore.save()` stores a SHA-256 hash in `meta.json.graph_hash`. The two computations never agree.

**Why DRIFT:** Staleness detection is meaningless if the freshness check uses a different algorithm than what was stored. Every graph would appear stale, and the feature would be useless noise.

**Action:** Request alignment: `--check-fresh` must compute the hash using the same algorithm that produced `meta.json.graph_hash`. Verify by reading both code paths.

### STOP: GraphStore still writes to a single flat file

**Scenario:** The executor modifies `GraphStore` to accept an `identity` parameter but internally still writes all graphs to `{root}/graph.toon`, ignoring the identity for directory creation. Tests pass because they only test single-contract scenarios.

**Why STOP:** This is the exact bug that caused the Plan 3.1c-12 Batch 1 failure. If sequential builds still land in the same file, the entire plan has failed to deliver its fundamental purpose. Agent A's graph will still disappear when Agent B builds.

**Action:** Escalate. Verify by inspecting `GraphStore.save()` to confirm the identity parameter is used in the directory path computation, not just accepted and discarded. Check that `self.root / identity / "graph.toon"` or equivalent appears in the write path.

## Reviewer Guidance

1. **Verify the directory structure, not just the API.** The single most important check is that `GraphStore.save()` actually creates `{root}/{identity}/graph.toon` — not just that it accepts an `identity` parameter. The Plan 3.1c-12 failure was caused by a flat file; the fix must be structural.

2. **meta.json is a cross-plan contract — check `source_contract` field.** Plan 03 depends on this field for stem-based `--graph` lookup. If `source_contract` is missing, Optional, or contains a path instead of a stem, Plan 03 will fail at runtime. Treat this as a hard interface verification, not a nice-to-have.

3. **Confirm `contract_identity()` is truly a single shared utility.** Grep for any inline hash computation in CLI files or `store.py`. The plan promises a single canonical function; if any caller computes identity independently, that is a divergence risk (different hashing under edge cases).

4. **Validate that `--check-fresh` exit code is exactly 42.** This is a machine-readable contract for CI and automation. Exit code 1 (generic error) would be indistinguishable from argument errors. The specific exit code 42 was chosen to be unambiguous.

5. **Check the callsite migration exhaustively, not just the listed files.** The plan lists 8 CLI/core files, but the read-path inventory may reveal additional callsites. A grep for `graph.json`, `graph.toon`, and `"graphs"` across all source files should return zero unaudited matches.
