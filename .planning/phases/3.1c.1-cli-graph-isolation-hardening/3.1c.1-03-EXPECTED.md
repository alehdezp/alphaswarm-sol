# Expected Results: Plan 03 — Query Routing Fix + --graph Flag

**Plan ID:** 3.1c.1-03
**Phase:** 3.1c.1-cli-graph-isolation-hardening
**Wave:** 2
**Depends on:** 3.1c.1-01 (vulndocs resolution), 3.1c.1-02 (per-contract graph storage + meta.json)

---

## Fundamental Deliverable

This plan delivers a query routing system that eliminates silent degradation (falling back to meaningless "nodes" results) and adds explicit graph targeting via `--graph` flag with stem-based lookup, so that agents always get either useful pattern-matched results or actionable error messages telling them exactly what went wrong and how to fix it. Without it, agents receive empty or meaningless query results with exit code 0, cannot distinguish "no matches" from "broken infrastructure," and have no way to target a specific contract's graph when multiple exist -- making the entire evaluation framework unreliable. Success means every CLI query either returns structured results with a machine-parseable `# result:` header or fails with a non-zero exit code and a recovery command. Failure means agents silently get zero results with no indication of what went wrong, reproducing the Plan 3.1c-12 Batch 1 calibration failure.

## Alignment Criteria

### AC-01: No silent degradation to "nodes" query kind [must_have]
**Source:** must_haves.truths[0]

- **Question:** Does the implementation prevent the CLI query command from ever silently falling back to the "nodes" query kind when patterns fail to load or match?
- **Aligned-if:** `intent.py` no longer has a code path where an empty `patterns` list causes `query_kind` to default to `"nodes"` for CLI-originated queries. When patterns fail to load, an error is emitted on stderr with exit code 1. When patterns load but 0 match, the structured header `# result: graph_nodes={N} matches=0` is emitted on stdout with exit code 0. The Python API still supports explicit `kind="nodes"` for backward compatibility.
- **Misaligned-if:** The "nodes" fallback still exists in the CLI path. Or the fallback was removed but a new silent degradation path was introduced (e.g., returning an empty result with exit code 0 and no header). Or pattern load failures produce exit code 0 instead of 1.

### AC-02: Error-when-ambiguous graph resolution [must_have]
**Source:** must_haves.truths[1]

- **Question:** When multiple graphs exist and `--graph` is not specified, does the CLI error with available stems instead of silently picking one?
- **Aligned-if:** `cli/main.py` implements graph discovery that scans `.vrs/graphs/` subdirectories, counts valid graphs (containing `graph.toon` without `.tmp`), and: (a) errors with exit 2 when 0 graphs found, (b) auto-selects when exactly 1 graph found, (c) errors with exit 1 and lists available stems on stderr when 2+ graphs found. No mtime-based selection logic exists.
- **Misaligned-if:** The code uses mtime, creation time, or any heuristic to auto-select among multiple graphs. Or the error message does not include the available stem names. Or the exit code is 0 when ambiguity is detected.

### AC-03: --graph flag with both direct path and stem lookup [must_have]
**Source:** must_haves.truths[2]

- **Question:** Does the `--graph` flag accept both a direct graph subdirectory path and a contract filename stem, with stem resolved against `meta.json` `source_contract` field?
- **Aligned-if:** `cli/main.py` defines a `--graph` option that: (a) when given a path like `.vrs/graphs/{hash}/`, uses it directly; (b) when given a stem like `Token`, loads `meta.json` from each graph subdirectory and matches the `source_contract` field; (c) on exactly 1 match, selects that graph; (d) on multiple matches (stem collision), exits 1 with disambiguation paths from `meta.json.contract_paths`; (e) on 0 matches, exits 1 with available graphs list and build suggestion command.
- **Misaligned-if:** Stem lookup does not use `meta.json` `source_contract` field (e.g., it uses directory names or file scanning). Or ambiguous stems silently pick the first match. Or the `--graph` flag only works on `query` but not on other graph-consuming commands (suggest, detect, schema-snapshot, etc.).

### AC-04: Agent-actionable error messages with recovery commands [must_have]
**Source:** must_haves.truths[3]

- **Question:** Does every error message include a concrete recovery command that an agent can execute to fix the problem?
- **Aligned-if:** Error messages contain embedded CLI commands: "no graphs found" includes `Run 'alphaswarm build-kg <path>' first`; "ambiguous stem" includes `use full path: {paths from meta.json}`; "non-existent stem" includes `Run 'alphaswarm build-kg contracts/{stem}.sol' to build`; pattern load failure includes `Run 'alphaswarm build-kg' from project root or set ALPHASWARM_VULNDOCS_DIR`.
- **Misaligned-if:** Error messages are generic (e.g., "graph not found" without recovery instructions). Or error messages go to stdout instead of stderr. Or errors exit 0 making them indistinguishable from success.

### AC-05: Structured stdout contract with result header [must_have]
**Source:** must_haves.truths[4]

- **Question:** Does successful query output always start with `# result: graph_nodes={N} matches={M}` on the first stdout line, followed by JSON lines, with no other `#`-prefixed lines?
- **Aligned-if:** The query output path in the code always emits the header line matching `^# result: graph_nodes=\d+ matches=\d+$` as the first stdout line on any successful query (including zero-match). Subsequent stdout lines are valid JSON objects (one per line) or empty. No other stdout line begins with `#`. All errors, warnings, and informational messages go to stderr exclusively.
- **Misaligned-if:** The header line is missing on some success paths (e.g., zero-match case). Or diagnostic/warning text is mixed into stdout. Or the header format deviates from the specified regex (e.g., spaces in wrong places, different field names). Or `Available:` lists appear on stdout instead of stderr.

### AC-06: Exit code semantics 0/1/2 consistently applied [must_have]
**Source:** must_haves.truths[5]

- **Question:** Are exit codes 0, 1, and 2 applied with consistent semantics across all graph-consuming CLI commands?
- **Aligned-if:** Exit 0 means query executed successfully (matches may be 0). Exit 1 means routing/argument error (ambiguous stem, non-existent stem, missing `--graph` when ambiguous). Exit 2 means graph load failure (file corrupt, missing, `.tmp` without `.toon`). These semantics are consistent across `query`, `suggest`, `schema-snapshot`, `detect`, and other graph-consuming commands.
- **Misaligned-if:** Different commands use different exit codes for the same error class. Or exit code 0 is returned on errors. Or new exit codes are introduced without following the 0/1/2 schema. Or some graph-consuming commands still use the old error handling.

### AC-07: Graph listing utility is shared and reusable [must_have]
**Source:** must_haves.artifacts[2] — `kg/store.py` with `list_available_graphs()`

- **Question:** Does `kg/store.py` (or a new discovery module) provide a shared `list_available_graphs()` utility that returns structured graph information usable by both error-when-ambiguous resolution and `--graph` flag lookup?
- **Aligned-if:** A function like `list_available_graphs(root: Path) -> list[GraphInfo]` exists, where each `GraphInfo` contains at minimum `identity`, `stem`, `source_contract`, `dir_path`, and `meta` (or equivalent fields). This function is imported and used by both the graph resolution logic and the `--graph` stem lookup logic in `cli/main.py` -- not duplicated.
- **Misaligned-if:** Graph discovery logic is inlined directly in `cli/main.py` without a reusable utility. Or two separate scan implementations exist for error-when-ambiguous vs `--graph` lookup. Or the function returns raw paths without structured metadata.

### AC-08: Test file exists with comprehensive coverage and no skip/xfail [must_have]
**Source:** must_haves.artifacts[3] — `tests/test_query_routing.py`

- **Question:** Does `tests/test_query_routing.py` exist with tests covering query routing, `--graph` flag, stdout contract, and exit codes, with zero `@pytest.mark.skip` or `@pytest.mark.xfail`?
- **Aligned-if:** The test file exists and contains tests for: (a) no-graphs exit 2, single-graph auto-select, multiple-graphs-no-flag exit 1; (b) `--graph` with direct path, unique stem, ambiguous stem, non-existent stem, schema version mismatch; (c) result header regex match, JSON line validity, no extra `#` lines; (d) exit code 0/1/2 for respective conditions. No `skip` or `xfail` markers. Tests validate outcomes (exit codes, stdout content, stderr content), not implementation internals.
- **Misaligned-if:** The test file is missing or a stub. Or tests use `@pytest.mark.skip` / `@pytest.mark.xfail`. Or tests mock everything and never exercise the actual query routing code path. Or tests only cover happy paths without error conditions.

### AC-09: meta.json schema_version validated at load time [context]
**Source:** CONTEXT.md D-3 — "Query router must validate meta.json schema_version at load time; fail fast with [error] meta.json schema version mismatch"

- **Question:** Does the `--graph` stem lookup validate `schema_version` from `meta.json` and fail fast if it does not match the expected version?
- **Aligned-if:** When loading `meta.json` during stem lookup, the code checks `schema_version` against an expected constant. If mismatched, it exits immediately with an error message containing "schema version mismatch" on stderr, without proceeding to match the stem.
- **Misaligned-if:** `schema_version` is read but not validated. Or validation failure is logged as a warning but execution continues. Or there is no `schema_version` check in the stem lookup path.

### AC-10: CLI I/O channel separation — stdout is data only [context]
**Source:** CONTEXT.md D-3 — "stdout = data only; stderr = all errors/warnings/info"

- **Question:** Is the I/O channel contract enforced such that stdout contains only the result header and JSON data lines, while all other output goes to stderr?
- **Aligned-if:** All `print()` or `sys.stdout.write()` calls in the query path output only the `# result:` header and JSON data lines. Error messages, `Available:` stem lists, warnings, and diagnostics use `sys.stderr.write()` or logging to stderr. Available stems are shown without truncation; if count exceeds 20, a "... and N more" note is added.
- **Misaligned-if:** Error messages or `Available:` lists appear on stdout. Or the code uses `print()` without explicit `file=sys.stderr` for non-data output. Or stem lists are truncated without the 20-item threshold.

### AC-11: Stem disambiguation preserves adversarial insight [improvement]
**Source:** IMPROVEMENT-DIGEST P2-IMP-10 (confirmed -> implemented) — "Stem disambiguation — error on collision with paths"

- **Question:** Does the stem disambiguation error message include the full contract paths from `meta.json` so agents can resolve the ambiguity?
- **Aligned-if:** When `--graph Token` matches multiple graphs, the error message includes the actual contract paths (e.g., `contracts/v1/Token.sol`, `contracts/v2/Token.sol`) read from `meta.json.contract_paths`, not just the stems or hash identifiers. The agent can copy-paste a path to resolve the ambiguity.
- **Misaligned-if:** The disambiguation message shows only stems (which are identical by definition of a collision) or only hash identifiers (which agents cannot map to contracts without additional lookups).

### AC-12: --graph applied to all graph-consuming commands, not just query [inferred]
**Source:** Plan objective and Tasks 2-3 — "Apply same resolution logic to suggest, schema_snapshot, detect, and other CLI commands"

- **Question:** Is the `--graph` flag and error-when-ambiguous resolution applied consistently to all graph-consuming CLI commands, not just `query`?
- **Aligned-if:** The `--graph` option and graph resolution logic (including error-when-ambiguous) are shared across `query`, `suggest`, `schema-snapshot`, `detect`, `label`, `beads`, and any other command that previously hardcoded a graph path. The resolution function is called from a single shared location, not duplicated per command.
- **Misaligned-if:** Only `query` has the `--graph` flag. Or other commands still hardcode `Path(".vrs/graphs/graph.json")`. Or graph resolution logic is copy-pasted into each command rather than shared.

## Drift Detection

### OK: Graph resolution utility placed in new file instead of store.py

**Scenario:** The executor creates `src/alphaswarm_sol/kg/discovery.py` instead of adding `list_available_graphs()` to `kg/store.py`, containing the same `GraphInfo` dataclass and discovery function.

**Why OK:** The plan explicitly mentions "Add `list_available_graphs()` to `kg/store.py` (or a new `kg/discovery.py`)" — both locations are acceptable. The key requirement is a shared, reusable utility, not its file location.

**Action:** Verify the function is imported and used by `cli/main.py` and is not duplicated.

### OK: Exit code 2 scope slightly broader than specified

**Scenario:** The executor uses exit code 2 for both graph load failures AND `meta.json` parse errors (e.g., malformed JSON), even though the spec only mentions file corruption and missing files.

**Why OK:** `meta.json` parse errors are conceptually graph infrastructure failures, consistent with exit code 2 semantics ("graph load failure"). The key invariant is that exit code 2 means "graph infrastructure problem" vs exit code 1 meaning "routing/argument problem."

**Action:** Verify the exit code semantics are internally consistent and documented in test assertions.

### DRIFT: Error messages lack embedded recovery commands

**Scenario:** The executor implements error-when-ambiguous correctly (exit 1 on multiple graphs) but the error message says "Error: multiple graphs found. Use --graph to specify." without listing available stems or including a build command.

**Why DRIFT:** CONTEXT.md D-3 and must_haves.truths[3] explicitly require agent-actionable errors with embedded recovery commands. A generic "use --graph" without listing available stems forces the agent to discover stems through another mechanism, defeating the purpose of actionable errors.

**Action:** Request the executor add available stems to the error message and include the `Available: {stems}` list on stderr.

### DRIFT: Silent fallback replaced with a different silent fallback

**Scenario:** The executor removes the "nodes" fallback but introduces a new fallback where a query with 0 matching patterns silently returns an empty JSON array on stdout with no `# result:` header and exit code 0.

**Why DRIFT:** This preserves the fundamental problem — agents get exit code 0 and no indication of what happened. The `# result:` header with `matches=0` is essential for agents to distinguish "zero matches on a valid graph" from "something went wrong silently."

**Action:** Request the `# result:` header be emitted for all successful queries, including zero-match cases.

### STOP: --graph flag works but error-when-ambiguous is missing

**Scenario:** The executor implements `--graph` with stem lookup correctly, but when multiple graphs exist and `--graph` is not specified, the code silently auto-selects the first graph found (e.g., alphabetically or by mtime).

**Why STOP:** This contradicts the fundamental purpose of the plan. The Plan 3.1c-12 Batch 1 failure was caused by exactly this behavior — agents querying the wrong graph without knowing it. Auto-selection among multiple graphs is the root cause this plan exists to eliminate. The `--graph` flag without error-when-ambiguous is cosmetic; agents will never use it if auto-selection "works."

**Action:** Escalate. The error-when-ambiguous behavior is the plan's core safety invariant. Without it, Plan 04 smoke tests will pass with false positives (agents get results, but from the wrong graph).

## Reviewer Guidance

1. **Verify the "nodes" fallback is truly eliminated from CLI paths.** This is the root cause of the Plan 3.1c-12 Batch 1 failure. Search `intent.py` for any code path where `query_kind` can become `"nodes"` when the query originates from the CLI `query` command. The Python API backward-compatibility path (explicit `kind="nodes"`) must still work — but the default/fallback path must not.

2. **Verify error-when-ambiguous is the DEFAULT, not an option.** The plan's safety value comes from refusing to guess. Check that when 2+ valid graphs exist and `--graph` is absent, execution halts with an error listing stems. There must be no configuration flag, environment variable, or code path that re-enables auto-selection.

3. **Verify the stdout/stderr channel separation is complete.** The `# result:` header and JSON data lines must be the ONLY content on stdout. Run a mental trace through the query code path and confirm no logging, warnings, `Available:` lists, or diagnostic messages reach stdout. This contract is consumed by Plan 04 agents and future CI pipelines.

4. **Verify `--graph` is applied to ALL graph-consuming commands, not just `query`.** The plan explicitly names `suggest`, `schema-snapshot`, `detect`, `label`, `beads` and others. Check that each command either shares the graph resolution function or has been updated. Any command still hardcoding `Path(".vrs/graphs/graph.json")` is a missed migration.

5. **Verify test coverage includes error paths, not just happy paths.** The test file must cover: 0 graphs (exit 2), ambiguous stem (exit 1 with paths), non-existent stem (exit 1 with available list), schema version mismatch (fail fast), pattern load failure (exit 1 with recovery command). Happy-path-only tests would miss the entire point of this plan.
