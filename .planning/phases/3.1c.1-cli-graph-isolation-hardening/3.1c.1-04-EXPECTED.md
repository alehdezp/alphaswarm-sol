# Expected Results: Plan 04 — CLI Isolation Smoke Tests with Agent Teams

## Fundamental Deliverable

This plan delivers empirical proof that the CLI fixes from Plans 01-03 actually work when real agents operate in Jujutsu-isolated workspaces, so that downstream phases (3.1c.2, 3.1c.3, 3.1f, 3.2) can trust the CLI as the agent-facing interface and the testing framework as the evaluation vehicle. Without it, the phase has fixed code but has no evidence that agents will use those fixes instead of falling back to raw Python imports — the exact failure mode that invalidated Plan 3.1c-12 Batch 1. Success means agents invoke CLI commands, receive non-empty correct results, produce no cross-contamination between concurrent builds, and all of this is recorded as persistent artifacts. Failure means the CLI fixes exist in source but agents still cannot or do not use them, cross-contamination persists, or the testing framework itself is broken — leaving every downstream phase building on unvalidated assumptions.

### AC-2: CLI returns non-empty, meaningful results [must_have]
**Source:** must_haves.truths[1] — "Agents GET correct results"

- **Misaligned-if:** Result headers show `graph_nodes=0` (graph not loaded) or all queries show `matches=0` for patterns that should match (query routing still broken). Alternatively, no `# result:` header appears in any agent output (header contract not implemented by Plan 03).

## Drift Detection

### OK: Agent uses slightly different query patterns than specified
**Scenario:** The plan specifies 3 CLI queries per contract. An agent runs 4 queries — the 3 specified plus an exploratory one — all via CLI, all returning non-empty results.
**Why OK:** The intent is to prove agents use the CLI and get correct results. Extra CLI queries strengthen the evidence. The 4 binary checks all pass.
**Action:** Verify all 4 CLI Usage Confirmation checks pass. Note the extra query but accept.

### OK: Different test contracts than expected
**Scenario:** The executor selects `SimpleVault.sol` and `UncheckedCalls.sol` instead of a reentrancy/access-control pair. Both contracts have genuinely discriminating patterns verified through cross-query.
**Why OK:** The plan leaves contract selection to Claude's discretion. What matters is that the two contracts have distinct, verifiable patterns — not which specific contracts are chosen.
**Action:** Verify `ground-truth-patterns.json` shows `cross_verified: true` and patterns are semantically sensible for the chosen contracts.

### DRIFT: Ground-truth patterns not pre-verified sequentially
**Scenario:** The executor jumps directly to concurrent agent builds without first establishing ground-truth patterns via sequential isolated builds. Cross-contamination results show `matches=0` but there is no `ground-truth-patterns.json` proving the patterns would have returned matches > 0 on the correct contract.
**Why DRIFT:** Without sequential pre-verification, `matches=0` is unfalsifiable — the pattern might return 0 on ALL contracts (including its own), making the cross-contamination "test" vacuous. The entire falsifiability design depends on ground-truth establishment.
**Action:** Request that Task 2 (sequential ground-truth verification) be completed before accepting Task 4 results. The concurrent test must be re-run after ground truth is established.

### DRIFT: Pre-validation tests exist but are not gating
**Scenario:** `test_cli_isolation_smoke.py` exists with pre-validation tests, but agents were spawned regardless of pre-validation outcomes. The test file runs independently but is not consulted before Task 3 execution.
**Why DRIFT:** Pre-validation exists to disambiguate "CLI broken" from "infrastructure broken." If agents are spawned without pre-validation passing, a CLI failure might actually be a WorkspaceManager failure, corrupting the diagnostic signal.
**Action:** Request evidence that pre-validation passed before agents were spawned. If pre-validation was not run first, request re-execution with the correct sequencing.

### STOP: Agents fall back to Python imports and results are accepted anyway
**Scenario:** Agent transcripts show `from alphaswarm_sol.kg import KnowledgeGraph` or direct Python graph construction. The executor notes this but proceeds to record the results as valid smoke test outcomes, marking the overall verdict as "PASS."
**Why STOP:** This is the exact Plan 3.1c-12 Batch 1 failure mode that triggered the entire Phase 3.1c.1. Accepting Python-import-based results means Plans 01-03 fixes are untested. The plan's fundamental purpose — proving agents USE the CLI — is violated. No downstream phase can trust these results.
**Action:** Escalate immediately. Results produced via Python imports are INVALID for this plan's purpose. All 4 CLI Usage Confirmation checks must be re-evaluated. AgentExecutionValidator must be run. The overall verdict must reflect the CLI usage violation.

## Reviewer Guidance

1. **Verify CLI usage before anything else.** The single most important check is AC-1: do agents actually invoke `alphaswarm build-kg` and `alphaswarm query` via bash, with zero Python imports? If this fails, nothing else matters — the plan's fundamental purpose is violated. Check agent transcripts or observation files for the specific command strings and anti-patterns.

2. **Confirm ground-truth falsifiability chain.** The cross-contamination test (AC-3) is only valid if ground-truth patterns were established first (AC-8). Verify the sequence: sequential builds -> pattern identification -> cross-verification -> concurrent test. If `ground-truth-patterns.json` is missing or `cross_verified` is false, the cross-contamination result is meaningless regardless of what it says.

3. **Distinguish real concurrent execution from sequential-labeled-concurrent.** Both AC-4 (concurrent same-contract) and AC-3 (cross-contamination) depend on actual concurrency. Look for evidence that agents or build commands ran simultaneously, not sequentially with a "concurrent" label applied retroactively.

4. **Check that pre-validation gated agent spawning.** AC-5 is about more than test existence — it is about execution sequencing. Pre-validation must have PASSED before agents were spawned. Look for evidence of this ordering in the test execution flow or smoke-test-summary.

5. **Watch for vacuous success on the happy path.** If all agents pass perfectly on first attempt, verify that Task 6 (interrogation protocol) correctly produced no artifacts rather than being skipped entirely. A clean pass is valid only if the interrogation protocol was prepared but not triggered — not if it was omitted from the implementation.
