# Plan 05a: Testing Governance Design

**Goal:** Define a phase-type taxonomy with testing requirements per type, a testing gate template for CONTEXT.md files, the mixed-type protocol, the mechanical snapshot protocol, and the 3.1c.2 bootstrap exception -- so that Plan 05b can roll these out to real phases with confidence.

**Trigger:** The cross-phase testing mandate (from CONTEXT.md) requires every phase to use the testing framework. But without a taxonomy, the same requirements apply to fundamentally different phase types (code changes vs evaluation runs vs documentation). This produces either meaningless compliance or impossible requirements.

**Context decisions:** Cross-Phase Testing Mandate (refined) in CONTEXT.md. Phase type taxonomy: CODE, EVALUATION, FRAMEWORK, SYNTHESIS. Mixed-type protocol. Enforcement via done criteria. 3.1c.2 bootstrap exception. Snapshot Protocol. All DECIDED and locked.

**Parallel execution:** This plan has NO dependencies on Plans 01-04. It can execute in parallel with Wave 3 (Plan 04). Its output is consumed by Plan 05b.

  <name>Task 1: Define Phase-Type Taxonomy</name>
  <objective>Create the canonical phase-type taxonomy document defining 4 phase types with their testing requirements.</objective>
  <what>
    Create `.planning/testing/rules/canonical/PHASE-TYPE-TAXONOMY.md` with:

    **Four phase types (per CONTEXT.md):**

    | Type | Definition | Testing Requirements |
    |------|-----------|---------------------|
    | CODE | Implements Python/Solidity changes | Agent Teams + AgentExecutionValidator all 12 checks + at least 1 red-path session |
    | EVALUATION | Runs agent sessions, measures quality | Dual-Opus evaluator + fabrication checks + baseline comparison |
    | FRAMEWORK | Modifies testing infrastructure | Snapshot-before/after diff + regression on prior baselines |
    | SYNTHESIS | VulnDocs, documentation, patterns | Human review checklist + pattern validation CLI pass, no Agent Teams required |

    **Mixed-type protocol:**
    - A phase may declare multiple types separated by `+` (e.g., `type: CODE+FRAMEWORK`)
    - Acceptance bar = UNION of requirements for all declared types
    - Primary type (listed first) determines which team owns the acceptance decision
    - Done criterion: the mixed-type rule has been applied to at least one existing phase (3.1c.1 as `CODE+FRAMEWORK`) and the resulting requirement list verified as non-contradictory

    **3.1c.2 bootstrap exception:**
    - Phases modifying testing infrastructure use a pre-modification snapshot for acceptance validation
    - The modified version cannot validate itself
    - Exception text: "use snapshot recorded per Snapshot Protocol; if `.vrs/snapshots/{phase}/pre-change-id.txt` is absent, the exception is void and standard FRAMEWORK requirements apply"

    **intelligence_contribution metadata field:**
    - Optional field in testing gate template
    - Documents which Tier 2 intelligence modules each phase's data feeds
    - Metadata annotation only, not an enforcement gate
    - Enables Phase 3.1c.3 to trace data lineage
  </what>
  <done_criteria>

    - All 4 types defined with specific, measurable testing requirements
    - Mixed-type protocol documented with 3.1c.1 as example
    - 3.1c.2 bootstrap exception documented with Snapshot Protocol reference
    - intelligence_contribution field documented
    - Non-contradictory: verify CODE+FRAMEWORK union for 3.1c.1 produces a coherent requirement list
  </done_criteria>
  <verify>

## must_haves

Derived from phase goal using goal-backward methodology:

1. **Phase-type taxonomy exists**: 4 types (CODE, EVALUATION, FRAMEWORK, SYNTHESIS) with measurable testing requirements per type. This is the foundation for the testing mandate.
2. **Mixed-type protocol works**: CODE+FRAMEWORK union for 3.1c.1 produces a non-contradictory requirement list. Plan 05b cannot apply the mandate without this.
3. **Testing gate template is copy-paste ready**: Phase planners can fill it in without interpretation. Plan 05b uses this template for each near-term phase.
4. **Snapshot Protocol is mechanical**: Exact commands, exact file paths, exact verification steps. No interpretive ambiguity.
5. **3.1c.2 bootstrap exception is actionable**: Clear conditions under which the exception applies and how to use it.
6. **TESTING-FRAMEWORK.md references the taxonomy**: Discoverable from the canonical testing framework document.

## Governance Compliance

- **GOV-01 (Derived checks):** Testing requirements per phase type are derived from phase characteristics (what kind of work each phase does), not arbitrary thresholds. CODE phases need Agent Teams because they change CLI behavior. SYNTHESIS phases don't because they produce documentation.
- **GOV-02 (Preconditions):** No unresolved preconditions. Taxonomy design derives from existing phase list and CONTEXT.md decisions. No external dependencies.
- **GOV-03 (HITL runbook):** Generated at `.vrs/debug/phase-3.1c.1/plan-phase/hitl-runbooks/governance-design.md` -- includes verification commands for taxonomy completeness, template fill-in exercise, and mixed-type union check.
- **GOV-05 (No fake tests):** This plan produces design artifacts, not test files. No skip/xfail concern.
- **GOV-07 (Phase checklists):** Plan contributes the testing gate template that all future phase entry checklists reference.
