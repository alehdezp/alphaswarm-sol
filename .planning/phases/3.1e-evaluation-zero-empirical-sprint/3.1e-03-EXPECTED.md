# 3.1e-03 Expected Results: Evaluator Assessment — Taxonomy + Validity + Adversarial Matrix

---

## Fundamental Deliverable

This plan delivers a data-driven evaluator assessment — failure taxonomy from real FP data, adversarial gaming archetypes from evaluator code-path analysis, and a validity/gameability matrix with LLM defense measurement — so that downstream phases (3.1c, 3.1f) know exactly which heuristic dimensions are trustworthy, which are gameable, and what failure modes the detection pipeline exhibits. Without it, 3.1c would design evaluation infrastructure on theoretical assumptions about heuristic validity and detector failure modes, repeating the v5.0 anti-pattern of building before observing. Success means the taxonomy emerges from clustering real FP data (not hand-designed categories), adversarial archetypes trace to specific evaluator code paths, and the validity matrix produces a falsifiable answer to "does LLM-as-judge defend against evaluator gaming?" Failure means categories are generic labels ungrounded in the 65-FP corpus, archetypes are invented without code-path traceability, or the validity/gameability assessment is produced without actually scoring adversarial transcripts against both heuristic and LLM rubrics.

## Alignment Criteria

### [must_have] Taxonomy discovered from data, not imposed

**Source:** must_haves.truths[0] — "Failure taxonomy discovered from data, not hand-designed categories"

**The question:** Does `descriptive_clusters.json` contain categories that emerged from undirected clustering of the 65-FP baseline data, rather than hand-designed categories?

**Aligned if:** The LLM prompt used for clustering (recoverable from transcripts or documented in the artifact) does NOT mention the hand-designed categories (`test_bug`, `llm_variance`, `transient`, `genuine_gap`). Cluster names are descriptive of root causes observed in the FP data (e.g., "modifier-recognition gap", "standard-function overfiring") rather than generic labels. The artifact contains a `source_baseline` field linking to the specific baseline data used.

**Misaligned if:** Cluster names match or closely paraphrase the hand-designed categories from CONTEXT.md pre-research. The LLM prompt contains leading category suggestions. Categories are generic enough to apply to any detection system (e.g., "false positive", "configuration error") without FP-data-specific grounding.

### [must_have] Validity coefficient measured per dimension

**Source:** must_haves.truths[1] — "Heuristic validity coefficient measured per dimension"

**The question:** Does `validity-matrix.json` contain a quantitative validity coefficient for each heuristic dimension, computed using the N-appropriate method (normalized inversion count for N < 10)?

**Aligned if:** `validity-matrix.json` contains per-dimension entries with a numeric validity coefficient, `validity_method: normalized_inversion_count`, the sample N, and pre-registered decision thresholds (< 0.2 high, 0.2-0.4 near zero, > 0.4 counter-productive). Each dimension has a `construct_alignment_note` explaining what the heuristic formula measures vs what the LLM rubric asks.

**Misaligned if:** Validity is stated qualitatively ("seems valid") without numeric coefficient. Method is Spearman rho applied to N < 10 samples (violating the N-appropriate decision from CONTEXT.md). Dimensions are assessed in aggregate rather than per-dimension. Decision thresholds are absent or differ from pre-registered values.

### [must_have] Adversarial archetypes traced to code paths

**Source:** must_haves.truths[2] — "Adversarial archetypes enumerated from evaluator code paths"

**The question:** Does `archetype-count.json` contain archetypes that each reference a specific evaluator code location, and are there at least 3 archetypes plus the mandatory LLM-Flatterer?

**Aligned if:** `archetype-count.json` contains N >= 3 entries, each with `mechanism_cluster` and `code_location` fields pointing to specific file:line references in `reasoning_evaluator.py`, `graph_value_scorer.py`, or `obs_bskg_query.py`. An `enumeration_sha256` field records the SHA256 of each source file at enumeration time. The LLM-Flatterer archetype is present and hand-crafted (not mechanism-derived). Fields include `coverage_complete`, `skipped_clusters`, and `enumerated_code_paths`.

**Misaligned if:** Archetypes are generic gaming strategies (e.g., "prompt injection") without tracing to specific evaluator code paths. Code locations are absent or point to files/lines that do not exist. LLM-Flatterer is missing. Archetype count is less than 3.

### [must_have] Gameability matrix with LLM defense measurement

**Source:** must_haves.truths[3] — "Gameability matrix produced with LLM identification of gaming mechanisms"

**The question:** Does the gameability matrix in `validity-matrix.json` show both heuristic and LLM scores for each archetype, with LLM mechanism identification results and a go/no-go verdict using the pre-registered ceil(N/2) threshold?

**Aligned if:** The gameability matrix has rows = archetypes and columns including `heuristic_score`, `LLM_score`, `gamed_heuristic?`, `gamed_LLM?`, `LLM_identifies_mechanism`. A `go_threshold_used` field equals `ceil(N/2)`. A falsifiable conclusion for Livshits is present ("LLM-as-judge [does|does not] provide meaningful defense..."). The three-state verdict (Go/Conditional/No-go) is documented with evidence.

**Misaligned if:** Matrix has heuristic scores but no LLM scores (Phase 2 was skipped without Plan 02 rubric freeze gate documentation). Go threshold is absent or differs from ceil(N/2). Livshits conclusion is missing or unfalsifiable. LLM_identifies_mechanism column is absent — the matrix only records scores without checking whether the LLM explains WHY a transcript is gaming.

## Drift Detection

### OK: Fewer than 6 adversarial archetypes generated

**Scenario:** The executor generates only 3-4 adversarial transcripts (plus LLM-Flatterer) instead of the maximum 6, because evaluator code-path enumeration found fewer distinct exploit mechanisms than the plan's expected list.

**Why OK:** The plan specifies "Minimum: 3 clusters. Maximum: 6 + LLM-Flatterer." Fewer archetypes discovered from code is a legitimate empirical finding, not a shortcut. The scope-reduction gate explicitly handles archetype_count <= 2.

**Action:** Verify `archetype-count.json` has `skipped_clusters` documenting which expected mechanisms were not found and why. Confirm N >= 3.

### OK: Phase 0 dominance ratio exceeds 75%

**Scenario:** One taxonomy cluster contains 80% of all FPs (e.g., "has_access_gate recognition failure" dominates). The executor annotates `corpus-limited` and proceeds.

**Why OK:** The plan explicitly defines this as a degradation annotation, not a blocking condition. Phase 1 and Phase 2 proceed unchanged. The finding "corpus is dominated by one failure mode" is itself valuable data for 3.1c.

**Action:** Verify the `corpus-limited` annotation is present in `descriptive_clusters.json`. Verify Phase 1 was not artificially constrained by the taxonomy result.

### DRIFT: Taxonomy categories match hand-designed categories

**Scenario:** The executor's `descriptive_clusters.json` contains categories named "test_bug", "llm_variance", "transient", and "genuine_gap" — matching the pre-existing hand-designed categories that CONTEXT.md explicitly says were created without observing real failures.

**Why DRIFT:** The plan's core invariant is that taxonomy is "discovered from data, not hand-designed categories." If the LLM prompt included or was biased toward these pre-existing categories, the discovery was contaminated. Even if the same categories emerge naturally, the prompt must not have mentioned them.

**Action:** Request the executor provide the LLM prompt used for clustering. If the prompt mentions the hand-designed categories, request re-run with undirected prompt. If categories emerged independently, request documentation of the prompt alongside the result.

### DRIFT: Validity matrix produced without LLM scoring

**Scenario:** The validity-matrix.json contains heuristic validity coefficients but the gameability matrix has empty LLM columns, with no documented Plan 02 rubric freeze gate check or degradation annotation.

**Why DRIFT:** Phase 2 explicitly requires Plan 02 rubric freeze as a gate. Proceeding without LLM scoring is allowed ONLY with `rubric-uncalibrated` annotation. Silent omission of LLM scoring breaks the core purpose of measuring whether LLM-as-judge defends against gaming.

**Action:** Request documentation of Plan 02 gate check. If Plan 02 rubric was not available, request `rubric-uncalibrated` annotation be added to all validity-matrix entries.

### STOP: Adversarial transcripts generated without evaluator code-path analysis

**Scenario:** The executor generates adversarial transcripts by imagining general gaming strategies (e.g., "repeat keywords", "add fake tool calls") without first reading `reasoning_evaluator.py`, `graph_value_scorer.py`, and `obs_bskg_query.py` to enumerate actual exploit mechanisms.

**Why STOP:** The plan's fundamental purpose is to discover gaming vectors FROM the evaluator's actual code paths, not from theoretical attack surfaces. Without code-path enumeration, the archetypes are speculative and cannot inform 3.1c about real evaluator weaknesses. The `enumeration_sha256` field exists specifically to prove source files were read at a fixed point in time.

**Action:** Escalate. The entire Phase 1 output is unreliable. Request re-execution starting from Step 1 code-path enumeration with SHA256 capture.

## Reviewer Guidance

**Priorities:**

1. **Verify taxonomy is data-derived, not imposed.** Check the LLM prompt used for Phase 0 clustering. The prompt must NOT mention hand-designed categories. This is the single most important alignment criterion — a contaminated taxonomy defeats the plan's core purpose.
2. **Verify adversarial archetypes trace to real code paths.** Each archetype in `archetype-count.json` must have a `code_location` referencing an actual line in evaluator source. Cross-check at least one code location against the actual source file.
3. **Verify Phase 2 gate on Plan 02 rubric freeze.** The validity matrix must reference Plan 02's rubric SHA256 or document `rubric-uncalibrated` degradation. This dependency is the critical cross-plan contract.
4. **Verify the gameability matrix has LLM mechanism identification, not just scores.** The `LLM_identifies_mechanism` column must show whether the LLM explained WHY a transcript is gaming, not just whether it scored low. This distinguishes "LLM rejects gaming" from "LLM detects gaming."
5. **Verify three-phase sequential execution.** Phase 0 -> Phase 1 -> Phase 2 ordering is a structural requirement. All artifacts should show temporal ordering consistent with this gating.
