# Phase 3.1e: Evaluation Zero -- Empirical Sprint - Research

**Researched:** 2026-02-24 (updated 2026-02-25 post-improvement-passes P1-P7)
**Domain:** Empirical evaluation experiments (detection delta, LLM-as-judge, adversarial stress testing, failure taxonomy, schema extraction)
**Confidence:** HIGH

## Summary

Phase 3.1e is an empirical sprint designed to answer questions that schemas cannot: Does the detection improvement loop produce measurable signal? Does the LLM-as-judge template produce informative scores? Can the heuristic evaluator be gamed? What failure categories emerge from real data? The phase exists because the project's adversarial review found ~1,185 LOC of proposed schemas before any real evaluation run -- the v5.0 anti-pattern of building infrastructure for unobserved phenomena.

The technical surface is well-bounded: 5 experiments (~430 LOC total), all operating on existing infrastructure (PatternEngine, ReasoningEvaluator, GraphValueScorer, BSKG builder) with a 7-contract DVDeFi corpus as the test bed. The primary risk is not technical complexity but measurement validity -- ensuring experiments answer what they claim to answer without circular reasoning.

**Primary recommendation:** Execute experiments in dependency order (Session 0 -> Plan 01 -> Plan 02 + Plan 03/04 Phase 0+1 in parallel -> Plan 03/04 Phase 2 -> Plan 05), treating each experiment's exit artifacts as hard gates. Use `claude -p --output-format json` for LLM scoring (not Agent Teams). Preserve per-function-firing granularity for delta measurement.

### Anti-Patterns to Avoid
- **Running detection via pytest cache:** Pytest's graph cache interferes with reproducibility. Build graphs explicitly in standalone scripts. Use `limit=200` (not default 50).
- **Using `fingerprint_graph` (legacy dict-based):** Use `graph_fingerprint` (OO, returns str). Also avoid `verify_determinism()` which takes legacy dict interface.
- **Conflating per-function and per-pattern counting:** Per-function for delta measurement (Plan 01). Per-pattern for regression detection (test suite). Document both.
- **Using calibration anchors as validity samples:** Rubric was tuned on anchors -- introduces circularity. Three-state validity: `{adversarial_validity: measured, normal_validity: deferred, anchor_validity: circular_excluded}`.
- **Applying Spearman rho at N < 10:** Use rank-ordering with inversion count instead. All 3.1e experiments operate at small N.
- **Citing `CORPUS` as a dict:** CORPUS is `list[GroundTruth]`. Pre-flight type check required. Build join: `corpus_by_contract = {gt.target_contract_label: set(gt.expected_patterns) for gt in CORPUS}`.
- **Modifying decision rules after execution starts:** DECISION-LOCK.md and pre-commitment.json are immutable after lock timestamp. Plan 05 verifies this constraint.
- **Assuming `--temperature` works with `claude -p`:** `--temperature` flag is UNAVAILABLE in Claude Code CLI v2.1.50. LLM scoring non-determinism is non-controllable. Document as a known confound.
- **Cross-scope cost comparison:** If before-run and after-run use different scopes (e.g., targeted_3 vs full_7), cost ratio is unreliable. Emit warning "cross-scope comparison: ratio unreliable" and escalate.
- **Skipping GATE A rationale adequacy:** Flip resolution rationale must be >= 20 words AND contain >= 1 falsifiable signal. GATE A rejection prevents contaminated calibration.

## Common Pitfalls

### Pitfall 1: Silent Truncation from Default Limit
**What goes wrong:** `build_lens_report` defaults to `limit=50`, silently truncating findings. The 3.1d-05 baseline used `limit=200`.
**Why it happens:** Default parameter mismatch between test and production use.
**How to avoid:** Always pass `limit=200` explicitly. Pre-flight check: compare finding counts with known 3.1d-05 baseline.
**Warning signs:** Significantly fewer findings than expected. Precision appears higher than 13.3% baseline.

### Pitfall 2: CORPUS Type Confusion
**What goes wrong:** Treating CORPUS as a dict when it's a `list[GroundTruth]`.
**Why it happens:** Cached transforms or assumption from other ground truth modules.
**How to avoid:** Pre-flight type check: `print(type(CORPUS), type(CORPUS[0]))` -- must be `list, GroundTruth`.
**Warning signs:** KeyError or AttributeError when accessing CORPUS elements.

### Pitfall 3: Detection Baseline Contradictions
**What goes wrong:** 3.1d-05 detection-baseline.md has internal contradictions: Executive Summary says 11 TP / 44 total (25% precision) vs Aggregate Table says 10 TP / 75 total (13.3% precision).
**Why it happens:** Different counting methodologies (per-pattern vs per-function) or stale summary.
**How to avoid:** Session 0 re-runs the pipeline fresh. Trust Session 0 numbers, not 3.1d-05 narrative.
**Warning signs:** Numbers matching one version but not the other.

### Pitfall 4: Heuristic Evaluator Count-Based Homogeneity
**What goes wrong:** All 7 non-default-30 dimensions in `_heuristic_dimension_score` are count-based proxies. Anchor transcripts that differ only in reasoning quality but have identical tool counts score identically.
**Why it happens:** Heuristic was designed as disposable proxy using tool diversity, not content analysis.
**How to avoid:** Design anchors where heuristic inputs (unique_tool_count, bskg_query_count) are equalized between GOOD and MEDIOCRE. This makes count-based limitation a feature of test design.
**Warning signs:** GOOD and MEDIOCRE anchors scoring identically on heuristic.

### Pitfall 5: Observation Hook False Events
**What goes wrong:** `obs_bskg_query.py:25` matches `'alphaswarm' in command` -- any Bash command containing 'alphaswarm' generates a bskg_query event, even without actual graph traversal.
**Why it happens:** Broad string matching without content validation.
**How to avoid:** For Plan 03/04, this IS an exploit to generate adversarial transcripts for. For Plan 02, awareness that GVS query_coverage can be inflated.
**Warning signs:** High query_coverage scores on transcripts with no substantive graph queries.

### Pitfall 6: Circular Validity from Anchor Reuse
**What goes wrong:** Using Plan 02 calibration anchors as non-adversarial validity samples creates circularity (rubric tuned on anchors).
**Why it happens:** Natural temptation to reuse existing scored transcripts.
**How to avoid:** Explicit three-state validity tracking: `{adversarial_validity: measured, normal_validity: deferred, anchor_validity: circular_excluded}`.
**Warning signs:** Unusually high validity coefficients on anchors.

### Pitfall 7: ReentrancyWithGuard as False Positive Attributor
**What goes wrong:** ReentrancyWithGuard has 12 FPs from 12 DISTINCT patterns. Fixing access-tierb-001 removes at most 1. Expecting aggregate improvement from single-pattern fix.
**Why it happens:** Confusing per-pattern improvement (access-tierb-001) with per-contract improvement (ReentrancyWithGuard total).
**How to avoid:** Use ReentrancyWithGuard as NEGATIVE CONTROL only (access-tierb-001 must NOT fire after fix). Primary test contracts: SideEntranceLenderPool + NaiveReceiverPool.
**Warning signs:** Expecting > 1 FP reduction on ReentrancyWithGuard from Plan 01.

### Pitfall 8: Pre-Run Limit Parameter Mismatch (Plan 01)
**What goes wrong:** Comparison-script computes delta between baseline-before.json and baseline-after.json, but they used different `limit` parameters, producing invalid delta.
**Why it happens:** Default limit (50) used in one run, explicit limit (200) in the other.
**How to avoid:** comparison-script.py reads `_run_params.limit` from BOTH files and ASSERTS equality before computing delta. Non-matching limits produce non-zero exit.
**Warning signs:** "ABORT: limit mismatch -- before={X} after={Y}" message.

### Pitfall 9: Claude CLI Temperature Non-Controllability
**What goes wrong:** Expecting `--temperature 0` for deterministic LLM scoring. Flag does not exist.
**Why it happens:** Claude Code CLI v2.1.50 does not expose temperature control.
**How to avoid:** Accept non-determinism as a confound. Run each scoring call multiple times if stability matters. Document variance as part of results.
**Warning signs:** CLI error on `--temperature` flag.

### Pitfall 10: Plan 05 Conditional Artifacts
**What goes wrong:** Expecting plan-05/ to always contain all 4 files. Three are CONDITIONAL.
**Why it happens:** sprint-block.md (compound failure), evaluation_observations_constrained.md (Track B), and 3.1c_deferred.md (adversarial-only validity) are generated only under specific failure conditions.
**How to avoid:** Check exit state: field-provenance.md is always present. Others depend on Plan 05 verdict track.
**Warning signs:** Missing conditional artifacts when the corresponding failure condition has been triggered.
