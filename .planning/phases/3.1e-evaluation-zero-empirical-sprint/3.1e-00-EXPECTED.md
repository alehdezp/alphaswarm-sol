# 3.1e-00 Expected Results: Session 0 — Measurement Prerequisites

## Fundamental Deliverable

This plan delivers all measurement prerequisites, baselines, decision locks, and rubrics required before Plans 01-05 can execute, so that every downstream experiment operates on verified, reproducible data with pre-committed decision rules. Without it, Plans 01-05 would build on unverified baselines, risk measurement drift, and allow post-hoc decision modification that invalidates experimental conclusions. Success means a fresh per-function detection baseline exists for 7 contracts, graph fingerprints prove reproducibility, property diagnostics reveal builder behavior, two LLM-injectable rubrics are ready for Plan 02, and all decision rules are locked with an immutable timestamp before any experiment runs. Failure means downstream plans execute against stale or contradictory data, counting methodology is ambiguous across plans, decision rules are invented retroactively, or rubrics do not exist when Plan 02 needs them.

## Drift Detection

### OK: Fewer than 65 FPs in fresh baseline

**Scenario:** The executor runs the detection pipeline and finds 58 FPs instead of the 65 reported in the prior detection baseline document.

**Why OK:** The prior detection-baseline.md had internal contradictions (11 TP / 44 denominator vs 10 TP / 75 total). The entire point of Session 0 is to produce a fresh, verified baseline. Different numbers from the contradictory prior document are expected and healthy.

**Action:** Verify the baseline-before.json has per-function granularity and the is_tp join logic is correct. The new count becomes the authoritative reference.

### OK: Only 2 rubric revision rounds instead of max 3

**Scenario:** The executor creates rubrics that satisfy the scoring band requirements on the first or second draft without needing all 3 permitted revision rounds.

**Why OK:** Max 3 rounds is a ceiling, not a target. Fewer rounds with correct content is preferable.

**Action:** Verify rubrics still have security-specific content with BSKG references, not that they merely have the required section headers.

### DRIFT: Eligible dimensions document omits heuristic formulas

**Scenario:** The executor lists dimension names (reasoning_depth, evidence_quality, etc.) but does not record the actual heuristic formula for each (e.g., `min(100, 30 + unique_tools * 20)`).

**Why DRIFT:** The formulas are essential for Plan 02 anchor design — anchors must equalize heuristic formula inputs between GOOD and MEDIOCRE. Without formulas recorded, Plan 02 cannot design properly constrained anchors.

**Action:** Request the executor to re-read `_heuristic_dimension_score` and record each formula. Also verify the testability gate result (all count-based or not).

### DRIFT: Graph fingerprints computed via fingerprint_graph() instead of graph_fingerprint()

**Scenario:** The executor imports `fingerprint_graph` (legacy function that returns a dict) instead of `graph_fingerprint` (current function that returns a 64-char hex string).

**Why DRIFT:** Plan 01 needs to compare fingerprints as simple string equality to verify graph determinism. A dict output changes the comparison logic and may not detect the same class of drift.

**Action:** Verify fingerprints in `graph-fingerprints.json` are 64-character hex strings (not JSON objects). If dicts, request re-run with `graph_fingerprint`.

### STOP: Decision lock artifacts created after baseline capture

**Scenario:** The executor ran Task 2 (baseline capture) before Task 1 (decision lock creation), or created DECISION-LOCK.md with placeholder content and filled it in after experiments started.

**Why STOP:** The entire experimental methodology depends on decision rules being immutable before execution. Post-hoc decision rules allow circular reasoning — interpreting results to fit desired conclusions. Plan 05 explicitly verifies `locked_at` precedes execution. This undermines the scientific integrity of the sprint.

**Action:** Escalate. The decision lock must be recreated with genuine pre-committed rules, and any experimental results generated before the lock must be re-evaluated against the now-locked rules.

## Reviewer Guidance

**Priorities:**

1. **Decision lock temporal integrity:** Verify DECISION-LOCK.md and pre-commitment.json were created as Task 1 (pre-flight) outputs, not retrofitted after Task 2. The `locked_at` timestamp in pre-commitment.json is the single most critical field in the entire plan — it is the anchor for Plan 05's sprint integrity verification.

2. **Baseline-before.json schema correctness:** Verify per-function granularity (not per-pattern deduplication), presence of `fired_conditions` with `explain=True` sub-keys, and `_corpus_version.sha256` for drift detection. Every downstream plan depends on this schema being correct.

3. **Property diagnostic actionability:** Verify the diagnostic produces findings that can guide Plan 01's pattern fix (not just "properties exist"). The three-state model (`true | false | absent`) and over-reporting/builder-gap flags are what make this diagnostic useful rather than ceremonial.

4. **Rubric security specificity:** Verify rubrics reference BSKG, node IDs, evidence chains, and security reasoning — not generic "good reasoning has clear structure" language. Plan 02's calibration depends on rubrics that discriminate security-domain reasoning quality.

5. **Counting policy verbatim extraction:** Verify counting-policy.md matches the CONTEXT.md "Finding Counting Policy" decision, not a paraphrase. Per-function vs per-pattern counting confusion across plans is a known risk that this artifact is specifically designed to prevent.
