# 3.1e-04 Expected Results: Extract Schemas From Reality

## Fundamental Deliverable

This plan delivers a field-provenance mapping that traces every proposed schema field to an observed experimental artifact from Plans 01-03/04, so that phase 3.1c receives only data-justified schema fields and knows exactly what data package it inherits (data_rich, partial_data, or blocked_recovery). Without it, 3.1c would either inherit speculative schemas (the v5.0 anti-pattern this entire phase exists to prevent) or enter blind without knowing which experiments succeeded. Success means every schema field has an evidence trail to a real artifact, contradictions between experiments are surfaced and resolved or escalated, unobserved phenomena are documented with falsifiable questions, and 3.1c has an explicit entry protocol. Failure means schemas contain speculative fields with no observed data justification, contradictions are papered over or ignored, the Unobserved Phenomena list contains topic labels instead of testable questions, or 3.1c does not know what data state it inherits.

## Alignment Criteria

### [must_have] No speculative schema fields

**Source:** must_haves.truths[0] — "Every schema field traces to an observed data artifact (no speculative fields)"

**The question:** Does every row in field-provenance.md cite a specific source_experiment and source_artifact that exists (or is annotated as degraded/incomplete with justification)?

**Aligned if:** Each field row in the provenance table has non-empty `source_experiment` and `source_artifact` columns pointing to actual files from Plans 01, 02, or 03/04. Fields whose source artifacts do not yet exist are annotated `data_state: pending` or `incomplete` with a phase target. No field has a blank or generic source.

**Misaligned if:** Fields exist with empty source columns, or with sources like "general best practice" or "will be needed later." Any field references an artifact that no experiment produced and lacks a pending/incomplete annotation.

### [must_have] Falsifiable Unobserved Phenomena

**Source:** must_haves.truths[1] — "Unobserved Phenomena list has falsifiable questions, not topic labels"

**The question:** Does each Unobserved Phenomena entry contain a concrete falsifiable question with an evidence anchor, or are entries merely topic labels?

**Aligned if:** Each entry has all six required fields: (a) phenomenon, (b) why it matters, (c) measurement difficulty, (d) target phase, (e) a concrete experimental question in "does X cause Y?" or "can we measure Z?" form, and (f) an evidence anchor citing a specific filename and finding. Entries with inconclusive experiments cite `INCONCLUSIVE: {experiment}, {artifact}, {why no signal}`.

**Misaligned if:** Entries follow the REJECT pattern from the plan: topic labels with no evidence cited and no testable question (e.g., "cross-contract interaction: matters because patterns interact: moderate: 3.1c"). Any entry missing the evidence anchor field.

### [must_have] Three contradiction pairs analyzed

**Source:** must_haves.truths[2] — "Three contradiction pairs explicitly checked and resolved or escalated"

**The question:** Are all three specified contradiction pairs (LLM_adds_signal vs validity coefficient, YAML-fix-sufficient vs builder-fix-required, rubric-calibration-succeeded vs rubric-uncalibrated) explicitly documented with findings from both sides and a resolution or escalation?

**Aligned if:** Three distinct contradiction sections exist, each containing: (a) Finding A with source and artifact, (b) Finding B with source and artifact, (c) explicit statement of whether resolvable from data, (d) if irresolvable: a specific arbitration question routed to 3.1c with data cited on both sides. Pair 2 includes verification that `addressed_count` is an integer for all action_map.json clusters (or documents the update step taken).

**Misaligned if:** Fewer than three pairs analyzed. Any pair lacks the structured (a)/(b)/(c)/(d) format. Contradictions are hand-waved with "these are compatible" without citing specific data. Pair 2 proceeds without checking `addressed_count` state.

## Drift Detection

### OK: Track B constrained rather than active

**Scenario:** Plan 02 produced a "Go (preliminary)" verdict rather than a confirmed "Go," so all Track B fields in field-provenance.md carry `confidence_qualifier: constrained` and an `evaluation_observations_constrained.md` was written instead of full integration.

**Why OK:** The plan explicitly defines graduated activation for Track B. Constrained mode is a pre-registered outcome with a defined response protocol. The provenance mapping is still complete — it just carries honest confidence annotations.

**Action:** Verify that `confidence_qualifier: constrained` is consistently applied to ALL Track B fields (not just some), and that `evaluation_observations_constrained.md` follows the required structured schema.

### OK: Fewer than expected taxonomy clusters in Track C

**Scenario:** Plan 03/04 produced only 2 taxonomy categories due to scope reduction, so Track C has fewer fields than a full run would produce, and some Unobserved Phenomena routing was skipped per pre-authorized degradation.

**Why OK:** The plan pre-authorizes Track C degradation when `scope_reduced=true` and fewer than 2 categories exist. Documenting the degradation honestly is better than inventing fields.

**Action:** Verify `archetype-count.json` actually shows `scope_reduced: true` and that the degradation is documented in field-provenance.md, not silently omitted.

### DRIFT: Contradiction pairs resolved without citing data from both sides

**Scenario:** All three contradiction pairs are marked "resolved" but the analysis references only one experiment's findings, dismissing the other side without engaging its specific data.

**Why DRIFT:** The contradiction analysis exists to surface genuine tensions between experiments. One-sided resolution suggests the executor treated it as a checkbox rather than an analytical exercise. 3.1c needs both sides to make informed decisions.

**Action:** Request re-analysis of any pair where Finding B lacks a specific artifact reference or quote. Both sides must cite concrete data.

### DRIFT: Unobserved Phenomena list contains topic labels disguised as questions

**Scenario:** Entries technically have question marks but are not falsifiable — e.g., "Is cross-contract interaction important?" instead of "Does pattern X fire differently when Contract A calls Contract B vs standalone?"

**Why DRIFT:** Topic labels with question marks are still topic labels. 3.1c cannot design experiments from "is X important?" — it needs "does X cause Y in measurable way Z?"

**Action:** Request rewrite of any entry where the question cannot be answered with a specific experiment design. Each question should imply what data would confirm or deny it.

### STOP: Pydantic models implemented instead of provenance mapping

**Scenario:** The executor wrote Python files containing Pydantic BaseModel classes for detection schemas, evaluation schemas, and taxonomy schemas — treating "extract schemas" as "implement schemas in code."

**Why STOP:** This is the exact v5.0 anti-pattern the entire phase exists to prevent. The plan explicitly states: "The output of this plan is a MARKDOWN TABLE documenting field provenance, NOT Pydantic class implementations." Building code artifacts before 3.1c confirms field stability repeats the failure that motivated phase 3.1e.

**Action:** Escalate. Delete Python schema files. Re-execute with focus on the provenance mapping document. The plan's fundamental purpose was to resist premature infrastructure.

## Reviewer Guidance

**Priorities:**

1. **Verify no speculative fields exist.** This is the plan's raison d'etre. For every row in field-provenance.md, check that `source_artifact` references a real file from Plans 01-03/04. Any field without observed data justification is a plan failure, not a minor gap.

2. **Verify contradiction analysis engages both sides with data.** Three pairs must each cite specific artifacts and findings from both experiments. One-sided dismissals or "compatible" hand-waves without data references indicate the analysis was performed as a formality.

3. **Verify Unobserved Phenomena are falsifiable, not topic labels.** Apply the REJECT/ACCEPT pattern from the plan. Each entry must have an evidence anchor citing a specific file and finding. "Cross-contract interaction: matters because patterns interact" is a failure; "action_map.json cluster X has no YAML-fixable action — reveals gap in Y" is a pass.

4. **Verify `phase_entry_mode` matches actual track status.** Cross-reference the claimed entry mode against Track A/B/C status documented in the file. The mode must be internally consistent — claiming `data_rich` when Track B is `absent` is misalignment.

5. **Verify output is a markdown provenance table, not code.** Any `.py` file with Pydantic models is an immediate STOP signal. The plan explicitly prohibits premature schema implementation.
