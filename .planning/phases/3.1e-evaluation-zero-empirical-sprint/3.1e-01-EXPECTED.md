# Expected Results: Plan 01 — Minimum Viable Loop

## Fundamental Deliverable

This plan delivers the first empirical evidence that detection precision metrics respond to a single pattern fix (access-tierb-001 YAML modification, post-fix baseline, delta comparison, and cost measurement) so that downstream phases (3.1f, 3.1c) can rely on a proven precondition: the improvement loop can detect improvement at all. Without it, 3.1f's multi-pattern sensitivity validation builds on an unverified assumption that metrics are responsive, and Plan 05's schema extraction has no Track A detection data to ground schema fields. Success means precision changes measurably after a targeted YAML fix, all existing TPs are preserved (zero regression), a reproducible comparison script classifies every finding's change status, and cost-per-FP-removed is quantified. Failure means the fix either breaks existing true positives, produces no measurable delta, or the before/after comparison is not reproducible — any of which would invalidate the metric responsiveness precondition that the entire phase depends on.

## Alignment Criteria

### [must_have] Measurable precision delta from single pattern fix

**Source:** must_haves.truths[0] — "Single pattern fix (access-tierb-001) produces measurable precision delta"

**The question:** Does the execution produce evidence that detection precision changed after modifying access-tierb-001?

**Aligned if:** `baseline-after.json` exists and the comparison script output shows `precision_delta != 0` (either Outcome A with >= 5pp or Outcome B with 0 < delta < 5pp). The YAML diff in `access-tierb-001-trust-assumption-violation.yaml` shows targeted condition changes (tightened exclusions, additional guards). `cost-summary.json` has `outcome` field set to "A" or "B" with non-null `fps_removed`.

**Misaligned if:** No precision delta recorded. Or the YAML file is unchanged and the plan claims Outcome C without documenting builder-change requirements exceeding 200 LOC AND no YAML-addressable subset. Or `baseline-after.json` is identical to `baseline-before.json` with no explanation.

### [must_have] Zero regression — all TPs preserved

**Source:** must_haves.truths[1] — "All existing TPs preserved after pattern fix (zero regression)"

**The question:** Does the comparison script confirm that no true positive was lost across the 7-contract corpus after the pattern fix?

**Aligned if:** The comparison script's delta table contains zero entries classified as `regression`. ## Drift Detection

### OK: Pattern fix addresses fewer FPs than expected

**Scenario:** The YAML fix removes 3 FPs instead of the expected 10+, producing a small but measurable precision delta (Outcome B). The executor documents the limited scope and routes remaining FPs to Plan 03/04 action_map as builder-fixable.

**Why OK:** The plan explicitly defines Outcome B as a valid result. Metric responsiveness is demonstrated even with a small delta. The routing to Plan 03/04 preserves the information for downstream use.

**Action:** Verify `cost-summary.json` has `outcome: "B"` and that `precision_delta > 0`. Verify action_map routing exists for unaddressed FP categories.

### OK: Comparison script uses slightly different aggregation method

**Scenario:** The executor computes precision as `TP / (TP + FP)` per-contract then averages, rather than computing a single global precision. Both methods show the same directional change.

**Why OK:** The plan specifies precision delta measurement but leaves the exact aggregation to executor discretion ("Claude's Discretion" in CONTEXT.md covers comparison harness implementation). As long as the method is documented and reproducible, the variant preserves intent.

**Action:** Verify the aggregation method is documented in the comparison script. Verify both before and after use the same method. Verify the delta direction matches expectations.

### DRIFT: Baseline-after generated with different limit or lens parameters

**Scenario:** The executor ran the detection pipeline with `limit=500` or a different lens set for `baseline-after.json`, producing incomparable results vs `baseline-before.json` which used `limit=200`.

**Why DRIFT:** The delta comparison is invalid if run parameters differ. The plan explicitly requires matching `_run_params: {limit: 200, lens: DEFAULT_LENSES}` between before and after baselines.

**Action:** Request re-run with matching parameters. Verify `_run_params` in both JSON files are identical.

### DRIFT: Graph fingerprint mismatch ignored

**Scenario:** The executor noticed that graph fingerprints for 2 contracts differed from Session 0 values but proceeded with delta computation anyway, noting the discrepancy in a comment.

**Why DRIFT:** The plan explicitly states: if ANY contract fingerprint differs, STOP. Do not compute delta. A fingerprint mismatch means the graph changed between runs, making the comparison unreliable. This is a determinism escalation, not a minor note.

**Action:** Request that the executor produce `fingerprint-mismatch.md` documenting which contracts differ. Delta computation must be deferred until determinism is resolved.

### STOP: Pattern fix modifies builder code instead of YAML

**Scenario:** The executor modified `kg/builder/` Python source code to change how `has_access_gate` is computed, rather than making a YAML-only pattern fix. The modification is 150 LOC of builder changes.

**Why STOP:** Plan 01 is scoped to YAML-only fixes. Builder changes belong to Plans 03/04 or 3.1c. Modifying the builder invalidates graph fingerprints, breaks Session 0 baselines, and exceeds the plan's authority. If builder changes were needed, Outcome C should have been declared (or the YAML-addressable subset should have been fixed first).

**Action:** Revert builder changes. Evaluate whether a YAML-only partial fix exists. If not, classify as Outcome C and produce cost-summary.json with null computed fields. Escalate builder fix requirements to Plan 03/04 action_map.

## Reviewer Guidance

**Priorities:**

1. **Zero regression is the hardest constraint.** Verify the comparison script's delta table shows zero `regression` entries AND the ReentrancyWithGuard negative control passes. A precision improvement that loses a TP is worse than no improvement at all.

2. **Graph determinism was maintained.** Verify that graph fingerprints match Session 0 values before any delta was computed. If fingerprints differ, the entire delta comparison is invalid regardless of what the numbers show.

3. **Baseline-after was pipeline-generated with matching parameters.** Confirm `_run_params` match between before and after baselines (`limit: 200`, `lens: DEFAULT_LENSES`). Confirm the same script/pipeline was used. A manually constructed baseline-after defeats the reproducibility requirement.

4. **Outcome classification follows pre-registered rules.** Cross-reference the measured `precision_delta` against DECISION-LOCK.md Plan 01 thresholds. The outcome (A/B/C) must be consistent with the data, not post-hoc rationalization.

5. **Cost measurement uses real timing, not estimates.** `compute_seconds` in `cost-summary.json` should be a measured value from the pipeline run, not a rough estimate or placeholder.
