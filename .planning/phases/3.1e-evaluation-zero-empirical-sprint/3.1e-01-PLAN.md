<objective>
Plan 01: Minimum Viable Loop — fix one pattern (access-tierb-001), re-run detection pipeline, measure precision delta.

Purpose: Prove detection metrics are RESPONSIVE to a single-pattern fix. This is NOT sensitivity validation (that requires multi-pattern corpus in 3.1f) — it establishes the precondition that metrics can detect improvement at all. Also produces first cost-per-FP-removed measurement for 3.1f loop economics.

Output: YAML fix + post-fix baseline + delta comparison + cost summary. Three possible outcomes:
- **Outcome A (success):** precision_delta > 0 AND tp_count unchanged on primary contracts → metrics are responsive
- **Outcome B (insufficient signal):** precision improves but < 5 percentage points → metric responsive but weak signal
- **Outcome C (fix scope exceeds sprint):** pattern fix requires > ~200 LOC of builder changes → defer to 3.1c
</objective>

    Create `.vrs/experiments/plan-01/comparison-script.py` that:
    - Reads baseline-before.json and baseline-after.json
    - Produces delta table: `(contract, pattern_id, function_name, before_is_tp, after_is_tp, change)` where change is one of: `fp_removed`, `tp_preserved`, `regression`, `unchanged`
    - Uses same is_tp join logic as session-0-baseline-script.py
    - Computes aggregate: FPs_removed, TPs_preserved, regressions, precision_before, precision_after, precision_delta
    - Includes a `negative_controls` array in baseline-before.json with pre-registered entries: `{ "contract": "ReentrancyWithGuard", "expected_tp": 0, "expected_fp_max": <N from Session 0> }`. The Session 0 value of N is determined empirically and recorded in Plan 00.
    - **BLOCKING negative control assertion:** before computing any delta metrics, the comparison script checks all negative control contracts. If any finding on ReentrancyWithGuard is classified as TP, the script halts immediately with: `"NEGATIVE CONTROL VIOLATION: unexpected TP on guarded contract — halting before metric comparison."` This is a hard stop, not a warning.
    - Non-zero FP count on the negative control is recorded and flagged as a known-FP entry in the delta table (not silently accepted).
    - ReentrancyWithGuard is NEGATIVE CONTROL: access-tierb-001 must NOT fire after fix. Other patterns on that contract remain unchanged. If access-tierb-001 still fires on ReentrancyWithGuard post-fix: REVERT the YAML change, document the revert reason in `revert-log.md`, and escalate to human before re-attempting. Do NOT classify any outcome until the negative control passes — this is a regression introduced by the fix, not Outcome B.
    - Primary test contracts: SideEntranceLenderPool + NaiveReceiverPool

    **Negative Control Halt Recovery**

    When the negative control halt fires, follow this three-step triage before retrying:

    1. **Counting policy check:** Re-run the counting policy rules (from the classification logic in the comparison script) against the specific flagged finding. Verify whether the finding was misclassified as TP due to a counting error (e.g., join key mismatch, stale ground-truth label). Log one-line verdict: `"COUNTING: <pass|fail> — <reason>"`.
    2. **Corpus classification check:** Verify that the corpus element classification confirms ReentrancyWithGuard is genuinely safe (i.e., the contract's ground-truth labels in the CORPUS mark it as guarded/non-vulnerable for the flagged pattern). Log one-line verdict: `"CORPUS: <confirmed-safe|misclassified> — <reason>"`.
    3. **Escalation decision:** If both checks pass (counting is correct AND corpus confirms safe), the finding is a true detection false-positive introduced by the pattern fix. The pattern fix must be revised to eliminate this false-positive before retry. Log: `"ESCALATE: true FP on negative control — pattern fix requires revision before retry"`.

    Each triage verdict is appended to `.vrs/experiments/plan-01/negative-control-triage.md` so the retry has causal context.
