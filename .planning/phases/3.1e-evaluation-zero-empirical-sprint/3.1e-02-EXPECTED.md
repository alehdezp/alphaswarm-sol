# 3.1e-02 Expected Results: First Real LLM Evaluation

## Fundamental Deliverable

This plan delivers the first real invocation of REASONING_PROMPT_TEMPLATE with rubric text, calibration anchors, and a three-instrument measurement protocol, so that the project can make an evidence-based go/no-go decision on whether LLM-as-judge adds signal beyond heuristic scoring for reasoning evaluation. Without it, phase 3.1c has no empirical basis for deciding how much to invest in LLM evaluation infrastructure, and the project continues building on assumptions about LLM utility that have never been tested. Success means the template was called with real rubric content, three calibration anchors were scored across ordering/calibration/divergence instruments, a hallucination check validated LLM evidence claims, the rubric was frozen with a SHA256 stamp, and a go/no-go verdict was produced using hypothesis scoring (H1+H2+H3) that cites a specific row from the pre-committed decision table. Failure means the template remains uncalled, the verdict is produced without empirical scoring data, or the go/no-go decision is made without the three-instrument protocol and hallucination check -- effectively rubber-stamping LLM investment without evidence.

## Drift Detection

### OK: Calibration incomplete after 3 revision rounds

**Scenario:** The executor attempted 3 rubric revision rounds but GOOD-BAD spread did not reach 30. They froze the best version, recorded `calibration-incomplete`, and proceeded to scoring.

**Why OK:** The plan explicitly allows this path. The rubric freeze protocol permits freezing at the best version after 3 rounds. Proceeding with an annotated incomplete calibration preserves the experiment's empirical value.

**Action:** Verify the annotation `calibration-incomplete` appears in the artifacts. Verify 3 revision rounds were actually attempted (check `llm-scores.jsonl` for multiple calibration entries). Verify scoring still used the three-instrument model despite incomplete calibration.

### OK: Session time guard triggered at 90 minutes

**Scenario:** Session B reached 90 minutes with calibration in progress. The executor froze at the current rubric version and proceeded directly to scoring.

**Why OK:** The plan includes a 90-minute session time guard. Completing partial scoring is more valuable than abandoning the session.

**Action:** Verify the time guard is documented. Verify the scoring that did occur follows the three-instrument protocol. Check if sufficient data exists for a hypothesis-scored verdict (even if some instruments have fewer data points).

### DRIFT: Verdict produced without hallucination check

**Scenario:** The executor completed pairwise ordering and scale calibration, produced a verdict, but skipped the hallucination check because the LLM scores looked consistent.

**Why DRIFT:** The hallucination check is a must_have truth. Consistent scores do not preclude fabricated evidence. A dimension could appear to show "LLM adds signal" when the LLM was actually hallucinating features not present in the anchors.

**Action:** Request the executor perform the hallucination check on the 3 strongest evidence claims per dimension and update the verdict if any fabrication is found.

### DRIFT: Go/no-go verdict without hypothesis scoring framework

**Scenario:** The executor produced `llm-recommendation.md` with a narrative verdict ("Based on our observation, LLM scoring shows promise") but without explicit H1, H2, H3 scores or citation of a decision table row.

**Why DRIFT:** The hypothesis scoring framework (H1+H2+H3 with weights, mapped to a decision table) is the pre-committed decision mechanism. A narrative verdict without the framework is post-hoc rationalization, exactly what pre-registration was designed to prevent.

**Action:** Request the executor compute explicit H1, H2, H3 scores from the existing scoring data and map the total to the decision table. The narrative may remain but must be anchored to the framework.

### STOP: Template called with placeholder or generic rubric text

**Scenario:** REASONING_PROMPT_TEMPLATE was invoked but the `{expected}` field contained generic text like "Good reasoning demonstrates clear thinking" rather than content derived from the security-specific rubrics (reasoning_depth_v1.0.txt / evidence_quality_v1.0.txt). Anchors were generic software analysis transcripts rather than smart contract security investigation transcripts.

**Why STOP:** The entire purpose of this plan is the FIRST REAL invocation -- real meaning rubrics grounded in the project's security-reasoning domain and anchors grounded in BSKG-based vulnerability investigation. Generic content makes the go/no-go decision meaningless for this project's specific needs. The result would not inform 3.1c about whether LLM evaluation adds signal for smart contract security reasoning specifically.

**Action:** Escalate. The rubrics from Plan 00 must be used. Anchors must reference real contract names from the baseline corpus with BSKG query syntax. Replan the scoring session with domain-grounded content.

1. Verify REASONING_PROMPT_TEMPLATE was actually called with domain-specific content (not generic text) by checking that `llm-scores.jsonl` has entries with `expected_sha256` values traceable to the Plan 00 rubric files and that anchor `{observed}` content references BSKG queries and real contract names.
