8. **Multi-agent information flow is out of scope** — All 3.1e experiments evaluate single-agent capabilities. Multi-agent debate is deferred to Phase 4. Rubrics and schemas from 3.1e may require adaptation for multi-agent transcripts.

**Evaluator Stress Test (EST) Framework (arxiv 2507.05619):**
- Invariance-based framework for detecting proxy gaming in RL and LLM alignment

### Plan 01: Minimum Viable Loop [HIGH confidence]
**Intent:** Establish detection metric responsiveness by fixing one pattern and measuring the delta — a necessary precondition for automated improvement in 3.1f, not a proof of full sensitivity.
**What it delivers:** Three-outcome measurement process: (a) pipeline reproducibility — run `build_lens_report()` twice on the SAME pre-built graph, tests detection pipeline reproducibility (NOT graph building), confirm same graph object produces identical JSON (if output differs on same object, that is a detection engine bug — stop and diagnose); (b) YAML exclusion effectiveness — measurable FP reduction without TP loss on 7-contract suite; (c) effort-to-improvement ratio — LOC changed per FP removed. Before/after FP counts on 7-contract suite, TP preservation confirmed. ~50 LOC YAML changes + ~20 LOC comparison script. Cost instrumentation at three levels: full 7-contract pipeline, targeted 2-3 contract run, lens-report-only on pre-built graphs. Cost deliverables: `.vrs/experiments/plan-01/cost-summary.json` (2 records: before and after runs) with fields `{level: "before"|"after", compute_seconds, wall_clock_seconds, contracts_processed, cost_per_FP_removed, LOC_per_FP_removed}`. Note: Plan 01 is LLM-free — `compute_seconds` replaces `token_count` (which would always be 0 here). Plans 02 and 03/04 record both `token_count` and `compute_seconds`. For 3.1f cost-ceiling decisions, establish a token-to-second exchange rate based on Plan 02 observed data (tokens / compute_seconds on first LLM call), enabling comparison with Plan 01's compute_seconds baseline.
**What problem it solves:** If metric responsiveness cannot be demonstrated, all improvement infrastructure (3.1c Plans 07-12, all of 3.1f) is premature. This is the go/no-go gate for the entire approach.
**Falsification criteria:** Outcome A: first diagnostic step — compare graph fingerprints from Session 0 build vs Plan 01 build using `compare_fingerprints()` from `alphaswarm_sol.kg.fingerprint`. If `identical=False`: graph construction non-determinism detected — do NOT classify as detection engine bug. Route to Assumption 7 and flag for systematic BSKG audit. If `identical=True`: Outcome A divergence is definitively a detection engine issue. Outcome A fails if identical runs produce different counts on same pre-built graph where graphs are fingerprint-identical (stop — detection engine bug; diagnose whether from pattern matching, graph difference, or classification logic). Additionally, Outcome A reproducibility comparison must use `limit=200` in both runs — do not mix limit values. Outcome B fails if zero FPs removed or any TP lost (revert). Mixed Outcome B (partial): if YAML fix removes subset S1 of FPs but leaves subset S2 untouched, pre-registered interpretation: Outcome B = CONFIRMED for S1 root cause, FINDING = S2 requires different intervention. Report: (a) FPs removed by root cause, (b) FPs remaining by root cause, (c) route S2 root causes to Plan 03/04 action_map. Outcome C: if fix requires >200 LOC: STOP. ## Implementation Decisions

### Experiment Before Infrastructure
All adversarial reviewers (4 rounds, 12 agents) converged: run experiments before building schemas. No Pydantic model without observed data justifying it.

### Pre-Sprint Decision Lock
All per-plan decision rules (outcome interpretation, flip_resolution defaults, scope gates) must be pre-committed BEFORE Session A execution. **DECISION-LOCK.md** (`.planning/phases/3.1e-evaluation-zero-empirical-sprint/DECISION-LOCK.md`): sections for each Plan (01, 02, 03/04, 05) with: (a) outcome variables observed, (b) decision rules in JSON format, (c) no-modification clause (rules locked before Session A execution). **pre-commitment.json** (`.planning/phases/3.1e-evaluation-zero-empirical-sprint/pre-commitment.json`): schema `{ "plan_01": {rules...}, "plan_02": {rules...}, "plan_03_04": {rules...}, "plan_05": {rules...}, "locked_at": "ISO timestamp", "locked_by": "executor name" }`. Plan 05 step 0 reads pre-commitment.json and verifies `locked_at` precedes first Session A execution — modification after lock timestamp triggers REJECT. Each plan's Pre-Commitment section reproduces applicable decision rules as 1:1 copy from DECISION-LOCK.md; executor signs off before plan execution.

### Pre-Registration Principle
Every experiment must specify, BEFORE execution: (a) what constitutes success, (b) what constitutes failure, (c) what the ambiguous-middle interpretation rule is. Written into plan description, not discovered during analysis.

### Pre-Registration Checklist
Before Session 0 executes, each plan must pass: (a) every named outcome has success/failure/ambiguous-middle interpretation; (b) every measurement instrument has at least one falsification condition that is not circular with the instrument's design; (c) the most likely PARTIAL outcome (mixed result across root causes) has a pre-registered interpretation; (d) every go/no-go threshold is consistent with all other plans' thresholds for the same metric; (e) held-out anchor coverage: does the held-out anchor differ from GOOD on at least one heuristic-formula-input dimension (unique_tool_count OR bskg_query_count)? If both GOOD and held-out have unique_tool_count=3 AND bskg_query_count=3, the held-out is not independent and cannot serve as a generalization test. Pre-verify this before calibration begins. Apply as final pre-session review step. Flag any gap found as blocking.

### LLM Call Method: claude -p for Scoring
For Plan 02 scoring calls, use `claude -p --output-format json`. Stateless single-prompt tasks where the question is "does this rubric produce informative scores?" must not have orchestration overhead as a confounding variable. Agent Teams deferred to 3.1c where orchestration IS the thing being tested.

### Evaluator Authority Axes
Axis A (reasoning process quality) is the declared scope. Axis B (conclusion correctness) requires ground-truth lookup, not LLM judgment. Separate concerns. Plan 02 delivers Axis A vs B boundary decision.

### Evaluation Layer Content Validation Policy
Observation layer (hooks) is proxy-only by design — event type and count, no content inspection. Heuristic layer (GVS/ReasoningEvaluator) is proxy-only by design. Content validation is the exclusive responsibility of the LLM evaluator layer (Axis A). This is a conscious tradeoff: layers 1-2 are gameable; layer 3 is the defense. Note: all 3.1e experiments validate single-agent capabilities only. Multi-agent transcript structure may differ — layer 3 rubric transferability to multi-agent is a 3.1c question.

### 7-Move Decomposition Status
Plan 02 evaluates LLM scoring against EXISTING evaluation contract dimensions. It does NOT validate the 7-move decomposition. The 7-move framework remains unimplemented and untested — its validity is a 3.1c design decision.

### Plan 01 Establishes Metric Responsiveness as Precondition — Not Sensitivity Validation
Plan 01 proves detection metrics are RESPONSIVE to a single-pattern fix on a known-over-firing pattern. Full sensitivity validation — that metrics distinguish degrees of improvement proportionally — requires the multi-pattern corpus in 3.1f. These are separate claims.

### Finding Counting Policy
For Plan 01 delta measurement: count per-function-firing. `_unique_patterns()` per-pattern-per-contract masks intra-contract co-fires — when a pattern fires on both TP and FP functions in the same contract, the harness counts only TP, making the FP invisible. Per-function required for accurate delta measurement. `_unique_patterns()` adequate for regression detection, insufficient for delta measurement. Session 0 task (a) records BOTH per-pattern and per-function counts. Pre-generate `counting-policy.md` before Session 0. Content: extract this decision verbatim. Path: `.vrs/experiments/counting-policy.md`.

### Plans 03 and 04 Merge
Same corpus, same technique, same customer. Phase 0 taxonomy is descriptive/diagnostic. Phase 1 adversarial archetypes are generated from evaluator code-path enumeration. Single-pass action framing is an anti-pattern here.

### Taxonomy and Adversarial Archetypes Are Separate Classification Systems
Phase 0 = what detector gets wrong (taxonomy). Phase 1 = how evaluator can be gamed (archetypes). Taxonomy categories annotate transcript domain scenarios, not archetype count. Taxonomy collapse does not constrain adversarial scope. Rich taxonomy does not expand archetype count without finding new code paths.

### N-Appropriate Statistical Decision Criteria
All correlation-based decision thresholds must specify the expected N and use N-appropriate statistical criteria. Thresholds from literature (typically N > 30) are not valid at N < 10. For N < 10: use rank-ordering with inversion count. For N >= 10: Spearman rho with standard thresholds.

### Fix Cost Classification in Merged Plan 03/04
Per failure category: YAML-fixable | builder-fixable | architectural | graph-property-error. Running across 65 FPs is ~0 additional LOC — one dimension on the same LLM prompt.

### Session 0 Ceiling
Session 0: 1 session max (~2 hours), 3 bounded tasks + instrumentation requirements. Time budget ~100 min core (5 min freed from pre-task simplification). Overflow rule: if task (a1) exceeds 35 min, record partial baseline, complete in Plan 01 first action.

### Measurement Methodology
3.1e uses per-function-firing for delta measurement (maximum sensitivity). EVMbench uses per-vulnerability for benchmark comparison. These serve different purposes. Plan 05 must document the mapping between per-function-firing counts (3.1e) and per-vulnerability counts (EVMbench). Phase 5 should document rationale if it departs from per-function-firing baseline — it should ADD per-vulnerability alongside, not replace.

### Budget Recount
Elapsed estimate: 8-12 sessions accounting for human review gates, potential Session 0 scope adjustments, and gate failures. Work-session estimate (zero failures, no scope changes): 6-7 sessions. Session 0 (1) + Plan 01 (1) + Plan 02 core (2-2.5) + stretch (0.5) + merged Plan 03/04 (1.5-2) + Plan 05 (0.5) = 6.5-7.5 core sessions. Critical path: Session 0 (1) → Plan 01 (1) → Plan 02 + Plan 03/04 Phases 0+1 in parallel (2-2.5) → Plan 03/04 Phase 2 (0.5-1, after Plan 02 freeze and Phase 1 quality gate) → Plan 05 (0.5) = 5-6 core sessions on critical path. Phase 1 quality gate evaluation is sequential AFTER Plan 02 rubric freeze. If quality gate fires, add ~30 min buffer before Phase 2 can start. Fallback scope lever for critical-path pressure: if Plan 02 runs over AND Plan 03/04 Phase 2 must start on schedule, option sequence: (a) complete calibration with best version [preferred], (b) reduce Plan 03/04 Phase 2 to Phase 1 only (skip gameability matrix — critical-path compression), (c) defer Plan 03/04 Phase 2 one week [does NOT block Plan 05 or 3.1c entry]. Pre-register which option applies if time pressure materializes. "Sprint" refers to experimental methodology — small, bounded, falsifiable experiments — not execution speed. Phase 0 MAY run in parallel with Plan 01 (requires only baseline data). Parallelism is available but each parallel stream may require separate session slots.

### Loop Economics Formula
Plan 01 computes `cost_per_FP_removed = total_pipeline_cost / FPs_removed` and `LOC_per_FP_removed = LOC_changed / FPs_removed`. Plan 01 is LLM-free: cost is measured in `compute_seconds`. Plans 02, 03/04 cost = token_count + compute_seconds. For 3.1f cost-ceiling decisions, establish a token-to-second exchange rate based on Plan 02 observed data (tokens / compute_seconds on first LLM call) to convert Plan 02+ costs to a single unit for comparison with Plan 01. These are Plan 01's contribution to 3.1f loop economics. No threshold set in 3.1e — threshold is a 3.1f decision.

### Heuristic Validity, Not Goodhart
The heuristic was designed as a disposable proxy. No agent is optimizing toward heuristic scores. The risk is misleading DIAGNOSIS (low validity), not Goodhart optimization. Plan 03 measures validity coefficient (Spearman rho per dimension), not gameability.

### Rubric Freeze Protocol
Rubric revision permitted during calibration round (max 3 targeted revisions). Freeze is atomic: SHA256 recorded immediately after successful calibration round, no edits permitted after. Final calibration-round scores are official scores. Post-freeze correction creates v1.1 requiring full re-calibration. Calibration chain logging: `(prompt_sha256, dimension, expected_sha256, observed_sha256, workflow_id, persona, llm_score, llm_evidence_count, cost_tokens, timestamp)` to `.vrs/experiments/plan-{NN}/llm-scores.jsonl`.

### No Verified Reference Point, No Experiment
No plan proceeds against an unverified reference point. Sprint speed does not override measurement validity.

### EVMbench as Optional Reference
EVMbench is potential reference, not required. Plan 05 succeeds without it. Patch/exploit methodologies deferred to Phase 5. Detection scoring: study only.

### Human Approval at Every Gate
No automated deployment of improvements. Every pattern fix, schema change, or infrastructure decision requires human review. No exceptions until Level 2 proven (Phase 4.1+).

### No Coupling to .planning/
VRS writes to `.vrs/`, MSD reads when ready. No bidirectional sync between testing infrastructure and planning infrastructure.

### Three-Instrument Measurement Model (Plan 02)
Instrument 1 (ordering): pairwise comparison. Instrument 2 (scale calibration): absolute scores for 3.1c data only. Instrument 3 (divergence evidence): LLM explanation content. Go/no-go uses Instruments 1 + 3. Instrument 2 is informational.

### Eligible Dimension Pool
Before Session 0, enumerate all primary-tier dimensions with heuristic handlers from `reasoning_evaluator.py`. Record as pre-flight checklist item. Primary: `reasoning_depth`, `evidence_quality`. The 7 non-default-30 dimensions in `_heuristic_dimension_score` as of 3.1e are all count-based proxies. Confirming this during pre-flight enables anchor design where heuristic formula inputs are equalized between GOOD and MEDIOCRE, making count-based limitations a feature of the test design. Fallback order pre-registered from pool.

### Plan 05 Schema Naming
Dimension names shared between heuristic and LLM layers are presumptively distinct constructs. Layer-qualified names (`heuristic_reasoning_depth`, `llm_reasoning_depth`) are the default for any dimension appearing in both heuristic and LLM scoring. Unification requires explicit validity evidence from validity-matrix.json case (a) with a `construct_note` annotation.
