## Goal

Ship only capabilities that pass philosophy-aligned gates with reproducible evidence artifacts.

## Why This Phase Exists

`docs/PHILOSOPHY.md` requires proof-token chains, markers, and validation gates. Shipping must enforce those requirements rather than rely on optimistic readiness statements.

1. GA readiness criteria are not yet unified with philosophy gates G0-G7.
2. Some shipped skill claims are ahead of demonstrated artifacts.
3. Marker and proof-token requirements are not consistently release-blocking.
4. Benchmark and debate evidence must be complete before capability claims are promoted.

## Plans (Validation-First Ordered Tasks)

### 8-01: Lock GA Gate Contract

- Define release blockers mapped to G0-G7 plus required orchestration markers.
- Require artifact links for every gate decision.

#### Reasoning

The GA gate contract is the single enforcement point that prevents premature shipping. It must be immutable once locked — any override mechanism (env var, CLI flag, lenient mode) creates a path to ship unvalidated capabilities. The contract references G0-G7 as defined in the GA Gate Contract section below, and each gate's pass/fail must be determined solely by artifact presence and validation, never by human judgment calls.

#### Expected Outputs

- `ga_gate_contract.json` — Machine-readable contract with gate IDs (G0-G7), required artifact paths, schema references, and checksums
- `gate_validator.py` — Deterministic validation script that reads `ga_gate_contract.json` and checks every artifact exists, passes schema validation, and has matching checksums
- `gate_validation_report.json` — Full pass/fail matrix with evidence IDs, generated by `gate_validator.py`
- `marker_contract.json` — Enumerated list of required orchestration markers with regex patterns for transcript validation
- Immutability guarantee: the locked contract file is read-only and tracked via git hash

#### Testing Strategy

| Output | Method 1 | Method 2 |
|--------|----------|----------|
| `ga_gate_contract.json` | JSON Schema validation against `schemas/ga_gate_contract.schema.json` | Round-trip test: load, serialize, reload, compare |
| `gate_validator.py` | Unit tests with synthetic passing and failing artifact sets | Mutation test: remove one artifact, confirm gate fails |
| `gate_validation_report.json` | Schema validation + assert every gate ID present | Cross-reference: every gate in contract appears in report |
| No-override guarantee | Grep codebase for `SKIP_GATES`, `--force`, `--lenient`, `manual_pass` — assert zero matches | Env var injection test: set `SKIP_GATES=true`, run validator, confirm it still fails on missing artifacts |
| Marker contract | Regex patterns match real transcript markers from Phase 3.2 | Negative test: transcript missing a marker fails validation |

#### Drift Detection

| Expected Behavior | Bypass Indicators | Enforcement |
|--------------------|-------------------|-------------|
| Gate validator has no skip/force/override flags | Grep for `argparse.*force`, `os.environ.*SKIP`, `--lenient`, `manual.*pass` in `gate_validator.py` | CI check: `rg -c 'SKIP\|force\|lenient\|manual.pass' gate_validator.py` must return 0 |
| Every gate requires at least one artifact ID | Gate with empty `required_artifacts` list | Schema enforces `minItems: 1` on `required_artifacts` array |
| Contract file is immutable after lock | Git diff shows changes to `ga_gate_contract.json` after lock commit | Pre-commit hook rejects modifications to locked contract file |
| Validation is fail-closed (missing artifact = FAIL) | Default return value is `True` or `pass` | Unit test: empty artifact directory produces all-FAIL report |

### 8-02: Run Pre-Ship Validation Sweep

- Execute health checks, E2E audit checks, debate checks, benchmark checks, and regression checks.
- Fail release if any gate lacks required evidence.
- Include calibration checks for evaluation bias (false-accept control on judged outputs).

#### Reasoning

The validation sweep is a clean-room re-execution of all gates in a fresh workspace with no cached results, no warm state, and no pre-existing `.vrs/` directories. This prevents the "works on my machine" failure mode where stale artifacts or environment pollution produce false passes. Calibration checks ensure that any LLM-judged outputs (debate verdicts, finding classifications) are not systematically biased toward acceptance.

#### Expected Outputs

- `sweep_environment.json` — Snapshot of clean workspace state before sweep: empty `.vrs/`, fresh clone hash, tool versions
- `sweep_results.json` — Aggregated pass/fail matrix for all gates (G0-G7) with individual artifact IDs and timestamps
- `calibration_report.json` — False-accept rate on known-safe contracts (should find zero real vulnerabilities) and false-reject rate on known-vulnerable contracts
- `sweep_transcript.log` — Full execution transcript with all orchestration markers present
- `sweep_checksums.json` — SHA-256 checksums of every generated artifact for reproducibility verification

#### Testing Strategy

| Output | Method 1 | Method 2 |
|--------|----------|----------|
| Clean workspace guarantee | Assert `.vrs/` directory is empty at sweep start; hash workspace state | Run sweep twice from fresh clones, compare `sweep_checksums.json` |
| `sweep_results.json` | Schema validation; every gate ID from contract present | Cross-reference with `gate_validation_report.json` — results must match |
| `calibration_report.json` | Run against `tests/contracts/*_safe.sol` — assert zero confirmed findings | Run against `tests/contracts/*_vuln.sol` — assert known vulns detected |
| Marker completeness | Parse `sweep_transcript.log` for all markers in `marker_contract.json` | Transcript validator (Phase 3.1-04) applied to sweep transcript |
| No cached results | Hash all inputs before and after sweep; no pre-existing output files | Environment variable `VRS_CLEAN_SWEEP=1` enforced, `.vrs/cache/` absent |

#### Drift Detection

| Expected Behavior | Bypass Indicators | Enforcement |
|--------------------|-------------------|-------------|
| Sweep runs in fresh workspace with no cached state | `.vrs/cache/` or `.vrs/graphs/` present before sweep starts | Pre-sweep assertion: `ls .vrs/` fails or directory is empty |
| All gates evaluated, none skipped | `sweep_results.json` has fewer entries than `ga_gate_contract.json` gates | Validator counts gates in contract vs gates in results; mismatch = FAIL |
| Calibration checks actually run on known-safe contracts | `calibration_report.json` missing or has zero entries | Schema requires `safe_contract_count >= 1` and `vuln_contract_count >= 1` |
| Failing checks block release | Sweep exits 0 despite gate failures | Unit test: inject a failing gate, assert sweep exit code != 0 |

### 8-03: Reconcile Skill and Registry Reality

- Ensure shipped/active labels match actual invocable and validated capabilities.
- Defer unsupported features to planned status.

#### Reasoning

The skill registry (`src/alphaswarm_sol/skills/registry.yaml`) is the source of truth for what the product claims to offer. Any skill marked "shipped" or "active" that cannot actually be invoked via Agent Teams + claude-code-controller is a false claim. This plan performs a live invocability test — not a mock test — against the real programmatic API to confirm every shipped skill loads, parses, and responds to a health-check invocation. Skills that fail are downgraded to "planned" with a tracking issue.

#### Expected Outputs

- `skill_invocability_report.json` — Per-skill pass/fail with invocation method, response time, and error details
- `registry_reconciliation_diff.json` — List of skills whose status changed (e.g., "shipped" -> "planned") with justification
- Updated `registry.yaml` — All status labels reflect actual invocability results
- `agent_catalog_reconciliation.json` — Cross-check that every agent referenced by a shipped skill exists in `src/alphaswarm_sol/agents/catalog.yaml`
- `skill_dependency_map.json` — For each shipped skill, the list of CLI commands, agents, and tools it requires, with availability status

#### Testing Strategy

| Output | Method 1 | Method 2 |
|--------|----------|----------|
| Skill invocability | Run each shipped skill via claude-code-controller programmatic API; assert exit 0 and valid response | Parse skill `.md` files for required CLI commands; run each command with `--help` to confirm availability |
| Registry accuracy | Compare `registry.yaml` status fields against `skill_invocability_report.json` | Snapshot test: diff current `registry.yaml` against last-known-good version |
| Agent catalog completeness | For each skill, extract agent references; assert all exist in `catalog.yaml` | Parse `catalog.yaml`; assert every agent has a valid model assignment |
| No phantom skills | Count skills with status "shipped" in registry; count skills that pass invocability; assert equal | Cross-reference shipped skills list with `src/alphaswarm_sol/shipping/skills/` directory contents |
| Dependency availability | For each entry in `skill_dependency_map.json`, run the dependency command | `alphaswarm tools status` covers CLI tool dependencies; compare against map |

#### Drift Detection

| Expected Behavior | Bypass Indicators | Enforcement |
|--------------------|-------------------|-------------|
| Invocability tested with real SDK, not mocks | Test file imports `unittest.mock` or `MagicMock` for SDK calls | Grep invocability test files for mock imports; assert zero matches |
| Non-functional skills downgraded to "planned" | `registry.yaml` has "shipped" skills not in `skill_invocability_report.json` pass list | CI job: parse both files, assert shipped set is subset of passed set |
| Skill files exist for all shipped entries | Registry references a skill path that does not exist on disk | Validation script: for each shipped skill, `os.path.exists(skill_path)` |
| Agent references resolve | Skill references agent ID not in `catalog.yaml` | Cross-reference script run in CI; missing agent = FAIL |

### 8-04: Publish Honest Release Artifacts

- Record benchmark report, debate transcripts, proof-token bundle, and limitations.
- Update README and docs to exact validated scope.

#### Reasoning

Release artifacts must be generated exclusively from pipeline outputs — never hand-written or hand-edited. This prevents cherry-picking favorable results, omitting limitation sections, or inflating metrics. Every artifact must trace to a pipeline run ID, and the limitations section is mandatory and non-removable. The 100%/100% rule applies: perfect metrics in any artifact trigger a fabrication investigation rather than celebration.

#### Expected Outputs

- `release/benchmark_report.json` — Generated by benchmark pipeline, includes precision/recall/F1 per vulnerability class, with pipeline run ID and checksums
- `release/debate_transcripts/` — Directory of raw debate transcripts (attacker/defender/verifier) from Phase 4, unedited, with file hashes
- `release/proof_tokens.jsonl` — Proof token bundle from `ProofTokenCollector`, each token with timestamp, graph hash, and evidence chain
- `release/limitations.md` — Auto-generated from pipeline gap analysis: unsupported vulnerability classes, known false-positive patterns, model dependency caveats
- `release/artifact_manifest.json` — Index of all release artifacts with SHA-256 checksums, pipeline run IDs, and generation timestamps
- Updated `README.md` — Claims section rewritten to match exact validated scope from `sweep_results.json`

#### Testing Strategy

| Output | Method 1 | Method 2 |
|--------|----------|----------|
| Benchmark report | Schema validation; precision/recall values are floats in [0,1] | Cross-reference with `sweep_results.json` G3 gate — numbers must match |
| Debate transcripts | Each transcript contains `[ATTACKER]`, `[DEFENDER]`, `[VERIFIER]` markers | File hashes match those recorded during Phase 4 runs |
| Proof tokens | Each token has required fields: `timestamp`, `graph_hash`, `evidence_chain` | `graph_hash` in tokens matches G2 graph hash from sweep |
| Limitations section present | `release/limitations.md` exists and has `> 0` bytes | Parse for required subsections: "Unsupported Vulnerability Classes", "Known False Positives", "Model Dependencies" |
| No hand-editing | Compare `artifact_manifest.json` checksums against actual file checksums | Git log for `release/` directory shows only automated commits (no manual edits post-generation) |
| 100%/100% check | Assert no metric in benchmark report equals exactly 1.0 for precision AND recall simultaneously | If triggered, generate `fabrication_investigation.json` with flagged metrics |

#### Drift Detection

| Expected Behavior | Bypass Indicators | Enforcement |
|--------------------|-------------------|-------------|
| Artifacts generated by pipeline, not hand-written | `release/` files have git author that is not the CI bot | Git log check: all files in `release/` committed by pipeline automation |
| Limitations section is mandatory and non-empty | `release/limitations.md` missing, empty, or contains only boilerplate | CI check: file exists, byte count > 200, contains required subsection headers |
| Benchmark numbers match pipeline outputs | `benchmark_report.json` metrics differ from `sweep_results.json` G3 evidence | Cross-validation script compares both files; mismatch = FAIL |
| No cherry-picked results | `benchmark_report.json` contains results for fewer contracts than the benchmark set | Count contracts in report vs contracts in benchmark input set; assert equal |
| Perfect metrics trigger investigation | `precision == 1.0 && recall == 1.0` for any category | Automated check in artifact validation; triggers `fabrication_investigation.json` generation |

### 8-05: Final Demo and Dossier

- Produce a reproducible `/vrs-audit` demo run.
- Build GA dossier with links to all evidence gates.

#### Reasoning

The final demo is the ultimate integration test — a real `/vrs-audit` execution against a real contract that exercises the full pipeline end-to-end. Reproducibility is non-negotiable: a second run on the same inputs must produce structurally equivalent outputs (same findings, same severity, same evidence chains). The GA dossier is the single document that links every gate (G0-G7) to its evidence artifacts, and every link must resolve — broken links mean the dossier is invalid.

#### Expected Outputs

- `demo/demo_transcript.log` — Full `/vrs-audit` transcript with all orchestration markers present
- `demo/demo_findings.json` — Structured findings output with evidence packets
- `demo/demo_checksums.json` — SHA-256 checksums of all demo outputs for reproducibility verification
- `demo/reproducibility_check.json` — Comparison of two independent runs: finding count, severity distribution, evidence chain overlap
- `ga_dossier.md` — Master document with links to all G0-G7 evidence artifacts, orchestration marker report, and rollout check results
- `ga_dossier_link_check.json` — Automated link validation report: every link in the dossier resolves to an existing file with matching checksum
- Release tag applied ONLY after `ga_dossier_link_check.json` shows zero broken links and `gate_validation_report.json` shows all gates passed

#### Testing Strategy

| Output | Method 1 | Method 2 |
|--------|----------|----------|
| Demo transcript | Transcript validator (Phase 3.1-04): all required markers present, duration > 5 seconds | Parse for `TaskCreate`, `TaskUpdate`, `[REPORT_GENERATED]` — all must appear |
| Demo findings | Schema validation against findings schema; every finding has evidence packet | Cross-reference finding graph node IDs against demo graph build artifact |
| Reproducibility | Run demo twice on same input; compare `demo_checksums.json` from both runs | `reproducibility_check.json` shows finding-level match rate >= defined threshold |
| GA dossier completeness | Parse `ga_dossier.md` for G0-G7 section headers; all 8 gates present | Every artifact link in dossier exists in `release/` or `demo/` directory |
| Link integrity | `ga_dossier_link_check.json` shows zero broken links | Manual spot-check: randomly select 3 links, verify file exists and checksum matches |
| Release tag timing | Git log shows tag created AFTER final `gate_validation_report.json` commit | Tag commit hash is a descendant of the report commit hash |

#### Drift Detection

| Expected Behavior | Bypass Indicators | Enforcement |
|--------------------|-------------------|-------------|
| Demo is a live `/vrs-audit` run, not scripted | Demo transcript lacks real tool execution times or shows suspiciously uniform timing | Duration bounds check: total demo time > 30 seconds; individual tool calls show variable timing |
| Reproducibility check actually re-runs the demo | `reproducibility_check.json` has identical timestamps for both runs (copy, not re-run) | Assert run-2 timestamp > run-1 timestamp + minimum expected duration |
| Release tag applied only after all gates pass | Tag exists but `gate_validation_report.json` has failing gates | Pre-tag hook: parse `gate_validation_report.json`, block tag if any gate status != "PASS" |
| GA dossier has no broken links | `ga_dossier_link_check.json` missing or not generated | CI job: generate link check report; block release if report absent or has broken links |
| Dossier is not hand-edited after generation | Git log shows manual commits to `ga_dossier.md` after pipeline generation | Checksum of `ga_dossier.md` recorded at generation time; pre-tag hook verifies checksum unchanged |

## Hard Delivery Kill-Switch Contract (Added 2026-02-10)

### HDG-12: Release Kill-Switch (Primary owner: 8-01, 8-02, 8-05)

**Why this is critical**
- A polished report with missing hard-gate evidence is worse than an explicit failure.
- The release path must fail loudly and return forensic artifacts instead of silently downgrading standards.

**Implementation contract**
- Introduce `hard_delivery_gate_report.json` generated by the pre-ship sweep:
  - includes status of HDG-01 through HDG-11
  - links each failed gate to blocking artifacts and rerun commands.
- Enforce kill-switch at two concrete control points:
  - **Pre-run gate (orchestrator entrypoint for `/vrs-audit`):** if any HDG in `hard_delivery_gate_report.json` is failed/missing, deny final audit publication.
  - **Final report schema gate:** if denied, output structured status `INCOMPLETE_HARD_GATE_FAILURE` with `failed_gate_ids`, `artifact_refs`, and `human_review_required=true`.
- Ban all bypass knobs (`--force-ship`, `SKIP_HARD_GATES`, manual overrides).
- Persist kill-switch events to `.vrs/debug/phase-8/kill-switch-events.jsonl`.

**Hard validation**
- Synthetic failure injection:
  - break one upstream gate artifact (e.g., missing exploit trace)
  - assert release sweep exits non-zero
  - assert `/vrs-audit` returns `INCOMPLETE_HARD_GATE_FAILURE` with correct gate ID.
- Manual-pass regression test:
  - set fake override env vars
  - assert kill-switch still blocks.
- Artifact integrity:
  - every failed gate entry must point to existing artifact path and checksum.

**Expected strict result**
- The system cannot ship a "successful" audit when hard evidence is missing.
- Failure mode is explicit, structured, and operator-actionable.

   - Freeze the GA gate contract (below) and publish it in `.planning/phases/8-ship-what-works/`.

## Release Blockers (Hard No-Go)

- Missing or invalid evidence for any gate in the GA contract.
- Any `/vrs-audit` run without required orchestration markers.
- Any finding labeled "confirmed" without full evidence packet.
- Any benchmark claim without reproducible scripts + outputs + checksums.
- Any debate claim without attacker/defender/verifier transcripts and graph queries.
- Any skill marked "shipped" that is not invocable or fails health checks.
- Any regression test drop vs the previous baseline without explicit acceptance note.
- Any failed or missing HDG-01..HDG-11 artifact in the Phase 8 hard delivery gate matrix.
- Any failed HDG-12 kill-switch validation (or missing kill-switch event/report artifacts).

## GA Gate Contract (Exact, Evidence-Required)

All gates below MUST pass for GA. Each gate requires evidence artifacts with IDs and
checksums. No "manual pass" allowed. If evidence is missing, the gate fails.

### Gate G0: Preflight (Environment + Tooling)

**Required Evidence**
- `preflight.json` with tool versions, CLI versions, model settings, dataset hashes.
- `[PREFLIGHT_PASS]` marker in transcript.
- `tools_status.json` from `alphaswarm tools status`.

**Pass Criteria**
- All required tools present, versions pinned or recorded.
- Dataset hashes match GA baseline.

### Gate G1: Evidence Integrity

**Required Evidence**
- Proof token bundle (`proof_tokens.jsonl` or equivalent) with timestamps.
- Evidence packet schema validation report (`evidence_schema_report.json`).

**Pass Criteria**
- 100% of findings have complete evidence packets:
  - graph node IDs
  - matched properties
  - operation sequences
  - source locations
- Proof tokens reference the same graph hash as G2.

### Gate G2: Graph Soundness

**Required Evidence**
- Graph build artifact (`graph_build.json` or `.vrs/graphs/*` metadata).
- `[GRAPH_BUILD_SUCCESS]` marker in transcript.
- Graph hash + node/edge counts.

**Pass Criteria**
- Graph hash stable for identical inputs.
- Node/edge counts within expected ranges for the audited contract set.

### Gate G3: Ground Truth Coverage

**Required Evidence**
- Benchmark output files (`benchmarks/RESULTS.md` + raw outputs).
- Coverage report (`coverage_report.json`) mapping findings to ground truth.

**Pass Criteria**
- Meets phase-defined coverage thresholds for the chosen benchmark set.
- No unreviewed "TODO" or placeholder ground truth entries.

### Gate G4: Mutation Robustness

**Required Evidence**
- Mutation test report (`mutation_report.json`) with seed and mutation list.
- Transcript markers for mutation runs.

**Pass Criteria**
- Mutation pass rate meets Phase 5 requirements.
- Failures are documented with accepted risk, or gate fails.

### Gate G5: Consensus & Variance

**Required Evidence**
- Multi-run consistency report (`consistency_report.json`) across k runs.
- Debate transcripts showing attacker/defender/verifier evidence usage.

**Pass Criteria**
- Consistency meets defined threshold (`pass^k`) for GA.
- Verdict variance within allowed tolerance.

### Gate G6: Regression Baseline

**Required Evidence**
- Regression test results (`pytest_report.xml` or `regression_report.json`).
- Baseline comparison artifact (`baseline_diff.json`).

**Pass Criteria**
- No regressions on GA baseline.
- Any allowed regressions explicitly documented in GA dossier.

### Gate G7: Continuous Health

**Required Evidence**
- Health check output (`health_check.json`) for shipped skills and tools.
- Trend snapshot (`trend_snapshot.json`) with last 3 runs.

**Pass Criteria**
- Health checks pass for all shipped skills.
- No negative trend on critical metrics.

### Orchestration Marker Contract (Required for All GA Audits)

**Must appear in transcript for each GA audit run:**
- `[PREFLIGHT_PASS]`
- `[GRAPH_BUILD_SUCCESS]`
- `[CONTEXT_READY]`
- `[TOOLS_COMPLETE]`
- `[DETECTION_COMPLETE]`
- `TaskCreate(task-id)`
- `TaskUpdate(task-id, verdict)`
- `[REPORT_GENERATED]`
- `[PROGRESS_SAVED]`

**Evidence**
- Full transcript with markers and a marker coverage report (`markers_report.json`).

## Rollout Checks (Post-Tag)

1. **Reproducibility Check**
   - Re-run `/vrs-audit` on the GA demo target in a clean workspace.
   - Compare outputs to GA artifacts via checksum.
2. **Invocability Check**
   - Run `/vrs-health-check` and `/vrs-audit` from a fresh user repo.
   - Confirm shipped skills load and execute without manual config edits.
3. **Evidence Traceability Check**
   - Randomly sample 3 findings and verify evidence packets trace to graph nodes and code.
4. **Docs Honesty Check**
   - Re-scan README and docs for claims not backed by GA evidence.
5. **Rollback Trigger Definition**
   - If any post-tag check fails, revert GA label and mark release as "candidate-only."
