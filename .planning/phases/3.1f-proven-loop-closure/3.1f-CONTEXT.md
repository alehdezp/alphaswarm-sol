<plans>
## Mini-Plans

### Plan 01: Failure Classifier + Gap Log Wiring [MEDIUM confidence]
**Intent:** Classify evaluation outcomes using the taxonomy discovered in 3.1e-04 and wire gap logging into EvaluationRunner.
**What it delivers:** Failure classifier (~80 LOC), gap log writer (~30 LOC), EvaluationRunner integration (~40 LOC), ~150 LOC total.
**What problem it solves:** Without classification, evaluation results are raw scores with no actionable categorization. The gap log creates a persistent record of genuine gaps that feed improvement proposals.
**Confidence:** MEDIUM -- depends on 3.1e-04 taxonomy output and 3.1e-05 schema format. Implementation is straightforward once inputs are known.

### Plan 02: Variant Tester [MEDIUM confidence]
**Intent:** Enable A/B testing of prompt variants using git branch isolation and BaselineManager comparison.
**What it delivers:** Thin wrapper over BaselineManager.check_batch() (~100 LOC), git branch management (~30 LOC), VariantReport model (~20 LOC), ~150 LOC total.
**What problem it solves:** No way to measure whether a prompt change actually improves evaluation outcomes. The variant tester provides controlled comparison: run same scenarios on old prompt vs new prompt, compare scores.
**Confidence:** MEDIUM -- BaselineManager API needs verification. Git branch isolation approach is simple and proven.

### Plan 03: E2E Proof + Goodhart Mitigation [MEDIUM confidence]
**Intent:** Prove the full improvement cycle works end-to-end and add Goodhart mitigation calibrated to 3.1e-03 findings.
**What it delivers:** If Goodhart risk proven: automated check (~45 LOC). If LLM resists gaming: human review guidance only. Improvement proposal YAML template (~40 lines). Full cycle proof documentation. ~100 LOC total.
**What problem it solves:** Proves the entire thesis: detect -> evaluate -> classify -> propose -> test variant -> verify improvement. Without this proof, the improvement loop is theoretical. Goodhart mitigation ensures the evaluator remains trustworthy.
**Confidence:** MEDIUM -- depends on 3.1e-03 results and successful execution of Plans 01-02. The E2E proof is the capstone.
