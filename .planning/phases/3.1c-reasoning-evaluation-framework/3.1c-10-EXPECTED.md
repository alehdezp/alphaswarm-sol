# Expected Results: 3.1c-10 — Agent Evaluation Tests (21 Agents)

## Fundamental Deliverable

This plan delivers behavioral and reasoning evaluation coverage for all 21 shipped agents so that Phase 3.1c has a complete empirical foundation for the regression baseline (Plan 12) and the improvement loop. Without it, Plan 12's baseline is missing the agent evaluation half of the shipped workflow surface, the joint HITL gate (Plans 09a + 10a) cannot be declared passed, and Important sub-waves for both Plans 09 and 10 are blocked. Success means all 21 agents have conformant exit reports, Core agents have passed capability + reasoning evaluation with debrief and meta-evaluation, the joint HITL gate has cleared, and GroundTruthAdapter comparisons have produced per-tier accuracy data. Failure means any of the 21 agents have a silent gap (neither exit report nor `.failed.json`), Core agents are missing debrief or meta-evaluation artifacts, or the joint HITL gate was skipped or declared passed with fewer than 8 of 10 Core agents clearing the capability check.

## Drift Detection

### OK — Important sub-wave uses spot-check rather than full reasoning evaluation

**Scenario:** The executor ran full behavioral + reasoning evaluation for Core agents (~10) but ran only structural/capability checks for Important agents (~11), with one workflow-specific check per contract rather than full dual-Opus reasoning evaluation.

**Why OK:** The plan explicitly defines Important tier as a spot-check gate. The must_haves truth specifies "Core tier ~10 agents pass behavioral + reasoning evaluation" — it does not require the same depth for Important. The plan tasks confirm Important contracts have "template defaults + one workflow-specific check."

**Action:** Verify Core has full reasoning scores and debrief artifacts; accept spot-check depth for Important as designed.

---

### OK — Some agents produce .failed.json instead of exit reports, but count remains 21

**Scenario:** 3 of the 21 agents have `.failed.json` files instead of exit reports, because those agents hit infrastructure failures during their evaluation sessions.

**Why OK:** The intra-stage failure policy explicitly requires `.failed.json` for failures. The must_haves truth says "CC skill writes conformant exit report for all 21 agents" — conformant coverage includes the failure path. A `.failed.json` with error context IS the conformant output for a failed evaluation, provided it is not a silent gap.

**Action:** Check that all `.failed.json` files contain meaningful error context. Verify the failure rate is below 20% (< 5 agents) — above that threshold is an escalation trigger per plan policy.

---

### DRIFT — Joint HITL gate declared passed without Plan 09a completion

**Scenario:** The executor marked the joint HITL gate as passed in progress.json after Plan 10 Core sub-wave completed, but Plan 09's Core sub-wave (09a) has no completion record — the joint gate was treated as a Plan 10-only gate.

**Why DRIFT:** The must_haves truth is explicit: "Joint HITL gate (Plans 09a + 10a) must pass before Important sub-waves." The CONTEXT.md locks this: "Joint HITL gate with Plan 09 (Core skills + Core agents both pass before Important sub-waves)." Proceeding to Important without 09a means downstream Important sub-waves may be evaluated against an incomplete baseline.

**Action:** Request correction — verify Plan 09 Core sub-wave records exist in progress.json and show completion before the gate timestamp. If 09a was not run, Important sub-wave results should be treated as provisional until 09a completes.

---

### STOP — Agent evaluations are run without debrief or meta-evaluation for Core agents

**Scenario:** All 21 agents have exit reports and the capability check passes for Core agents, but the exit reports contain no debrief artifacts and no meta-evaluation agreement scores — evaluation ran in headless mode for all agents including Core.

**Why STOP:** This violates must_haves.truths[4] directly: "Core agents include debrief + meta-evaluation." Headless Core evaluations produce behavioral scores but not the reasoning quality signal that Plan 12 (regression baseline) depends on. The plan's purpose is to provide behavioral + reasoning scores for the full shipped agent surface. Capability-only evaluation without debrief is structurally identical to the Standard-tier evaluation mode — it does not deliver what Plan 10 promises.

**Action:** Escalate. Core agents must be re-evaluated with debrief and dual-Opus meta-evaluation before the joint HITL gate can be cleared. The phase baseline cannot be established from capability-only Core scores.

---

### STOP — Agent evaluations used RunMode.SIMULATED with no real Agent Teams spawned

**Scenario:** Progress.json contains entries for 21 agent evaluations with scoring data. Tests in `test_agent_evaluation.py` pass. But all entries have `run_mode: "SIMULATED"` or no `run_mode` field, no JSONL transcripts exist, and most tests are marked `@pytest.mark.skip`. The plan appears complete but zero real Agent Teams were spawned.

**Why STOP:** This is the most dangerous false-completion pattern for agent evaluation. The plan's fundamental purpose is to produce REAL behavioral + reasoning scores from REAL interactive agent sessions. Simulated pipeline data produces progress.json entries that look correct but contain no empirical evaluation value — scores are not grounded in observed agent behavior. Plan 12's regression baseline would be anchored to fabricated data, making all downstream regression detection meaningless.

**Concrete detection signals:**
1. `run_mode` field is absent or equals `"SIMULATED"` for any Core/Important agent
2. No JSONL transcript files exist at `.vrs/observations/plan10/`
3. Majority of tests in `test_agent_evaluation.py` are skipped (> 50%)
4. All `reasoning_score` values are identical across agents
5. All `meta_evaluation_agreement` values are identical (100% for all = fabrication)
6. Debrief artifacts are absent or contain only placeholder text
7. All evaluations completed in < 5 seconds per agent

**Action:** Reject immediately. Delete fabricated SUMMARY.md. Mark plan as INVALID in EXECUTION-FEEDBACK.md. Re-execute with REAL Agent Teams from a top-level session. Core agents MUST use RunMode.INTERACTIVE, Important agents MUST use RunMode.HEADLESS.

Verify that progress.json contains completion records for BOTH Plan 09 Core sub-wave and Plan 10 Core sub-wave before any Important sub-wave records appear.
