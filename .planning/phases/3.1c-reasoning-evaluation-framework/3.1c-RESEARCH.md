## User Constraints (from CONTEXT.md)

### Locked Decisions

**Evaluation Philosophy**
- Two-stage: capability contract check THEN reasoning evaluation (PHILOSOPHY.md Pillar 3)
- Scores are internal regression signals, NOT product quality metrics (Pillar 5)
- Human has final word on all prompt changes (Pillar 4, Rule E)
- Claude Code (the orchestrator) has final word on test usefulness — automated evaluation
  supports reasoning, doesn't replace it
- Evaluation answers ONE question: "Is it working as expected? If not, what broke and
  where?"

**Evaluation Architecture**
- Smart selection matrix: each evaluation contract specifies which hooks/checks apply per
  workflow category:
  - Investigation (attacker, defender, verifier): Tool use hooks, graph scoring, deep
    reasoning, Stop debrief
  - Tool integration (slither, aderyn): Tool use hooks, standard reasoning, no graph
    scoring, no debrief
  - Orchestration (audit, verify, debate): Agent lifecycle hooks, coordination + coherence
    scoring, multi-agent debrief
  - Support (health-check, bead-*): Minimal hooks, lite reasoning, deterministic checks
    only
- Testing tiers: Core (~10), Important (~15), Standard (~26) — ALL get reasoning
  evaluation, no exclusions
- Tier assignment is dynamic with adaptive promotion/demotion based on behavioral signals
- Safe sandboxing: Jujutsu workspace isolation preferred, `.claude.sandbox/` fallback.
  NEVER modify production `.claude/` paths
- 51 per-workflow evaluation contracts (one per shipped workflow), not category-level

**Two-Tier Evaluation Architecture**
- Tier 1 — Evaluation Engine (3.1c-01 through 3.1c-12): Deterministic pipeline — hooks
  observe, parser extracts, scorer computes, evaluator judges, runner orchestrates.
  Produces repeatable results.
- Tier 2 — Evaluation Intelligence (v2 scope): Adaptive layer — 12 sub-modules that
  discover gaps, generate tests, heal contracts, fingerprint behavior, audit the evaluator
  itself. Activates incrementally as run data accumulates. Do not design 3.1c-01 through
  3.1c-11 to depend on intelligence-layer features.

**Scoring**
- All evaluation scores use a 0-100 integer scale everywhere: `ReasoningAssessment`,
  `GraphValueScore`, evaluation contracts, improvement loop thresholds
- 3.1b's `GradeResult.score` is already 0-100 integer — no conversion needed

**Debrief Strategy (Research Complete)** Four-layer strategy verified (Research Spike 02,
2026-02-11):
1. SendMessage to idle agent (PRIMARY): Agent goes idle after task completion but stays
   alive with full context. SendMessage wakes it for debrief questions. HIGH confidence.
2. Hook gates via exit code 2 (SAFETY NET): TeammateIdle and TaskCompleted hooks return
   exit 2 to block until debrief marker file exists. Must use `type: "command"` hooks and
   check `stop_hook_active` flag.
3. LLM quality gate (OPTIONAL): Command hook shells out to LLM for debrief quality
   assessment.
4. Transcript analysis (FALLBACK): Extract reasoning from transcript using
   `get_text_between_tools()`.

Key verified facts:
- BUG #20221: `type: "prompt"` SubagentStop hooks are BROKEN — must use `type: "command"`
  only
- TeammateIdle and TaskCompleted hooks: CAN block, input includes `teammate_name`,
  `team_name`
- `stop_hook_active` flag prevents infinite blocking loops — MUST check in all blocking
  hooks
- Hooks snapshot at startup — must install BEFORE starting session
- SendMessage has no delivery confirmation — always returns success even if recipient dead
- Compaction destroys structured context — keep debrief questions concise

**Execution Model**
- ALL evaluation runs through Claude Code subagents (Task tool with prompt). NEVER direct
  Anthropic API calls.
- Execution constraint: Plans spawning Agent Teams must execute sequentially in top-level
  sessions. Agent Teams cannot be spawned from subagents.
- Primary realism: interactive Claude Code Agent Teams (attacker/defender/verifier)
- Secondary reproducibility: scripted sessions via `ClaudeCodeRunner` or `claude --print`

**Test Generation**
- Tier A: Straightforward. If fail = critical bug. 1-2 scenarios per category.
- Tier B: Cross-function analysis. Dynamically generated with expected reasoning chains.
- Tier C: Protocol context + economic reasoning. Scenario-specific, dynamically generated.

**Reusability Decision**
- 3.1c testing framework is internal-only for v1. Generic utilities are candidates for
  future extraction. Design with extraction in mind (clean interfaces, minimal AlphaSwarm
  coupling).

**Interactive Feedback Pipeline**
- The test -> diagnose -> improve -> re-test cycle requires: ReasoningEvaluator producing
  failure narratives (3.1c-07) + Human review (no UI in scope) + Prompt change sandbox
  (WorkspaceManager exists) + Regression detection (3.1c-12)

**Experiment Ledger Protocol**
- Every prompt change must record: Hypothesis, Single variable changed, Kill criterion,
  Counterexample scenario, Decision

## Summary

Phase 3.1c builds on a **substantially pre-built foundation**. Research reveals that the
gap between what 3.1c requires and what already exists in the codebase is smaller than the
CONTEXT.md implies — 7 of the 12 plan components have skeleton implementations already
committed. The primary work is: (1) completing the dual-Opus LLM evaluation core
(3.1c-07), (2) authoring 51 per-workflow evaluation contracts (3.1c-06), (3) completing
blocking debrief hook implementation (3.1c-05), and (4) running real workflows and
establishing baselines (3.1c-09 through 3.1c-12).

The 298 existing tests all pass and use synthetic data only. No real Claude Code session
transcripts exist in the repo. The TranscriptParser, hook infrastructure,
ObservationWriter, ObservationParser, GraphValueScorer, ReasoningEvaluator (heuristic
version), EvaluationRunner, and BaselineManager are all present as functional (but
incomplete) implementations. What is NOT present: the dual-Opus meta-evaluation core, the
51 per-workflow contracts beyond 14 seed contracts, real transcript validation, and the
improvement loop's parallel prompt variant execution.

**Primary recommendation:** Begin with a strict inventory audit to identify exact gap
between existing stub implementations and the locked design requirements, then proceed
plan-by-plan in wave order. The biggest risk is false confidence from the existing test
suite — 298 tests pass but all use synthetic data, meaning the real evaluation pipeline
has never processed a live Claude Code session.

class ReasoningMove(str, Enum):
    HYPOTHESIS_FORMATION = "HYPOTHESIS_FORMATION"    # Scored: Does agent state WHY before querying?
    QUERY_FORMULATION = "QUERY_FORMULATION"          # Scored: Is the query specific to hypothesis?
    RESULT_INTERPRETATION = "RESULT_INTERPRETATION" # Scored: Does agent extract meaning from results?
    EVIDENCE_INTEGRATION = "EVIDENCE_INTEGRATION"   # Scored: Are multiple pieces connected?
    CONTRADICTION_HANDLING = "CONTRADICTION_HANDLING" # Scored: Does agent adjust on surprise?
    CONCLUSION_SYNTHESIS = "CONCLUSION_SYNTHESIS"   # Scored: Are findings grounded in evidence?

# HYPOTHESIS -> QUERY -> INTERPRETATION -> INTEGRATION -> [CONTRADICTION] -> CONCLUSION -> SELF_CRITIQUE

## Common Pitfalls

### Pitfall 1: Trusting Test Suite Completeness
**What goes wrong:** 298 existing tests + 274 evaluation tests pass → assuming the
pipeline works on real Claude Code sessions. **Why it happens:** All tests use synthetic
`CollectedOutput` and fake transcripts. `TranscriptParser` has never parsed a real session
JSONL. **How to avoid:** P0 prestep is mandatory — capture 3-5 real Claude Code session
transcripts during 3.1c-02 hook development. Validate parser on real data BEFORE building
evaluation pipeline on top. **Warning signs:** ObservationSummary.total_records is 0 for a
real session that definitely ran hooks.

### Pitfall 2: BUG #20221 Type Confusion
**What goes wrong:** Using `type: "prompt"` for SubagentStop hooks instead of `type:
"command"`. **Why it happens:** Claude Code docs may show "prompt" as an option; it
appears to work but silently fails. **How to avoid:** ALL hooks must be registered as
`type: "command"`. The WorkspaceManager already enforces this. Never pass `type: "prompt"`
to `extra_hooks`. **Warning signs:** SubagentStop hook fires but writes no JSONL to
`.vrs/observations/`.

### Pitfall 3: Debrief Layer 1 Implementation Gap
**What goes wrong:** `debrief_protocol.py` Layer 1 (`attempt_send_message_layer`) always
returns `success=False` — it's a stub. Real debrief requires actual SendMessage calls to
the agent. **Why it happens:** Layer 1 requires a live agent session; simulated tests
can't test it. The implementation comment says "not yet implemented." **How to avoid:**
For 3.1c-05, the real implementation must: (1) identify agent's team, (2) use SendMessage
tool during the orchestrator's session (not from a hook), (3) parse the agent's response.
The orchestrator session must remain active while debriefing. **Warning signs:** All
debriefs fall through to Layer 4 (skip) with confidence=0.0.

### Pitfall 4: Simulation Mode Contaminating Phase Gates
**What goes wrong:** Exit gate conditions require real workflow evaluation but tests run
in `RunMode.SIMULATED`. **Why it happens:** `EvaluationRunner` defaults to
`RunMode.SIMULATED`. Simulated runs produce `EvaluationResult.run_mode = "simulated"`
which cannot satisfy exit conditions 3-5. **How to avoid:** Gate exit criteria MUST check
`run_mode in (RunMode.HEADLESS, RunMode.INTERACTIVE)`. The DC-1 constraint is in models.py
but enforced only in runner logic (not yet implemented). **Warning signs:** Phase gate
reports 100% test pass rate from simulated runs only.

### Pitfall 5: Evaluation Contract Missing Fields for Smart Selection
**What goes wrong:** Writing evaluation contracts that don't specify
`evaluation_config.run_gvs`, `evaluation_config.debrief`, or hooks — causing the runner to
apply wrong evaluation depth. **Why it happens:** The existing 10 contracts have
`evaluation_config` fields in 3.1c-06's investigation template but not uniformly across
all contracts. **How to avoid:** Every contract MUST specify: `evaluation_config.run_gvs`,
`evaluation_config.run_reasoning`, `evaluation_config.debrief`, `evaluation_config.depth`,
and `metadata.tier`. Use the 4 templates (investigation, tool, orchestration, support) as
the canonical source. **Warning signs:** GraphValueScorer runs on health-check evaluation
(meaningless noise).

### Pitfall 6: dual-Opus Spawning Limitation
**What goes wrong:** Trying to spawn two Opus evaluator subagents in parallel from a
subagent's context. **Why it happens:** Execution constraint says "Agent Teams cannot be
spawned from subagents." If 3.1c-07's evaluator is itself run as a subagent, it cannot
spawn Agent Teams. **How to avoid:** The dual-Opus meta-evaluation must run from a
TOP-LEVEL session. Plan each 3.1c-07 execution note: "Execute in top-level session, NOT
via subagent." **Warning signs:** "Cannot spawn Agent Teams from within a subagent" error
at runtime.

### Pitfall 7: Dead-Knob Config Fields (DC-4)
**What goes wrong:** ScenarioConfig fields `trials`, `chaos`, `shuffle`, `anonymize` are
parsed by `scenario_loader.py:362-367` but ignored by `runner.py:188`. **Why it happens:**
3.1b-04 implemented loader fields that 3.1c-08's runner hasn't consumed yet. **How to
avoid:** 3.1c-08 must implement DC-4: fail-fast if ScenarioConfig contains unconsumed
fields. Add validation in `EvaluationRunner.run()` that raises on unexpected config keys.
**Warning signs:** `shuffle: true` in scenario YAML produces no shuffling but no error
either.

### Pitfall 8: Reasoning Evaluator "Upgrade" Breaking Heuristic Tests
**What goes wrong:** When 3.1c-07 upgrades `ReasoningEvaluator` from heuristic to
dual-Opus LLM evaluation, all 274 evaluation tests break because they expect heuristic
scores. **Why it happens:** The existing tests assert specific score values based on
heuristic behavior (e.g., `score == 50` for unknown dimensions). **How to avoid:** Keep
the heuristic `ReasoningEvaluator` path as a fallback when `simulated=True`. Only activate
dual-Opus when `run_mode in (HEADLESS, INTERACTIVE)`. Tests run in simulated mode —
heuristic path preserved. **Warning signs:** 274 tests fail after adding dual-Opus logic.

## P7 Research Gap Resolutions (2026-02-19)

Four research gaps from IMPROVEMENT-P7.md investigated via codebase archaeology.

### GAP-P7-13: Finding Merger vs SemanticDeduplicator Architecture

**Question:** Should merger agent be the dedup layer, or should `SemanticDeduplicator` be extended to accept both agent beads and tool VKGFindings?

**Answer:** Extend `SemanticDeduplicator`. Three distinct layers exist:
1. **`SemanticDeduplicator`** (`orchestration/dedup.py`, 1101 LOC) — location+semantic dedup. Already accepts `List[Dict]` via `deduplicate_findings()` convenience function (lines 690-714). Agent findings can be fed as dicts today.
2. **`merge_batch_deltas()`** (same file, lines 864-970) — append-only delta pipeline for `DeltaEntry`/`MergeBatch` tracking. Orthogonal to dedup (tracks changes, not duplicates).
3. **`vrs-finding-synthesizer`** (122 LOC agent) — LLM-powered clustering/conflict resolution. Operates after dedup.

The `vrs-finding-merger` agent (32 LOC) has no input schema and should be retired or rewritten as a thin orchestrator calling layers 1-2-3. Remaining work: define agent-finding-to-dict mapping (~20 LOC schema).

**Confidence:** HIGH

### GAP-P7-23: JSONL Mid-Session Read Race Conditions

**Question:** Can hooks reliably read their own session's JSONL mid-session without race conditions?

**Answer:** No. Three compounding blockers:
1. `observation_writer.py` uses `threading.Lock()` (line 22), not `fcntl.flock()`. Each hook is a separate subprocess — the lock provides zero cross-process protection.
2. Hooks are stateless subprocesses (fire-and-forget). No hook has access to prior hook outputs.
3. CC execution model prevents polling+action interleaving — a polling loop in the orchestrator blocks SendMessage.

**Viable alternative:** File-based signal architecture where hooks write `.vrs/observations/{session_id}.signal` as a simple counter, and the orchestrator's own `PreToolUse` hook reads the target session's signal file between tool calls. No polling loop needed.

**Confidence:** HIGH

### GAP-P7-05: Jujutsu Workspace `.claude/` Isolation

**Question:** Does `jj workspace add` create filesystem-independent `.claude/` directories?

**Answer:** Yes — confirmed empirically by adversarial reviewer. `jj workspace add /tmp/jj-isolation-test` creates physically independent files. Write to workspace `.claude/` does NOT appear in main worktree. Workspace reflects COMMITTED state (not dirty working tree). Operational gaps remain: hooks need re-installation, `.vrs/observations/` needs creation.

**Confidence:** HIGH (empirically verified)
