## Fundamental Deliverable

This plan delivers orchestrator evaluation infrastructure so that the multi-agent system (attacker + defender + verifier) can be verified as a coherent whole — not merely as three independently functioning agents. Without it, Plan 12 cannot establish a regression baseline for orchestration workflows, and Phase 3.2 (First Working Audit) proceeds without any empirical evidence that the full 3-agent pipeline preserves evidence integrity across handoffs. Success means: at least one Core orchestrator test exercises the complete pipeline (spawn through shutdown), a `fidelity_score` is produced for the attacker-to-verifier handoff edge, and cross-agent coherence scoring generates populated scores. Failure means the pytest wrappers pass but none produce non-None `fidelity_score` values, or the lifecycle coverage stops at individual agent capability without measuring cross-agent coordination.

## Drift Detection

### OK — Fidelity threshold produces WARN instead of PASS for some Core runs

**Scenario:** The executor implements `fidelity_score` correctly. For 2 of 3 Core orchestrator contracts, `fidelity_score` is 0.45 (below the 0.6 threshold). The tests emit WARN rather than FAIL for these contracts, and the Core HITL checkpoint documents the below-threshold scores.

**Why OK:** The plan explicitly specifies `>= 0.6` as a WARN threshold, not a hard FAIL. Evidence degrades in text serialization; the metric's purpose is to surface degradation, not block execution. The fidelity field is non-None (the metric is operational), which satisfies must_haves.truths[2]. WARN outcomes with documented scores are the expected behavior when real agent communication is observed.

**Action:** Verify that the threshold direction is correct (0.6 triggers WARN, not FAIL), and that the HITL checkpoint records the scores. No correction needed.

---

### DRIFT — Coherence scoring implemented as per-agent score average

**Scenario:** The executor adds a `coherence_score` field to the exit report schema. Inspection of the computation shows it is calculated as `(attacker_score + defender_score + verifier_score) / 3`. The lifecycle phases (spawn, assign, DM exchange, shutdown) are not individually measured.

**Why DRIFT:** The plan requires coherence scoring across the full lifecycle — spawn success, task assignment correctness, DM delivery, and shutdown. Averaging three independent agent reasoning scores produces a number but measures individual reasoning quality, not inter-agent coordination. This satisfies the structural check (non-None coherence field) but misses the intent: measuring whether the team functions as a unit. must_haves.truths[0] and [1] are both violated in substance.

**Action:** Request correction. The coherence scorer must be rewritten to operate on lifecycle phase events extracted from the orchestrator exit report, not on per-agent score aggregation.

---

### DRIFT — Prestep P1 ground truth templates exist but contain only placeholder tuples

**Scenario:** Ground truth template files for multi-agent contracts exist on disk. The evidence fidelity metric is implemented. However, the ground truth templates contain placeholder tuples like `("function_name", "reentrancy", "high")` rather than real expected attacker findings derived from known-vulnerable contracts.

**Why DRIFT:** The plan requires Prestep P1 completed before execution, and the co-design requirement means the metric spec must inform template content. Placeholder tuples make `fidelity_score` comparisons circular — any regex extraction will match trivially. The metric produces non-None scores (satisfying the letter of must_haves.truths[2]) but the scores measure nothing real.

**Action:** Flag at HITL checkpoint. Prestep P1 templates must be populated with real expected tuples from the orchestration ground truth contracts before the scores can be interpreted. Mark this as a prestep quality gap, not a plan execution failure.

### STOP — fidelity_score is added to models.py rather than output_collector.py

**Scenario:** The executor implements `EvidenceFlowEdge.fidelity_score` in `tests/workflow_harness/lib/models.py` (the Plan 01 module) rather than in `tests/workflow_harness/lib/output_collector.py` (the 3.1b dataclass). Tests pass because the field exists and assertions find it via import.

**Why STOP:** The plan explicitly constrains the implementation location: "3.1b dataclass, NOT Plan 01 models.py." This is not a style preference — it reflects an architectural boundary between the evaluation models layer and the observation/collection layer. Placing `fidelity_score` in `models.py` couples evidence fidelity tracking to the evaluation result model rather than the observation collection layer where cross-agent data flows are tracked. The key_link wiring (must_haves.key_links[0]) is severed — `output_collector.py` no longer participates in the data flow.

**Action:** Stop. Escalate to plan author. The field must be relocated to `output_collector.py` and the key_link must be verified before proceeding to Plan 12.
