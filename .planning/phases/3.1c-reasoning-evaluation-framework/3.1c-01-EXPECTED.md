## Fundamental Deliverable

This plan delivers the shared Pydantic type vocabulary for the entire evaluation framework — ~30+ models covering assessment, scoring, calibration, baseline, failure, and progress tracking — so that every downstream plan (02–12) has a stable, validated, import-safe foundation to build on. Without it, no other plan can instantiate evaluation results, write conformant exit reports, or enforce the DC-2/DC-3 architectural contracts that separate evaluation logic from domain code. Success means all v1 types are importable, the EvaluationPlugin protocol carries the `context` kwarg, `EvaluatorConstants.current()` is the single source of truth for evaluator config, `models.py` has zero imports from `alphaswarm.kg` or `alphaswarm.vulndocs`, and the `/vrs-test-suite` SKILL.md stub exists with `forced-eval: true` and delegation rules. Failure means downstream plans instantiate models that do not exist or raise import errors, evaluation results contain silent DC-2 contamination from domain code, or the skill stub is absent — leaving Plan 09 unable to fill in the `[TO-FILL: Plan 09]` markers.

## Drift Detection

### OK — Field stability tags applied as comments rather than type annotations
**Scenario:** The executor used Python comments (`# Track B: pending-real-transcript-validation`) instead of docstring annotations or a field metadata dict to tag Track B and Track C fields, because Python type annotations cannot carry freeform string tags without changing the runtime type.
**Why OK:** The must_haves require that stability tiers be *applied* — they do not prescribe the exact mechanism. Comments in the model definition are visible, readable by grep, and do not alter Pydantic's behavior. The substance is present.
**Action:** Verify the tags are present by running `grep -n "pending-real-transcript-validation\|structural-proxy" src/alphaswarm_sol/testing/evaluation/models.py` — if both track labels appear, classification is correct.

### OK — intelligence/ stubs contain only `pass` bodies with type signatures
**Scenario:** `tier_manager.py`, `contract_healer.py`, and `coverage_radar.py` each contain only a function signature for `is_active()` with a `pass` body and no logic, because Plan 01's scope is stubs for Wave 2+ to fill.
**Why OK:** Plan 01 explicitly scopes intelligence modules as stubs. A stub with the correct signature and `is_active(store: EvaluationStoreProtocol) -> bool` return type is the correct Wave 1 deliverable. Non-stub logic in Wave 1 would be out-of-scope scope creep.
**Action:** Confirm the stubs export `is_active` with the correct signature. No logic is expected or desired at this stage.

### DRIFT — EvaluatorConstants.current() hard-codes the prompt hash internally
**Scenario:** The executor implemented `EvaluatorConstants.current()` to import `REASONING_PROMPT_TEMPLATE` from `reasoning_evaluator.py` and compute `hashlib.sha256(...)` internally, so callers do not need to pass a hash. This makes the API simpler but violates P14-IMP-06.
**Why DRIFT:** The locked decision explicitly states the *caller* computes SHA256 and passes it as `prompt_template_hash: str`. Internal hash computation creates a hidden import from `reasoning_evaluator.py` into `models.py`, defeating DC-2's principle (models must not depend on evaluator implementation). This is not a cosmetic difference — it breaks the single-source-of-truth contract and makes Plan 12's calibration comparison fragile.
**Action:** Request correction. `models.py` must accept `prompt_template_hash` as a parameter. Remove the internal import from `reasoning_evaluator.py`. Verify with `grep "reasoning_evaluator" src/alphaswarm_sol/testing/evaluation/models.py` — expect zero matches.

### DRIFT — SKILL.md stub is complete instead of stub (all [TO-FILL] markers resolved)
**Scenario:** The executor filled in all Plan 09 content (budget ceilings, kill-file mechanism, circuit breaker, sub-wave sequencing) in the Wave 1 stub, reasoning that "more complete is better" and Plan 09 might not need to touch the file.
**Why DRIFT:** Plan 01's scope is explicitly the stub with `[TO-FILL: Plan 09]` markers. A prematurely complete SKILL.md is not verified against Plan 09's actual requirements (Gate 0 dry-run, pre-flight activation logic, intra-stage failure policy). Plan 09 may need to revise content written without the context Plan 08 provides. The cross-plan dependency between Plan 01 (stub) and Plan 09 (complete) is a deliberate wave-boundary contract.
**Action:** Check that at least the Plan 09-scoped content (per-workflow budget, kill-file, circuit breaker, sub-wave ordering) still has `[TO-FILL: Plan 09]` markers. If the executor filled them with content, flag for human review — the reviewer must assess whether the content matches Plan 09's verified requirements.

**Scenario:** The executor created `models.py` with 30+ class definitions that each contain only `pass` or a single `...` placeholder, treating the file as a schema outline rather than a working implementation. All tests pass because tests only check `isinstance()` against empty classes.

Every plan that depends on `EvaluationResult`, `ReasoningAssessment`, or `CalibrationConfig` would instantiate silently-invalid objects. The plan's fundamental deliverable — a working type foundation — was not delivered.

A STOP condition here blocks the entire 3.1c wave sequence.

## Reviewer Guidance

**Priorities:**

1. **DC-2 boundary is the highest-stakes criterion.** A single `from alphaswarm_sol.kg` import in `models.py` would contaminate every downstream plan that imports from the evaluation module. Verify with a direct grep before accepting any other results.

2. **EvaluatorConstants.current() import chain matters more than its API shape.** The method signature can be adjusted in later plans, but if `models.py` imports from `reasoning_evaluator.py` at Wave 1, the circular dependency propagates to Plans 07 and 12 where it becomes much harder to untangle. Check `grep "reasoning_evaluator" src/alphaswarm_sol/testing/evaluation/models.py` explicitly.

3. **The SKILL.md stub is a contract between Plan 01 and Plan 09 — verify it is a stub, not a complete implementation.** `[TO-FILL: Plan 09]` markers must be present for the Plan 09-owned content. If the markers are gone, assess whether the filled content is correct given Plan 09's verified requirements — this is the primary cross-plan wiring risk.

4. **`effective_score()` returning `None` on all-N/A is a specific edge case the plan calls out explicitly.** Verify `tests/evaluation/test_models.py` contains the all-N/A test case (not just the normal case). This edge case matters for Plan 08's pipeline behavior when a workflow has no applicable scoring dimensions.

5. **Model completeness: verify category coverage, not just count.** The plan specifies ~30+ models across 8 categories (Core Assessment, Calibration, Infrastructure, Detection, Progress, Scoring updates, Baseline, Debrief updates). A reviewer should spot-check at least one model per category rather than accepting a total count. An executor could satisfy "30 models" with 30 scoring sub-models and zero calibration models.
