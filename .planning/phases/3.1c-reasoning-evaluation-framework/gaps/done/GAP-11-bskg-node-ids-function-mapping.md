# GAP-11: BSKG Query Results Node ID to Function Name Mapping

**Created by:** improve-phase
**Source:** P7-IMP-14
**Priority:** HIGH
**Status:** resolved
**depends_on:** []

## Question

Do BSKG query results include node IDs that are mappable to function names? Specifically: when `alphaswarm query` returns results, does the output include structured node identifiers (not just text) that can be cross-referenced against ground truth entries containing `contract`, `function`, and `line_range` fields?

## Context

P7-IMP-14 proposes a "Graph-Reasoning Coherence" cross-scorer that cross-references cited node IDs against ground truth function/contract names. This requires BSKG query results to contain structured identifiers, not just freeform text. If node IDs are not present or not mappable, the entire cross-scorer design must change.

Affects: Plan 04 (GraphValueScorer +50 LOC), Plan 06 (investigation contracts), Plan 02 (must capture result node IDs).

## Research Approach

- Read the BSKG builder source code (`src/alphaswarm_sol/kg/builder/`) to understand node schema
- Run `uv run alphaswarm build-kg` on a test contract and examine output format
- Run `uv run alphaswarm query` to see result format
- Check if TOON format output includes node IDs with function-level granularity
- Look at GraphValueScorer code to understand what it currently expects

## Findings

**Confidence: HIGH**

### 1. Node IDs are structured and deterministic, but NOT directly decodable

Node IDs are generated by `node_id_hash()` in `src/alphaswarm_sol/kg/builder/helpers.py:217-230`:

```python
def node_id_hash(kind: str, name: str, file_path: str | None, line_start: int | None) -> str:
    raw = f"{kind}:{name}:{file_path}:{line_start}"
    return f"{kind}:{hashlib.sha1(raw.encode('utf-8')).hexdigest()[:12]}"
```

This produces IDs like `function:a1b2c3d4e5f6`. The ID format is `{kind}:{12-char-hash}` where the hash is a one-way SHA1 derived from `name`, `file_path`, and `line_start`. **You cannot extract the function name from the node ID alone.**

### 2. Query results return full Node objects with all mapping data

Every query result returns **full Node objects**, not just IDs. The `QueryExecutor._serialize_node()` method in `src/alphaswarm_sol/queries/executor.py:663-667` calls `node.to_dict()` which produces:

```json
{
    "id": "function:a1b2c3d4e5f6",
    "type": "Function",
    "label": "withdraw",
    "properties": {
        "file": "contracts/Vault.sol",
        "line_start": 42,
        "line_end": 67,
        "visibility": "external",
        "semantic_ops": ["..."]
    },
    "evidence": [
        {"file": "contracts/Vault.sol", "line_start": 42, "line_end": 67}
    ]
}
```

The `label` field is the function name. The `properties` dict contains `file`, `line_start`, and `line_end`. The parent contract name is derivable from `CONTAINS_FUNCTION` edges linking contract nodes to function nodes.

### 3. Pattern/finding results also contain node IDs

For pattern queries, findings include `node_id` fields (executor.py:285): `node_ids = {finding["node_id"] for finding in findings}`. These are resolved to full Node objects in the result payload.

### 4. Current evaluation pipeline does NOT capture structured node IDs from results

The `BSKGQuery` dataclass in `tests/workflow_harness/lib/transcript_parser.py:50-71` captures:
- `command`: full command string
- `query_type`: classified type
- `query_text`: the query string
- `result_snippet`: first 2000 chars of tool result (raw text truncation)
- `cited_in_conclusion`: heuristic boolean

**It does NOT parse the JSON result to extract structured node IDs or labels.** Cross-referencing cited node IDs against ground truth requires parsing the JSON from `result_snippet`.

### 5. GraphValueScorer does not use node IDs at all

The current `GraphValueScorer` (`tests/workflow_harness/graders/graph_value_scorer.py`) scores only:
- Query type coverage (did the agent issue expected query types?)
- Citation rate (regex: `node:|edge:|graph:|BSKG|build-kg` in response text)
- Graph-first compliance (Bash before Read in tool sequence)

No cross-referencing of node IDs against ground truth exists.

## Recommendation

**The cross-scorer design (P7-IMP-14) is feasible. The data is fully available and mappable. Implementation requires a specific extraction step in the observation layer.**

### Do this:

1. **Plan 02 (observation hooks):** Extend `BSKGQuery` in `transcript_parser.py` to parse `result_snippet` as JSON and extract `result_node_ids: list[str]` and `result_node_labels: list[str]` from `nodes[].id` and `nodes[].label`. This is ~20 LOC. Also extract `result_node_files: list[str]` from `nodes[].properties.file` for file-level matching.

2. **Plan 04 (GraphValueScorer):** Add a `ground_truth_coherence` scoring dimension. The logic:
   - Accept `ground_truth` from evaluation context (contract name, function name, line_range per expected finding).
   - For each ground truth entry, check if any `result_node_labels` from BSKG queries match the expected function name, AND if the function name or node ID appears cited in the agent's response text.
   - Score = (matched ground truth entries / total ground truth entries).
   - Redistribute weights: query_coverage 0.25, citation_rate 0.25, graph_first 0.25, ground_truth_coherence 0.25.
   - LOC estimate: +30 over current Plan 04 estimate (total: +80 LOC instead of +50).

3. **Plan 06 (investigation contracts):** Ground truth entries MUST use `function` (the function name as it appears in `node.label`) and `contract` (contract name). Do NOT use node IDs in ground truth -- they are hash-based and file-path-dependent. Match on `label`, not on hash.

4. **Do NOT attempt to reverse node ID hashes.** The mapping path is: query result JSON -> parse `nodes[].label` and `nodes[].properties.file` -> match against ground truth. Forward lookup only.

### Edge cases identified by adversarial review:

5. **TOON format guard:** `build-kg` defaults to `--format toon`, but `query` outputs JSON. The JSON extraction in BSKGQuery must guard on `query_type in {"query", "pattern-query"}` before calling `json.loads()`. Not all BSKGQuery entries in a session will be parseable as JSON.

6. **Truncation handling:** `BSKG_RESULT_TRUNCATE_LEN = 2000` (transcript_parser.py:23). A query returning many nodes may produce truncated JSON. `json.loads()` on truncated JSON raises `JSONDecodeError`. The extraction MUST wrap in try/except and return empty lists on parse failure. This means ~20 LOC estimate becomes ~30 LOC.

7. **Contract name not in query results:** `_serialize_node()` returns function node's `label` and `properties.file` â€” NOT the parent contract name. Getting the contract name requires traversing `CONTAINS_FUNCTION` edges. Ground truth schema should match on `function` + `file` rather than `function` + `contract`. Alternatively, derive contract name from file path (e.g., `contracts/Vault.sol` â†’ `Vault`).
