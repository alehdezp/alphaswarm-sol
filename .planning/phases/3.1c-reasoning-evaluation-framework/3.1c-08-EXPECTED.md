# Expected Results: 3.1c-08 — Evaluation Runner

## Fundamental Deliverable

This plan delivers `EvaluationRunner` — a standalone pipeline orchestrator that chains contract loading → observation parsing → graph value scoring → reasoning evaluation → deterministic checks → `EvaluationResult` → persistence — so that Plans 09-11 have a single, testable entry point for running any evaluation workflow against fixtures or live sessions. Without it, the skill and agent evaluation plans (09-11) have no callable pipeline to execute against, and regression baseline (Plan 12) has nothing to write to. Success means the runner executes the full pipeline on a standard fixture session (producing non-null `graph_value_score` and `reasoning_assessment`), correctly refuses invalid sessions before scoring begins, gates baseline writes on a strict three-condition AND, and the Scoring Quality Gate empirically confirms the scorer discriminates good from bad agent behavior. Failure means the runner produces `EvaluationResult` objects that contain no discriminating signal, accepts invalid sessions into the baseline, or imports CC skill code — making the evaluation framework structurally unreliable before tests ever run.

## Drift Detection

### OK — Mode-aware filtering implemented as a separate post-scoring pass

**Scenario:** The executor implements mode-aware dimension filtering as a post-scoring step that sets `applicable=False` on dimensions after GVS and reasoning evaluator return, rather than passing the mode into each grader directly.

**Why OK:** The must_haves require that mode-aware filtering is applied and that `ScoreCard.effective_score()` is used for all computations. Whether filtering happens before or after grader invocation is an implementation detail, as long as inapplicable dimensions are excluded from `effective_score()` and the graders themselves are still called with full inputs.

**Action:** Verify the test for "headless run skips interactive-only dimensions" confirms that `effective_score()` excludes the filtered dimensions — that is the observable contract, not the internal ordering.

---

### OK — Negative fixture directories contain minimal file sets (not full workflow outputs)

**Scenario:** The three negative fixture directories (`session_interrupted/`, `session_stale/`, `session_missing_debrief/`) contain only the files necessary to trigger their specific failure — for example, `session_missing_debrief/` contains a JSONL but no debrief file, rather than a full simulation of an interrupted session with partial scores.

**Why OK:** The plan specifies these fixtures test `_check_session_validity()` failure paths. The validity check reads the presence and basic structure of session files. Minimal fixtures that trigger the right failure mode are sufficient — they do not need to simulate a complete session's output.

**Action:** Confirm each fixture's file set is sufficient to trigger the named failure reason in the validity manifest (a missing debrief JSON is enough for `session_missing_debrief/`) and that tests assert the correct reason string.

---

### DRIFT — Scoring Quality Gate passes but fixtures are shallow

**Scenario:** The executor creates two fixture files where the "good" transcript has a few BSKG-like tool call records prepended and the "bad" transcript has none — constructed to pass the gate rather than sourced from a real agent run. The gate passes numerically but the fixtures do not represent real behavioral differences.

**Why DRIFT:** The CONTEXT.md decision (P14-IMP-14) is explicit: fixtures MUST come from real agent runs. The purpose of the quality gate is to empirically validate that the scoring stack discriminates real-world behavioral differences, not that it can distinguish hand-crafted inputs. If fixtures are synthetic, the gate is circular — it only proves the scorer responds to structural cues that were intentionally planted.

**Action:** Check `README.md` for provenance documentation. If it describes a generation script or says "synthetic," flag as misalignment and request the executor re-run vrs-attacker with both prompt variants to generate real transcripts.

---

### STOP — EvaluationRunner imports alphaswarm_sol.shipping

**Scenario:** The executor builds a working runner that produces correct `EvaluationResult` objects and passes all quality gate tests, but `evaluation_runner.py` imports `from alphaswarm_sol.shipping.skills.test_suite import coordinator` (or any other CC skill code) to handle session coordination.

**Why STOP:** This violates the fundamental isolation boundary the plan was built to enforce. The must_have truth is non-negotiable: the runner must be callable without the CC skill layer installed. If this import exists, Plans 09-11 will silently depend on the CC skill being installed in the test environment, making the evaluation framework non-portable and impossible to run in CI without the full product. The entire wave-5 plan fails if this isolation is broken.

**Action:** Do not accept this plan as complete. Request the executor remove the CC skill import, extract the needed logic into the test-harness layer, and re-verify with `grep "alphaswarm_sol.shipping" tests/workflow_harness/lib/evaluation_runner.py` returning empty.

## Reviewer Guidance

**Priority 1 — Verify the CC isolation boundary first.** Before checking anything else, run `grep -r "alphaswarm_sol.shipping" tests/workflow_harness/lib/evaluation_runner.py`. A single match is a STOP condition. This is the structural promise that makes all downstream plans viable.

**Priority 2 — Inspect the Scoring Quality Gate fixture provenance.** Read `tests/workflow_harness/fixtures/scoring_quality_gate/README.md`. Confirm it names a real contract, real run parameters, and the prompt modification used. If the README is absent or describes synthetic generation, the quality gate is circular and must be re-done with real agent runs before Plan 09 can be unblocked.

**Priority 3 — Trace the three-condition AND gate in source.** Read the `run()` method in `evaluation_runner.py` and locate the `update_baseline()` call site. The guard must check all three conditions (`manifest.valid`, `status == 'completed'`, `effective_score() is not None`) in the same conditional — not split across separate early returns or delegated into `BaselineManager`.

**Priority 4 — Verify negative fixtures assert baseline exclusion, not just validity.** The validity check returning `valid=False` is necessary but not sufficient. The tests must also assert that none of the three negative fixture sessions appear in the baseline store after `run()` completes. Check the test assertions, not just the fixture structure.

**Priority 5 — Confirm standard fixture produces non-null scores end-to-end.** Run `pytest tests/evaluation/test_evaluation_runner.py -k "standard_fixture" -v` and verify the output shows non-null `graph_value_score` and `reasoning_assessment` — not just that the test passes without assertion errors.
