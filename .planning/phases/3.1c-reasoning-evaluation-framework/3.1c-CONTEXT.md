### Evaluation Philosophy
- Two-stage: capability contract check THEN reasoning evaluation (PHILOSOPHY.md Pillar 3)
- Scores are internal regression signals, NOT product quality metrics (Pillar 5)
- Human has final word on all prompt changes (Pillar 4, Rule E)

- Evaluation answers ONE question: "Is it working as expected? If not, what broke and
  where?"

### Signal Validity Vocabulary (P17-SYN-01)
Three signal classes govern how scored fields are interpreted across the framework:
- **VALIDITY_CLAIM:** Asserts measurable quality. Must cite empirical evidence (e.g.,
  calibration data, cross-check results). Exit gates MAY threshold on these.
- **STRUCTURAL_PROXY:** Pipeline routing/guard signal with no ranking claim. Used for
  anti-fabrication triggers, zero-query guards, fast-fail gates. Exit gates treat these as
  health checks, NOT quality measurements.
- **PROVISIONAL:** Under validation — cannot gate decisions. Tagged
  `[pending-real-transcript-validation]` or equivalent. Consumers must handle absence
  gracefully.

Plan 01 scored fields: annotate each against these classes using Track A/B/C as starting
classification (Track A → VALIDITY_CLAIM candidates, Track B → PROVISIONAL, Track C →
STRUCTURAL_PROXY). Exit gate threshold references must declare which validity class they
require.

### Heuristic Scoring Demotion (P17-IMP-01, 3.1e-03 empirical basis)
3.1e Plan 03 establishes: ALL 7 heuristic dimensions have ZERO validity as ranking signal
vs LLM evaluation (NONE across all dimensions). Heuristics are demoted to STRUCTURAL_PROXY
role only — pipeline health checks, anti-fabrication guards, and fast-fail gates. They MUST
NOT contribute to evaluation scores. Heuristic structural guard uses (anti-fabrication,
zero-query detection) are explicitly preserved — 3.1e did not test these non-ranking use
cases. The dual-path framing is corrected: "LLM primary path with structural heuristic
guards — heuristics do not score, they gate."

### Process/Outcome Boundary Enforcement (P14-SYN-01)
The Grader Taxonomy separation between process evaluation (transcript graders) and outcome
evaluation (detection graders) is enforced at four touchpoints: (a) rubric text (P14-IMP-09
grep check — zero detection-outcome phrases in rubric text), (b) capability contract fields
(P14-IMP-16 — prohibited: `findings_count`, `vulnerabilities_found` as capability gates),
(c) fallback scoring protocol (P14-IMP-12 — detection accuracy explicitly excluded from
fallback options), (d) EvaluationResult visibility (P14-ADV-3-01 — DetectionSummary for
corpus runs). Any new scoring dimension, contract field, or improvement loop criterion that
references detection outcomes as a process-evaluation input MUST be flagged as an outcome
metric and routed through the External Detection Validation path, not the reasoning
evaluation path. Plan 06 exit criterion: "Process/Outcome boundary audit: grep all 51
contracts for fields referencing detection outcomes used as capability gates or reasoning
dimensions. Zero violations." Plan 12 Part 2 startup check: "Verify no
ExperimentLedgerEntry uses detection_accuracy as kill_criterion for a reasoning dimension
improvement."

### Evaluation Architecture
- Smart selection matrix: each evaluation contract specifies which hooks/checks apply per
  workflow category:
  - **Investigation** (attacker, defender, verifier): Tool use hooks, graph scoring, deep
    reasoning, Stop debrief
  - **Tool integration** (slither, aderyn): Tool use hooks, standard reasoning, no graph
    scoring, no debrief
  - **Orchestration** (audit, verify, debate): Agent lifecycle hooks, coordination +
    coherence scoring, multi-agent debrief
  - **Support** (health-check, bead-*): Minimal hooks, lite reasoning, deterministic
    checks only
- Testing tiers: Core (~10), Important (~15), Standard (~26) — ALL get reasoning
  evaluation, no exclusions
- Tier assignment is dynamic with adaptive promotion/demotion based on behavioral signals
- Safe sandboxing: Jujutsu workspace isolation preferred, `.claude.sandbox/` fallback.
  NEVER modify production `.claude/` paths
- 51 per-workflow evaluation contracts (one per shipped workflow), not category-level

### Two-Tier Evaluation Architecture
- **Tier 1 — Evaluation Engine (3.1c-01 through 3.1c-12):** Deterministic pipeline — hooks
  observe, parser extracts, scorer computes, evaluator judges, runner orchestrates.
  Produces repeatable results.
- **Tier 2 — Evaluation Intelligence (v2 scope):** Adaptive layer — 12 sub-modules that
  discover gaps, generate tests, heal contracts, fingerprint behavior, audit the evaluator
  itself. Activates incrementally as run data accumulates. Do not design 3.1c-01 through
  3.1c-11 to depend on intelligence-layer features.

### Scoring
- All evaluation scores use a **0-100 integer scale** everywhere: `ReasoningAssessment`,
  `GraphValueScore`, evaluation contracts, improvement loop thresholds
- 3.1b's `GradeResult.score` is already 0-100 integer — no conversion needed

### Debrief Strategy (Research Complete)
Four-layer strategy verified (Research Spike 02, 2026-02-11):

1. **SendMessage to idle agent (PRIMARY):** Agent goes idle after task completion but
   stays alive with full context. SendMessage wakes it for debrief questions. HIGH
   confidence.
2. **Hook gates via exit code 2 (SAFETY NET):** TeammateIdle and TaskCompleted hooks
   return exit 2 to block until debrief marker file exists. Must use `type: "command"`
   hooks and check `stop_hook_active` flag. Hook gate confidence is LOW unless Layer 1
   (SendMessage) also fired in the same session and populated answers — the gate blocks
   but does not independently collect responses.
3. **LLM quality gate (OPTIONAL):** Command hook shells out to LLM for debrief quality
   assessment.
4. **Transcript analysis (FALLBACK):** Extract reasoning from transcript using
   `get_text_between_tools()`.

**Key verified facts:**
- **BUG #20221:** `type: "prompt"` SubagentStop hooks are BROKEN — must use `type:
  "command"` only
- TeammateIdle and TaskCompleted hooks: CAN block, input includes `teammate_name`,
  `team_name`
- `stop_hook_active` flag prevents infinite blocking loops — MUST check in all blocking
  hooks
- Hooks snapshot at startup — must install BEFORE starting session
- SendMessage has no delivery confirmation — always returns success even if recipient dead
- Compaction destroys structured context — keep debrief questions concise
- PreCompact hook (auto-trigger only) sets `{session_id}.compacted` marker; debrief
  branches to 3 structural questions when marker present (P11-IMP-04)
- **Deferred research (P13-ADV-4-01):** When BUG #20221 is resolved, evaluate SubagentStop
  prompt-hook as Layer 1 alternative. Criteria: (a) agent context integrity at
  SubagentStop, (b) hook fire timing relative to transcript finalization, (c) exit code 2
  blocking availability. NOT a v1 implementation item
- **6th essential hook — `delegate_guard.py` (P15-IMP-22):** PreToolUse blocking hook
  (~25 LOC) enforcing delegate mode via config file. Reads blocked list from
  `delegate_guard_config.yaml` with `blocked_tools`, `blocked_patterns`, `allowed_reads`.
  Returns exit 2 on match. Plan 09 exit: negative test (blocks .sol Read) AND positive test
  (permits .vrs/ Read)

### Execution Model
- ALL evaluation runs through Claude Code subagents (Task tool with prompt). NEVER direct
  Anthropic API calls.
- **Execution constraint — session-type distinction (P15-IMP-17):** Agent Teams MUST
  execute sequentially in top-level sessions (shared session state). Headless `claude -p`
  sessions MAY run in parallel when: (a) each session writes to `{session_id}/`
  subdirectory (enforced by EvaluationRunner), (b) BaselineManager uses `fcntl.flock()`
  file locking for write operations, (c) circuit-breaker state is read-only during parallel
  execution — writes happen in coordinator only. Plan 08: `max_parallel_headless: int = 1`
  default, increment to 3 after isolation test passes.
- Primary realism: interactive Claude Code Agent Teams (attacker/defender/verifier)
- Secondary reproducibility: scripted sessions via `ClaudeCodeRunner` or `claude --print`

### ground_truth_rubric Specification
Type: `string`, optional at all tiers (Core SHOULD include, Standard MAY omit). Max
length: 500 tokens (~400 words). Format: free prose, one sentence per expected reasoning
process step. Rubric text MUST describe reasoning process expectations only (examples:
"Agent should query BSKG for external-call patterns before forming an initial hypothesis",
"Agent should explicitly evaluate whether state-write ordering is protected"). Rubric MUST
NOT encode detection outcomes ("Agent should find reentrancy"). **NON-CONFORMING:**
"Agent finds the reentrancy in withdraw()." **CONFORMING:** "Agent queries BSKG for
value-movement patterns before invoking hypothesis formation."
Plan 06 deliverable 8: Core contracts MUST include rubric text. Plan 06 exit criterion:
"All Core-tier `ground_truth_rubric` texts pass rubric conformance check — zero
detection-outcome phrases ('find', 'detect', 'identify the vulnerability') in rubric text."
Plan 07 deliverable 9: inject as `## Expected Reasoning Coverage\n{rubric}` in
REASONING_PROMPT_TEMPLATE between Dimension and observed behavior sections. When absent,
omit section entirely. Score mapping: rubric present + reasoning coverage spans all
specified process steps = 80+, partial coverage = 40-79, none covered = 0-39.
Store as named dimension `rubric_coverage` separate from other dimensions.
Scoring bands are PROVISIONAL (P15-IMP-10): Plan 07 reads from `evaluation_config.yaml`
instead of constants. 3.1e may recommend different bands.

### Grader Taxonomy (P13-IMP-16)
GVS and reasoning move scores are *transcript graders* — they assess execution quality
(how well the agent reasoned). Detection accuracy is an *outcome grader* — it assesses
task completion (did the agent find the vulnerability). `ground_truth_rubric` is a
transcript grader (process column). Its scoring bands measure reasoning move coverage, not
detection outcomes. These can diverge: an agent can reason well but miss a finding, or
find it by luck with poor reasoning. If Spearman rho(GVS, detection_accuracy) < 0.3
across Core investigation agents, this is a GVS failure condition requiring diagnosis — do
NOT auto-substitute detection accuracy. The rho < 0.3 threshold is a halt condition, not a
graceful fallback. Plan 12 Part 1 must compute rho and HALT if < 0.3.

### DC-1: Run-Mode Classification (3.1c-01 models)
`EvaluationResult` must include:

`alphaswarm_sol.kg` or `alphaswarm_sol.vulndocs`. BSKG-specific scoring

Plan 01 exit criteria: hard criterion "DC-2 import boundaries verified — all three grep checks return zero matches."

### DC-3: EvaluationPlugin Protocol (3.1c-01 models)
```python
class EvaluationPlugin(Protocol):
    name: str
    def score(self, output: CollectedOutput, context: dict[str, Any] | None = None) -> PluginScore: ...
    def explain(self, score: PluginScore) -> str: ...
```
**Note:** The `context` parameter fixes a present-day API violation —
`reasoning_evaluator.py:201` already passes `context=context` to plugins, but the Protocol

### FS-01: Improvement Loop Optimizes Wrong Direction Without External Anchor (IMP-07)
Day 1: Attacker scores 55 on CONCLUSION_SYNTHESIS. Day 2: Improvement adds "always cite
specific function names and line numbers." Day 3: Scores 72 — more specific citations.
Day 4: Agent over-cites on SAFE contract, constructs convincing FALSE vulnerability.
Evaluator scores 68. No regression detected (68 > 55 baseline). Agent is WORSE at
distinguishing real from false but scores HIGHER. Detection baseline already shows
Precision=13.3% — agents already produce ~6.5 false positives per true positive. Rewarding
specificity without truth checking amplifies this.

**Mitigations (defense-in-depth):**
1. **Calibration Anchor Protocol (ADV-03):** External ground truth validation before
   accepting any calibrated hints. Spearman rho > 0.6 on rank ordering.
2. **External Detection Validation (IMP-06):** Per-experiment kill criterion requires
   detection rate non-regression on corpus contracts.
3. **Evaluator Debate Protocol (IMP-14):** Evaluator B, seeing A's raw scores without
   reasoning, may independently score the false finding lower. Partial protection.

### FS-02: Late Zombie Artifact Contamination (IMP-20)
Core investigation spawns attacker + defender + verifier. Attacker enters unproductive loop.
45-min timeout fires. CC skill writes `status: interrupted` and starts next workflow. But
zombie attacker still runs, writing complete observations. Scoring pipeline processes it as
valid result, contaminating baseline.

**Mitigations (defense-in-depth):**
1. **SubagentStop hook termination detection (PRIMARY):** CC fires `SubagentStop` hook when
   subagent finishes (confirmed in official hook lifecycle: 14 events). CC skill installs
   SubagentStop hook that writes `{session_id}.completed` marker file. Provides definitive
   termination detection.
2. **Interrupted sentinel:** EvaluationResult at `interrupted` status NEVER written to
   baseline. CC skill writes `{session_id}.interrupted` sentinel in observation directory
   upon timeout. EvaluationRunner checks sentinel before scoring — if found, skip regardless
   of JSONL completeness. ~5 LOC.
3. **60-second grace period:** Before starting next workflow. Fallback if SubagentStop
   doesn't fire (subagent crash).

4. **SessionEnd marker (P11-IMP-01/ADV-1-01):** `obs_session_end.py` writes
   `{session_id}.session_ended` marker on normal termination. 4th SVM condition: absent
   marker = WARN not REJECT (graceful degradation for SIGKILL/crash). SessionEnd fires in
   both headless and interactive mode (confirmed). Cannot block (exit code 2 = N/A).

Plan 08 EvaluationRunner: add sentinel check (~5 LOC). Plan 02 ObservationWriter: ignore
write attempts after sentinel detected. Per-workflow observation subdirectory NOT required
— sentinel file sufficient and simpler.

### FS-03: Dead Agent — Layer 1 Returns Success, Artifact Never Written (ADV-5-01)
CC skill sends SendMessage for debrief. SendMessage returns success even if recipient dead
(CC platform behavior). Runner reads missing file and either crashes or silently falls
through with no record of Layer 1 attempt.

**Mitigation:** Post-SendMessage artifact-presence check with configurable wait (default
`debrief_wait_seconds: 60`). If absent after wait: write sentinel JSON `{layer_used:
"send_message_no_response", answers: [], confidence: 0.0}`, continue to Layer 3.
`DebriefResponseValidator` adds `layer_used="send_message_no_response"` signal.
`debrief_wait_seconds: int = 60` added to evaluation contract schema + Schema Hardening
Field Manifest.

### FS-04: Valid-Looking Artifacts from Invalid Session (SYN-02)
Three contamination guards protect the same invariant ("this session's artifacts are
trustworthy") but have no shared enforcement point. A zombie session may write complete
JSONL that passes the staleness guard (recent) and the session-id filter (same namespace),
while the interrupted sentinel exists.

**Mitigation:** Session Validity Manifest — before Plan 08 scores any session, construct
`SessionValidityManifest` by checking in order: (a) `{session_id}.interrupted` sentinel
absent (FS-02), (b) observation file mtime within staleness window (ADV-1-01), (c) debrief
artifact present OR sentinel JSON `send_message_no_response` written (FS-03), (d)
`{session_id}.session_ended` marker present (P11-IMP-01/ADV-1-01 — absent = WARN not
REJECT, graceful degradation for SIGKILL/crash). All four checks apply regardless of
run_mode. Score only if `manifest.valid`. If invalid, write
`EvaluationResult(status="invalid_session", reasons=manifest.reasons)` — never enters
baseline.

**Design rationale:** Do not remove any of these three mitigations. They prevent a
self-sealing Goodhart's Law loop where the framework produces consistent scores while being
systematically wrong about agent quality.

not empirically supported.

## Reasoning Evaluation Deep Dive
<!-- REFERENCE: Scoring methodology details. Consult when writing Plans 07, 08, 12. -->

### 7 Reasoning Move Types (scored independently 0-100)
1. `HYPOTHESIS_FORMATION`: States expected vulnerability and why
2. `QUERY_FORMULATION`: Constructs targeted graph query
3. `RESULT_INTERPRETATION`: Reads query results, extracts meaning
4. `EVIDENCE_INTEGRATION`: Connects multiple pieces into coherent picture
5. `CONTRADICTION_HANDLING`: Adjusts approach on unexpected results
6. `CONCLUSION_SYNTHESIS`: Derives specific findings from evidence
7. `SELF_CRITIQUE`: Questions own reasoning, identifies gaps

### Per-Category Move Applicability
Not all 7 move types apply to all workflow categories. Scoring non-applicable moves as
zero cascade-poisons the scorecard. Per-category scope:
- **Investigation:** All 7 moves scored
- **Tool Integration:** RESULT_INTERPRETATION + CONCLUSION_SYNTHESIS only; others N/A
- **Orchestration:** EVIDENCE_INTEGRATION + CONCLUSION_SYNTHESIS + SELF_CRITIQUE +
  CONTRADICTION_HANDLING (all DEFAULT_ON); HYPOTHESIS_FORMATION and QUERY_FORMULATION
  receive `applicable: False` automatically
- **Support:** Skip per-move evaluation entirely

Scoring aggregator uses `applicable: bool` on `DimensionScore` — N/A entries excluded
from weighted average. Templates include explicit skip directives for non-applicable
moves.

### Move Type Default Applicability Table
Score comparability is a hard requirement for the regression baseline (Plan 12). This
table declares which moves are applicable by default per workflow category. Individual
contracts override via explicit dimension entries with `applicable: false`. Plan 07
reads this table.

| Move Type | Investigation | Tool Integration | Orchestration | Support |
|-----------|:---:|:---:|:---:|:---:|
| HYPOTHESIS_FORMATION | DEFAULT_ON | DEFAULT_OFF | DEFAULT_OFF | N/A |
| QUERY_FORMULATION | DEFAULT_ON | DEFAULT_OFF | DEFAULT_OFF | N/A |
| RESULT_INTERPRETATION | DEFAULT_ON | DEFAULT_ON | DEFAULT_OFF | N/A |
| EVIDENCE_INTEGRATION | DEFAULT_ON | DEFAULT_OFF | DEFAULT_ON | N/A |
| CONTRADICTION_HANDLING | DEFAULT_ON | DEFAULT_OFF | DEFAULT_ON | N/A |
| CONCLUSION_SYNTHESIS | DEFAULT_ON | DEFAULT_ON | DEFAULT_ON | N/A |
| SELF_CRITIQUE | DEFAULT_ON | DEFAULT_OFF | DEFAULT_ON | N/A |

### Heuristic vs LLM Scoring (3.1e-03 empirical basis)
All 7 reasoning move types are scored by LLM evaluator ONLY. Heuristic dimensions have
ZERO validity as ranking signal (P17-IMP-01). Heuristics serve as structural guards
(anti-fabrication triggers, zero-query detection) classified as STRUCTURAL_PROXY — they do
not contribute to any move type score or scorecard computation.

### Cascade-Aware Scoring
Moves form a DAG: HYPOTHESIS -> QUERY -> INTERPRETATION -> INTEGRATION -> [CONTRADICTION]
-> CONCLUSION -> SELF_CRITIQUE. When upstream fails, downstream may score low as cascade
victim, not independent failure. Three modes: hard blocking (score=0 if prerequisite
failed), soft blocking (proportional to upstream health), attribution (classify root cause
vs cascade victim). SELF_CRITIQUE is never hard-blocked (cross-cutting). Based on
PRISM-Physics ancestor closure scoring.

### Self-Describing Failure Narratives
Every scored dimension produces: "what happened" (from transcript) + "what should have
happened" (evaluator counterfactual). Example: "QUERY_FORMULATION scored 35. Actual:
queried 'functions with external calls' (47 results). Ideal: should have queried
'functions calling untrusted addresses without reentrancy guards' (3 results)." These
pairs are the primary metaprompting signal.
