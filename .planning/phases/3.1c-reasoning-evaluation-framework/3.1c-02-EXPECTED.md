# Expected Results — Plan 3.1c-02: Transcript Capture, Hook Infrastructure, and Validation

## Fundamental Deliverable

This plan delivers 3+ real Claude Code session transcripts plus the 6 essential hook files that make future evaluation possible, so that Plans 03–12 have a verified data source and enforcement infrastructure to build on. Without it, all downstream evaluation plans are working against fabricated or synthetic data — a condition the phase explicitly banned after deleting all prior fake transcripts — and every hook in the pipeline is an untested stub. Success means the real_sessions directory contains JSONL files from distinct workflow categories, TranscriptParser.to_observation_summary() returns non-empty tool_counts on each of them, obs_precompact.py uses zero imports outside stdlib, and delegate_guard.py demonstrably blocks the calls its config prohibits. Failure means the directory is empty, the hook files are scaffolded placeholders with no real session validation, or obs_precompact.py imports TranscriptParser — which would have made it impossible to deploy the hook safely in production contexts.

## Drift Detection

### OK — obs_precompact.py is longer than 35 LOC due to error handling

**Scenario:** The executor implements `obs_precompact.py` at 65 LOC instead of the ~35 LOC target, adding try/except blocks for file-not-found, invalid JSON lines, and permission errors, plus logging to stderr.

**Why OK:** The constraint is stdlib-only imports and synchronous execution — not line count. Defensive error handling in a hook that runs at compaction time is appropriate. As long as no project imports appear and the file completes well within 5s, the extra LOC is acceptable variation.

**Action:** Verify no project imports were introduced. Verify the try/except blocks use stdlib exceptions only. No correction needed if these hold.

---

### OK — Golden file snapshots are from a subset of the 6 hooks

**Scenario:** Only 3 of the 6 hooks have golden file snapshots stored in `hook_schema_snapshots/`, because debrief_gate.py and debrief_task_complete.py are harder to trigger in isolation during a real session.

**Why OK:** The plan specifies golden files for the hooks that interact with real CC sessions. Debrief hooks fire on specific workflow completion events that may not occur in the 3 baseline transcripts. Partial snapshot coverage is acceptable if the hooks that do fire against the real sessions are all captured, and the missing snapshots are documented.

**Action:** Verify the hooks with the most platform-schema risk (especially PostToolUse hooks using `tool_response`) have snapshots. Document which hooks lack snapshots and why in the SUMMARY.md.

---

### DRIFT — real_sessions/ contains JSONL files but TranscriptParser returns empty tool_counts

**Scenario:** Three `.jsonl` files exist in `real_sessions/`. The files are real output from `claude -p`. However, `to_observation_summary()` returns `tool_counts = {}` on all three because the JSONL event structure from the actual CC version uses a field name the parser does not expect (e.g., `tool_use` vs `tool_call`).

**Why DRIFT:** The P0 hard gate is not purely "files exist" — it requires that TranscriptParser can extract meaningful data from them. Empty tool_counts on real transcripts means the parser is not connected to the actual data. Plans 03–12 would be building on a broken foundation. This is the exact gap P14-IMP-14 identified: known-good fixtures require validated extraction, not just file presence.

**Action:** Request that the executor run `to_observation_summary()` on each real transcript and document the non-empty tool_counts in the SUMMARY.md. If all return empty, the plan is not complete — the parser must be fixed before the plan is marked done.

---

### DRIFT — delegate_guard.py hardcodes blocked patterns instead of reading from config

**Scenario:** The executor implements `delegate_guard.py` with blocked patterns as Python constants at the top of the file (`BLOCKED_TOOLS = ["Read", "Write"]`) rather than loading them from `delegate_guard_config.yaml`. The blocking behavior works correctly in tests.

**Why DRIFT:** Config-driven enforcement is the explicit requirement (P15-IMP-22). Hardcoded patterns mean the guard cannot be adjusted without modifying the hook file itself, defeating the purpose of a config-driven enforcement system. The must_haves truth specifically says "blocks prohibited tool calls via config" — the "via config" is not incidental.

**Action:** Request correction to read from `delegate_guard_config.yaml`. The config file must also exist with at least the canonical test cases (`blocked_tools`, `allowed_reads` entries).

---

### STOP — 3 JSONL files exist in real_sessions/ but they were generated by a script, not captured from real CC sessions

**Scenario:** The executor writes a Python script that generates plausible-looking CC session JSONL with randomized tool names, timestamps, and node IDs, stores three files in `real_sessions/`, and passes the TranscriptParser validation checks. The files look like real transcripts but contain no actual agent reasoning.

**Why STOP:** This is the exact anti-pattern the phase banned when it "deleted all fabricated data." The fundamental purpose of this plan is to ground the entire evaluation framework in reality. Synthetic transcripts defeat that purpose entirely. Plans 03–12 would be calibrating evaluation rubrics against invented behavior. This is not a variation — it is a direct contradiction of the plan's raison d'etre.

**Action:** Escalate immediately. Require the executor to capture transcripts by actually running `claude -p` or interactive sessions. Verify files contain platform-specific fields (session IDs, actual model response patterns) that cannot easily be fabricated. Do not proceed to Plan 03 until this is resolved.

## Reviewer Guidance

**Priorities:**

1. **Verify P0 is genuinely complete before reviewing hooks.** Check that `real_sessions/` contains 3+ non-empty `.jsonl` files from distinct workflow categories AND that `to_observation_summary()` returns non-empty `tool_counts` on each. File existence alone is not P0 satisfaction. This is the single most important check — everything downstream depends on real data.

2. **Check obs_precompact.py for TranscriptParser imports.** Grep for `transcript_parser` in `obs_precompact.py`. Any match is a hard failure per the stdlib-only constraint. This is a deployment correctness issue, not a style preference.

3. **Verify delegate_guard.py uses config-driven patterns, not hardcoded constants.** Look for `delegate_guard_config.yaml` on disk and verify the hook reads from it. Check for `BLOCKED_TOOLS = [...]` style constants at module level — if present, the config-driven requirement was not honored.

4. **Confirm PostToolUse hooks use `tool_response` not `tool_output`.** This is the platform schema bug from P13-IMP-01. Grep for `tool_output` across all hook files — any match is a data-loss bug that will silently drop tool results from observation data.

5. **Confirm no `type: "prompt"` SubagentStop hooks appear in any hook file.** BUG #20221 locks this out. Search for `SubagentStop` and `type.*prompt` across the hook directory. If present, the plan contradicts a locked decision.
