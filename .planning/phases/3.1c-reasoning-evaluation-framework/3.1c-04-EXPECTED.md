# Expected Results: 3.1c-04 — Graph Value Scorer Fixes

## Fundamental Deliverable

This plan delivers a corrected and provisionally calibrated `GraphValueScorer` so that downstream plans (07, 08) receive signal-valid citation rate and graph-first compliance data instead of noise. Without it, the evaluator in Plan 07 receives false-positive graph-first verdicts (any Bash call counts) and citation rates contaminated by English prose matches, producing evaluation scores that correlate with text content rather than actual graph behavior. Success means the GVS uses TranscriptParser as its primary citation source, uses observation event timestamps to determine graph-first ordering, replaces the keyword-soup fallback with BSKG node ID structural matching, short-circuits cleanly when `run_gvs: false`, and correctly handles compacted sessions via PreCompact snapshot merge. Failure means graph_value_scorer.py exists and tests pass, but the file still guesses tool names to infer BSKG usage, still matches `graph:` in documentation prose, or still scores DEFAULT-30 for dimensions with no real handler.

## Drift Detection

### OK: `graduated_graph_first_score` computed but binary flag preserved

**Scenario:** The executor adds `graph_first_score: float` as a graduated proportion alongside the existing `graph_first_compliant: bool`, and both are populated. The graduated score is passed to Plan 07 context but does not replace the binary enforcement.

**Why OK:** The plan explicitly specifies this additive approach (P13-ADV-3-02). Adding graduated scoring without removing binary enforcement is exactly the intended design. The binary flag remains the enforcement mechanism; the graduated score is contextual signal.

**Action:** Verify both fields exist in `GraphValueScore`, that `graph_first_compliant` is still a `bool`, and that `graph_first_score` is a `float` in [0.0, 1.0].

---

### OK: `_check_citation_relevance()` crash guard added without full coherence implementation

**Scenario:** The executor adds the `if context is None or context.get('ground_truth_entry') is None: return DimensionScore(applicable=False)` guard to `_check_citation_relevance()` but does not implement the full cross-reference logic (deferred to later wave).

**Why OK:** The plan explicitly scopes this fix to a 3-LOC crash guard. The full coherence dimension implementation is noted in CONTEXT.md as requiring ground truth corpus access and is not a Wave 3 deliverable.

**Action:** Verify the guard exists and the function does not crash on `None` context. Note whether the full coherence logic is present or deferred.

---

### DRIFT: Fallback structural matching uses function name substrings, not node ID format

**Scenario:** The executor replaces the keyword-soup regex with a new regex that matches function names (`withdraw`, `transfer`, `approve`) as a proxy for BSKG nodes, rather than matching the actual BSKG node ID format (`[FCE]-contractName-functionName`).

**Why DRIFT:** This is a more sophisticated form of the same underlying problem — matching content-derived strings instead of structural identifiers. The CONTEXT.md explicitly identifies BSKG node IDs as the correct structural anchor. Function name matching would still produce false positives on any Solidity code that mentions those function names in discussion.

**Action:** Request correction. The fallback must match the canonical BSKG node ID format, not function name substrings. Check that the pattern or logic cross-references against actual observation data from Plan 02's BSKG output.

---

### DRIFT: DEFAULT-30 fix applied to dimensions but `applicable` field added incorrectly

**Scenario:** The executor returns `DimensionScore(score=0, applicable=False)` instead of simply `DimensionScore(applicable=False)`, or sets `applicable=False` but the aggregation logic in the scorer still uses `score=0` in the denominator.

**Why DRIFT:** The DEFAULT-30 fix is a BREAKING CHANGE (per plan: "merge MUST be followed by baseline reset"). If the score is set to 0 instead of truly excluded via `applicable=False`, the GVS aggregate computation is wrong — 0 still pulls the total downward rather than being excluded from the denominator.

**Action:** Verify that the GVS aggregation logic excludes `applicable=False` dimensions from the denominator entirely, and that the DEFAULT-30 dimensions do not contribute any numeric value to the final GVS score.

---

### STOP: `_check_graph_first()` refactored to accept `obs_summary` but falls back to tool name substring matching when obs_summary is absent

**Scenario:** The executor adds the `obs_summary` parameter and timestamp-based path, but retains the original `tool_sequence` substring inspection (`if "alphaswarm" in tool_name`) as the fallback when `obs_summary` is `None`. The plan tasks indicate "conservative `False`" when obs_summary is absent.

**Why STOP:** This is a direct contradiction of the plan's locked behavior and the plan's test case (c): "Read before BSKG = False". If the fallback is tool name inspection, non-BSKG Bash calls will still score as graph-first compliant when `obs_summary` is unavailable, which is precisely the false-positive bug this plan exists to fix. The "conservative `False`" requirement is explicit and non-negotiable.

**Action:** Escalate. The fundamental purpose of this plan is to eliminate the false-positive graph-first compliance path. An implementation that preserves the false-positive path as a fallback has not delivered the plan's core promise. The fix must return `False` (not the heuristic result) when `obs_summary` is absent.

## Reviewer Guidance

**Priority 1 — Verify the false-positive elimination.** The entire plan exists to fix `_check_graph_first()`. The single most important check: find the `_check_graph_first` function and confirm (a) it accepts `obs_summary`, (b) it uses timestamps from `bskg_query` events for ordering, and (c) when `obs_summary` is `None` or empty, it returns `False` — not a tool-name substring check. Test case (b) from the plan ("non-BSKG Bash before Read = False") is the decisive signal.

**Priority 2 — Confirm keyword-soup is gone from the citation fallback path.** Grep `graph_value_scorer.py` for `node:|edge:|graph:|BSKG`. If any of these patterns appear in a regex used for citation detection (not in comments documenting the old approach), the keyword-soup replacement was incomplete.

**Priority 3 — Verify DEFAULT-30 fix and score denominator exclusion.** Check that `exploit_path_construction`, `arbitration_quality`, `evidence_weighing`, `investigation_depth` return `applicable=False`. Then verify the GVS aggregation does not include `applicable=False` dimensions in the denominator. A score of 0 and `applicable=False` are not equivalent.

**Priority 4 — Confirm TranscriptParser is called first, not last.** Read the citation rate computation flow and confirm `graph_citation_rate()` is the primary branch, with structural node ID matching as the fallback. This is a locked architectural decision; inversion would contradict the CONTEXT.md.

**Priority 5 — PreCompact merge test must use a real compacted session shape.** The merge test is only meaningful if `precompact_snapshot` is structured as real observation data would produce it (from Plan 03's `ObservationParser`). A test that passes a hand-crafted dict with no resemblance to actual pre-compaction observation format does not verify the merge works in production.
