# Expected Results: 3.1c-12 — Calibration, Baseline, and 3.1f Handoff

**Plan:** 3.1c-12 | **Wave:** 7 | **Phase:** 3.1c-reasoning-evaluation-framework
**Depends on:** 3.1c-09, 3.1c-10, 3.1c-11
**Scope boundary (locked):** Parts 0-1 only. Parts 2-5 are 3.1f scope.

---

## Fundamental Deliverable

This plan delivers a calibrated regression baseline for all 51 evaluation workflows so that future prompt and pattern changes can be detected as regressions with statistical confidence. Without it, the tier-weighted regression detection system in `regression.py` has no ground truth to measure against, making every subsequent plan in 3.1c and 3.1f unable to distinguish improvement from noise. Success means: a Spearman rho > 0.6 is recorded against external ground truth, all 51 workflows have at least one baseline entry keyed by `(workflow_id, run_mode, debate_enabled, effort_level)`, Core workflows have 5 entries for variance computation, and the three handoff artifacts (`calibration_config.yaml`, `variance_summary.yaml`, `baseline_data/`) are schema-stable for 3.1f to consume without revisiting 3.1c decisions. Failure means: baseline entries exist but the calibration anchor was never validated (rho unknown or undocumented), variance data is absent or incomplete, improvement.py makes no reference to `CalibrationConfig`, or Parts 2-5 logic was implemented inside this plan — violating the scope boundary and contaminating the 3.1c/3.1f handoff.

## Drift Detection

### OK — Variance summary uses median absolute deviation instead of std dev

**Scenario:** The executor computes median absolute deviation (MAD) instead of standard deviation for per-dimension variance, citing robustness to outlier evaluation scores.

**Why OK:** The must_have specifies "per-dimension variance from 5x Core runs" — the specific statistic is implementation detail. What matters is: (1) a per-dimension scalar is present, (2) it was computed from the 5 Core runs, (3) `convergence_config.yaml` references it with the correct formula (so `convergence_threshold = max(2pt, 1.5 × MAD)` is explicit). If those three conditions hold, the downstream convergence detection still works correctly.

**Action:** Verify `variance_summary.yaml` documents which statistic was used and `convergence_config.yaml` references the same formula. Confirm deviation from std dev is documented in the summary so 3.1f inherits the correct formula.

**Scenario:** The executor ran 2 Core agents on 18 contracts (36 sessions) in the HITL-approved batch, computed an interim Spearman rho > 0.6, and documented the remaining 36 sessions as pending a second HITL approval.

**Why OK:** The must_have is that rho > 0.6 is computed and documented before calibrated hints. If the interim rho passes the gate with 36 sessions and a confidence interval is included, the downstream gate is satisfied. The full 72-session anchor is the plan target, but partial completion with documented continuation is not misalignment — it is the HITL checkpointing the plan explicitly describes.

**Action:** Verify `calibration_config.yaml` contains the interim rho value, a confidence interval, and a `run_count` showing 36 (not 72). Confirm the remaining sessions are scheduled, not abandoned.

### STOP — Baseline anchored to fabricated/simulated evaluation data from Plans 09-11

**Scenario:** Baseline entries exist for all 51 workflows. Calibration anchor was computed. Tests pass. But the baseline data originates from Plans 09-11 entries that used `RunMode.SIMULATED` — no real Agent Teams were spawned for the underlying evaluations. The baseline appears complete but is anchored to fabricated scores, not observed agent behavior.

**Why STOP:** The baseline is the foundation for ALL downstream regression detection and improvement loops. If the baseline scores come from simulated data:
- Regression detection will flag real Agent Teams scores as "regressions" (because they differ from fabricated baselines)
- Improvement variants cannot be meaningfully compared to baselines that don't represent real behavior
- The Spearman rho from calibration is meaningless if calibration runs were also simulated
- Anti-fabrication checks should have caught this, but if they were not implemented, the fake data flows through unchecked

This is the highest-impact failure mode: it doesn't just invalidate Plan 12 — it invalidates the entire 3.1f phase and everything downstream.

**Concrete detection signals:**
1. Plans 09/10/11 entries in progress.json have `run_mode: "SIMULATED"` or missing `run_mode`
2. No JSONL transcripts exist for Plans 09-11 evaluations
3. Calibration runs completed in < 5 seconds per session
4. All baseline scores are identical or suspiciously uniform
5. Anti-fabrication triggers (100% pass rate, scores at 100, duration < 5s) were not implemented
6. `PREREQUISITE CHECK` in Plan 12 was skipped or its results were ignored

**Action:** Reject immediately. This requires re-execution of Plans 09-11 with REAL Agent Teams FIRST, then re-execution of Plan 12 calibration and baseline with real data. No shortcuts.
