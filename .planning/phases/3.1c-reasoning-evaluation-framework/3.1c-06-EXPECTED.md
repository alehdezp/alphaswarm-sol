# Expected Results: 3.1c-06 — Evaluation Contracts

## Fundamental Deliverable

This plan delivers 51 per-workflow YAML evaluation contracts, a hardened JSON Schema, and a canonical dimension registry so that Plans 07, 08, 09, and 12 have a complete, schema-validated specification for what "good reasoning" looks like for every shipped workflow. Without it, the LLM reasoning evaluator (Plan 07) has no dimension whitelist to iterate, the evaluation pipeline (Plan 08) has no per-workflow hook configuration, and the regression baseline (Plan 12) cannot distinguish stub-quality from hand-authored Core contracts. Success means 51 YAML files validate against a schema that requires `evaluation_config` with 4 sub-fields, Core-tier contracts carry `ground_truth_rubric` text describing reasoning process only, and a grep of all 51 files finds zero capability fields that use detection outcomes as pass/fail thresholds. Failure means the files exist but contain placeholders that bypass schema enforcement — contracts with `evaluation_config: {}`, rubrics that say "agent should find reentrancy," or capability checks gated on `findings_count >= 1`.

## Drift Detection

### OK — Standard stubs authored without hand-crafting every field

**Scenario:** The executor created all 26 Standard-tier contracts from a template script that populated `workflow_id`, `category`, and `status: stub`, leaving all other fields at template defaults. The contracts are schema-valid but contain no workflow-specific content beyond the ID.

**Why OK:** CONTEXT.md explicitly specifies "Standard tier (~26): Create template-derived stubs during Plan 06. Full authoring deferred until after at least 1 real run." Template-derived stubs with valid schema and `status: stub` are exactly what was specified. The reviewer should verify `status: stub` is present and `run_reasoning: false`, not that the contracts contain hand-authored content.

**Action:** Verify Standard stubs have `status: stub`, `run_reasoning: false`, `reasoning_dimensions: []`. Confirm they are not accidentally promoted to `status: active`.

---

### OK — Schema field names differ from manifest but are structurally equivalent

**Scenario:** The schema uses `eval_config` instead of `evaluation_config` for the required object, and all 51 contracts were authored consistently with this name.

**Why OK:** If the executor made a consistent naming choice applied everywhere (schema + all contracts + contract_loader.py), and the four sub-fields (`run_gvs`, `run_reasoning`, `debrief`, `depth`) are all present under the chosen name, the structural intent is satisfied. The exact field name `evaluation_config` is specified in must_haves but the criterion is the required 4-sub-field structure.

**Action:** Flag the naming deviation for the reviewer's awareness. Verify the chosen name is used consistently across schema, all 51 contracts, and loader code. If inconsistent (schema says one thing, contracts say another), that is DRIFT.

---

### DRIFT — 51 files exist but Core contracts contain no ground_truth_rubric

**Scenario:** The executor created 51 YAML files that all validate against the hardened schema. The file count is correct and all schema fields are present. However, the Core-tier contracts (agent-vrs-attacker, agent-vrs-defender, etc.) have `ground_truth_rubric: ""` or the field is absent — the executor treated Core contracts as Important-tier stubs with extra fields.

**Why DRIFT:** The plan explicitly distinguishes Core contracts as "hand-authored with ground_truth_rubric." An empty rubric means Plan 07's evaluator has no rubric injection for its most important contracts. This directly breaks the Plan 07 dependency on rubric-based scoring (the `rubric_coverage` dimension). The 51-file count and schema compliance are necessary but not sufficient — Core contracts must contain substantive rubric text.

**Action:** Request the executor author rubric text for all Core-tier contracts. Rubric text must describe reasoning process steps, not detection outcomes. Verify against the CONFORMING example in CONTEXT.md: "Agent queries BSKG for value-movement patterns before invoking hypothesis formation."

---

### DRIFT — Schema hardening applied only to new contracts; original 10 not retrofitted

**Scenario:** The executor hardened the schema and correctly authored 41 new contracts using the new schema. However, the original 10 contracts were not updated — they still use `metadata.evaluation_depth` instead of `evaluation_config.depth`, and lack `status:`, `hooks:`, and `coverage_axes:` fields. The migration script exists but was not run.

**Why DRIFT:** CONTEXT.md is explicit: "Retrofit all 10 existing contracts (+ 4 templates) to include all four sub-fields." The cross-plan dependency to Plan 07 is that ALL contracts validate against the hardened schema — a plan 07 evaluator loading the old-format contracts will fail. This is a contract between plans, not an aesthetic preference.

**Action:** Request the executor run `scripts/migrate_contracts.py` and verify all 14 pre-existing files pass validation. Check that `migrate_deprecated()` was applied to move `metadata.evaluation_depth` to `evaluation_config.depth`.

---

### STOP — Contracts exist but capability checks use detection outcomes as pass criteria

**Scenario:** The executor created 51 files and they validate against the schema. However, Core and Important contracts have capability checks like: `id: vulnerability_found`, `expected_behavior: "Response contains at least one vulnerability with severity HIGH"`, `grader_type: model`. The Process/Outcome boundary was not audited.

**Why STOP:** This is the exact violation that P14-IMP-09 identified as "Rubric text MUST describe reasoning process, NOT detection outcomes." If detection outcomes are baked into capability gates, then an agent that fabricates findings passes the gate while an agent that reasons carefully but finds nothing fails. This inverts the evaluation framework's purpose. Plans 07, 08, and 09 will inherit corrupted contracts and produce evaluation results that measure the wrong thing. The entire framework's validity premise is violated.

**Action:** Do not proceed. Request a full Process/Outcome boundary audit: grep all 51 contracts for `findings_count`, `vulnerabilities_found`, and inspect all `grader_type: model` capability checks for outcome language. Every violation must be replaced with structural execution criteria before Plan 07 can begin.

## Reviewer Guidance

**Priority 1: Verify the Process/Outcome boundary across all 51 contracts.**
This is the highest-stakes correctness requirement. A grep for `findings_count`, `vulnerabilities_found`, and detection-outcome verbs in `expected_behavior` fields is the minimum check. Any violation means evaluation results from Plans 07-09 will measure the wrong thing. Check both `capability_checks` blocks AND `ground_truth_rubric` text.

**Priority 2: Confirm Core-tier contracts have substantive ground_truth_rubric text.**
The file count and schema compliance are necessary but not sufficient. Read the rubric text in the ~10 Core contracts. Does it describe agent reasoning steps (queries, hypothesis formation, evidence integration)? Or does it describe findings? This is the content that Plan 07's evaluator will inject into its scoring prompts — empty or outcome-based rubrics will corrupt reasoning scores.

**Priority 3: Validate schema hardening was applied atomically and completely.**
Check `.vrs/testing/schemas/evaluation_contract.schema.json` for `additionalProperties: false` at root, `evaluation_config` in top-level `required`, and the 4-sub-field `required` inside `evaluation_config`. Then check that all 10 original contracts (plus 4 templates) were retrofitted — the migration was not just applied to new contracts.

**Priority 4: Confirm Standard stubs are correctly marked and cannot accidentally pass reasoning evaluation.**
Verify that Standard-tier contracts have `status: stub`, `run_reasoning: false`, and `reasoning_dimensions: []`. Confirm the debrief-mode validator in `contract_loader.py` rejects `run_reasoning: true` + empty dimensions. This gate prevents the evaluation pipeline from wasting Opus calls on stub contracts.

**Priority 5: Verify dimension_registry.yaml is the authoritative source, not a documentation artifact.**
The registry must be a real YAML file at the specified path with 27+ dimension entries. Check that at least one Core contract's `active_dimensions` field references dimension names that appear in the registry. If the registry exists but no contract references it, it is documentation, not a wiring contract.
