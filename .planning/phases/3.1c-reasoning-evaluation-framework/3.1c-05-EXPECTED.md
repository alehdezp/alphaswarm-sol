# Expected Results — 3.1c-05: Debrief Protocol

## Fundamental Deliverable

This plan delivers a working four-layer debrief cascade (SendMessage disk-read path → blocking hook gates → LLM quality gate → transcript fallback) so that Plans 07 and 08 have agent self-report signal to feed the reasoning evaluator and baseline. Without it, Plans 07 and 08 have no debrief data: the evaluator can only assess agent outputs externally, losing the unique self-explanation signal that distinguishes reasoning quality from task completion. Success means `debrief_protocol.py` implements a real disk-read Layer 1, `debrief_gate.py` blocks with exit code 2 and a `stop_hook_active` guard, compaction-aware branching produces 3-question vs 7-question debriefs based on the session marker, and `DebriefResponseValidator` handles malformed responses and dead-agent detection — all tested and schema-validated. Failure means the hooks exit 0 unconditionally (the pre-existing stub behavior), no `.vrs/observations/{session_id}_debrief.json` artifact is produced in live mode, and Plans 07/08 silently receive empty or absent debrief data.

## Drift Detection

### OK — Polling constants differ from task spec

**Scenario:** The executor implements the timeout loop with `timeout_iterations=30` and `poll_interval=2.0` seconds (60s total) rather than the exact values mentioned in the task. The `stop_hook_active` guard is present, `sys.exit(2)` fires on exhaustion.

**Why OK:** The architectural contract (exit 2 on timeout, exit 0 on success, `stop_hook_active` guard first) is what matters. Specific numeric constants are implementation details that can be tuned without affecting the cascade contract. The `debrief_wait_seconds: int = 60` default mentioned in CONTEXT.md is the total, not the per-iteration value.

**Action:** Verify the total timeout is reasonable (default should be ~60s based on `debrief_wait_seconds: int = 60` from CONTEXT.md). Accept if the guard and exit codes are correct.

---

### OK — DebriefResponseValidator implemented as a module-level function, not a class

**Scenario:** The executor implements fence-stripping and dead-agent detection as a `validate_debrief_response(raw: str) -> DebriefResult` function rather than a class named `DebriefResponseValidator`.

**Why OK:** The must_have specifies behavior (fence-stripping, `layer_used` sentinel values, dead-agent detection), not the OOP structure. A function with the same behavior satisfies the criterion.

**Action:** Verify the function handles all three cases: valid fenced JSON, malformed JSON, absent artifact. Accept if the sentinel values match (`"send_message_malformed"`, `"send_message_no_response"`).

---

### DRIFT — debrief_gate.py blocks correctly but stop_hook_active guard is missing

**Scenario:** The executor implements real timeout polling with `sys.exit(2)` on exhaustion. The gate works correctly for normal flows. However, the `stop_hook_active` guard was omitted — the hook does not short-circuit when Claude Code signals it should not re-block.

**Why DRIFT:** This is a latent infinite-loop risk. In practice, during evaluation runs with multiple subagent stops, the missing guard will cause the hook to re-enter its blocking loop even when the stop event is the hook's own prior block resolving. This corrupts evaluation sessions silently — no immediate error, but stuck evaluations.

**Action:** Request correction before accepting. The `stop_hook_active` check is a locked CONTEXT.md decision (not optional) and is explicitly listed in must_haves.truths[1].

---

### DRIFT — Layer 1 calls SendMessage directly instead of reading from disk

**Scenario:** The executor implements `attempt_send_message_layer()` as a live SendMessage call from the Python runner. The artifact is not read from disk — instead the runner sends a message and waits for a response inline. This appears to work in some test scenarios.

**Why DRIFT:** The architectural contract from CONTEXT.md and must_haves.truths[0] is explicit: CC test orchestrators (Plans 09-11) call SendMessage during teardown and write the artifact; the Python runner reads the artifact from disk. Inverting this means the Python runner is now issuing CC API calls directly, breaking the boundary between the Python test layer and the CC orchestration layer. Plans 09-11 integration will break.

**Action:** Escalate. This is a boundary violation, not a style choice. The disk-read contract is the integration point with Plans 09-11.

---

### STOP — debrief_gate.py remains an exit-0 stub

**Scenario:** The executor delivers `debrief_protocol.py` with all four cascade layers defined and tests passing, but `debrief_gate.py` still exits 0 unconditionally (the pre-existing stub). The tests for `debrief_protocol.py` mock the hook layer so they pass without exercising the real gate.

**Why STOP:** The entire purpose of this plan is to fix the debrief from stub to working blocking hook. If `debrief_gate.py` still exits 0, the cascade has no blocking Layer 2 — agents can terminate before the debrief is captured. Plans 07 and 08 will receive empty or absent debrief signals. This is not a partial delivery; it is the pre-plan state with a passing test suite that validates nothing.

**Action:** Reject. Do not accept. Verify `debrief_gate.py` directly: `grep "sys.exit(2)" tests/workflow_harness/hooks/debrief_gate.py` must return at least one match.
