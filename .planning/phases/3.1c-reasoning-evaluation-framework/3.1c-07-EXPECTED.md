# Expected Results — Plan 3.1c-07: Reasoning Evaluator + Meta-Evaluation

**Phase:** 3.1c-reasoning-evaluation-framework
**Plan:** 07

## Fundamental Deliverable

This plan delivers a working LLM-based reasoning evaluator with blind-then-debate protocol and per-category prompt templates so that the 3.1c evaluation framework has a valid scoring signal replacing the heuristic evaluator whose 7 dimensions were empirically proven to have NONE validity in phase 3.1e. Without it, all downstream reasoning scores are noise — the evaluation pipeline (3.1c-08) has no scoring engine, and the skill/agent evaluation plans (3.1c-09, 3.1c-10) cannot produce meaningful quality judgments. Success means `_llm_evaluate_dimension()` produces non-degenerate per-dimension LLM scores, the blind-then-debate protocol resolves disagreements above 10 points, Phase 0 self-consistency and taxonomy gates are recorded as passed, and no code path falls back to keyword heuristic scoring. Failure means the evaluator exists as a file but reasons identically to the discarded heuristic — either because scoring is delegated to keyword matching via a fallback, because the debate protocol exists in name but both evaluators share session state, or because the self-consistency gate was never run and is unrecorded.

## Drift Detection

### OK — Rebuttal granularity pilot outcome changes debate structure

**Scenario:** The plan includes an explicit pre-implementation pilot (ADV-2-02) on 6 calibration transcripts comparing batch vs. per-dimension rebuttal granularity. The executor runs this pilot and finds batch rebuttal produces >80% specific rebuttals with no halo effect — so the debate protocol operates at batch granularity rather than per-dimension.

**Why OK:** This is a permitted deviation path explicitly designed into the plan. The pilot is required, the decision criteria are defined (>80% specific, deltas < 5pt), and the result must be recorded in an Implementation Decision entry with pilot data. The alignment criterion is that the decision was made from evidence and recorded — not which granularity was chosen.

**Action:** Verify the Implementation Decision "Evaluator Debate Granularity" entry exists with pilot data (transcript count, rebuttal specificity %, halo effect deltas). If missing, the pilot was skipped — request it.

---

### OK — Taxonomy reduced from 7 moves to fewer observable moves

**Scenario:** Phase 0 Gate (a) annotation reveals that real transcripts only show 4 distinguishable move boundaries. The executor reduces the active taxonomy and updates `DIMENSION_TO_MOVE_TYPES` to reference only the observable subset. The orchestration template references 3 moves instead of 4.

**Why OK:** The plan explicitly provides for this outcome: "If <4 distinguishable: merge to observable subset." The reduction is valid as long as it is recorded in `phase0_results.yaml` with Gate (a) outcome and the mapping and templates are consistently updated to reflect the reduction.

**Action:** Verify `phase0_results.yaml` Gate (a) shows `outcome: reduced` with the new subset listed, and that no template or mapping references removed move types.

---

### DRIFT — Phase 0 gates skipped; implementation proceeds directly to full evaluator

**Scenario:** The executor implements `_llm_evaluate_dimension()`, the debate protocol, and all four templates without first running or recording Phase 0 gates. The `phase0_results.yaml` file is absent or contains placeholder data. The rest of the implementation looks correct.

**Why DRIFT:** Phase 0 gates are a hard prerequisite — they exist because 3.1e proved real LLM behavior cannot be assumed from design. Skipping them means the evaluator may be well-implemented but built on unvalidated assumptions about transcript structure and LLM consistency. The self-consistency gate in particular gates whether debate is even safe to enable. The plan's checkpoint task (Task 2) is explicitly blocking and requires human verification of gate results before proceeding.

**Action:** Request Phase 0 execution on real transcripts. Do not accept the evaluator as aligned until `phase0_results.yaml` contains real numeric data for all three gates.

---

### DRIFT — evaluator_tier_used omitted; provenance tracking absent

**Scenario:** The evaluator works correctly for the DUAL tier case. The `EvaluatorDisagreement` model is complete. But `evaluator_tier_used` per-evaluation recording (P17-CSC-03) is missing — neither in the output model nor in the result files. The distribution check (UNAVAILABLE_SENTINEL >10% or DUAL <70% flags) is also absent.

**Why DRIFT:** Tier provenance is the mechanism by which the evaluation pipeline detects infrastructure degradation silently masquerading as evaluation signal. Without it, SINGLE_WITH_UNCERTAINTY results cannot be distinguished from DUAL results in downstream analysis, and the flag thresholds in CONTEXT.md (>10% unavailable = flag) cannot fire. This is a cross-plan contract: 3.1c-08 runner consumes `evaluator_tier_used` for suite-level health reporting.

**Action:** Verify `evaluator_tier_used` field presence in the evaluation output model and confirm it is set in all three fallback tier code paths.

---

### STOP — heuristic still the primary scoring path; LLM call is an annotation only

**Scenario:** The executor adds `_llm_evaluate_dimension()` and it produces a score, but the final `DimensionScore` value is computed by the existing keyword heuristic. The LLM result is stored as an annotation field (`llm_suggestion`) but the weighted average that feeds downstream gates uses the heuristic score. The heuristic is the `primary` scorer; LLM is `advisory`.

**Why STOP:** This inverts the entire purpose of Plan 07. Phase 3.1e proved heuristic dimensions have zero validity as ranking signals. A plan that installs LLM scoring as an annotation while preserving heuristic scoring as the actual signal is not an implementation of the plan — it is a refactoring of the status quo with extra steps. The Fundamental Deliverable is defeated. No downstream plan (3.1c-08, 3.1c-09, 3.1c-10) can produce valid results.

**Action:** Escalate immediately. Do not accept the delivery. The score that enters `DimensionScore.score` and contributes to the weighted average must come from `_llm_evaluate_dimension()`, not from any string-matching logic.

## Reviewer Guidance

**Priority 1 — Verify Phase 0 gate data is real, not constructed.**
The single highest failure risk is Phase 0 gates existing as a file with the right structure but containing invented data. Check that transcript identifiers in `phase0_results.yaml` match real files in the transcript store, that std dev values are consistent with the listed per-run scores, and that gate (a) annotation data references at least 2 transcripts.

**Priority 2 — Confirm no heuristic score path survives in any fallback tier.**
Read `reasoning_evaluator.py` and trace every code path that assigns a value to `DimensionScore.score`. Verify each path leads to either an LLM call result or the `UNAVAILABLE_SENTINEL` (no score emitted). The STRUCTURAL_PROXY label is the only permitted landing zone for keyword logic. One heuristic fallback in an `except` block defeats the plan entirely.

**Priority 3 — Validate debate protocol session isolation.**
Confirm both evaluators use separate `session_id` values and that `B_score_pre_exposure` is captured before A's explanation is transmitted to B. These are not separately testable from behavior — they require reading the subprocess invocation code to verify the session argument is distinct between evaluator A and evaluator B calls.

**Priority 4 — Check template-to-evaluator wiring on BOTH {expected} and {observed}.**
Run a trace through `_llm_evaluate_dimension()` for the case where `context['ground_truth_rubric']` is absent — verify the template renders with a graceful omission of the `## Expected Reasoning Coverage` section, not with a literal `{expected}` string. Then verify the case where it is present fills correctly.

**Priority 5 — Check 3.1c-08 interface contract is honored.**
Read the public method signature of `ReasoningEvaluator.evaluate()` and confirm it matches what 3.1c-08 expects: `transcript`, `contract`, `context` parameters, and return type containing `evaluator_tier_used` and `consistency_class`. A signature mismatch at this boundary will silently pass all unit tests but break the pipeline runner in wave 5.
