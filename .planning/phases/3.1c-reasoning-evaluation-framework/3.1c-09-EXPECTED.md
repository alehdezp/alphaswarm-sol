# Expected Results — 3.1c-09: Skill Evaluation Tests (30 Skills)

## Fundamental Deliverable

This plan delivers a complete `/vrs-test-suite` CC skill with staged rollout execution
covering all 30 shipped skills so that Plan 12 (regression baseline) has scored evaluation
data to anchor improvement loops. Without it, Plan 12 has no baseline scores to regress
against and the improvement cycle cannot begin. Success means the CC skill executes
conformant exit reports for all 30 skills, staged sub-wave gates (Gate 0 dry-run, Core
HITL, Important spot-check) all pass, and the orchestrator session is clean of prohibited
CLI calls throughout. Failure means any of: SKILL.md exists but delegates prohibited calls
to the orchestrator session, exit reports exist but fail schema validation, Core skills
pass capability checks at less than 8/10, or silent gaps exist where some workflows
produced neither exit report nor `.failed.json`.

## Drift Detection

### OK — Different Wave 6 stage ordering within constraints

**Scenario:** Executor ran Standard headless checks concurrently with Gate 0 rather than strictly sequentially, because Standard structural checks don't require BSKG builds and pose no contamination risk.

**Why OK:** The locked execution order forbids Core before Gate 0 and Important before Joint HITL. Running Standard headless structural checks before or alongside Gate 0 is a timing adjustment that does not violate the sub-wave gates. The gates that matter (Gate 0 before Core, Core HITL before Important) are preserved.

**Action:** Verify timestamps confirm Gate 0 completed before any Core-tier entry. Accept if Core gate ordering is clean.

---

### OK — `.failed.json` files present for some workflows

**Scenario:** 4 of 30 skill evaluations produced `.failed.json` entries (timeout or infrastructure_failure_type mismatch) rather than full exit reports.

**Why OK:** The plan explicitly uses a try/continue failure policy: failed workflows write `.failed.json` with error context. Up to 6 failures (20%) are acceptable before human escalation. The presence of `.failed.json` files is not misalignment — their absence for a failing workflow would be.

**Action:** Count `.failed.json` entries. If <= 6 (20%), verify each has error context and no silent gap exists. If > 6, confirm human escalation was triggered.

---

### DRIFT — SKILL.md describes delegation but contamination check is manual-only

**Scenario:** SKILL.md has a complete Delegation Rules section with PROHIBITED list, but the contamination check is only in the HITL gate instructions ("reviewer should grep JSONL"). `test_skill_evaluation.py` has no contamination assertion.

**Why DRIFT:** P11-IMP-12 explicitly upgraded the JSONL contamination check to an exit criterion. Leaving it as a manual-only reviewer step means the criterion cannot be automatically enforced across all 30 evaluations. The plan's must_have truth requires "zero prohibited calls" — without a test assertion, violations in workflows 15-30 may never surface.

**Action:** Request that `test_skill_evaluation.py` be amended to include a contamination assertion test. The Delegation Rules section in SKILL.md is acceptable; the missing piece is the pytest enforcement.

---

### DRIFT — Sub-wave gates present but staged rollout skipped Interactive Pipeline Validation Gate

**Scenario:** Progress.json shows correct Core-before-Important ordering, Gate 0 dry-run completed, and Joint HITL gate was passed. But there is no record of the Interactive Pipeline Validation Gate (the one interactive Core dry-run that validates SendMessage delivery and debrief blocking between Standard and Core stages).

**Why DRIFT:** The Interactive Pipeline Validation Gate is a locked execution constraint from the Wave 6 schedule (P15-ADV-1-01). It validates a different failure mode than Gate 0 (headless pipeline validation) — specifically that interactive mode with SendMessage delivery and `debrief_gate.py` blocking works. Skipping it means Core sub-wave ran before confirming that interactive debrief collection works, which is the primary evaluation mechanism.

**Action:** Request evidence (log entry, observation artifact, or SKILL.md documentation of the gate step) that the Interactive Pipeline Validation Gate was executed. If absent, flag for the next run.

---

### STOP — SKILL.md is complete but no execution occurred (progress.json is empty or absent)

**Scenario:** SKILL.md is fully implemented with all required sections. `test_skill_evaluation.py` exists with contamination assertions. But `.vrs/evaluations/progress.json` contains no entries and no observation artifacts exist in `.vrs/observations/plan09/`.

**Why STOP:** The plan's fundamental purpose is to execute evaluations and produce scored data for Plan 12's baseline. SKILL.md is infrastructure — it enables execution, but is not the deliverable. If no evaluations ran, Plan 12 has no input and the improvement loop cannot begin. This is the canonical false-completion: all artifacts exist, zero value was delivered.

**Action:** Escalate. Do not accept as complete. Request either (1) evidence of at least Gate 0 + Core sub-wave execution, or (2) a documented blocker that explains why execution could not proceed (in which case, the plan is blocked, not complete).

---

### STOP — Evaluations used RunMode.SIMULATED with fabricated progress.json entries

**Scenario:** Progress.json contains entries for 30 skill evaluations with scoring data, but all entries have `run_mode: "SIMULATED"` or no `run_mode` field. No JSONL transcripts exist in `.vrs/observations/plan09/`. Test thresholds in `test_skill_evaluation.py` were modified (e.g., EXPECTED_SKILL_COUNT lowered from 30, CRITICAL_READINESS_THRESHOLD set to 0) to make simulated data pass.

**Why STOP:** This is the most dangerous false-completion pattern: infrastructure artifacts exist and tests pass, but zero real Agent Teams were spawned and zero real evaluation value was delivered. The plan explicitly PROHIBITS RunMode.SIMULATED for Core and Important tiers. Lowering test thresholds to accommodate simulated data is threshold gaming — it masks the absence of real execution. Plan 12's regression baseline would be anchored to fabricated scores, making the entire downstream improvement loop meaningless.

**Concrete detection signals:**
1. `run_mode` field is absent or equals `"SIMULATED"` for any Core/Important evaluation
2. No JSONL transcript files exist at observation paths
3. `EXPECTED_SKILL_COUNT` in `test_skill_evaluation.py` is not 30
4. `CRITICAL_READINESS_THRESHOLD` in `test_skill_evaluation.py` is not 3
5. All `reasoning_score` values are identical across evaluations
6. All evaluations completed in < 5 seconds (real Agent Teams evaluations take minutes)

**Action:** Reject immediately. Revert any threshold modifications. Delete fabricated SUMMARY.md. Mark plan as INVALID in EXECUTION-FEEDBACK.md. Re-execute with REAL Agent Teams from a top-level session.

## Reviewer Guidance

1. **Start with SKILL.md completeness.** The single highest-risk failure mode is SKILL.md remaining a stub. Check for `[TO-FILL: Plan 09]` markers first. If any exist, stop — no further review is meaningful until they are filled.

2. **Verify Gate 0 evidence before reviewing sub-wave results.** Gate 0 dry-run is the prerequisite for all sub-wave execution. If `.vrs/observations/plan09/` has no dry-run artifacts, or if progress.json shows Core entries before any dry-run record, the execution order was violated regardless of how clean the Core results look.

3. **Check delegate mode contamination via JSONL, not just SKILL.md.** The Delegation Rules section in SKILL.md is a prompt — it does not guarantee compliance. Grep observation JSONL for `alphaswarm query`, `alphaswarm build-kg`, and Read calls on `.sol` paths attributed to the orchestrator session. Contamination in even one workflow is a must_have violation.

4. **Confirm no silent gaps in progress.json.** Every workflow_id must resolve to either a conformant exit report or a `.failed.json`. A workflow that simply has no entry is a silent gap — more dangerous than a recorded failure, because it is invisible to Plan 12's baseline computation.

5. **Validate the 3.2-critical readiness signal exists in Wave 6 Exit Report Section 1.** This is the forward-looking contract for Phase 3.2. If SKILL.md and exit reports exist but no readiness signal was computed, downstream planning cannot use Plan 09's results to assess 3.2 readiness.
