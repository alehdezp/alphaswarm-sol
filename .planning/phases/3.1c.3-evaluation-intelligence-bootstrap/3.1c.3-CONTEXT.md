<domain>
## Phase Boundary

**Goal:** Activate the Tier 2 Intelligence layer AND prove it works ‚Äî make the testing framework self-healing, self-improving, self-diagnostic, graph-aware, planning-integrated, and capable of real-time hotfixing. This phase builds 10 intelligence modules, deploys observation hooks (GAP-1), validates the framework against itself, and integrates with the MSD/GSD planning system. After this phase, the testing framework is a proven, self-correcting system ‚Äî not theoretical infrastructure.

**What this phase delivers:**
- Production deployment of 5 Claude Code observation hooks (PreToolUse, PostToolUse, SubagentStart, SubagentStop, Stop) + JSONL writer into `.claude/settings.json`
- Per-move reasoning decomposition starting with 3 moves (HYPOTHESIS_FORMATION, EVIDENCE_INTEGRATION, CONCLUSION_SYNTHESIS) ‚Äî remaining 4 deferred until 50+ evaluations
- Coverage radar wired to 4 evaluation contract templates (not 54 individual contracts) with severity-weighted gap prioritization
- Scenario synthesizer v1 that analyzes prompt text, cross-references test corpus, and generates scenario YAML stubs for manual review
- Contract healer with bimodal detection (ceiling/floor/zero-variance) + tier manager with demotion proposal YAML, both wired into runner post-hook
- Counterfactual replayer trajectory recording + fingerprinter schema with `is_active` stub (activates after 20 runs)
- **Framework self-validation** ‚Äî meta-test suite that RUNS the pipeline, tests both spawn paths (Task() vs TeamCreate), validates anti-fabrication triggers, and proves workspace rollback produces repeatable results
- **Graph-aware diagnostics** ‚Äî intelligence module that understands WHY graph queries fail, WHY patterns don't match, and recommends specific fixes (graph builder / query routing / pattern definition / contract structure)
- **Workspace rollback integration** ‚Äî Jujutsu snapshot/restore for repeatable test scenarios with regression anchoring and variance tracking
- **Recommendation engine** ‚Äî failure-to-fix mapping that translates evaluation failures into specific actionable fixes targeting prompts, patterns, graph builder, or query routing
- **MSD/GSD planning integration** ‚Äî bridge that translates test gaps into planning actions (/msd:insert-phase, /msd:improve-phase), with experiment-before-continue protocol
- **Real-time hotfix protocol** ‚Äî classification-based system that fixes small issues immediately (typos, missing imports, stale configs) via subagent spawning with safety guardrails

**What this phase does NOT deliver:**
- Metaprompting feedback system (GAP-4) ‚Äî deferred to 3.1f or post-3.1f
- Insight propagator (needs 20+ runs/workflow + multi-agent debate data ‚Äî post Phase 4)
- Evaluator improver (meta-improvement premature before basic loop works ‚Äî post 3.1f)
- Composition tester (needs working Agent Teams Debate ‚Äî post Phase 4)
- Adversarial auditor (needs mature evaluation data ‚Äî post 3.1f + cycles)
- Feedback ingester (blocked on real audit outcomes ‚Äî post Phase 3.2)
- Remaining 4 reasoning moves (QUERY_FORMULATION, RESULT_INTERPRETATION, CONTRADICTION_HANDLING, SELF_CRITIQUE ‚Äî add after 50 evaluations show which moves discriminate)
- Fully autonomous test generation (v1 generates stubs for manual review)
- Auto-applied hotfixes to production agent prompts (requires sandbox + human approval)

**Trigger:** 6-agent alignment review found that 8 of 10 Tier 2 Intelligence modules had no phase to deliver them. The testing framework was ~25% complete ‚Äî Tier 1 (Engine) plans were mostly done but Tier 2 (Intelligence) was 90% unbuilt.

**Why hooks belong HERE (not in 3.1c or 3.1c.1):**
- 3.1c Plan 01 DESIGNED the hooks (data structures, event types, JSONL format)
- 3.1c.1 fixes CLI so hooks have something real to capture
- 3.1c.2 hardened the harness so agents can't fabricate
- 3.1c.3 DEPLOYS the hooks because the intelligence modules (Plans 02-06) are their consumers
- Without deployed hooks, Tier 2 intelligence has no data to work with

### Plan 08: Graph-Aware Diagnostics Module [MEDIUM confidence]
**Intent:** Build an intelligence module that understands WHY graph queries fail, WHY patterns don't match, and HOW to fix graph/query/pattern issues
**What it delivers:**
- `graph_diagnostics.py` in `intelligence/` ‚Äî new module
- **Query failure analysis:** When `alphaswarm query` returns 0 results, diagnose:
  - Pattern not in vulndocs? ‚Üí "Missing pattern: add via /vrs:add-vulnerability"
  - Graph not built? ‚Üí "No graph found: run alphaswarm build-kg <contract>"
  - Query routing failed? ‚Üí "Query type X not supported: use --mode=raw or rephrase"
  - Contract too simple? ‚Üí "Graph has N nodes, insufficient for pattern X (requires M+ nodes)"
- **Pattern match analysis:** When a pattern should match but doesn't, analyze:
  - Missing semantic operations (e.g., TRANSFERS_VALUE_OUT not detected)
  - Missing edges (e.g., no control flow edge between guard check and transfer)
  - Label mismatch (e.g., function labeled ACCESS_CONTROL but pattern expects VALUE_TRANSFER)
- **Graph structure report:** For each contract graph, produce a structural summary:
  - Node count by type (function, modifier, state_var, event)
  - Edge count by type (calls, reads, writes, control_flow)
  - Semantic operation coverage (which operations detected vs which patterns expect)
  - Comparison to known-good graph structures from test corpus
- **Diagnostic recommendations:** Output structured JSON with:
  - `diagnosis`: what's wrong
  - `root_cause`: graph builder / query routing / pattern definition / contract structure
  - `fix`: specific actionable fix (e.g., "Add property X to builder", "Fix query routing for type Y")
  - `severity`: how many evaluations this affects
- `is_active()` gated on: graph build successful + at least 1 query failure observed
- Registered in `intelligence/__init__.py` module registry
**What problem it solves:** Framework scores agents poorly but doesn't understand that the graph itself is the problem. Without graph diagnostics, improvement efforts target prompts when the graph builder or query routing needs fixing.
**Confidence:** MEDIUM ‚Äî graph structure analysis is well-defined; mapping failures to fixes requires heuristics
**Depends on:** Plan 01 (needs hook data to see query failures), Plan 02 (needs per-move scores to correlate reasoning failures with graph issues)
**~LOC:** ~200

### Plan 09: Workspace Rollback Integration [MEDIUM confidence]
**Intent:** Use Jujutsu workspace snapshot/restore for REPEATABLE test scenarios with regression guarantee
**What it delivers:**
- **Snapshot/restore protocol:** Before each evaluation session:
  1. `jj workspace add eval-{id}` ‚Äî create isolated workspace
  2. `jj snapshot` ‚Äî capture clean state
  3. Run agent session in workspace (hooks, CLI, evaluation)
  4. Record results
  5. `jj restore` ‚Äî rollback to clean state
  6. Re-run SAME scenario ‚Üí verify results are within acceptable variance (non-deterministic tolerance)
  7. `jj workspace forget eval-{id}` ‚Äî cleanup
- **Repeatability metric:** For each scenario, track score variance across 3 runs. If variance > threshold, flag as non-repeatable and investigate (is it LLM variance? graph instability? hook timing?)
- **Regression anchor:** Use rollback to re-run against a FIXED contract+scenario pair. If score drops between framework versions, it's a framework regression (not agent or graph regression)
- **WorkspaceManager extensions:**
  - `snapshot()` method ‚Äî capture current workspace state as named snapshot
  - `restore(snapshot_name)` method ‚Äî rollback to named snapshot
  - `run_repeatable(scenario, n=3)` method ‚Äî run scenario N times with rollback between, return variance stats
- **Integration with evaluation_runner:** Runner's `run()` method accepts optional `workspace_id` parameter. If provided, all file operations are scoped to that workspace.
**What problem it solves:** Without workspace rollback, each test run modifies the environment (writes graphs, observations, hooks output). Subsequent runs in the same environment may produce different results due to stale artifacts. Rollback ensures clean-room execution.
**Confidence:** MEDIUM ‚Äî Jujutsu workspace rollback is well-documented; integration with evaluation runner requires care
**Depends on:** Plan 01 (hooks must be deployable per-workspace), Plan 07 (framework self-validation confirms runner works)
**~LOC:** ~120

### Plan 10: Recommendation Engine [MEDIUM confidence]
**Intent:** Build a system that doesn't just score agents but RECOMMENDS specific fixes to prompts, patterns, graph builder, and query routing ‚Äî with root cause mapping
**What it delivers:**
- `recommendation_engine.py` in `intelligence/` ‚Äî new module
- **Failure-to-fix mapping:** Maintains a mapping from failure types to fix categories:
  | Failure Type | Root Cause | Fix Target | Example Fix |
  |-------------|-----------|-----------|-------------|
  | Low HYPOTHESIS_FORMATION score | Agent doesn't form clear hypotheses | Agent prompt | Add "State your hypothesis before querying" to prompt |
  | Low EVIDENCE_INTEGRATION score | Agent ignores graph results | Agent prompt | Add "Cite specific node IDs from query results" |
  | Query returns 0 results | Pattern not in vulndocs | VulnDocs | "Run /vrs:add-vulnerability for category X" |
  | Graph missing edges | Builder doesn't extract relationship | Graph builder | "Add property X extraction to builder" |
  | Agent uses Python imports | CLI doesn't work from workspace | CLI | "Fix vulndocs resolution (3.1c.1 Plan 01)" |
  | Score variance > 30pts | Evaluation contract too vague | Evaluation contract | "Add specific rubric criteria for dimension Y" |
- **Recommendation output:** After each evaluation, produce `recommendation.json`:
  ```json
  {
    "workflow_id": "skill-vrs-investigate",
    "weakest_move": "EVIDENCE_INTEGRATION",
    "root_cause": "agent_prompt",
    "specific_fix": "Add 'You MUST cite node IDs from query results' to vrs-investigate SKILL.md",
    "confidence": "medium",
    "supporting_evidence": ["score=42", "move_scores={...}", "graph_diagnostics={...}"],
    "estimated_impact": "Score improvement of 10-20 points on EVIDENCE_INTEGRATION dimension"
  }
  ```
- **Aggregated recommendations:** After N evaluations, produce prioritized list:
  - Group by root_cause (prompt vs graph vs CLI vs evaluation)
  - Sort by impact √ó frequency
  - Output to `.vrs/evaluations/recommendations-report.json`
- **Agent interview integration:** When an agent fails, the recommendation engine formulates specific questions for the debrief (from 3.1c.2 Plan 05):
  - "Your EVIDENCE_INTEGRATION score was 42. What graph nodes did you cite? Why didn't you use the query results?"
  - "You ran 0 queries. What prevented you from querying the graph?"
  - Debrief responses are fed BACK into the recommendation engine to refine the failure-to-fix mapping
- `is_active()` gated on: 3+ evaluation results with per-move scores (needs Plan 02)
- Registered in `intelligence/__init__.py`
**What problem it solves:** Current framework identifies WHAT failed but not WHY or HOW TO FIX IT. Without specific recommendations, humans must manually diagnose every failure. This closes the gap between "we know it's broken" and "here's what to do."
**Confidence:** MEDIUM ‚Äî failure mapping is heuristic; quality improves with accumulated data
**Depends on:** Plan 02 (per-move scores), Plan 08 (graph diagnostics), Plan 05 (contract healer for evaluation contract fixes)
**~LOC:** ~200

### Plan 11: MSD/GSD Planning Integration [LOW-MEDIUM confidence]
**Intent:** Make the testing framework aware of the MSD/GSD planning system so it can suggest and create phases, plans, and improvements through the same framework used for development
**What it delivers:**

- **Skill awareness:** Framework knows about `/msd:insert-phase`, `/msd:improve-phase`, `/msd:plan-phase`, `/msd:add-phase` and their triggers. When a gap is identified, it generates a structured suggestion:
  ```json

    "suggested_phase": {
      "name": "Property Gap Sprint ‚Äî Round 2",

      "plans": [
  ```

- **Phase execution awareness:** After each phase executes, the framework:
  1. Reads phase CONTEXT.md exit criteria
  2. Runs relevant evaluation scenarios against the phase deliverables
  3. Produces a gap analysis: what was promised vs what was delivered
  4. If gaps found: suggests `/msd:improve-phase` with specific items to address
- **Experiment-before-continue protocol:** Before any infrastructure change, the framework can:

- **Integration with scenario_synthesizer:** When synthesizer (Plan 04) generates new scenarios that reveal gaps, planning_bridge translates those gaps into actionable MSD/GSD commands

**What problem it solves:** The testing framework and planning system are disconnected. Test results don't feed back into the planning process. When testing reveals a gap, a human must manually create phases/plans. This bridge enables the framework to suggest planning actions, reducing the manual gap.
**Confidence:** LOW-MEDIUM ‚Äî MSD/GSD skill invocation is well-defined; automated planning suggestions require careful validation to avoid noise
**Depends on:** Plans 03 (coverage radar), 04 (scenario synthesizer), 08 (graph diagnostics), 10 (recommendation engine)

### Plan 12: Real-Time Hotfix Protocol [LOW-MEDIUM confidence]

**What it delivers:**
- `hotfix_protocol.py` in `intelligence/` ‚Äî new module
- **Hotfix classification:** After each evaluation, classify failures into:
  | Classification | Action | Example |
  |---------------|--------|---------|
  | HOTFIX_NOW | Spawn subagent to fix immediately | "Missing import in test file", "Typo in pattern YAML" |
  | PLAN_NEXT | Add to current phase as new plan | "New intelligence module needed for X" |
  | PHASE_NEEDED | Suggest new phase via planning_bridge | "Major graph builder overhaul required" |
  | INVESTIGATE | Spawn investigation subagent | "Unclear why query returns wrong results" |
  | DEFER | Log for human review | "Architectural decision needed" |
- **Hotfix execution:**
  1. Framework detects a HOTFIX_NOW failure (small, well-defined, safe)
  2. Spawns a Task() subagent with focused context:
     - The specific file to fix
     - The exact failure description
     - Expected outcome after fix
     - Maximum scope: 1 file, < 20 LOC change
  3. Subagent applies fix
  4. Framework re-runs the evaluation scenario
  5. If fix works (score improves): commit the fix with attribution
  6. If fix doesn't work: revert, escalate to PLAN_NEXT
- **Safety guardrails:**
  - Hotfixes ONLY for test infrastructure, patterns, or evaluation contracts ‚Äî NEVER for production agent prompts (that requires sandbox + human approval)
  - Maximum 3 hotfix attempts per evaluation session ‚Äî prevent infinite fix loops
  - All hotfixes logged in `.vrs/evaluations/hotfix-log.jsonl` with before/after evidence
  - Human can disable hotfixing entirely via config flag
- **Investigation subagent:** When classification is INVESTIGATE:
  - Spawns an Explore subagent to read relevant code, check graph state, analyze transcripts
  - Subagent produces structured investigation report
  - Report feeds into recommendation engine (Plan 10)
- `is_active()`: always active (hotfixing doesn't need accumulated data)
- Registered in `intelligence/__init__.py`
**What problem it solves:** Currently all failures are logged and deferred. Simple issues (typos, missing imports, stale configs) could be fixed immediately, saving time. The classification system ensures only safe, small fixes are applied automatically.
**Confidence:** LOW-MEDIUM ‚Äî hotfix classification requires judgment; safety guardrails need careful design
**Depends on:** Plan 07 (framework self-validation ‚Äî must be able to verify fixes work), Plan 10 (recommendation engine provides the failure analysis)
**~LOC:** ~200
