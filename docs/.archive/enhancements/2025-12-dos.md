# BSKG Builder Enhancements (2025-12-29)

## Overview

This document describes the enhancements made to the AlphaSwarm.sol system based on the comprehensive DoS test suite analysis. All enhancements maintain **100% backward compatibility** with existing code.

## Motivation

The comprehensive DoS vulnerability testing identified 5 key limitations in the BSKG builder that prevented accurate detection of certain DoS attack patterns. These enhancements address those limitations while maintaining the system's deterministic, evidence-based approach.

## Enhancements Summary

### Priority 1: High Impact Enhancements

#### 1. `has_require_bounds` Property
**File**: `src/true_vkg/kg/builder.py`
**Method**: `_has_require_bounds(require_exprs, parameter_names)`

**Purpose**: Distinguish between truly unbounded loops and properly bounded pagination functions.

**Detection Logic**:
- Checks for `require()` statements that validate parameter bounds
- Looks for patterns like `require(param <= MAX)` or `require(end - start <= MAX)`
- Detects bound operators: `<=`, `<`, `>=`, `>`
- Identifies bound keywords: `max`, `limit`, `bound`, or numeric constants

**Example**:
```solidity
function paginatedAccess(uint256 start, uint256 end) public {
    require(end - start <= MAX_PAGE_SIZE);  // ← Detected as has_require_bounds = true
    for (uint256 i = start; i < end; i++) {
        // Safe pagination
    }
}
```

**Impact**: Reduces false positives for `dos-unbounded-loop` pattern.

---

#### 2. `uses_transfer` and `uses_send` Properties
**File**: `src/true_vkg/kg/builder.py`
**Methods**: `_uses_transfer(fn)`, `_uses_send(fn)`

**Purpose**: Detect usage of deprecated `.transfer()` and `.send()` methods that have fixed 2300 gas stipend.

**Detection Logic**:
- Scans function nodes for `.transfer(` and `.send(` expressions
- Checks IR (Intermediate Representation) for transfer/send operations
- Boolean flags set to `True` if detected

**Example**:
```solidity
function distribute() external {
    recipient.transfer(amount);  // ← uses_transfer = true
}

function tryPay() external {
    bool success = recipient.send(amount);  // ← uses_send = true
}
```

**Impact**: Enables detection of `dos-transfer-in-loop` pattern (SWC-113).

---

#### 3. `has_strict_equality_check` Property
**File**: `src/true_vkg/kg/builder.py`
**Method**: `_has_strict_equality_check(require_exprs)`

**Purpose**: Detect gridlock attack vectors caused by strict equality checks on `address(this).balance`.

**Detection Logic**:
- Looks for `require()` or `assert()` statements
- Checks for strict equality operator `==`
- Identifies balance checks: `address(this).balance`, `this.balance`

**Example**:
```solidity
function withdraw() public {
    require(address(this).balance == expectedBalance);  // ← has_strict_equality_check = true
    // Vulnerable to gridlock: attacker can forcibly send 1 wei to break equality
}
```

**Impact**: Enables detection of `dos-strict-equality` pattern (CWE-1077, Edgeware Lockdrop attack).

---

### Priority 2: Medium Impact Enhancements

#### 4. Enhanced `has_external_calls` Property
**File**: `src/true_vkg/kg/builder.py`
**Location**: Line 242 in `_add_functions()`

**Purpose**: Include low-level calls (`.call`, `.delegatecall`, `.staticcall`) in external call detection.

**Previous Logic**:
```python
"has_external_calls": bool(external_calls),  # Only high-level calls
```

**Enhanced Logic**:
```python
has_external_calls = bool(external_calls) or bool(low_level_calls)  # Both high and low-level
```

**Example**:
```solidity
function unsafeCall(address target) external {
    target.call{value: msg.value}("");  // ← Now detected as has_external_calls = true
}
```

**Impact**: More accurate detection of `dos-revert-failed-call` and reentrancy patterns.

---

#### 5. `state_mutability` Property (Normalized)
**File**: `src/true_vkg/kg/builder.py`
**Method**: `_normalize_state_mutability(fn)`

**Purpose**: Provide consistent state mutability classification for all functions.

**Detection Logic**:
1. Check Slither's `.pure` property → return "pure"
2. Check Slither's `.view` property → return "view"
3. Check Slither's `.payable` property → return "payable"
4. Check `state_mutability` attribute → normalize to standard values
5. Default → return "nonpayable"

**Valid Values**: `"pure"`, `"view"`, `"payable"`, `"nonpayable"`

**Example**:
```solidity
function getAllUsers() public view returns (User[] memory) {
    // state_mutability = "view"
}

function deposit() public payable {
    // state_mutability = "payable"
}
```

**Impact**: Enables detection of `dos-array-return-unbounded` pattern for view functions.
