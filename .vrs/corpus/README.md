# VRS Validation Corpus

This corpus provides the statistical foundation for GA validation (Phase 7.3). Built to ensure pattern detection accuracy with heavy emphasis on false positive prevention.

## Current Corpus Status

- **Total contracts:** 818
- **Safe contracts:** 569 (69.6%)
- **Vulnerable contracts:** 249 (30.4%)
- **Target ratio:** 80% safe / 20% vulnerable
- **Minimum size:** 150+ contracts (exceeded)

## Structure

```
.vrs/corpus/
  corpus.db                    # SQLite database (lean metadata + indexes)
  contracts/
    safe/                      # 80% - Audited-clean contracts for FP testing
      oz-contracts/            # OpenZeppelin (339 contracts)
      solady/                  # Vectorized's gas-optimized (109 contracts)
      solmate/                 # Transmissions11 building blocks (59 contracts)
      uniswap-v3/              # Battle-tested AMM (62 contracts)
    vulnerable/                # 20% - Contracts with known vulnerabilities
      dvdefi-v3/               # Damn Vulnerable DeFi (53 contracts)
      ethernaut/               # OpenZeppelin CTF (171 contracts)
      not-so-smart/            # Trail of Bits examples (25 contracts)
    adversarial/               # Reserved - Hand-crafted FP traps
    mutations/                 # Reserved - Programmatic variants
    recent-audits/             # Reserved - Post-2025 audit protocols
  ground-truth/
    {source}.yaml              # Per-source ground truth stubs
    audits-2025.yaml           # Audit-derived ground truth (16 entries)
  metadata/
    sources.yaml               # Source manifest with pinned commits
    urls.yaml                  # URL provenance ledger
```

## Database Schema (CorpusDB)

The database intentionally remains **lean** - it stores only what's needed for indexing and querying. Rich metadata lives in YAML sidecars for human readability and easy editing.

### contracts
| Column | Type | Description |
|--------|------|-------------|
| contract_id | TEXT | Primary key (stable hash: `{source}-{hash[:8]}`) |
| source_path | TEXT | Relative path from corpus root |
| category | TEXT | `safe`, `vulnerable`, `adversarial`, `mutations` |
| vulnerability_class | TEXT | Comma-separated vuln classes (nullable) |
| added_date | TEXT | ISO timestamp when added |
| corpus_version | TEXT | Corpus version (e.g., "v1") |

**Why contract_id is a hash:** Stable across re-ingestion. Derived from `source_id:relative_path:commit_sha`, so same file at same commit always gets same ID.

### ground_truth
| Column | Type | Description |
|--------|------|-------------|
| gt_id | TEXT | Primary key (e.g., "gt-penpie-reentrancy") |
| contract_id | TEXT | References contracts.contract_id |
| function_id | TEXT | Target function (nullable) |
| vulnerability_pattern | TEXT | Pattern ID (e.g., "reentrancy-classic") |
| severity | TEXT | `critical`, `high`, `medium`, `low` |
| code_location | TEXT | Location string (e.g., "Contract.sol:L42") |
| verified_by | TEXT | Who verified this label (nullable) |
| verification_date | TEXT | When verified (nullable) |

### test_runs
| Column | Type | Description |
|--------|------|-------------|
| run_id | TEXT | Primary key |
| run_date | TEXT | ISO timestamp |
| corpus_version | TEXT | Corpus version tested |
| model_used | TEXT | Model identifier (nullable) |
| precision | REAL | Overall precision |
| recall | REAL | Overall recall |
| recall_critical | REAL | Recall for critical findings |
| f1_score | REAL | F1 score |
| cost_usd | REAL | Run cost (nullable) |
| notes | TEXT | Notes (nullable) |

### corpus_metadata
| Key | Description |
|-----|-------------|
| schema_version | Current schema version (1.0) |

## DB vs YAML Sidecar

| Data | Location | Reason |
|------|----------|--------|
| Contract ID, path, category | DB | Fast querying, indexing |
| Source commit, vulnerability classes | DB | Stable reference |
| Detailed ground truth descriptions | YAML | Human-editable, version-controlled |
| Provenance URLs | YAML (urls.yaml) | Audit trail |
| Source manifest | YAML (sources.yaml) | Reproducible ingestion |

**Rule:** DB for machine access, YAML for human context.

## Ground Truth YAML Schema

Ground truth stubs are generated by `corpus_ingest.py` and follow this schema (aligned with `config_schema.py`):

```yaml
source_id: dvdefi-v3
repo_url: https://github.com/...
commit_sha: 64fddf9f...
category: vulnerable
vulnerability_classes:
  - reentrancy
  - access_control
revalidation_age_days: 90  # Entries older than this need re-verification
contracts:
  - path: VulnerableVault.sol
    has_vulnerability: true
    ground_truth:
      - pattern: reentrancy-classic
        severity: high
        location: withdraw:L42
        description: "CEI violation in withdraw function"
        confidence_min: 0.8
    notes: "Classic reentrancy in withdraw"
```

## Vulnerability Categories

Normalized from audit reports (maps to VulnDocs taxonomy):

- **reentrancy** - Classic, cross-function, cross-contract, read-only
- **access_control** - Missing checks, role bypass, ownership issues
- **oracle** - Price manipulation, stale data, sequencer issues
- **arithmetic** - Overflow, underflow, precision loss, rounding
- **flash_loan** - Flash loan enabled attacks
- **logic** - Invariant violations, state corruption, configuration errors
- **upgrade** - Storage collision, initialization issues
- **governance** - Proposal manipulation, voting issues
- **cross_chain** - Bridge vulnerabilities, message validation

## Ingestion

```bash
# Ingest all sources
uv run python scripts/corpus_ingest.py

# Preview only (dry-run)
uv run python scripts/corpus_ingest.py --dry-run

# Ingest specific category
uv run python scripts/corpus_ingest.py --category vulnerable

# Show statistics
uv run python scripts/corpus_ingest.py --stats

# List sources
uv run python scripts/corpus_ingest.py --list-sources
```

## Provenance

Every source must be logged in `metadata/urls.yaml` with:
- `url` - Repository or report URL
- `accessed_at` - ISO timestamp
- `query` - Search query that found this source
- `category` - Corpus category
- `agent` - Which agent/script performed ingestion
- `extracted` - Whether content was extracted
- `notes` - Commit hash, additional context
